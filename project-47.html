PROJECT 47: BUILDING A LAMBDA ARCHITECTURE — BATCH LAYER WITH HADOOP

🔴 HYPOTHETICAL PROBLEM

SCENARIO:
→ You are the chief data architect at a ride-sharing platform
→ Your Hadoop cluster processes 1.2PB of trip data daily (Project 46 optimized)
→ The batch pipeline produces these analytics tables every night:
    - Driver earnings per city per hour
    - Surge pricing zones (demand vs supply)
    - Rider wait time averages per neighborhood
    - Trip completion rates by driver rating
    - Revenue dashboards per market

EVERYTHING WORKS — BUT THE BUSINESS HAS A NEW DEMAND:

CEO (Monday morning meeting):
  "Our competitors show drivers their earnings in REAL-TIME.
   Our drivers see yesterday's numbers. They're switching platforms.
   
   I need:
   1. Real-time driver earnings (updated every 30 seconds)
   2. Live surge pricing (updated every minute)  
   3. Current wait times (updated every 2 minutes)
   4. BUT ALSO: accurate historical reports (daily/weekly/monthly)
   
   The real-time view doesn't need to be PERFECT — 
   close enough is fine. But the historical reports 
   MUST be 100% accurate for financial audits."

YOUR ANALYSIS:
→ Current batch pipeline: 100% accurate, but 6-12 hours old
→ Real-time requirement: ~30 seconds freshness, can be approximate
→ Historical requirement: 100% accurate, auditable
→ These two requirements CONFLICT:
    - Accuracy requires processing ALL data (batch — slow)
    - Freshness requires processing data AS IT ARRIVES (stream — fast)
    - You CANNOT have both from a single system

THE INSIGHT:
→ You need TWO systems working in PARALLEL:
    1. BATCH LAYER: Processes ALL data periodically → accurate results
    2. SPEED LAYER: Processes NEW data in real-time → fresh results
    3. SERVING LAYER: Merges both → accurate AND fresh

→ This is the LAMBDA ARCHITECTURE
→ Invented by Nathan Marz (creator of Apache Storm)
→ Hadoop is the NATURAL batch layer
→ HBase (Project 27) is the NATURAL serving layer

THREE FUNDAMENTAL PRINCIPLES:

  PRINCIPLE 1: IMMUTABLE MASTER DATASET
  → ALL raw events are stored in HDFS forever
  → NEVER modified or deleted
  → Append-only — new data is added, old data stays
  → If processing logic had a bug → RECOMPUTE from master dataset
  → This is the SINGLE SOURCE OF TRUTH

  PRINCIPLE 2: BATCH VIEWS ARE PRECOMPUTED
  → Batch layer runs MapReduce on ENTIRE master dataset
  → Produces "batch views" — precomputed query results
  → Stored in serving layer (HBase) for fast access
  → Recomputed periodically (daily, hourly, etc.)
  → ALWAYS accurate — processes ALL data

  PRINCIPLE 3: SPEED LAYER FILLS THE GAP
  → Between batch runs, new data arrives
  → Speed layer processes ONLY this new data
  → Produces "real-time views" — approximate, incremental
  → Stored in serving layer alongside batch views
  → When new batch view arrives → real-time view is DISCARDED
  → Real-time view = temporary bridge until next batch completes

═══════════════════════════════════════════════════════════
COMPONENT 2: BATCH VIEW COMPUTATION
═══════════════════════════════════════════════════════════

BATCH VIEW = function(ALL master data)

  For driver earnings:
    batch_view = MapReduce over ALL trip events
                 GROUP BY (driver_id, city, hour)
                 SUM(fare_amount)

  TWO STRATEGIES FOR BATCH COMPUTATION:

  STRATEGY A: FULL RECOMPUTATION
  → Process ALL historical data every batch run
  → Simplest logic — just re-run MapReduce on entire dataset
  → PROS: Always correct, handles late-arriving data, handles corrections
  → CONS: Expensive — processing months/years of data daily
  → WHEN: Dataset small enough, or computation must be 100% fresh

  STRATEGY B: INCREMENTAL + PERIODIC FULL
  → Daily batch: process only yesterday's data
  → Weekly batch: recompute last 7 days (catches late data)
  → Monthly batch: recompute last 30 days (handles corrections)
  → Yearly batch: full recomputation (master reconciliation)
  → PROS: Much faster daily runs
  → CONS: More complex, must handle late data windows
  → WHEN: Dataset too large for daily full recomputation

  FOR OUR RIDE-SHARING CASE:
  → 1.2PB daily × 365 days = ~400PB yearly master dataset
  → Full recomputation of 400PB daily = IMPOSSIBLE
  → Use Strategy B:
    - Nightly: process yesterday's 1.2PB (optimized per Project 46)
    - Weekly: reprocess last 7 days (8.4PB)
    - Monthly: reprocess last 30 days (36PB)
    - Batch view = merge(monthly_base, weekly_delta, daily_delta)

═══════════════════════════════════════════════════════════
COMPONENT 4: VIEW MERGING (THE QUERY FUNCTION)
═══════════════════════════════════════════════════════════

  query(metric, time_range) = 
      merge(batch_view(metric, time_range),
            realtime_view(metric, time_range))

  MERGE LOGIC DEPENDS ON THE METRIC:

  For ADDITIVE metrics (SUM, COUNT):
    result = batch_value + realtime_value
    
    Example: Driver earnings for today
    batch_view:    $245.50 (from midnight batch run, covers 00:00-06:00)
    realtime_view: $67.30  (from speed layer, covers 06:00-14:32)
    MERGED:        $312.80 (total for today so far)

  For NON-ADDITIVE metrics (AVG, PERCENTILE):
    Cannot simply average the averages
    
    Example: Average wait time
    batch_view:    avg=4.2min, count=1500 (covers 00:00-06:00)
    realtime_view: avg=3.8min, count=200  (covers 06:00-14:32)
    MERGED:        (4.2×1500 + 3.8×200) / (1500+200) = 4.15min
    → Must store COUNT alongside AVERAGE for correct merge

  For REPLACEMENT metrics (LATEST VALUE):
    result = realtime_value if exists, else batch_value
    
    Example: Current surge pricing zone
    batch_view:    surge=1.5× (computed at midnight)
    realtime_view: surge=2.3× (computed 30 seconds ago)
    MERGED:        2.3× (real-time is more current)

✅ WHAT SHOULD DO

1. DESIGN MASTER DATASET FOR REPROCESSABILITY
   → Every event must have: event_id, event_type, timestamp, payload
   → event_id enables deduplication during reprocessing
   → timestamp enables time-range partitioning
   → payload is the full raw event (no information loss)

2. MAKE BATCH VIEWS ATOMIC (SWAP, NOT UPDATE)
   → Never UPDATE a batch view in place
   → Compute NEW batch view → atomic swap → delete old view
   → This prevents queries from seeing partial/inconsistent views
   → In HBase: write to new table → rename → delete old table
   → Or: write to versioned column family → flip pointer

3. STORE ENOUGH STATE FOR CORRECT MERGING
   → For averages: store (sum, count) not just average
   → For percentiles: store histograms or sketches
   → For unique counts: store HyperLogLog sketches
   → Merging approximate data structures is a science

4. TIMESTAMP EVERYTHING
   → batch_view must record: "computed from data up to timestamp X"
   → realtime_view must record: "covers data from timestamp X to now"
   → Query logic uses timestamps to avoid double-counting

5. DESIGN FOR LATE-ARRIVING DATA
   → Trip event arrives 2 hours late (driver had no connectivity)
   → Next batch run picks it up from master dataset
   → Batch view automatically corrects
   → Speed layer may have missed it — that's OK
   → Lambda's self-healing property: batch always corrects speed

6. USE HBASE BULK LOAD FOR BATCH VIEWS (Project 36)
   → Batch MapReduce generates HFiles directly
   → LoadIncrementalHFiles loads them atomically
   → No write amplification on HBase RegionServers
   → Fastest way to load batch views (billions of rows in minutes)

🚫 WHAT SHOULD NOT DO

1. DO NOT MODIFY THE MASTER DATASET
   → No updates, no deletes, no in-place transformations
   → If data needs correction: append a correction event
   → Batch recomputation handles corrections automatically
   → Modifying master data = losing the source of truth

2. DO NOT LET SPEED LAYER RESULTS PERSIST FOREVER
   → Real-time views are TEMPORARY approximations
   → Once batch view covers the same time range → delete real-time view
   → If real-time views accumulate → data inconsistency grows
   → The batch layer is the AUTHORITY — speed layer is the BRIDGE

3. DO NOT DOUBLE-COUNT DURING MERGE
   → If batch view covers 00:00-06:00 and realtime covers 00:00-14:32
   → Merge MUST NOT add the 00:00-06:00 data twice
   → Realtime view should only cover data AFTER batch cutoff
   → Use watermarks/timestamps to delineate boundaries

4. DO NOT BUILD SPEED LAYER BEFORE BATCH LAYER
   → Batch layer is the FOUNDATION — always correct, always recoverable
   → Speed layer is the OPTIMIZATION — nice to have, not essential
   → Many teams successfully run Lambda with batch-only initially
   → Add speed layer when freshness requirement is proven real

5. DO NOT USE LAMBDA WHEN KAPPA SUFFICES
   → Kappa Architecture = speed layer only (no batch recomputation)
   → If your stream processing is reliable enough AND
     you don't need historical recomputation → Kappa is simpler
   → Lambda's strength is REPROCESSABILITY — if you don't need it,
     you don't need Lambda

🚫 WHEN NOT TO USE

LAMBDA ARCHITECTURE IS WRONG WHEN:

1. YOU DON'T NEED REAL-TIME
   → If batch-only (nightly reports) meets business needs
   → Adding speed layer = operational complexity for no benefit
   → Start with batch only, add speed layer when PROVEN needed

2. YOUR DATA IS SMALL
   → If entire dataset fits in PostgreSQL
   → Lambda overhead (HDFS + MapReduce + HBase + streaming) is absurd
   → Use a traditional OLTP + materialized views

3. STREAM PROCESSING ALONE IS SUFFICIENT
   → If you never need to reprocess historical data
   → If your stream processing is reliable (exactly-once)
   → Kappa Architecture (stream-only) is simpler
   → One codebase instead of two

4. YOU CANNOT MAINTAIN TWO PROCESSING CODEBASES
   → Lambda requires: batch logic (MapReduce) + stream logic (Storm/Spark)
   → Both must produce CONSISTENT results
   → Bug in one but not the other = data inconsistency
   → Team must maintain and test BOTH paths

5. LATENCY REQUIREMENT IS SUB-SECOND
   → Lambda's speed layer typically provides 1-30 second freshness
   → For sub-second (< 100ms) latency, use dedicated stream processor
   → Lambda adds overhead that prevents sub-second response

  HOW LAMBDA HANDLES THIS:

  Scenario A: Event arrives BEFORE batch processes Jan 15 partition
    → Event lands in /trips/master/2024/01/15/ (correct partition)
    → Batch job reads it normally → included in batch view
    → No problem

  Scenario B: Event arrives AFTER batch processes Jan 15
    → Event lands in /trips/master/2024/01/15/ (correct partition)
    → Today's batch already completed → event MISSED
    → Speed layer picks it up → adds to realtime view for Jan 15
    → Result: Jan 15 = batch_view(incomplete) + realtime_view(late event)
    → NEXT batch run (weekly recomputation) reprocesses Jan 15
    → Weekly batch view includes the late event → CORRECTED

  Scenario C: Event arrives DAYS late
    → Monthly batch recomputation catches it
    → Until then: realtime view bridges the gap (approximately)

  THIS IS LAMBDA'S SELF-HEALING PROPERTY:
  → Speed layer provides IMMEDIATE approximate answer
  → Batch layer EVENTUALLY provides correct answer
  → No manual intervention needed
  → The system CONVERGES to accuracy over time

═══════════════════════════════════════════════════════════
INTERNAL MECHANIC 4: BATCH VIEW VERSIONING
═══════════════════════════════════════════════════════════

  PROBLEM: During batch view swap, queries must not see partial data

  SOLUTION — DOUBLE-BUFFERING:

  HBase tables:
    driver_earnings_v1  ← currently serving queries
    driver_earnings_v2  ← being loaded with new batch view

  Swap process:
    1. Batch MapReduce generates HFiles for v2
    2. Bulk load HFiles into driver_earnings_v2
    3. Verify v2 is complete and correct
    4. ATOMIC SWAP: update pointer from v1 to v2
       → Application config: "active_table = v2"
       → Or: HBase table rename (not natively supported)
       → Or: Use a metadata table with version pointer
    5. Next batch writes to v1 (now the inactive table)

  ALTERNATIVE — COLUMN FAMILY VERSIONING:
    Single table: driver_earnings
    Column families: batch_v1, batch_v2
    
    Pointer stored in separate metadata row:
      Row: __metadata__
      Column: active_batch_version = v2
    
    Query logic:
      version = get(__metadata__, active_batch_version)
      data = get(driver_row, batch_{version}:total_fare)

  Though we focus on batch layer, here's how speed layer connects:

  EVENT FLOW:

⚠️ COMMON PITFALLS AND FAILURE MODES

PITFALL 1: DOUBLE-COUNTING IN MERGE
→ SYMPTOM: Dashboard shows earnings 2× actual
→ CAUSE: Batch view covers 00:00-23:59, realtime also covers 00:00-14:32
→ Merge adds both: $312 + $187 = $499 (but $187 is SUBSET of $312)
→ FIX: Track time boundaries precisely
→ After batch loads, DELETE realtime entries for covered period
→ Realtime should only cover time AFTER batch cutoff

PITFALL 2: BATCH VIEW INCONSISTENCY DURING SWAP
→ SYMPTOM: Queries return null or partial results during batch load
→ CAUSE: Bulk load takes 30 minutes, queries hit partially loaded table
→ FIX: Use double-buffering (v1/v2 tables)
→ Never load into the ACTIVE serving table
→ Load into inactive → verify → atomic pointer swap

PITFALL 3: MASTER DATASET CORRUPTION
→ SYMPTOM: Batch recomputation produces different results each time
→ CAUSE: Someone modified/deleted files in master dataset
→ FIX: Set HDFS permissions to READ-ONLY for master dataset
→ Only the ingestion process has WRITE permission
→ Use HDFS snapshots for point-in-time recovery

PITFALL 4: SPEED LAYER DIVERGES FROM BATCH
→ SYMPTOM: Realtime shows $187 but next batch shows $192 for same period
→ CAUSE: Speed layer has different logic than batch (code divergence)
→ THIS IS EXPECTED AND OK in Lambda — batch is authoritative
→ FIX: Minimize divergence by sharing logic libraries
→ Accept that speed layer is APPROXIMATE by design

PITFALL 5: BATCH JOB FAILS MIDWAY
→ SYMPTOM: New batch view not produced, realtime view grows stale
→ CAUSE: MapReduce job failed (OOM, data corruption, cluster issue)
→ IMPACT: Queries still work using old batch view + growing realtime view
→ FIX: Retry batch job, monitor for failures
→ Lambda is RESILIENT: old batch view + realtime still gives answers
→ Just not as accurate as fresh batch view would be

PITFALL 6: HBASE BULK LOAD COMPACTION STORM
→ SYMPTOM: HBase performance degrades after batch view load
→ CAUSE: Bulk load creates many HFiles → triggers major compaction
→ FIX: Schedule compaction during off-peak hours
→ Pre-split regions to distribute load (Project 35)
→ Rate-limit bulk load if cluster is serving live traffic

import org.apache.hadoop.conf.Configuration;
import org.apache.hadoop.conf.Configured;
import org.apache.hadoop.fs.Path;
import org.apache.hadoop.io.*;
import org.apache.hadoop.mapreduce.*;
import org.apache.hadoop.mapreduce.lib.input.FileInputFormat;
import org.apache.hadoop.mapreduce.lib.output.FileOutputFormat;
import org.apache.hadoop.util.Tool;
import org.apache.hadoop.util.ToolRunner;

import java.io.DataInput;
import java.io.DataOutput;
import java.io.IOException;

/**
 * BATCH LAYER: Compute driver earnings by city and hour.
 * 
 * Lambda principle: This job can be re-run on ANY time range
 * of the master dataset to recompute accurate batch views.
 * 
 * Input:  /trips/master/YYYY/MM/DD/trip_events/
 * Output: /trips/batch_views/driver_earnings/YYYY-MM-DD/
 * 
 * Output format: TSV
 *   driver_id \t city \t date \t hour \t total_fare \t trip_count \t total_distance
 */
public class DriverEarningsBatchJob extends Configured implements Tool {

    // ── Composite Key: driver_id + city + date + hour ──
    public static class EarningsKey implements WritableComparable<EarningsKey> {
        private Text driverId = new Text();
        private Text city = new Text();
        private Text date = new Text();
        private IntWritable hour = new IntWritable();

        public EarningsKey() {}

        public void set(String driverId, String city, String date, int hour) {
            this.driverId.set(driverId);
            this.city.set(city);
            this.date.set(date);
            this.hour.set(hour);
        }

        @Override
        public void write(DataOutput out) throws IOException {
            driverId.write(out); city.write(out);
            date.write(out); hour.write(out);
        }

        @Override
        public void readFields(DataInput in) throws IOException {
            driverId.readFields(in); city.readFields(in);
            date.readFields(in); hour.readFields(in);
        }

        @Override
        public int compareTo(EarningsKey o) {
            int cmp = driverId.compareTo(o.driverId);
            if (cmp != 0) return cmp;
            cmp = city.compareTo(o.city);
            if (cmp != 0) return cmp;
            cmp = date.compareTo(o.date);
            if (cmp != 0) return cmp;
            return hour.compareTo(o.hour);
        }

        @Override
        public int hashCode() {
            return driverId.hashCode() * 163 + city.hashCode() * 31 
                   + date.hashCode() * 7 + hour.hashCode();
        }

        @Override
        public boolean equals(Object o) {
            if (!(o instanceof EarningsKey)) return false;
            EarningsKey other = (EarningsKey) o;
            return driverId.equals(other.driverId) && city.equals(other.city)
                   && date.equals(other.date) && hour.equals(other.hour);
        }

        public String toRowKey() {
            // HBase row key format for serving layer (Project 35 design)
            return driverId + "#" + date + "#" + String.format("%02d", hour.get());
        }

        @Override
        public String toString() {
            return driverId + "\t" + city + "\t" + date + "\t" + hour;
        }
    }

    // ── Aggregation Value ──
    public static class EarningsValue implements Writable {
        private DoubleWritable totalFare = new DoubleWritable();
        private IntWritable tripCount = new IntWritable();
        private DoubleWritable totalDistance = new DoubleWritable();

        public EarningsValue() {}

        public void set(double fare, int trips, double distance) {
            totalFare.set(fare); tripCount.set(trips); totalDistance.set(distance);
        }

        public void add(EarningsValue other) {
            totalFare.set(totalFare.get() + other.totalFare.get());
            tripCount.set(tripCount.get() + other.tripCount.get());
            totalDistance.set(totalDistance.get() + other.totalDistance.get());
        }

        @Override
        public void write(DataOutput out) throws IOException {
            totalFare.write(out); tripCount.write(out); totalDistance.write(out);
        }

        @Override
        public void readFields(DataInput in) throws IOException {
            totalFare.readFields(in); tripCount.readFields(in); totalDistance.readFields(in);
        }

        @Override
        public String toString() {
            return String.format("%.2f\t%d\t%.2f", 
                totalFare.get(), tripCount.get(), totalDistance.get());
        }
    }

    // ── MAPPER: Parse trip events, emit (driver+city+hour, fare) ──
    public static class EarningsMapper
            extends Mapper<LongWritable, Text, EarningsKey, EarningsValue> {

        private final EarningsKey outKey = new EarningsKey();
        private final EarningsValue outValue = new EarningsValue();

        @Override
        protected void map(LongWritable key, Text value, Context context)
                throws IOException, InterruptedException {

            // Parse JSON trip event (simplified — use Jackson in production)
            String line = value.toString().trim();
            if (line.isEmpty() || !line.startsWith("{")) return;

            try {
                // Expected: {"driver_id":"D123","city":"NYC","timestamp":1705334400,
                //            "fare":24.50,"distance_km":8.3,"event_type":"trip_complete"}
                String driverId = extractJsonField(line, "driver_id");
                String city = extractJsonField(line, "city");
                String eventType = extractJsonField(line, "event_type");

                if (!"trip_complete".equals(eventType)) return;

                long timestamp = Long.parseLong(extractJsonField(line, "timestamp"));
                double fare = Double.parseDouble(extractJsonField(line, "fare"));
                double distance = Double.parseDouble(extractJsonField(line, "distance_km"));

                // Derive date and hour from timestamp
                java.time.Instant instant = java.time.Instant.ofEpochSecond(timestamp);
                java.time.ZonedDateTime zdt = instant.atZone(java.time.ZoneId.of("UTC"));
                String date = zdt.toLocalDate().toString();
                int hour = zdt.getHour();

                outKey.set(driverId, city, date, hour);
                outValue.set(fare, 1, distance);

                context.write(outKey, outValue);

            } catch (Exception e) {
                context.getCounter("PARSE_ERRORS", e.getClass().getSimpleName()).increment(1);
            }
        }

        private String extractJsonField(String json, String field) {
            // Simple JSON field extraction (use Jackson in production)
            String searchKey = "\"" + field + "\":";
            int idx = json.indexOf(searchKey);
            if (idx < 0) return "";
            idx += searchKey.length();
            
            if (json.charAt(idx) == '"') {
                int endIdx = json.indexOf('"', idx + 1);
                return json.substring(idx + 1, endIdx);
            } else {
                int endIdx = json.indexOf(',', idx);
                if (endIdx < 0) endIdx = json.indexOf('}', idx);
                return json.substring(idx, endIdx).trim();
            }
        }
    }

    // ── COMBINER: Local aggregation (same as Reducer — Project 9) ──
    // Earnings aggregation is associative + commutative → safe for Combiner
    public static class EarningsCombiner
            extends Reducer<EarningsKey, EarningsValue, EarningsKey, EarningsValue> {

        private final EarningsValue result = new EarningsValue();

        @Override
        protected void reduce(EarningsKey key, Iterable<EarningsValue> values, Context context)
                throws IOException, InterruptedException {
            result.set(0, 0, 0);
            for (EarningsValue val : values) {
                result.add(val);
            }
            context.write(key, result);
        }
    }

    // ── REDUCER: Final aggregation ──
    public static class EarningsReducer
            extends Reducer<EarningsKey, EarningsValue, EarningsKey, EarningsValue> {

        private final EarningsValue result = new EarningsValue();

        @Override
        protected void reduce(EarningsKey key, Iterable<EarningsValue> values, Context context)
                throws IOException, InterruptedException {
            result.set(0, 0, 0);
            for (EarningsValue val : values) {
                result.add(val);
            }
            context.write(key, result);
        }
    }

    @Override
    public int run(String[] args) throws Exception {
        if (args.length != 2) {
            System.err.println("Usage: DriverEarningsBatchJob <input_path> <output_path>");
            System.err.println("  input_path:  /trips/master/2024/01/15/trip_events/");
            System.err.println("  output_path: /trips/batch_views/driver_earnings/2024-01-15/");
            return 1;
        }

        Configuration conf = getConf();
        Job job = Job.getInstance(conf, "Lambda Batch: Driver Earnings");
        job.setJarByClass(DriverEarningsBatchJob.class);

        job.setMapperClass(EarningsMapper.class);
        job.setCombinerClass(EarningsCombiner.class);
        job.setReducerClass(EarningsReducer.class);

        job.setOutputKeyClass(EarningsKey.class);
        job.setOutputValueClass(EarningsValue.class);

        // Apply Project 46 optimizations
        job.setNumReduceTasks(500);

        FileInputFormat.addInputPath(job, new Path(args[0]));
        FileOutputFormat.setOutputPath(job, new Path(args[1]));

        return job.waitForCompletion(true) ? 0 : 1;
    }

    public static void main(String[] args) throws Exception {
        int exitCode = ToolRunner.run(new Configuration(), new DriverEarningsBatchJob(), args);
        System.exit(exitCode);
    }
}

import org.apache.hadoop.fs.Path;

import org.apache.hadoop.io.*;

import org.apache.hadoop.mapreduce.lib.input.TextInputFormat;

/**
 * SERVING LAYER LOADER: Converts batch view TSV to HBase HFiles
 * then bulk loads them for instant serving.
 * 
 * This implements the atomic batch view swap described in the
 * internal mechanics section.
 * 
 * Input:  /trips/batch_views/driver_earnings/2024-01-15/ (TSV from batch job)
 * Output: HFiles loaded into HBase table driver_earnings
 * 
 * Recall Project 36: Bulk load generates HFiles via MapReduce,
 * then LoadIncrementalHFiles moves them into HBase regions.
 */
public class BatchViewHBaseLoader extends Configured implements Tool {

    private static final byte[] CF_BATCH = Bytes.toBytes("batch");
    private static final byte[] COL_TOTAL_FARE = Bytes.toBytes("total_fare");
    private static final byte[] COL_TRIP_COUNT = Bytes.toBytes("trip_count");
    private static final byte[] COL_TOTAL_DISTANCE = Bytes.toBytes("total_distance");
    private static final byte[] COL_COMPUTED_AT = Bytes.toBytes("computed_at");
    private static final byte[] COL_DATA_THROUGH = Bytes.toBytes("data_through");

    /**
     * Mapper: Converts batch view TSV rows to HBase Put operations.
     * 
     * Input TSV: driver_id \t city \t date \t hour \t total_fare \t trip_count \t distance
     * Output: (row_key, Put) for HFileOutputFormat2
     */
    public static class HBaseMapper
            extends Mapper<LongWritable, Text, ImmutableBytesWritable, Put> {

        private String computedAt;
        private String dataThroughTimestamp;

        @Override
        protected void setup(Context context) {
            Configuration conf = context.getConfiguration();
            computedAt = conf.get("batch.computed.at", 
                java.time.Instant.now().toString());
            dataThroughTimestamp = conf.get("batch.data.through",
                java.time.LocalDate.now().minusDays(1).toString() + "T23:59:59Z");
        }

        @Override
        protected void map(LongWritable key, Text value, Context context)
                throws IOException, InterruptedException {

            String line = value.toString().trim();
            if (line.isEmpty()) return;

            String[] parts = line.split("\t");
            if (parts.length < 7) {
                context.getCounter("ERRORS", "MALFORMED_LINE").increment(1);
                return;
            }

            String driverId = parts[0];
            String city = parts[1];
            String date = parts[2];
            String hour = parts[3];
            String totalFare = parts[4];
            String tripCount = parts[5];
            String totalDistance = parts[6];

            // Row key design (recall Project 35):
            // driver_id#date#hour — enables range scans per driver per day
            String rowKey = driverId + "#" + date + "#" + String.format("%02d", Integer.parseInt(hour));

            byte[] rowBytes = Bytes.toBytes(rowKey);
            ImmutableBytesWritable hbaseKey = new ImmutableBytesWritable(rowBytes);

            Put put = new Put(rowBytes);
            put.addColumn(CF_BATCH, COL_TOTAL_FARE, Bytes.toBytes(totalFare));
            put.addColumn(CF_BATCH, COL_TRIP_COUNT, Bytes.toBytes(tripCount));
            put.addColumn(CF_BATCH, COL_TOTAL_DISTANCE, Bytes.toBytes(totalDistance));
            put.addColumn(CF_BATCH, COL_COMPUTED_AT, Bytes.toBytes(computedAt));
            put.addColumn(CF_BATCH, COL_DATA_THROUGH, Bytes.toBytes(dataThroughTimestamp));

            // Also store city for query enrichment
            put.addColumn(CF_BATCH, Bytes.toBytes("city"), Bytes.toBytes(city));

            context.write(hbaseKey, put);
        }
    }

    @Override
    public int run(String[] args) throws Exception {
        if (args.length < 3) {
            System.err.println("Usage: BatchViewHBaseLoader <input_path> <hfile_output> <table_name>");
            return 1;
        }

        String inputPath = args[0];
        String hfileOutput = args[1];
        String tableName = args[2];

        Configuration conf = getConf();
        conf.set("batch.computed.at", java.time.Instant.now().toString());
        conf.set("batch.data.through",
            java.time.LocalDate.now().minusDays(1).toString() + "T23:59:59Z");

        // ── PHASE 1: Generate HFiles ──
        System.out.println("=== PHASE 1: Generating HFiles ===");

        Job job = Job.getInstance(conf, "Lambda: Batch View → HBase HFiles");
        job.setJarByClass(BatchViewHBaseLoader.class);

        job.setMapperClass(HBaseMapper.class);
        job.setMapOutputKeyClass(ImmutableBytesWritable.class);
        job.setMapOutputValueClass(Put.class);

        job.setInputFormatClass(TextInputFormat.class);
        FileInputFormat.addInputPath(job, new Path(inputPath));

        // Configure HFileOutputFormat2
        Connection connection = ConnectionFactory.createConnection(conf);
        Table table = connection.getTable(TableName.valueOf(tableName));
        RegionLocator regionLocator = connection.getRegionLocator(TableName.valueOf(tableName));
        
        HFileOutputFormat2.configureIncrementalLoad(job, table, regionLocator);
        HFileOutputFormat2.setOutputPath(job, new Path(hfileOutput));

        boolean success = job.waitForCompletion(true);
        if (!success) {
            System.err.println("HFile generation FAILED");
            return 1;
        }

        System.out.println("HFiles generated at: " + hfileOutput);

        // ── PHASE 2: Bulk Load HFiles into HBase ──
        System.out.println("=== PHASE 2: Bulk Loading into HBase ===");

        LoadIncrementalHFiles bulkLoader = new LoadIncrementalHFiles(conf);
        bulkLoader.doBulkLoad(new Path(hfileOutput), (HTable) table);

        System.out.println("Bulk load COMPLETE into table: " + tableName);

        // ── PHASE 3: Update metadata ──
        System.out.println("=== PHASE 3: Updating batch metadata ===");

        Put metaPut = new Put(Bytes.toBytes("__batch_metadata__"));
        metaPut.addColumn(CF_BATCH, Bytes.toBytes("last_completed"),
            Bytes.toBytes(java.time.Instant.now().toString()));
        metaPut.addColumn(CF_BATCH, Bytes.toBytes("data_through"),
            Bytes.toBytes(conf.get("batch.data.through")));
        metaPut.addColumn(CF_BATCH, Bytes.toBytes("input_path"),
            Bytes.toBytes(inputPath));

        }

            result["total_fare"] += batch["batch:total_fare"]
            result["trip_count"] += batch["batch:trip_count"]
            result["total_distance"] += batch["batch:total_distance"]

                result["trip_count"] += realtime["realtime:trip_count"]
                result["total_distance"] += realtime["realtime:total_distance"]

            result["source"].append("realtime")
            result["freshness"] = realtime["realtime:last_updated"]
        
        elif not covered and not realtime:
            # No data available for this time range
            result["source"].append("none")
            result["freshness"] = "no_data"
        
        elif covered and not batch:
            # Batch should cover but has no data — driver was inactive
            result["source"].append("batch_empty")
            result["freshness"] = self.batch_metadata["last_completed"]

class BatchViewSwapper:
    """
    Manages atomic batch view swaps in HBase.
    
    Implements double-buffering:
    → Load new batch view into inactive table
    → Verify correctness
    → Swap active pointer
    → Clean realtime overlap
    """
    
    def __init__(self):
        self.active_version = "v1"
        self.versions = {"v1": {}, "v2": {}}
        self.metadata = {"active": "v1"}
    
    def get_inactive_version(self):
        """Get the version that is NOT currently serving."""
        return "v2" if self.active_version == "v1" else "v1"
    
    def load_new_batch_view(self, version, data):
        """Load new batch view data into specified version."""
        print(f"  Loading batch view into {version}...")
        self.versions[version] = data
        print(f"  Loaded {len(data)} entries into {version}")
    
    def verify_batch_view(self, version):
        """Verify new batch view before swapping."""
        data = self.versions[version]
        
        checks_passed = True
        
        # Check 1: Not empty
        if len(data) == 0:
            print(f"  ❌ VERIFY FAILED: {version} is empty")
            checks_passed = False
        
        # Check 2: Row count within expected range
        active_count = len(self.versions[self.active_version])
        new_count = len(data)
        
        if active_count > 0:
            change_pct = abs(new_count - active_count) / active_count * 100
            if change_pct > 50:
                print(f"  ⚠️  VERIFY WARNING: Row count changed by {change_pct:.0f}%")
                print(f"     Active ({self.active_version}): {active_count} rows")
                print(f"     New ({version}): {new_count} rows")
            else:
                print(f"  ✅ Row count change: {change_pct:.1f}% (acceptable)")
        
        # Check 3: Spot-check values
        for key, values in list(data.items())[:3]:
            fare = values.get("batch:total_fare", 0)
            if fare < 0:
                print(f"  ❌ VERIFY FAILED: Negative fare for {key}")
                checks_passed = False
        
        if checks_passed:
            print(f"  ✅ Verification PASSED for {version}")
        
        return checks_passed
    
    def swap(self):
        """Atomic swap: make inactive version the new active."""
        old_active = self.active_version
        new_active = self.get_inactive_version()
        
        print(f"\n  ── ATOMIC SWAP ──")
        print(f"  Old active: {old_active}")
        print(f"  New active: {new_active}")
        
        # The swap is a single metadata write — atomic
        self.active_version = new_active
        self.metadata["active"] = new_active
        self.metadata["swapped_at"] = datetime.utcnow().isoformat() + "Z"
        
        print(f"  ✅ Swap complete at {self.metadata['swapped_at']}")
        print(f"  Active version is now: {self.active_version}")
        
        return True
    
    def cleanup_realtime_overlap(self, realtime_views, batch_data_through):
        """
        Delete realtime entries that are now covered by the new batch view.
        
        This prevents double-counting in the merge function.
        """
        print(f"\n  ── REALTIME CLEANUP ──")
        print(f"  Batch covers data through: {batch_data_through}")
        
        keys_to_delete = []
        batch_cutoff = datetime.fromisoformat(batch_data_through.replace("Z", "+00:00"))
        
        for key, values in realtime_views.items():
            data_from = values.get("realtime:data_from", "")
            if data_from:
                event_time = datetime.fromisoformat(data_from.replace("Z", "+00:00"))
                # If realtime data is for a time BEFORE batch cutoff
                # AND there's no late-arriving aspect, delete it
                if event_time <= batch_cutoff:
                    keys_to_delete.append(key)
        
        for key in keys_to_delete:
            del realtime_views[key]
            print(f"  Deleted realtime entry: {key} (now covered by batch)")
        
        remaining = len(realtime_views)
        print(f"  Remaining realtime entries: {remaining} (post-batch data)")
        
        return len(keys_to_delete)

🔑 KEY TAKEAWAY

WHAT WE ESTABLISHED IN PROJECT 47:

✅ THE PROBLEM:
   → Business needs BOTH accurate historical data AND fresh real-time data
   → Batch processing (MapReduce) is accurate but hours old
   → Stream processing is fresh but potentially approximate
   → Single system cannot satisfy both requirements optimally

✅ THE SOLUTION — LAMBDA ARCHITECTURE:
   → Three layers working in parallel:
   
   BATCH LAYER (Hadoop MapReduce):
   → Processes ENTIRE master dataset periodically
   → Produces 100% accurate "batch views"
   → Stored in HBase for fast serving
   → Self-healing: recomputation corrects any errors
   
   SPEED LAYER (Stream Processing):
   → Processes new events as they arrive (seconds)
   → Produces approximate "realtime views"
   → Fills the gap between batch runs
   → Temporary — discarded when batch catches up
   
   SERVING LAYER (HBase):
   → Stores both batch and realtime views
   → Merges them at query time
   → query = merge(batch_view, realtime_view)
   → Provides low-latency responses (< 10ms)

✅ KEY PRINCIPLES:
   → IMMUTABLE MASTER DATASET: Raw events never modified
   → BATCH VIEWS ARE PRECOMPUTED: Not computed at query time
   → SPEED LAYER IS TEMPORARY: Batch always corrects speed
   → MERGE LOGIC HANDLES TIME BOUNDARIES: No double-counting
   → LATE-ARRIVING DATA: Batch recomputation self-heals

✅ MERGE LOGIC RULES:
   → ADDITIVE metrics (SUM, COUNT): batch + realtime
   → AVERAGE metrics: weighted average using stored counts
   → REPLACEMENT metrics: prefer realtime (more recent)
   → Time boundaries: track data_through to prevent overlap

✅ ATOMIC BATCH VIEW SWAP:
   → Double-buffering (v1/v2 tables)
   → Load new view into inactive table
   → Verify correctness
   → Atomic pointer swap
   → Clean realtime overlap for covered period
