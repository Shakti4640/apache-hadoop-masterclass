PROJECT 18: SORTING ALL EMPLOYEE SALARIES GLOBALLY ACROSS THE CLUSTER

🔴 HYPOTHETICAL PROBLEM

SCENARIO:
→ You are a compensation analyst at a multinational corporation
→ 85 million employee salary records across 40 countries
→ Data sits in HDFS as output from a Reduce-Side Join (Project 17)
   that merged employee profiles with payroll tables
→ Management demands a SINGLE SORTED FILE of all employees
   ordered by salary from lowest to highest

FORMAT (each line):
  EMP_ID | NAME | COUNTRY | DEPARTMENT | ANNUAL_SALARY_USD

SAMPLE:
  E00001 | Maria Santos   | Brazil  | Engineering | 42000
  E00002 | John Smith     | USA     | Marketing   | 78000
  E00003 | Akiko Tanaka   | Japan   | Finance     | 65000
  ...85 million more rows...

REQUIREMENT:
→ Output must be GLOBALLY sorted by salary
→ If you concatenate part-r-00000, part-r-00001, ..., part-r-00099
   the ENTIRE sequence must be in ascending salary order
→ part-r-00000 contains the LOWEST salaries
→ part-r-00099 contains the HIGHEST salaries

THE PROBLEM:
→ You run a standard MapReduce sort with 100 reducers
→ Each reducer's output IS sorted internally
→ BUT part-r-00000 might contain salaries 10K-500K
   AND part-r-00050 might ALSO contain salaries 15K-480K
→ The files OVERLAP — concatenation is NOT globally sorted

WHY THIS HAPPENS:
→ HashPartitioner (Project 10) distributes keys by hash
→ hash("42000") % 100 might send to reducer 7
→ hash("43000") % 100 might send to reducer 92
→ There is NO salary-range-based partitioning
→ Global sort is BROKEN by default

🔍 HOW TO APPROACH IT

STEP 1: Understand what "sorted" means in MapReduce

  ┌────────────────────────────────────────────────────────────┐
  │           DEFAULT MapReduce SORTING BEHAVIOR               │
  ├────────────────────────────────────────────────────────────┤
  │                                                            │
  │  WITHIN each reducer → keys ARE sorted (guaranteed)        │
  │  ACROSS reducers     → keys are NOT sorted (random hash)   │
  │                                                            │
  │  Reducer 0 output: [12K, 45K, 89K, 310K, 500K]           │
  │  Reducer 1 output: [8K, 22K, 77K, 290K, 410K]            │
  │  Reducer 2 output: [15K, 50K, 120K, 340K, 480K]          │
  │                                                            │
  │  Concatenated: 12K,45K,89K,310K,500K,8K,22K...           │
  │  ❌ NOT globally sorted                                    │
  └────────────────────────────────────────────────────────────┘

STEP 2: Understand what GLOBAL sort requires

  ┌────────────────────────────────────────────────────────────┐
  │           REQUIRED: RANGE-BASED PARTITIONING               │
  ├────────────────────────────────────────────────────────────┤
  │                                                            │
  │  Reducer 0 handles: salaries     0 -  50,000              │
  │  Reducer 1 handles: salaries 50,001 - 100,000             │
  │  Reducer 2 handles: salaries 100,001 - 500,000            │
  │                                                            │
  │  Reducer 0 output: [8K, 12K, 15K, 22K, 42K, 45K, 50K]   │
  │  Reducer 1 output: [65K, 77K, 78K, 89K]                  │
  │  Reducer 2 output: [120K, 290K, 310K, 340K, 410K, 500K]  │
  │                                                            │
  │  Concatenated: 8K,12K,15K,...,65K,77K,...,120K,290K...    │
  │  ✅ GLOBALLY sorted                                        │
  └────────────────────────────────────────────────────────────┘

STEP 3: The critical question

  HOW do we determine the range boundaries?
  → We DON'T know the salary distribution in advance
  → If most salaries are 30K-80K and we split evenly 0-500K,
    reducer 0 gets 90% of data (SKEW — see Project 33 preview)
  → We need to SAMPLE the data first to find good split points
  → THIS is exactly what TotalOrderPartitioner + InputSampler does

💡 HOW CAN WE SOLVE IT

HADOOP PROVIDES A BUILT-IN SOLUTION — THREE COMPONENTS:

┌─────────────────────────────────────────────────────────────────┐
│                 GLOBAL SORT MECHANISM                            │
│                                                                  │
│  COMPONENT 1: InputSampler                                      │
│  → Reads a SAMPLE of records from input data                    │
│  → Determines partition boundaries from the sample              │
│  → Writes boundaries to a "partition file"                      │
│                                                                  │
│  COMPONENT 2: TotalOrderPartitioner                             │
│  → Reads the partition file at job startup                      │
│  → For each map output key, uses BINARY SEARCH                  │
│    on the boundaries to determine which reducer gets it         │
│  → Replaces HashPartitioner (Project 10)                        │
│                                                                  │
│  COMPONENT 3: Standard MapReduce Sort                           │
│  → Each reducer receives a non-overlapping range of keys        │
│  → Each reducer sorts its range internally (default behavior)   │
│  → Result: GLOBAL SORT across all output files                  │
└─────────────────────────────────────────────────────────────────┘

THE FLOW:

  ┌──────────┐    sample     ┌──────────────┐    partition file
  │  INPUT   │ ───────────→  │ InputSampler │ ───────────────→  _partition.lst
  │  DATA    │               └──────────────┘                        │
  └────┬─────┘                                                       │
       │                                                             ▼
       │              ┌──────────────────────────────────────────────────┐
       │              │           TotalOrderPartitioner                  │
       │              │  reads _partition.lst at startup                 │
       │              │  boundaries: [50000, 100000, 200000]            │
       │              │                                                  │
       └──────────────│  key < 50000       → Reducer 0                  │
                      │  50000 ≤ key < 100000 → Reducer 1              │
                      │  100000 ≤ key < 200000 → Reducer 2             │
                      │  key ≥ 200000      → Reducer 3                  │
                      └──────────────────────────────────────────────────┘

✅ WHAT SHOULD DO

1. UNDERSTAND THE THREE SAMPLING STRATEGIES

   InputSampler provides THREE built-in samplers:

   ┌────────────────────────────────────────────────────────────────┐
   │ SAMPLER            │ HOW IT WORKS            │ WHEN TO USE     │
   ├────────────────────┼─────────────────────────┼─────────────────┤
   │ RandomSampler      │ Reads random records    │ General purpose │
   │ (freq, numSamples, │ with given probability  │ Most common     │
   │  maxSplits)        │ from random splits      │                 │
   ├────────────────────┼─────────────────────────┼─────────────────┤
   │ IntervalSampler    │ Reads records at fixed  │ Already roughly │
   │ (freq, maxSplits)  │ intervals from splits   │ sorted data     │
   ├────────────────────┼─────────────────────────┼─────────────────┤
   │ SplitSampler       │ Reads first N records   │ Quick & dirty   │
   │ (numSamples,       │ from first M splits     │ Testing only    │
   │  maxSplits)        │                         │                 │
   └────────────────────┴─────────────────────────┴─────────────────┘

   → RandomSampler is the SAFEST choice for unknown distributions
   → It gives the most representative partition boundaries

2. CHOOSE NUMBER OF REDUCERS WISELY
   → More reducers = more output files = finer-grained sort
   → But also more overhead and more partition boundaries needed
   → RULE: numReducers should produce output files of 1-4GB each
   → 85M records × ~60 bytes each ≈ 5GB → 2-5 reducers sufficient
   → For 500GB data → 100-200 reducers

3. ENSURE THE SORT KEY IS THE MAP OUTPUT KEY
   → TotalOrderPartitioner partitions based on MAP OUTPUT KEY
   → If you want to sort by salary, salary MUST be the key
   → Use an identity mapper that re-keys records by salary

4. USE DISTRIBUTED CACHE FOR THE PARTITION FILE
   → TotalOrderPartitioner reads partition file from Distributed Cache
     (covered in Project 15)
   → InputSampler.writePartitionFile() handles this automatically

🚫 WHAT SHOULD NOT DO

1. DO NOT hard-code partition boundaries
   → Salary distributions change over time
   → What works today creates massive skew tomorrow
   → ALWAYS sample — let the data tell you the boundaries

2. DO NOT use TotalOrderPartitioner with too few samples
   → If you sample 10 records from 85 million
   → Boundaries will be wildly inaccurate
   → RULE: Sample at least 10 × numReducers records
   → For 100 reducers → sample at least 1000 records
   → RandomSampler(0.001, 10000, 100) is a safe starting point

3. DO NOT confuse TotalOrderPartitioner with Secondary Sort (Project 19)
   → TotalOrderPartitioner: Global sort by ONE key across reducers
   → Secondary Sort: Sort by primary key THEN sub-sort by secondary key
   → They solve DIFFERENT problems — can be combined but separately

4. DO NOT forget that the sort key type must be WritableComparable
   → IntWritable, LongWritable, Text all implement it (Project 5)
   → Custom keys must implement compareTo() (Project 13)
   → TotalOrderPartitioner uses binary search — needs comparable keys

5. DO NOT skip the sampling step and use a uniform range
   → Salaries are NOT uniformly distributed
   → Most cluster between 30K-120K
   → Uniform split [0-100K, 100K-200K, ...] creates extreme skew
   → Reducer 0 gets 80% of data, others nearly empty

🚫 WHEN NOT TO USE

1. SMALL DATA THAT FITS IN ONE REDUCER
   → If 85 million records fit in a single reducer's memory/disk
   → Just use 1 reducer — output is automatically globally sorted
   → TotalOrderPartitioner overhead is unnecessary

2. WHEN YOU ONLY NEED TOP-N RESULTS
   → Sorting 85 million records to get top 10 highest salaries
   → WASTE — use a priority queue in mapper + single reducer
   → Global sort is O(N log N) — Top-N is O(N log K) where K=10

3. WHEN DOWNSTREAM ONLY READS ONE PARTITION AT A TIME
   → If consumers always query "salaries between 50K-60K"
   → Partitioning by range is useful but FULL SORT is unnecessary
   → Use a Custom Partitioner (Project 10) with known ranges

4. WHEN DATA IS ALREADY SORTED
   → If input comes from a previous global sort job
   → Re-sorting is pure waste of cluster resources

5. REAL-TIME SORTING NEEDS
   → TotalOrderPartitioner is a BATCH operation
   → Takes minutes to hours depending on data size
   → For real-time sorted views → use HBase (Project 27) row key design

⚙️ HOW IT ONLY WORKS — INTERNAL MECHANICS

═══════════════════════════════════════════════════════════════
INTERNAL MECHANIC 1: HOW InputSampler WORKS
═══════════════════════════════════════════════════════════════

Given: 85 million salary records, 4 reducers needed
Using: RandomSampler(probability=0.0001, maxSamples=10000, maxSplits=50)

STEP A — Select which input splits to sample
  → Input has 200 splits (200 blocks of 128MB each)
  → maxSplits=50 → randomly pick 50 of the 200 splits

STEP B — Sample records from selected splits
  → For each of the 50 splits:
    → Read each record
    → With probability 0.0001 (1 in 10,000), keep the record
    → Stop when maxSamples (10000) total records collected
  
  → Result: ~10000 salary values sampled
  
  Sample: [42000, 78000, 65000, 31000, 155000, 42500, 88000, ...]

STEP C — Sort the samples
  → Sort all 10000 sampled values:
  [12000, 12500, 13000, ..., 42000, 42500, ..., 155000, ..., 890000]

STEP D — Pick partition boundaries
  → Need 4 reducers → need 3 boundaries (N-1 boundaries for N reducers)
  → Divide sorted samples into 4 equal segments
  → Boundaries at indices: 2500, 5000, 7500
  
  samples[2500]  = 45000   → boundary 1
  samples[5000]  = 72000   → boundary 2
  samples[7500]  = 125000  → boundary 3

STEP E — Write partition file
  → Serialize boundaries to a SequenceFile: _partition.lst
  → Place in HDFS and add to Distributed Cache (Project 15)

  _partition.lst contains:
  ┌──────────────────────────┐
  │ Key[0]: IntWritable(45000)  │
  │ Key[1]: IntWritable(72000)  │
  │ Key[2]: IntWritable(125000) │
  └──────────────────────────┘

═══════════════════════════════════════════════════════════════
INTERNAL MECHANIC 2: HOW TotalOrderPartitioner WORKS AT RUNTIME
═══════════════════════════════════════════════════════════════

At Job Start:
  → TotalOrderPartitioner reads _partition.lst from Distributed Cache
  → Loads boundaries into a TRIE or sorted array for fast lookup
  → This happens ONCE per mapper JVM

For EVERY map output key-value pair:
  → getPartition(key, value, numReducers) is called
  → Performs BINARY SEARCH on boundaries array

  Example key: salary = 68000

  Boundaries: [45000, 72000, 125000]
  
  Binary search:
    → 68000 < 125000? YES → go left
    → 68000 < 72000?  YES → go left  
    → 68000 < 45000?  NO  → partition index = 1
  
  Result: partition 1 (second reducer)

  ┌───────────────────────────────────────────────────────────┐
  │  PARTITION ASSIGNMENT:                                     │
  │                                                            │
  │  salary < 45000          → Reducer 0  (part-r-00000)      │
  │  45000 ≤ salary < 72000  → Reducer 1  (part-r-00001)      │
  │  72000 ≤ salary < 125000 → Reducer 2  (part-r-00002)      │
  │  salary ≥ 125000         → Reducer 3  (part-r-00003)      │
  │                                                            │
  │  Each reducer sorts its range internally (default)         │
  │  Concatenation: part-r-00000 + 00001 + 00002 + 00003     │
  │  = GLOBALLY SORTED                                         │
  └───────────────────────────────────────────────────────────┘

═══════════════════════════════════════════════════════════════
INTERNAL MECHANIC 3: WHY BINARY SEARCH MATTERS
═══════════════════════════════════════════════════════════════

  → With 1000 reducers → 999 boundaries
  → HashPartitioner: O(1) — just hash % numReducers
  → TotalOrderPartitioner: O(log 999) ≈ 10 comparisons per record
  → For 85 million records: 850 million comparisons
  → Still fast because comparisons are cheap integer operations
  
  → For Text keys: TotalOrderPartitioner builds a TRIE
    (prefix tree) for even faster lookups than binary search
  → Trie lookup: O(key_length) instead of O(log N × key_length)

⚠️ COMMON PITFALLS AND FAILURE MODES

PITFALL 1: "Partition file not found" error at runtime
→ CAUSE: InputSampler.writePartitionFile() was not called before job.submit()
→ OR: The partition file path in TotalOrderPartitioner.setPartitionFile()
      does not match where writePartitionFile() wrote it
→ FIX: Always call writePartitionFile() AFTER setting job configuration
       and BEFORE job.waitForCompletion()

PITFALL 2: Extreme data skew despite using TotalOrderPartitioner
→ CAUSE: Too few samples for highly skewed distribution
→ EXAMPLE: 80% of salaries are between 40K-80K
           Sampling only 100 records might miss the tail distribution
→ FIX: Increase sample size — RandomSampler(0.001, 100000, 100)
→ ALSO: If one salary value has millions of duplicates,
        no partitioning can split WITHIN that value
        (all identical keys MUST go to same reducer)

PITFALL 3: Output files have wildly different sizes
→ CAUSE: Data distribution is not uniform and sample was too small
→ DIAGNOSIS: Check each part-r-NNNNN file size
→ FIX: Increase samples, or use more reducers to make ranges narrower

PITFALL 4: "Key type mismatch" ClassCastException
→ CAUSE: TotalOrderPartitioner expects map output key type
         to match the type used during sampling
→ If you sampled with IntWritable but mapper outputs LongWritable → CRASH
→ FIX: Ensure map output key class matches across sampler and mapper

PITFALL 5: Sorting by salary but salary is embedded in a Text line
→ CAUSE: If key is Text("42000"), sorting is LEXICOGRAPHIC not numeric
→ "9000" > "80000" because "9" > "8" in ASCII
→ FIX: Use IntWritable or LongWritable as map output key
→ OR: Pad with leading zeros — Text("0042000") sorts correctly
       but this is fragile — prefer numeric Writable types

PITFALL 6: Partition file generated on different data than actual input
→ CAUSE: You sampled dataset A but ran the sort job on dataset B
→ Boundaries make no sense for the actual data
→ FIX: ALWAYS sample from the SAME input path the job will process

📈 SCALABILITY CONSIDERATIONS

SCALING DIMENSION 1: NUMBER OF RECORDS

  ┌──────────────────┬───────────┬────────────────────────────────┐
  │ Records          │ Reducers  │ Sampling Recommendation        │
  ├──────────────────┼───────────┼────────────────────────────────┤
  │ 1 million        │ 2-5       │ RandomSampler(0.01, 5000, 10)  │
  │ 100 million      │ 10-50     │ RandomSampler(0.001, 50000, 50)│
  │ 1 billion        │ 50-200    │ RandomSampler(0.0001,100000,100│
  │ 10 billion       │ 200-1000  │ RandomSampler(0.00001,500000,  │
  │                  │           │                          200)   │
  └──────────────────┴───────────┴────────────────────────────────┘

SCALING DIMENSION 2: SAMPLING OVERHEAD

  → Sampling reads a FRACTION of total data
  → RandomSampler(0.0001, ...) reads 0.01% of records
  → For 10TB input: samples ~1GB worth of data
  → Sampling is a SEPARATE pass over the data (adds time)
  → But it's much faster than the actual sort job

SCALING DIMENSION 3: PARTITION FILE SIZE

  → Partition file contains (numReducers - 1) keys
  → 1000 reducers → 999 keys → tiny file (few KB)
  → Loaded into memory on EVERY mapper → negligible overhead
  → NOT a scaling concern

SCALING DIMENSION 4: BINARY SEARCH COST

  → O(log numReducers) per record
  → 1000 reducers → ~10 comparisons per record
  → 10 billion records → 100 billion comparisons
  → Integer comparison: ~1 nanosecond each
  → Total: ~100 seconds of pure partition computation
  → Dwarfed by I/O time — NOT a bottleneck

SCALING DIMENSION 5: THE REAL BOTTLENECK — SHUFFLE

  → Global sort requires ALL data to move through shuffle
  → 10TB input → up to 10TB network transfer during shuffle
  → THIS is where most time is spent
  → Mitigation: Use Combiner (Project 9) if applicable
    (not always applicable for sort — depends on whether
     you're aggregating or just reordering)
  → Mitigation: Compression (Project 14) on map output
    reduces shuffle data volume significantly

🔍 INTERPRETABILITY CONSIDERATIONS

HOW TO VERIFY YOUR GLOBAL SORT IS CORRECT:

1. CHECK OUTPUT FILE SIZES — should be roughly equal
   hdfs dfs -ls -h /output/sorted/
   → part-r-00000: ~1.2GB
   → part-r-00001: ~1.3GB
   → part-r-00002: ~1.1GB
   → part-r-00003: ~1.4GB
   → Roughly balanced = good sampling

2. CHECK BOUNDARY VALUES — first/last record of each file
   hdfs dfs -cat /output/sorted/part-r-00000 | head -1
   → Should show LOWEST salary in entire dataset
   
   hdfs dfs -cat /output/sorted/part-r-00000 | tail -1
   → Should be LESS THAN first record of part-r-00001
   
   hdfs dfs -cat /output/sorted/part-r-00001 | head -1
   → Should be GREATER THAN OR EQUAL to last of part-r-00000

3. CHECK JOB COUNTERS for data distribution
   → Map output records: should equal input records (identity mapper)
   → Per-reducer record counts: visible in YARN UI
   → Large imbalance indicates poor sampling

4. SAMPLING DIAGNOSTICS
   → Log the sampled boundaries before submitting job:
     Read _partition.lst and print boundary values
   → Verify boundaries match your expectation of data distribution
   → Example: If you KNOW most salaries are 30K-120K,
     boundaries should cluster in that range, not be uniform 0-500K

// PURPOSE: Globally sort 85M employee records by salary using
//          TotalOrderPartitioner + InputSampler
// BUILDS ON: Project 10 (Custom Partitioner), Project 17 (Join output)
// ══════════════════════════════════════════════════════════════════

import org.apache.hadoop.conf.Configuration;
import org.apache.hadoop.conf.Configured;
import org.apache.hadoop.fs.Path;
import org.apache.hadoop.io.IntWritable;
import org.apache.hadoop.io.LongWritable;
import org.apache.hadoop.io.NullWritable;
import org.apache.hadoop.io.Text;
import org.apache.hadoop.mapreduce.Job;
import org.apache.hadoop.mapreduce.Mapper;
import org.apache.hadoop.mapreduce.Reducer;
import org.apache.hadoop.mapreduce.lib.input.FileInputFormat;
import org.apache.hadoop.mapreduce.lib.input.TextInputFormat;
import org.apache.hadoop.mapreduce.lib.output.FileOutputFormat;
import org.apache.hadoop.mapreduce.lib.output.TextOutputFormat;
import org.apache.hadoop.mapreduce.lib.partition.InputSampler;
import org.apache.hadoop.mapreduce.lib.partition.TotalOrderPartitioner;
import org.apache.hadoop.util.Tool;
import org.apache.hadoop.util.ToolRunner;

import java.io.IOException;

public class SalaryGlobalSort extends Configured implements Tool {

    // ══════════════════════════════════════════════════════════
    // MAPPER: Re-key each record by salary (the sort field)
    // ══════════════════════════════════════════════════════════
    //
    // Input line format:
    //   E00001 | Maria Santos | Brazil | Engineering | 42000
    //
    // Map output:
    //   KEY = IntWritable(42000)
    //   VALUE = Text("E00001 | Maria Santos | Brazil | Engineering | 42000")
    //
    // WHY: TotalOrderPartitioner partitions by MAP OUTPUT KEY
    //      So the sort field MUST become the key

    public static class SalaryKeyMapper
            extends Mapper<LongWritable, Text, IntWritable, Text> {

        private IntWritable salaryKey = new IntWritable();

        @Override
        protected void map(LongWritable offset, Text line, Context context)
                throws IOException, InterruptedException {

            String record = line.toString().trim();

            // Skip empty lines and headers
            if (record.isEmpty() || record.startsWith("EMP_ID")) {
                return;
            }

            // Parse salary from last field
            // Format: EMP_ID | NAME | COUNTRY | DEPARTMENT | SALARY
            String[] fields = record.split("\\|");

            if (fields.length < 5) {
                // Malformed record — skip
                // Counter for debugging (Project 7)
                context.getCounter("QUALITY", "MALFORMED_RECORDS").increment(1);
                return;
            }

            try {
                int salary = Integer.parseInt(fields[4].trim());
                salaryKey.set(salary);

                // Emit salary as KEY, entire record as VALUE
                context.write(salaryKey, line);

                context.getCounter("QUALITY", "VALID_RECORDS").increment(1);
            } catch (NumberFormatException e) {
                context.getCounter("QUALITY", "BAD_SALARY_FORMAT").increment(1);
            }
        }
    }

    // ══════════════════════════════════════════════════════════
    // REDUCER: Identity — just output records in sorted order
    // ══════════════════════════════════════════════════════════
    //
    // Within each reducer, keys arrive SORTED (default behavior)
    // TotalOrderPartitioner ensures non-overlapping ranges
    // So we just emit each record as-is
    //
    // Output:
    //   KEY = Text("E00001 | Maria Santos | Brazil | Engineering | 42000")
    //   VALUE = NullWritable (we don't need a separate value)

    public static class SortedOutputReducer
            extends Reducer<IntWritable, Text, Text, NullWritable> {

        @Override
        protected void reduce(IntWritable salaryKey, Iterable<Text> records,
                              Context context)
                throws IOException, InterruptedException {

            // Multiple employees can have the SAME salary
            // Emit all of them — they are correctly grouped
            for (Text record : records) {
                context.write(record, NullWritable.get());
            }
        }
    }

    // ══════════════════════════════════════════════════════════
    // DRIVER: Wire everything together
    // ══════════════════════════════════════════════════════════

    @Override
    public int run(String[] args) throws Exception {

        if (args.length != 3) {
            System.err.println(
                "Usage: SalaryGlobalSort <input_path> <output_path> <num_reducers>");
            System.err.println(
                "Example: SalaryGlobalSort /data/employees /output/sorted 4");
            return -1;
        }

        String inputPath = args[0];
        String outputPath = args[1];
        int numReducers = Integer.parseInt(args[2]);

        Configuration conf = getConf();

        // ── STEP 1: Create and configure the job ──
        Job job = Job.getInstance(conf, "Global Sort - Employee Salaries");
        job.setJarByClass(SalaryGlobalSort.class);

        // Mapper and Reducer
        job.setMapperClass(SalaryKeyMapper.class);
        job.setReducerClass(SortedOutputReducer.class);

        // Map output types (what flows through shuffle)
        job.setMapOutputKeyClass(IntWritable.class);
        job.setMapOutputValueClass(Text.class);

        // Final output types
        job.setOutputKeyClass(Text.class);
        job.setOutputValueClass(NullWritable.class);

        // Input/Output formats
        job.setInputFormatClass(TextInputFormat.class);
        job.setOutputFormatClass(TextOutputFormat.class);

        // Paths
        FileInputFormat.addInputPath(job, new Path(inputPath));
        FileOutputFormat.setOutputPath(job, new Path(outputPath));

        // Number of reducers
        job.setNumReduceTasks(numReducers);

        // ── STEP 2: Set TotalOrderPartitioner ──
        // This REPLACES the default HashPartitioner (Project 10)
        job.setPartitionerClass(TotalOrderPartitioner.class);

        // ── STEP 3: Configure partition file path ──
        // TotalOrderPartitioner needs to know WHERE to read boundaries
        Path partitionFile = new Path(inputPath + "/_partition.lst");
        TotalOrderPartitioner.setPartitionFile(
            job.getConfiguration(), partitionFile);

        // ── STEP 4: Run InputSampler to create partition file ──
        //
        // RandomSampler parameters:
        //   freq = 0.001      → sample 0.1% of records
        //   numSamples = 10000 → collect at most 10000 samples
        //   maxSplits = 100    → sample from at most 100 input splits
        //
        // WHY these values:
        //   → 0.1% of 85M = 85K potential samples (capped at 10000)
        //   → 10000 samples for 4 reducers = 2500 samples per boundary
        //   → Statistically robust for most distributions

        System.out.println("=== PHASE 1: Sampling input data ===");
        System.out.println("Creating partition boundaries for "
            + numReducers + " reducers...");

        InputSampler.Sampler<IntWritable, Text> sampler =
            new InputSampler.RandomSampler<>(0.001, 10000, 100);

        // THIS is the critical call:
        // - Reads samples from input
        // - Sorts them
        // - Picks (numReducers - 1) evenly spaced boundaries
        // - Writes boundaries to partitionFile in HDFS
        // - Adds partitionFile to DistributedCache automatically
        InputSampler.writePartitionFile(job, sampler);

        System.out.println("Partition file written to: " + partitionFile);
        System.out.println("=== PHASE 2: Running global sort job ===");

        // ── STEP 5: Submit the job ──
        boolean success = job.waitForCompletion(true);

        if (success) {
            System.out.println("=== GLOBAL SORT COMPLETE ===");
            System.out.println("Output: " + outputPath);
            System.out.println("Files part-r-00000 through part-r-"
                + String.format("%05d", numReducers - 1)
                + " are globally sorted by salary");
            System.out.println("Concatenating them produces a fully "
                + "sorted dataset");
        }

        return success ? 0 : 1;
    }

    public static void main(String[] args) throws Exception {
        int exitCode = ToolRunner.run(new Configuration(),
            new SalaryGlobalSort(), args);
        System.exit(exitCode);
    }
}

🔑 KEY TAKEAWAY

WHAT PROJECT 18 ESTABLISHED:

✅ PROBLEM: Default MapReduce sorts WITHIN each reducer but NOT
   ACROSS reducers — HashPartitioner (Project 10) distributes
   keys by hash, not by value range

✅ SOLUTION: TotalOrderPartitioner + InputSampler
   → Sample data to discover distribution
   → Compute range boundaries
   → Partition by range instead of hash
   → Each reducer handles a non-overlapping range
   → Concatenation of outputs = globally sorted

✅ THREE SAMPLING STRATEGIES:
   → RandomSampler (safest, general purpose)
   → IntervalSampler (faster, for roughly sorted data)
   → SplitSampler (quick, for testing only)

✅ INTERNAL MECHANICS:
   → Sampling → sort samples → pick evenly-spaced boundaries
   → Boundaries stored in SequenceFile via DistributedCache (Proj 15)
   → Binary search (numeric) or Trie (text) at partition time
   → O(log R) per record where R = number of reducers

CONNECTION TO PREVIOUS PROJECTS:
   → Project 10: Introduced partitioners — this REPLACES
     HashPartitioner with TotalOrderPartitioner
   → Project 13: WritableComparable keys — TotalOrderPartitioner
     relies on compareTo() for binary search
   → Project 15: DistributedCache — partition file distributed
     to all mappers via this mechanism
   → Project 17: Reduce-Side Join output is the INPUT to this
     sort job — real pipeline thinking

WHAT WE HAVE NOT COVERED YET:
   → What if you need to sort by salary AND THEN by hire date
     within the same salary?
   → Global sort gives you ONE sort dimension
   → For TWO sort dimensions → Secondary Sort

🌉 BRIDGE TO PROJECT 19

PROJECT 18 sorted all employees by salary GLOBALLY.

But what if management asks:
"Group all transactions by CUSTOMER_ID, and within each
customer, sort transactions by DATE (newest first)"?

This requires TWO levels of ordering:
  → PRIMARY sort:   by customer_id (grouping)
  → SECONDARY sort: by date (ordering within group)

DEFAULT MapReduce cannot do this:
  → Reducer receives all values for a key
  → But values arrive in UNDEFINED ORDER
  → You get all transactions for customer C001
  → But dates are scrambled: [Mar, Jan, Dec, Jul]

PROJECT 19 introduces SECONDARY SORT:
  → Composite key: (customer_id, date)
  → Custom Partitioner: partition by customer_id ONLY
  → Custom SortComparator: sort by customer_id THEN date
  → Custom GroupingComparator: group by customer_id ONLY

This is the MOST COMPLEX MapReduce pattern —
and the final piece of the sorting puzzle before we move
to multi-output and job chaining.
