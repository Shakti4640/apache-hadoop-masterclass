PROJECT 1: THE OVERWHELMED LOG SERVER — WHY ONE MACHINE FAILS

🔴 HYPOTHETICAL PROBLEM

SCENARIO:
→ You are an engineer at a mid-size e-commerce company
→ Your web servers generate 50GB of access logs EVERY DAY
→ Management wants a daily report:
    - Total page views per product
    - Top 10 most visited pages
    - Count of unique visitors per hour
→ You have ONE server: 8 cores, 16GB RAM, 1TB disk

WHAT HAPPENS:
→ Day 1-30: Script runs in 20 minutes — fine
→ Day 31-90: Data grows to 1.5TB — disk full
→ You add more disk — now 4TB
→ Day 91-180: Script takes 4 hours — RAM chokes during sorting
→ Day 181-365: Script takes 14 hours — doesn't finish before next day's data arrives
→ Day 366: THE SYSTEM BREAKS — you cannot process data faster than it arrives

THIS IS THE EXACT PROBLEM HADOOP WAS BUILT TO SOLVE.

🔍 HOW TO APPROACH IT

STEP 1: Identify the bottleneck categories

┌─────────────────────────────────────────────────┐
│           SINGLE MACHINE BOTTLENECKS            │
├─────────────────────────────────────────────────┤
│ 1. STORAGE BOTTLENECK                           │
│    → Disk fills up — cannot store more data     │
│                                                 │
│ 2. COMPUTE BOTTLENECK                           │
│    → CPU/RAM cannot process fast enough         │
│                                                 │
│ 3. I/O BOTTLENECK                               │
│    → Disk read speed ~100-200 MB/s              │
│    → Reading 50GB takes ~5 minutes minimum      │
│    → Reading 5TB takes ~8 HOURS just to read    │
│                                                 │
│ 4. RELIABILITY BOTTLENECK                       │
│    → One disk fails = ALL data lost             │
│    → One machine crashes = ZERO processing      │
└─────────────────────────────────────────────────┘

STEP 2: Understand the two possible solutions

OPTION A: SCALE UP (Vertical Scaling)
→ Buy a BIGGER machine
→ 128 cores, 512GB RAM, 100TB RAID
→ Cost: $50,000 - $500,000
→ LIMIT: There is a ceiling — you cannot buy infinite hardware

OPTION B: SCALE OUT (Horizontal Scaling)
→ Buy MANY cheap machines
→ 100 machines × 8 cores × 16GB RAM × 1TB disk
→ Cost: $100,000 (commodity hardware)
→ LIMIT: You need SOFTWARE to coordinate them
→ THAT SOFTWARE IS HADOOP

💡 HOW CAN WE SOLVE IT

THE CORE INSIGHT:

Instead of bringing DATA to the PROGRAM (one big machine),
bring the PROGRAM to the DATA (many small machines).

┌─────────── TRADITIONAL APPROACH ───────────┐
│                                             │
│   [50GB Data] ──→ [ONE Machine] ──→ Result  │
│                    (bottleneck)              │
│                                             │
└─────────────────────────────────────────────┘

┌─────────── HADOOP APPROACH ────────────────┐
│                                             │
│   [10GB] → [Machine 1] → partial result ─┐ │
│   [10GB] → [Machine 2] → partial result ─┤ │
│   [10GB] → [Machine 3] → partial result ─┼→ COMBINE → Final Result
│   [10GB] → [Machine 4] → partial result ─┤ │
│   [10GB] → [Machine 5] → partial result ─┘ │
│                                             │
│   TIME: 5× faster (parallel processing)    │
│   STORAGE: 5× more (distributed storage)   │
│   RELIABILITY: If Machine 3 dies,           │
│                its data exists on Machine 1 │
│                (replication)                │
└─────────────────────────────────────────────┘

THIS REQUIRES TWO THINGS:
1. A DISTRIBUTED FILE SYSTEM   → HDFS (Hadoop Distributed File System)
2. A DISTRIBUTED COMPUTE MODEL → MapReduce

✅ WHAT COULD BE DONE

HADOOP SOLVES THIS WITH A 4-COMPONENT ARCHITECTURE:

┌──────────────────────────────────────────────────────────────────┐
│                     HADOOP ECOSYSTEM CORE                        │
├──────────────────────────────────────────────────────────────────┤
│                                                                  │
│  ┌─────────────┐   ┌─────────────┐   ┌──────────────────────┐  │
│  │   HDFS      │   │  MapReduce  │   │   YARN               │  │
│  │ (Storage)   │   │ (Processing)│   │ (Resource Management) │  │
│  └──────┬──────┘   └──────┬──────┘   └──────────┬───────────┘  │
│         │                 │                      │               │
│         └─────────────────┼──────────────────────┘               │
│                           │                                      │
│                  ┌────────┴────────┐                             │
│                  │  Hadoop Common  │                             │
│                  │  (Utilities)    │                             │
│                  └─────────────────┘                             │
└──────────────────────────────────────────────────────────────────┘

COMPONENT 1 — HDFS (Storage Layer)
→ Splits large files into BLOCKS (default 128MB)
→ Distributes blocks across machines
→ Replicates each block 3 times (fault tolerance)
→ One NameNode (master) tracks where blocks live
→ Many DataNodes (slaves) store actual blocks

COMPONENT 2 — MapReduce (Compute Layer)
→ MAP phase: Each machine processes its LOCAL data chunk
→ REDUCE phase: Results are combined across machines
→ You write the logic — Hadoop handles distribution

COMPONENT 3 — YARN (Resource Layer)
→ Decides WHICH machine runs WHICH task
→ Manages CPU and memory allocation
→ Ensures fair sharing among multiple jobs

COMPONENT 4 — Hadoop Common (Utility Layer)
→ Shared libraries and utilities
→ File system abstractions
→ RPC communication framework

✅ WHAT SHOULD DO

1. UNDERSTAND THE MASTER-SLAVE ARCHITECTURE FIRST

   ┌─────────────────────────────────────────────────────────────┐
   │                    HADOOP CLUSTER LAYOUT                     │
   │                                                              │
   │   ┌──────────────── MASTER NODE ──────────────────┐         │
   │   │  NameNode        (HDFS Master — metadata)     │         │
   │   │  ResourceManager  (YARN Master — scheduling)  │         │
   │   └───────────────────────────────────────────────┘         │
   │              │              │              │                  │
   │              ▼              ▼              ▼                  │
   │   ┌──────────────┐ ┌──────────────┐ ┌──────────────┐       │
   │   │ SLAVE NODE 1 │ │ SLAVE NODE 2 │ │ SLAVE NODE 3 │       │
   │   │              │ │              │ │              │         │
   │   │ DataNode     │ │ DataNode     │ │ DataNode     │        │
   │   │ NodeManager  │ │ NodeManager  │ │ NodeManager  │        │
   │   │              │ │              │ │              │         │
   │   │ [Block A]    │ │ [Block A']   │ │ [Block B]    │        │
   │   │ [Block C]    │ │ [Block B']   │ │ [Block C']   │        │
   │   └──────────────┘ └──────────────┘ └──────────────┘       │
   └─────────────────────────────────────────────────────────────┘

   MASTER NODE knows:
   → Which files exist
   → Which blocks make up each file
   → Which DataNodes hold which blocks
   → DOES NOT store actual data

   SLAVE NODES do:
   → Store actual data blocks
   → Execute computation tasks
   → Report health back to master

2. THINK "DATA LOCALITY" FROM DAY ONE
   → Hadoop moves COMPUTATION to where DATA already lives
   → NOT the other way around
   → This is the fundamental paradigm shift

3. ACCEPT THAT HARDWARE WILL FAIL
   → Hadoop ASSUMES machines will crash
   → Built-in replication handles this
   → This is not a bug — it is the DESIGN PHILOSOPHY

🚫 WHAT SHOULD NOT DO

1. DO NOT think of Hadoop as a database
   → Hadoop is NOT for random access queries
   → Hadoop is for BATCH processing of massive datasets
   → Querying one row from 1TB? Wrong tool.

2. DO NOT use Hadoop for small data
   → If your data fits in RAM of one machine → DON'T USE HADOOP
   → Hadoop overhead (JVM startup, task scheduling, network) 
     makes small jobs SLOWER than a simple Python script
   → RULE OF THUMB: If data < 10GB, question whether you need Hadoop

3. DO NOT ignore the Master as single point of failure
   → If NameNode dies, the ENTIRE cluster is inaccessible
   → (We address this in Project 44: NameNode HA)

4. DO NOT confuse Hadoop with real-time processing
   → MapReduce jobs take MINUTES to HOURS
   → Need sub-second responses? Wrong tool.
   → (We bridge to streaming in Project 47)

5. DO NOT try to modify files in HDFS
   → HDFS is WRITE-ONCE, READ-MANY
   → You cannot edit a file in place
   → You write a new file with the changes

🚫 WHEN NOT TO USE

HARD RULES — HADOOP IS THE WRONG TOOL WHEN:

1. DATA IS SMALL
   → Less than a few GB
   → Single machine handles it fine
   → Hadoop's overhead makes it SLOWER

2. LOW LATENCY IS REQUIRED
   → Need results in milliseconds or seconds
   → MapReduce minimum latency: ~30 seconds (JVM startup alone)
   → Use stream processing or in-memory databases instead

3. LOTS OF SMALL FILES
   → HDFS NameNode stores metadata for EVERY file in RAM
   → 1 million tiny files (1KB each) = 1GB data but NameNode overwhelmed
   → Each file's metadata costs ~150 bytes of NameNode memory
   → 100 million small files = 15GB of NameNode heap just for metadata

4. RANDOM READ/WRITE PATTERNS
   → HDFS optimized for sequential reads of large files
   → Random access is extremely slow
   → Use HBase (Project 27) or a traditional database

5. ITERATIVE ALGORITHMS
   → Machine learning algorithms that loop over data many times
   → MapReduce writes to disk EVERY iteration
   → Use Apache Spark (Project 48) instead

6. TIGHTLY COUPLED COMPUTATIONS
   → Tasks that need constant inter-node communication
   → MapReduce has ONE communication point (shuffle between Map and Reduce)
   → Use MPI or Spark for such workloads

INTERNAL MECHANIC 1: HOW HDFS STORES A FILE

Given: A 400MB file "access_log.txt"

Step 1 — Client contacts NameNode
   Client: "I want to store access_log.txt"
   NameNode: "OK, I'll split it into blocks"

Step 2 — Block splitting
   Block 1: bytes 0 - 128MB
   Block 2: bytes 128MB - 256MB
   Block 3: bytes 256MB - 384MB
   Block 4: bytes 384MB - 400MB (partial block, only 16MB)

Step 3 — NameNode decides placement (replication factor = 3)
   Block 1 → DataNode 1, DataNode 3, DataNode 5
   Block 2 → DataNode 2, DataNode 4, DataNode 1
   Block 3 → DataNode 3, DataNode 5, DataNode 2
   Block 4 → DataNode 4, DataNode 1, DataNode 3

Step 4 — Pipeline replication
   Client sends Block 1 to DataNode 1
   DataNode 1 simultaneously forwards to DataNode 3
   DataNode 3 simultaneously forwards to DataNode 5
   (NOT client sending 3 copies — it's a PIPELINE)

   ┌────────┐    ┌────────────┐    ┌────────────┐    ┌────────────┐
   │ Client │───→│ DataNode 1 │───→│ DataNode 3 │───→│ DataNode 5 │
   └────────┘    └────────────┘    └────────────┘    └────────────┘
                  (stores copy1)   (stores copy2)    (stores copy3)

Step 5 — NameNode records metadata
   Metadata stored IN MEMORY for fast lookups:
   {
     "access_log.txt": {
       "blocks": [
         {"id": "blk_001", "nodes": [DN1, DN3, DN5], "size": 128MB},
         {"id": "blk_002", "nodes": [DN2, DN4, DN1], "size": 128MB},
         {"id": "blk_003", "nodes": [DN3, DN5, DN2], "size": 128MB},
         {"id": "blk_004", "nodes": [DN4, DN1, DN3], "size": 16MB}
       ]
     }
   }

INTERNAL MECHANIC 2: HOW MAPREDUCE PROCESSES DATA

Given: Count page views per product from access_log.txt

Step 1 — INPUT SPLITS (one per block typically)
   Split 1 → assigned to Machine holding Block 1 (data locality!)
   Split 2 → assigned to Machine holding Block 2
   Split 3 → assigned to Machine holding Block 3
   Split 4 → assigned to Machine holding Block 4

Step 2 — MAP PHASE (runs in parallel on ALL 4 machines)
   Each mapper reads its local split line by line:
   
   Machine 1 reads Block 1:
     "GET /product/shoe" → emit("shoe", 1)
     "GET /product/hat"  → emit("hat", 1)
     "GET /product/shoe" → emit("shoe", 1)

   Machine 2 reads Block 2 simultaneously:
     "GET /product/hat"  → emit("hat", 1)
     "GET /product/bag"  → emit("bag", 1)

Step 3 — SHUFFLE & SORT (Hadoop handles this automatically)
   All values for the SAME KEY go to the SAME REDUCER:
   
   Reducer 1 receives: ("bag", [1])
   Reducer 2 receives: ("hat", [1, 1])
   Reducer 3 receives: ("shoe", [1, 1])

Step 4 — REDUCE PHASE (runs in parallel)
   Reducer 1: ("bag", [1])    → emit("bag", 1)
   Reducer 2: ("hat", [1, 1]) → emit("hat", 2)
   Reducer 3: ("shoe",[1, 1]) → emit("shoe", 2)

Step 5 — OUTPUT (written back to HDFS)
   part-r-00000: bag    1
   part-r-00001: hat    2
   part-r-00002: shoe   2

═══════════════════════════════════════════════════════════

INTERNAL MECHANIC 3: HEARTBEAT SYSTEM

   Every DataNode sends a HEARTBEAT to NameNode every 3 seconds
   
   ┌────────────┐  heartbeat (3s)  ┌──────────┐
   │ DataNode 1 │ ───────────────→ │          │
   │ DataNode 2 │ ───────────────→ │ NameNode │
   │ DataNode 3 │ ───────────────→ │          │
   │ DataNode 4 │ ─────── X ─────→ │          │
   └────────────┘  (missed!)       └──────────┘

   If NameNode receives no heartbeat for 10 minutes (default):
   → Declares DataNode 4 DEAD
   → Finds all blocks that were on DataNode 4
   → Re-replicates those blocks to other healthy DataNodes
   → Cluster HEALS ITSELF automatically

⚠️ COMMON PITFALLS AND FAILURE MODES

PITFALL 1: "I installed Hadoop but it's incredibly slow on my laptop"
→ REASON: Hadoop is designed for clusters, not single machines
→ Local/pseudo-distributed mode is for DEVELOPMENT ONLY
→ Real performance requires 3+ machines minimum

PITFALL 2: "NameNode ran out of memory"
→ REASON: Too many small files — each file/block uses ~150 bytes of heap
→ 200 million files × 150 bytes = 30GB heap needed
→ SOLUTION: Merge small files, use SequenceFile/HAR archives

PITFALL 3: "My MapReduce job runs but produces wrong results"
→ REASON: Block split happens at byte boundary, not line boundary
→ BUT — Hadoop's TextInputFormat handles this:
   → If a line spans two blocks, the mapper reading the FIRST block
     reads past the block boundary to complete the line
   → The mapper reading the SECOND block skips the partial first line
→ SOLUTION: Use standard InputFormats — they handle this correctly

PITFALL 4: "DataNode disk is full"
→ REASON: Replication factor 3 means 1TB of data needs 3TB of disk
→ Plus intermediate MapReduce data uses local disk
→ RULE: Plan for 3× raw data size + 30% overhead

PITFALL 5: "Cluster is up but no jobs can run"
→ REASON: YARN not started, or ResourceManager not configured
→ HDFS running ≠ Computation possible
→ HDFS and YARN are SEPARATE systems that must both be running

# ── PART A: INSTALL HADOOP (Pseudo-Distributed Mode on Ubuntu/Mac) ──

# Step 1: Prerequisites
sudo apt update
sudo apt install -y openjdk-11-jdk ssh rsync

# Verify Java
java -version
# Expected: openjdk version "11.x.x"

# Step 2: Set JAVA_HOME
echo 'export JAVA_HOME=/usr/lib/jvm/java-11-openjdk-amd64' >> ~/.bashrc
source ~/.bashrc

# Step 3: Download Hadoop 3.3.6
cd /opt
wget https://dlcdn.apache.org/hadoop/common/hadoop-3.3.6/hadoop-3.3.6.tar.gz
tar -xzf hadoop-3.3.6.tar.gz
mv hadoop-3.3.6 hadoop

# Step 4: Set Hadoop environment variables
cat >> ~/.bashrc << 'EOF'
export HADOOP_HOME=/opt/hadoop
export HADOOP_CONF_DIR=$HADOOP_HOME/etc/hadoop
export PATH=$PATH:$HADOOP_HOME/bin:$HADOOP_HOME/sbin
export HDFS_NAMENODE_USER=root
export HDFS_DATANODE_USER=root
export HDFS_SECONDARYNAMENODE_USER=root
export YARN_RESOURCEMANAGER_USER=root
export YARN_NODEMANAGER_USER=root
EOF
source ~/.bashrc

# Step 5: Configure SSH (Hadoop nodes communicate via SSH)
ssh-keygen -t rsa -P '' -f ~/.ssh/id_rsa
cat ~/.ssh/id_rsa.pub >> ~/.ssh/authorized_keys
chmod 600 ~/.ssh/authorized_keys
ssh localhost   # Should connect without password — type 'exit' to return

# Step 6: Set JAVA_HOME in Hadoop's env file
echo "export JAVA_HOME=/usr/lib/jvm/java-11-openjdk-amd64" >> $HADOOP_HOME/etc/hadoop/hadoop-env.sh
<!-- ── PART B: CONFIGURE HADOOP (4 Core Configuration Files) ── -->

<!-- FILE 1: $HADOOP_HOME/etc/hadoop/core-site.xml -->
<!-- PURPOSE: Tells Hadoop where the master NameNode lives -->
<configuration>
    <property>
        <name>fs.defaultFS</name>
        <value>hdfs://localhost:9000</value>
        <!-- This is the address of the NameNode -->
        <!-- ALL HDFS operations go through this address -->
    </property>
    <property>
        <name>hadoop.tmp.dir</name>
        <value>/opt/hadoop/tmp</value>
        <!-- Temporary directory for Hadoop data -->
    </property>
</configuration>

<!-- FILE 2: $HADOOP_HOME/etc/hadoop/hdfs-site.xml -->
<!-- PURPOSE: Configures HDFS behavior -->
<configuration>
    <property>
        <name>dfs.replication</name>
        <value>1</value>
        <!-- Replication factor = 1 for pseudo-distributed -->
        <!-- In production: 3 (default) -->
        <!-- We use 1 because we only have 1 machine -->
    </property>
    <property>
        <name>dfs.namenode.name.dir</name>
        <value>/opt/hadoop/data/namenode</value>
    </property>
    <property>
        <name>dfs.datanode.data.dir</name>
        <value>/opt/hadoop/data/datanode</value>
    </property>
</configuration>

<!-- FILE 3: $HADOOP_HOME/etc/hadoop/mapred-site.xml -->
<!-- PURPOSE: Tells MapReduce to use YARN as its execution engine -->
<configuration>
    <property>
        <name>mapreduce.framework.name</name>
        <value>yarn</value>
        <!-- MapReduce runs ON TOP of YARN -->
        <!-- YARN manages resources, MapReduce does computation -->
    </property>
    <property>
        <name>mapreduce.application.classpath</name>
        <value>$HADOOP_HOME/share/hadoop/mapreduce/*:$HADOOP_HOME/share/hadoop/mapreduce/lib/*</value>
    </property>
</configuration>

<!-- FILE 4: $HADOOP_HOME/etc/hadoop/yarn-site.xml -->
<!-- PURPOSE: Configures YARN resource management -->
<configuration>
    <property>
        <name>yarn.nodemanager.aux-services</name>
        <value>mapreduce_shuffle</value>
        <!-- Enables the shuffle service needed by MapReduce -->
    </property>
    <property>
        <name>yarn.nodemanager.env-whitelist</name>
        <value>JAVA_HOME,HADOOP_COMMON_HOME,HADOOP_HDFS_HOME,HADOOP_CONF_DIR,CLASSPATH_PREPEND_DISTCACHE,HADOOP_YARN_HOME,HADOOP_HOME,PATH,LANG,TZ,HADOOP_MAPRED_HOME</value>
    </property>
</configuration>

# ── PART E: PYTHON SCRIPT — PROVE WHY SINGLE MACHINE FAILS ──
# This demonstrates the PROBLEM that motivates Hadoop

import time
import os
import tempfile

def generate_fake_log(filepath, size_mb):
    """Generate a fake access log of specified size."""
    products = ["shoe", "hat", "bag", "shirt", "watch", "phone", "laptop"]
    line = "192.168.1.1 - - [01/Jan/2024:00:00:00] GET /product/{} HTTP/1.1 200\n"
    
    bytes_written = 0
    target_bytes = size_mb * 1024 * 1024
    
    with open(filepath, 'w') as f:
        i = 0
        while bytes_written < target_bytes:
            product = products[i % len(products)]
            entry = line.format(product)
            f.write(entry)
            bytes_written += len(entry)
            i += 1
    
    return bytes_written

def count_pageviews_single_machine(filepath):
    """Process log file on a SINGLE machine — the traditional way."""
    counts = {}
    
    start_time = time.time()
    
    with open(filepath, 'r') as f:
        for line in f:
            parts = line.strip().split()
            if len(parts) >= 7:
                # Extract product from URL: /product/shoe → shoe
                url = parts[6]
                product = url.split('/')[-1]
                counts[product] = counts.get(product, 0) + 1
    
    elapsed = time.time() - start_time
    return counts, elapsed

def simulate_distributed_processing(filepath, num_chunks):
    """
    Simulate what Hadoop does:
    Split the file into chunks and process each independently.
    Then combine results.
    This is NOT actual Hadoop — it demonstrates the CONCEPT.
    """
    file_size = os.path.getsize(filepath)
    chunk_size = file_size // num_chunks
    
    partial_results = []
    start_time = time.time()
    
    # ── MAP PHASE: Each "machine" processes its chunk ──
    with open(filepath, 'r') as f:
        for chunk_id in range(num_chunks):
            chunk_counts = {}
            bytes_read = 0
            
            for line in f:
                parts = line.strip().split()
                if len(parts) >= 7:
                    url = parts[6]
                    product = url.split('/')[-1]
                    chunk_counts[product] = chunk_counts.get(product, 0) + 1
                
                bytes_read += len(line)
                if bytes_read >= chunk_size and chunk_id < num_chunks - 1:
                    break
            
            partial_results.append(chunk_counts)
            print(f"  Mapper {chunk_id + 1} processed: {chunk_counts}")
    
    # ── REDUCE PHASE: Combine all partial results ──
    final_counts = {}
    for partial in partial_results:
        for product, count in partial.items():
            final_counts[product] = final_counts.get(product, 0) + count
    
    elapsed = time.time() - start_time
    return final_counts, elapsed

🔑 KEY TAKEAWAY

WHAT WE ESTABLISHED IN PROJECT 1:

✅ The PROBLEM: Single machines cannot handle data that grows
   faster than hardware improves
   
✅ The SOLUTION: Distribute data AND computation across
   many cheap machines
   
✅ The ARCHITECTURE: Master-Slave paradigm
   → NameNode (master) knows WHERE data is
   → DataNodes (slaves) STORE the actual data
   → ResourceManager (master) SCHEDULES computation
   → NodeManagers (slaves) EXECUTE computation
   
✅ The PHILOSOPHY:
   → Move computation to data (not data to computation)
   → Assume hardware WILL fail (replicate everything)
   → Write once, read many (immutable data)
   
✅ The SETUP: Hadoop running in pseudo-distributed mode
   with all 5 daemons verified
