PROJECT 13: COUNTING UNIQUE VISITORS TO A WEBSITE PER HOUR

🔴 HYPOTHETICAL PROBLEM

SCENARIO:
→ You are a data engineer at a major news website
→ 500 million page views per day from 20 million unique visitors
→ Web server logs arrive in standard Apache format:

  192.168.1.42 - - [15/Jan/2024:09:23:45 +0000] "GET /news/politics HTTP/1.1" 200 4523
  10.0.0.88 - - [15/Jan/2024:09:24:01 +0000] "GET /news/sports HTTP/1.1" 200 3891
  192.168.1.42 - - [15/Jan/2024:09:25:12 +0000] "GET /news/tech HTTP/1.1" 200 5102
  10.0.0.88 - - [15/Jan/2024:10:01:33 +0000] "GET /news/politics HTTP/1.1" 200 4523

→ Management wants: "How many UNIQUE visitors per HOUR?"

THE CHALLENGE:
→ Same IP visiting 50 pages in hour 09 = 1 unique visitor, not 50
→ Same IP visiting in hour 09 AND hour 10 = counted in BOTH hours
→ We need to GROUP BY (hour) and COUNT DISTINCT (ip) within each group

ATTEMPT 1: Use hour as key, IP as value
  Mapper emits: key="09", value="192.168.1.42"
  Reducer gets: key="09", values=["192.168.1.42", "10.0.0.88", "192.168.1.42"]
  Reducer must: collect ALL values into HashSet → count unique
  
  PROBLEM: For popular hours, reducer receives 100 million values
  → HashSet of 20 million IPs × ~40 bytes each = 800MB in memory
  → Reducer OOM crash

ATTEMPT 2: Use "hour|ip" as Text key
  Mapper emits: key="09|192.168.1.42", value=1
  Combiner: reduces duplicates locally (Project 9)
  Reducer gets: key="09|192.168.1.42", values=[1,1,1]
  
  PROBLEM: Each unique (hour, ip) pair becomes a SEPARATE reducer group
  → Reducer is called 20 million times for hour 09
  → We need a SECOND pass to count how many distinct keys per hour
  → Two jobs for what should be one operation

ATTEMPT 3: COMPOSITE KEY with custom grouping
  → Create a custom key class: HourIPKey(hour, ip)
  → Partition by HOUR only → all records for same hour go to same reducer
  → Sort by HOUR then IP → duplicates become adjacent
  → Group by HOUR → reducer sees all IPs for one hour together
  → Walk through sorted values, count when IP changes

  THIS IS THE COMPOSITE KEY PATTERN — the subject of this project.

🔍 HOW TO APPROACH IT

STEP 1: Understand why Text keys are insufficient

In Projects 4-12, all keys were simple types:
  → Text ("NORTH", "Accident", "Premium Plan")
  → IntWritable (year, temperature)

But our problem has TWO dimensions: HOUR and IP ADDRESS

Using Text "09|192.168.1.42" as key:
  → Hadoop treats it as ONE opaque string
  → Cannot partition by JUST the hour portion
  → Cannot sort by hour THEN by ip separately
  → Cannot group by JUST the hour portion

WE NEED: A key that Hadoop can DECOMPOSE into parts

STEP 2: Understand WritableComparable — the key contract

Every MapReduce key must implement WritableComparable<T>:

💡 HOW CAN WE SOLVE IT

SOLUTION: Composite Key with 3-Component Customization

✅ WHAT COULD BE DONE

APPROACH A: Composite key + grouping comparator (OUR CHOICE)
→ Custom WritableComparable key holding hour + IP
→ Custom Partitioner routing by hour
→ Custom GroupingComparator grouping by hour
→ Reducer walks sorted IPs, counts distinct
→ Memory efficient: no HashSet, just track "previous IP"
→ Works at ANY scale — constant memory per reducer

APPROACH B: Two-pass MapReduce (Project 21 chaining)
→ Job 1: Deduplicate (hour|ip) pairs → emit unique pairs
→ Job 2: Count unique IPs per hour
→ Correct but requires two full MapReduce cycles
→ Double the I/O and processing time

APPROACH C: Single pass with HashSet in reducer (Attempt 1 above)
→ Mapper: key=hour, value=ip
→ Reducer: collect all IPs into HashSet, count size
→ FAILS at scale: 20 million IPs in memory per hour
→ OOM for popular hours

APPROACH D: Use HyperLogLog approximation
→ Approximate distinct count with probabilistic data structure
→ Very memory efficient (~12KB per counter)
→ But: approximate, not exact
→ Good for monitoring dashboards, bad for billing reports
→ (Advanced technique beyond this project's scope)

WE CHOOSE APPROACH A because:
→ Exact results (not approximate)
→ Constant memory (no HashSet)
→ Single MapReduce job (efficient)
→ Demonstrates the full composite key pattern
→ Foundation for Secondary Sort (Project 19)

✅ WHAT SHOULD DO

1. IMPLEMENT ALL FIVE METHODS IN WritableComparable
   → write() — serialize fields in CONSISTENT order
   → readFields() — deserialize in SAME order as write()
   → compareTo() — define multi-field sort order
   → hashCode() — based on fields used for partitioning
   → equals() — consistent with compareTo()
   → toString() — human-readable for output files

2. MATCH SERIALIZATION ORDER EXACTLY
   → write(): write hour first, then ip
   → readFields(): read hour first, then ip
   → Mismatch → data corruption (hour and ip values swapped)
   → This is the #1 bug in custom Writable implementations

3. MAKE compareTo() CONSISTENT WITH equals()
   → If compareTo() returns 0, equals() must return true
   → If equals() returns true, compareTo() must return 0
   → Violation causes unpredictable sorting behavior

4. USE hashCode() BASED ON PARTITION FIELDS ONLY
   → Partitioner uses hour only → hashCode should use hour only
   → If hashCode includes IP, HashPartitioner distributes differently
     than our custom partitioner
   → Consistency prevents subtle routing bugs

5. REUSE KEY OBJECTS IN MAPPER
   → Create ONE HourIPKey instance, call setters per record
   → DO NOT create new HourIPKey() for every map() call
   → 500 million records × object creation = massive GC pressure
   → Hadoop's framework copies the key when needed

6. TEST SERIALIZATION ROUND-TRIP
   → Write key to bytes → read back → verify fields match
   → Do this BEFORE running on cluster
   → Serialization bugs are extremely hard to debug in distributed jobs

🚫 WHAT SHOULD NOT DO

1. DO NOT FORGET readFields() MATCHES write() ORDER
   → write(): out.writeUTF(hour); out.writeUTF(ip);
   → readFields(): hour = in.readUTF(); ip = in.readUTF();
   → WRONG: readFields(): ip = in.readUTF(); hour = in.readUTF();
   → No error thrown — just silently corrupted data
   → Hour becomes IP, IP becomes hour

2. DO NOT USE DEFAULT hashCode() FROM Object
   → Object.hashCode() is memory-address based
   → Same logical key → different hashCode across JVMs
   → Key ("09","192.168.1.42") on Mapper 1 gets hashCode 12345
   → Same key on Mapper 2 gets hashCode 67890
   → HashPartitioner sends to DIFFERENT reducers
   → Identical keys split across reducers = WRONG RESULTS
   → ALWAYS override hashCode() with field-based implementation

3. DO NOT MAKE MUTABLE FIELDS WITHOUT UNDERSTANDING CONSEQUENCES
   → Hadoop reuses key objects for efficiency
   → After context.write(key, value), framework may KEEP reference
   → If you modify key fields AFTER write, you corrupt the buffer
   → SAFE: create new key per write, OR let framework handle copying
   → Hadoop's MapOutputBuffer serializes immediately, so reuse is safe
     IF you set new values BEFORE next context.write()

4. DO NOT SKIP THE GROUPING COMPARATOR
   → Without GroupingComparator: each unique (hour,ip) = separate reduce()
   → Hour 09 with 20 million unique IPs = 20 million reduce() calls
   → Each call has ONE value → cannot count distinct
   → WITH GroupingComparator: hour 09 = ONE reduce() call
   → All 20 million IPs streamed through one call → count distinct

5. DO NOT CONFUSE SORT COMPARATOR WITH GROUPING COMPARATOR
   → SORT comparator: defines order of keys within reducer's input
     → Sort by hour THEN by ip → duplicates adjacent
   → GROUPING comparator: defines which keys are "same group"
     → Group by hour ONLY → all IPs for same hour in one reduce()
   → BOTH are needed for composite key pattern
   → Sort = ordering, Grouping = boundaries

6. DO NOT IMPLEMENT Writable INSTEAD OF WritableComparable
   → Writable = serializable but NOT comparable
   → Keys MUST be sortable → WritableComparable required
   → Values can be just Writable (no sorting needed)
   → Using Writable as key → compilation succeeds but
     runtime fails during sort phase

COMPOSITE KEY DECISION CHECKLIST:
  ✅ Need to partition by field A but sort by field A+B? → Composite key
  ✅ Need to group by field A but iterate sorted by B?   → Composite key
  ✅ Need to count distinct B values within each A?      → Composite key
  ✅ Need secondary sort within primary groups?          → Composite key
  ❌ Just need to group and aggregate by one field?      → Simple key

🚫 WHEN NOT TO USE

1. WHEN SIMPLE KEY SUFFICES
   → "Sum of sales per region" → key=region (Text) is enough
   → Adding composite key adds complexity for zero benefit
   → RULE: If you don't need custom partitioning OR custom grouping,
     don't use composite keys

2. WHEN APPROXIMATE COUNTS ARE ACCEPTABLE
   → Unique visitor counting for dashboards
   → HyperLogLog gives ~2% accuracy with 12KB memory
   → Much simpler than composite key pattern
   → Use when exactness is not business-critical

3. WHEN DATA FITS IN REDUCER MEMORY
   → If distinct values per group < 100,000
   → HashSet in reducer is simpler and works fine
   → 100,000 × 40 bytes = 4MB — trivial memory
   → Composite key pattern overkill for small cardinality

4. WHEN USING HIGHER-LEVEL TOOLS
   → Hive: SELECT hour, COUNT(DISTINCT ip) GROUP BY hour
   → Pig: DISTINCT then GROUP then COUNT
   → These tools generate composite key patterns automatically
   → (Projects 24, 26)
   → Write raw MapReduce composite keys only when you need
     maximum control or performance

⚙️ HOW IT ONLY WORKS — INTERNAL MECHANICS

═══════════════════════════════════════════════════════════
MECHANIC 1: SERIALIZATION/DESERIALIZATION CYCLE
═══════════════════════════════════════════════════════════

A composite key travels through MANY serialization cycles:

MAPPER SIDE:
  1. Mapper creates HourIPKey("09", "192.168.1.42")
  2. context.write(key, value)
  3. MapOutputBuffer calls key.write(DataOutput):
     → Writes to circular buffer as bytes:
     ┌─────────────────────────────────────────┐
     │ [2 bytes: hour length][hour bytes]       │
     │ [2 bytes: ip length][ip bytes]           │
     │ = [0x00 0x02][0x30 0x39]                │
     │   [0x00 0x0C][192.168.1.42 bytes]       │
     └─────────────────────────────────────────┘
  4. Buffer sorted by comparing SERIALIZED bytes
     → compareTo() called on deserialized keys
     → OR: RawComparator compares bytes directly (faster)

SPILL TO DISK:
  5. Buffer spills to local disk as sorted run
  6. Key bytes written as-is (already serialized)

SHUFFLE (NETWORK):
  7. Reducer requests partition data from mapper
  8. Key bytes sent over TCP connection
  9. Reducer receives bytes

REDUCER SIDE:
  10. Framework calls key.readFields(DataInput):
      → Reads bytes in SAME order as write():
      → hour = readUTF()  ← reads length prefix + string
      → ip = readUTF()    ← reads length prefix + string
  11. Grouping comparator checks: same group as previous key?
  12. If same group: add to current reduce() iteration
  13. If different group: start new reduce() call

TOTAL SERIALIZATION CYCLES PER KEY:
  → write() at mapper output: 1 time
  → Possibly re-read during merge of spill files: N times
  → write() during shuffle: 1 time
  → readFields() at reducer: 1 time
  → MINIMUM: 3 serialization operations per key
  → This is why efficient serialization matters at scale

═══════════════════════════════════════════════════════════
MECHANIC 2: SORT COMPARATOR vs GROUPING COMPARATOR
═══════════════════════════════════════════════════════════

These are TWO DIFFERENT comparators used at DIFFERENT stages:

SORT COMPARATOR (job.setSortComparatorClass):
  → Used DURING sort phase (map side AND reduce side merge)
  → Defines the TOTAL ORDER of keys
  → For our case: sort by hour first, then by IP
  → Result: keys are ordered as:
    (08, 10.0.0.1)
    (08, 10.0.0.88)
    (08, 192.168.1.42)
    (09, 10.0.0.1)      ← hour boundary
    (09, 10.0.0.88)
    (09, 192.168.1.42)
    (09, 192.168.1.42)  ← duplicate IP, adjacent
    (10, 10.0.0.88)     ← hour boundary

  DEFAULT: Uses key's compareTo() method
  → If HourIPKey.compareTo() sorts by hour then IP,
    no separate sort comparator needed

GROUPING COMPARATOR (job.setGroupingComparatorClass):
  → Used ONLY on reduce side
  → Defines which keys belong to the SAME reduce() call
  → For our case: group by hour ONLY (ignore IP)
  → Result:
    GROUP 1 (hour=08): (08,10.0.0.1), (08,10.0.0.88), (08,192.168.1.42)
    GROUP 2 (hour=09): (09,10.0.0.1), (09,10.0.0.88), (09,192.168.1.42), (09,192.168.1.42)
    GROUP 3 (hour=10): (10,10.0.0.88)

  → Each GROUP = one reduce() call
  → Within each call, keys arrive SORTED by IP (from sort phase)

  CRITICAL: Grouping comparator must be CONSISTENT with sort order
  → If sort puts hour 09 before hour 10,
    grouping must not mix hours 09 and 10 in same group
  → Grouping comparator must be a PREFIX of sort comparator
  → Sort: (hour, ip) → Group: (hour) ← hour is prefix ✓
  → Sort: (hour, ip) → Group: (ip) ← ip is NOT prefix ✗

  ┌────────────────────────────────────────────────────────┐
  │  SORT defines ORDER                                     │
  │  GROUPING defines BOUNDARIES                            │
  │                                                         │
  │  Sorted stream:                                         │
  │    (08,A) (08,B) (08,C) | (09,A) (09,B) (09,C) | ...  │
  │                         ↑                       ↑       │
  │                    group boundary          group boundary│
  │                                                         │
  │  Grouping comparator says:                              │
  │    "08 keys are same group, 09 keys are same group"     │
  │    Boundary = where hour changes                        │
  └────────────────────────────────────────────────────────┘

═══════════════════════════════════════════════════════════
MECHANIC 3: RawComparator — BYTE-LEVEL COMPARISON
═══════════════════════════════════════════════════════════

Hadoop offers an optimization: compare keys WITHOUT deserializing

Normal comparison:
  1. Deserialize key1 from bytes → HourIPKey object
  2. Deserialize key2 from bytes → HourIPKey object
  3. Call key1.compareTo(key2)
  → 2 object creations + 2 deserializations per comparison
  → During sort: millions of comparisons → millions of objects

RawComparator optimization:
  1. Compare bytes DIRECTLY without creating objects
  2. WritableComparator.compareBytes(b1, s1, l1, b2, s2, l2)
  → Zero object creation
  → 5-10× faster sorting

HOW TO ENABLE:
  → Extend WritableComparator instead of implementing RawComparator
  → Override compare(byte[] b1, int s1, int l1, byte[] b2, int s2, int l2)
  → For simple fields (ints, strings with length prefix): straightforward
  → For complex fields: may need to parse byte structure

For this project: we use object-level compareTo() for clarity
Project 19 (Secondary Sort) shows RawComparator optimization

═══════════════════════════════════════════════════════════
MECHANIC 4: HOW REDUCER ITERATES COMPOSITE KEY GROUPS
═══════════════════════════════════════════════════════════

When grouping comparator groups by hour only:

  reduce(HourIPKey key, Iterable<IntWritable> values, Context ctx)

  The 'key' parameter:
  → Set to the FIRST key in the group
  → For group hour=09: key = HourIPKey("09", "10.0.0.1") ← first sorted IP
  → Does NOT change as you iterate through values
  → To access individual IPs, you need a DIFFERENT approach

  The 'values' parameter:
  → Iterates through ALL values in the group
  → But we need the KEY (IP) for each value, not just the value

  TRICK: The framework reuses the KEY object
  → As you call values.iterator().next(), the framework
    ALSO updates the key object to match the current value
  → BUT: only if you access the key through the iterator
  → Standard Iterable<V> doesn't expose the key

  BETTER APPROACH FOR OUR PROBLEM:
  → Emit key=HourIPKey(hour, ip), value=Text(ip)
  → Value carries the IP redundantly
  → Reducer reads IP from value, not from key
  → Simpler and more reliable than framework key tricks

  OR: Use the sorted-adjacent-duplicates approach:
  → Since IPs are sorted within each hour group
  → Track "previousIP" — when IP changes, count new unique
  → Don't need to access individual keys at all

⚠️ COMMON PITFALLS AND FAILURE MODES

PITFALL 1: "Reducer produces wrong unique counts"
→ CAUSE: Grouping comparator not set
→ Without grouping comparator: each unique (hour,ip) = separate reduce()
→ Reducer sees ONE IP per call → counts 1 every time
→ FIX: job.setGroupingComparatorClass(HourGroupingComparator.class)

PITFALL 2: "Keys arrive in wrong order at reducer"
→ CAUSE: compareTo() has a bug
→ Common bug: comparing strings with < instead of compareTo()
→ "10" < "9" alphabetically but 10 > 9 numerically
→ FIX: Parse to int for numeric fields, or pad with zeros ("09", "10")
→ ALWAYS test compareTo() with unit tests before cluster run

PITFALL 3: "Serialization corrupts data — hours and IPs swapped"
→ CAUSE: write() and readFields() have different field order
→ write(): writeUTF(ip); writeUTF(hour);    ← IP first
→ readFields(): hour = readUTF(); ip = readUTF();  ← reads IP into hour
→ FIX: Document and verify field order matches exactly

PITFALL 4: "Some hours missing from output"
→ CAUSE: Partitioner sends multiple hours to same reducer
→ If 24 hours but only 4 reducers (from Project 10 pattern)
→ Each reducer handles ~6 hours — this is CORRECT
→ But if grouping comparator has a bug, hours might merge
→ VERIFY: Check that output has exactly 24 hour entries

PITFALL 5: "ClassNotFoundException for inner classes"
→ CAUSE: Custom comparator/partitioner as inner class of driver
→ Hadoop serializes class name for remote execution
→ Inner class name: "OuterClass$InnerClass"
→ If not included in JAR properly, remote node can't find it
→ FIX: Use static inner classes OR separate top-level classes
→ Ensure JAR includes all .class files

PITFALL 6: "hashCode() inconsistency causes data loss"
→ CAUSE: hashCode() includes IP field
→ Custom partitioner routes by hour
→ HashPartitioner (if accidentally used) routes by hashCode
→ If both partition methods are inconsistent, framework
   may lose track of which reducer has which data
→ FIX: hashCode() should be based on partitioning field (hour)

import org.apache.hadoop.io.WritableComparable;
import java.io.DataInput;
import java.io.DataOutput;
import java.io.IOException;

/**
 * Composite key holding HOUR and IP ADDRESS.
 * 
 * Sort order: by hour (ascending), then by IP (ascending).
 * This ensures duplicate IPs within same hour are ADJACENT.
 * 
 * Implements WritableComparable (not just Writable) because
 * MapReduce keys MUST be sortable.
 */
public class HourIPKey implements WritableComparable<HourIPKey> {

    private String hour;    // "00" to "23"
    private String ip;      // "192.168.1.42"

    // ── DEFAULT CONSTRUCTOR REQUIRED ──
    // Hadoop instantiates via reflection: Class.newInstance()
    // Then calls readFields() to populate
    // Without this: InstantiationException at runtime
    public HourIPKey() {
        this.hour = "";
        this.ip = "";
    }

    public HourIPKey(String hour, String ip) {
        this.hour = hour;
        this.ip = ip;
    }

    // ── GETTERS AND SETTERS ──
    // Setters allow mapper to REUSE one key object (no new allocation per record)
    public String getHour() { return hour; }
    public String getIp() { return ip; }

    public void set(String hour, String ip) {
        this.hour = hour;
        this.ip = ip;
    }


    // ════════════════════════════════════════════════════════
    // SERIALIZATION: write() and readFields()
    // ════════════════════════════════════════════════════════
    //
    // CRITICAL: Field order in write() MUST match readFields()
    //
    // writeUTF() writes: 2-byte length prefix + UTF-8 bytes
    // readUTF() reads:   2-byte length prefix + that many bytes
    //
    // This is called MILLIONS of times during a job:
    //   → Spilling map output to disk
    //   → Shuffling to reducers over network
    //   → Merging sorted runs on reducer side

    @Override
    public void write(DataOutput out) throws IOException {
        out.writeUTF(hour);   // Field 1: hour
        out.writeUTF(ip);     // Field 2: ip
        // ORDER MATTERS: readFields must read in this exact order
    }

    @Override
    public void readFields(DataInput in) throws IOException {
        hour = in.readUTF();  // Field 1: hour (matches write order)
        ip = in.readUTF();    // Field 2: ip (matches write order)
    }


    // ════════════════════════════════════════════════════════
    // COMPARISON: compareTo() — defines sort order
    // ════════════════════════════════════════════════════════
    //
    // Sort by HOUR first (primary sort)
    // Then by IP within same hour (secondary sort)
    //
    // This ensures:
    //   (09, 10.0.0.1) comes before (09, 192.168.1.42)
    //   (09, anything) comes before (10, anything)
    //   Duplicate IPs within same hour are ADJACENT

    @Override
    public int compareTo(HourIPKey other) {
        // Primary sort: by hour
        int hourCompare = this.hour.compareTo(other.hour);
        if (hourCompare != 0) {
            return hourCompare;
        }
        // Secondary sort: by IP (within same hour)
        return this.ip.compareTo(other.ip);
    }


    // ════════════════════════════════════════════════════════
    // hashCode() and equals() — for partitioning and grouping
    // ════════════════════════════════════════════════════════
    //
    // hashCode() based on HOUR only
    // WHY: Our custom partitioner routes by hour
    //      If HashPartitioner is accidentally used,
    //      it should still group same hours together
    //
    // equals() based on BOTH fields
    // WHY: Two keys are truly equal only if hour AND ip match

    @Override
    public int hashCode() {
        // Based on hour only — consistent with partitioning strategy
        return hour.hashCode();
    }

    @Override
    public boolean equals(Object obj) {
        if (this == obj) return true;
        if (obj == null || getClass() != obj.getClass()) return false;
        HourIPKey other = (HourIPKey) obj;
        return this.hour.equals(other.hour) && this.ip.equals(other.ip);
    }


    // ════════════════════════════════════════════════════════
    // toString() — used by TextOutputFormat for output files
    // ════════════════════════════════════════════════════════

    @Override
    public String toString() {
        return hour + "\t" + ip;
    }
}

import org.apache.hadoop.io.Text;
import org.apache.hadoop.mapreduce.Partitioner;

/**
 * Routes all records for the SAME HOUR to the SAME reducer.
 * 
 * Different from Project 10's RegionPartitioner:
 *   → Project 10: explicit mapping (NORTH→0, SOUTH→1)
 *   → This: hash-based on hour field only
 *   → We don't need specific hour→reducer mapping
 *   → We just need same hour → same reducer
 * 
 * Uses hash of hour to distribute across available reducers.
 */
public class HourPartitioner extends Partitioner<HourIPKey, Text> {

    @Override
    public int getPartition(HourIPKey key, Text value, int numReduceTasks) {
        // Partition by HOUR only — ignore IP
        // Same hour always goes to same reducer
        // Different hours may share a reducer (if numReducers < 24)
        return (key.getHour().hashCode() & Integer.MAX_VALUE) % numReduceTasks;
    }
}

import org.apache.hadoop.io.WritableComparable;
import org.apache.hadoop.io.WritableComparator;

/**
 * Tells the reducer framework: "Keys with the same HOUR
 * belong to the same reduce() call, regardless of IP."
 * 
 * WITHOUT this comparator:
 *   → Each unique (hour, ip) pair = separate reduce() call
 *   → Hour 09 with 20M unique IPs = 20M reduce() calls
 *   → Each call has 1 value → cannot count distinct
 * 
 * WITH this comparator:
 *   → All keys with hour=09 = ONE reduce() call
 *   → Reducer iterates through sorted IPs
 *   → Counts when IP changes = unique visitor count
 * 
 * IMPORTANT: This comparator must be CONSISTENT with sort order.
 * Sort: by (hour, ip) — grouping boundary: where hour changes.
 * This works because hour is a PREFIX of the sort key.
 */
public class HourGroupingComparator extends WritableComparator {

    // ── CONSTRUCTOR: Register the key class ──
    // WritableComparator needs to know which class to instantiate
    // for deserialization during comparison
    protected HourGroupingComparator() {
        super(HourIPKey.class, true);
        // 'true' means: create instances for comparison
        // Without this: NullPointerException during reduce
    }

    @Override
    @SuppressWarnings("rawtypes")
    public int compare(WritableComparable a, WritableComparable b) {
        HourIPKey key1 = (HourIPKey) a;
        HourIPKey key2 = (HourIPKey) b;

        // Compare by HOUR only — ignore IP
        // All keys with same hour → same reduce() call
        return key1.getHour().compareTo(key2.getHour());
    }
}

import org.apache.hadoop.conf.Configuration;
import org.apache.hadoop.conf.Configured;
import org.apache.hadoop.fs.Path;
import org.apache.hadoop.io.IntWritable;
import org.apache.hadoop.io.LongWritable;
import org.apache.hadoop.io.Text;
import org.apache.hadoop.mapreduce.Job;
import org.apache.hadoop.mapreduce.Mapper;
import org.apache.hadoop.mapreduce.Reducer;
import org.apache.hadoop.mapreduce.lib.input.FileInputFormat;
import org.apache.hadoop.mapreduce.lib.output.FileOutputFormat;
import org.apache.hadoop.util.Tool;
import org.apache.hadoop.util.ToolRunner;

import java.io.IOException;
import java.util.regex.Matcher;
import java.util.regex.Pattern;

public class UniqueVisitorCounter extends Configured implements Tool {

    // ════════════════════════════════════════════════════════
    // MAPPER: Parse Apache log → emit (HourIPKey, ip_as_text)
    // ════════════════════════════════════════════════════════
    //
    // Apache Combined Log Format:
    //   IP - - [DD/Mon/YYYY:HH:MM:SS +ZONE] "METHOD URL PROTO" STATUS SIZE
    //
    // Example:
    //   192.168.1.42 - - [15/Jan/2024:09:23:45 +0000] "GET /news HTTP/1.1" 200 4523
    //
    // We extract: IP and HH (hour)
    // Emit: key=HourIPKey(hour, ip), value=Text(ip)
    //
    // WHY value=ip (redundant with key)?
    //   → Grouping comparator makes reducer see keys grouped by hour
    //   → The 'key' parameter in reduce() is the FIRST key in group
    //   → To see individual IPs, we need them in the VALUE stream
    //   → Alternative: access key per iteration (framework-dependent trick)
    //   → Putting IP in value is SIMPLER and MORE PORTABLE

    public static class LogMapper
            extends Mapper<LongWritable, Text, HourIPKey, Text> {

        // ── Compiled ONCE in setup (Project 11 lesson) ──
        private Pattern logPattern;
        private HourIPKey outputKey = new HourIPKey();  // Reuse single instance
        private Text outputValue = new Text();

        enum MapCounters {
            TOTAL_LINES,
            PARSED_OK,
            PARSE_FAILED,
            EMPTY_LINE
        }

        @Override
        protected void setup(Context context) {
            // Apache log pattern — compile ONCE
            // Group 1: IP address
            // Group 2: Hour (HH from timestamp)
            logPattern = Pattern.compile(
                "^(\\S+)\\s+\\S+\\s+\\S+\\s+\\[(\\d{2}/\\w{3}/\\d{4}):(\\d{2}):\\d{2}:\\d{2}.*"
            );
            // Group 1: IP (e.g., 192.168.1.42)
            // Group 2: Date (e.g., 15/Jan/2024)
            // Group 3: Hour (e.g., 09)
        }

        @Override
        protected void map(LongWritable offset, Text line, Context context)
                throws IOException, InterruptedException {

            context.getCounter(MapCounters.TOTAL_LINES).increment(1);

            String logLine = line.toString().trim();
            if (logLine.isEmpty()) {
                context.getCounter(MapCounters.EMPTY_LINE).increment(1);
                return;
            }

            Matcher matcher = logPattern.matcher(logLine);

            if (!matcher.matches()) {
                context.getCounter(MapCounters.PARSE_FAILED).increment(1);
                return;
            }

            String ip = matcher.group(1);
            String hour = matcher.group(3);  // "09", "10", "23", etc.

            // ── Set composite key fields (reuse object) ──
            outputKey.set(hour, ip);
            outputValue.set(ip);

            context.write(outputKey, outputValue);
            context.getCounter(MapCounters.PARSED_OK).increment(1);
        }
    }


    // ════════════════════════════════════════════════════════
    // COMBINER: Deduplicate (hour, ip) pairs locally
    // ════════════════════════════════════════════════════════
    //
    // Before shuffle: many duplicate (hour, ip) pairs from same mapper
    // Combiner emits each unique (hour, ip) only ONCE
    //
    // NOTE: Combiner uses the FULL composite key for grouping
    //       (not the grouping comparator — that's reducer only)
    //       So combiner sees: key=(09, 192.168.1.42), values=[ip, ip, ip]
    //       Emits: key=(09, 192.168.1.42), value=ip (once)
    //
    // This REDUCES shuffle volume significantly:
    //   Without combiner: 500M records shuffled
    //   With combiner: ~200M unique (hour, ip) pairs shuffled

    public static class DeduplicatingCombiner
            extends Reducer<HourIPKey, Text, HourIPKey, Text> {

        @Override
        protected void reduce(HourIPKey key, Iterable<Text> values, Context context)
                throws IOException, InterruptedException {
            // Emit the key-value ONCE regardless of how many duplicates
            // We only need the first value (all values are same IP)
            context.write(key, values.iterator().next());
        }
    }


    // ════════════════════════════════════════════════════════
    // REDUCER: Count unique IPs per hour
    // ════════════════════════════════════════════════════════
    //
    // Thanks to grouping comparator: ONE reduce() call per HOUR
    // Thanks to sort comparator: IPs arrive sorted within each hour
    // Thanks to combiner: many duplicates already eliminated
    //
    // Algorithm:
    //   Track previousIP
    //   For each value (IP):
    //     If IP != previousIP → new unique visitor
    //     Update previousIP
    //   Emit: hour → unique_count
    //
    // MEMORY: O(1) — only stores one previous IP string
    // Compare with HashSet: O(unique_visitors) — potentially GB

    public static class UniqueCountReducer
            extends Reducer<HourIPKey, Text, Text, IntWritable> {

        private Text hourOutput = new Text();
        private IntWritable countOutput = new IntWritable();

        enum ReduceCounters {
            HOURS_PROCESSED,
            TOTAL_UNIQUE_VISITORS,
            TOTAL_RECORDS_SEEN
        }

        @Override
        protected void reduce(HourIPKey key, Iterable<Text> ips, Context context)
                throws IOException, InterruptedException {

            // key.getHour() = the hour for this entire group
            String currentHour = key.getHour();
            String previousIP = null;
            int uniqueCount = 0;
            long totalRecords = 0;

            for (Text ipText : ips) {
                totalRecords++;
                String currentIP = ipText.toString();

                // ── Because IPs are SORTED, duplicates are ADJACENT ──
                // When IP changes → new unique visitor
                if (!currentIP.equals(previousIP)) {
                    uniqueCount++;
                    previousIP = currentIP;
                }
            }

            // Emit: hour → unique visitor count
            hourOutput.set(currentHour);
            countOutput.set(uniqueCount);
            context.write(hourOutput, countOutput);

            // Track counters
            context.getCounter(ReduceCounters.HOURS_PROCESSED).increment(1);
            context.getCounter(ReduceCounters.TOTAL_UNIQUE_VISITORS).increment(uniqueCount);
            context.getCounter(ReduceCounters.TOTAL_RECORDS_SEEN).increment(totalRecords);
        }
    }


    // ════════════════════════════════════════════════════════
    // DRIVER: Wire all components together
    // ════════════════════════════════════════════════════════

    @Override
    public int run(String[] args) throws Exception {
        if (args.length != 2) {
            System.err.println("Usage: UniqueVisitorCounter <input> <output>");
            return 1;
        }

        Configuration conf = getConf();
        Job job = Job.getInstance(conf, "Unique Visitors Per Hour - Composite Key Pattern");
        job.setJarByClass(UniqueVisitorCounter.class);

        // ── Mapper ──
        job.setMapperClass(LogMapper.class);

        // ── Combiner (deduplicates locally) ──
        job.setCombinerClass(DeduplicatingCombiner.class);

        // ── Reducer ──
        job.setReducerClass(UniqueCountReducer.class);
        job.setNumReduceTasks(6);  // 6 reducers for 24 hours (4 hours each)

        // ── CUSTOM PARTITIONER: Route by hour only ──
        // From Project 10 concept — now partitioning on composite key's sub-field
        job.setPartitionerClass(HourPartitioner.class);

        // ── CUSTOM GROUPING COMPARATOR: Group by hour only ──
        // THIS IS THE KEY NEW CONCEPT
        // Without this: each (hour, ip) pair = separate reduce() call
        // With this: all (hour, *) keys = one reduce() call per hour
        job.setGroupingComparatorClass(HourGroupingComparator.class);

        // ── Map output types (composite key + text value) ──
        job.setMapOutputKeyClass(HourIPKey.class);
        job.setMapOutputValueClass(Text.class);

        // ── Final output types (hour text + unique count) ──
        job.setOutputKeyClass(Text.class);
        job.setOutputValueClass(IntWritable.class);

        // ── Input/Output paths ──
        FileInputFormat.addInputPath(job, new Path(args[0]));
        FileOutputFormat.setOutputPath(job, new Path(args[1]));

        // ── Run ──
        boolean success = job.waitForCompletion(true);

        if (success) {
            System.out.println("\n╔══════════════════════════════════════════════╗");
            System.out.println("║     UNIQUE VISITOR ANALYSIS COMPLETE          ║");
            System.out.println("╠══════════════════════════════════════════════╣");

            long hoursProcessed = job.getCounters()
                .findCounter(UniqueCountReducer.ReduceCounters.HOURS_PROCESSED)
                .getValue();
            long totalUnique = job.getCounters()
                .findCounter(UniqueCountReducer.ReduceCounters.TOTAL_UNIQUE_VISITORS)
                .getValue();
            long totalRecords = job.getCounters()
                .findCounter(UniqueCountReducer.ReduceCounters.TOTAL_RECORDS_SEEN)
                .getValue();
            long mapInput = job.getCounters()
                .findCounter(LogMapper.MapCounters.PARSED_OK)
                .getValue();

            System.out.printf("║  Hours processed:          %,d%n", hoursProcessed);
            System.out.printf("║  Total unique visitors:    %,d%n", totalUnique);
            System.out.printf("║  Records at reducer:       %,d%n", totalRecords);
            System.out.printf("║  Records at mapper:        %,d%n", mapInput);
            System.out.printf("║  Combiner dedup savings:   %,d records%n",
                    mapInput - totalRecords);
            System.out.printf("║  Dedup ratio:              %.1f%%%n",
                    (mapInput - totalRecords) * 100.0 / Math.max(1, mapInput));
            System.out.println("╚══════════════════════════════════════════════╝\n");
        }

        return success ? 0 : 1;
    }


    public static void main(String[] args) throws Exception {
        int exitCode = ToolRunner.run(new Configuration(),
                                      new UniqueVisitorCounter(), args);
        System.exit(exitCode);
    }
}

# writeUTF writes: 2-byte big-endian length + UTF-8 bytes

def write_utf(s):
    encoded = s.encode('utf-8')
    length = len(encoded)
    return struct.pack('>H', length) + encoded

def read_utf(data, offset):
    length = struct.unpack('>H', data[offset:offset+2])[0]
    s = data[offset+2:offset+2+length].decode('utf-8')
    return s, offset + 2 + length
