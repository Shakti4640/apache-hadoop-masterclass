💡 HOW CAN WE SOLVE IT

DIAGNOSTIC FRAMEWORK — THE 5-STEP PROCESS:

STEP 1: CHECK JOB STATUS
  → Did the job succeed, fail, or is it still running?
  → yarn application -list -appStates ALL
  → Quick answer: SUCCESS / FAILED / RUNNING / KILLED

STEP 2: READ FRAMEWORK COUNTERS
  → How many records were read? Written?
  → How many bytes transferred during shuffle?
  → How long did map vs. reduce take?
  → These counters exist for EVERY job automatically

STEP 3: READ CUSTOM COUNTERS
  → How many records passed your validation?
  → How many were filtered? Why?
  → How many had parse errors?
  → These counters exist only if YOUR code created them

STEP 4: IDENTIFY PROBLEM TASKS
  → Which specific mapper/reducer was slow or failed?
  → How many attempts were made before success/failure?
  → Was the slowness on one task or all tasks?

STEP 5: READ TASK LOGS
  → What exception caused the failure?
  → What was the last record processed before crash?
  → What does stderr show?

DIAGNOSTIC RULES OF THUMB:

Rule 1: Map input records ≈ number of lines in input files
  → If much LESS: wrong input path or files empty
  → If much MORE: reading compressed files without proper codec

Rule 2: Map output records ≈ Map input records (for most jobs)
  → If much LESS: mapper is filtering aggressively (check custom counters)
  → If ZERO: mapper emits nothing (bug in parsing logic)

Rule 3: Reduce input records = Map output records - Combine savings
  → If combiner used: Reduce input records < Map output records (good!)
  → Without combiner: Reduce input records = Map output records

Rule 4: Reduce shuffle bytes = data transferred over network
  → This is the MOST EXPENSIVE metric for performance
  → Minimize this with Combiners (Project 9)

Rule 5: GC time > 10% of CPU time = memory pressure
  → Too many objects being created and discarded
  → Review Writable object reuse (Project 4 lesson)

Rule 6: Spilled Records >> Map output records = insufficient sort buffer
  → Records spilled to disk multiple times
  → Increase mapreduce.task.io.sort.mb (default 100MB)

Rule 7: Data-local map tasks should be close to total map tasks
  → If most mappers are NOT data-local: cluster is overloaded
  → Or HDFS block placement is suboptimal

CUSTOM COUNTER DESIGN PRINCIPLES:

1. GROUP counters by functional area:
   → "DataQuality" group: malformed, filtered, valid counts
   → "BusinessLogic" group: specific condition tracking
   → "Performance" group: custom timing measurements

2. COUNTER NAMING should be self-explanatory:
   → BAD:  context.getCounter("C1", "x").increment(1)
   → GOOD: context.getCounter("DataQuality", "missing_temperature_field").increment(1)

3. USE ENUM-based counters for compile-time safety:
   → Define an enum with all counter names
   → context.getCounter(MyCounters.MALFORMED_LINES).increment(1)
   → Prevents typos in counter names
   → Hadoop groups enum counters by enum class name

4. TRACK RATIOS, not just absolute numbers:
   → valid_records / total_records = data quality percentage
   → filtered_records / total_records = filter impact
   → map_output_records / map_input_records = selectivity

✅ WHAT SHOULD DO

1. ALWAYS check counters after every job — even successful ones
   → A job can succeed with 0 records processed (empty output)
   → A job can succeed but miss 50% of data (filter bug)
   → Counters are the ONLY way to verify correctness at scale

2. ALWAYS define an ENUM for your custom counters
   → Compile-time type safety (no typos)
   → Grouped together in counter output
   → Self-documenting code

3. USE the Job object to read counters programmatically
   → job.getCounters().findCounter(group, name).getValue()
   → Build automated validation after job completion
   → Fail the pipeline if counters show data quality issues

4. CHECK task attempt count in Web UI
   → If a task was attempted 4 times before succeeding:
     something is flaky (network, disk, data)
   → Even if the job succeeded, investigate
   → Default max attempts: 4 for mappers, 4 for reducers

5. ENABLE log aggregation for post-mortem analysis
   → yarn.log-aggregation-enable = true in yarn-site.xml
   → Logs collected from all NodeManagers after job completes
   → Available via: yarn logs -applicationId <app_id>

6. USE the Job History Server for completed jobs
   → Start: mapred --daemon start historyserver
   → URL: http://localhost:19888
   → Shows completed jobs even after cluster restart
   → Preserves counter values and task details

🚫 WHAT SHOULD NOT DO

1. DO NOT use System.out.println for debugging in production
   → stdout goes to individual task attempt log files
   → With 1000 mappers: your output is scattered across 1000 log files
   → Finding your debug message: needle in a haystack
   → USE INSTEAD: Counters for numeric tracking
   → USE INSTEAD: LOG4J logger with appropriate log level

2. DO NOT ignore failed task attempts that eventually succeeded
   → If mapper attempt_0 fails but attempt_1 succeeds:
     the job reports SUCCESS
   → But the failure indicates a problem (bad data, flaky node, etc.)
   → Next time, you might not be lucky with the retry
   → INVESTIGATE why attempt_0 failed

3. DO NOT create thousands of unique counter names dynamically
   → BAD: context.getCounter("Products", productName).increment(1)
   → With 50,000 products: 50,000 counters → huge memory overhead
   → Counters are aggregated in ApplicationMaster memory
   → Too many counters → ApplicationMaster OOM
   → USE INSTEAD: Fixed set of counters (valid, invalid, filtered, etc.)
   → LIMIT: Keep total counter count under ~500

4. DO NOT rely on wall-clock time alone for performance analysis
   → Job took 10 minutes, but was it CPU-bound or I/O-bound?
   → Check: CPU time spent vs. elapsed time
   → Check: GC time vs. CPU time
   → Check: Shuffle bytes vs. map output bytes

5. DO NOT skip checking Data-local map tasks counter
   → If 0 out of 100 mappers are data-local: SERIOUS PROBLEM
   → All data being transferred over network to remote mappers
   → Job will be many times slower than necessary
   → CAUSE: Cluster overloaded, or data placement issue

6. DO NOT ignore Spilled Records counter
   → Spilled records = data written to disk during sort
   → If spilled records >> map output records: multiple spills happening
   → Each spill = extra disk I/O = slower job
   → FIX: Increase mapreduce.task.io.sort.mb

🚫 WHEN NOT TO USE

1. DO NOT use counters for per-record debugging
   → Counters are AGGREGATE numbers (totals across all tasks)
   → Cannot tell you "line 5732 in file X was malformed"
   → For per-record debugging: use task attempt logs with conditional logging
   → Or: run a small sample locally first

2. DO NOT use the Web UI for automated monitoring
   → Web UI is for human investigation
   → For automated alerting: use programmatic counter API
   → Or: parse yarn logs output in scripts

3. DO NOT rely on elapsed wall time for comparison
   → Cluster load varies (other jobs competing for resources)
   → A job that took 5 minutes today might take 15 minutes tomorrow
   → Compare CPU TIME counters for consistent metrics
   → Compare record throughput (records / CPU time) not wall time

4. DO NOT keep Job History Server data forever
   → History files accumulate in HDFS at /mr-history/
   → Old history consumes HDFS space
   → Configure retention: mapreduce.jobhistory.max-age-ms
   → Default: 7 days in some configurations

⚠️ COMMON PITFALLS AND FAILURE MODES

PITFALL 1: "Job shows map 100% reduce 0% for a long time"
→ DIAGNOSIS: Check Reduce shuffle bytes counter
→ If shuffle bytes is HUGE: too much data being transferred
→ CAUSE: No combiner, or mapper emitting too many records
→ FIX: Add combiner (Project 9), or filter more in mapper
→ ALTERNATIVE: reduce 0-33% is shuffle, stuck at 33% = one slow mapper

PITFALL 2: "4 mapper tasks took 30 seconds each, 1 took 45 minutes"
→ DIAGNOSIS: Data skew — one mapper got much more work
→ CAUSE 1: One input file is much larger than others
→ CAUSE 2: One key has vastly more values (hot key)
→ CHECK: Web UI → Map Tasks → sort by elapsed time
→ FIX: Ensure even input splits, salt hot keys (Project 33)

PITFALL 3: "Job failed with 'Container killed by YARN for exceeding memory'"
→ DIAGNOSIS: Task used more memory than its container limit
→ CAUSE: Large in-mapper HashMap (Project 6 pattern)
→ CAUSE: Accumulating values in reducer instead of streaming
→ CHECK: Physical memory snapshot counter
→ FIX: Increase mapreduce.map.memory.mb or reduce data structure size

PITFALL 4: "GC time is 40% of CPU time"
→ DIAGNOSIS: Too many Java objects being created
→ CAUSE: Creating new Text/IntWritable per record (Project 4 warning)
→ CAUSE: String concatenation in tight loops
→ CHECK: GC time elapsed counter vs. CPU time spent counter
→ FIX: Reuse Writable objects, minimize object allocation

    public static class DiagnosticMapper
            extends Mapper<LongWritable, Text, Text, LongWritable> {
        
        private final Text outputKey = new Text();
        private final LongWritable outputValue = new LongWritable();
        
        // Columns: caller_id(0), receiver_id(1), type(2), 
        // duration_sec(3), cost_usd(4), timestamp(5), tower_id(6)
        private static final int COL_TYPE = 2;
        private static final int COL_DURATION = 3;
        private static final int COL_COST = 4;
        private static final int EXPECTED_COLUMNS = 7;
        
        private long mapStartTime;
        
        @Override
        protected void setup(Context context) throws IOException, InterruptedException {
            mapStartTime = System.currentTimeMillis();
            long setupTime = System.currentTimeMillis() - mapStartTime;
            context.getCounter(PerformanceMetrics.MAPPER_SETUP_TIME_MS).increment(setupTime);
        }
        
        @Override
        protected void map(LongWritable key, Text value, Context context)
                throws IOException, InterruptedException {
            
            String line = value.toString();
            
            // Track every record
            context.getCounter(DataQuality.TOTAL_RECORDS).increment(1);
            
            // Skip headers
            if (line.startsWith("caller_id") || line.startsWith("transaction_id")) {
                context.getCounter(DataQuality.HEADER_LINES_SKIPPED).increment(1);
                return;
            }
            
            try {
                String[] fields = line.split(",");
                
                // Validate field count
                if (fields.length < EXPECTED_COLUMNS) {
                    context.getCounter(DataQuality.MISSING_FIELDS).increment(1);
                    context.getCounter(DataQuality.MALFORMED_LINES).increment(1);
                    return;
                }
                
                // Extract and validate call type
                String callType = fields[COL_TYPE].trim();
                
                if (callType.isEmpty()) {
                    context.getCounter(DataQuality.EMPTY_KEY_FIELD).increment(1);
                    context.getCounter(DataQuality.MALFORMED_LINES).increment(1);
                    return;
                }
                
                // Track business metrics by call type
                switch (callType) {
                    case "voice_call":
                        context.getCounter(BusinessMetrics.VOICE_CALLS).increment(1);
                        break;
                    case "sms":
                        context.getCounter(BusinessMetrics.SMS_MESSAGES).increment(1);
                        break;
                    case "data":
                        context.getCounter(BusinessMetrics.DATA_SESSIONS).increment(1);
                        break;
                    case "roaming_call":
                        context.getCounter(BusinessMetrics.ROAMING_CALLS).increment(1);
                        break;
                    case "intl_call":
                        context.getCounter(BusinessMetrics.INTERNATIONAL_CALLS).increment(1);
                        break;
                    default:
                        context.getCounter(BusinessMetrics.UNKNOWN_CALL_TYPES).increment(1);
                        break;
                }
                
                // Parse and validate duration
                long duration = Long.parseLong(fields[COL_DURATION].trim());
                if (duration < 0) {
                    context.getCounter(DataQuality.NEGATIVE_VALUES).increment(1);
                    return;
                }
                if (duration > 86400) { // > 24 hours
                    context.getCounter(DataQuality.OUT_OF_RANGE_VALUES).increment(1);
                    return;
                }
                
                context.getCounter(BusinessMetrics.TOTAL_DURATION_SECONDS).increment(duration);
                
                // Parse and validate cost
                double cost = Double.parseDouble(fields[COL_COST].trim());
                if (cost < 0) {
                    context.getCounter(DataQuality.NEGATIVE_VALUES).increment(1);
                    return;
                }
                
                // Track cost in cents (avoid floating point counter issues)
                long costCents = Math.round(cost * 100);
                context.getCounter(BusinessMetrics.TOTAL_COST_CENTS).increment(costCents);
                
                // Track high-value transactions
                if (cost > 25.0) {
                    context.getCounter(BusinessMetrics.HIGH_VALUE_TRANSACTIONS).increment(1);
                }
                
                // Emit valid record
                outputKey.set(callType);
                outputValue.set(duration);
                context.write(outputKey, outputValue);
                
                context.getCounter(DataQuality.VALID_RECORDS).increment(1);
                
            } catch (NumberFormatException e) {
                context.getCounter(DataQuality.PARSE_ERRORS).increment(1);
                context.getCounter(DataQuality.MALFORMED_LINES).increment(1);
            } catch (ArrayIndexOutOfBoundsException e) {
                context.getCounter(DataQuality.MISSING_FIELDS).increment(1);
                context.getCounter(DataQuality.MALFORMED_LINES).increment(1);
            } catch (Exception e) {
                context.getCounter(DataQuality.PARSE_ERRORS).increment(1);
            }
        }
        
        @Override
        protected void cleanup(Context context) throws IOException, InterruptedException {
            long cleanupTime = System.currentTimeMillis() - mapStartTime;
            context.getCounter(PerformanceMetrics.MAPPER_CLEANUP_TIME_MS).increment(cleanupTime);
        }
    }
    
    
    // ================================================================
    // REDUCER: Sum durations per call type
    // ================================================================
    
    public static class DiagnosticReducer
            extends Reducer<Text, LongWritable, Text, Text> {
        
        private final Text outputValue = new Text();
        
        @Override
        protected void reduce(Text key, Iterable<LongWritable> values, Context context)
                throws IOException, InterruptedException {
            
            long totalDuration = 0;
            long count = 0;
            long maxDuration = Long.MIN_VALUE;
            long minDuration = Long.MAX_VALUE;
            
            for (LongWritable val : values) {
                long duration = val.get();
                totalDuration += duration;
                count++;
                if (duration > maxDuration) maxDuration = duration;
                if (duration < minDuration) minDuration = duration;
            }
            
            double avgDuration = count > 0 ? (double) totalDuration / count : 0;
            
            String result = String.format(
                "count:%d|total_sec:%d|avg_sec:%.1f|max_sec:%d|min_sec:%d",
                count, totalDuration, avgDuration, maxDuration, minDuration
            );
            
            outputValue.set(result);
            context.write(key, outputValue);
        }
    }

    print(f"AUTOMATED DIAGNOSTIC ANALYSIS")
    print(f"{'='*70}")
    
    issues = []
    warnings = []
    healthy = []
    
    # ── Check 1: Input records ──
    map_in = counters.get("map_input_records", 0)
    if map_in == 0:
        issues.append("CRITICAL: Map input records = 0 → Wrong input path or empty files")
    elif map_in < 100:
        warnings.append(f"WARNING: Only {map_in} input records → Is input path correct?")
    else:
        healthy.append(f"Input records: {map_in:,} ✓")
    
    # ── Check 2: Map output vs input ──
    map_out = counters.get("map_output_records", 0)
    if map_in > 0:
        output_ratio = map_out / map_in * 100
        if output_ratio < 1:
            issues.append(f"CRITICAL: Only {output_ratio:.2f}% of records emitted → Mapper filtering everything")
        elif output_ratio < 50:
            warnings.append(f"WARNING: Only {output_ratio:.1f}% of records emitted → Check filter logic")
        elif output_ratio < 90:
            warnings.append(f"NOTE: {output_ratio:.1f}% of records emitted → Moderate filtering")
        else:
            healthy.append(f"Map output ratio: {output_ratio:.1f}% ✓")
    
    # ── Check 3: Reduce input vs map output ──
    reduce_in = counters.get("reduce_input_records", 0)
    if map_out > 0 and reduce_in > 0:
        if reduce_in < map_out:
            savings = (1 - reduce_in / map_out) * 100
            healthy.append(f"Combiner savings: {savings:.1f}% reduction in shuffle ✓")
        elif reduce_in == map_out:
            warnings.append(f"NOTE: No combiner used — reduce input equals map output")
    
    # ── Check 4: Spilled records ──
    spilled = counters.get("spilled_records", 0)
    if map_out > 0 and spilled > 0:
        spill_ratio = spilled / map_out
        if spill_ratio > 3:
            issues.append(f"PERFORMANCE: Spill ratio = {spill_ratio:.1f}× → Multiple disk spills")
            issues.append(f"  FIX: Increase mapreduce.task.io.sort.mb (default 100MB)")
        elif spill_ratio > 2:
            warnings.append(f"NOTE: Spill ratio = {spill_ratio:.1f}× → Some extra disk I/O")
        else:
            healthy.append(f"Spill ratio: {spill_ratio:.1f}× ✓ (minimal extra I/O)")
    
    # ── Check 5: GC pressure ──
    gc_time = counters.get("gc_time_ms", 0)
    cpu_time = counters.get("cpu_time_ms", 1)  # avoid div by zero
    if cpu_time > 0:
        gc_ratio = gc_time / cpu_time * 100
        if gc_ratio > 20:
            issues.append(f"PERFORMANCE: GC time = {gc_ratio:.1f}% of CPU → Severe memory pressure")
            issues.append(f"  FIX: Reuse Writable objects, reduce HashMap size")
        elif gc_ratio > 10:
            warnings.append(f"NOTE: GC time = {gc_ratio:.1f}% of CPU → Moderate memory pressure")
        else:
            healthy.append(f"GC overhead: {gc_ratio:.1f}% ✓ (healthy)")
    
    # ── Check 6: Data locality ──
    data_local = counters.get("data_local_maps", 0)
    launched_maps = counters.get("launched_maps", 0)
    if launched_maps > 0:
        locality_pct = data_local / launched_maps * 100
        if locality_pct < 50:
            issues.append(f"PERFORMANCE: Only {locality_pct:.0f}% data-local mappers → Heavy network I/O")
            issues.append(f"  FIX: Reduce cluster load or increase HDFS replication")
        elif locality_pct < 80:
            warnings.append(f"NOTE: {locality_pct:.0f}% data-local mappers → Some network overhead")
        else:
            healthy.append(f"Data locality: {locality_pct:.0f}% ✓")
    
    # ── Check 7: Shuffle volume ──
    shuffle_bytes = counters.get("reduce_shuffle_bytes", 0)
    hdfs_read = counters.get("hdfs_bytes_read", 1)
    if hdfs_read > 0 and shuffle_bytes > 0:
        shuffle_ratio = shuffle_bytes / hdfs_read * 100
        if shuffle_ratio > 200:
            issues.append(f"PERFORMANCE: Shuffle = {shuffle_ratio:.0f}% of input → Mapper output too large")
            issues.append(f"  FIX: Add combiner (Project 9), filter more in mapper")
        elif shuffle_ratio > 100:
            warnings.append(f"NOTE: Shuffle = {shuffle_ratio:.0f}% of input → Consider combiner")
        else:
            healthy.append(f"Shuffle efficiency: {shuffle_ratio:.0f}% of input ✓")
    
    # ── Check 8: Data quality ──
    total_records = counters.get("total_records", 0)
    valid_records = counters.get("valid_records", 0)
    malformed = counters.get("malformed_lines", 0)
    parse_errors = counters.get("parse_errors", 0)
    
    if total_records > 0:
        quality_pct = valid_records / total_records * 100
        if quality_pct < 90:
            issues.append(f"DATA QUALITY: Only {quality_pct:.1f}% valid → Investigate data source")
        elif quality_pct < 99:
            warnings.append(f"DATA QUALITY: {quality_pct:.1f}% valid → Minor data issues")
        else:
            healthy.append(f"Data quality: {quality_pct:.2f}% valid ✓")
        
        if malformed > 0:
            malformed_pct = malformed / total_records * 100
            warnings.append(f"Malformed lines: {malformed:,} ({malformed_pct:.2f}%)")
        
        if parse_errors > 0:
            error_pct = parse_errors / total_records * 100
            warnings.append(f"Parse errors: {parse_errors:,} ({error_pct:.2f}%)")
    
    # ── Print report ──
    if issues:
        print(f"\n  🔴 ISSUES ({len(issues)}):")
        for issue in issues:
            print(f"     {issue}")
    
    if warnings:
        print(f"\n  🟡 WARNINGS ({len(warnings)}):")
        for warning in warnings:
            print(f"     {warning}")
    
    if healthy:
        print(f"\n  🟢 HEALTHY ({len(healthy)}):")
        for item in healthy:
            print(f"     {item}")
    
    # ── Overall grade ──
    print(f"\n  {'─'*55}")
    if len(issues) == 0 and len(warnings) <= 2:
        print(f"  OVERALL: ✅ JOB IS HEALTHY")
    elif len(issues) == 0:
        print(f"  OVERALL: ⚠️  JOB SUCCEEDED WITH WARNINGS — Review recommended")
    elif len(issues) <= 2:
        print(f"  OVERALL: 🔶 JOB HAS ISSUES — Action needed")
    else:
        print(f"  OVERALL: 🔴 JOB HAS CRITICAL ISSUES — Investigate immediately")
    
    return issues, warnings, healthy
