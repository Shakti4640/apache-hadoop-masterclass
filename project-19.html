PROJECT 19: GROUPING TRANSACTIONS BY CUSTOMER THEN SORTING BY DATE WITHIN EACH

🔴 HYPOTHETICAL PROBLEM

SCENARIO:
→ You are a data engineer at a retail banking company
→ 120 million transaction records across 3 years
→ Data was cleaned via Map-Only ETL (Project 11) and enriched
   via Reduce-Side Join (Project 17) with customer profiles
→ Now the fraud analytics team needs a SPECIFIC output:

REQUIREMENT:
→ All transactions GROUPED by customer_id
→ Within each customer's transactions, sorted by
   transaction_date DESCENDING (newest first)
→ Fraud models scan each customer's history chronologically
   from most recent backward — ORDER WITHIN GROUP matters

INPUT FORMAT (one line per transaction):
  CUST_ID | TXN_DATE   | TXN_ID   | AMOUNT  | MERCHANT      | CATEGORY
  C00001  | 2024-03-15 | T8830012 | 249.99  | Amazon        | Shopping
  C00001  | 2024-01-02 | T7710034 | 15.50   | Starbucks     | Food
  C00001  | 2024-06-21 | T9920045 | 1200.00 | BestBuy       | Electronics
  C00002  | 2024-04-10 | T8840078 | 89.00   | Netflix       | Entertainment
  C00002  | 2024-02-28 | T7750091 | 340.00  | Delta Airlines| Travel
  ...120 million more rows...

DESIRED OUTPUT (grouped by customer, sorted by date DESC within):
  C00001  | 2024-06-21 | T9920045 | 1200.00 | BestBuy       | Electronics
  C00001  | 2024-03-15 | T8830012 | 249.99  | Amazon        | Shopping
  C00001  | 2024-01-02 | T7710034 | 15.50   | Starbucks     | Food
  C00002  | 2024-04-10 | T8840078 | 89.00   | Netflix       | Entertainment
  C00002  | 2024-02-28 | T7750091 | 340.00  | Delta Airlines| Travel

THE PROBLEM WITH DEFAULT MapReduce:
→ If key = customer_id, reducer gets ALL transactions for that customer
→ BUT values arrive in UNDEFINED order
→ You get: C00001 → [Mar-15, Jun-21, Jan-02] (RANDOM order)
→ You NEED: C00001 → [Jun-21, Mar-15, Jan-02] (DATE DESC order)

NAIVE SOLUTION: Sort values inside reducer using a List
→ Load ALL values into ArrayList, sort, then emit
→ FAILS for customers with millions of transactions (OutOfMemoryError)
→ Customer C00001 might have 500K transactions — cannot buffer all in RAM

CORRECT SOLUTION: SECONDARY SORT
→ Make Hadoop's shuffle/sort phase do the sorting FOR you
→ Values arrive at reducer ALREADY in date-descending order
→ Zero in-memory buffering needed
→ Streaming through values — constant memory usage

🔍 HOW TO APPROACH IT

STEP 1: Understand the four components of Secondary Sort

  DEFAULT MapReduce has:
    → Map output key determines PARTITIONING (which reducer)
    → Map output key determines SORTING (order within reducer)
    → Map output key determines GROUPING (which values go to same reduce call)

  ALL THREE use the SAME key — this is the limitation.

  SECONDARY SORT decouples these three functions:

  ┌─────────────────────────────────────────────────────────────────┐
  │              SECONDARY SORT — FOUR CUSTOM COMPONENTS            │
  ├─────────────────────────────────────────────────────────────────┤
  │                                                                 │
  │  1. COMPOSITE KEY                                               │
  │     → Key = (customer_id, txn_date) instead of just customer_id │
  │     → WritableComparable (Project 13)                           │
  │                                                                 │
  │  2. CUSTOM PARTITIONER                                          │
  │     → Partition by customer_id ONLY (ignore date)               │
  │     → All transactions for same customer → SAME reducer         │
  │     → Replaces HashPartitioner on composite key (Project 10)    │
  │                                                                 │
  │  3. SORT COMPARATOR (Key Comparator)                            │
  │     → Sort by customer_id ASC, THEN by txn_date DESC            │
  │     → Controls the ORDER keys arrive at reducer                 │
  │     → This is what makes values "pre-sorted"                    │
  │                                                                 │
  │  4. GROUPING COMPARATOR                                         │
  │     → Group by customer_id ONLY (ignore date)                   │
  │     → Tells reducer: "these composite keys belong to SAME group"│
  │     → One reduce() call per customer, not per (customer, date)  │
  │                                                                 │
  └─────────────────────────────────────────────────────────────────┘

STEP 2: Visualize how these four interact

  MAP OUTPUT (before shuffle):
    (C00001, 2024-03-15) → "T8830012 | 249.99 | Amazon | Shopping"
    (C00001, 2024-01-02) → "T7710034 | 15.50  | Starbucks | Food"
    (C00001, 2024-06-21) → "T9920045 | 1200.00| BestBuy | Electronics"
    (C00002, 2024-04-10) → "T8840078 | 89.00  | Netflix | Entertainment"
    (C00002, 2024-02-28) → "T7750091 | 340.00 | Delta | Travel"

  AFTER PARTITIONER (customer_id only):
    Reducer 0 gets: ALL C00001 keys AND ALL C00002 keys (same partition)
    (Determined by hash(customer_id) % numReducers)

  AFTER SORT COMPARATOR (customer_id ASC, date DESC):
    (C00001, 2024-06-21)  ← newest first within C00001
    (C00001, 2024-03-15)
    (C00001, 2024-01-02)  ← oldest last within C00001
    (C00002, 2024-04-10)  ← newest first within C00002
    (C00002, 2024-02-28)

  AFTER GROUPING COMPARATOR (customer_id only):
    reduce(C00001, [val_jun21, val_mar15, val_jan02])  ← ONE call
    reduce(C00002, [val_apr10, val_feb28])              ← ONE call

    Values arrive in DATE DESC order because sort comparator
    ordered the composite keys that way, and grouping comparator
    lumps all same-customer keys into one reduce() invocation

💡 HOW CAN WE SOLVE IT

THE FOUR-PIECE PUZZLE:

  ┌──────────────────────────────────────────────────────────────┐
  │                                                              │
  │   MAPPER                                                     │
  │   Input: raw transaction line                                │
  │   Output: CompositeKey(custId, date) → Text(full record)    │
  │                                                              │
  │         │                                                    │
  │         ▼                                                    │
  │   ┌──────────────────────────────────────────┐              │
  │   │ PARTITIONER                              │              │
  │   │ getPartition(compositeKey, value, numR)  │              │
  │   │ → hash(compositeKey.custId) % numR       │              │
  │   │ IGNORES the date field entirely           │              │
  │   │ ENSURES: same customer → same reducer    │              │
  │   └──────────────────┬───────────────────────┘              │
  │                      │                                       │
  │                      ▼                                       │
  │   ┌──────────────────────────────────────────┐              │
  │   │ SORT COMPARATOR                          │              │
  │   │ compare(compositeKey1, compositeKey2)     │              │
  │   │ → First: compare custId ASC              │              │
  │   │ → If same custId: compare date DESC      │              │
  │   │ CONTROLS: physical order on disk/network  │              │
  │   └──────────────────┬───────────────────────┘              │
  │                      │                                       │
  │                      ▼                                       │
  │   ┌──────────────────────────────────────────┐              │
  │   │ GROUPING COMPARATOR                      │              │
  │   │ compare(compositeKey1, compositeKey2)     │              │
  │   │ → Compare custId ONLY                    │              │
  │   │ → If same custId → return 0 (same group) │              │
  │   │ CONTROLS: what constitutes ONE reduce()   │              │
  │   │           call's key-values group         │              │
  │   └──────────────────┬───────────────────────┘              │
  │                      │                                       │
  │                      ▼                                       │
  │   REDUCER                                                    │
  │   reduce(compositeKey, Iterable<values>)                    │
  │   → Values arrive ALREADY sorted by date DESC              │
  │   → Just iterate and emit — no in-memory sorting needed     │
  │                                                              │
  └──────────────────────────────────────────────────────────────┘

✅ WHAT SHOULD DO

1. IMPLEMENT CompositeKey AS WritableComparable (Project 13 pattern)
   → Two fields: customerId (Text) + transactionDate (Text)
   → Implement write(), readFields(), compareTo(), hashCode(), equals()
   → compareTo() defines the NATURAL sort order
   → But SortComparator OVERRIDES this for shuffle-sort phase

2. UNDERSTAND THE DIFFERENCE BETWEEN THREE COMPARISONS

   ┌───────────────────┬──────────────────────┬──────────────────────┐
   │ COMPONENT         │ COMPARES             │ PURPOSE              │
   ├───────────────────┼──────────────────────┼──────────────────────┤
   │ CompositeKey      │ custId ASC,          │ Default/natural      │
   │ .compareTo()      │ date DESC            │ ordering             │
   ├───────────────────┼──────────────────────┼──────────────────────┤
   │ SortComparator    │ custId ASC,          │ Controls shuffle     │
   │                   │ date DESC            │ sort order           │
   │                   │                      │ (overrides natural)  │
   ├───────────────────┼──────────────────────┼──────────────────────┤
   │ GroupingComparator│ custId ONLY          │ Controls which keys  │
   │                   │ (ignores date)       │ form ONE reduce call │
   └───────────────────┴──────────────────────┴──────────────────────┘

   → SortComparator and compareTo() often do the SAME thing
   → But GroupingComparator MUST be different — only compares custId

3. USE RawComparator FOR PERFORMANCE
   → Default compareTo() deserializes bytes → creates objects → compares
   → RawComparator compares SERIALIZED BYTES directly
   → Avoids object creation during shuffle (millions of comparisons)
   → WritableComparator.define() provides this optimization

4. TEST WITH SMALL DATA FIRST
   → Secondary sort has 4 interacting components
   → A bug in ANY ONE breaks the entire output
   → Test with 20 records across 3 customers before scaling

🚫 WHAT SHOULD NOT DO

1. DO NOT forget the GroupingComparator
   → Without it, each UNIQUE composite key gets its OWN reduce() call
   → (C00001, 2024-06-21) → reduce() call 1
   → (C00001, 2024-03-15) → reduce() call 2 (SEPARATE!)
   → (C00001, 2024-01-02) → reduce() call 3 (SEPARATE!)
   → You wanted ONE call with ALL three values for C00001
   → GroupingComparator tells Hadoop: "same custId = same group"

2. DO NOT partition by the COMPOSITE key
   → HashPartitioner on (custId, date) might send:
     (C00001, 2024-06-21) → Reducer 0
     (C00001, 2024-03-15) → Reducer 3 (DIFFERENT reducer!)
   → Customer's transactions SPLIT across reducers
   → Custom Partitioner MUST hash on custId ONLY

3. DO NOT confuse sort order with grouping order
   → SortComparator: custId ASC, date DESC
   → GroupingComparator: custId ONLY
   → If GroupingComparator also compared dates:
     Each (custId, date) pair becomes its own group
     Defeats the entire purpose

4. DO NOT buffer values in reducer to sort them
   → The WHOLE POINT of secondary sort is to avoid this
   → If you find yourself writing:
       List<Text> allValues = new ArrayList<>();
       for (Text val : values) { allValues.add(val); }
       Collections.sort(allValues);
   → You are doing it WRONG — secondary sort makes values
     arrive pre-sorted, just iterate and emit

5. DO NOT use Secondary Sort when TotalOrderPartitioner suffices
   → Project 18 handles single-key global sort
   → Secondary Sort is for TWO-LEVEL ordering:
     group by X, sort within group by Y
   → Using Secondary Sort for simple global sort is overengineering

6. DO NOT ignore that the KEY passed to reduce() is the FIRST key in group
   → When GroupingComparator groups (C00001, 2024-06-21) and
     (C00001, 2024-03-15) together
   → The key argument in reduce() is the FIRST composite key seen
   → Which is (C00001, 2024-06-21) because SortComparator put it first
   → If you need custId only, extract it from compositeKey.getCustomerId()

🚫 WHEN NOT TO USE

1. SINGLE SORT DIMENSION
   → Just sorting by salary? Use TotalOrderPartitioner (Project 18)
   → Secondary Sort adds complexity with zero benefit

2. SMALL VALUE SETS PER KEY
   → If each customer has at most 10 transactions
   → Sorting 10 items in reducer memory is trivial
   → Secondary Sort's 4-component complexity is overkill
   → Simple: collect in ArrayList, sort, emit

3. NO ORDERING NEEDED WITHIN GROUPS
   → Standard MapReduce grouping is sufficient
   → Adding SortComparator/GroupingComparator wastes development time

4. WHEN USING HIGHER-LEVEL TOOLS
   → Hive: DISTRIBUTE BY custId SORT BY txnDate DESC
   → Pig: GROUP BY custId; FOREACH ORDER BY txnDate DESC
   → Spark: groupByKey + sortBy within partitions
   → These tools implement Secondary Sort internally
   → Writing raw MapReduce Secondary Sort is rarely needed in 2024
   → BUT understanding it deeply helps debug those tools

5. THREE OR MORE SORT LEVELS
   → Secondary Sort handles TWO levels: group by X, sort by Y
   → Need: group by X, sub-group by Y, sort by Z?

⚙️ HOW IT ONLY WORKS — INTERNAL MECHANICS

═══════════════════════════════════════════════════════════════
INTERNAL MECHANIC 1: THE SHUFFLE-SORT PIPELINE IN DETAIL
═══════════════════════════════════════════════════════════════

Without Secondary Sort (DEFAULT):

  Mapper output buffer (in memory):
    → Sorted by KEY using key.compareTo()
    → Spilled to disk when buffer fills
    → Partitioned by HashPartitioner(key)
  
  Shuffle (network transfer):
    → Each reducer pulls its partition from all mappers
  
  Reduce-side merge-sort:
    → Merges all pulled data, sorted by key.compareTo()
    → Groups consecutive equal keys into one reduce() call
    → "Equal" defined by GroupingComparator (default: key.compareTo() == 0)

With Secondary Sort:

  Mapper output buffer:
    → Sorted by COMPOSITE KEY using ★SortComparator★
    → NOT key.compareTo() — SortComparator OVERRIDES it
    → Spilled to disk when buffer fills
    → Partitioned by ★CustomPartitioner★ (custId only)

  Shuffle:
    → Each reducer pulls its partition (same customer → same reducer)
  
  Reduce-side merge-sort:
    → Merges all pulled data using ★SortComparator★
    → (custId ASC, date DESC)
    → Groups consecutive keys using ★GroupingComparator★
    → GroupingComparator only compares custId
    → So all (C00001, any_date) keys grouped together
    → Within that group, keys arrive in date DESC order
    → Because SortComparator already arranged them that way

═══════════════════════════════════════════════════════════════
INTERNAL MECHANIC 2: WHY GROUPING COMPARATOR IS THE SECRET
═══════════════════════════════════════════════════════════════

  After sort, the reduce-side merge produces this stream:

  Position 1: KEY=(C00001, 2024-06-21) VALUE="T99..."
  Position 2: KEY=(C00001, 2024-03-15) VALUE="T88..."
  Position 3: KEY=(C00001, 2024-01-02) VALUE="T77..."
  Position 4: KEY=(C00002, 2024-04-10) VALUE="T88..."
  Position 5: KEY=(C00002, 2024-02-28) VALUE="T77..."

  Hadoop's reduce framework scans this stream:

  ┌─────────────────────────────────────────────────────────┐
  │ START: current_group_key = Position 1 key               │
  │                                                         │
  │ Position 1: GroupingComparator(pos1.key, current) == 0  │
  │   → Same group → add value to iterator                 │
  │                                                         │
  │ Position 2: GroupingComparator(pos2.key, current) == 0  │
  │   → (C00001,2024-03-15) vs (C00001,2024-06-21)        │
  │   → Compare custId only: C00001 == C00001 → SAME GROUP │
  │   → Add value to iterator                              │
  │                                                         │
  │ Position 3: GroupingComparator(pos3.key, current) == 0  │
  │   → C00001 == C00001 → SAME GROUP                     │
  │   → Add value to iterator                              │
  │                                                         │
  │ Position 4: GroupingComparator(pos4.key, current) != 0  │
  │   → C00002 != C00001 → NEW GROUP                      │
  │   → CALL reduce(key=(C00001,2024-06-21),               │
  │                  values=[val_jun, val_mar, val_jan])    │
  │   → Start new group with Position 4 as current         │
  │                                                         │
  │ Position 5: GroupingComparator(pos5.key, current) == 0  │
  │   → C00002 == C00002 → SAME GROUP                     │
  │                                                         │
  │ END OF STREAM:                                          │
  │   → CALL reduce(key=(C00002,2024-04-10),               │
  │                  values=[val_apr, val_feb])             │
  └─────────────────────────────────────────────────────────┘

  KEY INSIGHT:
  → The key passed to reduce() is the FIRST composite key in group
  → For C00001: key = (C00001, 2024-06-21) — the NEWEST date
  → Because SortComparator put date DESC, newest comes first
  → Values iterate in the SAME order as SortComparator defined

═══════════════════════════════════════════════════════════════
INTERNAL MECHANIC 3: SERIALIZATION DURING SORT
═══════════════════════════════════════════════════════════════

  During shuffle-sort, Hadoop compares keys MILLIONS of times.
  
  DEFAULT path (SLOW):
    raw bytes → deserialize to CompositeKey object
    → call compareTo() → return int
    → Repeat for every comparison
    → Object creation overhead × millions = SLOW

  OPTIMIZED path with RawComparator (FAST):
    raw bytes → compare bytes directly without deserialization
    → No object creation
    → Byte-level comparison

  HOW CompositeKey is serialized:
  
  ┌──────────────────────────────────────────────────┐
  │ BYTE LAYOUT of CompositeKey("C00001", "2024-06") │
  │                                                    │
  │ [4 bytes: custId length = 6]                      │
  │ [6 bytes: "C00001"]                               │
  │ [4 bytes: date length = 10]                       │
  │ [10 bytes: "2024-06-21"]                          │
  │                                                    │
  │ Total: 24 bytes per composite key                 │
  └──────────────────────────────────────────────────┘

  RawComparator for SORT:
    → Read custId length from bytes 0-3
    → Compare custId bytes directly (memcmp equivalent)
    → If equal: read date length, compare date bytes
    → Reverse comparison for DESC order on date

  RawComparator for GROUPING:
    → Read custId length from bytes 0-3
    → Compare custId bytes ONLY
    → IGNORE everything after custId
    → Return 0 if custId bytes match

═══════════════════════════════════════════════════════════════
INTERNAL MECHANIC 4: MEMORY MODEL — WHY THIS IS EFFICIENT
═══════════════════════════════════════════════════════════════

  NAIVE APPROACH (sort in reducer):
    Customer with 500K transactions:
    → Load 500K Text objects into ArrayList
    → Each Text ~100 bytes → 50MB in heap
    → Sort: O(500K × log(500K)) = ~9.5M comparisons
    → Memory: 50MB per customer group
    → 1000 customers processed sequentially → peaks at 50MB
    → BUT: If one customer has 10M transactions → 1GB → OOM

  SECONDARY SORT APPROACH:
    → Sorting happens in MAP-SIDE SPILL and REDUCE-SIDE MERGE
    → These use DISK-BACKED merge-sort (external sort)
    → Memory usage: only the merge buffer size
      (configured by io.sort.mb, default 100MB — see Project 46)
    → Can sort UNLIMITED data — limited only by disk space
    → Reducer iterates through values ONE AT A TIME
    → Memory: O(1) per record — constant regardless of group size

  ┌─────────────────────────────────────────────────┐
  │           MEMORY COMPARISON                      │
  │                                                  │
  │  Naive sort in reducer:                         │
  │    500K txns × 100B = 50MB per customer         │
  │    10M txns × 100B = 1GB per customer → OOM     │
  │                                                  │
  │  Secondary Sort:                                │
  │    ANY number of txns → ~0 extra memory         │
  │    Sorting done during shuffle (disk-backed)    │
  │    Reducer streams through pre-sorted values    │
  └─────────────────────────────────────────────────┘

⚠️ COMMON PITFALLS AND FAILURE MODES

PITFALL 1: "All records for one customer go to different reduce() calls"
→ CAUSE: Forgot to set GroupingComparator
→ Without it: each unique (custId, date) pair = separate reduce() call
→ FIX: job.setGroupingComparatorClass(CustomerGroupingComparator.class)

PITFALL 2: "Records are grouped correctly but NOT sorted within group"
→ CAUSE: SortComparator not set or not comparing date field
→ Default sort uses CompositeKey.compareTo()
→ If compareTo() only compares custId → no date ordering
→ FIX: Ensure SortComparator compares custId THEN date
→ OR: Ensure compareTo() does both and don't set separate SortComparator

PITFALL 3: "Customer's transactions split across multiple reducers"
→ CAUSE: Using default HashPartitioner on composite key
→ hash(C00001, 2024-06-21) ≠ hash(C00001, 2024-03-15)
→ Different composite keys → potentially different partitions
→ FIX: Custom Partitioner that hashes ONLY custId

PITFALL 4: "ClassCastException in SortComparator or GroupingComparator"
→ CAUSE: Comparator class extends WritableComparator but forgot to call
   super(CompositeKey.class, true) in constructor
→ The 'true' parameter enables raw byte comparison
→ Without it: Hadoop cannot instantiate the key class for comparison

PITFALL 5: "Date sorting is wrong — 2024-02 comes after 2024-12"
→ CAUSE: Comparing dates as strings with wrong format
→ "2024-02-28" < "2024-12-01" lexicographically (correct for ASC)
→ BUT: If date format is "2/28/2024" vs "12/1/2024"
   → "2/28" > "12/1" because "2" > "1" in ASCII
→ FIX: ALWAYS use ISO format YYYY-MM-DD for string date comparison
→ OR: Convert to epoch milliseconds (LongWritable) for numeric comparison

PITFALL 6: "Reducer key shows wrong date for the group"
→ CAUSE: The key in reduce(key, values) is the FIRST composite key
→ With date DESC sort: first key has the NEWEST date
→ If you expected the OLDEST date as the group key → wrong assumption
→ FIX: Extract only custId from the composite key
→ DO NOT rely on the date component of the reduce key

PITFALL 7: "Values iterator can only be traversed ONCE"
→ CAUSE: Hadoop's reduce values Iterable is backed by disk streaming
→ You CANNOT iterate twice:
     for (Text v : values) { ... }   // first pass — works
     for (Text v : values) { ... }   // second pass — EMPTY
→ FIX: If you need two passes, collect into a list (but watch memory)
→ Better: Design logic to work in single pass

📈 SCALABILITY CONSIDERATIONS

SCALING DIMENSION 1: NUMBER OF UNIQUE CUSTOMERS (GROUP KEYS)

  ┌──────────────────┬─────────────┬──────────────────────────┐
  │ Unique Customers │ Reducers    │ Notes                    │
  ├──────────────────┼─────────────┼──────────────────────────┤
  │ 1,000            │ 2-5         │ Partitioner distributes  │
  │                  │             │ evenly enough             │
  ├──────────────────┼─────────────┼──────────────────────────┤
  │ 1,000,000        │ 10-50       │ Good distribution likely │
  ├──────────────────┼─────────────┼──────────────────────────┤
  │ 100,000,000      │ 50-200      │ Watch for hot customers  │
  │                  │             │ with millions of txns    │
  └──────────────────┴─────────────┴──────────────────────────┘

SCALING DIMENSION 2: TRANSACTIONS PER CUSTOMER (GROUP SIZE)

  → Secondary Sort handles ANY group size efficiently
  → 100 txns per customer → trivial
  → 10M txns per customer → still works (disk-backed merge-sort)
  → The ONLY concern: reducer processing time per group
  → If reducer does O(N) work per value → linear, fine
  → If reducer does O(N²) work → quadratic explosion for big groups

SCALING DIMENSION 3: COMPOSITE KEY OVERHEAD

  → Each map output record now has a LARGER key
  → Default: key = "C00001" (6 bytes)
  → Composite: key = "C00001" + "2024-06-21" (16+ bytes)
  → ~2.5x key size increase
  → For 120M records: ~1.2GB extra key data through shuffle
  → Manageable but noticeable — use compression (Project 14)

SCALING DIMENSION 4: COMPARISON COST

  → SortComparator runs during:
    1. Map-side spill sort (per mapper)
    2. Map-side merge (combining spill files)
    3. Reduce-side merge (combining mapper outputs)
  → Total comparisons ≈ N × log(N) × constant
  → 120M records → ~120M × 27 ≈ 3.2 billion comparisons
  → RawComparator: ~3ns per comparison → ~10 seconds total
  → Object-based compareTo(): ~30ns per → ~100 seconds total
  → RawComparator is 10x faster — USE IT for large datasets

import org.apache.hadoop.io.WritableComparable;
import org.apache.hadoop.io.WritableComparator;

import java.io.DataInput;
import java.io.DataOutput;
import java.io.IOException;

public class CustomerDateKey implements WritableComparable<CustomerDateKey> {

    private String customerId;
    private String transactionDate;  // YYYY-MM-DD format for correct string comparison

    // Default constructor REQUIRED by Hadoop serialization framework
    public CustomerDateKey() {
        this.customerId = "";
        this.transactionDate = "";
    }

    public CustomerDateKey(String customerId, String transactionDate) {
        this.customerId = customerId;
        this.transactionDate = transactionDate;
    }

    // ── Getters ──
    public String getCustomerId() {
        return customerId;
    }

    public String getTransactionDate() {
        return transactionDate;
    }

    // ── Setters (reuse objects to reduce GC pressure) ──
    public void set(String customerId, String transactionDate) {
        this.customerId = customerId;
        this.transactionDate = transactionDate;
    }

    // ══════════════════════════════════════════════════════
    // SERIALIZATION: How this key is written to bytes
    // (Project 13 covered this pattern)
    // ══════════════════════════════════════════════════════

    @Override
    public void write(DataOutput out) throws IOException {
        out.writeUTF(customerId);
        out.writeUTF(transactionDate);
    }

    @Override
    public void readFields(DataInput in) throws IOException {
        customerId = in.readUTF();
        transactionDate = in.readUTF();
    }

    // ══════════════════════════════════════════════════════
    // NATURAL ORDERING: customerId ASC, transactionDate DESC
    // This is the DEFAULT comparison if no SortComparator set
    // ══════════════════════════════════════════════════════

    @Override
    public int compareTo(CustomerDateKey other) {
        // First: compare customerId ascending
        int cmp = this.customerId.compareTo(other.customerId);
        if (cmp != 0) {
            return cmp;
        }

        // Second: compare transactionDate DESCENDING (reverse)
        // Reverse: other.date compared to this.date
        // YYYY-MM-DD format ensures lexicographic = chronological
        return other.transactionDate.compareTo(this.transactionDate);
    }

    // ══════════════════════════════════════════════════════
    // hashCode and equals: MUST be consistent with compareTo
    // hashCode based on BOTH fields for correct behavior
    // in HashMap/HashSet if ever used outside MapReduce
    // ══════════════════════════════════════════════════════

    @Override
    public int hashCode() {
        // Use ONLY customerId for hashCode
        // This aligns with Partitioner which hashes custId only
        // Avoids subtle bugs if hashCode used in partitioning logic
        return customerId.hashCode();
    }

    @Override
    public boolean equals(Object obj) {
        if (this == obj) return true;
        if (obj == null || getClass() != obj.getClass()) return false;
        CustomerDateKey other = (CustomerDateKey) obj;
        return customerId.equals(other.customerId)
            && transactionDate.equals(other.transactionDate);
    }

    @Override
    public String toString() {
        return customerId + "\t" + transactionDate;
    }

    // ══════════════════════════════════════════════════════
    // REGISTER RAW COMPARATOR for performance
    // Allows Hadoop to compare serialized bytes directly
    // without deserializing to CustomerDateKey objects
    // ══════════════════════════════════════════════════════

    static {
        WritableComparator.define(
            CustomerDateKey.class,
            new CustomerDateComparator()
        );
    }
}

// FILE 2: CustomerDateComparator.java (SORT COMPARATOR)
// PURPOSE: Controls the ORDER of keys during shuffle-sort
//          Sort by customerId ASC, then transactionDate DESC
// ══════════════════════════════════════════════════════════════════

import org.apache.hadoop.io.WritableComparable;
import org.apache.hadoop.io.WritableComparator;

public class CustomerDateComparator extends WritableComparator {

    // ══════════════════════════════════════════════════════
    // Constructor: MUST call super with key class and true
    // The 'true' parameter creates a default instance of the key
    // which is needed for deserialization during comparison
    // ══════════════════════════════════════════════════════

    protected CustomerDateComparator() {
        super(CustomerDateKey.class, true);
    }

    // ══════════════════════════════════════════════════════
    // COMPARISON LOGIC:
    // 1. Compare customerId ascending (alphabetical)
    // 2. If same customer, compare date DESCENDING (newest first)
    //
    // This method is called MILLIONS of times during sort
    // Keep it as fast as possible
    // ══════════════════════════════════════════════════════

    @Override
    @SuppressWarnings("rawtypes")
    public int compare(WritableComparable a, WritableComparable b) {
        CustomerDateKey key1 = (CustomerDateKey) a;
        CustomerDateKey key2 = (CustomerDateKey) b;

        // Primary sort: customerId ASCENDING
        int cmp = key1.getCustomerId().compareTo(key2.getCustomerId());
        if (cmp != 0) {
            return cmp;
        }

        // Secondary sort: transactionDate DESCENDING
        // DESCENDING = reverse the comparison order
        // key2.date.compareTo(key1.date) instead of key1.date.compareTo(key2.date)
        return key2.getTransactionDate().compareTo(key1.getTransactionDate());
    }
}

// FILE 3: CustomerGroupingComparator.java (GROUPING COMPARATOR)
// PURPOSE: Controls which composite keys form ONE reduce() call
//          Group by customerId ONLY — ignore transactionDate
//
// THIS IS THE MOST CRITICAL PIECE OF SECONDARY SORT
// Without this, each unique (custId, date) pair gets its own
// reduce() call — defeating the purpose entirely
// ══════════════════════════════════════════════════════════════════

import org.apache.hadoop.io.WritableComparable;
import org.apache.hadoop.io.WritableComparator;

public class CustomerGroupingComparator extends WritableComparator {

    protected CustomerGroupingComparator() {
        super(CustomerDateKey.class, true);
    }

    // ══════════════════════════════════════════════════════
    // GROUPING LOGIC:
    // Compare ONLY customerId
    // Return 0 when customerIds match → "same group"
    // Return non-zero → "new group, trigger new reduce() call"
    //
    // The reduce framework scans sorted keys sequentially:
    //   (C00001, 2024-06-21) ← start group
    //   (C00001, 2024-03-15) ← grouping says "same" (custId match)
    //   (C00001, 2024-01-02) ← grouping says "same"
    //   (C00002, 2024-04-10) ← grouping says "different" → new group
    // ══════════════════════════════════════════════════════

    @Override
    @SuppressWarnings("rawtypes")
    public int compare(WritableComparable a, WritableComparable b) {
        CustomerDateKey key1 = (CustomerDateKey) a;
        CustomerDateKey key2 = (CustomerDateKey) b;

        // Compare ONLY customerId — ignore transactionDate entirely
        return key1.getCustomerId().compareTo(key2.getCustomerId());
    }
}

// FILE 4: CustomerIdPartitioner.java (CUSTOM PARTITIONER)
// PURPOSE: Ensure all transactions for same customer go to same reducer
//          Partition by customerId ONLY — ignore transactionDate
// BUILDS ON: Project 10 (Custom Partitioner concept)
// ══════════════════════════════════════════════════════════════════

import org.apache.hadoop.io.Text;
import org.apache.hadoop.mapreduce.Partitioner;

public class CustomerIdPartitioner extends Partitioner<CustomerDateKey, Text> {

    // ══════════════════════════════════════════════════════
    // PARTITIONING LOGIC:
    // Hash ONLY the customerId field of the composite key
    // This ensures:
    //   (C00001, 2024-06-21) → same partition
    //   (C00001, 2024-03-15) → same partition  ← SAME reducer
    //   (C00001, 2024-01-02) → same partition
    //
    // Without this, HashPartitioner hashes the ENTIRE composite key:
    //   hash(C00001, 2024-06-21) might → Reducer 2
    //   hash(C00001, 2024-03-15) might → Reducer 7  ← DIFFERENT!
    // ══════════════════════════════════════════════════════

    @Override
    public int getPartition(CustomerDateKey key, Text value,
                            int numPartitions) {

        // hashCode of customerId, made positive with bitwise AND
        // Math.abs() has edge case: Math.abs(Integer.MIN_VALUE) = negative
        // Bitwise AND with MAX_VALUE always produces non-negative
        return (key.getCustomerId().hashCode() & Integer.MAX_VALUE)
               % numPartitions;
    }
}

// ══════════════════════════════════════════════════════════════════

import org.apache.hadoop.conf.Configuration;
import org.apache.hadoop.conf.Configured;
import org.apache.hadoop.fs.Path;
import org.apache.hadoop.io.LongWritable;
import org.apache.hadoop.io.NullWritable;
import org.apache.hadoop.io.Text;
import org.apache.hadoop.mapreduce.Job;
import org.apache.hadoop.mapreduce.Mapper;
import org.apache.hadoop.mapreduce.Reducer;
import org.apache.hadoop.mapreduce.lib.input.FileInputFormat;
import org.apache.hadoop.mapreduce.lib.input.TextInputFormat;
import org.apache.hadoop.mapreduce.lib.output.FileOutputFormat;
import org.apache.hadoop.mapreduce.lib.output.TextOutputFormat;
import org.apache.hadoop.util.Tool;
import org.apache.hadoop.util.ToolRunner;

import java.io.IOException;

public class TransactionSecondarySort extends Configured implements Tool {

    // ══════════════════════════════════════════════════════
    // MAPPER: Parse transaction records and emit
    //         CompositeKey(custId, date) → full record
    // ══════════════════════════════════════════════════════

    public static class TransactionMapper
            extends Mapper<LongWritable, Text, CustomerDateKey, Text> {

        // Reuse key object to reduce garbage collection
        // (same pattern as Project 5 — Writable object reuse)
        private CustomerDateKey compositeKey = new CustomerDateKey();

        @Override
        protected void map(LongWritable offset, Text line, Context context)
                throws IOException, InterruptedException {

            String record = line.toString().trim();

            // Skip empty lines and header
            if (record.isEmpty() || record.startsWith("CUST_ID")) {
                return;
            }

            // Parse: CUST_ID | TXN_DATE | TXN_ID | AMOUNT | MERCHANT | CATEGORY
            String[] fields = record.split("\\|");

            if (fields.length < 6) {
                context.getCounter("QUALITY", "MALFORMED_RECORDS").increment(1);
                return;
            }

            String customerId = fields[0].trim();
            String transactionDate = fields[1].trim();

            // Validate date format (YYYY-MM-DD) for correct lexicographic sorting
            if (!transactionDate.matches("\\d{4}-\\d{2}-\\d{2}")) {
                context.getCounter("QUALITY", "BAD_DATE_FORMAT").increment(1);
                return;
            }

            // Set composite key: (customerId, transactionDate)
            compositeKey.set(customerId, transactionDate);

            // Emit: KEY = composite key, VALUE = full original record
            context.write(compositeKey, line);
            context.getCounter("QUALITY", "VALID_RECORDS").increment(1);
        }
    }

    // ══════════════════════════════════════════════════════
    // REDUCER: Iterate through pre-sorted values and emit
    //
    // BY THE TIME VALUES ARRIVE HERE:
    // → All transactions for ONE customer are together (GroupingComparator)
    // → Within this customer, transactions are sorted by date DESC
    //   (SortComparator arranged this during shuffle)
    // → We just iterate and emit — ZERO sorting logic needed
    // ══════════════════════════════════════════════════════

    public static class SortedTransactionReducer
            extends Reducer<CustomerDateKey, Text, NullWritable, Text> {

        @Override
        protected void reduce(CustomerDateKey key, Iterable<Text> transactions,
                              Context context)
                throws IOException, InterruptedException {

            // The 'key' here is the FIRST composite key in this group
            // Which means key.getTransactionDate() = the NEWEST date
            // (because SortComparator put date DESC first)

            String currentCustomer = key.getCustomerId();
            int transactionCount = 0;

            // Just iterate — values arrive pre-sorted by date DESC
            for (Text transaction : transactions) {
                context.write(NullWritable.get(), transaction);
                transactionCount++;
            }

            // Track group sizes for monitoring
            context.getCounter("GROUPS", "TOTAL_CUSTOMER_GROUPS").increment(1);

            // Detect potentially skewed groups (preview of Project 33)
            if (transactionCount > 100000) {
                context.getCounter("GROUPS", "LARGE_GROUPS_OVER_100K").increment(1);
            }
        }
    }

    // ══════════════════════════════════════════════════════
    // DRIVER: Wire all four components together
    // ══════════════════════════════════════════════════════

    @Override
    public int run(String[] args) throws Exception {

        if (args.length != 3) {
            System.err.println(
                "Usage: TransactionSecondarySort <input> <output> <numReducers>");
            System.err.println(
                "Example: TransactionSecondarySort /data/transactions /output/sorted 4");
            return -1;
        }

        Configuration conf = getConf();
        Job job = Job.getInstance(conf, "Secondary Sort - Customer Transactions by Date");
        job.setJarByClass(TransactionSecondarySort.class);

        // ── Mapper ──
        job.setMapperClass(TransactionMapper.class);

        // ── Map output types ──
        // The composite key is what flows through shuffle
        job.setMapOutputKeyClass(CustomerDateKey.class);
        job.setMapOutputValueClass(Text.class);

        // ════════════════════════════════════════════════════
        // THE FOUR SECONDARY SORT CONFIGURATIONS
        // These four lines are the ENTIRE secret
        // Remove ANY ONE and the output will be wrong
        // ════════════════════════════════════════════════════

        // 1. PARTITIONER: Route by customerId only
        //    Ensures same customer → same reducer
        job.setPartitionerClass(CustomerIdPartitioner.class);

        // 2. SORT COMPARATOR: Sort by custId ASC, date DESC
        //    Controls physical ordering during shuffle-sort
        job.setSortComparatorClass(CustomerDateComparator.class);

        // 3. GROUPING COMPARATOR: Group by custId only
        //    Tells reducer framework: same custId = one reduce() call
        job.setGroupingComparatorClass(CustomerGroupingComparator.class);

        // 4. REDUCER: Process pre-sorted groups
        job.setReducerClass(SortedTransactionReducer.class);

        // ── Final output types ──
        job.setOutputKeyClass(NullWritable.class);
        job.setOutputValueClass(Text.class);

        // ── Input/Output ──
        job.setInputFormatClass(TextInputFormat.class);
        job.setOutputFormatClass(TextOutputFormat.class);

        FileInputFormat.addInputPath(job, new Path(args[0]));
        FileOutputFormat.setOutputPath(job, new Path(args[1]));

        // ── Number of reducers ──
        job.setNumReduceTasks(Integer.parseInt(args[2]));

        // ── Submit and wait ──
        System.out.println("=== SECONDARY SORT: Customer Transactions ===");
        System.out.println("Partitioner:          CustomerIdPartitioner (custId only)");
        System.out.println("SortComparator:       CustomerDateComparator (custId ASC, date DESC)");
        System.out.println("GroupingComparator:   CustomerGroupingComparator (custId only)");
        System.out.println("Reducers:             " + args[2]);

        boolean success = job.waitForCompletion(true);

        if (success) {
            long totalGroups = job.getCounters()
                .findCounter("GROUPS", "TOTAL_CUSTOMER_GROUPS").getValue();
            long largeGroups = job.getCounters()
                .findCounter("GROUPS", "LARGE_GROUPS_OVER_100K").getValue();
            long validRecords = job.getCounters()
                .findCounter("QUALITY", "VALID_RECORDS").getValue();

            System.out.println("\n=== RESULTS ===");
            System.out.println("Total records processed: " + validRecords);
            System.out.println("Total customer groups:   " + totalGroups);
            System.out.println("Large groups (>100K):    " + largeGroups);
        }

        return success ? 0 : 1;
    }

    public static void main(String[] args) throws Exception {
        int exitCode = ToolRunner.run(
            new Configuration(), new TransactionSecondarySort(), args);
        System.exit(exitCode);
    }
}

import org.apache.hadoop.io.WritableComparator;
import org.apache.hadoop.io.WritableUtils;

public class CustomerDateRawComparator extends WritableComparator {

    // ══════════════════════════════════════════════════════
    // RAW BYTE COMPARISON
    //
    // CustomerDateKey serialization layout (writeUTF format):
    //   [2 bytes: custId UTF length]
    //   [N bytes: custId UTF data]
    //   [2 bytes: date UTF length]
    //   [M bytes: date UTF data]
    //
    // writeUTF writes 2-byte length prefix followed by modified UTF-8 data
    // We compare byte arrays directly using WritableComparator.compareBytes()
    // ══════════════════════════════════════════════════════

    protected CustomerDateRawComparator() {
        super(CustomerDateKey.class);
    }

    @Override
    public int compare(byte[] b1, int s1, int l1,
                       byte[] b2, int s2, int l2) {

        // ── Parse first field (customerId) from both byte arrays ──

        // writeUTF stores 2-byte length prefix
        int custId1Len = readUnsignedShort(b1, s1);
        int custId2Len = readUnsignedShort(b2, s2);

        // Compare customerId bytes (ascending)
        int cmp = compareBytes(
            b1, s1 + 2, custId1Len,    // skip 2-byte length prefix
            b2, s2 + 2, custId2Len
        );

        if (cmp != 0) {
            return cmp;  // Different customers — ascending order
        }

        // ── Same customer — parse second field (transactionDate) ──

        // Date starts after: 2 bytes (custId length) + custId data
        int date1Start = s1 + 2 + custId1Len;
        int date2Start = s2 + 2 + custId2Len;

        int date1Len = readUnsignedShort(b1, date1Start);
        int date2Len = readUnsignedShort(b2, date2Start);

        // Compare date bytes DESCENDING (reverse order)
        // Swap b1/b2 positions for descending
        return compareBytes(
            b2, date2Start + 2, date2Len,   // NOTE: b2 first = DESCENDING
            b1, date1Start + 2, date1Len
        );
    }

    private int readUnsignedShort(byte[] bytes, int offset) {
        return ((bytes[offset] & 0xFF) << 8) | (bytes[offset + 1] & 0xFF);
    }
}
