PROJECT 44: NAMENODE HIGH AVAILABILITY — SURVIVING A MASTER NODE CRASH

🔴 HYPOTHETICAL PROBLEM

SCENARIO:
→ You are the on-call platform engineer at a financial trading analytics firm
→ Your Hadoop cluster processes $2.3 billion worth of trade data daily
→ You deployed HDFS Federation (Project 43) with 3 NameNodes:
    - ns1: /trades, /positions       (critical — trading analytics)
    - ns2: /risk-models              (critical — risk calculations)
    - ns3: /compliance, /audit-logs  (regulatory — legally required)

TUESDAY 2:47 AM — YOUR PHONE RINGS:
→ NameNode for ns1 (namenode1-host) crashed
→ Hardware failure: motherboard burned out
→ ns1 manages /trades and /positions
→ IMPACT:
    - ALL trade analytics pipelines are DEAD
    - Morning risk reports cannot be generated
    - Trading desk opens in 4 hours — they NEED yesterday's P&L
    - Compliance team cannot access audit logs... wait, those are on ns3 — SAFE
      (Federation isolation from Project 43 saved compliance)

WHAT HAPPENS NEXT (WITHOUT HA):
→ 2:47 AM: NameNode process dies
→ 2:50 AM: Monitoring alerts fire
→ 3:15 AM: You arrive at laptop, assess damage
→ 3:30 AM: Hardware team confirms motherboard is dead
→ 4:00 AM: You find a spare server, begin provisioning
→ 5:30 AM: New server ready, Hadoop installed
→ 5:45 AM: Copy FSImage backup from secondary storage
→ 6:00 AM: Replay EditLogs from backup
→ 6:35 AM: NameNode starts, enters safe mode
→ 6:55 AM: All DataNodes report blocks, safe mode exits
→ 7:00 AM: CLUSTER BACK — 4 HOURS AND 13 MINUTES OF DOWNTIME

BUSINESS IMPACT:
→ 4+ hours of complete inaccessibility for /trades and /positions
→ Morning risk report MISSED — regulatory violation
→ Trading desk operated without analytics for first 2 hours
→ Estimated business impact: $1.8 million in operational risk

THE FUNDAMENTAL QUESTION:
→ Can we have a SECOND NameNode that is ALWAYS READY to take over
   within SECONDS, not HOURS?
→ YES — That is NameNode High Availability (HA).

🔍 HOW TO APPROACH IT

STEP 1: Understand what makes NameNode recovery SLOW without HA

Recall from Project 2:
→ NameNode stores metadata in two persistent structures:
  1. FSImage: Complete snapshot of filesystem metadata (written periodically)
  2. EditLog: Sequential log of every metadata change since last FSImage

STEP 2: Identify what HA must solve

For INSTANT failover, the Standby NameNode must have:
  ✅ Up-to-date FSImage + EditLogs (no replay delay)
  ✅ Up-to-date block→DataNode mapping (no block report wait)
  ✅ Automatic failure detection (no human in the loop)
  ✅ Automatic promotion to Active (no manual commands)
  ✅ Fencing of the old Active (prevent split-brain)

💡 HOW CAN WE SOLVE IT

HA INTRODUCES FOUR NEW COMPONENTS:

═══════════════════════════════════════════════════════════
COMPONENT 1: JOURNAL NODES (Shared Edit Log Storage)
═══════════════════════════════════════════════════════════

PROBLEM:
→ Active NameNode writes EditLog entries for every metadata change
→ Standby NameNode needs these edits IN REAL-TIME to stay synchronized
→ If Active writes to local disk, Standby cannot read them
→ If Active writes to NFS shared storage, NFS becomes single point of failure

SOLUTION — JOURNAL NODES:
→ A quorum of lightweight daemons (typically 3 or 5)
→ Active NameNode writes each edit to ALL JournalNodes simultaneously
→ Write succeeds if MAJORITY acknowledge (quorum write)
→ Standby NameNode reads edits from JournalNodes continuously

  Active NN writes edit #5001:
    → Send to JN1: ✅ ACK
    → Send to JN2: ✅ ACK
    → Send to JN3: ❌ timeout
    → 2 of 3 ACKed = QUORUM MET = write SUCCESS

  Standby NN reads:
    → Polls JournalNodes every 2 seconds (configurable)
    → Fetches new edits: #5001
    → Applies edit #5001 to its in-memory namespace
    → Standby is now 2 seconds behind Active (worst case)

WHY QUORUM:
→ 3 JournalNodes: tolerates 1 failure (need 2 of 3)
→ 5 JournalNodes: tolerates 2 failures (need 3 of 5)
→ Formula: can tolerate (N-1)/2 failures with N JournalNodes
→ SAME principle as ZooKeeper quorum

═══════════════════════════════════════════════════════════
COMPONENT 2: ZOOKEEPER ENSEMBLE (Failure Detection + Election)
═══════════════════════════════════════════════════════════

PURPOSE:
→ Determines WHICH NameNode is Active at any given time
→ Only ONE NameNode can be Active — enforced by ZooKeeper lock
→ If Active NN fails, ZooKeeper releases the lock
→ Standby NN acquires the lock and becomes Active

HOW IT WORKS:
→ Both NNs compete for an EPHEMERAL ZNODE in ZooKeeper
→ Winner becomes Active, loser stays Standby
→ Ephemeral znode disappears if the owning session dies
→ When Active NN's session dies → lock released → Standby claims it

  ZooKeeper znode: /hadoop-ha/ns1/ActiveStandbyElectorLock
  
  Normal state:
    Active NN holds: /hadoop-ha/ns1/ActiveStandbyElectorLock (ephemeral)
    Standby NN watches this znode for deletion
  
  Failure:
    Active NN crashes → ZK session times out → ephemeral znode deleted
    Standby NN's watcher fires → Standby creates the znode → becomes Active

═══════════════════════════════════════════════════════════
COMPONENT 3: ZKFC (ZooKeeper Failover Controller)
═══════════════════════════════════════════════════════════

PURPOSE:
→ Runs as a SEPARATE PROCESS on each NameNode machine
→ Monitors the LOCAL NameNode's health
→ Communicates with ZooKeeper to manage Active/Standby state

THREE RESPONSIBILITIES:

  1. HEALTH MONITORING
     → Periodically calls NameNode's health check RPC
     → Detects: NN crashed, NN hung (GC pause), NN unreachable
  
  2. ZOOKEEPER SESSION MANAGEMENT
     → Maintains a session with ZooKeeper
     → Creates/monitors the election znode
  
  3. FENCING TRIGGER
     → Before promoting Standby to Active,
       MUST ensure the old Active is truly dead
     → Triggers FENCING METHOD to kill the old Active

  │  Machine A (Active NN)        Machine B (Standby NN)  │
  │                                                         │
  │  ┌──────────┐  ┌──────┐     ┌──────────┐  ┌──────┐   │
  │  │ NameNode │←─│ ZKFC │     │ NameNode │←─│ ZKFC │   │
  │  │ (Active) │  │      │     │(Standby) │  │      │   │
  │  └──────────┘  └──┬───┘     └──────────┘  └──┬───┘   │
  │                    │                           │        │
  │                    └───────────┬───────────────┘        │
  │                                │                        │

═══════════════════════════════════════════════════════════
COMPONENT 4: FENCING (Split-Brain Prevention)
═══════════════════════════════════════════════════════════

THE MOST CRITICAL SAFETY MECHANISM IN HA:

PROBLEM — SPLIT-BRAIN:
→ What if Active NN is not DEAD but just SLOW (long GC pause)?
→ ZKFC thinks it's dead → promotes Standby to Active
→ Old "Active" recovers from GC pause → now TWO Active NameNodes
→ Both accept client writes → METADATA CORRUPTION → DATA LOSS

SOLUTION — FENCING:
→ Before Standby becomes Active, it must GUARANTEE the old Active
   cannot accept any more writes
→ This is called "fencing" the old Active

FENCING METHODS (in order of preference):

  Method 1: sshfence (Default)
  → SSH into the old Active's machine
  → Kill the NameNode process: kill -9 <pid>
  → REQUIRES: passwordless SSH between NN machines
  → RISK: SSH may fail if network partition caused the "failure"

  Method 2: shell (Custom script)
  → Run a custom script to fence the old Active
  → Examples:
    - Power off the machine via IPMI/iLO
    - Disable the network port via switch API
    - Write to a shared file that the old NN checks
  → Most reliable: hardware-level fencing (power off)

  Method 3: PowerFence (STONITH — Shoot The Other Node In The Head)
  → Uses IPMI to hard-power-off the old Active's machine
  → MOST RELIABLE — hardware cannot lie
  → Used in mission-critical financial systems

  CRITICAL RULE:
  → If ALL fencing methods fail, failover DOES NOT PROCEED
  → Better to stay down than to have split-brain
  → This is a SAFETY decision, not a convenience decision

✅ WHAT SHOULD DO

1. DEPLOY ZOOKEEPER BEFORE CONFIGURING HA
   → ZooKeeper must be running BEFORE formatting HA NameNodes
   → ZKFC cannot start without a ZooKeeper ensemble
   → Use odd number of ZK nodes: 3 (tolerates 1 failure) or 5 (tolerates 2)

2. TEST FENCING METHODS BEFORE GOING LIVE
   → Configure sshfence AND a backup fencing method
   → Test manually:
     ssh namenode1-host "kill -9 $(jps | grep NameNode | awk '{print $1}')"
   → If SSH fails, the backup method MUST work
   → NEVER go to production with untested fencing

3. PLACE NAMENODE MACHINES IN DIFFERENT RACK/POWER DOMAINS
   → Active NN on Rack A, Standby NN on Rack B
   → If Rack A loses power, Standby on Rack B takes over
   → Same rack = same failure domain = HA is useless

4. MONITOR ZKFC PROCESS SEPARATELY FROM NAMENODE
   → ZKFC crash ≠ NameNode crash
   → But ZKFC crash = no automatic failover capability
   → Alert if ZKFC dies even if NameNode is healthy
   → Restart ZKFC immediately

5. CONFIGURE DATANODE TO REPORT TO BOTH NAMENODES
   → This happens automatically when dfs.nameservices lists HA namenodes
   → Verify: DataNode logs show registration with BOTH Active and Standby
   → If DataNode only reports to Active, Standby has stale block map

6. SET APPROPRIATE ZOOKEEPER SESSION TIMEOUT
   → Too short: False failovers during GC pauses
   → Too long: Slow failure detection
   → Recommended: ha.zookeeper.session-timeout.ms = 10000 (10 seconds)
   → This means: 10 seconds to detect failure + fencing time

7. PERFORM PLANNED FAILOVER FOR MAINTENANCE
   → Before patching Active NN: trigger graceful failover
   → hdfs haadmin -ns ns1 -failover nn1 nn2
   → This is SAFE: no fencing needed, no data loss risk
   → Patch the now-Standby, then failover back if desired

🚫 WHAT SHOULD NOT DO

1. DO NOT RUN WITHOUT FENCING CONFIGURED
   → HA without fencing = guaranteed split-brain eventually
   → Hadoop REFUSES to start HA without at least one fencing method
   → If you see "no fencing method configured" → STOP AND FIX

2. DO NOT CO-LOCATE ACTIVE AND STANDBY ON SAME MACHINE
   → Defeats the entire purpose of HA
   → Machine failure takes out BOTH NameNodes
   → Even for development/testing: use separate VMs at minimum

3. DO NOT USE SHARED NFS FOR EDITLOGS IN HA MODE
   → Hadoop 1.x used shared NFS — replaced by JournalNodes in Hadoop 2.x+
   → NFS is a single point of failure
   → NFS has no quorum semantics — cannot prevent split-brain writes
   → JournalNodes are the ONLY supported shared storage for HA

4. DO NOT MANUALLY PROMOTE STANDBY WITHOUT FENCING
   → hdfs haadmin -transitionToActive nn2 --forcemanual
   → This bypasses fencing — EXTREMELY DANGEROUS
   → Only use if you have physically verified the old Active is dead
   → In production: NEVER use --forcemanual in automation

5. DO NOT IGNORE JOURNALNODE HEALTH
   → If 2 of 3 JournalNodes die, Active NN CANNOT write edits
   → Active NN will shut itself down (safety measure)
   → Monitor JournalNodes as critically as NameNodes
   → JN failure = ticking time bomb for the entire namespace

6. DO NOT SET ZOOKEEPER SESSION TIMEOUT TOO LOW
   → ZK session timeout < NN GC pause time = false failovers
   → If NN has 8-second GC pauses and ZK timeout is 5 seconds
     → ZK thinks NN is dead → triggers failover → NN recovers
     → Now you have split-brain risk
   → RULE: ZK timeout > 2× worst-case GC pause time

7. DO NOT FORGET TO FORMAT ZKFC BEFORE FIRST START
   → hdfs zkfc -formatZK must run ONCE before starting ZKFC
   → Creates the necessary znodes in ZooKeeper
   → Without this: ZKFC crashes on startup with confusing errors

🚫 WHEN NOT TO USE

HARD RULES — NameNode HA IS WRONG WHEN:

1. SINGLE-NODE PSEUDO-DISTRIBUTED MODE
   → You need at minimum 2 machines for HA
   → Plus 3 for JournalNodes (can co-locate with other services)
   → If you only have 1 machine, HA is physically impossible

2. SHORT-LIVED EPHEMERAL CLUSTERS
   → Cluster spins up, runs batch job, shuts down in 2 hours
   → HA overhead (ZK, JN, ZKFC) wastes resources
   → If NN fails, just restart the cluster

3. DATA CAN BE FULLY REGENERATED QUICKLY
   → If all HDFS data is a cache/copy of data in S3/database
   → And regeneration takes < 30 minutes
   → Manual recovery may be acceptable (with business approval)

4. BUDGET DOES NOT ALLOW ADDITIONAL MACHINES
   → HA requires: 2 NN machines + 3 JN machines minimum
   → If budget is constrained, invest in:
     - Regular FSImage backups
     - Documented manual recovery procedure
     - Tested recovery drills
   → This is RISK ACCEPTANCE, not a technical solution

⚙️ HOW IT ONLY WORKS — INTERNAL MECHANICS

═══════════════════════════════════════════════════════════
INTERNAL MECHANIC 1: JOURNALNODE WRITE PROTOCOL
═══════════════════════════════════════════════════════════

When Active NameNode processes a metadata operation (e.g., create file):

  Client: "Create /trades/2024-01-15/orders.parquet"
  
  Active NN:
    Step 1: Validate request (permissions, path exists, etc.)
    Step 2: Assign new file ID and block IDs
    Step 3: Create EditLog entry:
      {
        "txid": 500001,
        "op": "OP_ADD",
        "path": "/trades/2024-01-15/orders.parquet",
        "replication": 3,
        "blockSize": 134217728,
        "timestamp": 1705334400000
      }
    
    Step 4: Write edit to JournalNode quorum (SYNCHRONOUS)
    
      ┌───────────┐     write(txid=500001)     ┌──────┐
      │ Active NN │ ─────────────────────────→ │ JN1  │ → ACK ✅
      │           │ ─────────────────────────→ │ JN2  │ → ACK ✅
      │           │ ─────────────────────────→ │ JN3  │ → timeout ❌
      └───────────┘                             └──────┘
      
      Quorum: 2 of 3 ACKed → WRITE SUCCEEDS
      
    Step 5: Apply edit to in-memory namespace
    Step 6: Respond to client: "File created successfully"

  CRITICAL ORDERING:
  → Edit MUST be written to JN quorum BEFORE responding to client
  → This ensures the Standby will eventually see this edit
  → If quorum write fails → client gets an error → edit is lost
  → This is a STRONG CONSISTENCY guarantee

═══════════════════════════════════════════════════════════
INTERNAL MECHANIC 2: STANDBY EDIT TAILING
═══════════════════════════════════════════════════════════

Standby NameNode continuously reads new edits from JournalNodes:

  ┌──────────────┐    poll every 2 sec    ┌──────┐
  │ Standby NN   │ ←─────────────────── │ JN1  │
  │              │                        │ JN2  │
  │ Last applied:│                        │ JN3  │
  │ txid=500000  │                        └──────┘
  └──────────────┘
  
  Poll result: "New edits available: txid 500001"
  
  Standby NN:
    Step 1: Fetch edit txid=500001 from JournalNode
    Step 2: Apply to in-memory namespace:
            Create file entry /trades/2024-01-15/orders.parquet
    Step 3: Update last-applied: txid=500001
    Step 4: Wait 2 seconds, poll again

  EDIT TAILING CONFIGURATION:
    dfs.ha.tail-edits.period = 2000ms (default: how often to poll)
    
  IN-PROGRESS EDIT TAILING (Hadoop 3.x):
    → Standby can tail edits from the CURRENT in-progress log segment
    → Not just finalized segments
    → Reduces lag from minutes to seconds
    → Configured via: dfs.ha.tail-edits.in-progress = true

  STANDBY LAG:
    → Best case: 0-2 seconds behind Active
    → Worst case: ~1 edit log segment behind (if tailing finalized only)
    → During failover: Standby must apply any remaining unapplied edits
    → This typically takes < 1 second

═══════════════════════════════════════════════════════════
INTERNAL MECHANIC 3: DATANODE DUAL REPORTING
═══════════════════════════════════════════════════════════

WITHOUT HA (recall Project 2):
  DataNode → heartbeat → 1 NameNode
  DataNode → block report → 1 NameNode

WITH HA:
  DataNode → heartbeat → Active NN
  DataNode → heartbeat → Standby NN (simultaneously)
  DataNode → block report → Active NN
  DataNode → block report → Standby NN (simultaneously)

  ┌────────────┐
  │ DataNode   │──heartbeat (3s)──→ Active NN
  │            │──heartbeat (3s)──→ Standby NN
  │            │
  │            │──block report (6hr)──→ Active NN
  │            │──block report (6hr)──→ Standby NN
  └────────────┘

  WHY THIS MATTERS:
  → Standby NN has COMPLETE block→DataNode mapping at ALL times
  → During failover, Standby does NOT need to wait for block reports
  → This is what makes failover take SECONDS instead of MINUTES
  → Recall: Without HA, block report collection takes 5-20 minutes

  IMPLEMENTATION DETAIL:
  → DataNode discovers both NNs from hdfs-site.xml configuration
  → DataNode sends identical heartbeats/reports to both
  → Active NN processes and responds with commands (replicate, delete)
  → Standby NN processes but does NOT respond with commands
    (Standby is read-only — it never tells DataNodes to do anything)

═══════════════════════════════════════════════════════════
INTERNAL MECHANIC 4: AUTOMATIC FAILOVER SEQUENCE
═══════════════════════════════════════════════════════════

Complete timeline of an automatic failover:

  T+0.000s: Active NN on Machine A crashes (process dies)
  
  T+0.000s: ZKFC-A detects local NN health check failure
             (ZKFC calls NN's monitorHealth() RPC — no response)
  
  T+0.500s: ZKFC-A retries health check — still no response
  
  T+1.000s: ZKFC-A declares local NN UNHEALTHY
             ZKFC-A releases its ZooKeeper session
             → Ephemeral znode /hadoop-ha/ns1/ActiveBreadCrumb disappears
  
  T+1.100s: ZooKeeper notifies ZKFC-B (on Machine B):
             "The Active lock has been released"
  
  T+1.200s: ZKFC-B begins failover process:
             Step 1: Attempt to FENCE the old Active
  
  T+1.300s: ZKFC-B runs sshfence:
             ssh machine-a "fuser -k -9 /path/to/namenode/pidfile"
             → Machine A's NameNode process is killed (if not already dead)
             → sshfence returns SUCCESS
  
  T+2.000s: Fencing confirmed — old Active is guaranteed dead
  
  T+2.100s: ZKFC-B tells Standby NN on Machine B:
             "Transition to Active state"
  
  T+2.200s: Standby NN on Machine B:
             Step 1: Apply any remaining unapplied edits from JournalNodes
                     (typically < 2 seconds worth of edits)
             Step 2: Finalize the current edit log segment
             Step 3: Start a new edit log segment as Active writer
             Step 4: Begin accepting client requests
  
  T+2.500s: NEW ACTIVE NN is serving clients
  
  T+3.000s: DataNodes' next heartbeat goes to new Active
             → DataNodes automatically follow the new Active
             → No DataNode restart needed
  
  T+5.000s: Clients' next RPC goes to new Active
             → HDFS client library has built-in HA retry logic
             → Client automatically retries on the new Active
             → Client MAY see a brief IOException, then succeeds
  
  TOTAL FAILOVER TIME: ~2.5 SECONDS (NN perspective)
  TOTAL CLIENT-VISIBLE OUTAGE: ~5-30 SECONDS
  (depends on client retry configuration and ZK session timeout)

═══════════════════════════════════════════════════════════
INTERNAL MECHANIC 5: CLIENT-SIDE HA PROXY
═══════════════════════════════════════════════════════════

HOW CLIENTS FIND THE ACTIVE NAMENODE:

Without HA:
  Client connects to: hdfs://namenode-host:9000
  → Single address — if it's down, client fails

With HA:
  Client connects to: hdfs://ns1
  → ns1 is a LOGICAL name, not a physical host
  → Client library resolves ns1 using hdfs-site.xml:
    dfs.namenode.rpc-address.ns1.nn1 = machine-a:9000
    dfs.namenode.rpc-address.ns1.nn2 = machine-b:9000
  
  → Client uses a FAILOVER PROXY PROVIDER:
    dfs.client.failover.proxy.provider.ns1 = 
      org.apache.hadoop.hdfs.server.namenode.ha.ConfiguredFailoverProxyProvider
  
  → ConfiguredFailoverProxyProvider:
    Step 1: Try nn1 (machine-a:9000)
    Step 2: If nn1 responds with "I am Standby" → try nn2
    Step 3: If nn1 is unreachable → try nn2
    Step 4: If nn2 responds with "I am Active" → use nn2
    Step 5: Cache the Active address for future calls
    Step 6: If cached Active fails → re-discover

  RETRY CONFIGURATION:
    dfs.client.failover.max.attempts = 15
    dfs.client.failover.sleep.base.millis = 500
    dfs.client.failover.sleep.max.millis = 15000

INTERNAL MECHANIC 6: CHECKPOINTING IN HA MODE
═══════════════════════════════════════════════════════════

WITHOUT HA:
→ SecondaryNameNode periodically merges FSImage + EditLogs
→ SecondaryNameNode is NOT a backup (common misconception)
→ It only reduces EditLog size for faster restart

WITH HA:
→ SecondaryNameNode is NO LONGER NEEDED
→ Standby NameNode performs checkpointing automatically
→ Standby already has up-to-date namespace in memory
→ Periodically writes a new FSImage and sends it to Active

  Standby NN checkpoint process:
    Step 1: Apply all pending edits to in-memory namespace
    Step 2: Write FSImage to local disk
    Step 3: Upload FSImage to Active NN via HTTP
    Step 4: Active NN replaces its old FSImage
    Step 5: Active NN can now delete old EditLog segments

  Configuration:
    dfs.namenode.checkpoint.period = 3600 (seconds — every 1 hour)
    dfs.namenode.checkpoint.txns = 1000000 (or after 1M transactions)

  BENEFIT:
  → No separate SecondaryNameNode process needed
  → Checkpointing is more reliable (Standby has COMPLETE namespace)
  → Active NN's EditLog stays small → faster recovery

⚠️ COMMON PITFALLS AND FAILURE MODES

PITFALL 1: SPLIT-BRAIN — TWO ACTIVE NAMENODES
→ SYMPTOM: Data corruption, inconsistent reads, missing blocks
→ CAUSE: Fencing failed but failover proceeded anyway
→ THIS SHOULD NEVER HAPPEN if fencing is configured correctly
→ Hadoop prevents this: if ALL fencing methods fail, failover ABORTS
→ BUT: If you use --forcemanual, you bypass this safety
→ RULE: NEVER use --forcemanual in production scripts

PITFALL 2: ZKFC NOT RUNNING — SILENT HA DEGRADATION
→ SYMPTOM: Active NN crashes but no failover happens
→ CAUSE: ZKFC process died silently hours ago
→ EFFECT: You THINK you have HA but you don't
→ FIX: Monitor ZKFC process independently
→ Alert: "ZKFC not running on namenode host" = CRITICAL alert

PITFALL 3: JOURNALNODE QUORUM LOST
→ SYMPTOM: Active NN cannot write edits → shuts itself down
→ CAUSE: 2 of 3 JournalNodes crashed
→ EFFECT: ENTIRE namespace goes down (both Active and Standby)
→ FIX: 
   - Use 5 JournalNodes for critical clusters (tolerates 2 failures)
   - Place JNs in different failure domains
   - Monitor JN health with same priority as NN health

PITFALL 4: FALSE FAILOVER DURING GC PAUSE
→ SYMPTOM: Failover triggered, then original NN recovers → chaos
→ CAUSE: ZK session timeout < GC pause duration
→ TIMELINE:
    T+0s: Active NN enters 15-second GC pause
    T+10s: ZK session timeout (10s) → ZKFC triggers failover
    T+12s: Fencing kills old Active (even though it would have recovered)
    T+15s: GC pause would have ended — but NN is already dead
→ FIX: Set ha.zookeeper.session-timeout.ms > 2× worst GC pause
→ ALSO: Tune GC to reduce pause times (Project 23)

PITFALL 5: STANDBY FALLS TOO FAR BEHIND
→ SYMPTOM: Failover succeeds but takes 30+ seconds for Standby to catch up
→ CAUSE: Edit tailing period too long, or burst of metadata operations
→ FIX:
   - dfs.ha.tail-edits.period = 1000 (1 second instead of default)
   - dfs.ha.tail-edits.in-progress = true (tail current segment)
   - Monitor: Standby lag via JMX metric "LastAppliedTxId"
   - Alert if gap between Active and Standby txid > 10000

PITFALL 6: DATANODE DOES NOT REGISTER WITH STANDBY
→ SYMPTOM: After failover, new Active reports 0 live DataNodes
→ CAUSE: DataNode configuration does not list both NN addresses
→ CAUSE: Network firewall blocks DataNode → Standby NN communication
→ FIX: Verify dfs.nameservices and dfs.ha.namenodes.ns1 in DataNode config
→ FIX: Verify firewall allows DataNode to reach BOTH NN hosts on port 9000

PITFALL 7: ZOOKEEPER ENSEMBLE FAILURE
→ SYMPTOM: ZKFC cannot connect to ZooKeeper → no failover possible
→ CAUSE: ZK ensemble lost quorum (2 of 3 ZK nodes down)
→ EFFECT: HA is disabled — running in "hope mode"
→ ACTIVE NN continues working fine (ZK failure doesn't kill Active)
→ BUT: if Active then fails, NO automatic failover
→ FIX: Use 5 ZK nodes for production, monitor ZK independently

📈 SCALABILITY CONSIDERATIONS

HA OVERHEAD ANALYSIS:

1. JOURNALNODE WRITE LATENCY
   → Every metadata operation adds JN quorum write latency
   → Typical JN write: 1-5ms (SSD), 5-15ms (HDD)
   → Impact: ~5-10% increase in metadata operation latency
   → For most workloads: imperceptible
   → For workloads creating millions of tiny files: measurable

2. DATANODE DUAL REPORTING OVERHEAD
   → Each DataNode sends heartbeats and block reports to TWO NNs
   → Heartbeat overhead: negligible (small packet every 3 seconds)
   → Block report overhead: significant during full block reports
     - 5000 DataNodes × 2 NNs = 10,000 block reports per interval
     - Each block report can be 10-100MB for DataNodes with millions of blocks
   → Mitigation: Stagger block reports, increase interval

3. STANDBY NN RESOURCE USAGE
   → Standby consumes nearly identical RAM as Active
   → Standby CPU usage: 30-50% of Active (no client requests, only edit replay)
   → Network: continuous edit tailing from JournalNodes
   → Disk: periodic FSImage checkpointing

4. JOURNALNODE DISK USAGE
   → Stores EditLog segments until cleaned up after checkpointing
   → Typical: 1-10GB per JN (depends on checkpoint frequency)
   → Use SSD for JN storage: write latency directly impacts NN throughput

🔍 INTERPRETABILITY CONSIDERATIONS

MONITORING HA-SPECIFIC METRICS:

1. NAMENODE HA STATE
   → JMX: Hadoop:service=NameNode,name=NameNodeStatus
   → Key metric: "State" = "active" or "standby"
   → Alert if: Both NNs report "standby" (no Active = cluster down)
   → Alert if: Both NNs report "active" (SPLIT-BRAIN = critical)

2. EDIT LAG BETWEEN ACTIVE AND STANDBY
   → Active NN JMX: "LastWrittenTransactionId" = 500050
   → Standby NN JMX: "LastAppliedOrWrittenTransactionId" = 500048
   → Lag = 500050 - 500048 = 2 transactions
   → Alert if lag > 10000 (Standby falling behind)
   → Alert if lag > 100000 (Standby critically behind — failover will be slow)

3. JOURNALNODE SYNC STATUS
   → JN JMX: Hadoop:service=JournalNode,name=JournalNodeInfo
   → Key metric: "LastWrittenTxId" per JN
   → All JNs should be within a few transactions of each other
   → Alert if any JN is > 1000 transactions behind others

4. ZKFC HEALTH
   → No built-in JMX for ZKFC
   → Monitor process existence: jps | grep DFSZKFailoverController
   → Monitor ZooKeeper znode:
     zkCli.sh -server zk1:2181 get /hadoop-ha/ns1/ActiveStandbyElectorLock
   → Shows which NN currently holds the Active lock

5. FENCING HISTORY
   → ZKFC logs: grep "fencing" in ZKFC log file
   → Shows every fencing attempt: success/failure, method used, duration
   → Review after every failover event

6. WEB UI INDICATORS
   → Active NN: http://active-nn:9870 → shows "active" in header
   → Standby NN: http://standby-nn:9870 → shows "standby" in header
   → Standby UI is READ-ONLY but shows namespace state
   → Useful for verifying Standby is synchronized

        echo "[$TIMESTAMP] ⚠️ WARNING: JournalNode down on $jn_host" >> $LOG_FILE
        $ALERT_SCRIPT "WARNING" "JournalNode down on $jn_host" 2>/dev/null
    fi
done

if [ $JN_ALIVE -lt 2 ]; then
    echo "[$TIMESTAMP] ❌ CRITICAL: JournalNode quorum LOST ($JN_ALIVE/3 alive)" >> $LOG_FILE
    $ALERT_SCRIPT "CRITICAL" "JournalNode quorum lost. Only $JN_ALIVE/3 alive. Active NN will shutdown if it needs to write." 2>/dev/null
fi

# ── CHECK 4: ZooKeeper Ensemble Health ──
ZK_ALIVE=0
for zk_host in jn1-host jn2-host jn3-host; do
    zk_status=$(echo "ruok" | nc -w 2 $zk_host 2181 2>/dev/null)
    if [ "$zk_status" = "imok" ]; then
        ZK_ALIVE=$((ZK_ALIVE + 1))
    else
        echo "[$TIMESTAMP] ⚠️ WARNING: ZooKeeper down on $zk_host" >> $LOG_FILE
    fi
done

if [ $ZK_ALIVE -lt 2 ]; then
    echo "[$TIMESTAMP] ❌ CRITICAL: ZooKeeper quorum LOST ($ZK_ALIVE/3 alive)" >> $LOG_FILE
    $ALERT_SCRIPT "CRITICAL" "ZooKeeper quorum lost. Automatic failover DISABLED." 2>/dev/null
fi

# ── CHECK 5: Edit Lag (Standby synchronization) ──
STANDBY_NN=$([ "$nn1_state" = "standby" ] && echo "nn1-host" || echo "nn2-host")
ACTIVE_NN=$([ "$nn1_state" = "active" ] && echo "nn1-host" || echo "nn2-host")

# Fetch last written txid from Active
active_txid=$(curl -s "http://$ACTIVE_NN:9870/jmx?qry=Hadoop:service=NameNode,name=NameNodeInfo" 2>/dev/null | \
    python3 -c "import sys,json; data=json.load(sys.stdin); print(data['beans'][0].get('LastWrittenTransactionId',0))" 2>/dev/null || echo "0")

# Fetch last applied txid from Standby
standby_txid=$(curl -s "http://$STANDBY_NN:9870/jmx?qry=Hadoop:service=NameNode,name=FSNamesystem" 2>/dev/null | \
    python3 -c "import sys,json; data=json.load(sys.stdin); print(data['beans'][0].get('LastWrittenTransactionId',0))" 2>/dev/null || echo "0")

if [ "$active_txid" != "0" ] && [ "$standby_txid" != "0" ]; then
    EDIT_LAG=$((active_txid - standby_txid))
    echo "[$TIMESTAMP] Edit lag: $EDIT_LAG txns (Active=$active_txid, Standby=$standby_txid)" >> $LOG_FILE
    
    if [ $EDIT_LAG -gt 100000 ]; then
        $ALERT_SCRIPT "WARNING" "Standby NN edit lag is $EDIT_LAG transactions. Failover will be slow." 2>/dev/null
    fi
fi

echo "[$TIMESTAMP] HA health check complete" >> $LOG_FILE
#!/usr/bin/env python3
# ════════════════════════════════════════════════════════════════
# FILE 7: ha_dashboard_collector.py
# Collects HA metrics for Grafana/Prometheus dashboards
# ════════════════════════════════════════════════════════════════

"""
PURPOSE:
Collects all HA-related metrics from NameNodes, JournalNodes,
ZooKeeper, and ZKFC into a structured format for monitoring dashboards.

Outputs metrics in Prometheus exposition format for scraping.
"""

import json
import urllib.request
import socket
import time
import sys

class HAMetricsCollector:
    """
    Collects NameNode HA health metrics from JMX endpoints.
    
    Architecture recall:
    → Active NN + ZKFC on Machine A
    → Standby NN + ZKFC on Machine B
    → 3 JournalNodes (shared edit log)
    → 3 ZooKeeper nodes (failover coordination)
    """
    
    def __init__(self, config):
        self.config = config
        self.metrics = []
    
    def _fetch_jmx(self, host, port, query):
        """Fetch JMX metrics from a Hadoop daemon."""
        url = f"http://{host}:{port}/jmx?qry={query}"
        try:
            with urllib.request.urlopen(url, timeout=10) as response:
                data = json.loads(response.read().decode())
                return data.get('beans', [])
        except Exception as e:
            return None
    
    def _check_port(self, host, port, timeout=3):
        """Check if a TCP port is reachable."""
        try:
            sock = socket.socket(socket.AF_INET, socket.SOCK_STREAM)
            sock.settimeout(timeout)
            result = sock.connect_ex((host, port))
            sock.close()
            return result == 0
        except Exception:
            return False
    
    def _add_metric(self, name, value, labels=None):
        """Add a metric in Prometheus format."""
        label_str = ""
        if labels:
            pairs = [f'{k}="{v}"' for k, v in labels.items()]
            label_str = "{" + ",".join(pairs) + "}"
        self.metrics.append(f"hadoop_ha_{name}{label_str} {value}")
    
    def collect_namenode_metrics(self):
        """Collect NameNode HA state and performance metrics."""
        for nn_id, nn_config in self.config['namenodes'].items():
            host = nn_config['host']
            http_port = nn_config['http_port']
            labels = {"namenode": nn_id, "host": host}
            
            # Check liveness
            alive = 1 if self._check_port(host, http_port) else 0
            self._add_metric("namenode_alive", alive, labels)
            
            if not alive:
                self._add_metric("namenode_active", 0, labels)
                continue
            
            # Get HA state
            beans = self._fetch_jmx(host, http_port, "Hadoop:service=NameNode,name=NameNodeStatus")
            if beans:
                state = beans[0].get('State', 'unknown')
                self._add_metric("namenode_active", 1 if state == 'active' else 0, labels)
                self._add_metric("namenode_standby", 1 if state == 'standby' else 0, labels)
            
            # Get namespace metrics
            beans = self._fetch_jmx(host, http_port, "Hadoop:service=NameNode,name=FSNamesystem")
            if beans:
                fsn = beans[0]
                self._add_metric("files_total", fsn.get('FilesTotal', 0), labels)
                self._add_metric("blocks_total", fsn.get('BlocksTotal', 0), labels)
                self._add_metric("capacity_used_bytes", fsn.get('CapacityUsed', 0), labels)
                self._add_metric("capacity_total_bytes", fsn.get('CapacityTotal', 0), labels)
                self._add_metric("last_written_txid", fsn.get('LastWrittenTransactionId', 0), labels)
                self._add_metric("under_replicated_blocks", fsn.get('UnderReplicatedBlocks', 0), labels)
                self._add_metric("missing_blocks", fsn.get('MissingBlocks', 0), labels)
                self._add_metric("live_datanodes", fsn.get('NumLiveDataNodes', 0), labels)
                self._add_metric("dead_datanodes", fsn.get('NumDeadDataNodes', 0), labels)
                self._add_metric("stale_datanodes", fsn.get('NumStaleDataNodes', 0), labels)
            
            # Get heap metrics
            beans = self._fetch_jmx(host, http_port, "java.lang:type=Memory")
            if beans:
                heap = beans[0].get('HeapMemoryUsage', {})
                self._add_metric("heap_used_bytes", heap.get('used', 0), labels)
                self._add_metric("heap_max_bytes", heap.get('max', 0), labels)
                if heap.get('max', 0) > 0:
                    pct = (heap.get('used', 0) / heap.get('max', 1)) * 100
                    self._add_metric("heap_used_percent", round(pct, 1), labels)
            
            # Get GC metrics
            beans = self._fetch_jmx(host, http_port, "java.lang:type=GarbageCollector,name=*")
            if beans:
                total_gc_time = sum(b.get('CollectionTime', 0) for b in beans)
                total_gc_count = sum(b.get('CollectionCount', 0) for b in beans)
                self._add_metric("gc_time_millis_total", total_gc_time, labels)
                self._add_metric("gc_count_total", total_gc_count, labels)
            
            # Get RPC metrics
            beans = self._fetch_jmx(host, http_port, "Hadoop:service=NameNode,name=RpcActivityForPort*")
            if beans:
                rpc = beans[0]
                self._add_metric("rpc_queue_time_avg_ms", rpc.get('RpcQueueTimeAvgTime', 0), labels)
                self._add_metric("rpc_processing_time_avg_ms", rpc.get('RpcProcessingTimeAvgTime', 0), labels)
                self._add_metric("rpc_num_open_connections", rpc.get('NumOpenConnections', 0), labels)
    
    def collect_edit_lag(self):
        """Calculate edit transaction lag between Active and Standby."""
        active_txid = 0
        standby_txid = 0
        active_nn = None
        standby_nn = None
        
        for nn_id, nn_config in self.config['namenodes'].items():
            host = nn_config['host']
            http_port = nn_config['http_port']
            
            beans = self._fetch_jmx(host, http_port, "Hadoop:service=NameNode,name=NameNodeStatus")
            if beans:
                state = beans[0].get('State', 'unknown')
                
                fsn_beans = self._fetch_jmx(host, http_port, "Hadoop:service=NameNode,name=FSNamesystem")
                if fsn_beans:
                    txid = fsn_beans[0].get('LastWrittenTransactionId', 0)
                    
                    if state == 'active':
                        active_txid = txid
                        active_nn = nn_id
                    elif state == 'standby':
                        standby_txid = txid
                        standby_nn = nn_id
        
        if active_txid > 0 and standby_txid > 0:
            lag = active_txid - standby_txid
            self._add_metric("edit_lag_transactions", lag, 
                           {"active": active_nn or "unknown", 
                            "standby": standby_nn or "unknown"})
    
    def collect_journalnode_metrics(self):
        """Collect JournalNode health metrics."""
        alive_count = 0
        
        for jn_id, jn_config in self.config['journalnodes'].items():
            host = jn_config['host']
            http_port = jn_config.get('http_port', 8480)
            labels = {"journalnode": jn_id, "host": host}
            
            alive = 1 if self._check_port(host, http_port) else 0
            self._add_metric("journalnode_alive", alive, labels)
            
            if alive:
                alive_count += 1
                
                beans = self._fetch_jmx(host, http_port, 
                         "Hadoop:service=JournalNode,name=JournalNodeInfo")
                if beans:
                    # Journal-specific metrics if available
                    pass
        
        total_jns = len(self.config['journalnodes'])
        quorum_size = (total_jns // 2) + 1
        quorum_healthy = 1 if alive_count >= quorum_size else 0
        
        self._add_metric("journalnode_alive_total", alive_count)
        self._add_metric("journalnode_quorum_healthy", quorum_healthy)
    
    def collect_zookeeper_metrics(self):
        """Collect ZooKeeper ensemble health."""
        alive_count = 0
        leader_count = 0
        
        for zk_id, zk_config in self.config['zookeepers'].items():
            host = zk_config['host']
            port = zk_config.get('port', 2181)
            labels = {"zookeeper": zk_id, "host": host}
            
            # Send "ruok" 4-letter command
            try:
                sock = socket.socket(socket.AF_INET, socket.SOCK_STREAM)
                sock.settimeout(3)
                sock.connect((host, port))
                sock.sendall(b"ruok")
                response = sock.recv(1024).decode().strip()
                sock.close()
                
                alive = 1 if response == "imok" else 0
            except Exception:
                alive = 0
            
            self._add_metric("zookeeper_alive", alive, labels)
            
            if alive:
                alive_count += 1
                
                # Check if leader or follower
                try:
                    sock = socket.socket(socket.AF_INET, socket.SOCK_STREAM)
                    sock.settimeout(3)
                    sock.connect((host, port))
                    sock.sendall(b"srvr")
                    response = sock.recv(4096).decode()
                    sock.close()
                    
                    if "leader" in response.lower():
                        self._add_metric("zookeeper_is_leader", 1, labels)
                        leader_count += 1
                    else:
                        self._add_metric("zookeeper_is_leader", 0, labels)
                except Exception:
                    pass
        
        total_zks = len(self.config['zookeepers'])
        quorum_size = (total_zks // 2) + 1
        
        self._add_metric("zookeeper_alive_total", alive_count)
        self._add_metric("zookeeper_quorum_healthy", 1 if alive_count >= quorum_size else 0)
        self._add_metric("zookeeper_leader_count", leader_count)
    
    def collect_all(self):
        """Run all metric collectors and return Prometheus-format output."""
        self.metrics = []
        self.metrics.append(f"# HELP hadoop_ha NameNode HA metrics")
        self.metrics.append(f"# TYPE hadoop_ha gauge")
        self.metrics.append(f"hadoop_ha_collection_timestamp {int(time.time())}")
        
        self.collect_namenode_metrics()
        self.collect_edit_lag()
        self.collect_journalnode_metrics()
        self.collect_zookeeper_metrics()
        
        return "\n".join(self.metrics)

ha_status() {

    nn2_state=$(hdfs haadmin -getServiceState nn2 2>/dev/null || echo "UNREACHABLE")

    echo "    nn2 (nn2-host): $nn2_state"
    echo ""

    for nn_host in nn1-host nn2-host; do
        safemode=$(curl -s "http://$nn_host:9870/jmx?qry=Hadoop:service=NameNode,name=NameNodeInfo" 2>/dev/null | \
            python3 -c "import sys,json; d=json.load(sys.stdin); print(d['beans'][0].get('Safemode',''))" 2>/dev/null)
        if [ -n "$safemode" ] && [ "$safemode" != "" ]; then

        fi
    done
    
    # Process check
    echo ""
    echo "  Processes:"
    for nn_host in nn1-host nn2-host; do

        echo "    $nn_host: NameNode=$nn_pid, ZKFC=$zkfc_pid"
    done
    
    echo ""
    echo "  JournalNodes:"
    for jn_host in jn1-host jn2-host jn3-host; do

    echo ""
    echo "  ZooKeeper:"

        zk_resp=$(echo "ruok" | nc -w 2 $zk_host 2181 2>/dev/null || echo "DOWN")
        echo "    $zk_host: $zk_resp"
    done
}

ha_failover() {
    echo "═══ PLANNED FAILOVER ═══"
    echo ""
    
    # Determine current Active
    nn1_state=$(hdfs haadmin -getServiceState nn1 2>/dev/null || echo "UNREACHABLE")
    nn2_state=$(hdfs haadmin -getServiceState nn2 2>/dev/null || echo "UNREACHABLE")
    
    if [ "$nn1_state" = "active" ]; then
        FROM="nn1"
        TO="nn2"
    elif [ "$nn2_state" = "active" ]; then
        FROM="nn2"
        TO="nn1"
    else
        echo "  ❌ Cannot determine Active NameNode. Aborting."
        echo "     nn1=$nn1_state, nn2=$nn2_state"
        exit 1
    fi
    
    echo "  Current Active: $FROM"
    echo "  Failover to:    $TO"
    echo ""
    
    # Pre-failover checks
    echo "  Pre-failover checks:"
    
    # Check Standby is healthy
    to_state=$(hdfs haadmin -getServiceState $TO 2>/dev/null)
    if [ "$to_state" != "standby" ]; then
        echo "    ❌ Target $TO is not in standby state (state=$to_state). Aborting."
        exit 1
    fi
    echo "    ✅ Target $TO is in standby state"
    
    # Check JournalNode quorum
    jn_alive=0
    for jn_host in jn1-host jn2-host jn3-host; do
        jn_check=$(ssh -o ConnectTimeout=5 $jn_host "jps 2>/dev/null | grep JournalNode" 2>/dev/null)
        [ -n "$jn_check" ] && jn_alive=$((jn_alive + 1))
    done
    if [ $jn_alive -lt 2 ]; then
        echo "    ❌ JournalNode quorum unhealthy ($jn_alive/3). Aborting."
        exit 1
    fi
    echo "    ✅ JournalNode quorum healthy ($jn_alive/3)"
    
    # Check ZooKeeper quorum
    zk_alive=0
    for zk_host in jn1-host jn2-host jn3-host; do
        zk_resp=$(echo "ruok" | nc -w 2 $zk_host 2181 2>/dev/null)
        [ "$zk_resp" = "imok" ] && zk_alive=$((zk_alive + 1))
    done
    if [ $zk_alive -lt 2 ]; then
        echo "    ❌ ZooKeeper quorum unhealthy ($zk_alive/3). Aborting."
        exit 1
    fi
    echo "    ✅ ZooKeeper quorum healthy ($zk_alive/3)"
    
    echo ""
    read -p "  Proceed with failover $FROM → $TO? (yes/no): " confirm
    if [ "$confirm" != "yes" ]; then
        echo "  Failover cancelled."
        exit 0
    fi
    
    echo ""
    echo "  Executing failover..."
    FAILOVER_START=$(date +%s%N)
    
    hdfs haadmin -failover $FROM $TO 2>&1 | sed 's/^/    /'
    
    FAILOVER_END=$(date +%s%N)
    FAILOVER_MS=$(( (FAILOVER_END - FAILOVER_START) / 1000000 ))
    
    echo ""
    echo "  Failover completed in ${FAILOVER_MS}ms"
    
    # Post-failover verification
    sleep 3
    new_from_state=$(hdfs haadmin -getServiceState $FROM 2>/dev/null)
    new_to_state=$(hdfs haadmin -getServiceState $TO 2>/dev/null)
    
    echo ""
    echo "  Post-failover states:"
    echo "    $FROM: $new_from_state (was active)"
    echo "    $TO:   $new_to_state (was standby)"
    
    if [ "$new_to_state" = "active" ] && [ "$new_from_state" = "standby" ]; then
        echo ""
        echo "  ✅ Failover SUCCESSFUL"
    else
        echo ""
        echo "  ❌ Failover may have FAILED — verify manually"
    fi
}

check_fencing() {
    echo "═══ FENCING CONNECTIVITY TEST ═══"
    echo ""
    echo "  Testing SSH fencing between NameNode machines..."
    echo ""
    
    # Test SSH from nn1 to nn2
    echo "  nn1-host → nn2-host:"
    ssh -o ConnectTimeout=10 nn1-host "ssh -o ConnectTimeout=5 -o BatchMode=yes nn2-host 'echo SSH_OK'" 2>/dev/null
    if [ $? -eq 0 ]; then
        echo "    ✅ SSH fencing path works"
    else
        echo "    ❌ SSH fencing WILL FAIL — fix passwordless SSH"
    fi
    
    # Test SSH from nn2 to nn1
    echo "  nn2-host → nn1-host:"
    ssh -o ConnectTimeout=10 nn2-host "ssh -o ConnectTimeout=5 -o BatchMode=yes nn1-host 'echo SSH_OK'" 2>/dev/null
    if [ $? -eq 0 ]; then
        echo "    ✅ SSH fencing path works"
    else
        echo "    ❌ SSH fencing WILL FAIL — fix passwordless SSH"
    fi
    
    echo ""
    echo "  Testing that fuser command exists on both hosts..."
    for nn_host in nn1-host nn2-host; do
        fuser_exists=$(ssh -o ConnectTimeout=5 $nn_host "which fuser 2>/dev/null" || echo "NOT FOUND")
        if [ "$fuser_exists" != "NOT FOUND" ]; then
            echo "    $nn_host: ✅ fuser found at $fuser_exists"
        else
            echo "    $nn_host: ❌ fuser NOT found — install psmisc package"
        fi
    done
}

save_namespace() {
    echo "═══ TRIGGER CHECKPOINT ON STANDBY ═══"
    echo ""
    
    # Find Standby NN
    nn1_state=$(hdfs haadmin -getServiceState nn1 2>/dev/null || echo "UNREACHABLE")
    nn2_state=$(hdfs haadmin -getServiceState nn2 2>/dev/null || echo "UNREACHABLE")
    
    if [ "$nn1_state" = "standby" ]; then
        STANDBY_HOST="nn1-host"
        STANDBY_ID="nn1"
    elif [ "$nn2_state" = "standby" ]; then
        STANDBY_HOST="nn2-host"
        STANDBY_ID="nn2"
    else
        echo "  ❌ Cannot find Standby NameNode. Aborting."
        exit 1
    fi
    
    echo "  Standby NameNode: $STANDBY_ID ($STANDBY_HOST)"
    echo "  Triggering saveNamespace (FSImage checkpoint)..."
    echo ""
    
    # In HA mode, the Standby performs checkpointing automatically
    # But we can trigger it manually via dfsadmin
    hdfs dfsadmin -saveNamespace 2>&1 | sed 's/^/    /'
    
    echo ""
    echo "  ✅ Checkpoint triggered"
    echo "  FSImage saved on Standby and uploaded to Active"
}

usage() {
    echo ""
    echo "Usage: $0 <scenario>"
    echo ""
    echo "Scenarios:"
    echo "  1  — Active NN dead, Standby alive, auto-failover failed"
    echo "  2  — Both NNs dead, JournalNodes alive"
    echo "  3  — Active NN dead, ZKFC dead, need manual promotion"
    echo "  4  — Split-brain detected — CRITICAL"
    echo "  5  — JournalNode quorum lost — Active NN shutting down"

scenario_1() {
    echo ""
    echo "═══ SCENARIO 1: Active Dead + Auto-Failover Failed ═══"
    echo ""
    echo "DIAGNOSIS:"
    echo "  → Active NN process died"
    echo "  → ZKFC did not trigger failover (ZKFC may be dead or ZK unreachable)"
    echo "  → Standby NN is healthy but still in standby state"
    echo ""
    echo "PROCEDURE:"
    echo ""
    
    # Step 1: Verify Active is truly dead
    echo "[Step 1] Verify Active NN is truly dead"
    echo "  Checking nn1-host..."
    nn1_alive=$(ssh -o ConnectTimeout=5 nn1-host "jps 2>/dev/null | grep '^[0-9]* NameNode$'" 2>/dev/null || echo "")
    echo "  Checking nn2-host..."
    nn2_alive=$(ssh -o ConnectTimeout=5 nn2-host "jps 2>/dev/null | grep '^[0-9]* NameNode$'" 2>/dev/null || echo "")
    
    echo "  nn1-host NameNode: $([ -n '$nn1_alive' ] && echo 'RUNNING' || echo 'DEAD')"
    echo "  nn2-host NameNode: $([ -n '$nn2_alive' ] && echo 'RUNNING' || echo 'DEAD')"
    
    # Determine which is dead and which is alive
    nn1_state=$(hdfs haadmin -getServiceState nn1 2>/dev/null || echo "UNREACHABLE")
    nn2_state=$(hdfs haadmin -getServiceState nn2 2>/dev/null || echo "UNREACHABLE")
    
    echo "  nn1 state: $nn1_state"
    echo "  nn2 state: $nn2_state"
    
    DEAD_NN=""
    ALIVE_NN=""
    ALIVE_HOST=""
    
    if [ "$nn1_state" = "UNREACHABLE" ] && [ "$nn2_state" = "standby" ]; then
        DEAD_NN="nn1"
        ALIVE_NN="nn2"
        ALIVE_HOST="nn2-host"
    elif [ "$nn2_state" = "UNREACHABLE" ] && [ "$nn1_state" = "standby" ]; then
        DEAD_NN="nn2"
        ALIVE_NN="nn1"
        ALIVE_HOST="nn1-host"
    else
        echo ""
        echo "  ❌ Cannot determine dead/alive NNs. Manual investigation required."
        echo "     If both are standby, one may have lost ZK session."
        return 1
    fi
    
    echo ""
    echo "  Dead NN:  $DEAD_NN"
    echo "  Alive NN: $ALIVE_NN ($ALIVE_HOST)"
    
    # Step 2: Manually fence the dead NN (extra safety)
    echo ""
    echo "[Step 2] Manual fencing — ensure dead NN cannot come back as Active"
    
    DEAD_HOST=$([ "$DEAD_NN" = "nn1" ] && echo "nn1-host" || echo "nn2-host")
    
    echo "  Attempting to kill any NameNode process on $DEAD_HOST..."
    ssh -o ConnectTimeout=10 $DEAD_HOST "
        NN_PID=\$(jps 2>/dev/null | grep '^[0-9]* NameNode$' | awk '{print \$1}')
        if [ -n \"\$NN_PID\" ]; then
            kill -9 \$NN_PID
            echo '  Killed NameNode PID: '\$NN_PID
        else
            echo '  No NameNode process found (already dead)'
        fi
        
        ZKFC_PID=\$(jps 2>/dev/null | grep DFSZKFailoverController | awk '{print \$1}')
        if [ -n \"\$ZKFC_PID\" ]; then
            kill -9 \$ZKFC_PID
            echo '  Killed ZKFC PID: '\$ZKFC_PID
        fi
    " 2>/dev/null || echo "  ⚠️  Cannot SSH to $DEAD_HOST — machine may be completely down"
    
    echo "  ✅ Fencing complete (or machine unreachable)"
    
    # Step 3: Manually transition Standby to Active
    echo ""
    echo "[Step 3] Manual transition: $ALIVE_NN → Active"
    echo ""
    echo "  ⚠️  WARNING: Using --forcemanual bypasses automatic fencing"
    echo "  We have already manually fenced in Step 2"
    echo ""
    
    read -p "  Proceed with manual transition? (yes/no): " confirm
    if [ "$confirm" != "yes" ]; then
        echo "  Aborted."
        return 1
    fi
    
    hdfs haadmin -transitionToActive --forcemanual $ALIVE_NN 2>&1 | sed 's/^/    /'
    
    sleep 3
    
    new_state=$(hdfs haadmin -getServiceState $ALIVE_NN 2>/dev/null || echo "UNREACHABLE")
    echo ""
    echo "  $ALIVE_NN state after transition: $new_state"
    
    if [ "$new_state" = "active" ]; then
        echo "  ✅ Manual failover SUCCESSFUL — $ALIVE_NN is now Active"
    else
        echo "  ❌ Manual failover FAILED — escalate to Hadoop admin"
    fi
    
    # Step 4: Recovery instructions
    echo ""
    echo "[Step 4] Post-Recovery Actions"
    echo "  1. Fix/replace the dead machine ($DEAD_HOST)"
    echo "  2. Start NameNode on $DEAD_HOST: hdfs --daemon start namenode"
    echo "     (It will auto-join as Standby)"
    echo "  3. Start ZKFC on $DEAD_HOST: hdfs --daemon start zkfc"
    echo "  4. Start ZKFC on $ALIVE_HOST: hdfs --daemon start zkfc"
    echo "     (if it was killed during manual fencing)"
    echo "  5. Verify: hdfs haadmin -getServiceState nn1"
    echo "  6. Verify: hdfs haadmin -getServiceState nn2"
    echo "  7. Run ha_failover_test.sh to confirm HA is restored"
}

scenario_4() {
    echo ""
    echo "═══ SCENARIO 4: SPLIT-BRAIN DETECTED — CRITICAL ═══"
    echo ""
    echo "  ⚠️⚠️⚠️  THIS IS THE MOST DANGEROUS SCENARIO  ⚠️⚠️⚠️"
    echo ""
    echo "  Two Active NameNodes can cause METADATA CORRUPTION"
    echo "  and PERMANENT DATA LOSS"
    echo ""
    echo "IMMEDIATE ACTIONS:"
    echo ""
    
    nn1_state=$(hdfs haadmin -getServiceState nn1 2>/dev/null || echo "UNREACHABLE")
    nn2_state=$(hdfs haadmin -getServiceState nn2 2>/dev/null || echo "UNREACHABLE")
    
    echo "  Current states: nn1=$nn1_state, nn2=$nn2_state"
    
    if [ "$nn1_state" != "active" ] || [ "$nn2_state" != "active" ]; then
        echo ""
        echo "  Split-brain NOT confirmed. Only one Active detected."
        echo "  False alarm — but investigate why this alert was triggered."
        return 0
    fi
    
    echo ""
    echo "  ❌ CONFIRMED: BOTH nn1 AND nn2 ARE ACTIVE"
    echo ""
    echo "  PROCEDURE:"
    echo "  1. IMMEDIATELY stop one NameNode to prevent further divergence"
    echo "  2. Choose the NN with HIGHER LastWrittenTransactionId to keep"
    echo ""
    
    # Get transaction IDs from both
    nn1_txid=$(curl -s "http://nn1-host:9870/jmx?qry=Hadoop:service=NameNode,name=FSNamesystem" 2>/dev/null | \
        python3 -c "import sys,json; d=json.load(sys.stdin); print(d['beans'][0].get('LastWrittenTransactionId',0))" 2>/dev/null || echo "0")
    nn2_txid=$(curl -s "http://nn2-host:9870/jmx?qry=Hadoop:service=NameNode,name=FSNamesystem" 2>/dev/null | \
        python3 -c "import sys,json; d=json.load(sys.stdin); print(d['beans'][0].get('LastWrittenTransactionId',0))" 2>/dev/null || echo "0")
    
    echo "  nn1 LastWrittenTransactionId: $nn1_txid"
    echo "  nn2 LastWrittenTransactionId: $nn2_txid"
    echo ""
    
    if [ "$nn1_txid" -ge "$nn2_txid" ]; then
        KEEP="nn1"
        KILL="nn2"
        KILL_HOST="nn2-host"
    else
        KEEP="nn2"
        KILL="nn1"
        KILL_HOST="nn1-host"
    fi
    
    echo "  RECOMMENDATION: Keep $KEEP (higher txid), kill $KILL"
    echo ""
    
    read -p "  KILL $KILL on $KILL_HOST? This is IRREVERSIBLE. (yes/no): " confirm
    if [ "$confirm" != "yes" ]; then
        echo "  Aborted. MANUAL INTERVENTION REQUIRED IMMEDIATELY."
        return 1
    fi
    
    # Kill the NN with fewer transactions
    echo "  Killing $KILL on $KILL_HOST..."
    ssh $KILL_HOST "
        NN_PID=\$(jps | grep '^[0-9]* NameNode$' | awk '{print \$1}')
        kill -9 \$NN_PID
        ZKFC_PID=\$(jps | grep DFSZKFailoverController | awk '{print \$1}')
        [ -n \"\$ZKFC_PID\" ] && kill -9 \$ZKFC_PID
    " 2>/dev/null
    
    echo "  ✅ $KILL killed on $KILL_HOST"
    echo ""
    echo "  POST-SPLIT-BRAIN RECOVERY:"
    echo "  1. Check JournalNode edit logs for conflicting transactions"
    echo "  2. Run: hdfs fsck / -files -blocks to check for corruption"
    echo "  3. If corruption found: restore from last known good backup"
    echo "  4. Bootstrap new Standby from the surviving Active:"
    echo "     ssh $KILL_HOST 'hdfs namenode -bootstrapStandby -force'"
    echo "  5. Restart $KILL as Standby"
    echo "  6. Restart ZKFC on both machines"
    echo "  7. Investigate ROOT CAUSE: why did fencing fail?"
    echo "  8. FILE AN INCIDENT REPORT"
}

scenario_5() {
    echo ""
    echo "═══ SCENARIO 5: JournalNode Quorum Lost ═══"
    echo ""
    echo "  When JN quorum is lost, Active NN cannot write edits"
    echo "  Active NN will shut itself down as a safety measure"
    echo ""
    
    echo "  Checking JournalNode status..."
    echo ""
    
    JN_STATUS=()
    for jn_host in jn1-host jn2-host jn3-host; do
        jn_alive=$(ssh -o ConnectTimeout=5 $jn_host "jps 2>/dev/null | grep JournalNode" 2>/dev/null || echo "")
        if [ -n "$jn_alive" ]; then
            echo "  $jn_host: ✅ JournalNode ALIVE"
            JN_STATUS+=("ALIVE")
        else
            echo "  $jn_host: ❌ JournalNode DEAD"
            JN_STATUS+=("DEAD")
        fi
    done
    
    ALIVE_COUNT=0
    for s in "${JN_STATUS[@]}"; do
        [ "$s" = "ALIVE" ] && ALIVE_COUNT=$((ALIVE_COUNT + 1))
    done
    
    echo ""
    echo "  JournalNodes alive: $ALIVE_COUNT/3"
    echo "  Quorum required:    2/3"
    echo ""
    
    if [ $ALIVE_COUNT -ge 2 ]; then
        echo "  ✅ Quorum is healthy. If Active NN shut down, restart it."
        echo "  Command: ssh <active-nn-host> 'hdfs --daemon start namenode'"
        return 0
    fi
    
    echo "  RECOVERY PROCEDURE:"
    echo ""
    echo "  Step 1: Restart dead JournalNodes FIRST"
    
    IDX=0
    for jn_host in jn1-host jn2-host jn3-host; do
        if [ "${JN_STATUS[$IDX]}" = "DEAD" ]; then
            echo "    Restarting JournalNode on $jn_host..."
            ssh -o ConnectTimeout=10 $jn_host "hdfs --daemon start journalnode" 2>/dev/null && \
                echo "    ✅ JournalNode restarted on $jn_host" || \
                echo "    ❌ Failed to restart on $jn_host — machine may be down"
        fi
        IDX=$((IDX + 1))
    done
    
    sleep 5
    
    echo ""
    echo "  Step 2: Verify quorum restored"
    NEW_ALIVE=0
    for jn_host in jn1-host jn2-host jn3-host; do
        jn_check=$(ssh -o ConnectTimeout=5 $jn_host "jps 2>/dev/null | grep JournalNode" 2>/dev/null || echo "")
        [ -n "$jn_check" ] && NEW_ALIVE=$((NEW_ALIVE + 1))
    done
    echo "  JournalNodes alive now: $NEW_ALIVE/3"
    
    if [ $NEW_ALIVE -ge 2 ]; then
        echo "  ✅ Quorum restored"
        echo ""
        echo "  Step 3: Restart NameNodes"
        echo "    Start Active first:  ssh <active-host> 'hdfs --daemon start namenode'"
        echo "    Then Standby:        ssh <standby-host> 'hdfs --daemon start namenode'"
        echo "    Then ZKFC on both:   hdfs --daemon start zkfc"
    else
        echo "  ❌ Quorum NOT restored — need to fix JournalNode machines"
        echo "     If JN data is lost, you may need to:"
        echo "     1. Initialize new JN from existing NN's edit logs"
        echo "     2. Or restore JN data from backup"
        echo "     3. Contact Hadoop support for data recovery"
    fi
}

set -e

CLUSTER_ID="production-cluster-001"

echo "╔══════════════════════════════════════════════════════════════╗"
echo "║  FEDERATION + HA FULL CLUSTER SETUP                          ║"
echo "║  3 Namespaces × 2 NameNodes = 6 NameNode Processes          ║"
echo "╚══════════════════════════════════════════════════════════════╝"

# ── STEP 1: Start ZooKeeper ──
echo ""
echo "[1/9] Starting ZooKeeper Ensemble"
for zk_host in jn1-host jn2-host jn3-host; do
    ssh $zk_host "/opt/zookeeper/bin/zkServer.sh start" 2>/dev/null
    echo "  ✅ ZK on $zk_host"
done
sleep 5

# ── STEP 2: Start JournalNodes ──
echo ""
echo "[2/9] Starting JournalNodes"
for jn_host in jn1-host jn2-host jn3-host; do
    ssh $jn_host "mkdir -p /opt/hadoop/data/journalnode && hdfs --daemon start journalnode"
    echo "  ✅ JN on $jn_host"
done
sleep 3

# ── STEP 3: Format and Start Active NameNodes (one per namespace) ──
echo ""
echo "[3/9] Formatting Active NameNodes"

declare -A NS_ACTIVE_HOSTS=(
    ["ns1"]="ns1-active-host"
    ["ns2"]="ns2-active-host"
    ["ns3"]="ns3-active-host"
)

for ns in ns1 ns2 ns3; do
    active_host="${NS_ACTIVE_HOSTS[$ns]}"
    echo "  Formatting $ns Active on $active_host..."
    ssh $active_host "
        mkdir -p /opt/hadoop/data/namenode
        hdfs namenode -format -clusterid $CLUSTER_ID -force 2>/dev/null
    "
    echo "  ✅ $ns Active formatted"
done

# ── STEP 4: Start Active NameNodes ──
echo ""
echo "[4/9] Starting Active NameNodes"
for ns in ns1 ns2 ns3; do
    active_host="${NS_ACTIVE_HOSTS[$ns]}"
    ssh $active_host "hdfs --daemon start namenode"
    echo "  ✅ $ns Active started on $active_host"
done
sleep 5

# ── STEP 5: Bootstrap and Start Standby NameNodes ──

   → Fencing: Guarantees old Active is dead before Standby promotes

✅ KEY INTERNAL MECHANICS:
   → Active writes every edit to JN quorum SYNCHRONOUSLY
   → Standby tails edits from JNs every 1-2 seconds
   → DataNodes report blocks to BOTH Active and Standby
   → Standby performs FSImage checkpointing (replaces SecondaryNameNode)
   → Client uses ConfiguredFailoverProxyProvider to auto-discover Active

✅ THE CRITICAL SAFETY RULE:
   → FENCING must succeed before Standby becomes Active
   → Without fencing = split-brain risk = metadata corruption = data loss
