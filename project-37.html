PROJECT 37: STREAMING MAPREDUCE WITH PYTHON — HADOOP STREAMING API

🔴 HYPOTHETICAL PROBLEM

SCENARIO:
→ You are the data engineering lead at a media analytics company
→ Your team of 8 data engineers: 6 know Python, 2 know Java
→ Current pipeline: Java MapReduce jobs processing 500GB of daily web logs
→ Business needs NEW analytics fast:
    - Sentiment analysis on user comments (Python NLTK library)
    - URL pattern extraction (Python regex expertise on team)
    - Session reconstruction from clickstream data
    - Geographic IP resolution (Python GeoIP2 library)

THE FRICTION:
→ Java developers are backlogged — 3-month queue for new MR jobs
→ Python team can write the LOGIC in 2 days
→ But they don't know Java, Hadoop API, Writable interfaces,
  InputFormat, OutputFormat, Driver configuration...
→ Rewriting every Python prototype in Java wastes months

THE QUESTION:
→ Can the Python team write MapReduce jobs WITHOUT learning Java?
→ Can they use Python libraries (NLTK, GeoIP2, pandas) inside Hadoop?
→ What's the performance cost compared to native Java MapReduce?
→ How does data flow between Hadoop (Java) and Python processes?

THE ANSWER: Hadoop Streaming API
→ Write mapper in Python (or Ruby, Perl, bash, C++, anything)
→ Write reducer in Python
→ Hadoop launches your scripts as separate processes
→ Data flows via stdin/stdout — the UNIX pipe model
→ Your code never imports a single Hadoop class

🔍 HOW TO APPROACH IT

STEP 1: Understand the stdin/stdout protocol

  THE STREAMING CONTRACT:

  ┌─────────────────────────────────────────────────────────────┐
  │                 STREAMING PROTOCOL                           │
  │                                                              │
  │  MAPPER:                                                     │
  │  → Receives: one line of input per line on stdin             │
  │  → Each line: the raw text from the input file               │
  │  → Must output: key\tvalue\n to stdout                       │
  │  → TAB character (\t) separates key from value               │
  │  → One output pair per line                                  │
  │  → Can output ZERO lines (filter) or MANY lines (explode)   │
  │                                                              │
  │  REDUCER:                                                    │
  │  → Receives: key\tvalue\n lines on stdin                     │
  │  → Lines are SORTED by key (Hadoop's shuffle did this)       │
  │  → All lines with SAME key are CONSECUTIVE                   │
  │  → Must output: key\tvalue\n to stdout                       │
  │  → Must detect key changes yourself (no grouping API)        │
  │                                                              │
  │  COMBINER:                                                   │
  │  → Same interface as reducer                                 │
  │  → Runs on map side (Project 9 concept)                      │
  │  → Must be idempotent (may run 0, 1, or multiple times)     │
  │                                                              │
  └─────────────────────────────────────────────────────────────┘

  DATA FLOW EXAMPLE:

  Input file line:     "2024-01-15 10:30:00 user_42 GET /product/shoes 200"
                                │
                                ▼ (piped to mapper stdin)
  Mapper reads:        "2024-01-15 10:30:00 user_42 GET /product/shoes 200\n"
  Mapper prints:       "/product/shoes\t1\n"
                                │
                                ▼ (Hadoop sorts by key)
  Sorted stream:       "/product/bags\t1\n"
                       "/product/bags\t1\n"
                       "/product/shoes\t1\n"
                       "/product/shoes\t1\n"
                       "/product/shoes\t1\n"
                                │
                                ▼ (piped to reducer stdin)
  Reducer reads lines, detects key changes:
    current_key = "/product/bags", count = 2
    → prints: "/product/bags\t2\n"
    current_key = "/product/shoes", count = 3
    → prints: "/product/shoes\t3\n"

STEP 2: Understand how Hadoop launches your scripts

  ┌─────────────────────────────────────────────────────────────┐
  │           HADOOP STREAMING EXECUTION MODEL                   │
  │                                                              │
  │  hadoop jar hadoop-streaming.jar                             │
  │    -input /data/logs/                                        │
  │    -output /data/output/                                     │
  │    -mapper mapper.py                                         │
  │    -reducer reducer.py                                       │
  │    -file mapper.py                                           │
  │    -file reducer.py                                          │
  │                                                              │
  │  WHAT HAPPENS:                                               │
  │                                                              │
  │  1. Hadoop ships mapper.py and reducer.py to ALL task nodes  │
  │     (-file flag copies files to distributed cache)           │
  │                                                              │
  │  2. For each input split, Hadoop launches a MAP TASK:        │
  │     → JVM process starts                                     │
  │     → JVM forks a child process: python mapper.py            │
  │     → JVM reads input split, writes each line to child stdin │
  │     → Child reads stdin, processes, writes to stdout         │
  │     → JVM reads child stdout, parses key\tvalue              │
  │     → JVM handles partitioning, sorting, spilling to disk    │
  │                                                              │
  │  3. Shuffle phase: identical to Java MapReduce               │
  │     → Sort by key                                            │
  │     → Transfer to reducer nodes                              │
  │                                                              │
  │  4. For each reduce partition, Hadoop launches REDUCE TASK:  │
  │     → JVM forks child process: python reducer.py             │
  │     → JVM feeds sorted key\tvalue lines to child stdin       │
  │     → Child reads, aggregates, writes to stdout              │
  │     → JVM reads stdout, writes to HDFS output                │
  │                                                              │
  └─────────────────────────────────────────────────────────────┘

✅ WHAT SHOULD DO

1. ALWAYS START WITH #!/usr/bin/env python3
   → Shebang line tells OS how to execute the script
   → Use env for portability across different Python installations
   → Make scripts executable: chmod +x mapper.py

2. USE sys.stdin FOR INPUT, sys.stdout FOR OUTPUT
   → Do NOT use input() — it adds prompts and buffering
   → sys.stdin iterates line by line efficiently
   → sys.stdout.write() or print() for output
   → print() adds newline automatically — preferred

3. HANDLE MALFORMED INPUT GRACEFULLY
   → Real data has corrupt lines, encoding errors, missing fields
   → Wrap processing in try/except
   → Skip bad lines — don't crash the entire task
   → Use stderr for error logging (goes to task log, not output)

4. STRIP NEWLINES FROM INPUT
   → sys.stdin includes trailing \n
   → Always: line.strip() before processing
   → Forgetting this: keys have invisible \n → wrong grouping

5. USE -file TO SHIP SCRIPTS AND DEPENDENCIES
   → -file mapper.py ships the mapper script
   → -file my_library.py ships helper modules
   → -file model.pkl ships machine learning models
   → -file config.json ships configuration files
   → All files appear in the task's working directory

6. TEST LOCALLY BEFORE SUBMITTING TO HADOOP
   → echo "test data" | python mapper.py | sort | python reducer.py
   → This simulates the ENTIRE MapReduce pipeline locally
   → Catches 90% of bugs without waiting for YARN job startup

7. SET -D stream.num.map.output.key.fields FOR COMPOSITE KEYS
   → Default: first field before TAB is the key
   → For composite keys: "user\ttime\tvalue"
   → Set key fields = 2 → key is "user\ttime", value is "value"

🚫 WHAT SHOULD NOT DO

1. DO NOT USE print() WITH EXTRA FORMATTING
   → print(f"key={key}, value={value}") → WRONG
   → Hadoop expects EXACTLY: key\tvalue\n
   → Any deviation → parsing fails → wrong results or crash

2. DO NOT ASSUME REDUCER RECEIVES GROUPED VALUES
   → Java reducer: reduce(key, Iterable<values>) — pre-grouped
   → Streaming reducer: receives FLAT sorted lines
   → YOU must detect when the key changes
   → Missing this → all values merged into one giant group

3. DO NOT FORGET TO FLUSH OUTPUT IN REDUCER
   → Reducer must emit the LAST group after stdin ends
   → Common bug: last key's aggregation never output
   → Always handle the final group after the for loop

4. DO NOT USE HEAVY IMPORTS AT MODULE LEVEL WITHOUT CACHING
   → import nltk at top → loaded ONCE per task (OK)
   → But: downloading NLTK data on every task → 100 downloads
   → Pre-download data, ship via -file or -archives

5. DO NOT IGNORE ENCODING ISSUES
   → HDFS files may contain non-UTF-8 bytes
   → Python 3 stdin defaults to system encoding
   → Set: PYTHONIOENCODING=utf-8 in environment
   → Or: open stdin in binary mode and decode manually

6. DO NOT WRITE BINARY OUTPUT
   → Streaming protocol is TEXT-based
   → Binary output → Hadoop cannot parse key\tvalue
   → For binary output needs: use Java MapReduce (Project 32)

7. DO NOT USE GLOBAL VARIABLES FOR STATE ACROSS LINES IN MAPPER
   → Each mapper processes ONE split independently
   → State within one mapper is fine (accumulating per split)
   → But: no state sharing BETWEEN mappers
   → Reducer handles cross-mapper aggregation

🚫 WHEN NOT TO USE

1. WHEN PROCESSING BINARY FILES
   → Streaming reads TEXT lines from stdin
   → Binary data (Project 32 sensor files) cannot be piped as text
   → WORKAROUND: Convert binary to text first (Project 36 pattern)
   → But this adds an extra step — Java InputFormat is cleaner

2. WHEN YOU NEED CUSTOM PARTITIONING
   → Streaming uses HashPartitioner by default
   → Custom partitioner (Project 10) requires Java
   → PARTIAL WORKAROUND: -D stream.num.map.output.key.fields
     controls which fields are the partition key
   → But: no custom hash function

3. WHEN PERFORMANCE IS CRITICAL
   → Streaming overhead: fork process, pipe data, parse text
   → Java MR: in-process, binary serialization, zero parsing
   → Streaming is typically 2-5x SLOWER than Java for same logic
   → For TB-scale production with tight SLAs: use Java

4. WHEN YOU NEED MULTIPLE OUTPUTS (MultipleOutputs)
   → Project 20's MultipleOutputs API is Java-only
   → Streaming writes to single stdout → single output
   → WORKAROUND: Encode output destination in key, post-process

5. WHEN REDUCER NEEDS COMPLEX STATE MANAGEMENT
   → Streaming reducer is a flat stream of sorted lines
   → No built-in grouping, no setup()/cleanup() lifecycle
   → Complex state machines are error-prone in streaming
   → Java reducer with proper lifecycle methods is safer

  │  │ 1. Open split      │               │ 1. Read stdin  │ │

  │  → JVM and Python are SEPARATE OS processes               │
  │  → Data crosses process boundary via OS pipes             │
  │  → Each line is serialized to text, piped, deserialized   │
  │  → This is the OVERHEAD of streaming                      │
  │                                                           │
  │  PERFORMANCE COST:                                        │
  │  → Per-line: ~1-5 microseconds for pipe + parse           │

  │  → Plus: Python is ~10x slower than Java for parsing      │
  │  → Net effect: 2-5x slower than native Java MR            │
  │                                                           │
  └──────────────────────────────────────────────────────────┘

MECHANIC 2: KEY PARSING FROM STDOUT
═══════════════════════════════════════════════════════════════

  Hadoop reads mapper stdout and must determine key vs value:

  DEFAULT BEHAVIOR:
    Line: "shoes\t42\n"
    Key:   "shoes"
    Value: "42"
    Split at FIRST tab character

  CONFIGURABLE:
    -D stream.map.output.field.separator=,
    → Line: "shoes,42\n" → Key: "shoes", Value: "42"

    -D stream.num.map.output.key.fields=2
    → Line: "shoes\tred\t42\n"
    → Key: "shoes\tred" (first 2 fields)
    → Value: "42" (remaining)

  IF NO TAB FOUND:
    → Entire line becomes the key
    → Value is empty string
    → This is valid but often a bug

MECHANIC 3: REDUCER INPUT GUARANTEE
═══════════════════════════════════════════════════════════════

  Hadoop guarantees reducer stdin receives lines SORTED by key:

  ┌──────────────────────────────────────────────────────────┐
  │  REDUCER STDIN (guaranteed sorted):                       │
  │                                                           │
  │  apple\t1         ← Group 1: "apple"                     │
  │  apple\t1         ←                                       │
  │  apple\t1         ←                                       │
  │  banana\t1        ← Group 2: "banana" (key changed!)     │
  │  banana\t1        ←                                       │
  │  cherry\t1        ← Group 3: "cherry"                    │
  │                                                           │
  │  YOUR REDUCER MUST:                                       │
  │  → Track current key                                      │
  │  → Accumulate values while key is same                    │
  │  → When key changes: emit result for previous key         │
  │  → After stdin ends: emit result for LAST key             │
  │                                                           │
  └──────────────────────────────────────────────────────────┘

  JAVA REDUCER EQUIVALENT:
    reduce("apple", [1, 1, 1]) → emit("apple", 3)
    reduce("banana", [1, 1])   → emit("banana", 2)
    reduce("cherry", [1])      → emit("cherry", 1)

  STREAMING REDUCER MUST DO THE GROUPING MANUALLY:
    current_key = None
    current_count = 0
    for line in stdin:
        key, value = line.split('\t')
        if key != current_key:
            if current_key is not None:
                print(f"{current_key}\t{current_count}")
            current_key = key
            current_count = 0
        current_count += int(value)
    # DON'T FORGET THE LAST GROUP!
    if current_key is not None:
        print(f"{current_key}\t{current_count}")

⚠️ COMMON PITFALLS AND FAILURE MODES

PITFALL 1: "Reducer produces wrong counts — always too low"
→ CAUSE: Forgot to emit the LAST key group after stdin ends
→ The for loop ends → last accumulated result never printed
→ SOLUTION: Always add final emit after the loop (shown in code)

PITFALL 2: "Task fails with: python: command not found"
→ CAUSE: Python not installed on DataNodes, or wrong path
→ SOLUTION: Use full path: /usr/bin/python3
→ Or: ship a virtual environment with -archives

PITFALL 3: "All output lines have key = entire line"
→ CAUSE: Mapper output doesn't contain a TAB character
→ Hadoop treats entire line as key, value is empty
→ SOLUTION: Ensure mapper prints exactly: key\tvalue\n
→ Debug: add to mapper: print(repr(output_line), file=sys.stderr)

PITFALL 4: "UnicodeDecodeError on certain input lines"
→ CAUSE: Input contains non-UTF-8 bytes
→ Python 3 sys.stdin defaults to system encoding
→ SOLUTION: Set environment variable:
  -D mapreduce.map.env="PYTHONIOENCODING=utf-8"
→ Or: read sys.stdin.buffer and decode with errors='replace'

PITFALL 5: "Job succeeds but output is empty"
→ CAUSE: Mapper script has a bug that crashes silently
→ Python exception goes to stderr (task log), not stdout
→ SOLUTION: Check task logs in YARN UI
→ Always test locally first: cat input.txt | python mapper.py

PITFALL 6: "Streaming job is 10x slower than expected"
→ CAUSE: Python startup + library import on every task
→ Each mapper launches a new Python process
→ import pandas takes 2-3 seconds per task
→ 1000 mappers × 3 seconds = 50 minutes of pure import time
→ SOLUTION: Minimize imports, use lightweight libraries
→ Or: increase split size to reduce number of mappers

   KEY INSIGHT: When processing logic is HEAVY (NLP, ML inference),
   Python can actually be FASTER because:
   → Python libraries (NLTK, scikit-learn) are C-optimized
   → The pipe overhead is negligible vs. processing time
   → Java equivalents may not exist or are less optimized

2. SCALING BOTTLENECK: PYTHON STARTUP TIME
   → Each task forks a Python process: ~0.5-2 seconds
   → With 10,000 mappers: 5,000-20,000 seconds of startup
   → SOLUTION: Increase split size → fewer, longer-running mappers
   → -D mapreduce.input.fileinputformat.split.minsize=536870912
     (512MB splits instead of 128MB → 4x fewer mappers)

3. MEMORY USAGE
   → Each Python process: ~50-200MB baseline
   → Plus your code + libraries
   → Container memory must accommodate JVM + Python process
   → Set: -D mapreduce.map.memory.mb=2048 (2GB per mapper)
   → JVM gets ~1GB, Python gets ~1GB

#!/usr/bin/env python3
# ═══════════════════════════════════════════════════════════════
# FILE 1: mapper_wordcount.py — BASIC STREAMING MAPPER
# ═══════════════════════════════════════════════════════════════
# Equivalent of Project 4's Java WordCount mapper.
# Reads lines from stdin, emits word\t1 to stdout.

"""
STREAMING PROTOCOL:
  INPUT:  One line of text per line on stdin
  OUTPUT: key\tvalue\n on stdout
  ERRORS: Print to stderr (goes to task log)
"""

import sys

def mapper():
    """
    Read input lines from stdin.
    For each word, emit: word\t1
    """
    for line in sys.stdin:

# ═══════════════════════════════════════════════════════════════
# FILE 2: reducer_wordcount.py — BASIC STREAMING REDUCER
# ═══════════════════════════════════════════════════════════════
# Equivalent of Project 4's Java WordCount reducer.
# Receives sorted key\tvalue lines, aggregates counts per key.

import sys

def reducer():
    """
    Read sorted key\tvalue lines from stdin.
    Sum values for each key group.
    Emit: key\ttotal_count
    
    CRITICAL: Must handle key changes manually.
    Hadoop guarantees sorted input, but does NOT group for you.
    """
    current_word = None
    current_count = 0
    
    for line in sys.stdin:
        line = line.strip()
        
        if not line:
            continue
        
        try:
            # Parse key\tvalue
            parts = line.split('\t', 1)  # Split on FIRST tab only
            
            if len(parts) != 2:
                print("reporter:counter:StreamingErrors,ReducerParseError,1",
                      file=sys.stderr)
                continue
            
            word = parts[0]
            count = int(parts[1])
            
            # ── Key change detection ──
            if word == current_word:
                # Same key: accumulate
                current_count += count
            else:
                # New key: emit previous group's result
                if current_word is not None:
                    print(f"{current_word}\t{current_count}")
                
                # Start new group
                current_word = word
                current_count = count
        
        except ValueError as e:
            print(f"ERROR: Cannot parse count: {line[:100]}", file=sys.stderr)
            print("reporter:counter:StreamingErrors,ReducerValueError,1",
                  file=sys.stderr)
    
    # ═══════════════════════════════════════════════
    # CRITICAL: Emit the LAST key group!
    # Without this, the last word's count is lost.
    # This is the #1 streaming reducer bug.
    # ═══════════════════════════════════════════════
    if current_word is not None:
        print(f"{current_word}\t{current_count}")

if __name__ == "__main__":
    reducer()

# FILE 3: mapper_log_analysis.py — WEB LOG PARSER
# ═══════════════════════════════════════════════════════════════
# Parses Apache/Nginx access logs using Python regex.
# Demonstrates: Python library advantage over Java.

import sys
import re
from datetime import datetime

# Compiled regex for Apache Combined Log Format
# Much easier in Python than Java Pattern/Matcher
LOG_PATTERN = re.compile(
    r'(\S+)\s+'           # IP address
    r'\S+\s+'             # ident
    r'\S+\s+'             # user
    r'\[([^\]]+)\]\s+'    # timestamp
    r'"(\S+)\s+'          # method
    r'(\S+)\s+'           # URL
    r'\S+"\s+'            # protocol
    r'(\d{3})\s+'         # status code
    r'(\d+|-)\s*'         # response size
    r'"([^"]*)"?\s*'      # referrer (optional)
    r'"([^"]*)"?'         # user agent (optional)
)

def parse_log_line(line):
    """Parse an Apache Combined Log Format line."""
    match = LOG_PATTERN.match(line)
    if not match:
        return None
    
    groups = match.groups()
    
    # Parse timestamp
    try:
        timestamp = datetime.strptime(
            groups[1].split()[0], '%d/%b/%Y:%H:%M:%S')
        hour = timestamp.strftime('%Y-%m-%d-%H')
    except (ValueError, IndexError):
        hour = 'unknown'
    
    return {
        'ip': groups[0],
        'timestamp': groups[1],
        'hour': hour,
        'method': groups[2],
        'url': groups[3],
        'status': int(groups[4]),
        'size': int(groups[5]) if groups[5] != '-' else 0,
        'referrer': groups[6] if len(groups) > 6 else '-',
        'user_agent': groups[7] if len(groups) > 7 else '-'
    }

def extract_page_category(url):
    """Extract page category from URL path."""
    if '/product/' in url:
        return 'product_page'
    elif '/category/' in url:
        return 'category_page'
    elif '/cart' in url:
        return 'cart'
    elif '/checkout' in url:
        return 'checkout'
    elif '/search' in url:
        return 'search'
    elif url == '/' or url == '/index.html':
        return 'homepage'
    elif '/api/' in url:
        return 'api_call'
    else:
        return 'other'

def mapper():
    """
    Emit multiple analysis dimensions per log line:
    1. hourly_traffic: hour\tpage_views,bytes,errors
    2. url_category: category\tcount
    3. status_code: status\tcount
    """
    for line in sys.stdin:
        line = line.strip()
        if not line:
            continue
        
        try:
            parsed = parse_log_line(line)
            if parsed is None:
                print("reporter:counter:LogAnalysis,UnparsableLines,1",

                continue
            
            print("reporter:counter:LogAnalysis,ParsedLines,1",

            is_error = 1 if parsed['status'] >= 400 else 0

            print(f"STATUS|{parsed['status']}\t1")
            
        except Exception as e:
            print(f"ERROR: {e}", file=sys.stderr)
            print("reporter:counter:LogAnalysis,MapperErrors,1",
                  file=sys.stderr)

# FILE 4: reducer_log_analysis.py — MULTI-DIMENSION AGGREGATION
# ═══════════════════════════════════════════════════════════════
# Handles multiple key prefixes from the log analysis mapper.
# Demonstrates complex streaming reducer logic.

import sys

def reducer():
    """
    Process sorted lines with prefixed keys:
    HOURLY|2024-01-15-10\t1,bytes,errors
    CATEGORY|product_page\t1
    STATUS|200\t1
    
    Aggregate differently based on prefix.
    """
    current_key = None
    
    # Aggregation accumulators
    count = 0
    total_bytes = 0
    total_errors = 0
    
    for line in sys.stdin:
        line = line.strip()
        if not line:
            continue
        
        try:
            parts = line.split('\t', 1)
            if len(parts) != 2:
                continue
            
            key = parts[0]
            value = parts[1]
            
            if key != current_key:
                # ── Emit previous group ──
                if current_key is not None:
                    emit_result(current_key, count, total_bytes, total_errors)
                
                # ── Reset for new group ──
                current_key = key
                count = 0
                total_bytes = 0
                total_errors = 0
            
            # ── Accumulate based on key prefix ──
            if key.startswith('HOURLY|'):
                # Value format: count,bytes,errors
                vals = value.split(',')
                count += int(vals[0])
                total_bytes += int(vals[1]) if len(vals) > 1 else 0
                total_errors += int(vals[2]) if len(vals) > 2 else 0
            else:
                # CATEGORY and STATUS: simple count
                count += int(value)
        
        except Exception as e:
            print(f"ERROR: {e}", file=sys.stderr)
    
    # ── Emit last group ──
    if current_key is not None:
        emit_result(current_key, count, total_bytes, total_errors)

def emit_result(key, count, total_bytes, total_errors):
    """Format output based on key type."""
    if key.startswith('HOURLY|'):
        hour = key.split('|', 1)[1]
        avg_size = total_bytes // max(1, count)
        error_rate = round(total_errors / max(1, count) * 100, 2)
        print(f"HOURLY\t{hour}\t{count}\t{total_bytes}\t{avg_size}\t{total_errors}\t{error_rate}%")
    elif key.startswith('CATEGORY|'):
        category = key.split('|', 1)[1]
        print(f"CATEGORY\t{category}\t{count}")
    elif key.startswith('STATUS|'):
        status = key.split('|', 1)[1]
        print(f"STATUS\t{status}\t{count}")

POSITIVE_WORDS = set([
    'good', 'great', 'excellent', 'amazing', 'wonderful', 'fantastic',
    'love', 'best', 'happy', 'perfect', 'beautiful', 'awesome',
    'recommend', 'satisfied', 'quality', 'fast', 'easy'
])
NEGATIVE_WORDS = set([
    'bad', 'terrible', 'horrible', 'awful', 'worst', 'hate',
    'poor', 'slow', 'broken', 'useless', 'disappointed', 'waste',
    'never', 'refund', 'complaint', 'defective', 'scam'
])

def simple_sentiment(text):
    """Fallback sentiment when NLTK not available."""
    words = text.lower().split()
    pos = sum(1 for w in words if w in POSITIVE_WORDS)
    neg = sum(1 for w in words if w in NEGATIVE_WORDS)
    total = pos + neg
    if total == 0:
        return 0.0, 'neutral'
    score = (pos - neg) / total
    if score > 0.1:
        return score, 'positive'
    elif score < -0.1:
        return score, 'negative'
    else:
        return score, 'neutral'

def nltk_sentiment(text, analyzer):
    """NLTK VADER sentiment analysis."""
    scores = analyzer.polarity_scores(text)
    compound = scores['compound']
    if compound >= 0.05:
        return compound, 'positive'
    elif compound <= -0.05:
        return compound, 'negative'
    else:
        return compound, 'neutral'

import sys

def reducer():
    """
    Aggregate sentiment per product:
    - Count positive/negative/neutral
    - Average sentiment score
    - Average rating
    - Overall sentiment label
    """
    current_product = None
    pos_count = 0
    neg_count = 0
    neu_count = 0
    total_score = 0.0
    total_rating = 0.0
    total_count = 0
    
    for line in sys.stdin:
        line = line.strip()
        if not line:
            continue
        
        try:
            parts = line.split('\t', 1)
            if len(parts) != 2:
                continue
            
            product_id = parts[0]
            value_parts = parts[1].split(',')
            
            if len(value_parts) < 3:
                continue
            
            label = value_parts[0]
            score = float(value_parts[1])
            rating = float(value_parts[2])
            
            # ── Key change detection ──
            if product_id != current_product:
                if current_product is not None:
                    emit_product_sentiment(
                        current_product, pos_count, neg_count, neu_count,
                        total_score, total_rating, total_count)
                
                current_product = product_id
                pos_count = neg_count = neu_count = 0
                total_score = total_rating = 0.0
                total_count = 0
            
            # ── Accumulate ──
            if label == 'positive':
                pos_count += 1
            elif label == 'negative':
                neg_count += 1
            else:
                neu_count += 1
            
            total_score += score
            total_rating += rating
            total_count += 1
        
        except Exception as e:
            print(f"ERROR: {e}", file=sys.stderr)
    
    # ── Last product ──
    if current_product is not None:
        emit_product_sentiment(
            current_product, pos_count, neg_count, neu_count,
            total_score, total_rating, total_count)

def emit_product_sentiment(product, pos, neg, neu, total_score, total_rating, count):
    """Emit aggregated sentiment for one product."""
    if count == 0:
        return
    
    avg_score = total_score / count
    avg_rating = total_rating / count
    
    # Determine overall sentiment
    if pos > neg and pos > neu:
        overall = 'POSITIVE'
    elif neg > pos and neg > neu:
        overall = 'NEGATIVE'
    else:
        overall = 'MIXED'
    
    print(f"{product}\t{overall}\t{count}\t{pos}\t{neg}\t{neu}\t"
          f"{avg_score:.4f}\t{avg_rating:.1f}")

    Reduces: mapper emits 1000 × "the\t1"
    To:      combiner emits 1 × "the\t1000"
    Savings: 999 fewer records in shuffle.
    """
    current_word = None
    current_count = 0
    
    for line in sys.stdin:
        line = line.strip()
        if not line:
            continue
        
        try:
            word, count_str = line.split('\t', 1)
            count = int(count_str)
            
            if word == current_word:
                current_count += count
            else:
                if current_word is not None:
                    print(f"{current_word}\t{current_count}")
                current_word = word
                current_count = count
        except (ValueError, IndexError):
            continue
    
    if current_word is not None:
        print(f"{current_word}\t{current_count}")

if __name__ == "__main__":
    combiner()

# ═══════════════════════════════════════════════════════════════
# Demonstrates: composite key + secondary sort in streaming.
# Groups clickstream events by user, sorted by timestamp.
#
# Key trick: emit user_id\ttimestamp\taction
# Configure Hadoop to partition by user_id only,
# but sort by user_id + timestamp.

import sys
import json

def mapper():
    """
    Input: JSON clickstream events
    Output: user_id\ttimestamp\taction,url,duration
    
    The composite key (user_id\ttimestamp) ensures:
    → All events for same user go to same reducer (partition by user_id)
    → Events arrive sorted by timestamp (sort by full key)
    """
    for line in sys.stdin:
        line = line.strip()
        if not line:
            continue
        
        try:
            event = json.loads(line)
            
            user_id = event.get('user_id', 'unknown')
            timestamp = event.get('timestamp', '0')
            action = event.get('action', 'unknown')
            url = event.get('url', '-')
            duration = event.get('duration_sec', 0)
            
            # Composite key: user_id + timestamp
            # Hadoop will sort by both, partition by user_id only
            # (configured via stream.num.map.output.key.fields)
            print(f"{user_id}\t{timestamp}\t{action},{url},{duration}")
            
        except json.JSONDecodeError:
            print("reporter:counter:Session,InvalidJSON,1", file=sys.stderr)

if __name__ == "__main__":
    mapper()
#!/usr/bin/env python3
# ═══════════════════════════════════════════════════════════════
# FILE 10 (CONTINUED): reducer_session.py
# ═══════════════════════════════════════════════════════════════

import sys

SESSION_TIMEOUT_SEC = 1800  # 30 minutes

def reducer():
    """
    Input: sorted by user_id, then timestamp
    Detect sessions based on time gaps.
    Emit: user_id \t session_id \t start \t end \t page_count \t total_duration \t pages
    """
    current_user = None
    session_id = 0
    session_start = None
    session_end = None
    session_pages = []
    last_timestamp = 0
    
    for line in sys.stdin:
        line = line.strip()
        if not line:
            continue
        
        try:
            parts = line.split('\t')
            if len(parts) < 3:
                continue
            
            user_id = parts[0]
            timestamp = int(parts[1])
            value_parts = parts[2].split(',')
            action = value_parts[0] if len(value_parts) > 0 else 'unknown'
            url = value_parts[1] if len(value_parts) > 1 else '-'
            duration = int(value_parts[2]) if len(value_parts) > 2 else 0
            
            # ── User changed: emit all pending sessions ──
            if user_id != current_user:
                if current_user is not None:
                    emit_session(current_user, session_id,
                                session_start, session_end, session_pages)
                
                # Reset for new user
                current_user = user_id
                session_id = 1
                session_start = timestamp
                session_end = timestamp
                session_pages = [(action, url, duration)]
                last_timestamp = timestamp
                continue
            
            # ── Same user: check for session boundary ──
            time_gap = timestamp - last_timestamp
            
            if time_gap > SESSION_TIMEOUT_SEC:
                # Gap exceeds threshold → emit current session, start new one
                emit_session(current_user, session_id,
                            session_start, session_end, session_pages)
                
                session_id += 1
                session_start = timestamp
                session_end = timestamp
                session_pages = [(action, url, duration)]
            else:
                # Same session: extend it
                session_end = timestamp
                session_pages.append((action, url, duration))
            
            last_timestamp = timestamp
        
        except Exception as e:
            print(f"ERROR: {e}", file=sys.stderr)
            print("reporter:counter:Session,ReducerErrors,1", file=sys.stderr)
    
    # ── Emit last user's last session ──
    if current_user is not None:
        emit_session(current_user, session_id,
                    session_start, session_end, session_pages)


def emit_session(user_id, session_id, start_ts, end_ts, pages):
    """Emit a complete session summary."""
    if not pages:
        return
    
    page_count = len(pages)
    total_duration = end_ts - start_ts
    
    # Build page flow string: "homepage→product→cart→checkout"
    page_flow = '→'.join(p[1].split('/')[-1] or 'home' for p in pages)
    
    # Detect conversion: did user reach checkout?
    urls = [p[1] for p in pages]
    converted = any('/checkout' in u for u in urls)
    
    # Detect bounce: only 1 page viewed
    is_bounce = page_count == 1
    
    # Session classification
    if converted:
        session_type = 'CONVERSION'
    elif is_bounce:
        session_type = 'BOUNCE'
    elif page_count >= 5:
        session_type = 'ENGAGED'
    else:
        session_type = 'BROWSE'
    
    print(f"{user_id}\t{session_id}\t{start_ts}\t{end_ts}\t"
          f"{page_count}\t{total_duration}\t{session_type}\t"
          f"{converted}\t{page_flow}")
    
    print("reporter:counter:Session,TotalSessions,1", file=sys.stderr)
    print(f"reporter:counter:Session,{session_type},1", file=sys.stderr)


if __name__ == "__main__":
    reducer()
