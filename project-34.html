PROJECT 34: BUILDING A STAR SCHEMA DATA WAREHOUSE ON HIVE WITH ORC

🔴 HYPOTHETICAL PROBLEM

SCENARIO:
→ You are the data warehouse architect at a multinational retail company
→ Business analysts need to answer questions like:
    - "What was total revenue by product category per region last quarter?"
    - "Which stores had declining sales month-over-month?"
    - "What is the average basket size by customer segment on weekends?"
    - "Top 10 products by revenue in stores where footfall dropped 20%?"
→ Current state: ALL data sits in flat CSV files on HDFS
→ Analysts write MapReduce jobs for EVERY query — takes 2 days per report

THE FLAT FILE NIGHTMARE:

THE QUESTION:
→ How do you design a proper dimensional data warehouse on Hive
  that transforms 45-minute queries into 30-second queries
  using star schema, ORC, bloom filters, vectorized execution,
  and the cost-based optimizer?

STEP 2: Understand WHY star schema is faster

  QUERY: "Total revenue by product category for Q1 2024"
  
  FLAT FILE APPROACH:
  → Scan ALL 2TB of data
  → Parse every field of every row
  → Filter date strings that start with "2024-01" through "2024-03"
  → Group by product_category column
  → TIME: ~45 minutes
  
  STAR SCHEMA ON ORC APPROACH:
  → Fact table PARTITIONED by year/month
    → Only read partitions 2024-01, 2024-02, 2024-03 (~150GB vs 2TB)
  → ORC COLUMNAR storage
    → Only read product_key and total_amount columns (~15GB vs 150GB)
  → ORC BLOOM FILTER on product_key
    → Skip stripes that don't contain relevant product keys
  → JOIN with dim_product is tiny (10,000 products)
    → Map-side join via broadcast (Project 16 pattern)
  → VECTORIZED execution
    → Process 1024 rows at a time instead of one-by-one
  → TIME: ~25 seconds

  SPEEDUP BREAKDOWN:
  ┌─────────────────────────────────┬───────────┬──────────────┐
  │ OPTIMIZATION                    │ DATA READ │ CUMULATIVE   │
  ├─────────────────────────────────┼───────────┼──────────────┤
  │ Full scan (baseline)            │ 2,000 GB  │ 2,000 GB     │
  │ + Partition pruning (Q1 only)   │ 150 GB    │ 150 GB       │

✅ WHAT SHOULD DO

1. USE SURROGATE KEYS IN DIMENSION TABLES
   → Natural keys (product SKU, store code) can change
   → Surrogate keys (auto-increment integers) are stable
   → Fact table references surrogate keys → joins are on INTs (fast)
   → Natural keys stored as attributes in dimension table

2. DESIGN DATE DIMENSION WITH PRE-COMPUTED ATTRIBUTES
   → Don't derive "is_weekend" at query time from date string
   → Pre-compute: year, quarter, month, week, day_of_week, is_weekend, is_holiday
   → Queries become simple WHERE clauses instead of date functions

3. PARTITION FACT TABLES BY DATE (Project 25 concept)
   → Most queries filter by date range
   → Partition by year/month: fact_sales/year=2024/month=01/
   → Partition pruning skips irrelevant months entirely

4. BUCKET FACT TABLES BY JOIN KEYS
   → CLUSTERED BY (product_key) INTO 32 BUCKETS
   → Enables bucket map join (same buckets on both sides)
   → Reduces shuffle for large-to-large joins

5. ENABLE ALL ORC OPTIMIZATIONS
   → Bloom filters on join columns
   → Stripe-level min/max statistics
   → SNAPPY compression (fast decode)
   → Row-level index every 10,000 rows

6. USE COST-BASED OPTIMIZER (CBO)
   → Analyze tables: ANALYZE TABLE fact_sales COMPUTE STATISTICS
   → CBO chooses optimal join order, join strategy
   → Can turn 10-minute query into 30-second query automatically

7. ENABLE VECTORIZED EXECUTION
   → Process 1024 rows in a batch instead of row-by-row
   → 2-10x speedup on scan-heavy queries
   → Works with ORC and Parquet natively

🚫 WHAT SHOULD NOT DO

1. DO NOT DENORMALIZE EVERYTHING INTO ONE GIANT TABLE
   → "Just put everything in one table" seems simpler
   → But: updating a store name requires rewriting ALL transactions
   → And: columnar storage is LESS effective with 50+ columns
   → Star schema balances normalization and query performance

2. DO NOT USE TEXT FORMAT FOR WAREHOUSE TABLES
   → Text: no compression, no column pruning, no predicate pushdown
   → ORC gives 4-10x compression + 10-100x query speedup
   → Project 30 established this — apply it here

3. DO NOT SKIP STATISTICS COLLECTION
   → Without ANALYZE TABLE, CBO has no information
   → Falls back to rule-based optimizer → suboptimal plans
   → A 5-minute ANALYZE can save hours of query time daily

4. DO NOT OVER-PARTITION
   → Partition by day = 365 partitions/year = manageable
   → Partition by day AND store AND product = millions of partitions
   → Too many partitions → NameNode metadata explosion (Project 1 pitfall)
   → RULE: < 10,000 total partitions per table

5. DO NOT FORGET TO SET TBLPROPERTIES FOR ORC
   → Default ORC settings are conservative
   → Explicitly set bloom filters, compression, stripe size
   → Small differences in settings → large performance differences

6. DO NOT USE INSERT INTO FOR IDEMPOTENT LOADS
   → INSERT INTO appends → re-running creates duplicates
   → INSERT OVERWRITE PARTITION replaces → safe to re-run
   → Critical for pipeline reliability (Project 31 lesson)

🚫 WHEN NOT TO USE

1. WHEN YOU NEED REAL-TIME UPDATES
   → Hive is designed for batch analytics, not OLTP
   → Even with ACID: transaction throughput is low
   → Need real-time inserts? → HBase (Project 27) or Kafka
   → Feed Hive warehouse from HBase via batch export

2. WHEN QUERIES ARE POINT LOOKUPS
   → "Get order #12345" → Hive scans entire partition
   → Use HBase for random access (Project 27)
   → Hive is for ANALYTICAL queries (aggregations, joins, scans)

3. WHEN DATA VOLUME IS SMALL
   → < 10GB → PostgreSQL or DuckDB is 100x simpler
   → Hive overhead (metastore, YARN, JVM) not justified
   → Star schema design is universal — works on any database

4. WHEN SCHEMA IS COMPLETELY UNKNOWN
   → Star schema requires understanding business entities
   → If you don't know what facts and dimensions exist
   → Start with raw zone exploration FIRST, then design schema

5. WHEN SUB-SECOND QUERY LATENCY IS REQUIRED
   → Hive queries take seconds to minutes minimum
   → Even with all optimizations: 5-30 seconds for complex queries
   → Need < 1 second? → Druid, ClickHouse, or pre-aggregated HBase

MECHANIC 1: ORC FILE INTERNAL STRUCTURE
═══════════════════════════════════════════════════════════════

  From Project 30, we know ORC is columnar.
  Now we go DEEPER into its optimization structures:

  ┌───────────────────────────────────────────────────────────┐
  │                   ORC FILE ANATOMY                         │
  │                                                            │
  │  ┌──────────────────────────────────────────────────────┐ │
  │  │ STRIPE 1 (default 64MB — configurable)               │ │
  │  │ ┌──────────────────────────────────────────────────┐ │ │
  │  │ │ STREAM: Column 1 (product_key) data              │ │ │
  │  │ │ STREAM: Column 2 (total_amount) data             │ │ │
  │  │ │ STREAM: Column 3 (quantity) data                 │ │ │
  │  │ │ ...                                              │ │ │
  │  │ ├──────────────────────────────────────────────────┤ │ │
  │  │ │ ROW INDEX (every 10,000 rows):                   │ │ │
  │  │ │   product_key: min=1001, max=5999                │ │ │
  │  │ │   total_amount: min=0.50, max=9999.99            │ │ │
  │  │ │   quantity: min=1, max=100                       │ │ │
  │  │ ├──────────────────────────────────────────────────┤ │ │
  │  │ │ BLOOM FILTER INDEX:                              │ │ │
  │  │ │   product_key: bloom[1001,1002,...,5999]          │ │ │
  │  │ │   → "Is product_key=3456 POSSIBLY in this stripe?"│ │ │
  │  │ │   → YES: read stripe                             │ │ │
  │  │ │   → NO: SKIP stripe entirely                     │ │ │
  │  │ └──────────────────────────────────────────────────┘ │ │
  │  ├──────────────────────────────────────────────────────┤ │
  │  │ STRIPE 2 (another 64MB)                              │ │
  │  │   ... same structure ...                             │ │
  │  ├──────────────────────────────────────────────────────┤ │
  │  │ STRIPE N                                             │ │
  │  ├──────────────────────────────────────────────────────┤ │
  │  │ FILE FOOTER:                                         │ │
  │  │   Schema definition                                  │ │
  │  │   Number of rows                                     │ │
  │  │   Statistics per column (min, max, sum, count, nulls)│ │
  │  │   Stripe locations and sizes                         │ │
  │  └──────────────────────────────────────────────────────┘ │
  └───────────────────────────────────────────────────────────┘

  QUERY: WHERE product_key = 3456

  Step 1: Read FILE FOOTER → Check file-level statistics
    → product_key range: 1001 to 50000
    → 3456 is within range → file MIGHT contain it

  Step 2: Read STRIPE FOOTER for each stripe
    → Stripe 1: product_key min=1001, max=5999 → 3456 IN RANGE
    → Stripe 2: product_key min=6000, max=12000 → 3456 NOT IN RANGE → SKIP
    → Stripe 3: product_key min=12001, max=18000 → SKIP

  Step 3: Check BLOOM FILTER for Stripe 1
    → bloom.test(3456) → TRUE (might exist) → READ STRIPE 1
    → bloom.test(9999) → FALSE (definitely not here) → SKIP

  Step 4: Read ROW INDEX within Stripe 1
    → Row group 0 (rows 0-9999): product_key min=1001, max=2500 → SKIP
    → Row group 1 (rows 10000-19999): product_key min=2501, max=4000 → READ
    → Row group 2 (rows 20000-29999): product_key min=4001, max=5999 → SKIP

  RESULT: Read only ~10,000 rows out of potentially millions

MECHANIC 2: VECTORIZED QUERY EXECUTION
═══════════════════════════════════════════════════════════════

  TRADITIONAL (Row-at-a-time):
  ┌───────────────────────────────────────────────────────────┐
  │ for each row:                                              │
  │   read row from ORC                                        │
  │   deserialize row into Java objects                        │
  │   evaluate WHERE predicate                                 │
  │   if matches:                                              │
  │     compute aggregate                                      │
  │     serialize result                                       │
  │                                                            │
  │ OVERHEAD PER ROW:                                          │
  │   → Virtual method dispatch for type checking              │
  │   → Object creation and garbage collection                 │
  │   → Branching for null checks                              │
  │   → CPU cache misses (objects scattered in heap)           │
  │                                                            │
  │ For 1 billion rows: 1 billion iterations of overhead       │
  └───────────────────────────────────────────────────────────┘

  VECTORIZED (Batch of 1024 rows):
  ┌───────────────────────────────────────────────────────────┐
  │ for each batch of 1024 rows:                               │
  │   read column vectors from ORC:                            │
  │     long[] product_keys = [1001, 1002, ..., 2024]          │
  │     double[] amounts = [49.99, 12.50, ..., 89.99]          │
  │     boolean[] nulls = [false, false, ..., false]           │
  │                                                            │
  │   evaluate WHERE on entire vector:                         │
  │     for i in 0..1023:                                      │
  │       selected[i] = (product_keys[i] == 3456)              │
  │     // This is a tight loop on primitive arrays            │
  │     // CPU can SIMD optimize and prefetch cache lines      │
  │                                                            │
  │   compute aggregate on selected rows:                      │
  │     for i in 0..1023:                                      │
  │       if selected[i]: sum += amounts[i]                    │
  │                                                            │
  │ ADVANTAGE:                                                 │
  │   → No object creation (primitive arrays)                  │
  │   → No virtual dispatch (direct array access)              │
  │   → CPU cache-friendly (contiguous memory)                 │
  │   → SIMD-compatible tight loops                            │
  │   → 2-10x faster than row-at-a-time                       │
  └───────────────────────────────────────────────────────────┘

MECHANIC 3: COST-BASED OPTIMIZER (CBO)
═══════════════════════════════════════════════════════════════

  WITHOUT CBO (Rule-Based):
    Query: SELECT ... FROM fact_sales f
           JOIN dim_product p ON f.product_key = p.product_key
           JOIN dim_store s ON f.store_key = s.store_key
           WHERE s.region = 'WEST'
    
    Rule-based: Joins in ORDER WRITTEN
    → Join fact_sales (2B rows) with dim_product (10K rows) → 2B rows
    → Join result (2B rows) with dim_store (500 rows) → THEN filter
    → WASTE: Processed 2B rows before filtering by region

  WITH CBO (Statistics-Aware):
    CBO knows:
    → fact_sales: 2,000,000,000 rows
    → dim_product: 10,000 rows  
    → dim_store: 500 rows, region='WEST' selects 125 stores
    → fact_sales partitioned: Q1 2024 = 150,000,000 rows
    
    CBO decides:
    1. Filter dim_store WHERE region='WEST' → 125 rows
    2. Broadcast join dim_store (125 rows) with fact_sales → eliminates 75% of facts
    3. Then join result with dim_product (10K rows)
    → MUCH less data processed

  CBO STATISTICS INCLUDE:
  ┌─────────────────────────────────────┐
  │ TABLE-LEVEL:                        │
  │   → Number of rows                  │
  │   → Total size in bytes             │
  │   → Number of files                 │
  │                                     │
  │ COLUMN-LEVEL:                       │
  │   → Number of distinct values (NDV) │
  │   → Min / Max values                │
  │   → Average length (for strings)    │
  │   → Number of nulls                 │
  │   → Histogram (value distribution)  │
  └─────────────────────────────────────┘

MECHANIC 4: ACID TRANSACTIONS ON ORC (Hive 3+)
═══════════════════════════════════════════════════════════════

  TRADITIONAL HIVE: Write-once, append-only
  → Cannot UPDATE or DELETE individual rows
  → Changing a dimension attribute requires full table rewrite
  
  ACID HIVE (on ORC):
  → Supports INSERT, UPDATE, DELETE, MERGE
  → Uses delta files for modifications:
  
  ┌───────────────────────────────────────────────────────────┐
  │ BASE FILE (original data):                                 │
  │   base_0000001/bucket_00000                                │
  │   Contains: original rows                                  │
  │                                                            │
  │ DELTA FILES (modifications):                               │
  │   delta_0000002_0000002/bucket_00000                       │
  │   Contains: INSERT operations (new rows)                   │
  │                                                            │
  │   delete_delta_0000003_0000003/bucket_00000                │
  │   Contains: DELETE operations (row IDs to remove)          │

  │ ON COMPACTION: Merge base + all deltas into new base       │

⚠️ COMMON PITFALLS AND FAILURE MODES

PITFALL 1: "Bloom filter didn't speed up my query"
→ CAUSE: Bloom filter only helps for equality predicates (= or IN)
→ Range predicates (BETWEEN, >, <) use min/max statistics instead
→ Bloom filter on low-cardinality column (e.g., region with 4 values) is useless
→ BEST FOR: High-cardinality join columns (product_key, customer_key)

PITFALL 2: "CBO chose a terrible plan"
→ CAUSE: Statistics are stale or never collected
→ ANALYZE TABLE must be run after significant data changes
→ Without statistics: CBO estimates row counts as 0 → wrong decisions
→ SOLUTION: Schedule ANALYZE TABLE after each major load

PITFALL 3: "ACID table queries are slow"
→ CAUSE: Too many small delta files from frequent inserts
→ Each query must merge base + all deltas on read
→ 1000 delta files = 1000 file opens per query
→ SOLUTION: Run compaction regularly
→ ALTER TABLE fact_sales COMPACT 'major'

PITFALL 4: "Vectorization not working despite being enabled"
→ CAUSE: Unsupported data types or expressions
→ Complex UDFs, DECIMAL type in some versions, nested types
→ CHECK: EXPLAIN VECTORIZATION query shows if vectorized
→ SOLUTION: Simplify expressions, use supported types

PITFALL 5: "Partition pruning not happening"
→ CAUSE: Predicate uses function on partition column
→ WHERE YEAR(sale_date) = 2024 → NO pruning (function applied)
→ WHERE year = 2024 → YES pruning (direct partition column)
→ SOLUTION: Design partition columns to match query patterns

PITFALL 6: "Join produces wrong results after bucketing"
→ CAUSE: Bucket count mismatch between fact and dimension
→ Bucket map join requires SAME number of buckets or multiple
→ fact: 32 buckets, dim: 16 buckets → OK (32 is multiple of 16)
→ fact: 32 buckets, dim: 10 buckets → NOT compatible
→ SOLUTION: Choose bucket counts that are powers of 2

📈 SCALABILITY CONSIDERATIONS

1. FACT TABLE GROWTH MANAGEMENT
   ┌───────────────────────────┬──────────────────────────────┐
   │ DATA VOLUME               │ PARTITION STRATEGY            │
   ├───────────────────────────┼──────────────────────────────┤
   │ < 100GB/year              │ Partition by year             │

   │                           │ + bucket by join key (64-128) │
   └───────────────────────────┴──────────────────────────────┘

2. DIMENSION TABLE SIZE LIMITS FOR MAP-SIDE JOIN
   → Map-side join (Project 16): dimension must fit in memory
   → Default threshold: 25MB (hive.auto.convert.join.noconditionaltask.size)
   → dim_product (10K rows) → ~2MB → easily fits
   → dim_customer (50M rows) → ~5GB → too large for broadcast
   → SOLUTION: Filter dimension first, or use bucket join

3. ORC STRIPE SIZE TUNING
   → Default: 64MB per stripe
   → Smaller stripes → better predicate pushdown granularity
   → Larger stripes → less overhead, better compression
   → RECOMMENDATION: 64MB for most cases, 256MB for scan-heavy workloads

4. STATISTICS COLLECTION COST
   → Full table ANALYZE on 2B rows: ~30 minutes
   → Column-level with histograms: ~60 minutes
   → Schedule during off-peak hours
   → For partitioned tables: ANALYZE TABLE ... PARTITION(year=2024, month=01)
   → Only analyze new/changed partitions

SELECT
    CAST(date_format(d, 'yyyyMMdd') AS INT)     AS date_key,
    d                                            AS full_date,
    YEAR(d)                                      AS year,
    QUARTER(d)                                   AS quarter,
    MONTH(d)                                     AS month,

    weekofyear(d)                                AS week_of_year,
    DAY(d)                                       AS day_of_month,
    -- 1=Monday through 7=Sunday
    CASE dayofweek(d)
        WHEN 1 THEN 7  -- Sunday → 7
        ELSE dayofweek(d) - 1
    END                                          AS day_of_week,
    date_format(d, 'EEEE')                      AS day_name,
    CASE WHEN dayofweek(d) IN (1, 7) THEN true ELSE false END AS is_weekend,
    false                                        AS is_holiday,
    NULL                                         AS holiday_name,
    -- Fiscal year: starts April 1
    CASE WHEN MONTH(d) >= 4 THEN YEAR(d) ELSE YEAR(d) - 1 END AS fiscal_year,
    CASE
        WHEN MONTH(d) IN (4,5,6) THEN 1
        WHEN MONTH(d) IN (7,8,9) THEN 2
        WHEN MONTH(d) IN (10,11,12) THEN 3
        ELSE 4
    END                                          AS fiscal_quarter
FROM (
    SELECT date_add('2015-01-01', pos) AS d
    FROM (
        SELECT posexplode(split(space(7304), ' ')) AS (pos, val)
        -- 7305 days = 20 years (2015-2034)
    ) numbers
) dates;

        t.quantity * t.unit_price * (1 - t.discount_pct / 100) 
        - t.quantity * p.unit_cost, 2
    ) AS profit,
    ROUND(t.quantity * t.unit_price * (1 - t.discount_pct / 100) * 0.08, 2) AS tax_amount,
    
    -- Transaction metadata
    t.payment_method,
    CAST(t.transaction_time AS TIMESTAMP),
    
    -- Partition columns (MUST be last in SELECT for dynamic partitioning)
    YEAR(CAST(t.sale_date AS DATE))  AS year,
    MONTH(CAST(t.sale_date AS DATE)) AS month

FROM stg_transactions t

-- Join with date dimension
JOIN dim_date d
    ON d.full_date = CAST(t.sale_date AS DATE)

-- Join with store dimension (current version only — SCD2)
JOIN dim_store s
    ON s.store_id = t.store_id
    AND s.is_current = true

-- Join with product dimension (current version only)
JOIN dim_product p
    ON p.product_id = t.product_id
    AND p.is_current = true

-- Join with customer dimension (current version only)
JOIN dim_customer c
    ON c.customer_id = t.customer_id
    AND c.is_current = true

-- Data quality: skip invalid records
WHERE t.transaction_id IS NOT NULL
  AND t.quantity > 0
  AND t.unit_price > 0
  AND t.store_id IS NOT NULL
  AND t.product_id IS NOT NULL;

-- ══════════════════════════════════════════════════════════
-- QUERY 2: Month-over-month revenue trend by region
-- OPTIMIZATIONS USED:
--   → Uses pre-aggregated agg_daily_sales (small table)
--   → No fact table scan needed at all!
--   → Window function for MoM comparison
-- ══════════════════════════════════════════════════════════

SELECT
    region,
    year,
    month,
    monthly_revenue,
    prev_month_revenue,
    ROUND(
        (monthly_revenue - prev_month_revenue) / 
        NULLIF(prev_month_revenue, 0) * 100, 2
    ) AS mom_growth_pct
FROM (
    SELECT
        region,
        year,
        month,
        SUM(total_revenue) AS monthly_revenue,
        LAG(SUM(total_revenue)) OVER (
            PARTITION BY region 
            ORDER BY year, month
        ) AS prev_month_revenue
    FROM agg_daily_sales
    WHERE year IN (2023, 2024)
    GROUP BY region, year, month
) trend
ORDER BY region, year, month;


-- ══════════════════════════════════════════════════════════
-- QUERY 3: Top 10 products with declining sales
-- OPTIMIZATIONS USED:
--   → Pre-aggregate comparison (current vs previous month)
--   → CBO chooses optimal join order
--   → Partition pruning for both months
-- ══════════════════════════════════════════════════════════

WITH current_month AS (
    SELECT product_key, SUM(total_amount) AS current_revenue
    FROM fact_sales
    WHERE year = 2024 AND month = 1
    GROUP BY product_key
),
previous_month AS (
    SELECT product_key, SUM(total_amount) AS previous_revenue
    FROM fact_sales
    WHERE year = 2023 AND month = 12
    GROUP BY product_key
)
SELECT
    p.product_name,
    p.category,
    p.brand,
    cm.current_revenue,
    pm.previous_revenue,
    ROUND(
        (cm.current_revenue - pm.previous_revenue) / 
        NULLIF(pm.previous_revenue, 0) * 100, 2
    ) AS revenue_change_pct
FROM current_month cm
JOIN previous_month pm ON cm.product_key = pm.product_key
JOIN dim_product p ON cm.product_key = p.product_key AND p.is_current = true
WHERE cm.current_revenue < pm.previous_revenue
ORDER BY revenue_change_pct ASC
LIMIT 10;

-- QUERY 4: Weekend vs. Weekday performance by store type
-- OPTIMIZATIONS USED:
--   → Date dimension pre-computed is_weekend (no function call)
--   → Multiple dimension joins — CBO optimizes order
--   → Bloom filters on all foreign keys
-- ══════════════════════════════════════════════════════════

SELECT
    st.store_type,
    d.is_weekend,
    COUNT(DISTINCT f.transaction_id) AS transactions,
    SUM(f.total_amount) AS revenue,
    ROUND(AVG(f.total_amount), 2) AS avg_transaction_value,
    COUNT(DISTINCT f.customer_key) AS unique_customers,
    ROUND(
        SUM(f.total_amount) / COUNT(DISTINCT f.customer_key), 2
    ) AS revenue_per_customer
FROM fact_sales f
JOIN dim_date d ON f.date_key = d.date_key
JOIN dim_store st ON f.store_key = st.store_key AND st.is_current = true
WHERE f.year = 2024 AND f.month = 1
GROUP BY st.store_type, d.is_weekend
ORDER BY st.store_type, d.is_weekend;

-- QUERY 5: Customer segmentation analysis
-- OPTIMIZATIONS USED:
--   → Uses pre-aggregated agg_monthly_customer
--   → Avoids scanning billion-row fact table
--   → Simple aggregation on small summary table
-- ══════════════════════════════════════════════════════════

    COUNT(*) AS customer_count,

FROM agg_monthly_customer
WHERE year = 2024 AND month = 1
GROUP BY segment
ORDER BY avg_monthly_spend DESC;
