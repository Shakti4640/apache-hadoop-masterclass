═══════════════════════════════════════════════════════════
TUNING AREA 1: MAP-SIDE SORT BUFFER (THE BIGGEST WIN)
═══════════════════════════════════════════════════════════

HOW MAP OUTPUT WORKS INTERNALLY:

  Map function emits (key, value) pairs
  ↓
  Pairs go into an IN-MEMORY CIRCULAR BUFFER
  (controlled by mapreduce.task.io.sort.mb — default 100MB)
  ↓
  When buffer is X% full (mapreduce.map.sort.spill.percent — default 0.80)
  ↓
  SPILL: Sort the buffer contents by key, write to LOCAL DISK
  ↓
  Continue filling buffer while spill writes to disk (double-buffering)
  ↓
  When map task completes, MERGE all spill files into ONE sorted file
  ↓
  This merged file is the map output — served to reducers during shuffle

  ┌───────────────────────────────────────────────────────────┐
  │  MAP TASK MEMORY LAYOUT                                    │
  │                                                            │
  │  ┌─────────────────────────────────────────────┐          │
  │  │         SORT BUFFER (io.sort.mb)            │          │
  │  │                                              │          │
  │  │  ┌─────────┐ ┌─────────┐ ┌─────────┐       │          │
  │  │  │ Record 1│ │ Record 2│ │ Record 3│ ...   │          │
  │  │  │ key+val │ │ key+val │ │ key+val │       │          │
  │  │  └─────────┘ └─────────┘ └─────────┘       │          │
  │  │                                              │          │
  │  │  ┌──────────────────────────────────────┐   │          │
  │  │  │ ACCOUNTING SPACE (io.sort.record.pct)│   │          │
  │  │  │ 16 bytes per record (key len, val len│   │          │
  │  │  │ partition, position)                  │   │          │
  │  │  └──────────────────────────────────────┘   │          │
  │  └─────────────────────────────────────────────┘          │
  │                                                            │
  │  When 80% full → SPILL to disk                            │
  │                                                            │
  │  SPILL FILE 1: sorted (key1, val1), (key2, val2)...      │
  │  SPILL FILE 2: sorted (key5, val5), (key6, val6)...      │
  │  SPILL FILE 3: sorted (key9, val9), (key10, val10)...    │
  │                                                            │
  │  FINAL MERGE: merge-sort all spill files into ONE file    │
  │  (controlled by mapreduce.task.io.sort.factor — default 10│
  │   = merge at most 10 files at once)                        │
  └───────────────────────────────────────────────────────────┘

WHY THIS MATTERS AT PETABYTE SCALE:

  Current settings (defaults):
    io.sort.mb = 100MB
    Map output per task ≈ 500MB (for our clickstream data)
    Spills per map task = 500MB / (100MB × 0.80) = ~6 spills
    Total map tasks = 2.8PB / 128MB = ~22,000 map tasks
    Total spills = 22,000 × 6 = 132,000 spill operations
    Each spill = disk write + disk read (during merge)
    Total unnecessary disk I/O: ENORMOUS

  Optimized settings:
    io.sort.mb = 512MB
    Map output per task ≈ 500MB
    Spills per map task = 500MB / (512MB × 0.80) = ~1 spill (or zero!)
    Total spills = 22,000 × 1 = 22,000 (6× FEWER)
    MASSIVE reduction in disk I/O

  RULE OF THUMB:
    io.sort.mb should be large enough that MOST map tasks
    produce only 1 spill (ideally zero spills)
    
    Formula: io.sort.mb ≈ (average map output bytes per task) / 0.80
    
    For our case: 500MB / 0.80 ≈ 625MB → set to 640MB
    
    BUT: io.sort.mb comes from the MAP TASK HEAP
    If map task heap = 1024MB and io.sort.mb = 640MB
    Only 384MB left for map function logic — may cause OOM
    
    SOLUTION: Increase map task memory:
    mapreduce.map.memory.mb = 2048
    mapreduce.map.java.opts = -Xmx1536m
    io.sort.mb = 768

TUNING AREA 2: SHUFFLE OPTIMIZATION
═══════════════════════════════════════════════════════════

THE SHUFFLE IS THE MOST EXPENSIVE PHASE AT SCALE

For our 2.8PB job:
  → Map output: 2.1PB
  → After combiner (Project 9): 1.9PB
  → This 1.9PB must travel across the NETWORK from map nodes to reduce nodes
  → 3,000 DataNodes × 10Gbps network = theoretical 3.75TB/s
  → 1.9PB / 3.75TB/s = 8.4 minutes theoretical minimum
  → ACTUAL: 45+ minutes due to disk I/O bottlenecks

SHUFFLE TUNING PARAMETERS:

  1. REDUCE SLOWSTART (mapreduce.job.reduce.slowstart.completedmaps)
     → Default: 0.05 (reducers start shuffling when 5% of maps complete)
     → For large jobs: set to 0.80
     → WHY: Starting reducers too early wastes container resources
       They sit idle waiting for map outputs
     → At 0.80: reducers start when 80% of maps are done
       Most map outputs are ready → shuffle completes faster

  2. SHUFFLE PARALLELISM (mapreduce.reduce.shuffle.parallelcopies)
     → Default: 5 (each reducer fetches from 5 mappers simultaneously)
     → For large clusters: set to 20-50
     → More parallel copies = faster shuffle
     → BUT: too many = network congestion
     → SWEET SPOT: 20-30 for clusters > 1000 nodes

  3. SHUFFLE MEMORY (mapreduce.reduce.shuffle.input.buffer.percent)
     → Default: 0.70 (70% of reduce heap used for shuffle buffer)
     → For shuffle-heavy jobs: increase to 0.80-0.90
     → More memory for shuffle = fewer disk spills on reduce side
     → BUT: leaves less memory for reduce function

  4. SHUFFLE MERGE THRESHOLD (mapreduce.reduce.shuffle.merge.percent)
     → Default: 0.66 (merge starts when 66% of shuffle buffer is full)
     → For large jobs: increase to 0.80
     → Delays merge → more data in memory → fewer merge passes

TUNING AREA 3: INTERMEDIATE COMPRESSION (HUGE WIN)
═══════════════════════════════════════════════════════════

MAP OUTPUT COMPRESSION:
→ Compress data BETWEEN map and reduce (during shuffle)
→ Reduces network transfer by 2-5× (depending on data and codec)
→ Trades CPU for network bandwidth — almost always a good trade

  Without compression:
    Map output: 2.1PB → shuffled as 2.1PB over network
    
  With Snappy compression:
    Map output: 2.1PB → compressed to ~0.7PB → shuffled as 0.7PB
    SAVINGS: 1.4PB less network transfer
    COST: ~5% more CPU on map tasks (Snappy is very fast)
    NET EFFECT: 2-3× faster shuffle

  CODEC SELECTION:
  ┌─────────────┬────────────┬────────────────┬─────────────────┐
  │ Codec       │ Speed      │ Compression    │ Best For         │
  ├─────────────┼────────────┼────────────────┼─────────────────┤
  │ Snappy      │ Very fast  │ Moderate (2-3×)│ Shuffle (DEFAULT)│
  │ LZ4         │ Very fast  │ Moderate (2-3×)│ Shuffle          │
  │ Zstd        │ Fast       │ Good (3-5×)    │ Shuffle if CPU   │
  │             │            │                │ is not bottleneck│
  │ Gzip        │ Slow       │ Best (5-8×)    │ NEVER for shuffle│
  │             │            │                │ (too slow)       │
  └─────────────┴────────────┴────────────────┴─────────────────┘

TUNING AREA 4: JVM REUSE
═══════════════════════════════════════════════════════════

PROBLEM:
→ Each map/reduce task launches a NEW JVM
→ JVM startup: 2-5 seconds
→ 22,000 map tasks × 3 seconds = 18 HOURS of cumulative JVM startup
→ (Parallelized, but still significant overhead per task slot)

SOLUTION — JVM REUSE:
→ mapreduce.job.jvm.numtasks = -1 (unlimited reuse)
→ One JVM runs MULTIPLE sequential tasks
→ JVM starts once, processes task 1, then task 2, then task 3...
→ Eliminates startup overhead for tasks 2, 3, 4...

  WITHOUT JVM REUSE (default):
    [JVM start 3s] [Task 1: 60s] [JVM exit]
    [JVM start 3s] [Task 2: 60s] [JVM exit]
    [JVM start 3s] [Task 3: 60s] [JVM exit]
    Total: 189 seconds for 3 tasks (5% overhead)

  WITH JVM REUSE:
    [JVM start 3s] [Task 1: 60s] [Task 2: 60s] [Task 3: 60s] [JVM exit]
    Total: 183 seconds for 3 tasks (1.6% overhead)

  AT SCALE (22,000 tasks, 3,000 task slots):
    Without reuse: ~22,000 JVM starts
    With reuse:    ~3,000 JVM starts (one per slot)
    Savings: ~19,000 × 3 seconds = 15.8 HOURS of cumulative JVM time

  CAVEAT:
  → JVM reuse can cause memory leaks if map/reduce code leaks
  → Stale state from previous task may affect next task
  → Use with well-tested code only

═══════════════════════════════════════════════════════════
TUNING AREA 5: UBER TASKS (SMALL JOB OPTIMIZATION)
═══════════════════════════════════════════════════════════

FOR SMALL JOBS ONLY (not our 2.8PB job):

PROBLEM:
→ Small job: 5 map tasks, 1 reduce task, finishes in 30 seconds
→ YARN overhead: request containers (5s), launch AM (3s), 
  allocate containers (5s), launch tasks (3s)
→ Overhead: 16 seconds for a 30-second job = 35% overhead

SOLUTION — UBER MODE:
→ Run the ENTIRE job inside the ApplicationMaster container
→ No separate task containers needed
→ Eliminates container allocation overhead completely

  mapreduce.job.ubertask.enable = true
  mapreduce.job.ubertask.maxmaps = 9
  mapreduce.job.ubertask.maxreduces = 1
  mapreduce.job.ubertask.maxbytes = 134217728 (128MB)

  IF job has ≤ 9 maps AND ≤ 1 reduce AND input ≤ 128MB:
    → Run as uber task (inside AM)
    → 2-3× faster for small jobs

  FOR OUR 2.8PB JOB:
    → 22,000 map tasks → NOT eligible for uber mode
    → Uber mode helps the THOUSANDS of small ad-hoc queries
      that run alongside the big ETL pipeline

TUNING AREA 6: SPECULATIVE EXECUTION
═══════════════════════════════════════════════════════════

PROBLEM — THE STRAGGLER EFFECT:
→ 22,000 map tasks complete in 4 minutes average
→ BUT: 50 tasks take 25 minutes (slow disk, network issue, GC)
→ Job cannot finish until ALL tasks complete
→ 50 stragglers extend job by 21 minutes (25 - 4)

SOLUTION — SPECULATIVE EXECUTION:
→ Hadoop detects tasks running significantly slower than average
→ Launches BACKUP COPIES of slow tasks on different nodes
→ Whichever copy finishes first wins — other is killed
→ Eliminates stragglers at the cost of some extra resource usage

  mapreduce.map.speculative = true
  mapreduce.reduce.speculative = true

  WHEN IT HELPS:
  → Heterogeneous hardware (some nodes faster than others)
  → Intermittent disk/network issues on specific nodes
  → GC pauses on specific JVMs

  WHEN IT HURTS:
  → Homogeneous cluster with consistent performance
  → Reduce tasks that write to external systems (duplicate writes!)
  → Resource-constrained cluster (backup tasks consume resources)

  FOR OUR JOB:
  → Enable for map tasks (safe — map tasks are idempotent)
  → DISABLE for reduce tasks in Stage 4 (join with external data)
  → Enable for reduce tasks in Stages 1, 2, 3 (safe)

✅ WHAT SHOULD DO

1. PROFILE BEFORE TUNING — USE JOB COUNTERS
   → Run baseline job with default settings
   → Record these counters:
     - SPILLED_RECORDS (high = increase io.sort.mb)
     - COMBINE_INPUT_RECORDS vs COMBINE_OUTPUT_RECORDS (combiner effectiveness)
     - SHUFFLED_MAPS, SHUFFLE_BYTES (shuffle volume)
     - GC_TIME_MILLIS (GC overhead)
     - MAP_OUTPUT_BYTES vs MAP_OUTPUT_MATERIALIZED_BYTES (compression ratio)
   → These counters TELL YOU what to tune

2. TUNE ONE PARAMETER AT A TIME
   → Change io.sort.mb → run → measure → record improvement
   → Then change compression → run → measure → record
   → NEVER change 5 things at once — you won't know what helped

3. USE SNAPPY FOR INTERMEDIATE COMPRESSION
   → Fastest codec with decent compression
   → Native library required (hadoop checknative)
   → Verify: hadoop checknative -a | grep snappy

4. MONITOR DISK I/O DURING JOBS
   → Run iostat on DataNodes during MapReduce execution
   → If disk utilization > 90%: io.sort.mb is too small (too many spills)
   → If network utilization > 80%: enable shuffle compression

5. SET REDUCE TASKS BASED ON DATA VOLUME
   → Rule of thumb: 1 reduce task per 1-2GB of reduce input
   → For 1.9PB shuffle data: ~1,000-2,000 reduce tasks
   → Too few reducers: each processes too much data (slow)
   → Too many reducers: overhead of scheduling + small output files
   → Formula: num_reducers = min(total_shuffle_bytes / 1GB, total_reduce_slots × 0.95)

6. ENABLE TIMELINE SERVER FOR DETAILED PROFILING
   → Provides per-task breakdown: shuffle time, sort time, reduce time
   → Identifies individual straggler tasks
   → Shows container allocation delays

🚫 WHAT SHOULD NOT DO

1. DO NOT SET io.sort.mb LARGER THAN 60% OF MAP TASK HEAP
   → io.sort.mb = 768MB but map heap = 1024MB → OOM guaranteed
   → RULE: io.sort.mb < (mapreduce.map.java.opts Xmx × 0.60)
   → Leave room for map function logic, framework overhead, GC

2. DO NOT ENABLE SPECULATIVE EXECUTION FOR NON-IDEMPOTENT TASKS
   → If reduce task writes to external database/HBase
   → Speculative copy = duplicate writes = data corruption
   → ONLY enable for tasks with HDFS-only output

3. DO NOT USE GZIP FOR INTERMEDIATE COMPRESSION
   → Gzip compression ratio is great (5-8×)
   → BUT: Gzip is 10× slower than Snappy
   → The CPU cost exceeds the network savings
   → Gzip for FINAL output (storage savings) — Snappy for shuffle

4. DO NOT SET reduce.slowstart TOO HIGH ON SMALL CLUSTERS
   → slowstart = 0.95 on a 10-node cluster
   → Reducers wait until 95% of maps complete
   → If one map is slow → ALL reducers wait
   → For small clusters: 0.50-0.70 is better
   → For large clusters (1000+ nodes): 0.80-0.90

5. DO NOT FORGET TO SIZE REDUCE TASK MEMORY
   → Reducers receive ALL values for a key
   → If one key has billions of values → reduce task OOM
   → This is data SKEW (Project 33)
   → Set reduce memory higher than map memory:
     mapreduce.reduce.memory.mb = 4096 (vs map 2048)
     mapreduce.reduce.java.opts = -Xmx3072m

6. DO NOT OVER-PARALLELIZE SHUFFLE
   → shuffle.parallelcopies = 100 on a congested network
   → Creates 100 HTTP connections per reducer
   → Network switch overwhelmed → retries → slower than fewer copies
   → MEASURE network utilization before increasing beyond 30

🚫 WHEN NOT TO USE

WHEN TUNING MAPREDUCE IS THE WRONG ANSWER:

1. ALGORITHM IS WRONG
   → No amount of tuning fixes O(n²) when O(n log n) exists
   → Profile the MAP/REDUCE functions first
   → If map function does 10 regex matches per record when 1 suffices
   → Fix the CODE, not the configuration

2. DATA SKEW IS THE ROOT CAUSE
   → One reduce task processes 80% of the data
   → Tuning shuffle won't help — that one task is the bottleneck
   → Use salting/custom partitioner (Project 33)

3. JOB SHOULD BE SPARK INSTEAD OF MAPREDUCE
   → Iterative algorithms (ML training)
   → Interactive queries
   → DAG-based pipelines with many stages
   → We address this in Project 48

4. CLUSTER IS GENUINELY UNDERSIZED
   → 100 nodes trying to process 2.8PB
   → No tuning overcomes physics
   → Need more hardware or less data (pre-filter)

5. I/O FORMAT IS THE BOTTLENECK
   → Processing 2.8PB of raw JSON
   → JSON parsing is inherently slow
   → Convert to Parquet/ORC first (Project 30)
   → Columnar format + predicate pushdown = 10× faster reads

INTERNAL MECHANIC 1: SORT-SPILL-MERGE IN DETAIL
═══════════════════════════════════════════════════════════

When a map task emits a record via context.write(key, value):

  Step 1: SERIALIZE
    → Key and value are serialized to bytes
    → Writable.write() called on both key and value
    → Serialized bytes stored in sort buffer

  Step 2: PARTITION
    → Partitioner.getPartition(key, value, numReducers) called
    → Returns partition number (0 to numReducers-1)
    → Stored in accounting metadata alongside the record

  Step 3: BUFFER
    → Serialized bytes written to circular buffer (kvbuffer)
    → Accounting metadata written to separate section (kvmeta)
    → kvbuffer layout:
      
      ┌──────────────────────────────────────────────────┐
      │                  kvbuffer                         │
      │  (size = mapreduce.task.io.sort.mb × 1024 × 1024)│
      │                                                   │
      │  ┌─────────────────────────────────┐              │
      │  │  DATA SECTION (1 - record.pct)  │              │
      │  │  Serialized key+value bytes     │              │
      │  │  Grows →                        │              │
      │  └─────────────────────────────────┘              │
      │                                                   │
      │  ┌─────────────────────────────────┐              │
      │  │  META SECTION (record.pct)      │              │
      │  │  16 bytes per record:           │              │
      │  │  [partition|keyStart|valStart|   │              │
      │  │   valLength]                    │              │
      │  │  ← Grows                        │              │
      │  └─────────────────────────────────┘              │
      └──────────────────────────────────────────────────┘
      
      io.sort.record.percent (default 0.05):
      → 5% of sort buffer reserved for metadata
      → If records are small (short keys): increase to 0.10-0.15
      → If records are large (long values): keep at 0.05

  Step 4: SPILL CHECK
    → After each write, check:
      used_bytes / total_bytes > spill.percent (default 0.80)?
    → If YES: trigger SPILL

  Step 5: SPILL (runs in background thread)
    → QuickSort the buffer contents by (partition, key)
    → For each partition:
      → If Combiner exists: run Combiner on sorted records
        (reduces data volume BEFORE writing to disk)
      → If map output compression enabled: compress
      → Write sorted, optionally combined+compressed data to:
        ${mapred.local.dir}/taskTracker/jobcache/<jobid>/<taskid>/output/spill<N>.out
    → Create spill index file: partition offsets within spill file

  Step 6: MERGE (after map task completes all input)
    → If only 1 spill file: rename to final output (no merge needed)
    → If multiple spill files: merge-sort them
    → io.sort.factor (default 10): max files to merge simultaneously
    → If 30 spill files and factor=10:
      Pass 1: merge files 1-10 → temp1
      Pass 2: merge files 11-20 → temp2
      Pass 3: merge files 21-30 → temp3
      Pass 4: merge temp1+temp2+temp3 → final output
    → OPTIMIZATION: increase io.sort.factor to 100
      → Merges all 30 files in ONE pass
      → Fewer disk reads/writes

  Step 7: MAP OUTPUT FILE
    → Final merged file on local disk
    → Partitioned and sorted within each partition
    → Ready for reducers to fetch via HTTP

INTERNAL MECHANIC 3: SPECULATIVE EXECUTION ALGORITHM
═══════════════════════════════════════════════════════════

How Hadoop detects stragglers:

  1. Track progress of ALL tasks of the same type (map or reduce)
  2. Calculate AVERAGE progress rate across all tasks
  3. For each running task, estimate completion time:
     estimated_completion = elapsed_time / progress
  4. Calculate average estimated completion for ALL tasks
  5. If a task's estimated completion > 
     average_completion × SPECULATION_MULTIPLIER:
     → Mark as STRAGGLER
     → Launch backup copy on a different node
  6. First copy to complete wins → other copy killed

  PROGRESS CALCULATION:
    Map task progress:
      progress = bytes_read / total_input_bytes
      
    Reduce task progress (three phases):
      shuffle_phase:  0.00 to 0.33 (proportional to shuffled data)
      sort_phase:     0.33 to 0.66 (proportional to merge progress)
      reduce_phase:   0.66 to 1.00 (proportional to processed keys)

  SPECULATION THRESHOLD:
    Default multiplier: 1.5× (task must be 50% slower than average)
    Configurable: mapreduce.job.speculative.speculative-cap-running-tasks
    → Limits how many speculative tasks can run simultaneously
    → Default: 0.1 (at most 10% of tasks can be speculative)

⚠️ COMMON PITFALLS AND FAILURE MODES

PITFALL 1: OOM AFTER INCREASING io.sort.mb
→ SYMPTOM: Map tasks fail with OutOfMemoryError
→ CAUSE: io.sort.mb + map function memory > task heap
→ FIX: Increase map task memory proportionally
→ FORMULA:
   mapreduce.map.java.opts Xmx ≥ io.sort.mb + 512MB (for framework + user code)
   mapreduce.map.memory.mb ≥ Xmx × 1.3 (YARN container > JVM heap)
→ Example: io.sort.mb=768 → Xmx=1280m → memory.mb=1664

PITFALL 2: SHUFFLE FETCH FAILURES AT SCALE
→ SYMPTOM: "Too many fetch-failures" → reduce task fails
→ CAUSE: Map output served via HTTP; under heavy load, connections timeout
→ FIX:
   mapreduce.reduce.shuffle.max-fetch-retries = 10 (default 3)
   mapreduce.reduce.shuffle.fetch-retry-timeout-ms = 30000
   mapreduce.reduce.shuffle.connect.timeout = 180000 (3 min)
→ ALSO: Ensure ShuffleHandler has enough threads:
   mapreduce.shuffle.max.threads = 0 (auto-detect based on cores)

PITFALL 3: REDUCE SIDE OOM DURING MERGE
→ SYMPTOM: Reduce task OOM during shuffle/merge phase
→ CAUSE: Too much shuffle data held in memory
→ FIX: Decrease shuffle.input.buffer.percent (0.70 → 0.50)
→ OR: Increase reduce task memory
→ TRADEOFF: Less in-memory merge = more disk I/O = slower merge

PITFALL 4: SPECULATIVE TASKS CONSUMING ALL RESOURCES
→ SYMPTOM: Cluster at 100% utilization, pending tasks pile up
→ CAUSE: Speculative execution launched too many backup tasks
→ FIX: mapreduce.job.speculative.speculative-cap-running-tasks = 0.05
→ Limits speculative tasks to 5% of total tasks
→ OR: Disable speculation if cluster is resource-constrained

PITFALL 5: JVM REUSE CAUSES MEMORY LEAKS
→ SYMPTOM: Tasks get slower over time within same JVM
→ CAUSE: User code allocates memory that isn't freed between tasks
→ FIX: Set jvm.numtasks = 10-20 (not unlimited)
→ JVM recycled after N tasks — limits leak accumulation
→ OR: Fix the memory leak in user code (proper cleanup in close())

PITFALL 6: COMPRESSION NATIVE LIBRARY NOT LOADED
→ SYMPTOM: Compression "enabled" but no compression actually happening
→ CAUSE: Snappy/LZ4 native library not installed
→ CHECK: hadoop checknative -a
→ Expected: snappy: true, lz4: true
→ If false: install native libraries and set:
   java.library.path = $HADOOP_HOME/lib/native

NETWORK BANDWIDTH PLANNING:

  Shuffle volume = map_output_bytes × (1 - combiner_reduction) × (1 - compression_ratio)

    Map output: 2.1PB
    Combiner reduces by 10%: 2.1PB × 0.90 = 1.89PB
    Snappy compression 3×: 1.89PB / 3 = 0.63PB
    
    Network needed: 0.63PB in 30 minutes (target shuffle time)
    = 0.63 × 1024 × 1024 GB / 1800 seconds
    = 367 GB/s aggregate cluster bandwidth
    
    With 3000 nodes × 10Gbps each = 3750 GB/s available
    Utilization: 367 / 3750 = 9.8% — well within capacity
    
    WITHOUT compression:
    Network needed: 1.89PB / 1800s = 1100 GB/s
    Utilization: 1100 / 3750 = 29.3% — still OK but 3× higher

/**
 * Clickstream Sessionizer — Groups user events into sessions.
 * 
 * A session = sequence of events from one user with < 30 min gap.
 * 
 * Input: tab-separated (userId, timestamp, eventType, pageUrl, ...)
 * Output: tab-separated (userId, sessionId, sessionStart, sessionEnd, 
 *         eventCount, pageViews, clicks)
 * 
 * PERFORMANCE DESIGN DECISIONS:
 * 1. Combiner NOT used — cannot combine across session boundaries
 * 2. Secondary sort by timestamp within each userId (Project 19)
 * 3. Custom Writable for compact serialization
 * 4. Map output designed for maximum compression ratio
 */
public class ClickstreamSessionizer extends Configured implements Tool {

    // ── Custom Writable for compact event representation ──
    public static class EventWritable implements WritableComparable<EventWritable> {
        private long timestamp;
        private byte eventType;  // 0=pageview, 1=click, 2=scroll, 3=video, 4=ad
        private Text pageUrl = new Text();

        public EventWritable() {}

        public EventWritable(long timestamp, byte eventType, String pageUrl) {
            this.timestamp = timestamp;
            this.eventType = eventType;
            this.pageUrl.set(pageUrl);
        }

        @Override
        public void write(java.io.DataOutput out) throws IOException {
            out.writeLong(timestamp);
            out.writeByte(eventType);
            pageUrl.write(out);
        }

        @Override
        public void readFields(java.io.DataInput in) throws IOException {
            timestamp = in.readLong();
            eventType = in.readByte();
            pageUrl.readFields(in);
        }

        @Override
        public int compareTo(EventWritable other) {
            return Long.compare(this.timestamp, other.timestamp);
        }

        public long getTimestamp() { return timestamp; }
        public byte getEventType() { return eventType; }
        public String getPageUrl() { return pageUrl.toString(); }
    }

    // ── MAPPER ──
    // Parse raw events, emit (userId, event)
    // Designed for SPEED — minimal object creation
    public static class SessionMapper 
            extends Mapper<LongWritable, Text, Text, EventWritable> {

        private final Text userId = new Text();
        private final EventWritable event = new EventWritable();

        // Reuse objects to minimize GC pressure
        // (Critical at 2.8PB scale — billions of map calls)

        @Override
        protected void map(LongWritable key, Text value, Context context)
                throws IOException, InterruptedException {
            
            String line = value.toString();
            
            // Fast tab-split (avoid String.split which creates array)
            int tab1 = line.indexOf('\t');
            if (tab1 < 0) return;  // malformed line
            
            int tab2 = line.indexOf('\t', tab1 + 1);
            if (tab2 < 0) return;
            
            int tab3 = line.indexOf('\t', tab2 + 1);
            if (tab3 < 0) return;
            
            // Extract fields without creating substring objects
            userId.set(line.substring(0, tab1));
            
            long timestamp;
            try {
                timestamp = Long.parseLong(line.substring(tab1 + 1, tab2));
            } catch (NumberFormatException e) {
                context.getCounter("PARSE_ERRORS", "BAD_TIMESTAMP").increment(1);
                return;
            }
            
            String eventTypeStr = line.substring(tab2 + 1, tab3);
            byte eventType = parseEventType(eventTypeStr);
            
            String pageUrl = (tab3 + 1 < line.length()) 
                ? line.substring(tab3 + 1) 
                : "";
            
            event.timestamp = timestamp;
            event.eventType = eventType;
            event.pageUrl.set(pageUrl);
            
            context.write(userId, event);
        }

        private byte parseEventType(String type) {
            switch (type) {
                case "pageview": return 0;
                case "click":   return 1;
                case "scroll":  return 2;
                case "video":   return 3;
                case "ad":      return 4;
                default:        return -1;
            }
        }
    }

    // ── REDUCER ──
    // Sessionize events for each user
    // Events arrive sorted by timestamp (secondary sort)
    public static class SessionReducer
            extends Reducer<Text, EventWritable, Text, Text> {

        private static final long SESSION_TIMEOUT_MS = 30 * 60 * 1000; // 30 minutes
        private final Text outputKey = new Text();
        private final Text outputValue = new Text();
        private final StringBuilder sb = new StringBuilder(256);

        @Override
        protected void reduce(Text userId, Iterable<EventWritable> events, 
                Context context) throws IOException, InterruptedException {
            
            // Collect events (they arrive sorted by timestamp)
            long sessionStart = -1;
            long lastTimestamp = -1;
            int sessionId = 0;
            int eventCount = 0;
            int pageViews = 0;
            int clicks = 0;

            for (EventWritable event : events) {
                long ts = event.getTimestamp();
                
                // New session detection
                if (lastTimestamp == -1 || (ts - lastTimestamp) > SESSION_TIMEOUT_MS) {
                    // Emit previous session (if exists)
                    if (sessionStart > 0) {
                        emitSession(context, userId.toString(), sessionId, 
                                   sessionStart, lastTimestamp, 
                                   eventCount, pageViews, clicks);
                    }
                    
                    // Start new session
                    sessionId++;
                    sessionStart = ts;
                    eventCount = 0;
                    pageViews = 0;
                    clicks = 0;
                }
                
                // Accumulate session metrics
                eventCount++;
                if (event.getEventType() == 0) pageViews++;
                if (event.getEventType() == 1) clicks++;
                lastTimestamp = ts;
            }
            
            // Emit final session
            if (sessionStart > 0) {
                emitSession(context, userId.toString(), sessionId, 
                           sessionStart, lastTimestamp, 
                           eventCount, pageViews, clicks);
            }
        }

        private void emitSession(Context context, String userId, int sessionId,
                long sessionStart, long sessionEnd, 
                int eventCount, int pageViews, int clicks) 
                throws IOException, InterruptedException {
            
            outputKey.set(userId);
            
            sb.setLength(0);
            sb.append(sessionId).append('\t')
              .append(sessionStart).append('\t')
              .append(sessionEnd).append('\t')
              .append(sessionEnd - sessionStart).append('\t')  // duration
              .append(eventCount).append('\t')
              .append(pageViews).append('\t')
              .append(clicks);
            
            outputValue.set(sb.toString());
            context.write(outputKey, outputValue);
        }
    }

    @Override
    public int run(String[] args) throws Exception {
        if (args.length != 2) {
            System.err.println("Usage: ClickstreamSessionizer <input> <output>");
            return 1;
        }

        Configuration conf = getConf();
        Job job = Job.getInstance(conf, "Clickstream Sessionizer — Stage 2");
        job.setJarByClass(ClickstreamSessionizer.class);

        // Mapper
        job.setMapperClass(SessionMapper.class);
        job.setMapOutputKeyClass(Text.class);
        job.setMapOutputValueClass(EventWritable.class);

        // Reducer
        job.setReducerClass(SessionReducer.class);
        job.setOutputKeyClass(Text.class);
        job.setOutputValueClass(Text.class);

        // NO COMBINER — cannot combine across session boundaries
        // (Combiner would corrupt sessionization logic)

        // Input/Output
        FileInputFormat.addInputPath(job, new Path(args[0]));
        FileOutputFormat.setOutputPath(job, new Path(args[1]));

        // ═══════════ PERFORMANCE TUNING VIA CODE ═══════════
        // These override mapred-site.xml for THIS specific job
        // Use when different stages need different tuning

        // Number of reduce tasks — sized for 2.8PB input
        // Each reducer processes ~1GB of shuffled data
        job.setNumReduceTasks(2000);

        // Confirm critical settings are applied
        System.out.println("=== PERFORMANCE SETTINGS ===");
        System.out.println("io.sort.mb:          " + conf.getInt("mapreduce.task.io.sort.mb", -1));
        System.out.println("map.output.compress:  " + conf.getBoolean("mapreduce.map.output.compress", false));
        System.out.println("compress.codec:       " + conf.get("mapreduce.map.output.compress.codec", "none"));
        System.out.println("reduce.slowstart:     " + conf.getFloat("mapreduce.job.reduce.slowstart.completedmaps", -1));
        System.out.println("shuffle.parallelcopy: " + conf.getInt("mapreduce.reduce.shuffle.parallelcopies", -1));
        System.out.println("jvm.numtasks:         " + conf.getInt("mapreduce.job.jvm.numtasks", -1));
        System.out.println("map.speculative:      " + conf.getBoolean("mapreduce.map.speculative", false));
        System.out.println("num.reducers:         " + job.getNumReduceTasks());
        System.out.println("============================");

        return job.waitForCompletion(true) ? 0 : 1;
    }

    public static void main(String[] args) throws Exception {
        int exitCode = ToolRunner.run(new Configuration(), new ClickstreamSessionizer(), args);
        System.exit(exitCode);
    }
}

    Reads job counters and diagnoses performance bottlenecks.

    def __init__(self, job_id):

    def _parse_counter_output(self, output):
        """Parse mapred job -status output for counters."""
        current_group = None
        for line in output.split('\n'):
            line = line.strip()
            if '=' in line and current_group:
                parts = line.split('=')
                if len(parts) == 2:
                    counter_name = parts[0].strip()
                    counter_value = int(parts[1].strip())
                    self.counters[f"{current_group}.{counter_name}"] = counter_value
            elif line and not line.startswith('-'):
                current_group = line

    def _analyze_spill_ratio(self):

    def _analyze_compression(self):
        """Analyze map output compression effectiveness."""
        raw_bytes = self.counters.get("MAP_OUTPUT_BYTES", 0)
        materialized = self.counters.get("MAP_OUTPUT_MATERIALIZED_BYTES", 0)
        
        if raw_bytes == 0:
            return
        
        if materialized == 0:
            print(f"  ── COMPRESSION ANALYSIS ──")
            print(f"    ❌ Map output compression is DISABLED")
            print(f"       Raw map output: {raw_bytes / (1024**4):.1f} PB")
            print(f"       All {raw_bytes / (1024**4):.1f} PB transferred during shuffle")
            self.recommendations.append({
                "priority": "CRITICAL",
                "parameter": "mapreduce.map.output.compress",
                "current": "false",
                "recommended": "true (with SnappyCodec)",
                "reason": f"Shuffle transfers {raw_bytes / (1024**4):.1f}PB uncompressed"
            })
            print()
            return
        
        compression_ratio = materialized / raw_bytes
        savings_pct = (1 - compression_ratio) * 100
        saved_bytes = raw_bytes - materialized
        
        print(f"  ── COMPRESSION ANALYSIS ──")
        print(f"  Raw map output:          {raw_bytes / (1024**4):.2f} PB")
        print(f"  Materialized (compressed):{materialized / (1024**4):.2f} PB")
        print(f"  Compression ratio:       {compression_ratio:.3f} ({savings_pct:.1f}% savings)")
        print(f"  Network traffic saved:   {saved_bytes / (1024**4):.2f} PB")
        print()
        
        if compression_ratio < 0.40:
            print(f"    ✅ Excellent compression ({savings_pct:.0f}% reduction)")
        elif compression_ratio < 0.60:
            print(f"    ✅ Good compression ({savings_pct:.0f}% reduction)")
        else:
            print(f"    ⚠️  Poor compression ({savings_pct:.0f}% reduction)")
            print(f"       Consider: data may already be compressed, or try Zstd codec")
        print()
    
    def _analyze_combiner(self):
        """Analyze combiner effectiveness."""
        combine_in = self.counters.get("COMBINE_INPUT_RECORDS", 0)
        combine_out = self.counters.get("COMBINE_OUTPUT_RECORDS", 0)
        
        print(f"  ── COMBINER ANALYSIS ──")
        
        if combine_in == 0:
            map_out = self.counters.get("MAP_OUTPUT_RECORDS", 0)
            shuffle_bytes = self.counters.get("REDUCE_SHUFFLE_BYTES", 0)
            
            print(f"    ⚠️  No combiner configured")
            print(f"       Map output records: {map_out:,}")
            print(f"       All records shipped to reducers via shuffle")
            
            if map_out > 1_000_000_000:
                self.recommendations.append({
                    "priority": "HIGH",
                    "parameter": "job.setCombinerClass()",
                    "current": "none",
                    "recommended": "Add combiner if operation is associative+commutative",
                    "reason": f"{map_out:,} records shuffled without local reduction"
                })
        else:
            reduction = 1 - (combine_out / combine_in) if combine_in > 0 else 0
            print(f"  Combiner input:  {combine_in:,}")
            print(f"  Combiner output: {combine_out:,}")
            print(f"  Reduction:       {reduction:.1%}")
            
            if reduction > 0.5:
                print(f"    ✅ Combiner very effective ({reduction:.0%} reduction)")
            elif reduction > 0.2:
                print(f"    ✅ Combiner moderately effective ({reduction:.0%} reduction)")
            else:
                print(f"    ⚠️  Combiner not very effective ({reduction:.0%} reduction)")
        print()
    
    def _analyze_gc_overhead(self):
        """Analyze garbage collection overhead."""
        gc_time = self.counters.get("GC_TIME_MILLIS", 0)

        print(f"  ── GC OVERHEAD ANALYSIS ──")

        print(f"  Total CPU time:  {cpu_time / 1000:,.0f} seconds")
        print(f"  GC overhead:     {gc_pct:.1f}%")
        print()
        
        if gc_pct < 10:
            print(f"    ✅ GC overhead acceptable ({gc_pct:.1f}% < 10%)")
        elif gc_pct < 20:

            self.recommendations.append({
                "priority": "MEDIUM",
                "parameter": "mapreduce.map.java.opts / reduce.java.opts",
                "current": "current heap",

                "reason": f"GC overhead {gc_pct:.1f}% — wasting {gc_time/1000:.0f}s on GC"

            print(f"    ❌ GC overhead excessive ({gc_pct:.1f}%)")

                "priority": "HIGH",

                "current": "current heap",

                "reason": f"GC overhead {gc_pct:.1f}% — {gc_pct:.0f}% of computation wasted"
            })
        print()
    
    def _analyze_data_locality(self):
        """Analyze data locality of map tasks."""
        total_maps = self.counters.get("TOTAL_LAUNCHED_MAPS", 0)
        data_local = self.counters.get("DATA_LOCAL_MAPS", 0)
        rack_local = self.counters.get("RACK_LOCAL_MAPS", 0)
        
        if total_maps == 0:
            return
        
        data_local_pct = (data_local / total_maps) * 100
        rack_local_pct = (rack_local / total_maps) * 100
        remote_pct = 100 - data_local_pct - rack_local_pct
        
        print(f"  ── DATA LOCALITY ANALYSIS ──")
        print(f"  Total map tasks:    {total_maps:,}")
        print(f"  Data-local:         {data_local:,} ({data_local_pct:.1f}%)")
        print(f"  Rack-local:         {rack_local:,} ({rack_local_pct:.1f}%)")
        print(f"  Remote:             {total_maps - data_local - rack_local:,} ({remote_pct:.1f}%)")
        print()
        
        if data_local_pct >= 90:
            print(f"    ✅ Excellent data locality ({data_local_pct:.0f}%)")
        elif data_local_pct >= 70:
            print(f"    ⚠️  Moderate data locality ({data_local_pct:.0f}%)")
            self.recommendations.append({
                "priority": "LOW",
                "parameter": "cluster / scheduling",
                "current": f"{data_local_pct:.0f}% data-local",
                "recommended": "Check HDFS replication, rack awareness, scheduler delay",
                "reason": "Lower data locality = more network I/O for map input"
            })
        else:
            print(f"    ❌ Poor data locality ({data_local_pct:.0f}%)")
            self.recommendations.append({
                "priority": "HIGH",
                "parameter": "yarn.scheduler.capacity.node-locality-delay",
                "current": f"{data_local_pct:.0f}% data-local",
                "recommended": "Increase locality delay, check HDFS balancer",
                "reason": "Poor locality significantly increases map input read time"
            })
        print()

        print()
    
    def _analyze_disk_io(self):

        if local_io_ratio < 0.5:
            print(f"    ✅ Local disk I/O well-controlled")
        elif local_io_ratio < 1.0:
            print(f"    ⚠️  Local disk I/O is {local_io_ratio:.1f}× HDFS input")
            print(f"       Spill files and shuffle merges generating significant disk I/O")

            print(f"    ❌ Local disk I/O is {local_io_ratio:.1f}× HDFS input!")

            self.recommendations.append({
                "priority": "CRITICAL",
                "parameter": "mapreduce.task.io.sort.mb",
                "current": "too small",
                "recommended": "Double current value",
                "reason": f"Local I/O {local_io_ratio:.1f}× input — disk is bottleneck"
            })
        print()
    
    def _analyze_speculative(self):

        print(f"  Speculated tasks: {speculated:,} ({spec_pct:.2f}%)")

            print(f"    ✅ Healthy speculation rate ({spec_pct:.2f}%)")

            print(f"    ⚠️  Moderate speculation ({spec_pct:.1f}%) — some stragglers")
        else:
            print(f"    ❌ High speculation ({spec_pct:.1f}%) — many stragglers")
            print(f"       Check for hardware issues or data skew")
        print()
