PROJECT 16: ENRICHING ORDERS WITH CUSTOMER NAMES — THE MAP-SIDE JOIN

🔴 HYPOTHETICAL PROBLEM

SCENARIO:
→ You are a data engineer at a large online retail company
→ Two datasets need to be combined for a quarterly business report:

DATASET 1 — Orders (LARGE: 800 million records, 120GB on HDFS)
  ORDER_ID|CUSTOMER_ID|PRODUCT_ID|QUANTITY|PRICE|ORDER_DATE
  ORD-0000001|C00542|P1023|2|49.99|2024-01-15
  ORD-0000002|C01287|P0891|1|129.50|2024-01-15
  ORD-0000003|C00542|P0456|3|19.99|2024-01-16

DATASET 2 — Customers (SMALL: 500,000 records, 60MB on HDFS)
  CUSTOMER_ID|NAME|EMAIL|TIER|SIGNUP_DATE|STATE
  C00542|Maria Santos|maria@email.com|GOLD|2020-03-15|CA
  C01287|James Chen|james@email.com|SILVER|2021-08-22|NY

DATASET 3 — Products (SMALL: 50,000 records, 8MB on HDFS)
  PRODUCT_ID|PRODUCT_NAME|CATEGORY|BRAND
  P1023|Wireless Mouse|Electronics|TechPro
  P0891|Running Shoes|Footwear|SpeedFit
  P0456|USB Cable|Electronics|TechPro

REQUIRED OUTPUT (Fully Enriched Order Report):
  ORD-0000001|C00542|Maria Santos|GOLD|CA|P1023|Wireless Mouse|Electronics|2|49.99|2024-01-15

THE CHALLENGE:
→ Project 15 showed enrichment with ONE lookup table
→ Now we need TWO lookup tables simultaneously
→ Customer file (60MB) + Product file (8MB) = 68MB in cache
→ Both fit in memory → Map-Side Join is still feasible
→ But we need a FORMAL, STRUCTURED approach
   that scales beyond ad-hoc HashMap loading

WHY NOT REDUCE-SIDE JOIN?
→ 800 million order records would ALL shuffle over network
→ 120GB × network transfer = massive overhead
→ Customer + Product files are SMALL — cache is perfect
→ Map-Side Join: ZERO shuffle, ZERO reducer, maximum speed

THE REAL QUESTION:
→ How do we formalize the Map-Side Join as a reusable pattern?
→ How do we handle MULTIPLE lookup tables cleanly?
→ What are the formal rules that determine map-side vs reduce-side?
→ What happens with LEFT JOIN vs INNER JOIN semantics?

🔍 HOW TO APPROACH IT

STEP 1: Understand the formal definition of Map-Side Join

Map-Side Join = joining datasets where ALL processing happens
in the mapper, with NO reducer involvement.

PRECONDITIONS (ALL must be true):
  ✅ One dataset is LARGE (the "probe" side — orders)
  ✅ Other dataset(s) are SMALL (the "build" side — customers, products)
  ✅ Small datasets fit in mapper memory as HashMaps
  ✅ Join key exists in both datasets
  ✅ No cross-record dependencies needed

STEP 2: Understand join semantics

JOIN TYPES IN MAP-SIDE JOIN:

┌──────────────────────────────────────────────────────────────────┐
│  INNER JOIN:                                                      │
│  → Only emit if BOTH lookup tables have matching keys             │
│  → Order with unknown customer OR unknown product → SKIP          │
│  → Output count ≤ input count                                     │
│                                                                   │
│  LEFT JOIN (most common):                                         │
│  → Always emit the order record                                   │
│  → If customer not found → use "UNKNOWN" placeholders             │
│  → If product not found → use "UNKNOWN" placeholders              │
│  → Output count = input count (minus invalid records)             │
│  → THIS IS WHAT PROJECT 15 DID (left join with UNKNOWN fallback)  │
│                                                                   │
│  FULL OUTER JOIN:                                                 │
│  → Cannot do with Map-Side Join alone                             │
│  → Customers with no orders would never appear                    │
│  → Requires Reduce-Side Join (Project 17)                         │
└──────────────────────────────────────────────────────────────────┘

STEP 3: Understand multi-table cache loading

Project 15 loaded ONE cache file. Now we load TWO:

  setup():
    1. Load customers.txt → customerMap<customerId, customerInfo>
    2. Load products.txt  → productMap<productId, productInfo>
    Total memory: ~180MB (60MB × ~2.5× overhead + 8MB × ~2.5×)
    
  map():
    1. Parse order record → extract customerId, productId
    2. customerInfo = customerMap.get(customerId)   // O(1)
    3. productInfo = productMap.get(productId)       // O(1)
    4. Emit enriched record with both lookups

💡 HOW CAN WE SOLVE IT

SOLUTION: Multi-Table Map-Side Join via Distributed Cache

✅ WHAT COULD BE DONE

APPROACH A: Multi-cache Map-Side Join, map-only (OUR CHOICE)
→ Both lookup tables in distributed cache
→ Two HashMaps in mapper setup()
→ Zero reducers — pure enrichment
→ Fastest execution, simplest architecture

APPROACH B: Map-Side Join + Reducer for aggregation
→ Same cache-based enrichment in mapper
→ Add reducer for: "Total revenue per customer tier per state"
→ Mapper enriches AND emits aggregation key
→ Reducer performs group-by aggregation
→ One job does both enrichment AND analysis

APPROACH C: Two separate jobs chained (Project 21)
→ Job 1: Map-Side Join with customers → enriched_v1
→ Job 2: Map-Side Join with products on enriched_v1 → enriched_v2
→ Works but reads/writes data twice → 2× I/O overhead
→ Single job with two caches is much more efficient

APPROACH D: Reduce-Side Join (Project 17)
→ All three datasets as mapper input
→ Tag each record with source
→ Reducer matches by join key
→ Works for ANY size datasets
→ Overkill here — lookup tables are small

WE CHOOSE APPROACH A because:
→ Demonstrates the formal multi-table map-side join pattern
→ Maximum performance (zero shuffle)
→ Clean, reusable architecture
→ Extends Project 15 naturally
→ Prepares for Reduce-Side Join comparison (Project 17)

✅ WHAT SHOULD DO

1. VALIDATE ALL CACHE FILES FIT IN MEMORY TOGETHER
   → Customer HashMap: 60MB × 2.5 overhead = ~150MB
   → Product HashMap: 8MB × 2.5 overhead = ~20MB
   → Combined: ~170MB
   → Mapper heap (1GB default): 170MB = 17% → comfortable
   → RULE: Total cache HashMaps should be < 50% of mapper heap

2. LOAD CACHE FILES IN DETERMINISTIC ORDER
   → Always load customers first, products second
   → If one fails, you know which → easier debugging
   → Log each load: "Loaded 500,000 customers in 1.2s"
   → Log each load: "Loaded 50,000 products in 0.3s"

3. USE SEPARATE COUNTER GROUPS PER LOOKUP TABLE
   → CUSTOMER_HIT, CUSTOMER_MISS
   → PRODUCT_HIT, PRODUCT_MISS
   → After job: identify WHICH table has data gaps
   → "98% customer hit, 85% product hit" → product catalog is stale

4. DEFINE JOIN TYPE EXPLICITLY IN CONFIGURATION
   → Pass join type as configuration parameter:
     conf.set("join.type", "LEFT")  or "INNER"
   → Mapper reads config → applies appropriate logic
   → Makes job reusable for different business requirements

5. VALIDATE JOIN KEY TYPES MATCH ACROSS DATASETS
   → Orders: CUSTOMER_ID = "C00542" (with prefix)
   → Customers: CUSTOMER_ID = "C00542" (with prefix)
   → If one has "542" and other has "C00542" → zero matches
   → ALWAYS normalize keys: trim, case-normalize, strip prefixes

6. TEST WITH KNOWN DATA WHERE EXPECTED OUTPUT IS CALCULABLE
   → Create 10 orders, 5 customers, 5 products
   → Manually compute expected enriched output
   → Run job → compare → verify before scaling

🚫 WHAT SHOULD NOT DO

1. DO NOT LOAD SAME CACHE FILE TWICE WITH DIFFERENT NAMES
   → job.addCacheFile("/path/customers.txt#cust")
   → job.addCacheFile("/path/customers.txt#customers")
   → Same file downloaded twice → wastes bandwidth and disk
   → Use ONE symlink name consistently

2. DO NOT USE REDUCE-SIDE JOIN WHEN MAP-SIDE WORKS
   → If smaller dataset fits in memory → map-side wins
   → Reduce-side shuffles ALL data over network
   → For 120GB orders + 68MB lookups:
     → Map-side: 68MB network transfer (cache distribution)
     → Reduce-side: ~120GB network transfer (shuffle)
   → 1,764× more network with reduce-side

3. DO NOT IGNORE NULL SAFETY IN MULTI-LOOKUP
   → customerInfo = customerMap.get(custId)  // may be null
   → productInfo = productMap.get(prodId)    // may also be null
   → INNER JOIN: skip if EITHER is null
   → LEFT JOIN: use UNKNOWN if EITHER is null
   → NEVER call .split() on null → NullPointerException

4. DO NOT HARDCODE CACHE FILE PATHS IN MAPPER
   → Mapper should use SYMLINK names only
   → Driver controls which files are cached
   → Mapper says: new FileReader("customers") — not full HDFS path
   → This decouples mapper from data location

5. DO NOT FORGET THAT Map-Side Join CANNOT DO FULL OUTER JOIN
   → Customers with ZERO orders never appear in output
   → Because mapper iterates orders, not customers
   → For "all customers including those with no orders":
     → Use Reduce-Side Join (Project 17)
     → Or separate job to find customers with no orders

6. DO NOT MIX MULTIPLE CACHE FILES INTO ONE HASHMAP
   → Separate HashMap per lookup table
   → customerMap for customers, productMap for products
   → Mixing causes key collision: what if customer ID "C001" and
     product ID "C001" exist? Different entities, same key

🚫 WHEN NOT TO USE

MAP-SIDE JOIN IS WRONG WHEN:

1. BOTH DATASETS ARE LARGE
   → 100GB orders + 80GB customer history
   → 80GB HashMap = impossible in mapper memory
   → Must use Reduce-Side Join

2. FULL OUTER JOIN REQUIRED
   → "Show all customers, even those with no orders"
   → Map-Side iterates orders → customers without orders never appear
   → Must use Reduce-Side Join with tagged records

3. JOIN KEY IS NOT SIMPLE EQUALITY
   → "Join where order.date BETWEEN customer.start AND customer.end"
   → Range joins cannot use HashMap (exact key match only)
   → Must use Reduce-Side Join with custom logic

4. LOOKUP DATA CHANGES DURING JOB
   → Customer tier changes mid-job
   → Cache is static snapshot → stale joins for later mappers
   → Use HBase for live lookups (Project 27)

5. TOO MANY LOOKUP TABLES EXCEED MEMORY
   → customers (60MB) + products (8MB) + warehouses (5MB)
     + promotions (2MB) + shipping_rates (1MB) = 76MB → fine
   → But: 10 tables × 50MB each = 500MB → risky
   → Each HashMap has ~2.5× overhead → 1.25GB → exceeds heap

MECHANIC 1: MULTI-FILE DISTRIBUTED CACHE INTERNALS
═══════════════════════════════════════════════════════════

When driver adds multiple cache files:
  job.addCacheFile(new URI("/data/customers.txt#customers"))
  job.addCacheFile(new URI("/data/products.txt#products"))

Configuration stores:
  "mapreduce.job.cache.files" = "/data/customers.txt#customers,/data/products.txt#products"

NodeManager localization process (per node):
  1. Parse comma-separated list of cache URIs
  2. For EACH file:
     a. Check if already in local cache (by path + modification time)
     b. If not: download from HDFS to local cache directory
     c. Create symlink in task working directory
  3. All files localized BEFORE task JVM starts

Task working directory layout:
  ┌─────────────────────────────────────────────┐
  │  /task/working/directory/                     │
  │    customers → /local/cache/abc/customers.txt │
  │    products  → /local/cache/def/products.txt  │
  │    (symlinks to actual cached files)          │
  └─────────────────────────────────────────────┘

IMPORTANT: Order of addCacheFile() calls determines order in
getCacheFiles() array, but symlink names are independent.
Always use symlink names, never array indices.

MECHANIC 4: WHY MAP-SIDE JOIN BEATS REDUCE-SIDE FOR THIS CASE
═══════════════════════════════════════════════════════════

QUANTITATIVE COMPARISON for our scenario:

MAP-SIDE JOIN:
  Cache distribution: 68MB × 50 nodes = 3.4GB (once, before job)
  Shuffle: 0 bytes
  Disk I/O: Read 120GB orders (input splits)
  Network: 3.4GB total
  Reducers: 0
  Wall time estimate: ~15 minutes

REDUCE-SIDE JOIN:
  Shuffle: 120GB orders + 68MB lookups = ~120GB
  At 1Gbps per node, 50 nodes: 120GB / (50 × 125MB/s) = ~19 seconds
  But with contention: ~5-10 minutes just for shuffle
  Plus: sort on reduce side, reduce processing
  Disk I/O: Read 120GB + write 120GB intermediate + read 120GB at reduce
  Network: 120GB
  Reducers: ~100 (each processing ~1.2GB)
  Wall time estimate: ~45 minutes

  Map-Side is 3× faster for this workload.
  The advantage grows with larger order datasets.

⚠️ COMMON PITFALLS AND FAILURE MODES

PITFALL 1: "Second cache file not found — only first loads"
→ CAUSE: Symlink names conflict or wrong name in mapper
→ job.addCacheFile(...#customers)
→ job.addCacheFile(...#products)
→ Mapper opens: new FileReader("customer") ← typo: missing 's'
→ FIX: Print all symlinks in setup() for debugging:
   File workDir = new File(".");
   for (String f : workDir.list()) {
       System.err.println("Available file: " + f);
   }

PITFALL 2: "Join produces zero matches despite valid data"
→ CAUSE: Key format mismatch between datasets
→ Orders: CUSTOMER_ID = "C00542"
→ Customers: CUSTOMER_ID = " C00542" (leading space)
→ HashMap.get("C00542") misses " C00542"
→ FIX: Trim ALL keys when loading AND when looking up
→ FIX: Log first 3 keys from each source for visual comparison

PITFALL 3: "OutOfMemoryError with two large cache files"
→ CAUSE: Combined HashMap size exceeds mapper heap
→ Customer HashMap: 150MB + Product HashMap: 20MB = 170MB
→ Plus JVM overhead + processing buffers = needs ~500MB
→ Default heap may be 256MB → OOM
→ FIX: conf.set("mapreduce.map.java.opts", "-Xmx1g")
→ FIX: conf.setInt("mapreduce.map.memory.mb", 1536)

PITFALL 4: "Enriched output has misaligned fields"
→ CAUSE: Lookup value contains the delimiter character
→ Customer name: "Smith, Jr." and delimiter is comma
→ Output: "ORD-001,Smith, Jr.,GOLD" → extra field
→ FIX: Use pipe delimiter (|) which rarely appears in data
→ FIX: Or escape/quote values containing delimiter

PITFALL 5: "Some mappers have cache, others don't"
→ CAUSE: Cache file deleted from HDFS during job execution
→ First mappers localize successfully
→ Later mappers fail to download → FileNotFoundException
→ FIX: Never modify/delete cache files while jobs reference them
→ FIX: Use immutable dated files: customers_20240115.txt

PITFALL 6: "INNER JOIN drops too many records"
→ CAUSE: Stale lookup tables
→ New customers not in cache file → orders dropped
→ New products not in cache file → orders dropped
→ DIAGNOSIS: Check CACHE_MISS counter percentage
→ FIX: Refresh cache files before each job run
→ FIX: Use LEFT JOIN and track UNKNOWN count for monitoring

  │ CLUSTER SIZE         │ MAP-SIDE     │ REDUCE-SIDE   │

  Map-Side scales nearly linearly with nodes.
  Reduce-Side limited by shuffle bottleneck.

HOW TO VERIFY MULTI-TABLE MAP-SIDE JOIN:

1. PER-TABLE HIT/MISS COUNTERS
   ┌──────────────────────────────────────────────┐
   │  CUSTOMER_HIT:   780,000,000  (97.5%)        │
   │  CUSTOMER_MISS:   20,000,000  (2.5%)         │
   │  PRODUCT_HIT:    795,000,000  (99.4%)        │
   │  PRODUCT_MISS:     5,000,000  (0.6%)         │
   │                                               │
   │  INNER_JOIN_MATCH: 776,000,000 (97.0%)       │
   │  INNER_JOIN_SKIP:   24,000,000 (3.0%)        │
   │  (or LEFT_JOIN_PARTIAL: records with ≥1 miss) │
   └──────────────────────────────────────────────┘

2. SPOT-CHECK ENRICHED RECORDS
   → Pick 5 order IDs manually
   → Look up customer and product in original files
   → Verify enriched output matches
   → Automated: Python script compares samples

3. FIELD COUNT VALIDATION
   → Input: 6 fields per order
   → Customer adds: 4 fields (name, email, tier, state)
   → Product adds: 3 fields (name, category, brand)
   → Output: 6 + 4 + 3 = 13 fields (or subset)
   → Count fields in output: awk -F'|' '{print NF}' | sort -u
   → Should show single value: 13

4. RECORD COUNT RECONCILIATION
   → LEFT JOIN: output records = input records (minus invalid)
   → INNER JOIN: output records ≤ input records
   → output > input → BUG (duplicate emission)

import org.apache.hadoop.conf.Configuration;
import org.apache.hadoop.conf.Configured;
import org.apache.hadoop.fs.Path;
import org.apache.hadoop.io.LongWritable;
import org.apache.hadoop.io.NullWritable;
import org.apache.hadoop.io.Text;
import org.apache.hadoop.mapreduce.Job;
import org.apache.hadoop.mapreduce.Mapper;
import org.apache.hadoop.mapreduce.lib.input.FileInputFormat;
import org.apache.hadoop.mapreduce.lib.output.FileOutputFormat;
import org.apache.hadoop.util.Tool;
import org.apache.hadoop.util.ToolRunner;

import java.io.BufferedReader;
import java.io.FileReader;
import java.io.IOException;
import java.net.URI;
import java.util.HashMap;
import java.util.Map;

public class MapSideJoinJob extends Configured implements Tool {

    // Join type constants
    public static final String JOIN_TYPE_KEY = "join.type";
    public static final String JOIN_INNER = "INNER";
    public static final String JOIN_LEFT = "LEFT";

    // ════════════════════════════════════════════════════════
    // MAPPER: Multi-table Map-Side Join
    // ════════════════════════════════════════════════════════

    public static class JoinMapper
            extends Mapper<LongWritable, Text, NullWritable, Text> {

        // ── TWO separate lookup tables ──
        private Map<String, String[]> customerMap;  // custId → [name, email, tier, state]
        private Map<String, String[]> productMap;   // prodId → [name, category, brand]

        private Text enrichedRecord = new Text();
        private String joinType;

        // ── Per-table counters ──
        enum JoinCounters {
            TOTAL_ORDERS,
            CUSTOMER_CACHE_LOADED,
            PRODUCT_CACHE_LOADED,
            CUSTOMER_HIT,
            CUSTOMER_MISS,
            PRODUCT_HIT,
            PRODUCT_MISS,
            INNER_JOIN_MATCH,
            INNER_JOIN_SKIP,
            LEFT_JOIN_FULL_MATCH,
            LEFT_JOIN_PARTIAL_MATCH,
            ENRICHED_OUTPUT,
            INVALID_ORDER,
            HEADER_SKIPPED,
            CUSTOMER_LOAD_MS,
            PRODUCT_LOAD_MS
        }

        @Override
        protected void setup(Context context) throws IOException {
            Configuration conf = context.getConfiguration();
            joinType = conf.get(JOIN_TYPE_KEY, JOIN_LEFT);

            System.err.println("Join type: " + joinType);

            // ══════════════════════════════════════════
            // LOAD CACHE FILE 1: Customers
            // ══════════════════════════════════════════
            long startTime = System.currentTimeMillis();
            customerMap = new HashMap<>();

            try (BufferedReader reader = new BufferedReader(new FileReader("customers"))) {
                String line;
                while ((line = reader.readLine()) != null) {
                    line = line.trim();
                    if (line.isEmpty() || line.startsWith("CUSTOMER_ID")) continue;

                    String[] parts = line.split("\\|");
                    if (parts.length >= 6) {
                        String custId = parts[0].trim();
                        String[] info = new String[] {
                            parts[1].trim(),  // name
                            parts[2].trim(),  // email
                            parts[3].trim(),  // tier
                            parts[5].trim()   // state (index 5, skipping signup_date)
                        };
                        customerMap.put(custId, info);
                    }
                }
            }

            long customerLoadTime = System.currentTimeMillis() - startTime;
            context.getCounter(JoinCounters.CUSTOMER_CACHE_LOADED)
                   .increment(customerMap.size());
            context.getCounter(JoinCounters.CUSTOMER_LOAD_MS)
                   .increment(customerLoadTime);

            System.err.println("Loaded " + customerMap.size()
                + " customers in " + customerLoadTime + "ms");

            // ══════════════════════════════════════════
            // LOAD CACHE FILE 2: Products
            // ══════════════════════════════════════════
            startTime = System.currentTimeMillis();
            productMap = new HashMap<>();

            try (BufferedReader reader = new BufferedReader(new FileReader("products"))) {
                String line;
                while ((line = reader.readLine()) != null) {
                    line = line.trim();
                    if (line.isEmpty() || line.startsWith("PRODUCT_ID")) continue;

                    String[] parts = line.split("\\|");
                    if (parts.length >= 4) {
                        String prodId = parts[0].trim();
                        String[] info = new String[] {
                            parts[1].trim(),  // product_name
                            parts[2].trim(),  // category
                            parts[3].trim()   // brand
                        };
                        productMap.put(prodId, info);
                    }
                }
            }

            long productLoadTime = System.currentTimeMillis() - startTime;
            context.getCounter(JoinCounters.PRODUCT_CACHE_LOADED)
                   .increment(productMap.size());
            context.getCounter(JoinCounters.PRODUCT_LOAD_MS)
                   .increment(productLoadTime);

            System.err.println("Loaded " + productMap.size()
                + " products in " + productLoadTime + "ms");
            System.err.println("Total cache memory: ~"
                + ((customerMap.size() * 200 + productMap.size() * 150) / (1024 * 1024))
                + " MB estimated");
        }


        @Override
        protected void map(LongWritable offset, Text line, Context context)
                throws IOException, InterruptedException {

            String record = line.toString().trim();
            if (record.isEmpty()) return;

            if (record.startsWith("ORDER_ID")) {
                context.getCounter(JoinCounters.HEADER_SKIPPED).increment(1);
                return;
            }

            context.getCounter(JoinCounters.TOTAL_ORDERS).increment(1);

            // Parse order: ORDER_ID|CUSTOMER_ID|PRODUCT_ID|QUANTITY|PRICE|ORDER_DATE
            String[] fields = record.split("\\|");
            if (fields.length != 6) {
                context.getCounter(JoinCounters.INVALID_ORDER).increment(1);
                return;
            }

            String orderId   = fields[0].trim();
            String custId    = fields[1].trim();
            String prodId    = fields[2].trim();
            String quantity  = fields[3].trim();
            String price     = fields[4].trim();
            String orderDate = fields[5].trim();

            // ══════════════════════════════════════════
            // LOOKUP 1: Customer details
            // ══════════════════════════════════════════
            String[] custInfo = customerMap.get(custId);
            boolean customerFound = (custInfo != null);

            if (customerFound) {
                context.getCounter(JoinCounters.CUSTOMER_HIT).increment(1);
            } else {
                context.getCounter(JoinCounters.CUSTOMER_MISS).increment(1);
            }

            // ══════════════════════════════════════════
            // LOOKUP 2: Product details
            // ══════════════════════════════════════════
            String[] prodInfo = productMap.get(prodId);
            boolean productFound = (prodInfo != null);

            if (productFound) {
                context.getCounter(JoinCounters.PRODUCT_HIT).increment(1);
            } else {
                context.getCounter(JoinCounters.PRODUCT_MISS).increment(1);
            }

            // ══════════════════════════════════════════
            // APPLY JOIN SEMANTICS
            // ══════════════════════════════════════════

            if (joinType.equals(JOIN_INNER)) {
                // INNER JOIN: Skip if EITHER lookup misses
                if (!customerFound || !productFound) {
                    context.getCounter(JoinCounters.INNER_JOIN_SKIP).increment(1);
                    return;
                }
                context.getCounter(JoinCounters.INNER_JOIN_MATCH).increment(1);
            } else {
                // LEFT JOIN: Always emit, use UNKNOWN for misses
                if (customerFound && productFound) {
                    context.getCounter(JoinCounters.LEFT_JOIN_FULL_MATCH).increment(1);
                } else {
                    context.getCounter(JoinCounters.LEFT_JOIN_PARTIAL_MATCH).increment(1);
                }
            }

            // ── Build enriched output ──
            String custName  = customerFound ? custInfo[0] : "UNKNOWN";
            String custEmail = customerFound ? custInfo[1] : "UNKNOWN";
            String custTier  = customerFound ? custInfo[2] : "UNKNOWN";
            String custState = customerFound ? custInfo[3] : "UNKNOWN";

            String prodName     = productFound ? prodInfo[0] : "UNKNOWN";
            String prodCategory = productFound ? prodInfo[1] : "UNKNOWN";
            String prodBrand    = productFound ? prodInfo[2] : "UNKNOWN";

            // Output: ORDER_ID|CUST_ID|CUST_NAME|TIER|STATE|PROD_ID|PROD_NAME|CATEGORY|BRAND|QTY|PRICE|DATE
            String enriched = String.join("|",
                orderId, custId, custName, custTier, custState,
                prodId, prodName, prodCategory, prodBrand,
                quantity, price, orderDate
            );

            enrichedRecord.set(enriched);
            context.write(NullWritable.get(), enrichedRecord);
            context.getCounter(JoinCounters.ENRICHED_OUTPUT).increment(1);
        }


        @Override
        protected void cleanup(Context context) throws IOException {
            System.err.println("Join complete. Customer entries: " + customerMap.size()
                + ", Product entries: " + productMap.size());
            customerMap.clear();
            customerMap = null;
            productMap.clear();
            productMap = null;
        }
    }


    // ════════════════════════════════════════════════════════
    // DRIVER
    // ════════════════════════════════════════════════════════

    @Override
    public int run(String[] args) throws Exception {
        if (args.length < 5) {
            System.err.println("Usage: MapSideJoinJob <orders_input> <output> "
                + "<customers_cache> <products_cache> <join_type>");
            System.err.println("  join_type: INNER | LEFT");
            return 1;
        }

        String ordersPath    = args[0];
        String outputPath    = args[1];
        String customersPath = args[2];
        String productsPath  = args[3];
        String joinType      = args[4].toUpperCase();

        if (!joinType.equals(JOIN_INNER) && !joinType.equals(JOIN_LEFT)) {
            System.err.println("Invalid join type: " + joinType);
            System.err.println("Use: INNER or LEFT");
            return 1;
        }

        Configuration conf = getConf();
        conf.set(JOIN_TYPE_KEY, joinType);

        Job job = Job.getInstance(conf,
            "Map-Side Join (" + joinType + "): Orders × Customers × Products");
        job.setJarByClass(MapSideJoinJob.class);

        job.setMapperClass(JoinMapper.class);
        job.setNumReduceTasks(0);  // Map-only (Project 11)

        job.setOutputKeyClass(NullWritable.class);
        job.setOutputValueClass(Text.class);

        // ══════════════════════════════════════════════
        // ADD BOTH CACHE FILES
        // ══════════════════════════════════════════════
        job.addCacheFile(new URI(customersPath + "#customers"));
        job.addCacheFile(new URI(productsPath + "#products"));

        System.out.println("═══ MAP-SIDE JOIN CONFIGURATION ═══");
        System.out.println("  Join type:      " + joinType);
        System.out.println("  Orders input:   " + ordersPath);
        System.out.println("  Customer cache: " + customersPath + " → symlink: customers");
        System.out.println("  Product cache:  " + productsPath + " → symlink: products");
        System.out.println("  Output:         " + outputPath);
        System.out.println("  Reducers:       0 (map-only)");
        System.out.println("════════════════════════════════════");

        FileInputFormat.addInputPath(job, new Path(ordersPath));
        FileOutputFormat.setOutputPath(job, new Path(outputPath));

        boolean success = job.waitForCompletion(true);

        if (success) {
            printJoinReport(job, joinType);
        }

        return success ? 0 : 1;
    }


    private void printJoinReport(Job job, String joinType) throws IOException {
        System.out.println("\n╔══════════════════════════════════════════════════════╗");
        System.out.println("║          MAP-SIDE JOIN REPORT (" + joinType + ")                  ║");
        System.out.println("╠══════════════════════════════════════════════════════╣");

        long total = getCounter(job, "TOTAL_ORDERS");
        long custLoaded = getCounter(job, "CUSTOMER_CACHE_LOADED");
        long prodLoaded = getCounter(job, "PRODUCT_CACHE_LOADED");
        long custHit = getCounter(job, "CUSTOMER_HIT");
        long custMiss = getCounter(job, "CUSTOMER_MISS");
        long prodHit = getCounter(job, "PRODUCT_HIT");
        long prodMiss = getCounter(job, "PRODUCT_MISS");
        long enriched = getCounter(job, "ENRICHED_OUTPUT");
        long custLoadMs = getCounter(job, "CUSTOMER_LOAD_MS");
        long prodLoadMs = getCounter(job, "PRODUCT_LOAD_MS");

        System.out.printf("║  Cache: %,d customers loaded in %,d ms%n", custLoaded, custLoadMs);
        System.out.printf("║  Cache: %,d products loaded in %,d ms%n", prodLoaded, prodLoadMs);
        System.out.println("║──────────────────────────────────────────────────────║");
        System.out.printf("║  Total orders:        %,d%n", total);
        System.out.printf("║  Enriched output:     %,d%n", enriched);
        System.out.println("║──────────────────────────────────────────────────────║");
        System.out.printf("║  Customer HIT:  %,d (%.1f%%)%n", custHit, pct(custHit, custHit + custMiss));
        System.out.printf("║  Customer MISS: %,d (%.1f%%)%n", custMiss, pct(custMiss, custHit + custMiss));
        System.out.printf("║  Product HIT:   %,d (%.1f%%)%n", prodHit, pct(prodHit, prodHit + prodMiss));
        System.out.printf("║  Product MISS:  %,d (%.1f%%)%n", prodMiss, pct(prodMiss, prodHit + prodMiss));

        if (joinType.equals(JOIN_INNER)) {
            long match = getCounter(job, "INNER_JOIN_MATCH");
            long skip = getCounter(job, "INNER_JOIN_SKIP");
            System.out.println("║──────────────────────────────────────────────────────║");
            System.out.printf("║  INNER JOIN matched: %,d (%.1f%%)%n", match, pct(match, total));
            System.out.printf("║  INNER JOIN skipped: %,d (%.1f%%)%n", skip, pct(skip, total));
        } else {
            long full = getCounter(job, "LEFT_JOIN_FULL_MATCH");
            long partial = getCounter(job, "LEFT_JOIN_PARTIAL_MATCH");
            System.out.println("║──────────────────────────────────────────────────────║");
            System.out.printf("║  LEFT JOIN full match:    %,d (%.1f%%)%n", full, pct(full, total));
            System.out.printf("║  LEFT JOIN partial match: %,d (%.1f%%)%n", partial, pct(partial, total));
        }

        System.out.println("╚══════════════════════════════════════════════════════╝\n");
    }

    private long getCounter(Job job, String name) throws IOException {
        return job.getCounters()
                  .findCounter(JoinMapper.JoinCounters.valueOf(name))
                  .getValue();
    }

    private double pct(long part, long total) {
        return total == 0 ? 0.0 : part * 100.0 / total;
    }

    public static void main(String[] args) throws Exception {
        System.exit(ToolRunner.run(new Configuration(), new MapSideJoinJob(), args));
    }
}

import os

def cross_validate_join():
    """
    Independently perform the same join in Python
    and compare with Hadoop's output.
    """
    
    print("═" * 70)
    print("CROSS-VALIDATION: Python Join vs Hadoop Map-Side Join")
    print("═" * 70)
    
    # ── Load customer lookup ──
    customers = {}
    with open("/tmp/customers.txt", "r") as f:
        for line in f:
            line = line.strip()
            if line.startswith("CUSTOMER_ID") or not line:
                continue
            parts = line.split("|")
            if len(parts) >= 6:
                cid = parts[0].strip()
                customers[cid] = {
                    "name": parts[1].strip(),
                    "email": parts[2].strip(),
                    "tier": parts[3].strip(),
                    "state": parts[5].strip()
                }
    
    print(f"\n  Customers loaded: {len(customers):,}")
    
    # ── Load product lookup ──
    products = {}
    with open("/tmp/products.txt", "r") as f:
        for line in f:
            line = line.strip()
            if line.startswith("PRODUCT_ID") or not line:
                continue
            parts = line.split("|")
            if len(parts) >= 4:
                pid = parts[0].strip()
                products[pid] = {
                    "name": parts[1].strip(),
                    "category": parts[2].strip(),
                    "brand": parts[3].strip()
                }
    
    print(f"  Products loaded:  {len(products):,}")
    
    # ── Perform LEFT JOIN in Python ──
    python_left_results = []
    cust_hits = 0
    cust_misses = 0
    prod_hits = 0
    prod_misses = 0
    
    with open("/tmp/orders.txt", "r") as f:
        for line in f:
            line = line.strip()
            if line.startswith("ORDER_ID") or not line:
                continue
            
            parts = line.split("|")
            if len(parts) != 6:
                continue
            
            order_id = parts[0].strip()
            cust_id = parts[1].strip()
            prod_id = parts[2].strip()
            quantity = parts[3].strip()
            price = parts[4].strip()
            order_date = parts[5].strip()
            
            # Customer lookup
            cust = customers.get(cust_id)
            if cust:
                cust_hits += 1
                cust_name = cust["name"]
                cust_tier = cust["tier"]
                cust_state = cust["state"]
            else:
                cust_misses += 1
                cust_name = "UNKNOWN"
                cust_tier = "UNKNOWN"
                cust_state = "UNKNOWN"
            
            # Product lookup
            prod = products.get(prod_id)
            if prod:
                prod_hits += 1
                prod_name = prod["name"]
                prod_category = prod["category"]
                prod_brand = prod["brand"]
            else:
                prod_misses += 1
                prod_name = "UNKNOWN"
                prod_category = "UNKNOWN"
                prod_brand = "UNKNOWN"
            
            enriched = "|".join([
                order_id, cust_id, cust_name, cust_tier, cust_state,
                prod_id, prod_name, prod_category, prod_brand,
                quantity, price, order_date
            ])
            python_left_results.append(enriched)
    
    print(f"\n  Python LEFT JOIN results: {len(python_left_results):,}")
    print(f"  Customer hits:  {cust_hits:,} ({cust_hits * 100 / (cust_hits + cust_misses):.1f}%)")
    print(f"  Customer misses: {cust_misses:,}")
    print(f"  Product hits:   {prod_hits:,} ({prod_hits * 100 / (prod_hits + prod_misses):.1f}%)")
    print(f"  Product misses: {prod_misses:,}")
    
    # ── Compare with Hadoop output ──
    hadoop_output_file = "/tmp/hadoop_left_join_p16.txt"
    
    # Download Hadoop output
    os.system("hdfs dfs -getmerge /user/root/project16/output_left/part-m-* " + hadoop_output_file + " 2>/dev/null")
    
    if not os.path.exists(hadoop_output_file):
        print("\n  ⚠️  Cannot download Hadoop output for comparison")
        print("  Run the Hadoop job first, then re-run this script")
        return
    
    hadoop_results = []
    with open(hadoop_output_file, "r") as f:
        for line in f:
            line = line.strip()
            if line:
                hadoop_results.append(line)
    
    print(f"\n  Hadoop LEFT JOIN results: {len(hadoop_results):,}")
    
    # ── Record count comparison ──
    print(f"\n── RECORD COUNT ──")
    if len(python_left_results) == len(hadoop_results):
        print(f"  ✅ Record counts match: {len(python_left_results):,}")
    else:
        print(f"  ❌ Mismatch: Python={len(python_left_results):,}, Hadoop={len(hadoop_results):,}")
    
    # ── Content comparison ──
    print(f"\n── CONTENT COMPARISON ──")
    
    # Sort both for deterministic comparison
    python_sorted = sorted(python_left_results)
    hadoop_sorted = sorted(hadoop_results)
    
    mismatches = 0
    max_show = 5
    
    if len(python_sorted) == len(hadoop_sorted):
        for i, (py_line, hd_line) in enumerate(zip(python_sorted, hadoop_sorted)):
            if py_line != hd_line:
                mismatches += 1
                if mismatches <= max_show:
                    print(f"  Mismatch at record {i}:")
                    print(f"    Python: {py_line[:100]}")
                    print(f"    Hadoop: {hd_line[:100]}")
        
        if mismatches == 0:
            print(f"  ✅ ALL {len(python_sorted):,} RECORDS MATCH EXACTLY")
            print(f"     Python join and Hadoop Map-Side Join produce identical results")
        else:
            print(f"  ❌ {mismatches:,} mismatches found")
    else:
        # Compare by order ID
        python_by_id = {line.split("|")[0]: line for line in python_left_results}
        hadoop_by_id = {line.split("|")[0]: line for line in hadoop_results}
        
        only_python = set(python_by_id.keys()) - set(hadoop_by_id.keys())
        only_hadoop = set(hadoop_by_id.keys()) - set(python_by_id.keys())
        
        if only_python:
            print(f"  Records only in Python: {len(only_python)}")
            for oid in list(only_python)[:3]:
                print(f"    {oid}")
        
        if only_hadoop:
            print(f"  Records only in Hadoop: {len(only_hadoop)}")
            for oid in list(only_hadoop)[:3]:
                print(f"    {oid}")
        
        # Compare common records
        common_ids = set(python_by_id.keys()) & set(hadoop_by_id.keys())
        for oid in sorted(list(common_ids))[:1000]:
            if python_by_id[oid] != hadoop_by_id[oid]:
                mismatches += 1
                if mismatches <= max_show:
                    print(f"  Content mismatch for {oid}:")
                    print(f"    Python: {python_by_id[oid][:100]}")
                    print(f"    Hadoop: {hadoop_by_id[oid][:100]}")
        
        if mismatches == 0:
            print(f"  ✅ All {len(common_ids):,} common records match")
        else:
            print(f"  ❌ {mismatches:,} content mismatches in common records")
    
    # ── Cleanup ──
    os.remove(hadoop_output_file)
    
    # ── Performance estimation ──
    print(f"\n── PERFORMANCE ESTIMATE AT SCALE ──")
    print(f"  {'Scale':<25} {'Map-Side (cache)':<20} {'Reduce-Side (shuffle)'}")
    print(f"  {'-'*65}")
    
    scales = [
        ("10K orders (test)",       10_000,      0.07),
        ("1M orders",               1_000_000,   7),
        ("100M orders",             100_000_000, 700),
        ("800M orders (production)",800_000_000, 5600),
    ]
    
    cache_mb = 68  # customers + products
    
    for name, orders, input_gb in scales:
        # Map-side: cache distribution + input read
        map_side_network_gb = cache_mb * 50 / 1024  # 50 nodes
        map_side_minutes = input_gb * 0.5 + 0.5  # ~0.5 min/GB + overhead
        
        # Reduce-side: full shuffle

        reduce_side_minutes = input_gb * 1.5 + 2  # ~1.5 min/GB + overhead
        
        print(f"  {name:<25} {map_side_minutes:<6.1f} min ({map_side_network_gb:.1f}GB net)   "
              f"{reduce_side_minutes:<6.1f} min ({reduce_side_network_gb:.0f}GB net)")
    
    print(f"\n  CONCLUSION: Map-Side Join is 3-5× faster because:")
    print(f"    → Cache file (68MB) distributed once")
    print(f"    → Zero shuffle of order data")
    print(f"    → Zero reducer overhead")
    print(f"    → Advantage grows with order dataset size")

🔑 KEY TAKEAWAY

WHAT PROJECT 16 ESTABLISHED:

✅ MAP-SIDE JOIN = Distributed Cache + HashMap + Map-Only Job
   → Small dataset(s) in cache → loaded into HashMap(s) in setup()
   → Large dataset as normal MapReduce input
   → Each record enriched via O(1) HashMap lookups
   → Zero shuffle, zero reducers → maximum performance

✅ MULTI-TABLE JOINS:
   → Multiple cache files: job.addCacheFile() for each
   → Separate HashMap per lookup table
   → Each HashMap loaded independently in setup()
   → Combined memory must fit in mapper heap

✅ JOIN SEMANTICS:
   → INNER JOIN: skip records with ANY missing lookup
   → LEFT JOIN: emit all records, use UNKNOWN for misses
   → FULL OUTER JOIN: NOT possible with map-side join
   → Join type as configuration parameter → reusable job

✅ PERFORMANCE ADVANTAGE:
   → Network: 68MB cache vs 120GB shuffle (1,764× less)
   → Time: 3-5× faster than reduce-side for same workload
   → Advantage grows with larger main dataset

🌉 BRIDGE TO PROJECT 17

PROJECT 16 showed Map-Side Join — the fastest join strategy
when one dataset is small enough to cache.

But what happens when BOTH datasets are large?

PROJECT 17 asks: "We need to join 120GB of orders with 80GB
of product reviews. Neither fits in mapper memory. How do we
join them without distributed cache?"

This introduces:
→ Reduce-Side Join — the universal join strategy
→ Record tagging — marking records with their source
→ MultipleInputs — reading different datasets with different mappers
→ Reducer-based matching — grouping by join key
→ Custom grouping for multi-source data

The foundation: Project 16 showed WHY map-side is faster.
Project 17 shows WHAT to do when map-side isn't possible.
Together they form the complete Hadoop join toolkit.
