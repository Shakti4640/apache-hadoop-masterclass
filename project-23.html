PROJECT 23: TUNING MAP AND REDUCE TASK MEMORY TO STOP THE CLUSTER CRASHING

🔴 HYPOTHETICAL PROBLEM

SCENARIO:
→ You are a performance engineer at a logistics company
→ 50-node cluster managed by YARN (Project 22)
→ Three critical jobs keep FAILING or WASTING resources:

FAILURE 1: "Container killed by YARN for exceeding memory limits"
→ MapReduce job processing 800 million shipping records
→ Reduce tasks aggregate all shipments per warehouse (3000 warehouses)
→ Top 5 warehouses have 50 million records each
→ Reducer loads all values for one warehouse into memory
→ Container allocated: 2048MB
→ Reducer uses: 2400MB → KILLED by YARN
→ MapReduce retries 4 times → same result → JOB FAILS
→ Wasted: 3 hours of cluster time, zero output

FAILURE 2: "Job succeeds but wastes 60% of cluster memory"
→ MapReduce job processing 100 million small log entries
→ Each mapper processes a 128MB block → needs ~300MB RAM
→ Container allocated: 4096MB (someone set this "to be safe")
→ 200 mappers × (4096MB - 300MB wasted) = 759GB WASTED
→ Other teams' jobs sit in ACCEPTED state for hours
→ Cluster utilization: 40% real work, 60% wasted padding

FAILURE 3: "Job runs but is 5x slower than expected"
→ MapReduce job with 500 mappers and 20 reducers
→ Speculative execution enabled (default)
→ 3 mappers are slow (processing skewed input splits)
→ YARN launches 3 SPECULATIVE copies of those mappers
→ Now 503 mapper containers running → cluster overloaded
→ All mappers slow down due to resource contention
→ More speculative copies launched → cascade effect
→ Job that should take 30 minutes takes 2.5 hours

ROOT CAUSE OF ALL THREE:
→ Memory settings are WRONG
→ Nobody calculated the RIGHT container size
→ Nobody understood the relationship between:
    Container memory (YARN level)
    JVM heap memory (Java level)
    JVM overhead (metaspace, threads, buffers)
    Task-specific memory needs (data structures in your code)

STEP 2: Understand the CRITICAL RELATIONSHIP

  Container Memory = JVM Heap + JVM Overhead

  ┌───────────────────────────────────────────────────────────┐
  │                                                           │
  │  Container: 2048 MB (mapreduce.map.memory.mb)            │
  │  ┌────────────────────────────┬────────────────────────┐ │
  │  │      JVM Heap: 1536 MB    │  JVM Overhead: 512 MB  │ │
  │  │  (mapreduce.map.java.opts │  (implicit — NOT       │ │
  │  │   = -Xmx1536m)           │   configurable)        │ │
  │  │                           │                        │ │
  │  │  Your data structures,    │  Thread stacks,        │ │
  │  │  sort buffers,            │  metaspace, JIT cache, │ │
  │  │  cached data              │  GC overhead, NIO      │ │
  │  └────────────────────────────┴────────────────────────┘ │
  │                                                           │
  │  THE GOLDEN RULE:                                         │
  │  JVM Heap (-Xmx) should be ~75% of Container Memory     │
  │                                                           │
  │  Container 1024MB → -Xmx768m                             │
  │  Container 2048MB → -Xmx1536m                            │
  │  Container 4096MB → -Xmx3072m                            │
  │  Container 8192MB → -Xmx6144m                            │
  │                                                           │
  │  WHY 75%? The remaining 25% covers JVM overhead:         │
  │  → ~200-400MB for typical MapReduce tasks                │
  │  → More for tasks with many threads or NIO buffers       │
  │                                                           │
  └───────────────────────────────────────────────────────────┘

STEP 3: Understand what happens when the math is wrong

  SCENARIO A: Heap too close to container limit
    Container: 2048MB, Heap: -Xmx2000m
    JVM overhead: ~400MB
    Total: 2000 + 400 = 2400MB > 2048MB container
    RESULT: Container KILLED by YARN NodeManager
    ERROR: "Container [pid=12345,containerID=container_xxx]
            is running beyond physical memory limits.
            Current usage: 2.3 GB of 2 GB physical memory used.
            Killing container."

  SCENARIO B: Heap too small for the data
    Container: 2048MB, Heap: -Xmx512m
    Reducer loading 50M records into HashMap
    HashMap needs 1200MB
    RESULT: java.lang.OutOfMemoryError: Java heap space
    Container NOT killed by YARN — JVM crashes internally

  SCENARIO C: Container too large for the workload
    Container: 8192MB, Heap: -Xmx6144m
    Mapper processes 128MB block → needs 300MB heap
    RESULT: Job succeeds but wastes 5.8GB per mapper
    200 mappers × 5.8GB = 1.16TB wasted cluster-wide

💡 HOW CAN WE SOLVE IT

SOLUTION: Calculate memory settings based on WORKLOAD ANALYSIS

  ┌─────────────────────────────────────────────────────────────┐
  │         MEMORY CALCULATION FRAMEWORK                        │
  │                                                             │
  │  STEP A: Determine task memory needs                       │
  │    → What data structures does your code create?           │
  │    → How much data does each task process?                 │
  │    → Does it buffer all values or stream them?             │
  │                                                             │
  │  STEP B: Set JVM heap = task needs + 30% safety margin     │
  │    → If task needs 800MB → heap = 1024MB                  │
  │    → If task needs 1500MB → heap = 2048MB                 │
  │                                                             │
  │  STEP C: Set container = JVM heap / 0.75                   │
  │    → heap 1024MB → container = 1024 / 0.75 ≈ 1365        │
  │    → Round UP to allocation increment (usually 1024MB)     │
  │    → container = 2048MB                                    │
  │                                                             │
  │  STEP D: Set java.opts = -Xmx{heap}m                      │
  │    → container 2048MB → java.opts = -Xmx1536m             │
  │                                                             │
  │  STEP E: Verify against NodeManager capacity               │
  │    → Container must be ≤ yarn.scheduler.maximum-allocation │
  │    → Container must be ≤ yarn.nodemanager.resource.memory  │
  │                                                             │
  └─────────────────────────────────────────────────────────────┘

  FOR OUR THREE FAILURES:

  FAILURE 1 FIX (Reducer OOM on large warehouse groups):
    → Reducer needs ~2400MB for 50M records
    → JVM heap: 2400 × 1.3 = 3120MB → round to 3072MB
    → Container: 3072 / 0.75 = 4096MB
    → mapreduce.reduce.memory.mb = 4096
    → mapreduce.reduce.java.opts = -Xmx3072m
    → BETTER FIX: Don't buffer all values — use Secondary Sort

  FAILURE 2 FIX (Mapper over-allocated):

    → JVM heap: 300 × 1.3 = 390MB → round to 512MB
    → Container: 512 / 0.75 = 683MB → round to 1024MB
    → mapreduce.map.memory.mb = 1024
    → mapreduce.map.java.opts = -Xmx768m
    → Savings: (4096-1024) × 200 = 614GB freed for other jobs

  FAILURE 3 FIX (Speculative execution cascade):
    → Disable speculative execution for this job
    → mapreduce.map.speculative = false
    → mapreduce.reduce.speculative = false
    → OR: Fix the root cause — skewed input splits

✅ WHAT SHOULD DO

1. ALWAYS SET BOTH container memory AND java.opts TOGETHER

   ┌────────────────────────────────────────────────────────────┐
   │  MAPPER MEMORY SETTINGS (must be paired):                  │
   │                                                            │
   │  mapreduce.map.memory.mb = 2048      ← YARN container    │
   │  mapreduce.map.java.opts = -Xmx1536m ← JVM heap inside   │
   │                                                            │
   │  REDUCER MEMORY SETTINGS (must be paired):                 │
   │                                                            │
   │  mapreduce.reduce.memory.mb = 4096   ← YARN container    │
   │  mapreduce.reduce.java.opts = -Xmx3072m ← JVM heap       │
   │                                                            │
   │  APPLICATION MASTER MEMORY:                                │
   │                                                            │
   │  yarn.app.mapreduce.am.resource.mb = 2048                 │
   │  yarn.app.mapreduce.am.command-opts = -Xmx1536m           │
   │                                                            │
   │  RULE: java.opts Xmx = container.mb × 0.75                │
   └────────────────────────────────────────────────────────────┘

2. SET DIFFERENT MEMORY FOR MAPPERS AND REDUCERS
   → Mappers usually need LESS memory (processing one block)
   → Reducers often need MORE memory (aggregating across blocks)
   → Don't use same settings for both unless workload is identical

   TYPICAL PATTERN:
     Mapper:  container=1024MB, heap=768MB  (light processing)
     Reducer: container=2048MB, heap=1536MB (aggregation)

3. PROFILE BEFORE TUNING
   → Run job with generous memory first
   → Check ACTUAL memory usage in YARN UI or counters:
     - Physical Memory Snapshot (bytes): peak RSS
     - Virtual Memory Snapshot (bytes): peak virtual
     - COMMITTED_HEAP_BYTES counter: heap committed by JVM
   → Then right-size based on actual usage

4. ACCOUNT FOR SORT BUFFER MEMORY
   → MapReduce sort buffer (map side):
     mapreduce.task.io.sort.mb = 100 (default)
     This is INSIDE the JVM heap
     Increase for jobs with heavy map output
   → If sort buffer too small: excessive disk spills → slow job
   → If sort buffer too large: less heap for your data structures

5. ROUND CONTAINER SIZE TO ALLOCATION INCREMENT
   → YARN allocates in increments (minimum-allocation-mb)
   → If minimum = 1024MB and you request 1500MB
   → YARN rounds UP to 2048MB
   → Request exactly on boundaries to avoid waste:
     1024, 2048, 3072, 4096, 5120, 6144, 7168, 8192

6. CONSIDER THE CLUSTER-WIDE IMPACT
   → Larger containers = fewer containers per node
   → Fewer containers = fewer parallel tasks
   → More parallelism (small containers) vs more memory (large containers)
   → BALANCE based on workload characteristics

🚫 WHAT SHOULD NOT DO

1. DO NOT set -Xmx equal to container memory
   → Container: 2048MB, -Xmx2048m
   → JVM overhead (~400MB) pushes total past container limit
   → Container KILLED by YARN → task fails → job may fail
   → ALWAYS leave 25% headroom for JVM overhead

2. DO NOT set -Xmx LARGER than container memory
   → Container: 2048MB, -Xmx3072m
   → JVM starts fine (heap not fully used initially)
   → As heap grows toward 3072MB → exceeds 2048MB container
   → Container KILLED mid-execution (worst timing)
   → Wasted hours of work

3. DO NOT blindly increase memory when job fails with OOM
   → First question: WHY is memory needed?
   → If reducer buffers all values → fix the algorithm (Project 19)
   → If Distributed Cache HashMap too large → use Reduce-Side Join (Project 17)
   → If sort buffer too small → increase io.sort.mb specifically
   → Throwing memory at a bad algorithm is expensive and fragile

4. DO NOT use the same memory settings for all jobs
   → "Company standard: all containers 4096MB"
   → Light ETL mapper needs 500MB → wastes 3596MB per container
   → Heavy aggregation reducer needs 6000MB → still fails
   → TUNE PER JOB based on workload analysis

5. DO NOT forget to tune ApplicationMaster memory for large jobs
   → Default AM memory: 1536MB
   → AM tracks state for ALL tasks in the job
   → Job with 10,000 mappers → AM needs significant memory
   → AM OOM → entire job fails after all tasks complete
   → RULE: For >5000 tasks, set AM memory to 4096MB+

6. DO NOT enable speculative execution for ALL jobs blindly
   → Speculative execution launches DUPLICATE tasks
   → Good for: cluster with heterogeneous hardware (slow nodes)
   → Bad for: jobs that write to external systems (duplicate writes)
   → Bad for: resource-constrained clusters (wastes containers)
   → Bad for: reduce tasks (expensive to re-shuffle)
   → RULE: Enable for mappers on heterogeneous clusters
   → RULE: Disable for reducers in most cases

7. DO NOT ignore virtual memory settings
   → Java uses significantly more virtual memory than physical
   → Default vmem-pmem-ratio: 2.1
   → Container 2048MB physical → 4300MB virtual limit
   → Java with compressed oops, NIO buffers → can exceed virtual limit
   → RECOMMENDATION: Disable vmem check (Project 22 did this)
   → yarn.nodemanager.vmem-check-enabled = false

🚫 WHEN NOT TO USE

1. DO NOT TUNE MEMORY before understanding the algorithm
   → If reducer collects all values into ArrayList → fix the code first
   → Secondary Sort (Project 19) eliminates in-memory sorting
   → Combiner (Project 9) reduces data before reducer
   → Tuning memory is a BAND-AID for algorithmic problems

2. DO NOT TUNE on pseudo-distributed mode and expect same on cluster
   → Pseudo-distributed: one JVM per daemon, shared physical RAM
   → Cluster: separate machines, different OS overhead
   → ALWAYS validate tuning on actual cluster or representative subset

3. DO NOT OVER-TUNE for rare edge cases
   → If 1 out of 3000 warehouses has 50M records (skew)
   → Don't set ALL reducer containers to 8192MB
   → Fix the skew (Project 33) or use Secondary Sort (Project 19)
   → Right-size for the COMMON case, handle outliers differently

4. DO NOT USE speculative execution when:
   → Jobs have side effects (writing to databases, sending emails)
   → Cluster is at >80% utilization (no spare capacity for specs)
   → You have data skew (speculative copies also hit same skew)
   → Reduce tasks are involved (shuffle is expensive to redo)

  │                                                         │
  │  IF io.sort.mb too SMALL:                              │
  │    → Many spills to disk (e.g., 50 spill files)       │
  │    → Each spill = disk write                           │
  │    → Merge phase reads all spills back = more disk I/O │
  │    → SLOW mapper due to excessive disk I/O             │
  │                                                         │
  │  IF io.sort.mb too LARGE:                              │
  │    → Fewer spills (maybe just 1-2)                    │
  │    → BUT less heap for your data structures           │
  │    → Risk of OOM if your code also needs memory       │
  │                                                         │
  │  IDEAL: io.sort.mb sized so mapper produces 1-2 spills│
  │  CALCULATION:                                          │
  │    mapper_output_size ≈ input_split × selectivity     │
  │    io.sort.mb ≈ mapper_output_size / 2                │
  │    (2 spills is good balance)                          │
  └─────────────────────────────────────────────────────────┘

═══════════════════════════════════════════════════════════════
INTERNAL MECHANIC 4: HOW SPECULATIVE EXECUTION WORKS
═══════════════════════════════════════════════════════════════

  ApplicationMaster tracks task progress:

  ┌─────────────────────────────────────────────────────────┐
  │  TASK PROGRESS TRACKING (inside MRAppMaster)            │
  │                                                         │
  │  Mapper 001: progress=0.95 (95% done)   ← normal      │
  │  Mapper 002: progress=0.90              ← normal       │
  │  Mapper 003: progress=0.20              ← STRAGGLER!   │
  │  Mapper 004: progress=0.88              ← normal       │
  │  ...                                                    │
  │  Average progress: 0.87                                │
  │                                                         │
  │  SPECULATIVE EXECUTION ALGORITHM:                      │
  │                                                         │
  │  For each running task:                                │
  │    estimated_finish = start_time +                     │
  │      (current_time - start_time) / progress            │
  │                                                         │
  │    Mapper 003:                                         │
  │      Running for: 30 minutes                           │
  │      Progress: 0.20                                    │
  │      Estimated finish: 30 / 0.20 = 150 minutes total  │
  │      Estimated remaining: 120 minutes                  │
  │                                                         │
  │    Average mapper: finishes in ~35 minutes             │
  │                                                         │
  │    IF estimated_remaining >> average_remaining:         │
  │      → Launch SPECULATIVE COPY on a different node     │
  │      → BOTH original and copy run simultaneously       │
  │      → Whichever finishes FIRST wins                   │
  │      → The other is KILLED                             │
  │                                                         │
  │  RESOURCE COST:                                         │
  │    → Speculative copy needs its own container           │
  │    → Same memory + CPU as original                     │
  │    → Effectively DOUBLES resource usage for that task   │
  │                                                         │
  │  WHEN IT HELPS:                                         │
  │    → Straggler caused by slow hardware (bad disk)      │
  │    → Speculative copy on healthy node finishes faster   │
  │    → Job completes minutes earlier                     │
  │                                                         │
  │  WHEN IT HURTS:                                         │
  │    → Straggler caused by DATA SKEW (50M records)       │
  │    → Speculative copy processes SAME skewed data       │
  │    → Also slow → wasted container for no benefit       │
  │    → Plus: stole a container from another useful task  │
  └─────────────────────────────────────────────────────────┘

═══════════════════════════════════════════════════════════════
INTERNAL MECHANIC 5: REDUCE-SIDE MEMORY DURING SHUFFLE
═══════════════════════════════════════════════════════════════

  Reducer has THREE memory-consuming phases:

  PHASE 1 — SHUFFLE (Copy):
    → Pulls map output from completed mappers over network
    → Buffers in memory up to threshold, then spills to disk
    → Config: mapreduce.reduce.shuffle.input.buffer.percent = 0.70
    → Meaning: 70% of reducer heap used for shuffle buffer
    → For heap 1536MB: 1075MB for shuffle buffer

  PHASE 2 — SORT (Merge):
    → Merges shuffled data (memory + disk spills)
    → Config: mapreduce.reduce.merge.inmem.threshold
    → Disk-backed merge sort (Project 19 internal mechanics)

  PHASE 3 — REDUCE (Your code):
    → Iterates through sorted key-value groups
    → YOUR code's memory needs apply here
    → Config: mapreduce.reduce.input.buffer.percent = 0.0
    → Default 0.0: ALL shuffle data flushed to disk before reduce
    → Increase to keep some in memory (faster but riskier)

  ┌────────────────────────────────────────────────────────┐
  │  REDUCER HEAP: 1536 MB                                 │
  │                                                        │
  │  During SHUFFLE phase:                                │
  │  ┌──────────────────────────┬────────────────────┐    │
  │  │ Shuffle buffer: 1075 MB │ Other: 461 MB      │    │
  │  │ (70% of heap)           │ (GC, framework)    │    │
  │  └──────────────────────────┴────────────────────┘    │
  │                                                        │
  │  During REDUCE phase:                                 │
  │  ┌──────────────────────────┬────────────────────┐    │
  │  │ Your code: ~1200 MB     │ Framework: ~336 MB │    │
  │  │ (data structures,       │ (I/O buffers,      │    │
  │  │  accumulators)          │  serialization)    │    │
  │  └──────────────────────────┴────────────────────┘    │
  │                                                        │
  │  KEY INSIGHT: Shuffle and reduce DON'T overlap         │
  │  Shuffle buffer is released before reduce phase starts │
  │  So peak memory = max(shuffle_peak, reduce_peak)       │
  └────────────────────────────────────────────────────────┘

⚠️ COMMON PITFALLS AND FAILURE MODES

PITFALL 1: "Container killed" but OOM error NOT in logs
→ CAUSE: Physical memory exceeded but no Java OOM
→ JVM heap was fine, but off-heap (native memory) exceeded container
→ Common with large NIO buffers, snappy compression native libs
→ DIAGNOSIS: Check NodeManager logs for container kill message
→ FIX: Increase container memory without increasing -Xmx
   Container: 3072MB, -Xmx1536m (50% ratio instead of 75%)
   Gives 1536MB for off-heap — generous for native memory

PITFALL 2: "java.lang.OutOfMemoryError: Java heap space" in reducer
→ CAUSE: Reducer buffering all values for a key into memory
→ Common pattern: collecting into ArrayList then processing
→ DIAGNOSIS: Check which key has the most values (data skew)
→ FIX A: Use Secondary Sort (Project 19) to stream values
→ FIX B: Use Combiner (Project 9) to pre-aggregate
→ FIX C: Last resort — increase heap, but address root cause

PITFALL 3: "GC overhead limit exceeded"
→ CAUSE: JVM spending >98% of time in garbage collection
→ Usually means heap is JUST BARELY enough — constantly full
→ DIAGNOSIS: Add -verbose:gc to java.opts to see GC behavior
→ FIX: Increase heap by 50% or reduce data structures size
→ java.opts = "-Xmx1536m -verbose:gc -XX:+PrintGCDetails"

PITFALL 4: "Mapper succeeds but is 10x slower than expected"
→ CAUSE: io.sort.mb too small → dozens of disk spills
→ Each spill = sort + write to disk
→ Merge phase reads all spills = massive disk I/O
→ DIAGNOSIS: Check counter "Spilled Records" — if >> Map output records
   it means data was spilled multiple times
→ FIX: Increase mapreduce.task.io.sort.mb
→ FORMULA: io.sort.mb ≈ estimated_map_output / 2

PITFALL 5: "Job takes 2x longer with speculative execution"
→ CAUSE: Speculative copies consume containers needed by real tasks
→ On a busy cluster: speculative copy takes a container from reducer
→ Reducer delayed → job delayed
→ DIAGNOSIS: Check YARN UI for speculative task attempts
→ FIX: Disable speculative execution:
   mapreduce.map.speculative = false
   mapreduce.reduce.speculative = false

PITFALL 6: "Container memory set correctly but job STILL killed"
→ CAUSE: Memory-mapped files or native libraries not counted in heap
→ Snappy/LZ4 compression libraries use native memory
→ HDFS client uses NIO direct buffers
→ These are OUTSIDE -Xmx but INSIDE container physical memory
→ FIX: Increase container-to-heap ratio from 75% to 60%
   Container 4096MB → -Xmx2458m (60%) instead of -Xmx3072m (75%)

    → 1024MB containers: all 600 run in 1 wave → fast

    → NET EFFECT depends on whether job is memory-bound or CPU-bound

  MEMORY BUDGET ALLOCATION:
  ┌──────────────────────────────────┬─────────────┐
  │ Component                        │ Memory      │

  Extra resources: 25 × 2048MB = 50GB
  Cluster impact: 50GB / 2500GB = 2% overhead
  → Acceptable for potential 30% speedup if stragglers are HW-caused
  → Wasteful if stragglers are data-skew-caused (copies also slow)

import org.apache.hadoop.conf.Configuration;
import org.apache.hadoop.conf.Configured;
import org.apache.hadoop.fs.Path;
import org.apache.hadoop.io.IntWritable;
import org.apache.hadoop.io.LongWritable;
import org.apache.hadoop.io.Text;
import org.apache.hadoop.mapreduce.Job;
import org.apache.hadoop.mapreduce.Mapper;
import org.apache.hadoop.mapreduce.Reducer;
import org.apache.hadoop.mapreduce.lib.input.FileInputFormat;
import org.apache.hadoop.mapreduce.lib.output.FileOutputFormat;
import org.apache.hadoop.util.Tool;
import org.apache.hadoop.util.ToolRunner;

import java.io.IOException;

public class MemoryTuningDemo extends Configured implements Tool {

    // ══════════════════════════════════════════════════════
    // MAPPER: Process shipping records, emit (warehouse, 1)
    // Memory profile: LIGHT — processes one line at a time
    // ══════════════════════════════════════════════════════

    public static class ShipmentMapper
            extends Mapper<LongWritable, Text, Text, IntWritable> {

        private Text warehouseKey = new Text();
        private static final IntWritable ONE = new IntWritable(1);

        @Override
        protected void map(LongWritable offset, Text line, Context context)
                throws IOException, InterruptedException {

            String record = line.toString().trim();
            if (record.isEmpty() || record.startsWith("SHIPMENT_ID")) return;

            // Format: SHIPMENT_ID | WAREHOUSE | DATE | WEIGHT | DESTINATION
            String[] fields = record.split("\\|");
            if (fields.length < 5) {
                context.getCounter("QUALITY", "MALFORMED").increment(1);
                return;
            }

            String warehouse = fields[1].trim();
            warehouseKey.set(warehouse);
            context.write(warehouseKey, ONE);

            // Track memory for analysis
            if (context.getCounter("QUALITY", "VALID").getValue() % 1000000 == 0) {
                Runtime rt = Runtime.getRuntime();
                long usedMB = (rt.totalMemory() - rt.freeMemory()) / (1024 * 1024);
                long maxMB = rt.maxMemory() / (1024 * 1024);
                context.setStatus(String.format(
                    "Heap: %dMB / %dMB (%.1f%%)",
                    usedMB, maxMB, (double) usedMB / maxMB * 100));
            }

            context.getCounter("QUALITY", "VALID").increment(1);
        }
    }

    // ══════════════════════════════════════════════════════
    // COMBINER: Pre-aggregate counts per warehouse per mapper
    // Dramatically reduces data flowing to reducer
    // (Project 9 pattern — reduces reducer memory pressure)
    // ══════════════════════════════════════════════════════

    public static class ShipmentCombiner
            extends Reducer<Text, IntWritable, Text, IntWritable> {

        private IntWritable partialSum = new IntWritable();

        @Override
        protected void reduce(Text warehouse, Iterable<IntWritable> counts,
                              Context context)
                throws IOException, InterruptedException {

            int sum = 0;
            for (IntWritable count : counts) {
                sum += count.get();
            }
            partialSum.set(sum);
            context.write(warehouse, partialSum);
        }
    }

    // ══════════════════════════════════════════════════════
    // REDUCER: Aggregate warehouse shipment counts
    // Memory profile: LIGHT with Combiner (streaming sum)
    //                 HEAVY without Combiner (many values per key)
    // ══════════════════════════════════════════════════════

    public static class ShipmentReducer
            extends Reducer<Text, IntWritable, Text, Text> {

        private Text resultValue = new Text();

        @Override
        protected void reduce(Text warehouse, Iterable<IntWritable> counts,
                              Context context)
                throws IOException, InterruptedException {

            long totalShipments = 0;

            // STREAMING aggregation — O(1) memory regardless of group size
            // This is the CORRECT pattern — no ArrayList buffering
            for (IntWritable count : counts) {
                totalShipments += count.get();
            }

            // Track memory periodically
            Runtime rt = Runtime.getRuntime();
            long usedMB = (rt.totalMemory() - rt.freeMemory()) / (1024 * 1024);
            long maxMB = rt.maxMemory() / (1024 * 1024);

            resultValue.set(String.format(
                "shipments=%d | reducer_heap=%dMB/%dMB",
                totalShipments, usedMB, maxMB));

            context.write(warehouse, resultValue);

            // Warn if heap usage is high
            double heapPct = (double) usedMB / maxMB * 100;
            if (heapPct > 80) {
                context.getCounter("MEMORY", "HIGH_HEAP_REDUCERS").increment(1);
            }

            context.getCounter("OUTPUT", "WAREHOUSES").increment(1);
        }
    }

    // ══════════════════════════════════════════════════════
    // DRIVER: Configure memory settings based on profile
    // ══════════════════════════════════════════════════════

    @Override
    public int run(String[] args) throws Exception {

        if (args.length < 4) {
            System.err.println(
                "Usage: MemoryTuningDemo <input> <output> <numReducers> <profile>");
            System.err.println("  profile: light | medium | heavy | custom");
            System.err.println();
            System.err.println("  light:  Mapper 1024MB, Reducer 1024MB (small data)");
            System.err.println("  medium: Mapper 1024MB, Reducer 2048MB (moderate aggregation)");
            System.err.println("  heavy:  Mapper 2048MB, Reducer 4096MB (large groups)");
            System.err.println("  custom: Read from -D properties");
            return -1;
        }

        String inputPath = args[0];
        String outputPath = args[1];
        int numReducers = Integer.parseInt(args[2]);
        String profile = args[3].toLowerCase();

        Configuration conf = getConf();
        Job job = Job.getInstance(conf, "Memory Tuning Demo — " + profile);
        job.setJarByClass(MemoryTuningDemo.class);

        // Mapper + Combiner + Reducer
        job.setMapperClass(ShipmentMapper.class);
        job.setCombinerClass(ShipmentCombiner.class);
        job.setReducerClass(ShipmentReducer.class);

        // Types
        job.setMapOutputKeyClass(Text.class);
        job.setMapOutputValueClass(IntWritable.class);
        job.setOutputKeyClass(Text.class);
        job.setOutputValueClass(Text.class);

        // I/O
        FileInputFormat.addInputPath(job, new Path(inputPath));
        FileOutputFormat.setOutputPath(job, new Path(outputPath));
        job.setNumReduceTasks(numReducers);

        // ════════════════════════════════════════════════════════
        // MEMORY CONFIGURATION — THE CORE OF THIS PROJECT
        // ════════════════════════════════════════════════════════

        switch (profile) {
            case "light":
                configureLight(job);
                break;
            case "medium":
                configureMedium(job);
                break;
            case "heavy":
                configureHeavy(job);
                break;
            case "custom":
                configureCustom(job);
                break;
            default:
                System.err.println("Unknown profile: " + profile);
                return -1;
        }

        // Print configuration summary
        printMemoryConfig(job, profile);

        boolean success = job.waitForCompletion(true);

        if (success) {
            printMemoryReport(job);
        }

        return success ? 0 : 1;
    }

    // ════════════════════════════════════════════════════════
    // MEMORY PROFILES
    // ════════════════════════════════════════════════════════

    private void configureLight(Job job) {
        Configuration conf = job.getConfiguration();

        // MAPPER: Light — text parsing, no caching
        conf.setInt("mapreduce.map.memory.mb", 1024);
        conf.set("mapreduce.map.java.opts", "-Xmx768m");
        // Ratio: 768/1024 = 75% — standard

        // REDUCER: Light — streaming aggregation with Combiner
        conf.setInt("mapreduce.reduce.memory.mb", 1024);
        conf.set("mapreduce.reduce.java.opts", "-Xmx768m");

        // Sort buffer: small output per mapper
        conf.setInt("mapreduce.task.io.sort.mb", 100);

        // Speculative: ON for light workloads (fast tasks, low risk)
        conf.setBoolean("mapreduce.map.speculative", true);
        conf.setBoolean("mapreduce.reduce.speculative", false);

        // AM: default is sufficient
        conf.setInt("yarn.app.mapreduce.am.resource.mb", 1536);
        conf.set("yarn.app.mapreduce.am.command-opts", "-Xmx1024m");
    }

    private void configureMedium(Job job) {
        Configuration conf = job.getConfiguration();

        // MAPPER: Medium — still light per-record processing
        conf.setInt("mapreduce.map.memory.mb", 1024);
        conf.set("mapreduce.map.java.opts", "-Xmx768m");

        // REDUCER: Medium — moderate aggregation
        conf.setInt("mapreduce.reduce.memory.mb", 2048);
        conf.set("mapreduce.reduce.java.opts", "-Xmx1536m");
        // Ratio: 1536/2048 = 75%

        // Sort buffer: moderate map output
        conf.setInt("mapreduce.task.io.sort.mb", 150);

        // Speculative: ON for mappers, OFF for reducers
        conf.setBoolean("mapreduce.map.speculative", true);
        conf.setBoolean("mapreduce.reduce.speculative", false);

        // Shuffle tuning
        conf.setFloat("mapreduce.reduce.shuffle.input.buffer.percent", 0.70f);

        // AM: default
        conf.setInt("yarn.app.mapreduce.am.resource.mb", 1536);
        conf.set("yarn.app.mapreduce.am.command-opts", "-Xmx1024m");
    }

    private void configureHeavy(Job job) {
        Configuration conf = job.getConfiguration();

        // MAPPER: Heavy — large Distributed Cache or complex processing
        conf.setInt("mapreduce.map.memory.mb", 2048);
        conf.set("mapreduce.map.java.opts", "-Xmx1536m");

        // REDUCER: Heavy — large groups, significant aggregation
        conf.setInt("mapreduce.reduce.memory.mb", 4096);
        conf.set("mapreduce.reduce.java.opts", "-Xmx3072m");
        // Ratio: 3072/4096 = 75%

        // Sort buffer: large map output expected
        conf.setInt("mapreduce.task.io.sort.mb", 256);

        // Speculative: OFF — heavy tasks waste too many resources if duplicated
        conf.setBoolean("mapreduce.map.speculative", false);
        conf.setBoolean("mapreduce.reduce.speculative", false);

        // Shuffle tuning: give more buffer for large shuffle
        conf.setFloat("mapreduce.reduce.shuffle.input.buffer.percent", 0.80f);
        conf.setInt("mapreduce.reduce.shuffle.parallelcopies", 10);

        // AM: larger for potentially many tasks
        conf.setInt("yarn.app.mapreduce.am.resource.mb", 2048);
        conf.set("yarn.app.mapreduce.am.command-opts", "-Xmx1536m");

        // Slow start: reducers wait for more mappers before starting
        // Prevents reducers from sitting idle during shuffle
        conf.setFloat("mapreduce.job.reduce.slowstart.completedmaps", 0.80f);
    }

    private void configureCustom(Job job) {
        // Custom profile reads from -D properties passed at command line
        // Example:
        //   hadoop jar ... -Dmapreduce.map.memory.mb=3072 \
        //                  -Dmapreduce.map.java.opts="-Xmx2304m" \
        //                  -Dmapreduce.reduce.memory.mb=6144 \

    private int extractXmx(String javaOpts) {
        // Extract -Xmx value from string like "-Xmx1536m -verbose:gc"
        if (javaOpts == null) return 0;
        String[] parts = javaOpts.split("\\s+");
        for (String part : parts) {
            if (part.startsWith("-Xmx")) {
                String value = part.substring(4);
                if (value.endsWith("m") || value.endsWith("M")) {
                    return Integer.parseInt(value.substring(0, value.length() - 1));
                } else if (value.endsWith("g") || value.endsWith("G")) {
                    return Integer.parseInt(value.substring(0, value.length() - 1)) * 1024;
                }
            }
        }
        return 0;
    }

import math

def calculate_memory_settings(
    task_type,
    estimated_data_mb,
    uses_distributed_cache_mb=0,
    uses_in_memory_collections=False,
    num_threads=1,
    uses_compression=False,
    node_memory_gb=64,
    node_yarn_memory_gb=50,
    node_vcores=12
):
    """
    Calculate optimal YARN container and JVM heap settings.
    
    Parameters:
    -----------
    task_type: "mapper" or "reducer"
    estimated_data_mb: Estimated data processed per task (MB)
    uses_distributed_cache_mb: Size of Distributed Cache loaded into memory
    uses_in_memory_collections: Whether task buffers data in collections
    num_threads: Number of threads per task (usually 1)
    uses_compression: Whether compression codecs are used (adds native memory)
    node_memory_gb: Physical RAM per node
    node_yarn_memory_gb: RAM allocated to YARN per node
    node_vcores: VCores per node
    """
    
    print(f"\n{'='*60}")
    print(f"MEMORY CALCULATION: {task_type.upper()}")
    print(f"{'='*60}")
    
    # Step 1: Estimate base memory need
    if task_type == "mapper":
        # Mapper processes one input split
        # Memory = input split overhead + sort buffer + framework
        base_data_mb = estimated_data_mb * 0.5  # ~50% of input held in memory
        sort_buffer_mb = min(estimated_data_mb * 0.6, 512)  # Optimal sort buffer
        framework_mb = 100  # MapReduce framework overhead
    else:
        # Reducer receives shuffled data
        # Memory = shuffle buffer + reduce data structures + framework
        base_data_mb = estimated_data_mb * 0.3  # Streaming reduces need less
        if uses_in_memory_collections:
            base_data_mb = estimated_data_mb * 1.5  # Buffering all values
        sort_buffer_mb = 0  # No sort buffer on reduce side
        framework_mb = 150  # Reducer framework slightly heavier
    
    # Step 2: Add Distributed Cache overhead
    cache_mb = uses_distributed_cache_mb
    
    # Step 3: Add thread overhead
    thread_mb = num_threads * 1  # ~1MB stack per thread
    
    # Step 4: Calculate heap need
    heap_need_mb = base_data_mb + sort_buffer_mb + cache_mb + framework_mb + thread_mb
    
    # Step 5: Add 30% safety margin
    heap_with_margin_mb = heap_need_mb * 1.3
    
    # Step 6: Round up to nice number
    heap_mb = int(math.ceil(heap_with_margin_mb / 256) * 256)
    heap_mb = max(heap_mb, 512)  # Minimum 512MB heap
    
    # Step 7: Calculate container size (heap / 0.75)
    container_mb_exact = heap_mb / 0.75
    
    # Step 8: Round up to allocation increment (1024MB)
    container_mb = int(math.ceil(container_mb_exact / 1024) * 1024)
    container_mb = max(container_mb, 1024)  # Minimum 1024MB container
    
    # Step 9: Recalculate heap to fit container at 75%
    final_heap_mb = int(container_mb * 0.75)
    
    # Step 10: Add native memory for compression
    if uses_compression:
        native_overhead_mb = 200  # Snappy/LZ4 native libraries
    else:
        native_overhead_mb = 50  # Minimal native overhead
    
    jvm_overhead_mb = container_mb - final_heap_mb
    
    # Step 11: Verify fits on node
    max_container_per_node_by_mem = int(node_yarn_memory_gb * 1024 / container_mb)
    max_container_per_node_by_vcore = node_vcores
    max_containers_per_node = min(max_container_per_node_by_mem,
                                  max_container_per_node_by_vcore)
    
    # Print calculation breakdown
    print(f"\n  INPUT PARAMETERS:")
    print(f"    Estimated data per task:     {estimated_data_mb} MB")
    print(f"    Distributed Cache size:      {uses_distributed_cache_mb} MB")
    print(f"    In-memory collections:       {'Yes' if uses_in_memory_collections else 'No'}")
    print(f"    Compression:                 {'Yes' if uses_compression else 'No'}")
    
    print(f"\n  CALCULATION BREAKDOWN:")
    print(f"    Base data memory:            {base_data_mb:.0f} MB")
    if task_type == "mapper":
        print(f"    Sort buffer (io.sort.mb):    {sort_buffer_mb:.0f} MB")
    print(f"    Distributed Cache:           {cache_mb} MB")
    print(f"    Framework overhead:          {framework_mb} MB")
    print(f"    Thread stacks:               {thread_mb} MB")
    print(f"    ────────────────────────────────────────")
    print(f"    Raw heap need:               {heap_need_mb:.0f} MB")
    print(f"    + 30% safety margin:         {heap_with_margin_mb:.0f} MB")
    print(f"    Rounded heap:                {heap_mb} MB")
    
    print(f"\n  RECOMMENDED SETTINGS:")
    print(f"    ┌────────────────────────────────────────────────────┐")
    print(f"    │ Container:  {container_mb} MB                       ")
    print(f"    │ JVM Heap:   -Xmx{final_heap_mb}m                   ")
    print(f"    │ JVM Overhead: {jvm_overhead_mb} MB (implicit)       ")
    print(f"    │ Heap Ratio: {final_heap_mb/container_mb*100:.0f}%   ")
    print(f"    └────────────────────────────────────────────────────┘")
    
    if task_type == "mapper":
        print(f"\n  CONFIGURATION LINES:")
        print(f"    mapreduce.map.memory.mb = {container_mb}")
        print(f"    mapreduce.map.java.opts = -Xmx{final_heap_mb}m")
        print(f"    mapreduce.task.io.sort.mb = {int(sort_buffer_mb)}")
    else:
        print(f"\n  CONFIGURATION LINES:")
        print(f"    mapreduce.reduce.memory.mb = {container_mb}")
        print(f"    mapreduce.reduce.java.opts = -Xmx{final_heap_mb}m")
    
    print(f"\n  NODE CAPACITY IMPACT:")
    print(f"    Max containers per node:     {max_containers_per_node}")
    print(f"    (limited by {'memory' if max_container_per_node_by_mem < max_container_per_node_by_vcore else 'vcores'})")
    
    return {
        "container_mb": container_mb,
        "heap_mb": final_heap_mb,
        "sort_mb": int(sort_buffer_mb) if task_type == "mapper" else 0,
        "max_per_node": max_containers_per_node
    }

🔑 KEY TAKEAWAY

WHAT PROJECT 23 ESTABLISHED:

✅ PROBLEM: Wrong memory settings cause THREE failure modes:
   1. Container killed by YARN (heap + overhead > container)
   2. Java OOM inside JVM (data structures exceed heap)
   3. Resource waste (containers over-provisioned)

✅ THE THREE MEMORY LAYERS:
   Layer 1: YARN Container (mapreduce.{map|reduce}.memory.mb)
     → Hard limit enforced by NodeManager monitor
     → Container KILLED if physical memory exceeds this
   Layer 2: JVM Heap (mapreduce.{map|reduce}.java.opts = -Xmx)
     → Heap for Java objects, sort buffers, your data
     → OutOfMemoryError if exceeded
   Layer 3: JVM Overhead (implicit, NOT configurable)
     → Thread stacks, metaspace, JIT cache, NIO buffers
     → ~25% of container memory needed for this

✅ THE GOLDEN RULE:
   JVM Heap (-Xmx) = Container Memory × 0.75
   → 25% headroom for JVM overhead
   → Going above 85% risks container kill
   → Going below 50% wastes cluster resources

✅ SORT BUFFER (mapreduce.task.io.sort.mb):
   → Lives INSIDE the JVM heap
   → Too small → excessive disk spills → slow job
   → Too large → less heap for your code → OOM risk
   → Optimal: produces 1-2 spill files per mapper

✅ SPECULATIVE EXECUTION:
   → Launches duplicate tasks for stragglers
   → Helps: slow hardware (healthy copy finishes first)
   → Hurts: data skew (copy equally slow), resource waste
   → Default: ON for mappers — evaluate before keeping

✅ MEMORY CALCULATION FORMULA:
   1. Estimate task data need
   2. Add Distributed Cache, sort buffer, framework overhead
   3. Add 30% safety margin → JVM heap
   4. Container = heap / 0.75 → round up to 1024 boundary
   5. Verify: container ≤ max-allocation-mb
