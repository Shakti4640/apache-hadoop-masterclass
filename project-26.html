PROJECT 26: BUILDING A DATA FLOW SCRIPT WITHOUT JAVA — APACHE PIG

🔴 HYPOTHETICAL PROBLEM

SCENARIO:
→ You are a data engineer at the same ride-sharing company (Project 25)
→ Raw ride data lands daily in HDFS as messy CSV files
→ Before the data reaches the Hive warehouse (Project 24-25),
   it needs an ETL pipeline:

   STEP 1: Load raw CSV from HDFS
   STEP 2: Filter out corrupted records (missing fields, negative fares)
   STEP 3: Normalize city names (new_york → New_York, NYC → New_York)
   STEP 4: Compute derived fields (fare_per_km, ride_duration_category)
   STEP 5: Join with a driver ratings lookup file
   STEP 6: Aggregate daily summaries per city
   STEP 7: Write cleaned data to Hive-ready ORC partitioned directories

THREE WAYS TO BUILD THIS:

OPTION A — Java MapReduce (Projects 4-6):
→ 3 chained MapReduce jobs (Project 21)
→ ~400 lines of Java per job = 1,200+ lines
→ Compile, package JAR, deploy, debug, repeat
→ 2 weeks of development for a senior engineer

OPTION B — HiveQL (Projects 24-25):
→ SQL is great for queries and aggregations
→ But multi-step ETL with UDFs, conditional logic,
   branching pipelines? SQL gets UGLY and hard to maintain
→ Nested subqueries, LATERAL VIEW, temporary tables everywhere

OPTION C — Apache Pig:
→ 40-60 lines of Pig Latin script
→ Each line = one transformation step
→ Reads like a recipe: "load this, filter that, group by this, store there"
→ Compiles to optimized MapReduce automatically
→ Development time: 2 DAYS instead of 2 weeks

THE PROBLEM IS NOT COMPUTATION — IT'S DEVELOPER PRODUCTIVITY.
Pig exists because writing MapReduce for every ETL task is painful.

STEP 2: Understand Pig's mental model — DATAFLOW, not SQL

SQL (Hive) thinks in SETS:
   "Give me the result of combining these conditions on these tables"
   → You describe WHAT you want
   → The engine decides HOW to execute

Pig thinks in STEPS:
   "First load, then filter, then transform, then group, then store"
   → You describe HOW data flows through transformations
   → Each step produces a named intermediate dataset (called a RELATION)
   → You control the order of operations

MapReduce thinks in FUNCTIONS:
   "Here is my map function, here is my reduce function"
   → You describe the lowest-level mechanics
   → Maximum control, minimum productivity

STEP 3: Map our ETL requirements to Pig operators

   Load CSV              → LOAD ... USING PigStorage(',')
   Filter bad records    → FILTER ... BY (fare > 0 AND city IS NOT NULL)
   Normalize strings     → FOREACH ... GENERATE UPPER(city), ...
   Compute derived cols  → FOREACH ... GENERATE fare/distance AS fare_per_km
   Join with lookup      → JOIN rides BY driver_id, ratings BY driver_id
   Aggregate per city    → GROUP ... BY city; FOREACH ... GENERATE SUM(fare)
   Write output          → STORE ... INTO '/output' USING PigStorage(',')

💡 HOW CAN WE SOLVE IT

SOLUTION: Pig Latin — A Step-by-Step Dataflow Language

CORE CONCEPT — Everything is a RELATION:

   A relation = a named bag of tuples (like a table of rows)
   Each Pig Latin statement takes one or more relations as input
   and produces a NEW relation as output

   raw_data = LOAD '/input/rides.csv' ...;          ← relation
   clean_data = FILTER raw_data BY fare > 0;         ← new relation from raw_data
   enriched = JOIN clean_data BY id, lookup BY id;   ← new relation from two inputs
   grouped = GROUP enriched BY city;                  ← new relation
   result = FOREACH grouped GENERATE ...;             ← new relation
   STORE result INTO '/output/';                      ← terminal action

   NOTHING EXECUTES until STORE (or DUMP) is called.
   Pig builds a LOGICAL PLAN, then optimizes, then generates MapReduce jobs.

✅ WHAT SHOULD DO

1. THINK IN DATA PIPELINES, NOT QUERIES
   → Each line transforms data one step
   → Name your relations meaningfully
   → Bad:  A = LOAD; B = FILTER A; C = GROUP B;
   → Good: raw_rides = LOAD; clean_rides = FILTER raw_rides;
           city_groups = GROUP clean_rides;

2. USE SCHEMAS WHENEVER POSSIBLE
   → Pig can work WITHOUT schemas (untyped)
   → But with schemas, Pig catches errors EARLY
   → LOAD '/data' USING PigStorage(',')
     AS (ride_id:long, rider_id:long, fare:double, city:chararray);

3. USE ILLUSTRATE FOR DEBUGGING BEFORE FULL RUNS
   → ILLUSTRATE relation_name;
   → Shows sample data flowing through each step
   → Catches logic errors WITHOUT running full MapReduce job

4. USE DESCRIBE AND EXPLAIN LIBERALLY
   → DESCRIBE relation; → shows schema at each step
   → EXPLAIN relation;  → shows the MapReduce plan

5. TEST IN LOCAL MODE FIRST
   → pig -x local script.pig
   → Runs entirely on local filesystem, no Hadoop needed
   → Fast iteration during development

6. USE FOREACH ... GENERATE FOR COLUMN TRANSFORMATIONS
   → This is the workhorse of Pig
   → Equivalent to SELECT with computed columns in SQL

🚫 WHAT SHOULD NOT DO

1. DO NOT USE DUMP IN PRODUCTION
   → DUMP prints entire relation to console
   → On 1TB of data, this crashes your terminal
   → DUMP is for debugging small datasets only
   → Use STORE for production output

2. DO NOT FORGET THAT NOTHING EXECUTES UNTIL STORE/DUMP
   → Every statement before STORE just builds the logical plan
   → If you have 50 lines of Pig Latin but no STORE → nothing happens
   → Common mistake: "My script runs instantly but produces no output"

3. DO NOT IGNORE DATA TYPES
   → Pig defaults to bytearray (untyped) if no schema specified
   → Arithmetic on bytearray silently converts → can produce wrong results
   → Comparisons on untyped data may sort lexicographically, not numerically
   → "9" > "10" as strings (9 > 1 character comparison) but 9 < 10 as numbers
   → ALWAYS define schemas

4. DO NOT USE ORDER BY ON MASSIVE DATASETS CASUALLY
   → ORDER BY in Pig = global sort = single reducer bottleneck
   → On 1TB of data, this takes hours
   → Use LIMIT after ORDER BY when you need "top N"
   → Or use Pig's built-in TOP function

5. DO NOT NEST GROUP INSIDE FOREACH UNLESS YOU UNDERSTAND NESTED BLOCKS
   → Pig supports nested FOREACH with inner GROUP/FILTER/ORDER
   → Powerful but confusing — logic errors are hard to spot
   → Master simple GROUP + FOREACH first

6. DO NOT TREAT PIG AS A REPLACEMENT FOR HIVE
   → Pig is for ETL pipelines and data transformations
   → Hive is for interactive querying and reporting
   → Using Pig for ad-hoc analytics = slower development than Hive
   → Using Hive for complex multi-step ETL = uglier code than Pig
   → Each tool has its sweet spot (see "When to Use What" below)

7. DO NOT LOAD COMPRESSED FILES WITHOUT CHECKING SPLITTABILITY
   → Pig respects the same compression rules as MapReduce (Project 14)
   → GZip files → single mapper per file (no parallel processing)
   → BZip2 files → splittable → multiple mappers
   → Snappy with container format (SequenceFile) → splittable

🚫 WHEN NOT TO USE

DO NOT USE PIG WHEN:

1. YOU NEED SUB-SECOND QUERY RESPONSES
   → Pig compiles to MapReduce → minimum latency ~30 seconds
   → Even with Tez backend → still seconds, not milliseconds
   → Use HBase (Project 27) or Impala/Presto for low-latency

2. YOU ALREADY HAVE A CLEAN HIVE WAREHOUSE
   → Data is already in partitioned, bucketed ORC tables (Project 25)
   → Just write HiveQL — no need for another tool
   → Pig adds value BEFORE data reaches the warehouse

3. YOUR TEAM ONLY KNOWS SQL
   → Pig Latin is its own language — learning curve exists
   → Hive gives SQL-fluent teams immediate productivity
   → Don't force Pig on a SQL-native team

4. YOU NEED REAL-TIME STREAM PROCESSING
   → Pig is batch-only — no streaming mode
   → Use Kafka + Flink or Spark Streaming

5. THE PIPELINE IS TRIVIAL
   → "Load CSV, filter nulls, store as Parquet"
   → One Hive CTAS (CREATE TABLE AS SELECT) does this in one line
   → Pig adds unnecessary complexity for simple transforms

6. APACHE PIG IS BEING DEPRECATED IN YOUR ECOSYSTEM
   → As of 2024, Pig usage is declining
   → Apache Spark has largely replaced Pig for ETL
   → New projects should consider Spark (Project 48) for greenfield work
   → Pig knowledge still valuable for maintaining legacy pipelines

MECHANIC 1: PIG LATIN COMPILATION PIPELINE
═══════════════════════════════════════════════════════════

Your Script:
   raw = LOAD '/data/rides.csv' USING PigStorage(',') AS (id:long, fare:double, city:chararray);
   clean = FILTER raw BY fare > 0;
   grouped = GROUP clean BY city;
   result = FOREACH grouped GENERATE group AS city, SUM(clean.fare) AS total;
   STORE result INTO '/output/city_revenue';

STEP 1 — PARSING
   Pig parser reads each line
   Builds an Abstract Syntax Tree (AST)
   Validates syntax (not semantics yet)

STEP 2 — LOGICAL PLAN
   ┌──────────────────────────────────────┐
   │         LOGICAL PLAN (DAG)           │
   │                                      │
   │   LOLoad (/data/rides.csv)           │
   │      │                               │
   │      ▼                               │
   │   LOFilter (fare > 0)                │
   │      │                               │
   │      ▼                               │
   │   LOCogroup (GROUP BY city)          │
   │      │                               │
   │      ▼                               │
   │   LOForEach (SUM, project)           │
   │      │                               │
   │      ▼                               │
   │   LOStore (/output/city_revenue)     │
   └──────────────────────────────────────┘

STEP 3 — LOGICAL OPTIMIZER
   Applies optimization rules:
   
   Rule 1: PUSH FILTERS DOWN
   → If filter can be applied earlier, move it closer to LOAD
   → Reduces data flowing through subsequent operators
   
   Rule 2: PUSH PROJECTIONS DOWN (column pruning)
   → If only fare and city are used, don't read other columns
   → Tells PigStorage to skip unused columns
   
   Rule 3: COMBINE OPERATORS
   → Adjacent FOREACH statements may be merged into one
   
   Rule 4: PUSH LIMITS DOWN
   → LIMIT before ORDER BY can be optimized

STEP 4 — PHYSICAL PLAN
   Logical operators → Physical operators
   
   LOLoad → POLoad
   LOFilter → POFilter
   LOCogroup → POLocalRearrange + POGlobalRearrange + POPackage
   LOForEach → POForEach
   LOStore → POStore

STEP 5 — MAPREDUCE PLAN (or Tez DAG)
   Physical plan → actual MapReduce jobs
   
   ┌──────────────────────────────────────────────┐
   │           MAPREDUCE JOB 1                     │
   │                                               │
   │   MAP PHASE:                                  │
   │     POLoad → POFilter → POLocalRearrange      │
   │     (load, filter, prepare keys for shuffle)  │
   │                                               │
   │   SHUFFLE:                                    │
   │     Sort by city key                          │
   │                                               │
   │   REDUCE PHASE:                               │
   │     POPackage → POForEach(SUM) → POStore      │
   │     (group, aggregate, write output)          │
   │                                               │
   └──────────────────────────────────────────────┘

   Simple scripts = 1 MapReduce job
   Complex scripts (multiple GROUPs, JOINs) = multiple chained MR jobs
   With Tez backend: single DAG, no intermediate HDFS writes

MECHANIC 2: HOW GROUP BY WORKS IN PIG
═══════════════════════════════════════════════════════════

INPUT RELATION (clean):
   (1, 25.50, "New_York")
   (2, 18.00, "Chicago")
   (3, 32.00, "New_York")
   (4, 15.00, "Chicago")
   (5, 45.00, "New_York")

AFTER: grouped = GROUP clean BY city;

RESULT (grouped):
   ("Chicago",  {(2, 18.00, "Chicago"),  (4, 15.00, "Chicago")})
   ("New_York", {(1, 25.50, "New_York"), (3, 32.00, "New_York"), (5, 45.00, "New_York")})

SCHEMA OF grouped:
   grouped: {
       group: chararray,          ← the GROUP BY key (renamed to "group")
       clean: {                   ← BAG of all tuples in this group
           (id: long, fare: double, city: chararray)
       }
   }

KEY INSIGHT:
→ "group" is a special field containing the grouping key
→ "clean" is the ORIGINAL relation name — contains all rows for that group
→ This is WHY you write: SUM(clean.fare) — you're reaching INTO the bag

AFTER: result = FOREACH grouped GENERATE group AS city, SUM(clean.fare) AS total;

RESULT (result):
   ("Chicago",  33.00)
   ("New_York", 102.50)

MECHANIC 3: HOW JOIN WORKS IN PIG
═══════════════════════════════════════════════════════════

TWO RELATIONS:
   rides:   (1, 101, 25.50)   -- ride_id, driver_id, fare
   drivers: (101, "Alice", 4.8)  -- driver_id, name, rating

AFTER: joined = JOIN rides BY driver_id, drivers BY driver_id;

PIG internally:
   1. REARRANGES both relations by the join key (driver_id)
      → rides:   key=101, value=(1, 101, 25.50)
      → drivers: key=101, value=(101, "Alice", 4.8)
   
   2. SHUFFLES to same reducer (same as Reduce-Side Join — Project 17)
   
   3. MATCHES tuples with same key
   
   4. PRODUCES: (1, 101, 25.50, 101, "Alice", 4.8)
      → ALL fields from BOTH relations concatenated

JOIN TYPES IN PIG:
   JOIN a BY k, b BY k;                    → Inner join (default)
   JOIN a BY k LEFT OUTER, b BY k;         → Left outer join
   JOIN a BY k RIGHT OUTER, b BY k;        → Right outer join
   JOIN a BY k FULL OUTER, b BY k;         → Full outer join

REPLICATED JOIN (Map-Side — like Project 16):
   joined = JOIN rides BY driver_id, drivers BY driver_id USING 'replicated';
   → Small table (drivers) broadcast to all mappers via distributed cache
   → No shuffle needed — MUCH faster
   → CONSTRAINT: last relation in JOIN must be the SMALL one

MECHANIC 4: LAZY EVALUATION — NOTHING RUNS UNTIL STORE/DUMP
═══════════════════════════════════════════════════════════

   a = LOAD '/data/huge_file.csv' ...;     -- no execution
   b = FILTER a BY x > 0;                  -- no execution
   c = GROUP b BY y;                        -- no execution
   d = FOREACH c GENERATE ...;             -- no execution
   
   -- At this point, ZERO MapReduce jobs have been submitted
   -- Pig has only built a logical plan in memory
   
   STORE d INTO '/output/';                -- NOW everything executes
   
   -- Pig compiles the entire chain into MapReduce jobs
   -- Submits them to YARN
   -- Monitors until completion

WHY LAZY?
→ Allows the optimizer to see the ENTIRE pipeline before executing
→ Can push filters down, prune columns, combine stages
→ Much more efficient than executing line by line

⚠️ COMMON PITFALLS AND FAILURE MODES

PITFALL 1: "My Pig script runs but produces empty output"
→ CAUSE 1: Schema mismatch — column positions wrong in LOAD
   → CSV has: ride_id,rider_id,fare,city
   → Your schema says: AS (ride_id:long, fare:double, rider_id:long, city:chararray)
   → fare and rider_id are SWAPPED → filter "fare > 0" checks rider_id instead
   → All records may be filtered out
→ CAUSE 2: Data type mismatch
   → Column is string "25.50" but schema says int → Pig casts to null → filtered out
→ FIX: Use ILLUSTRATE and DESCRIBE after each step to verify

PITFALL 2: "GROUP BY produces weird nested structures I can't flatten"
→ CAUSE: Not understanding the bag-within-tuple structure after GROUP
→ After GROUP, each row is: (group_key, {bag of original tuples})
→ You MUST use FOREACH ... GENERATE to extract fields from the bag
→ Common mistake: trying to access fields directly without going through bag
→ CORRECT:
   grouped = GROUP rides BY city;
   result = FOREACH grouped GENERATE
       group AS city,
       SUM(rides.fare) AS total_fare;   -- rides.fare reaches INTO the bag
→ WRONG:
   result = FOREACH grouped GENERATE
       group AS city,
       SUM(fare) AS total_fare;   -- "fare" is ambiguous/undefined here

PITFALL 3: "Pig script generates 200 MapReduce jobs for a simple pipeline"
→ CAUSE: Multiple STORE statements + multiple GROUP/JOIN operations
→ Each GROUP or JOIN typically creates a new MapReduce boundary
→ Each STORE triggers execution of its upstream chain
→ FIX: Use Tez execution engine (pig -x tez) — merges into single DAG
→ FIX: Minimize unnecessary intermediate STOREs

PITFALL 4: "REPLICATED join fails with 'out of memory' error"
→ CAUSE: The "small" table is too large to fit in mapper memory
→ REPLICATED join loads the LAST table entirely into each mapper's RAM
→ If that table is 5GB and mapper has 2GB heap → OutOfMemoryError
→ FIX: Increase mapper memory (Project 23 YARN tuning)
→ OR: Use default join (shuffle-based) — slower but memory-safe

PITFALL 5: "CROSS between two tables takes forever / crashes"
→ CAUSE: CROSS produces a CARTESIAN PRODUCT
→ Table A (1M rows) CROSS Table B (1M rows) = 1 TRILLION rows
→ This is almost NEVER what you want
→ FIX: Use JOIN instead of CROSS
→ If you truly need cross product, filter heavily BEFORE crossing

PITFALL 6: "ORDER BY on large data runs for hours with one reducer"
→ CAUSE: By default ORDER BY uses single reducer for total ordering
→ Pig does use sampling (like Project 18) but final merge is still costly
→ FIX: If you only need top N, use:
   sorted = ORDER data BY fare DESC;
   top10 = LIMIT sorted 10;
   → Pig optimizes LIMIT + ORDER to avoid full sort
→ OR: Use Pig's built-in TOP function:
   top10 = TOP(10, 2, data);  -- top 10, by field index 2 (fare)

PITFALL 7: "UDF works in local mode but fails on cluster"
→ CAUSE: UDF JAR not registered or not in distributed cache
→ FIX: REGISTER /path/to/myudf.jar; at the top of script
→ Pig ships registered JARs to all task nodes automatically

import os

class PigScriptGenerator:
    """
    Generates Pig Latin ETL scripts from a pipeline configuration.
    Demonstrates how production teams template their Pig workflows.
    """
    
    def __init__(self, job_name, parallel=4):
        self.job_name = job_name
        self.parallel = parallel
        self.lines = []
        self.relation_counter = 0
        self.current_relation = None
    
    def _add_line(self, line):
        self.lines.append(line)
    
    def _add_blank(self):
        self.lines.append('')
    
    def add_header(self):
        """Add configuration and comments."""
        self._add_line(f"-- ═══════════════════════════════════════")
        self._add_line(f"-- AUTO-GENERATED PIG SCRIPT: {self.job_name}")
        self._add_line(f"-- ═══════════════════════════════════════")
        self._add_blank()
        self._add_line(f"SET default_parallel {self.parallel};")
        self._add_line(f"SET job.name '{self.job_name}';")
        self._add_line(f"SET pig.tmpfilecompression true;")
        self._add_line(f"SET pig.tmpfilecompression.codec gz;")
        self._add_blank()
    
    def add_load(self, relation_name, hdfs_path, schema_fields, delimiter=','):
        """
        Generate a LOAD statement.
        
        schema_fields: list of tuples [(name, type), ...]
        Example: [('id', 'long'), ('name', 'chararray'), ('fare', 'double')]
        """
        schema_str = ', '.join([f'{name}:{dtype}' for name, dtype in schema_fields])
        
        self._add_line(f"-- Load data from {hdfs_path}")
        self._add_line(f"{relation_name} = LOAD '{hdfs_path}'")
        self._add_line(f"    USING PigStorage('{delimiter}')")
        self._add_line(f"    AS ({schema_str});")
        self._add_blank()
        
        self.current_relation = relation_name
        return relation_name
    
    def add_filter(self, output_name, input_name, conditions):
        """
        Generate a FILTER statement.
        
        conditions: list of strings that will be ANDed together
        Example: ['fare > 0', 'city IS NOT NULL', 'distance_km > 0']
        """
        condition_str = '\n    AND '.join(conditions)
        
        self._add_line(f"-- Filter: remove invalid records")
        self._add_line(f"{output_name} = FILTER {input_name} BY")
        self._add_line(f"    {condition_str};")
        self._add_blank()
        
        self.current_relation = output_name
        return output_name
    
    def add_projection(self, output_name, input_name, fields, computed_fields=None):
        """
        Generate a FOREACH GENERATE statement.
        
        fields: list of field names to keep
        computed_fields: list of tuples [(expression, alias, type), ...]
        """
        all_fields = list(fields)
        if computed_fields:
            for expr, alias, dtype in computed_fields:
                all_fields.append(f"{expr} AS {alias}:{dtype}")
        
        fields_str = ',\n    '.join(all_fields)
        
        self._add_line(f"-- Transform and project fields")
        self._add_line(f"{output_name} = FOREACH {input_name} GENERATE")
        self._add_line(f"    {fields_str};")
        self._add_blank()
        
        self.current_relation = output_name
        return output_name
    
    def add_group_aggregate(self, output_name, input_name, group_keys,
                            aggregations):
        """
        Generate GROUP BY + FOREACH with aggregations.
        
        group_keys: list of field names or single string
        aggregations: list of tuples [(func, field, alias), ...]
        Example: [('COUNT', input_name, 'cnt'), ('SUM', 'fare', 'total')]
        """
        if isinstance(group_keys, list):
            group_str = f"({', '.join(group_keys)})"
            flatten_str = f"FLATTEN(group) AS ({', '.join(group_keys)})"
        else:
            group_str = group_keys
            flatten_str = f"group AS {group_keys}"
        
        group_rel = f"{output_name}_grp"
        
        agg_parts = [flatten_str]
        for func, field, alias in aggregations:
            if '.' not in field and field != input_name:
                field = f"{input_name}.{field}"
            agg_parts.append(f"{func}({field}) AS {alias}")
        
        agg_str = ',\n    '.join(agg_parts)
        
        self._add_line(f"-- Group by {group_str} and aggregate")
        self._add_line(f"{group_rel} = GROUP {input_name} BY {group_str};")
        self._add_blank()
        self._add_line(f"{output_name} = FOREACH {group_rel} GENERATE")
        self._add_line(f"    {agg_str};")
        self._add_blank()
        
        self.current_relation = output_name
        return output_name
    
    def add_join(self, output_name, left_rel, left_key, right_rel, right_key,
                 join_type='default'):
        """
        Generate a JOIN statement.
        
        join_type: 'default', 'replicated', 'skewed'
        """
        using_clause = ""
        if join_type != 'default':
            using_clause = f" USING '{join_type}'"
        
        self._add_line(f"-- Join {left_rel} with {right_rel} ({join_type})")
        self._add_line(
            f"{output_name} = JOIN {left_rel} BY {left_key}, "
            f"{right_rel} BY {right_key}{using_clause};"
        )
        self._add_blank()
        
        self.current_relation = output_name
        return output_name
    
    def add_order(self, output_name, input_name, order_field, direction='DESC'):
        """Generate an ORDER BY statement."""
        self._add_line(f"-- Sort by {order_field} {direction}")
        self._add_line(
            f"{output_name} = ORDER {input_name} BY {order_field} {direction};"
        )
        self._add_blank()
        
        self.current_relation = output_name
        return output_name
    
    def add_limit(self, output_name, input_name, n):
        """Generate a LIMIT statement."""
        self._add_line(f"-- Take top {n} records")
        self._add_line(f"{output_name} = LIMIT {input_name} {n};")
        self._add_blank()
        
        self.current_relation = output_name
        return output_name
    
    def add_store(self, relation_name, hdfs_path, delimiter='\t'):
        """Generate rmf + STORE statement."""
        self._add_line(f"-- Store output to {hdfs_path}")
        self._add_line(f"rmf {hdfs_path};")
        self._add_line(
            f"STORE {relation_name} INTO '{hdfs_path}' "
            f"USING PigStorage('{delimiter}');"
        )
        self._add_blank()
    
    def generate(self):
        """Return the complete Pig Latin script as string."""
        return '\n'.join(self.lines)
    
    def save(self, filepath):
        """Save the script to a file."""
        script = self.generate()
        with open(filepath, 'w') as f:
            f.write(script)
        print(f"Pig script saved to: {filepath}")
        print(f"Run with: pig -x mapreduce {filepath}")
        return filepath
