🔍 HOW TO APPROACH IT

STEP 1: Understand WHAT kind of database HBase is

HBase is NOT a relational database.
HBase is NOT a replacement for MySQL.
HBase is a COLUMN-FAMILY-ORIENTED, DISTRIBUTED, 
SORTED MAP built on top of HDFS.

Think of HBase as a GIANT SORTED DICTIONARY:

   Key → Value
   
   But the "Value" is not a single thing — it's organized into:
   → Column Families (groups of related columns)
   → Columns (individual fields within a family)
   → Versions (each cell can have multiple timestamped values)

STEP 3: Understand the HBase data model conceptually

TRADITIONAL RELATIONAL TABLE (MySQL):
┌──────────┬───────────┬───────┬──────┬─────────┐
│ rider_id │ name      │ email │ city │ rides   │
├──────────┼───────────┼───────┼──────┼─────────┤
│ 1001     │ Alice     │ a@b.c │ NYC  │ 847     │
│ 1002     │ Bob       │ b@c.d │ CHI  │ 523     │
└──────────┴───────────┴───────┴──────┴─────────┘
→ Fixed columns, every row has same structure
→ Adding a column = ALTER TABLE (expensive schema change)

HBASE TABLE:
┌──────────┬──────────────────────────┬──────────────────────────┐
│ Row Key  │ Column Family: profile   │ Column Family: stats     │
│          ├────────┬────────┬────────┼────────┬────────┬────────┤
│          │ name   │ email  │ city   │ rides  │ avg_f  │ last_d │
├──────────┼────────┼────────┼────────┼────────┼────────┼────────┤
│ 1001     │ Alice  │ a@b.c  │ NYC    │ 847    │ 23.50  │ 03-22  │
│ 1002     │ Bob    │ b@c.d  │ CHI    │ 523    │ 18.20  │ 03-21  │
│ 1003     │ Carol  │        │ LON    │ 12     │ 42.00  │ 03-20  │
└──────────┴────────┴────────┴────────┴────────┴────────┴────────┘
→ Row Key is the PRIMARY lookup mechanism (sorted lexicographically)
→ Column Families are defined at table creation (few, fixed)
→ Columns within families are DYNAMIC (can differ per row)
→ Carol has no email — that cell simply doesn't exist (sparse storage)
→ Adding new columns = just write to them (no schema change)

💡 HOW CAN WE SOLVE IT

SOLUTION: Design an HBase schema for rider profiles and driver dashboards

TABLE 1: rider_profiles
┌────────────────────────────────────────────────────────────────────────┐
│ Row Key: rider_id (zero-padded: "0000001001")                         │
│                                                                        │
│ Column Family "info" (rarely changing profile data):                   │
│   info:name = "Alice"                                                  │
│   info:email = "alice@example.com"                                     │
│   info:signup_date = "2020-03-15"                                      │
│   info:home_city = "New_York"                                          │
│                                                                        │
│ Column Family "stats" (frequently updated ride statistics):            │
│   stats:total_rides = "847"                                            │
│   stats:avg_fare = "23.50"                                             │
│   stats:total_spent = "19904.50"                                       │
│   stats:last_ride_date = "2024-03-22"                                  │
│   stats:favorite_city = "New_York"                                     │
│   stats:favorite_payment = "card"                                      │
│                                                                        │
│ Column Family "recent" (last N rides for fraud detection):             │
│   recent:ride_20240322_143022 = "New_York|25.50|8.2km|card"           │
│   recent:ride_20240321_091545 = "New_York|18.00|5.1km|wallet"         │
│   recent:ride_20240320_182230 = "Chicago|32.00|12.5km|card"           │
│   ... (dynamic column names — each ride is a new column)               │
└────────────────────────────────────────────────────────────────────────┘

WHY THIS DESIGN WORKS:

1. LOOKUP BY RIDER_ID: Get("0000001001") → instant (sorted index)
2. GET PROFILE ONLY: Get("0000001001", "info") → reads only info family
3. GET STATS ONLY: Get("0000001001", "stats") → reads only stats family
4. UPDATE AFTER RIDE: Put("0000001001", "stats:total_rides", "848")
   → Updates ONE cell, doesn't rewrite entire row
5. ADD RIDE HISTORY: Put("0000001001", "recent:ride_20240323_...", "...")
   → New column created dynamically — no schema change
6. FRAUD CHECK: Scan("0000001001", "recent") → get all recent rides
   → All recent rides for this rider in one fast read

WHY THREE COLUMN FAMILIES:
→ "info" = read often, written rarely → optimize for read caching
→ "stats" = read and written frequently → optimize for write throughput
→ "recent" = many small writes, scanned together → optimize for locality
→ Each column family is stored in SEPARATE files on HDFS
→ Reading "info" doesn't load "recent" data (I/O efficiency)

✅ WHAT SHOULD DO

1. DESIGN ROW KEY CAREFULLY — IT IS EVERYTHING
   → Row key is the ONLY indexed field in HBase
   → ALL lookups start from the row key
   → Data is sorted by row key GLOBALLY across the cluster
   → BAD row key = bad performance, hot spots, unusable table
   → (Deep row key design covered in Project 35)

2. KEEP COLUMN FAMILIES FEW (2-5 maximum)
   → Each column family = separate storage file (HFile)
   → Too many families = too many files = too many file handles
   → Group columns by ACCESS PATTERN, not by logical meaning
   → "Things read together should be stored together"

3. USE COLUMN QUALIFIERS (column names) FOR DYNAMIC DATA
   → Columns within a family can be created on the fly
   → No ALTER TABLE needed
   → Store time-series data as column names:
     "metric:2024-03-22" = "value"

4. USE VERSIONING STRATEGICALLY
   → Each cell can store multiple versions (timestamps)
   → Default: 1 version (only latest value)
   → For audit trail: set VERSIONS => 5 (keep last 5 values)
   → Useful for tracking changes without separate history table

5. PRE-SPLIT TABLES FOR KNOWN KEY RANGES
   → By default, new table has ONE region on ONE server
   → All writes go to that ONE server = hot spot
   → Pre-split into regions if you know the key distribution
   → (Detailed in Project 35)

6. USE BATCH OPERATIONS FOR BULK READS/WRITES
   → Single Get/Put for individual lookups
   → Batch Get/Put for loading many rows
   → Scan for range queries within a key range

🚫 WHAT SHOULD NOT DO

1. DO NOT USE HBASE AS A GENERAL-PURPOSE RELATIONAL DATABASE
   → No SQL (unless you add Phoenix on top)
   → No JOINs between tables (you must denormalize)
   → No secondary indexes by default
   → No complex WHERE clauses
   → If you need SQL → use Hive (Project 24) or a real RDBMS

2. DO NOT CREATE TOO MANY COLUMN FAMILIES
   → Each family = separate flush and compaction cycle
   → If family A has 1 million rows and family B has 10 rows:
     → When A flushes to disk, B is ALSO flushed
     → Tiny B files accumulate → compaction overhead
   → RULE: 2-3 column families for most tables

3. DO NOT USE MONOTONICALLY INCREASING ROW KEYS
   → Row keys like: 1, 2, 3, 4, 5, ... (auto-increment)
   → Or timestamps: 20240322143022, 20240322143023, ...
   → ALL writes go to the LAST region (highest key range)
   → ONE RegionServer handles ALL writes → HOT SPOT
   → Other servers sit idle → wasted cluster resources
   → SOLUTION: Hash prefix, salting, reversed timestamps
   → (Detailed in Project 35)

4. DO NOT SCAN THE ENTIRE TABLE
   → HBase is NOT for full table scans — that's Hive's job
   → Scan with start/stop row keys to limit range
   → scan 'table' without limits = read EVERY row = SLOW
   → If you need analytics on all data → export to Hive

5. DO NOT STORE LARGE VALUES IN CELLS
   → HBase cells are designed for values < 10MB
   → Storing images, PDFs, or large BLOBs = poor performance
   → Large cells cause memory pressure during reads
   → SOLUTION: Store large files in HDFS, store the PATH in HBase

6. DO NOT IGNORE COMPACTION
   → HBase writes to in-memory store (MemStore) first
   → Flushes to disk as HFiles
   → Many small HFiles accumulate → read performance degrades
   → Compaction merges HFiles → must be monitored
   → Minor compaction: merge a few small files (automatic)
   → Major compaction: merge ALL files (resource intensive, schedule off-peak)

7. DO NOT MIX HEAVY BATCH AND REAL-TIME ON SAME TABLE
   → Bulk loading millions of rows while serving real-time gets
   → Bulk writes cause region splits, compactions, GC pauses
   → Real-time reads suffer during these operations
   → SOLUTION: Use Bulk Load via HFile (Project 36) for batch ingestion

🚫 WHEN NOT TO USE

DO NOT USE HBASE WHEN:

1. YOU NEED SQL AND JOINS
   → HBase has NO native SQL (Phoenix adds it but with limitations)
   → JOINs must be done at application level
   → If your queries are: SELECT a.*, b.* FROM a JOIN b ON a.id = b.id
   → Use Hive (Project 24) or a relational database

2. DATA SIZE IS SMALL (< 100 million rows)
   → HBase overhead (ZooKeeper, RegionServers, Master) is significant
   → For < 100M rows, PostgreSQL or MySQL is simpler and faster
   → HBase shines at BILLIONS of rows

3. YOU ONLY DO BATCH ANALYTICS
   → If all queries are: "total revenue last quarter"
   → Hive does this better, cheaper, and simpler
   → HBase adds operational complexity with no benefit for batch

4. YOU NEED STRONG MULTI-ROW TRANSACTIONS
   → HBase guarantees atomicity only at SINGLE ROW level
   → No multi-row, multi-table transactions (no ACID across rows)
   → If you need: "transfer $50 from account A to account B atomically"
   → Use a traditional RDBMS

5. YOUR ACCESS PATTERN IS UNKNOWN
   → HBase requires DESIGNING the row key for your specific access pattern
   → If you don't know how data will be queried → wrong tool
   → Use Hive (schema-on-read, flexible queries) until patterns emerge

6. YOU NEED FULL-TEXT SEARCH
   → "Find all riders whose name contains 'Ali'"
   → HBase has NO full-text search capability
   → Use Elasticsearch or Apache Solr alongside HBase

WHEN CLIENT REQUESTS: Get("0000001001")
→ "0000001001" falls in Region A's range
→ Client knows Region A is on RegionServer 1
→ Client talks DIRECTLY to RegionServer 1
→ NO Master involved in read/write operations
→ This is why reads are fast — direct connection to the right server

HOW CLIENT KNOWS WHICH REGION:
→ Step 1: Client asks ZooKeeper: "Where is the META table?"
→ Step 2: ZooKeeper says: "META is on RegionServer 3"
→ Step 3: Client reads META table: 
          "rider_profiles, row 0000001001 → Region A → RegionServer 1"
→ Step 4: Client caches this mapping
→ Step 5: Client goes DIRECTLY to RegionServer 1
→ Future lookups skip steps 1-3 (cached)

AUTO-SPLITTING:
→ When a region grows beyond threshold (default 10GB)
→ HBase SPLITS it into two regions
→ Master reassigns the new region to balance load
→ This is AUTOMATIC — no manual intervention

═══════════════════════════════════════════════════════════
MECHANIC 3: WRITE PATH — How a PUT works
═══════════════════════════════════════════════════════════

Client: Put("0000001001", "stats:total_rides", "848")

STEP 1 — Write to WAL (Write-Ahead Log)
   → Before anything else, write to WAL on HDFS
   → WAL = crash recovery journal
   → If RegionServer crashes, WAL replays unfinished writes
   → Sequential write to HDFS — fast

STEP 2 — Write to MemStore (in-memory)
   → MemStore = sorted in-memory buffer (one per column family per region)
   → Key-value pair inserted into sorted structure
   → Client receives "SUCCESS" response immediately
   → Data is NOT yet on disk as HFile

STEP 3 — MemStore Flush (when MemStore is full)
   → Default threshold: 128MB per MemStore
   → Sorted data written to disk as a NEW HFile
   → HFile stored on HDFS (immutable — never modified)
   → WAL entries for flushed data can be cleaned up

   ┌──────────┐     ┌───────────┐     ┌──────────────┐
   │ Client   │────→│ WAL(HDFS) │     │              │
   │ Put()    │     └───────────┘     │              │
   │          │────→│ MemStore  │────→│ HFile (HDFS) │
   │          │     │ (RAM)     │flush│ (immutable)  │
   └──────────┘     └───────────┘     └──────────────┘
   
   Response sent after Step 2 (WAL + MemStore)
   NOT after HFile write — this is why writes are fast

STEP 4 — Compaction (background process)
   → Multiple HFiles accumulate for each column family
   → Minor compaction: merge a few recent HFiles into one
   → Major compaction: merge ALL HFiles into one
     → Removes deleted data (tombstone markers)
     → Removes expired versions
     → Resource intensive — schedule during off-peak

═══════════════════════════════════════════════════════════
MECHANIC 4: READ PATH — How a GET works
═══════════════════════════════════════════════════════════

Client: Get("0000001001", "stats")

STEP 1 — Check BlockCache (read cache in RAM)
   → Recently read data blocks are cached
   → If found → return immediately (fastest path)

STEP 2 — Check MemStore
   → Data recently written but not yet flushed
   → If found → return (second fastest)

STEP 3 — Check HFiles on disk
   → May need to check MULTIPLE HFiles
   → Each HFile has a BLOOM FILTER (Project 25 ORC concept)
     → "Does this row key POSSIBLY exist in this file?"
     → If bloom filter says NO → skip file entirely
     → If bloom filter says MAYBE → read the file
   → HFiles have BLOCK INDEXES for fast seeking
   → Read the specific block containing the row key

STEP 4 — Merge results
   → Combine data from BlockCache + MemStore + HFiles
   → Return the latest version of each cell

   │ Client   │────→│ BlockCache  │──→ HIT? Return

│           │   ├── stats/                ← column family "stats"

│           └── region_def456/

═══════════════════════════════════════════════════════════
MECHANIC 6: CELL VERSIONING
═══════════════════════════════════════════════════════════

Each cell in HBase is identified by:
   (RowKey, ColumnFamily:ColumnQualifier, Timestamp) → Value

Example: Updating a rider's city over time:

   ("0000001001", "info:city", T=1000) → "Chicago"     ← oldest
   ("0000001001", "info:city", T=2000) → "New_York"    ← middle
   ("0000001001", "info:city", T=3000) → "Los_Angeles"  ← latest

Get with no timestamp → returns "Los_Angeles" (latest)
Get with timestamp=2000 → returns "New_York" (specific version)
Get with versions=3 → returns all three values

VERSIONS setting per column family:
→ VERSIONS => 1: only latest value kept (default)
→ VERSIONS => 5: keep last 5 values
→ VERSIONS => MAX: keep all (careful — storage grows forever)

⚠️ COMMON PITFALLS AND FAILURE MODES

PITFALL 1: "All writes go to one RegionServer — others are idle"
→ CAUSE: Sequential row keys (auto-increment or timestamp prefix)
→ All new keys fall in the LAST region's range
→ Only ONE RegionServer handles all writes
→ FIX: Salt/hash the row key prefix (detailed in Project 35)
→ EXAMPLE: Instead of "20240322_1001", use "3_20240322_1001"
           where "3" = hash(1001) % num_regions

PITFALL 2: "Read performance degrades over time"
→ CAUSE: Too many HFiles accumulate without compaction
→ Each GET must check multiple HFiles → slower reads
→ FIX: Monitor HFile count per region
→ Tune compaction settings:
   hbase.hstore.compaction.min = 3 (trigger minor after 3 files)
   hbase.hstore.compaction.max = 10 (compact up to 10 files at once)
→ Schedule major compactions during off-peak hours

PITFALL 3: "RegionServer crashes with OutOfMemoryError"
→ CAUSE 1: Too many regions per RegionServer
   → Each region has MemStore per column family
   → 1000 regions × 3 families × 128MB = 384GB heap needed
   → RULE: ~100-200 regions per RegionServer maximum
→ CAUSE 2: Large scan results not released
   → Client opens scan, reads 10 million rows into memory
→ FIX: Tune region count, MemStore size, use scan caching/batching

PITFALL 4: "Scan returns unexpected results or misses rows"
→ CAUSE: Row key design doesn't match scan pattern
→ HBase scans are LEXICOGRAPHIC on row key bytes
→ "9" > "10" as strings (byte comparison)
→ FIX: Zero-pad numeric keys: "0009" < "0010"
→ FIX: Use consistent key encoding (discussed in Project 35)

PITFALL 5: "Table creation hangs or fails"
→ CAUSE: ZooKeeper not running or unreachable
→ HBase depends on ZooKeeper for ALL coordination
→ FIX: Verify ZooKeeper is running: echo ruok | nc localhost 2181
→ Should respond with "imok"

PITFALL 6: "Delete doesn't free disk space immediately"
→ CAUSE: HBase deletes are TOMBSTONE MARKERS, not physical deletes
→ Data is marked for deletion but remains in HFiles
→ Only removed during MAJOR COMPACTION
→ FIX: Run major_compact after bulk deletes
→ Be patient — this is by design for consistency

class RiderProfileService:
    """
    Production-style HBase client for rider profile management.
    Demonstrates all CRUD operations via Python.
    """

    def __init__(self, host='localhost', port=9090):
        self.host = host
        self.port = port
        self.connection = None
        self.table = None
        self.TABLE_NAME = 'rider_profiles'

    def connect(self):
        """Establish connection to HBase via Thrift."""
        try:
            self.connection = happybase.Connection(
                host=self.host,
                port=self.port,
                timeout=10000
            )
            self.connection.open()
            print(f"[CONNECTED] HBase at {self.host}:{self.port}")
            self.table = self.connection.table(self.TABLE_NAME)
            return True
        except Exception as e:
            print(f"[ERROR] Cannot connect to HBase Thrift: {e}")
            print("  Ensure HBase Thrift server is running:")
            print("  hbase thrift start -p 9090")
            return False

    def disconnect(self):
        """Close connection."""
        if self.connection:
            self.connection.close()
            print("[DISCONNECTED]")

    def create_table_if_not_exists(self):
        """Create rider_profiles table if it doesn't exist."""
        tables = self.connection.tables()
        table_bytes = self.TABLE_NAME.encode()

        if table_bytes in tables:
            print(f"[EXISTS] Table '{self.TABLE_NAME}' already exists")
            return

        print(f"[CREATE] Creating table '{self.TABLE_NAME}'...")
        self.connection.create_table(
            self.TABLE_NAME,
            {
                'info': dict(
                    max_versions=1,
                    bloom_filter_type='ROW'
                ),
                'stats': dict(
                    max_versions=3,
                    bloom_filter_type='ROW'
                ),
                'recent': dict(
                    max_versions=1,
                    bloom_filter_type='ROWCOL',
                    time_to_live=7776000  # 90 days
                )
            }
        )
        self.table = self.connection.table(self.TABLE_NAME)
        print(f"[CREATE] Table created successfully")

    # ─── PUT OPERATIONS ───

    def put_rider_profile(self, rider_id, name, email=None,
                          signup_date=None, home_city=None):
        """Insert or update a rider's profile info."""
        row_key = self._format_key(rider_id)
        data = {b'info:name': name.encode()}

        if email:
            data[b'info:email'] = email.encode()
        if signup_date:
            data[b'info:signup_date'] = signup_date.encode()
        if home_city:
            data[b'info:home_city'] = home_city.encode()

        self.table.put(row_key, data)

    def put_rider_stats(self, rider_id, total_rides=None, avg_fare=None,
                        total_spent=None, last_ride_date=None):
        """Update rider statistics."""
        row_key = self._format_key(rider_id)
        data = {}

        if total_rides is not None:
            data[b'stats:total_rides'] = str(total_rides).encode()
        if avg_fare is not None:
            data[b'stats:avg_fare'] = f"{avg_fare:.2f}".encode()
        if total_spent is not None:
            data[b'stats:total_spent'] = f"{total_spent:.2f}".encode()
        if last_ride_date:
            data[b'stats:last_ride_date'] = last_ride_date.encode()

        if data:
            self.table.put(row_key, data)

    def add_recent_ride(self, rider_id, ride_timestamp, city,
                        fare, distance, payment):
        """Add a ride to the recent rides history."""
        row_key = self._format_key(rider_id)
        col_name = f"recent:ride_{ride_timestamp}".encode()
        value = f"{city}|{fare}|{distance}|{payment}".encode()

        self.table.put(row_key, {col_name: value})

    # ─── GET OPERATIONS ───

    def get_rider_full(self, rider_id):
        """Get complete rider profile (all column families)."""
        row_key = self._format_key(rider_id)
        row = self.table.row(row_key)
        return self._decode_row(row_key, row)

    def get_rider_info(self, rider_id):
        """Get only profile info (info family)."""
        row_key = self._format_key(rider_id)
        row = self.table.row(row_key, columns=[b'info'])
        return self._decode_row(row_key, row)

    def get_rider_stats(self, rider_id):
        """Get only rider statistics."""
        row_key = self._format_key(rider_id)
        row = self.table.row(row_key, columns=[b'stats'])
        return self._decode_row(row_key, row)

    def get_recent_rides(self, rider_id):
        """Get recent ride history (for fraud detection)."""
        row_key = self._format_key(rider_id)
        row = self.table.row(row_key, columns=[b'recent'])
        rides = []
        for key, value in row.items():
            col_name = key.decode()
            ride_data = value.decode()
            parts = ride_data.split('|')
            if len(parts) == 4:
                rides.append({
                    'ride_id': col_name.replace('recent:', ''),
                    'city': parts[0],
                    'fare': float(parts[1]),
                    'distance': float(parts[2]),
                    'payment': parts[3]
                })
        return rides

    def batch_get_riders(self, rider_ids, columns=None):
        """Get multiple riders at once."""
        results = {}
        rows = self.table.rows(
            [self._format_key(rid) for rid in rider_ids],
            columns=columns
        )
        for row_key, row_data in rows:
            results[row_key.decode()] = self._decode_row(row_key, row_data)
        return results

    # ─── SCAN OPERATIONS ───

    def scan_riders_range(self, start_id, stop_id, columns=None, limit=100):
        """Scan a range of rider IDs."""
        start_key = self._format_key(start_id)
        stop_key = self._format_key(stop_id)

        results = []
        for row_key, row_data in self.table.scan(
            row_start=start_key,
            row_stop=stop_key,
            columns=columns,
            limit=limit
        ):
            results.append(self._decode_row(row_key, row_data))
        return results

    def scan_riders_prefix(self, prefix, columns=None, limit=100):
        """Scan riders whose key starts with a prefix."""
        results = []
        for row_key, row_data in self.table.scan(
            row_prefix=prefix.encode(),
            columns=columns,
            limit=limit
        ):
            results.append(self._decode_row(row_key, row_data))
        return results

    # ─── DELETE OPERATIONS ───

    def delete_rider(self, rider_id):
        """Delete entire rider profile."""
        row_key = self._format_key(rider_id)
        self.table.delete(row_key)

    def delete_rider_column(self, rider_id, column):
        """Delete a specific column from a rider."""
        row_key = self._format_key(rider_id)
        self.table.delete(row_key, columns=[column.encode()])

    # ─── HELPER METHODS ───

    def _format_key(self, rider_id):
        """Format rider ID as zero-padded row key."""
        if isinstance(rider_id, bytes):
            return rider_id
        return f"{int(rider_id):010d}".encode()

    def _decode_row(self, row_key, row_data):
        """Decode HBase row into a Python dictionary."""
        if not row_data:
            return None
        decoded = {
            'row_key': row_key.decode() if isinstance(row_key, bytes)
                       else row_key
        }
        for key, value in row_data.items():
            col = key.decode() if isinstance(key, bytes) else key
            val = value.decode() if isinstance(value, bytes) else value
            decoded[col] = val
        return decoded

def run_simulation_demo():
    """Fallback demo when HBase is not available."""
    print("\n" + "=" * 70)
    print("SIMULATION MODE — Demonstrating HBase concepts without server")
    print("=" * 70)

    # Simulate HBase in-memory with sorted dictionary
    hbase_sim = {}

    def sim_put(row_key, column, value):
        if row_key not in hbase_sim:
            hbase_sim[row_key] = {}
        hbase_sim[row_key][column] = value

    def sim_get(row_key, columns=None):
        row = hbase_sim.get(row_key, {})
        if columns:
            return {k: v for k, v in row.items() if k in columns}
        return row

    def sim_scan(start, stop):
        return {k: v for k, v in sorted(hbase_sim.items())
                if start <= k < stop}

    # Insert data
    print("\n[SIMULATION] Inserting 5 riders...")
    for i in range(1001, 1006):
        key = f"{i:010d}"
        sim_put(key, "info:name", f"Rider_{i}")
        sim_put(key, "info:home_city", ["NYC", "CHI", "LON", "MUM", "DEL"][i-1001])
        sim_put(key, "stats:total_rides", str((i - 1000) * 100))

    # Single GET
    print("\n[SIMULATION] GET rider 1001:")
    result = sim_get("0000001001")
    for col, val in result.items():
        print(f"  {col}: {val}")

    # Range SCAN
    print("\n[SIMULATION] SCAN riders 1002 to 1004:")
    scan_results = sim_scan("0000001002", "0000001005")
    for key, data in scan_results.items():
        print(f"  {key}: {data.get('info:name')} "
              f"({data.get('stats:total_rides')} rides)")

    # Demonstrate the KEY CONCEPT
    print(f"""
    ┌────────────────────────────────────────────────────────────┐
    │  HBase is essentially a SORTED DICTIONARY:                 │
    │                                                            │
    │  "0000001001" → {{"info:name": "Alice", "stats:rides": 100}} │

    │  Direct key access → single lookups are instant            │
    │  Column families → separate storage for different patterns │
    │  Distributed across RegionServers → scales horizontally    │

WHAT WE ESTABLISHED IN PROJECT 27:

✅ HBASE FILLS THE REAL-TIME GAP IN HADOOP
   → HDFS/Hive/Pig = batch processing (minutes to hours)
   → HBase = random read/write (milliseconds)
   → Same HDFS storage underneath — not a separate system
   → Billions of rows with sub-10ms lookups

✅ HBASE DATA MODEL — Column-family oriented, sparse, versioned
   → Row Key = ONLY index = MOST CRITICAL design decision
   → Column Families = few, fixed at creation, separate storage
   → Column Qualifiers = dynamic, unlimited, schema-free
   → Cells = versioned with timestamps
   → Sparse = missing columns cost ZERO storage

✅ HBASE ARCHITECTURE — Master + RegionServers + ZooKeeper
   → HBase Master: DDL, region assignment, load balancing
   → RegionServers: actual data serving (reads/writes)
   → ZooKeeper: coordination, META table location, failure detection
   → Clients talk DIRECTLY to RegionServers (not through Master)
   → Regions auto-split as data grows

✅ WRITE PATH: Client → WAL → MemStore → (flush) → HFile on HDFS
✅ READ PATH: Client → BlockCache → MemStore → HFile (with bloom filters)

✅ OPERATIONS: put, get, scan, delete, increment
   → put = insert or update (same operation)
   → get = single-row lookup by exact row key
   → scan = range query between start and stop row keys
   → delete = tombstone marker (physical removal at compaction)
   → increment = atomic counter operation
