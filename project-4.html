   /telecom/raw/call_records/2024/01/call_records_2024-01-05.csv

→ Management asks: "How many calls of EACH TYPE happened this week?"
   → voice_call: ???
   → sms: ???
   → data: ???
   → roaming_call: ???
   → intl_call: ???

WHAT YOU TRIED:
→ hdfs dfs -cat /telecom/raw/call_records/2024/01/* | awk -F',' '{print $3}' | sort | uniq -c
→ This WORKS but:
   → Streams ALL data through network to your machine (Project 3 warning)
   → 100MB works — 100TB does NOT

WHAT YOU NEED:
→ A program that runs ON THE CLUSTER
→ Each machine processes ITS LOCAL blocks (data locality — Project 1)
→ Results are combined automatically
→ If a machine fails, Hadoop retries on another machine
→ THIS IS MAPREDUCE

🔍 HOW TO APPROACH IT

THE MAPREDUCE MENTAL MODEL:

Think of counting call types like counting votes in an election:

TRADITIONAL (one person counts all votes):
→ One person opens ALL ballot boxes
→ Reads every ballot one by one
→ Keeps a tally sheet
→ Takes HOURS for millions of votes

MAPREDUCE (many people count in parallel):

PHASE 1 — MAP (Distribute ballot boxes to many counters):
→ Counter 1 gets Box A → reads each ballot → writes tally marks
→ Counter 2 gets Box B → reads each ballot → writes tally marks
→ Counter 3 gets Box C → reads each ballot → writes tally marks
→ ALL COUNTERS WORK SIMULTANEOUSLY

PHASE 2 — SHUFFLE (Sort tally marks by candidate):
→ ALL tally marks for "voice_call" go to ONE desk
→ ALL tally marks for "sms" go to ANOTHER desk
→ ALL tally marks for "data" go to ANOTHER desk
→ HADOOP DOES THIS AUTOMATICALLY — you don't code this

PHASE 3 — REDUCE (Sum up tally marks per candidate):
→ Desk 1 receives ALL "voice_call" tallies → adds them → TOTAL
→ Desk 2 receives ALL "sms" tallies → adds them → TOTAL
→ Desk 3 receives ALL "data" tallies → adds them → TOTAL

RESULT:
→ voice_call: 1,234,567
→ sms: 2,345,678
→ data: 3,456,789

MAPPING TO HADOOP:
→ Ballot boxes = HDFS blocks (128MB each — from Project 2)
→ Counters = Mapper tasks (one per block typically)
→ Desks = Reducer tasks
→ Tally marks = Key-Value pairs

✅ WHAT COULD BE DONE

UNDERSTAND THE TYPE SYSTEM BEFORE WRITING CODE:

MapReduce does NOT use standard Java types (String, Integer, etc.)
It uses HADOOP WRITABLE TYPES for efficient serialization:

┌──────────────────────┬───────────────────────┬──────────────────────┐
│ JAVA TYPE            │ HADOOP WRITABLE       │ WHY SPECIAL TYPE?    │
├──────────────────────┼───────────────────────┼──────────────────────┤
│ String               │ Text                  │ Efficient UTF-8      │
│                      │                       │ serialization        │
├──────────────────────┼───────────────────────┼──────────────────────┤
│ int                  │ IntWritable           │ Fixed 4-byte binary  │
│                      │                       │ (not string "12345") │
├──────────────────────┼───────────────────────┼──────────────────────┤
│ long                 │ LongWritable          │ Fixed 8-byte binary  │
├──────────────────────┼───────────────────────┼──────────────────────┤
│ float                │ FloatWritable         │ Fixed 4-byte binary  │
├──────────────────────┼───────────────────────┼──────────────────────┤
│ double               │ DoubleWritable        │ Fixed 8-byte binary  │
├──────────────────────┼───────────────────────┼──────────────────────┤
│ boolean              │ BooleanWritable       │ 1-byte binary        │
├──────────────────────┼───────────────────────┼──────────────────────┤
│ null                 │ NullWritable          │ Singleton, zero bytes│
│                      │                       │ (when key or value   │
│                      │                       │  not needed)         │
└──────────────────────┴───────────────────────┴──────────────────────┘

WHY NOT REGULAR JAVA TYPES?
→ MapReduce shuffles MILLIONS of key-value pairs across the network
→ Java serialization (Serializable) is SLOW and BLOATED
→ Hadoop Writables are:
  → Compact: IntWritable = exactly 4 bytes (Java Integer = 16+ bytes)
  → Fast: Direct byte-level read/write (no reflection)
  → Reusable: Same object can be rewritten (reduces garbage collection)

FOR OUR PROBLEM:
→ Mapper output key:   Text           (call type: "voice_call", "sms", etc.)
→ Mapper output value: IntWritable    (count: always 1)
→ Reducer output key:  Text           (call type)
→ Reducer output value: IntWritable   (total count)

DEFAULT INPUT FORMAT — TextInputFormat:
→ Reads input file LINE BY LINE
→ KEY given to mapper:   LongWritable  (byte offset of line in file)
→ VALUE given to mapper: Text          (the actual line content)
→ We IGNORE the key (byte offset) and PARSE the value (CSV line)

✅ WHAT SHOULD DO

1. ALWAYS define output types EXPLICITLY in the Driver
   → job.setMapOutputKeyClass(Text.class)
   → job.setMapOutputValueClass(IntWritable.class)
   → job.setOutputKeyClass(Text.class)
   → job.setOutputValueClass(IntWritable.class)
   → Mismatch between declared types and actual output = RUNTIME ERROR

2. ALWAYS handle malformed data in the Mapper
   → Real data has corrupt lines, missing fields, wrong formats
   → If mapper throws exception on one line, the ENTIRE task fails
   → Hadoop retries failed tasks (default: 4 attempts)
   → 4 failures on same bad line = JOB FAILS
   → USE try-catch and SKIP bad records

3. ALWAYS make output directory NOT exist before running
   → MapReduce REFUSES to write to existing output directory
   → Safety feature: prevents accidentally overwriting previous results
   → Delete output dir before running: hdfs dfs -rm -r /output/path

4. REUSE Writable objects in Mapper/Reducer
   → DO: Create Text and IntWritable ONCE as class fields
   → DO: Call .set() to change values
   → DO NOT: Create new Text("voice_call") for every line
   → REASON: Millions of lines = millions of objects = garbage collection death

5. UNDERSTAND the Mapper's map() method signature:
   map(LongWritable key, Text value, Context context)
   → key: byte offset (usually IGNORED)
   → value: one line of input text
   → context: how you emit output → context.write(outKey, outValue)

6. UNDERSTAND the Reducer's reduce() method signature:
   reduce(Text key, Iterable<IntWritable> values, Context context)
   → key: one unique key (e.g., "voice_call")
   → values: ALL values for that key from ALL mappers
   → context: how you emit output → context.write(outKey, outValue)
   → ⚠️ values iterator can only be traversed ONCE — do NOT store it

🚫 WHAT SHOULD NOT DO

1. DO NOT iterate over values more than once in Reducer
   → The Iterable<IntWritable> is a STREAMING iterator
   → Backed by disk/network — NOT an in-memory list
   → Second iteration returns NOTHING (empty)
   → If you need multiple passes: accumulate into a local collection first

   WRONG:
     for (IntWritable val : values) { sum += val.get(); }  // first pass OK
     for (IntWritable val : values) { count++; }           // EMPTY — returns nothing!
   
   RIGHT:
     for (IntWritable val : values) {
         sum += val.get();
         count++;
     }

2. DO NOT hold references to value objects from the iterator
   → Hadoop REUSES the same IntWritable object across iterations
   → values: [1, 1, 1] internally = same object with .set(1) called 3 times
   → Storing references gives you 3 pointers to the SAME object
   
   WRONG:
     List<IntWritable> stored = new ArrayList<>();
     for (IntWritable val : values) { stored.add(val); }
     // stored contains 3 references to the SAME object!
   
   RIGHT:
     List<Integer> stored = new ArrayList<>();
     for (IntWritable val : values) { stored.add(val.get()); }
     // stored contains 3 independent int values

3. DO NOT write to an existing output directory
   → Job will IMMEDIATELY fail with FileAlreadyExistsException
   → Always delete output dir before running
   → This is INTENTIONAL — prevents result corruption

4. DO NOT skip the header line by checking line number
   → Mappers process BLOCKS, not files
   → Block 2's first line is NOT the CSV header — it is data
   → Only Block 1's first line is the header
   → CHECK CONTENT, not position:
     if (value.toString().startsWith("caller_id")) return; // skip header

5. DO NOT use System.out.println for debugging in production
   → stdout goes to task attempt logs (hard to find)
   → Use Hadoop Counters instead (covered in Project 7)
   → For development: context.getCounter("Debug", "bad_lines").increment(1)

6. DO NOT set number of reducers to 0 accidentally
   → job.setNumReduceTasks(0) → Map-only job (no reduction!)
   → Output will be UNSORTED mapper output
   → Useful in specific cases (Project 11) but NOT for aggregation

🚫 WHEN NOT TO USE

MAPREDUCE IS THE WRONG TOOL WHEN:

1. DATA FITS IN MEMORY OF ONE MACHINE
   → Our 100MB telecom data? Pandas does it in 2 seconds
   → MapReduce overhead (JVM, scheduling, shuffle): ~30 seconds minimum
   → THRESHOLD: If data < 1GB, question MapReduce necessity
   → USE INSTEAD: Python/Pandas, awk, grep

2. YOU NEED ITERATIVE PROCESSING
   → Machine learning training (100 iterations over same data)
   → Each MapReduce job writes to disk → reads from disk → writes to disk
   → 100 iterations × disk I/O = painfully slow
   → USE INSTEAD: Apache Spark (Project 48)

3. YOU NEED REAL-TIME RESULTS
   → MapReduce minimum latency: ~30-60 seconds
   → Most jobs take minutes to hours
   → USE INSTEAD: Kafka Streams, Apache Flink, Storm

4. YOUR LOGIC REQUIRES GLOBAL STATE
   → Mapper A needs to know what Mapper B found
   → MapReduce mappers are ISOLATED — no communication
   → Only communication: through the shuffle (key-value routing)
   → USE INSTEAD: Spark with broadcast variables, or custom MPI

5. THE COMPUTATION IS TRIVIAL
   → "Just grep for ERROR lines in logs"
   → hdfs dfs -cat /logs/* | grep ERROR
   → Or use Hadoop Streaming with grep (Project 37)
   → Full Java MapReduce is overkill for simple filters

⚠️ COMMON PITFALLS AND FAILURE MODES

PITFALL 1: "Job fails with ClassNotFoundException"
→ REASON: Mapper/Reducer classes not found in JAR
→ CAUSE: Forgot to include inner classes or wrong package
→ FIX: Ensure JAR contains ALL classes:
   jar tf CallTypeCount.jar  ← verify contents
→ FIX: Use job.setJarByClass(DriverClass.class) in Driver

PITFALL 2: "Output directory already exists — job refuses to start"
→ REASON: MapReduce safety feature (Project 4 rule)
→ FIX: Delete before running:
   hdfs dfs -rm -r /output/path
→ OR: Add deletion code in Driver before job.waitForCompletion()

PITFALL 3: "Job completes but output is empty (0 bytes)"
→ POSSIBLE CAUSES:
  → Mapper emits nothing (all lines fail parsing)
  → Input path is wrong (no input files found)
  → Header line matches all data lines (filter too aggressive)
→ DEBUG: Check job counters:
  → "Map input records" should be > 0
  → "Map output records" should be > 0
  → If map input = 0: wrong input path
  → If map output = 0: mapper logic bug

PITFALL 4: "Type mismatch error during shuffle"
→ REASON: Mapper declares output types different from what it emits
→ EXAMPLE: Driver says setMapOutputValueClass(IntWritable.class)
   but mapper emits context.write(key, new Text("hello"))
→ Error: "Type mismatch in value from map"
→ FIX: Ensure declared types match actual emitted types EXACTLY

PITFALL 5: "Job runs but produces wrong counts"
→ CAUSE 1: CSV parsing error (splitting on wrong delimiter)
→ CAUSE 2: Header line counted as data record
→ CAUSE 3: Extra whitespace in parsed values
   "voice_call " ≠ "voice_call" (trailing space = different key!)
→ FIX: Always trim() parsed values
→ FIX: Print first few mapper outputs to verify (use counters)

PITFALL 6: "Reducer receives fewer values than expected"
→ REASON: NOT a bug — values iterator is LAZY
→ If you store the Iterable and iterate later: it may be closed
→ Always iterate IMMEDIATELY in the reduce() method
→ See "What Should Not Do" section above for details

PITFALL 7: "Job hangs at map 100% reduce 0%"
→ REASON: Shuffle phase is transferring data
→ Reducers start pulling map output BEFORE all mappers finish
→ But reduce() method only starts after ALL map output received
→ PATIENCE: For large data, shuffle can take significant time
→ Monitor: ResourceManager UI shows shuffle progress

📈 SCALABILITY CONSIDERATIONS

HOW MAPREDUCE SCALES WITH DATA SIZE:

┌──────────────────┬──────────┬────────────┬──────────────────────────┐
│ Data Size        │ Mappers  │ Time       │ Notes                    │
├──────────────────┼──────────┼────────────┼──────────────────────────┤
│ 100 MB (5 files) │ 5        │ ~45 sec    │ Dominated by JVM startup │
│                  │          │            │ and job scheduling       │
├──────────────────┼──────────┼────────────┼──────────────────────────┤
│ 1 GB             │ 8        │ ~1 min     │ Actual processing begins │
│                  │          │            │ to matter                │
├──────────────────┼──────────┼────────────┼──────────────────────────┤
│ 10 GB            │ 80       │ ~3 min     │ Good parallelism         │
├──────────────────┼──────────┼────────────┼──────────────────────────┤
│ 100 GB           │ 800      │ ~8 min     │ Network shuffle becomes  │
│                  │          │            │ significant              │
├──────────────────┼──────────┼────────────┼──────────────────────────┤
│ 1 TB             │ 8,000    │ ~20 min    │ Scheduling overhead      │
│                  │          │            │ noticeable               │
├──────────────────┼──────────┼────────────┼──────────────────────────┤
│ 10 TB            │ 80,000   │ ~45 min    │ Cluster-wide operation   │
├──────────────────┼──────────┼────────────┼──────────────────────────┤
│ 100 TB           │ 800,000  │ ~2 hours   │ Need Combiner (Proj 9)  │
│                  │          │            │ to reduce shuffle        │
└──────────────────┴──────────┴────────────┴──────────────────────────┘

→ Mappers scale LINEARLY with data (more blocks = more mappers)
→ Shuffle cost grows with mapper output volume
→ Reducer count should match output volume, not input volume
→ For count-by-type: even with 100TB input, output is only 5 lines
  → 1 reducer is sufficient

OVERHEAD BREAKDOWN:
→ JVM startup: ~5-10 seconds per task
→ Job scheduling: ~10-20 seconds
→ Task initialization: ~2-5 seconds per task
→ TOTAL OVERHEAD: ~30-60 seconds minimum for ANY job
→ This is why MapReduce is wrong for small data (Project 1 lesson)

// FILE: CallTypeCount.java
// PROJECT 4: First MapReduce Job — Count Call Types
// ══════════════════════════════════════════════════════════════════
// 
// INPUT:  CSV files with columns:
//   caller_id,receiver_id,type,duration_sec,cost_usd,timestamp,tower_id
//
// OUTPUT: Each call type with its total count:
//   data         723456
//   intl_call    23456
//   voice_call   542103
//
// ══════════════════════════════════════════════════════════════════

import java.io.IOException;

import org.apache.hadoop.conf.Configuration;
import org.apache.hadoop.fs.Path;
import org.apache.hadoop.fs.FileSystem;
import org.apache.hadoop.io.IntWritable;
import org.apache.hadoop.io.LongWritable;
import org.apache.hadoop.io.Text;
import org.apache.hadoop.mapreduce.Job;
import org.apache.hadoop.mapreduce.Mapper;
import org.apache.hadoop.mapreduce.Reducer;
import org.apache.hadoop.mapreduce.lib.input.FileInputFormat;
import org.apache.hadoop.mapreduce.lib.input.TextInputFormat;
import org.apache.hadoop.mapreduce.lib.output.FileOutputFormat;
import org.apache.hadoop.mapreduce.lib.output.TextOutputFormat;


public class CallTypeCount {

    // ================================================================
    // MAPPER CLASS
    // ================================================================
    // Called ONCE per input line (per CSV row)
    // INPUT:  key=byte offset (ignored), value=one CSV line
    // OUTPUT: key=call_type (Text), value=1 (IntWritable)
    
    public static class CallTypeMapper 
            extends Mapper<LongWritable, Text, Text, IntWritable> {
        
        // ── REUSABLE OUTPUT OBJECTS (created ONCE, reused per line) ──
        // This avoids creating millions of objects for millions of lines
        // Hadoop serializes and sends these — new objects = garbage collection pressure
        private final Text outputKey = new Text();
        private final IntWritable outputValue = new IntWritable(1);
        
        // ── Column index for call type in our CSV ──
        // caller_id(0), receiver_id(1), type(2), duration(3), cost(4), timestamp(5), tower(6)
        private static final int CALL_TYPE_COLUMN = 2;
        
        @Override
        protected void map(LongWritable key, Text value, Context context)
                throws IOException, InterruptedException {
            
            // Get the line as a string
            String line = value.toString();
            
            // ── SKIP CSV HEADER ──
            // Cannot use line number (mappers process blocks, not files)
            // Must check CONTENT to identify header
            if (line.startsWith("caller_id")) {
                // This is a header line — skip it
                // Increment a counter so we can verify in job output
                context.getCounter("CallTypeCount", "header_lines_skipped").increment(1);
                return;
            }
            
            // ── PARSE CSV LINE ──
            // Use try-catch to handle malformed lines gracefully
            try {
                String[] fields = line.split(",");
                
                // Validate we have enough columns
                if (fields.length < 7) {
                    context.getCounter("CallTypeCount", "malformed_lines").increment(1);
                    return;
                }
                
                // Extract call type and TRIM whitespace
                String callType = fields[CALL_TYPE_COLUMN].trim();
                
                // Validate call type is not empty
                if (callType.isEmpty()) {
                    context.getCounter("CallTypeCount", "empty_type_lines").increment(1);
                    return;
                }
                
                // ── EMIT KEY-VALUE PAIR ──
                // Set the reusable objects (NOT creating new ones)
                outputKey.set(callType);
                // outputValue is always 1 (set in constructor)
                
                // Write to context — this is how mapper output reaches the shuffle
                context.write(outputKey, outputValue);
                
            } catch (Exception e) {
                // Any parsing error — skip this line, don't crash the job
                context.getCounter("CallTypeCount", "parse_errors").increment(1);
            }
        }
    }
    
    
    // ================================================================
    // REDUCER CLASS
    // ================================================================
    // Called ONCE per unique key (per call type)
    // INPUT:  key=call_type, values=iterable of all 1s for this type
    // OUTPUT: key=call_type, value=total count
    
    public static class CallTypeReducer 
            extends Reducer<Text, IntWritable, Text, IntWritable> {
        
        // Reusable output value object
        private final IntWritable outputValue = new IntWritable();
        
        @Override
        protected void reduce(Text key, Iterable<IntWritable> values, Context context)
                throws IOException, InterruptedException {
            
            // ── SUM ALL VALUES FOR THIS KEY ──
            // values contains ALL the 1s emitted by ALL mappers for this call type
            // 
            // CRITICAL: Iterate ONCE only — this is a streaming iterator
            // CRITICAL: Don't store references to IntWritable objects — they are REUSED
            
            int totalCount = 0;
            
            for (IntWritable val : values) {
                totalCount += val.get();  // .get() extracts the int from IntWritable
            }
            
            // ── EMIT FINAL RESULT ──
            outputValue.set(totalCount);
            context.write(key, outputValue);
            
            // This produces one output line per call type:
            // "voice_call\t542103"
            // The \t (tab) separator is default for TextOutputFormat
        }
    }
    
    
    // ================================================================
    // DRIVER (main method)
    // ================================================================
    // Configures and launches the MapReduce job
    
    public static void main(String[] args) throws Exception {
        
        // ── VALIDATE ARGUMENTS ──
        if (args.length != 2) {
            System.err.println("Usage: CallTypeCount <input_path> <output_path>");
            System.err.println("  input_path:  HDFS path to input CSV files");
            System.err.println("  output_path: HDFS path for output (must NOT exist)");
            System.err.println("");
            System.err.println("Example:");
            System.err.println("  hadoop jar CallTypeCount.jar /telecom/raw/call_records/2024/01/ /telecom/output/call_type_counts");
            System.exit(1);
        }
        
        String inputPath = args[0];
        String outputPath = args[1];
        
        // ── CREATE CONFIGURATION ──
        Configuration conf = new Configuration();
        
        // ── DELETE OUTPUT DIRECTORY IF EXISTS ──
        // Safety: MapReduce refuses to overwrite existing output
        // We handle this programmatically
        FileSystem fs = FileSystem.get(conf);
        Path outPath = new Path(outputPath);
        if (fs.exists(outPath)) {
            System.out.println("Output path exists. Deleting: " + outputPath);
            fs.delete(outPath, true);  // true = recursive
        }
        
        // ── CREATE AND CONFIGURE JOB ──
        Job job = Job.getInstance(conf, "Call Type Count");
        
        // Tell Hadoop which JAR contains our classes
        // This JAR will be distributed to all nodes that run our tasks
        job.setJarByClass(CallTypeCount.class);
        
        // ── SET MAPPER AND REDUCER CLASSES ──
        job.setMapperClass(CallTypeMapper.class);
        job.setReducerClass(CallTypeReducer.class);
        
        // ── SET OUTPUT KEY/VALUE TYPES ──
        // These are the FINAL output types (from reducer)
        job.setOutputKeyClass(Text.class);
        job.setOutputValueClass(IntWritable.class);
        
        // ── SET MAP OUTPUT TYPES (if different from reducer output) ──
        // In our case they are the SAME, but it is good practice to set explicitly
        // If map output types differ from reduce output: you MUST set these
        job.setMapOutputKeyClass(Text.class);
        job.setMapOutputValueClass(IntWritable.class);
        
        // ── SET INPUT AND OUTPUT FORMATS ──
        // TextInputFormat: reads lines, key=byte offset, value=line text
        // TextOutputFormat: writes key\tvalue per line
        job.setInputFormatClass(TextInputFormat.class);
        job.setOutputFormatClass(TextOutputFormat.class);
        
        // ── SET INPUT AND OUTPUT PATHS ──
        FileInputFormat.addInputPath(job, new Path(inputPath));
        FileOutputFormat.setOutputPath(job, outPath);
        
        // ── SET NUMBER OF REDUCERS ──
        // We only have 5 call types — 1 reducer is plenty
        job.setNumReduceTasks(1);
        
        // ── LAUNCH JOB AND WAIT ──
        System.out.println("============================================");
        System.out.println("Starting Call Type Count MapReduce Job");
        System.out.println("Input:  " + inputPath);
        System.out.println("Output: " + outputPath);
        System.out.println("============================================");
        
        long startTime = System.currentTimeMillis();
        boolean success = job.waitForCompletion(true);  // true = print progress
        long endTime = System.currentTimeMillis();
        
        System.out.println("============================================");
        if (success) {
            System.out.println("JOB COMPLETED SUCCESSFULLY");
            System.out.println("Time: " + (endTime - startTime) / 1000 + " seconds");
            System.out.println("Output: " + outputPath);
        } else {
            System.out.println("JOB FAILED");
        }
        System.out.println("============================================");
        
        System.exit(success ? 0 : 1);
    }
}
