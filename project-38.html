PROJECT 38: UNIT TESTING MAPREDUCE WITHOUT A CLUSTER — MRUNIT AND LOCAL RUNNERS

🔴 HYPOTHETICAL PROBLEM

SCENARIO:
→ You lead a 6-person data engineering team at a logistics company
→ Your team maintains 23 MapReduce jobs (Java) and 11 Streaming jobs (Python, Project 37)
→ These jobs process 2TB of shipment data nightly
→ Current development workflow:

   Developer writes code on laptop
        ↓
   Packages JAR
        ↓
   Uploads to staging cluster (shared by 6 people)
        ↓
   Runs full job on 50GB test dataset
        ↓
   Waits 25 minutes for results
        ↓
   Discovers a NullPointerException on line 47
        ↓
   Fixes one character
        ↓
   Re-uploads, re-runs, waits another 25 minutes
        ↓
   REPEAT 8 TIMES PER BUG

ACTUAL COST:
→ Each developer wastes 3-4 hours/day on this cycle
→ 6 developers × 3.5 hours × $80/hour = $1,680/DAY wasted
→ Staging cluster costs $12,000/month
→ Jobs that pass "testing" still break in production
   because edge cases were never checked
→ Last month: A reducer received NULL value for a key
   → Entire nightly pipeline failed at 3 AM
   → On-call engineer spent 4 hours debugging
   → Root cause: One malformed CSV line among 800 million

THE QUESTION:
→ Can we test Mapper logic in 2 SECONDS without any cluster?
→ Can we test Reducer logic with specific edge-case inputs?
→ Can we test the full Map→Shuffle→Reduce pipeline locally?
→ Can we catch that NULL value bug BEFORE deployment?
→ Can we test Python streaming scripts (Project 37) the same way?

✅ WHAT COULD BE DONE

COMPREHENSIVE TESTING STRATEGY FOR MAPREDUCE:

1. UNIT TEST EVERY MAPPER with known inputs
   → Normal case: well-formed input line → expected output pair
   → Edge case: malformed line → no output (or counter increment)
   → Boundary case: empty string, extremely long line
   → Type case: numeric overflow, special characters

2. UNIT TEST EVERY REDUCER with crafted value lists
   → Single value: key with one element
   → Multiple values: key with many elements → correct aggregation
   → Large group: key with 10,000 values → memory handling
   → Empty group: should not happen but defensive coding

3. PIPELINE TEST for end-to-end correctness
   → Known input dataset → known expected output
   → Verify key grouping and sorting
   → Verify combiner does not change final result (Project 9)

4. COUNTER VALIDATION (Project 7)
   → Assert that bad-record counters match expected count
   → Assert that custom business counters are correct

5. INTEGRATION TEST for real HDFS + YARN behavior
   → File format parsing with actual block boundaries
   → Custom InputFormat (Project 32) with real splits
   → Output file structure (MultipleOutputs, Project 20)

6. STREAMING SCRIPT TESTS for Python jobs (Project 37)
   → Unit test mapper.py with mocked stdin
   → Unit test reducer.py with pre-sorted mocked stdin
   → Integration test with pipe: mapper | sort | reducer

✅ WHAT SHOULD DO

1. WRITE TESTS BEFORE WRITING THE MAPPER/REDUCER (TDD)
   → Define expected input → output pairs first
   → Then implement the logic to make tests pass
   → This catches 90% of bugs before any cluster is involved

2. TEST MALFORMED DATA AGGRESSIVELY
   → The #1 cause of production MapReduce failures:
     unexpected data format
   → Create specific test cases for:
     - Missing fields
     - Wrong delimiters
     - Unicode characters
     - Empty lines
     - Header rows mixed with data
     - Numeric fields containing letters

3. USE COUNTERS AS TESTABLE ASSERTIONS
   → From Project 7: counters track job metrics
   → In tests: verify counter values match expected counts
   → Example: "MALFORMED_RECORDS counter should be 3
     when input has 3 bad lines"

4. ISOLATE COMBINER TESTING
   → A combiner (Project 9) must be idempotent
   → Test: run reducer with combiner output as input
   → Result must be identical to running without combiner

5. MAKE TESTS PART OF BUILD PIPELINE
   → mvn test runs ALL MRUnit tests
   → If any test fails → JAR is NOT built
   → If JAR is not built → cannot deploy to cluster
   → ZERO chance of deploying untested code

6. KEEP TEST DATA FILES IN src/test/resources/
   → Small representative samples (10-50 lines)
   → Include edge cases in the sample
   → Version-controlled alongside code

🚫 WHAT SHOULD NOT DO

1. DO NOT test only the "happy path"
   → If you only test well-formed input, you WILL get
     production failures from malformed data
   → Real data is ALWAYS messier than you expect

2. DO NOT use the real cluster for testing logic
   → Cluster testing is for PERFORMANCE and INTEGRATION
   → Logic testing must happen locally in milliseconds
   → If you need a cluster to find a NullPointerException,
     your testing strategy is broken

3. DO NOT skip testing the Shuffle behavior
   → MapDriver tests mapper ONLY
   → ReduceDriver tests reducer ONLY
   → Neither tests whether keys are grouped correctly
   → USE MapReduceDriver for pipeline correctness

4. DO NOT ignore WritableComparable serialization in tests
   → From Project 13: custom keys must serialize/deserialize correctly
   → A key that works in MapDriver might fail during shuffle
     if readFields()/write() have bugs
   → Test serialization round-trip explicitly

5. DO NOT mock Hadoop framework classes manually
   → MRUnit exists precisely to avoid this
   → Hand-written mocks for Context, Reporter, etc. are
     fragile and miss framework behaviors
   → Use the provided test drivers

6. DO NOT treat MiniDFSCluster tests as unit tests
   → They take 5-30 seconds each
   → Run them in a separate test phase (integration-test)
   → Run MRUnit tests in the fast unit-test phase
   → Maven Failsafe plugin for integration, Surefire for unit

🚫 WHEN NOT TO USE

1. DO NOT USE MRUnit for performance testing
   → MRUnit runs everything in a single JVM
   → No parallelism, no network, no disk I/O patterns
   → Performance testing requires real cluster or MiniYARNCluster

2. DO NOT USE MiniDFSCluster for every test
   → Startup cost: 5-30 seconds per test class
   → Use ONLY when testing HDFS-specific behavior
   → Pure logic tests should use MRUnit (milliseconds)

3. DO NOT USE MiniYARNCluster in CI/CD for every commit
   → Takes 30-60 seconds per test
   → Run on nightly builds or pre-release
   → Run MRUnit on every commit (fast feedback)

4. DO NOT USE MRUnit for Spark jobs
   → MRUnit is Hadoop MapReduce specific
   → Spark has its own testing patterns
   → (Bridge to Project 48)

5. DO NOT USE these tools to test cluster configuration
   → Security (Project 41), HA (Project 44), YARN queues (Project 39)
   → These require real multi-node environments
   → Use Docker/Vagrant-based integration environments

6. DO NOT USE MRUnit if your Hadoop version is very new
   → MRUnit was retired from Apache in 2016
   → Still works with Hadoop 2.x and most 3.x
   → For newest Hadoop: use MiniMRCluster + JUnit directly
   → Alternative: hadoop-minicluster dependency

═══════════════════════════════════════════════════════════
MECHANIC 1: HOW MRUNIT WORKS INTERNALLY
═══════════════════════════════════════════════════════════

MRUnit does NOT start any Hadoop daemons.
It creates MOCK implementations of the Hadoop framework.

WHAT HAPPENS WHEN YOU CALL mapDriver.withInput(...).withOutput(...).runTest():

Step 1 — MRUnit creates a MockMapContext
   → Implements MapContext interface
   → Captures all context.write() calls into an internal list
   → No network, no HDFS, no serialization to disk

Step 2 — MRUnit instantiates YOUR Mapper class
   → Calls mapper.setup(mockContext) if overridden
   → Calls mapper.map(key, value, mockContext) with YOUR input
   → Calls mapper.cleanup(mockContext) if overridden

Step 3 — MRUnit collects output
   → Every context.write(key, value) is captured
   → Output stored as List<Pair<KeyOut, ValOut>>

Step 4 — MRUnit compares actual vs. expected
   → Your withOutput(expectedKey, expectedValue) declarations
   → Compared against actual captured output
   → Order-sensitive or order-insensitive depending on configuration
   → If mismatch → JUnit assertion failure with detailed diff

INTERNAL DATA FLOW:

   ┌──────────────────────────────────────────────────┐
   │  MapDriver                                       │
   │                                                  │
   │  withInput(key, value)                           │
   │       │                                          │
   │       ▼                                          │
   │  ┌─────────────────┐                             │
   │  │  YOUR Mapper     │                            │
   │  │  .map(k, v, ctx) │                            │
   │  └────────┬────────┘                             │
   │           │ ctx.write(outK, outV)                │
   │           ▼                                      │
   │  ┌─────────────────────────┐                     │
   │  │ MockMapContext          │                     │
   │  │ captures: [(outK,outV)] │                     │
   │  └────────────┬────────────┘                     │
   │               │                                  │
   │               ▼                                  │
   │  withOutput(expectedK, expectedV)                │
   │       │                                          │
   │       ▼                                          │
   │  ASSERT: captured == expected                    │
   │  → PASS or FAIL with diff                        │
   └──────────────────────────────────────────────────┘

═══════════════════════════════════════════════════════════
MECHANIC 2: HOW MapReduceDriver SIMULATES THE FULL PIPELINE
═══════════════════════════════════════════════════════════

When you use MapReduceDriver (not just MapDriver or ReduceDriver):

Step 1 — Run all inputs through Mapper
   → Same as MapDriver internally
   → Collects all (key, value) pairs from mapper

Step 2 — SIMULATE SHUFFLE AND SORT
   → Groups all values by key
   → Sorts keys using the key's compareTo() method
   → If custom SortComparator defined → uses that
   → If custom GroupingComparator defined → uses that
   → THIS IS THE CRITICAL STEP — catches sorting bugs

   Mapper output: [(B,2), (A,1), (B,3), (A,4)]
                          ↓ shuffle & sort
   Reducer input:  A → [1, 4]
                   B → [2, 3]

Step 3 — If Combiner is set, run it BEFORE reducer
   → Applies combiner to each group
   → Output feeds into reducer
   → Tests that combiner is truly transparent

Step 4 — Run each group through Reducer
   → Same as ReduceDriver internally
   → Collects all reducer output

Step 5 — Compare final output against expected
   → Exact same assertion mechanism as MapDriver

MECHANIC 4: HOW PYTHON STREAMING TESTS WORK
═══════════════════════════════════════════════════════════

From Project 37: Streaming uses stdin/stdout protocol.

⚠️ COMMON PITFALLS AND FAILURE MODES

PITFALL 1: "MRUnit test passes but job fails on cluster"
→ CAUSE: MRUnit does not test serialization round-trip
→ Your Writable's write() and readFields() may have bugs
→ MRUnit passes objects directly — no serialization in unit tests
→ FIX: Add explicit serialization round-trip test:
   Write object to bytes → Read back → Assert equals original

PITFALL 2: "Tests pass with MRUnit but reducer gets wrong values"
→ CAUSE: Custom partitioner (Project 10) sends keys to wrong reducer
→ MRUnit MapReduceDriver does NOT use custom partitioner by default
→ FIX: Test partitioner separately with direct getPartition() calls
→ AND: Use MiniYARNCluster for integration test with real partitioning

PITFALL 3: "MiniDFSCluster fails to start — port already in use"
→ CAUSE: Previous test run crashed without cleanup
→ NameNode/DataNode ports still bound
→ FIX: Use random ports in test configuration
→ FIX: Implement @After cleanup that calls cluster.shutdown()
→ FIX: Use JUnit TemporaryFolder rule for data directories

PITFALL 4: "MRUnit dependency conflicts with Hadoop 3.x"
→ CAUSE: MRUnit was last released for Hadoop 2.x
→ FIX: Use mrunit 1.1.0 with Hadoop 2 classifier
→ ALTERNATIVE: For Hadoop 3.x, use hadoop-minicluster directly
→ ALTERNATIVE: Write custom test harnesses (shown in code below)

PITFALL 5: "MiniYARNCluster test takes 2 minutes"
→ CAUSE: Full YARN startup including ApplicationMaster negotiation
→ FIX: Use @BeforeClass (not @Before) — start cluster ONCE per class
→ FIX: Run multiple test methods against same cluster instance
→ FIX: Set small block size (512 bytes) for test data

PITFALL 6: "Mapper test passes but real job emits nothing"
→ CAUSE: InputFormat parsing differs from test input format
→ MRUnit feeds exact Text objects to mapper
→ Real job uses TextInputFormat which creates (offset, line) pairs
→ FIX: Ensure test input matches what InputFormat actually provides
→ Key is LongWritable (byte offset), NOT line number

PITFALL 7: "Counter test fails intermittently"
→ CAUSE: Tests sharing static state or counter groups
→ MRUnit creates fresh counters per runTest() call
→ But if YOUR code uses static variables → state leaks between tests
→ FIX: Never use static mutable state in Mapper/Reducer classes

🔍 INTERPRETABILITY CONSIDERATIONS

WHAT TEST FAILURES TELL YOU:

1. MapDriver FAILURE:
   "Expected output (shoe, 1) but got (shoe, 2)"
   → Your mapper is double-counting
   → Or your parsing logic extracts wrong field

2. ReduceDriver FAILURE:
   "Expected output (shoe, 10) but got (shoe, 7)"
   → Your aggregation logic is wrong

package com.logistics;

import org.apache.hadoop.io.DoubleWritable;
import org.apache.hadoop.io.LongWritable;
import org.apache.hadoop.io.Text;
import org.apache.hadoop.mapreduce.Mapper;

import java.io.IOException;

public class ShipmentMapper extends Mapper<LongWritable, Text, Text, DoubleWritable> {

    // Reusable objects to avoid garbage collection pressure
    // (Pattern from Project 4 — object reuse in mappers)
    private final Text destinationKey = new Text();
    private final DoubleWritable costValue = new DoubleWritable();

    // Track if this is the first line (header detection)
    private boolean headerSkipped = false;

    @Override
    protected void map(LongWritable key, Text value, Context context)
            throws IOException, InterruptedException {

        String line = value.toString().trim();

        // ── Handle empty lines ──
        if (line.isEmpty()) {
            context.getCounter(ShipmentCounters.EMPTY_LINES).increment(1);
            return;
        }

        // ── Skip CSV header ──
        if (!headerSkipped && line.startsWith("shipment_id")) {
            headerSkipped = true;
            return;
        }
        headerSkipped = true;

        context.getCounter(ShipmentCounters.TOTAL_RECORDS).increment(1);

        // ── Parse CSV fields ──
        String[] fields = line.split(",");

        // Validate field count
        if (fields.length < 6) {
            context.getCounter(ShipmentCounters.MISSING_FIELDS).increment(1);
            context.getCounter(ShipmentCounters.MALFORMED_RECORDS).increment(1);
            return;
        }

        String destination = fields[2].trim();
        String costStr = fields[4].trim();

        // ── Validate cost ──
        double cost;
        try {
            cost = Double.parseDouble(costStr);
        } catch (NumberFormatException e) {
            context.getCounter(ShipmentCounters.MALFORMED_RECORDS).increment(1);
            return;
        }

        // ── Business rule: cost cannot be negative ──
        if (cost < 0) {
            context.getCounter(ShipmentCounters.NEGATIVE_COST).increment(1);
            context.getCounter(ShipmentCounters.MALFORMED_RECORDS).increment(1);
            return;
        }

        // ── Validate weight (optional but tracked) ──
        try {
            double weight = Double.parseDouble(fields[3].trim());
            if (weight <= 0) {
                context.getCounter(ShipmentCounters.INVALID_WEIGHT).increment(1);
                // Still process — weight issue is non-fatal
            }
        } catch (NumberFormatException e) {
            context.getCounter(ShipmentCounters.INVALID_WEIGHT).increment(1);
        }

        // ── Emit valid record ──
        destinationKey.set(destination.toLowerCase());
        costValue.set(cost);
        context.write(destinationKey, costValue);
        context.getCounter(ShipmentCounters.VALID_RECORDS).increment(1);
    }
}

package com.logistics;

import org.apache.hadoop.io.DoubleWritable;
import org.apache.hadoop.io.Text;
import org.apache.hadoop.mapreduce.Reducer;

import java.io.IOException;

public class ShipmentReducer extends Reducer<Text, DoubleWritable, Text, DoubleWritable> {

    private final DoubleWritable totalCost = new DoubleWritable();

    @Override
    protected void reduce(Text key, Iterable<DoubleWritable> values, Context context)
            throws IOException, InterruptedException {

        double sum = 0.0;
        int count = 0;

        for (DoubleWritable val : values) {
            sum += val.get();
            count++;
        }

        // Defensive: if no values received, do not emit
        if (count == 0) {
            return;
        }

        totalCost.set(Math.round(sum * 100.0) / 100.0); // Round to 2 decimal places
        context.write(key, totalCost);
    }
}

            if (file.getPath().getName().startsWith("part-")) {
                BufferedReader reader = new BufferedReader(
                    new InputStreamReader(fs.open(file.getPath())));
                String line;
                while ((line = reader.readLine()) != null) {
                    String[] parts = line.split("\t");
                    if (parts.length == 2) {
                        results.put(parts[0], Double.parseDouble(parts[1]));
                    }
                }
                reader.close();

def mapper_process(input_stream, output_stream):
    """
    Mapper: Parse shipment CSV and emit destination\tcost
    Protocol: Read from input_stream, write to output_stream
    Same as Project 37 stdin/stdout pattern
    """
    for line in input_stream:
        line = line.strip()

        # Skip empty lines
        if not line:
            # Write to stderr for counter simulation
            sys.stderr.write("reporter:counter:Shipment,EmptyLines,1\n")
            continue

        # Skip header
        if line.startswith("shipment_id"):
            continue

        fields = line.split(",")

        # Validate field count
        if len(fields) < 6:
            sys.stderr.write("reporter:counter:Shipment,MalformedRecords,1\n")
            continue

        destination = fields[2].strip().lower()
        cost_str = fields[4].strip()

        # Validate cost
        try:
            cost = float(cost_str)
            if cost < 0:
                sys.stderr.write("reporter:counter:Shipment,NegativeCost,1\n")
                continue
        except ValueError:
            sys.stderr.write("reporter:counter:Shipment,MalformedRecords,1\n")
            continue

        # Emit key\tvalue
        output_stream.write(f"{destination}\t{cost}\n")

def reducer_process(input_stream, output_stream):
    """
    Reducer: Sum costs per destination
    Input is PRE-SORTED by key (Hadoop guarantees this after shuffle)
    Protocol: Read tab-separated key\tvalue from input_stream
    """
    current_key = None
    current_sum = 0.0

    for line in input_stream:
        line = line.strip()
        if not line:
            continue

        parts = line.split("\t")
        if len(parts) != 2:
            continue

        key = parts[0]
        try:
            value = float(parts[1])
        except ValueError:
            continue

        if current_key == key:
            current_sum += value
        else:
            # Emit previous group
            if current_key is not None:
                output_stream.write(f"{current_key}\t{round(current_sum, 2)}\n")
            current_key = key
            current_sum = value

    # Emit last group
    if current_key is not None:
        output_stream.write(f"{current_key}\t{round(current_sum, 2)}\n")

        Simulate: echo input | python mapper.py | sort | python reducer.py

            ['python3', self.mapper_file.name],
            stdin=subprocess.PIPE,
            stdout=subprocess.PIPE,

            stdin=subprocess.PIPE,
            stdout=subprocess.PIPE,
            text=True

            ['python3', self.reducer_file.name],
            stdin=subprocess.PIPE,
            stdout=subprocess.PIPE,
            stderr=subprocess.PIPE,
            text=True
        )
        reducer_out, reducer_err = reducer_proc.communicate(input=sorted_out)

        output_lines = [l for l in reducer_out.strip().split("\n") if l]
        return output_lines
