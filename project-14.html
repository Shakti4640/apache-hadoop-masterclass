PROJECT 14: COMPRESSING OUTPUT TO SAVE CLUSTER STORAGE COSTS

🔴 HYPOTHETICAL PROBLEM

SCENARIO:
→ You are a data engineer at a ride-sharing company
→ Your Hadoop cluster has 500TB total storage across 50 DataNodes
→ HDFS replication factor = 3 → effective capacity = ~167TB
→ Daily data pipeline produces:
    - Raw trip logs:           20GB/day → 60GB with replication
    - Cleaned trip data:       15GB/day → 45GB with replication
    - Aggregated summaries:    5GB/day  → 15GB with replication
    - Intermediate MR output:  30GB/day → 90GB with replication
→ Total daily storage consumption: 210GB/day
→ 30-day retention: 210GB × 30 = 6.3TB/month

THE CRISIS:
→ After 18 months: 113TB consumed → cluster at 68% capacity
→ Growth rate accelerating (more riders, more data)
→ New hardware procurement takes 3 months
→ Budget for new hardware: not approved yet

IMMEDIATE OPTIONS:
  Option A: Delete old data → Compliance says NO (7-year retention)
  Option B: Reduce replication → Risk says NO (fault tolerance)
  Option C: COMPRESS the data → YES — same data, less bytes

THE MATH:
  → Text data compresses 5-10× with Gzip
  → 20GB raw → 2-4GB compressed
  → Daily consumption drops from 210GB to ~40-60GB
  → Buys 12+ months of runway without new hardware

BUT:
  → Which compression codec?
  → Where to compress (map output? final output? both?)
  → Will compressed files still work as MapReduce input?
  → What about CPU overhead of compression/decompression?
  → CRITICAL: Can Hadoop split compressed files for parallel processing?

🔍 HOW TO APPROACH IT

STEP 1: Understand WHERE compression can be applied

There are THREE compression points in a MapReduce job:

STEP 3: Understand WHY splittability is CRITICAL

This is the MOST IMPORTANT concept in Hadoop compression.

💡 HOW CAN WE SOLVE IT

THE SPLITTABILITY PROBLEM:

Recall from Project 12 (InputFormat mechanics):
→ Hadoop splits large files into 128MB blocks
→ Each block → one mapper
→ 10GB file → ~80 mappers running in parallel

WHAT HAPPENS WITH NON-SPLITTABLE COMPRESSED FILES?

┌──────────────────────────────────────────────────────────────┐
│  UNCOMPRESSED 10GB FILE:                                      │
│  ┌─────┐ ┌─────┐ ┌─────┐ ┌─────┐      ┌─────┐              │
│  │128MB│ │128MB│ │128MB│ │128MB│ ···  │128MB│              │
│  │Map 1│ │Map 2│ │Map 3│ │Map 4│      │Map80│              │
│  └─────┘ └─────┘ └─────┘ └─────┘      └─────┘              │
│  → 80 mappers in parallel → finishes in ~2 minutes           │
│                                                               │
│  GZIP COMPRESSED 10GB → 1.5GB .gz FILE:                       │
│  ┌─────────────────────────────────────────────┐              │
│  │          ENTIRE 1.5GB = ONE MAPPER           │              │
│  │  Cannot split: any byte depends on previous  │              │
│  │  Must decompress sequentially from start      │              │
│  └─────────────────────────────────────────────┘              │
│  → 1 mapper for entire file → finishes in ~30 minutes         │
│  → 79 idle mappers wasting cluster resources                  │
│                                                               │
│  BZIP2 COMPRESSED 10GB → 1.2GB .bz2 FILE:                    │
│  ┌─────┐ ┌─────┐ ┌─────┐ ┌─────┐      ┌────┐               │
│  │Split│ │Split│ │Split│ │Split│ ···  │Spl │               │
│  │Map 1│ │Map 2│ │Map 3│ │Map 4│      │Map N│              │
│  └─────┘ └─────┘ └─────┘ └─────┘      └────┘               │
│  → Multiple mappers → finishes in ~5 minutes                  │
│  → BZip2 has block boundaries → splittable                    │
│  → But: BZip2 decompression is CPU-heavy                      │
└──────────────────────────────────────────────────────────────┘

WHY GZIP IS NOT SPLITTABLE:
→ Gzip uses DEFLATE algorithm with sliding window
→ Any byte in the compressed stream may reference previous bytes
→ Cannot start decompression at arbitrary offset
→ Must read from beginning of file

WHY BZIP2 IS SPLITTABLE:
→ BZip2 compresses in independent BLOCKS (100KB-900KB each)
→ Each block starts with magic bytes: 0x314159265359
→ Hadoop can find these boundaries → start new split at any block
→ Each mapper decompresses its blocks independently

SOLUTION FOR OUR PROBLEM:

┌────────────────────────────────────────────────────────────┐
│  COMPRESSION STRATEGY:                                      │
│                                                             │
│  INTERMEDIATE (map output → shuffle):                       │
│    → Use SNAPPY                                             │
│    → WHY: Fastest compress/decompress                       │
│    → WHY: Intermediate data is temporary — ratio less       │
│           important than speed                              │
│    → Splittability irrelevant (intermediate never re-split) │
│                                                             │
│  FINAL OUTPUT (reducer → HDFS):                             │
│    → Use GZIP for archival data (best ratio)                │
│    → Use SNAPPY for data read by next MR job (fast read)    │
│    → Use BZIP2 only when splittability + ratio both needed  │
│                                                             │
│  INPUT (reading compressed files):                          │
│    → Hadoop auto-detects codec from file extension           │
│    → .gz → GzipCodec → single mapper per file               │
│    → .bz2 → BZip2Codec → multiple mappers per file          │
│    → .snappy → SnappyCodec → single mapper per file          │
│    → No code changes needed — transparent decompression      │
└────────────────────────────────────────────────────────────┘

✅ WHAT COULD BE DONE

APPROACH A: Compress final output only (storage savings)
→ Enable output compression in driver
→ Simplest — one configuration change
→ Saves HDFS storage immediately
→ Next job reading this output: auto-decompresses

APPROACH B: Compress intermediate only (network savings)
→ Enable map output compression
→ Reduces shuffle volume by 50-80%
→ Speeds up jobs bottlenecked by network
→ No impact on final output size

APPROACH C: Compress BOTH intermediate AND final (OUR CHOICE)
→ Maximum benefit: less network AND less storage
→ Independent configurations
→ Different codecs for each (Snappy intermediate, Gzip output)
→ Most production clusters use this approach

APPROACH D: Use SequenceFile with block compression
→ SequenceFile is Hadoop's native binary format
→ Supports block compression within the file
→ Splittable even with Snappy/Gzip compression
→ (Covered in Project 30 — file formats)

✅ WHAT SHOULD DO

1. CHOOSE CODEC BASED ON USE CASE — NOT "BEST COMPRESSION"

   ┌────────────────────────────────┬──────────────────────┐
   │ USE CASE                       │ RECOMMENDED CODEC     │
   ├────────────────────────────────┼──────────────────────┤
   │ Map output (intermediate)      │ Snappy or LZ4        │
   │ → Speed matters most           │ (fastest)             │
   ├────────────────────────────────┼──────────────────────┤
   │ Final output, read by next job │ Snappy                │
   │ → Balance of speed + ratio     │ (fast decompress)     │
   ├────────────────────────────────┼──────────────────────┤
   │ Final output, archival/cold    │ Gzip or Zstandard     │
   │ → Ratio matters most           │ (best compression)    │
   ├────────────────────────────────┼──────────────────────┤
   │ Final output, input to next    │ BZip2 or LZO          │
   │ job AND must be splittable     │ (splittable codecs)   │
   ├────────────────────────────────┼──────────────────────┤
   │ Hive/query-optimized tables    │ Snappy with ORC/Parquet│
   │ → Columnar + compression       │ (Project 30)          │
   └────────────────────────────────┴──────────────────────┘

2. ENABLE INTERMEDIATE COMPRESSION ON EVERY JOB
   → Almost always beneficial
   → Snappy adds ~2% CPU overhead
   → Saves 50-70% shuffle data
   → Net result: jobs run FASTER (less network wait)

3. TEST COMPRESSION RATIO ON YOUR ACTUAL DATA
   → Compression ratio varies wildly by data type
   → Repetitive text logs: 10× compression with Gzip
   → Random binary data: 1.1× compression (barely helps)
   → Pre-compressed data (images, video): 0.99× (gets BIGGER)
   → Test on representative sample before committing

4. USE MANY SMALL GZIP FILES INSTEAD OF ONE LARGE GZIP FILE
   → One 10GB .gz file → 1 mapper (no parallelism)
   → 100 × 100MB .gz files → 100 mappers (full parallelism)
   → When generating compressed output: use multiple reducers
   → Each reducer produces one compressed file

5. VERIFY CODEC IS AVAILABLE ON ALL CLUSTER NODES
   → Snappy: included in most Hadoop distributions
   → LZO: requires separate installation + license consideration
   → Gzip: always available (uses Java's built-in GZIPOutputStream)
   → Missing codec on any node → task failure on that node

🚫 WHAT SHOULD NOT DO

1. DO NOT USE GZIP FOR LARGE FILES THAT WILL BE MapReduce INPUT
   → 50GB Gzip file → 1 mapper → 50 minutes
   → 50GB uncompressed → 400 mappers → 2 minutes
   → If file must be Gzip AND large: split before compressing
   → OR use BZip2 (splittable) if CPU budget allows

2. DO NOT COMPRESS ALREADY-COMPRESSED DATA
   → Compressing .jpg, .png, .mp4, .zip files
   → Result: file gets BIGGER (compression overhead + no reduction)
   → Gzip header + metadata > compression savings on random data

3. DO NOT USE BZIP2 FOR INTERMEDIATE COMPRESSION
   → BZip2 is 10-50× slower than Snappy for compress/decompress
   → Intermediate data is temporary — doesn't need best ratio
   → Snappy: 250MB/s compress → shuffles 5× faster than BZip2
   → BZip2 for intermediate turns fast jobs into slow jobs

4. DO NOT FORGET THAT COMPRESSION USES CPU
   → Cluster may be CPU-bound, not I/O-bound
   → Adding compression to CPU-bound job makes it SLOWER
   → Monitor: if CPU usage already > 80%, compression hurts
   → Monitor: if network/disk wait is high, compression helps

5. DO NOT ASSUME ALL CODECS ARE INSTALLED
   → Snappy native library: may need hadoop-snappy-native
   → LZO: GPL-licensed, not in Apache distribution
   → If codec missing: job fails with ClassNotFoundException
   → Test with: hadoop checknative -a

6. DO NOT MIX COMPRESSION EXPECTATIONS ACROSS PIPELINE
   → Job 1 outputs Gzip
   → Job 2 expects uncompressed input
   → Actually fine — Hadoop auto-detects by extension
   → BUT: Job 2 gets only 1 mapper per Gzip file (splittability!)
   → Plan pipeline compression strategy holistically

🔀 WHEN TO USE WHAT

┌──────────────────────────────────┬──────────────────────────────────┐
│ SITUATION                        │ COMPRESSION CHOICE                │
├──────────────────────────────────┼──────────────────────────────────┤
│ Network is bottleneck            │ Snappy intermediate compression  │
│ (shuffle takes > 50% of job)     │ (biggest bang for least CPU)     │
├──────────────────────────────────┼──────────────────────────────────┤
│ Storage is bottleneck            │ Gzip output compression          │

├──────────────────────────────────┼──────────────────────────────────┤
│ Output is input to next MR job   │ Snappy in SequenceFile           │
│ (pipeline of jobs)               │ (splittable + fast)              │
│                                  │ OR many small Gzip files         │
├──────────────────────────────────┼──────────────────────────────────┤
│ Output is cold archival data     │ Gzip or BZip2                    │
│ (rarely read again)              │ (maximum compression)            │
├──────────────────────────────────┼──────────────────────────────────┤
│ CPU is bottleneck                │ NO compression                   │
│ (complex mapper/reducer logic)   │ (or very light: LZ4)             │
├──────────────────────────────────┼──────────────────────────────────┤
│ Hive tables for frequent queries │ Snappy with ORC or Parquet       │
│                                  │ (Project 30)                     │
├──────────────────────────────────┼──────────────────────────────────┤
│ Single large file must remain    │ BZip2 (splittable, good ratio)   │
│ splittable after compression     │ OR LZO with index                │
└──────────────────────────────────┴──────────────────────────────────┘

🚫 WHEN NOT TO USE

DO NOT COMPRESS WHEN:

1. DATA IS ALREADY COMPRESSED OR BINARY
   → Images, videos, ZIP files, encrypted data
   → Compression ratio ≈ 1.0 (no savings)
   → CPU wasted on futile compression attempt

2. CPU IS THE BOTTLENECK
   → Complex regex parsing, ML inference in mapper
   → CPU at 90% → adding compression pushes to 100%
   → Job slows down despite less I/O
   → Profile first: is job I/O-bound or CPU-bound?

3. DATA IS VERY SMALL
   → 10MB file → compressed to 2MB → saves 8MB × 3 replication = 24MB
   → 24MB savings is irrelevant on a 500TB cluster
   → Compression setup overhead may exceed savings

4. LOW-LATENCY REQUIREMENT
   → Compression adds latency to every read/write
   → For time-critical jobs, uncompressed may be faster
   → Snappy latency is minimal, Gzip/BZip2 latency is significant

5. DEBUGGING PHASE
   → Compressed output cannot be easily inspected
   → hdfs dfs -cat file.gz shows binary garbage
   → Need: hdfs dfs -cat file.gz | gunzip
   → During development: keep output uncompressed
   → Enable compression when moving to production

MECHANIC 1: HOW HADOOP AUTO-DETECTS COMPRESSION
═══════════════════════════════════════════════════════════

When Hadoop reads an input file, it checks the file extension:

  CompressionCodecFactory.getCodec(Path file):
    ".gz"     → GzipCodec
    ".bz2"    → BZip2Codec
    ".snappy" → SnappyCodec
    ".lz4"    → Lz4Codec
    ".lzo"    → LzoCodec (if installed)
    ".zst"    → ZStandardCodec
    (none)    → null (uncompressed)

  If codec found:
    → InputFormat wraps the input stream in a DecompressorStream
    → RecordReader reads decompressed bytes transparently
    → Mapper sees UNCOMPRESSED data — no code changes needed

  ┌──────────────┐    ┌───────────────────┐    ┌──────────────┐
  │ HDFS File     │ →  │ DecompressorStream│ →  │ RecordReader │
  │ data.txt.gz   │    │ (GzipCodec)       │    │ (lines)      │
  │ (compressed)  │    │ decompresses      │    │ (uncompressed)│
  └──────────────┘    └───────────────────┘    └──────┬───────┘
                                                       │
                                                       ▼
                                               ┌──────────────┐
                                               │   MAPPER      │
                                               │ sees plain    │
                                               │ text lines    │
                                               └──────────────┘

═══════════════════════════════════════════════════════════
MECHANIC 2: INTERMEDIATE COMPRESSION (MAP OUTPUT)
═══════════════════════════════════════════════════════════

When mapreduce.map.output.compress = true:

NORMAL FLOW (no compression):
  Mapper → MapOutputBuffer (100MB) → Spill to disk → Shuffle raw

COMPRESSED FLOW:
  Mapper → MapOutputBuffer (100MB) → COMPRESS → Spill to disk → Shuffle compressed
  
  At reducer: DECOMPRESS → Merge-sort → Reduce

  WHERE compression happens:
  ┌──────────────────────────────────────────────────────────┐
  │  MapOutputBuffer reaches 80% full                         │
  │    ↓                                                      │
  │  Sort buffer by (partition, key) — same as always         │
  │    ↓                                                      │
  │  Run Combiner (if set, from Project 9) — same as always   │
  │    ↓                                                      │
  │  ┌────────────────────────────────────┐                   │
  │  │ NEW STEP: Compress each partition  │                   │
  │  │ section before writing to disk     │                   │
  │  │                                    │                   │
  │  │ Partition 0 data → compress → disk │                   │
  │  │ Partition 1 data → compress → disk │                   │
  │  │ Partition 2 data → compress → disk │                   │
  │  └────────────────────────────────────┘                   │
  │    ↓                                                      │
  │  Spill file on local disk is SMALLER                      │
  │    ↓                                                      │
  │  During shuffle: compressed data sent over network        │
  │  → Less bytes transferred = faster shuffle                │
  │    ↓                                                      │
  │  Reducer receives compressed data                         │
  │  → Decompresses during merge-sort phase                   │
  └──────────────────────────────────────────────────────────┘

  SAVINGS CALCULATION:
  → Map output: 10GB uncompressed
  → With Snappy: ~4GB compressed (60% reduction)
  → Shuffle transfers 4GB instead of 10GB
  → On 1Gbps network: 32 seconds saved vs 80 seconds
  → CPU cost: ~0.5 seconds for Snappy compression
  → NET BENEFIT: 47.5 seconds faster

═══════════════════════════════════════════════════════════
MECHANIC 4: SPLITTABLE COMPRESSION INTERNALS
═══════════════════════════════════════════════════════════

WHY GZIP CANNOT BE SPLIT:

  Gzip uses DEFLATE with a 32KB sliding window:
  
  ┌──────────────────────────────────────────────────────┐
  │  Compressed byte at position 5000:                    │
  │  "This refers to data at position 4800"               │
  │                                                       │
  │  Compressed byte at position 5,000,000:               │
  │  "This refers to data at position 4,999,800"          │
  │                                                       │
  │  To decompress byte 5,000,000:                        │
  │  → Must have decompressed byte 4,999,800              │
  │  → Which needs byte 4,999,600                         │
  │  → Which needs byte 4,999,400                         │
  │  → ... all the way back to byte 0                     │
  │                                                       │
  │  CANNOT start decompression at arbitrary offset        │
  └──────────────────────────────────────────────────────┘

WHY BZIP2 CAN BE SPLIT:

  BZip2 compresses in independent blocks:
  
  ┌────────────────────────────────────────────────────────┐
  │  BZip2 file structure:                                  │
  │                                                         │
  │  [File Header]                                          │
  │  [Block 1: magic=0x314159265359][compressed data]       │
  │  [Block 2: magic=0x314159265359][compressed data]       │
  │  [Block 3: magic=0x314159265359][compressed data]       │
  │  ...                                                    │
  │  [File Footer]                                          │
  │                                                         │
  │  Each block is INDEPENDENTLY decompressible              │
  │  Block 3 does NOT depend on Block 1 or Block 2          │
  │                                                         │
  │  Hadoop's BZip2Codec.getSplits():                       │
  │  → Scans file for magic byte sequences                  │
  │  → Creates split at each block boundary                 │
  │  → Each mapper decompresses its blocks independently    │
  └────────────────────────────────────────────────────────┘

═══════════════════════════════════════════════════════════
MECHANIC 5: CODEC CLASS HIERARCHY
═══════════════════════════════════════════════════════════

  CompressionCodec (interface)
    ├── GzipCodec
    │     → createOutputStream() → GZIPOutputStream
    │     → createInputStream() → GZIPInputStream
    │     → getDefaultExtension() → ".gz"
    │
    ├── BZip2Codec
    │     → createOutputStream() → BZip2CompressorOutputStream
    │     → createInputStream() → BZip2CompressorInputStream
    │     → getDefaultExtension() → ".bz2"
    │     → implements SplittableCompressionCodec ← KEY
    │
    ├── SnappyCodec
    │     → Uses native Snappy library (JNI)
    │     → createOutputStream() → SnappyCompressorOutputStream
    │     → getDefaultExtension() → ".snappy"
    │
    ├── Lz4Codec
    │     → Uses native LZ4 library
    │     → getDefaultExtension() → ".lz4"
    │
    └── ZStandardCodec
          → Uses native Zstandard library
          → getDefaultExtension() → ".zstd"

  SplittableCompressionCodec extends CompressionCodec
    → Adds: createInputStream(seekableIn, start, end, readMode)
    → Allows starting decompression at arbitrary offset
    → Only BZip2Codec and LzoCodec implement this

⚠️ COMMON PITFALLS AND FAILURE MODES

PITFALL 1: "Job reads 50GB Gzip file with only 1 mapper"
→ CAUSE: Gzip is not splittable
→ 50GB .gz file → entire file = one InputSplit = one mapper
→ FIX: Either:
  a) Store as many small .gz files (not one large)
  b) Use BZip2 instead (splittable)
  c) Store in SequenceFile with block compression (Project 30)
  d) Decompress before storing on HDFS

PITFALL 2: "SnappyCodec not found — ClassNotFoundException"
→ CAUSE: Snappy native library not installed on cluster nodes
→ Hadoop Snappy requires: libsnappy.so on Linux
→ CHECK: hadoop checknative -a
→ Expected: snappy: true /path/to/libsnappy.so
→ FIX: Install snappy-devel on all nodes
       OR use Gzip (pure Java, always available)

PITFALL 3: "Compressed output is LARGER than uncompressed"
→ CAUSE: Data is already compressed or highly random
→ Compression adds headers + metadata
→ Random data: Gzip output ≈ 100.1% of input
→ FIX: Test compression ratio on sample before enabling
→ Skip compression for binary/encrypted data

PITFALL 4: "Job runs slower with compression enabled"
→ CAUSE: Job is CPU-bound, not I/O-bound
→ Compression adds CPU load
→ If CPU was already bottleneck → overall slower
→ DIAGNOSIS: Check YARN counters:
  → "CPU time spent (ms)" vs "GC time" vs "Shuffle bytes"
  → If shuffle bytes are small → compression helps little
  → If CPU time is high → compression hurts
→ FIX: Profile before enabling compression

PITFALL 5: "Compressed intermediate output causes reduce failures"
→ CAUSE: Reducer cannot decompress (codec mismatch or missing)
→ Map side compresses with SnappyCodec
→ Reducer node doesn't have Snappy native library
→ ERROR: java.lang.UnsatisfiedLinkError: snappy
→ FIX: Ensure codec libraries on ALL nodes, not just some

PITFALL 6: "Cannot read compressed output with hdfs dfs -cat"
→ CAUSE: -cat outputs raw bytes (compressed)
→ FIX: hdfs dfs -text /path/to/file.gz
→ The -text command auto-detects codec and decompresses
→ OR: hdfs dfs -cat /path/to/file.gz | gunzip

import org.apache.hadoop.conf.Configuration;
import org.apache.hadoop.conf.Configured;
import org.apache.hadoop.fs.Path;

import org.apache.hadoop.io.compress.GzipCodec;

public class CompressedTripAnalyzer extends Configured implements Tool {

    // ════════════════════════════════════════════════════════
    // MAPPER: Parse trip records → emit (zone, fare)
    // ════════════════════════════════════════════════════════
    //
    // Input format (pipe-delimited trip records):
    //   TRIP_ID|DRIVER_ID|PICKUP_ZONE|DROPOFF_ZONE|DISTANCE|FARE|TIP|TIMESTAMP
    //   T001|D042|Manhattan|Brooklyn|8.5|24.50|5.00|2024-01-15 09:30:00
    //
    // IMPORTANT: If input is compressed (.gz, .bz2, .snappy),
    // Hadoop decompresses AUTOMATICALLY — mapper sees plain text.
    // ZERO code changes needed to read compressed input.

    public static class TripMapper
            extends Mapper<LongWritable, Text, Text, DoubleWritable> {

        private Text zoneKey = new Text();
        private DoubleWritable fareValue = new DoubleWritable();

        enum TripCounters {
            TOTAL_RECORDS,
            VALID_TRIPS,
            INVALID_RECORDS,
            HEADER_SKIPPED
        }

        @Override
        protected void map(LongWritable offset, Text line, Context context)
                throws IOException, InterruptedException {

            context.getCounter(TripCounters.TOTAL_RECORDS).increment(1);

            String record = line.toString().trim();
            if (record.isEmpty()) return;

            // Skip header
            if (record.startsWith("TRIP_ID")) {
                context.getCounter(TripCounters.HEADER_SKIPPED).increment(1);
                return;
            }

            String[] fields = record.split("\\|");
            if (fields.length != 8) {
                context.getCounter(TripCounters.INVALID_RECORDS).increment(1);
                return;
            }

            String pickupZone = fields[2].trim();
            String fareStr = fields[5].trim();

            try {
                double fare = Double.parseDouble(fareStr);
                if (fare < 0) {
                    context.getCounter(TripCounters.INVALID_RECORDS).increment(1);
                    return;
                }

                zoneKey.set(pickupZone);
                fareValue.set(fare);
                context.write(zoneKey, fareValue);
                context.getCounter(TripCounters.VALID_TRIPS).increment(1);

            } catch (NumberFormatException e) {
                context.getCounter(TripCounters.INVALID_RECORDS).increment(1);
            }
        }
    }


    // ════════════════════════════════════════════════════════
    // COMBINER + REDUCER: Sum fares per pickup zone
    // ════════════════════════════════════════════════════════
    //
    // Combiner and Reducer are IDENTICAL for sum operations
    // (from Project 9 — combiner safe for associative operations)
    //
    // Compression does NOT affect combiner/reducer logic.
    // Compression is handled by the FRAMEWORK, not your code.

    public static class FareSumReducer
            extends Reducer<Text, DoubleWritable, Text, DoubleWritable> {

        private DoubleWritable totalFare = new DoubleWritable();

        @Override
        protected void reduce(Text zone, Iterable<DoubleWritable> fares,
                              Context context) throws IOException, InterruptedException {
            double sum = 0.0;
            for (DoubleWritable fare : fares) {
                sum += fare.get();
            }
            totalFare.set(Math.round(sum * 100.0) / 100.0);
            context.write(zone, totalFare);
        }
    }


    // ════════════════════════════════════════════════════════
    // DRIVER: Configure compression at all three points
    // ════════════════════════════════════════════════════════

    @Override
    public int run(String[] args) throws Exception {

        if (args.length < 3) {
            System.err.println("Usage: CompressedTripAnalyzer <input> <output> <codec>");
            System.err.println("  codec: none | gzip | snappy | bzip2");
            System.err.println();
            System.err.println("Examples:");
            System.err.println("  CompressedTripAnalyzer /input /output none");
            System.err.println("  CompressedTripAnalyzer /input /output gzip");
            System.err.println("  CompressedTripAnalyzer /input /output snappy");
            System.err.println("  CompressedTripAnalyzer /input /output bzip2");
            return 1;
        }

        String inputPath = args[0];
        String outputPath = args[1];
        String codecChoice = args[2].toLowerCase();

        Configuration conf = getConf();
        Job job = Job.getInstance(conf, "Trip Fare Analysis - Compression: " + codecChoice);
        job.setJarByClass(CompressedTripAnalyzer.class);

        // ── Standard MR configuration ──
        job.setMapperClass(TripMapper.class);
        job.setCombinerClass(FareSumReducer.class);
        job.setReducerClass(FareSumReducer.class);
        job.setOutputKeyClass(Text.class);
        job.setOutputValueClass(DoubleWritable.class);
        job.setNumReduceTasks(3);


        // ══════════════════════════════════════════════════════
        // COMPRESSION CONFIGURATION — THE CORE OF PROJECT 14
        // ══════════════════════════════════════════════════════

        Class<? extends CompressionCodec> codecClass = null;

        switch (codecChoice) {
            case "none":
                // ── NO COMPRESSION ──
                // Baseline for comparison
                System.out.println("COMPRESSION: DISABLED (baseline)");
                break;

            case "gzip":
                codecClass = GzipCodec.class;
                System.out.println("COMPRESSION: GZIP");
                System.out.println("  → Best compression ratio (5-10×)");
                System.out.println("  → Slower compress/decompress");
                System.out.println("  → Output NOT splittable");
                System.out.println("  → Output files: part-r-*.gz");
                break;

            case "snappy":
                codecClass = SnappyCodec.class;
                System.out.println("COMPRESSION: SNAPPY");
                System.out.println("  → Fastest compress/decompress");
                System.out.println("  → Moderate ratio (2-3×)");
                System.out.println("  → Output NOT splittable");
                System.out.println("  → Output files: part-r-*.snappy");
                break;

            case "bzip2":
                codecClass = BZip2Codec.class;
                System.out.println("COMPRESSION: BZIP2");
                System.out.println("  → Best ratio + SPLITTABLE");
                System.out.println("  → Slowest compress/decompress");
                System.out.println("  → Output IS splittable (unique!)");
                System.out.println("  → Output files: part-r-*.bz2");
                break;

            default:
                System.err.println("Unknown codec: " + codecChoice);
                System.err.println("Use: none | gzip | snappy | bzip2");
                return 1;
        }

        if (codecClass != null) {
            // ─────────────────────────────────────────────
            // POINT 2: INTERMEDIATE COMPRESSION (Map Output)
            // ─────────────────────────────────────────────
            // Compresses mapper output BEFORE shuffle
            // Reduces network transfer volume
            // Always use fast codec (Snappy) for intermediate
            // regardless of final output codec choice

            conf.setBoolean("mapreduce.map.output.compress", true);
            conf.setClass("mapreduce.map.output.compress.codec",
                    SnappyCodec.class,  // ALWAYS Snappy for intermediate
                    CompressionCodec.class);

            // NOTE: We use Snappy for intermediate even when final
            // output is Gzip. Different codecs for different purposes:
            //   Intermediate: speed matters (Snappy)
            //   Final output: ratio matters (Gzip)

            // ─────────────────────────────────────────────
            // POINT 3: OUTPUT COMPRESSION (Final Output)
            // ─────────────────────────────────────────────
            // Compresses reducer output written to HDFS
            // Uses the codec specified by user

            FileOutputFormat.setCompressOutput(job, true);
            FileOutputFormat.setOutputCompressorClass(job, codecClass);

            // COMPRESSION TYPE for SequenceFile output (not used here):
            // SequenceFileOutputFormat.setOutputCompressionType(job,
            //     SequenceFile.CompressionType.BLOCK);
            // → NONE: no compression
            // → RECORD: compress each record independently
            // → BLOCK: compress blocks of records (best ratio + splittable)
        }

        // ── NOTE ON POINT 1: INPUT COMPRESSION ──
        // No configuration needed!
        // If input files have .gz/.bz2/.snappy extension,
        // Hadoop auto-detects and decompresses.
        // Our mapper code is IDENTICAL regardless of input format.


        // ── Input/Output paths ──
        FileInputFormat.addInputPath(job, new Path(inputPath));
        FileOutputFormat.setOutputPath(job, new Path(outputPath));

        // ── Run and report ──
        long startTime = System.currentTimeMillis();
        boolean success = job.waitForCompletion(true);
        long elapsed = System.currentTimeMillis() - startTime;

        if (success) {
            System.out.println("\n╔═══════════════════════════════════════════════╗");
            System.out.printf("║  COMPRESSION: %-10s                        ║%n", codecChoice.toUpperCase());
            System.out.printf("║  Job time:    %,d ms                         ║%n", elapsed);
            System.out.printf("║  Map input bytes:    %,d%n",
                job.getCounters()
                   .findCounter("org.apache.hadoop.mapreduce.FileInputFormatCounter",
                                "BYTES_READ").getValue());
            System.out.printf("║  Map output bytes:   %,d%n",
                job.getCounters()
                   .findCounter("org.apache.hadoop.mapreduce.TaskCounter",
                                "MAP_OUTPUT_BYTES").getValue());
            System.out.printf("║  Map output materialized: %,d%n",
                job.getCounters()
                   .findCounter("org.apache.hadoop.mapreduce.TaskCounter",
                                "MAP_OUTPUT_MATERIALIZED_BYTES").getValue());
            System.out.printf("║  HDFS bytes written: %,d%n",
                job.getCounters()
                   .findCounter("org.apache.hadoop.mapreduce.FileOutputFormatCounter",
                                "BYTES_WRITTEN").getValue());
            System.out.println("║");

            long mapOut = job.getCounters()
                .findCounter("org.apache.hadoop.mapreduce.TaskCounter",
                             "MAP_OUTPUT_BYTES").getValue();
            long mapMat = job.getCounters()
                .findCounter("org.apache.hadoop.mapreduce.TaskCounter",
                             "MAP_OUTPUT_MATERIALIZED_BYTES").getValue();

            if (mapOut > 0 && mapMat > 0 && mapMat < mapOut) {
                double intermediateRatio = (double) mapOut / mapMat;
                System.out.printf("║  Intermediate compression ratio: %.1f×%n",
                        intermediateRatio);
                System.out.printf("║  Shuffle savings: %.1f%%%n",
                        (1.0 - (double) mapMat / mapOut) * 100);
            }

            System.out.println("╚═══════════════════════════════════════════════╝\n");
        }

        return success ? 0 : 1;
    }


    public static void main(String[] args) throws Exception {
        int exitCode = ToolRunner.run(new Configuration(),
                                      new CompressedTripAnalyzer(), args);
        System.exit(exitCode);
    }

import os

def analyze_compression_options(filepath, sample_size_mb=10):
    """
    Test all available compression codecs on a sample of your data.
    Produces a recommendation based on actual measurements.
    """
    
    print("═" * 70)
    print("COMPRESSION ANALYSIS FOR YOUR DATA")
    print("═" * 70)
    
    file_size = os.path.getsize(filepath)
    file_size_mb = file_size / (1024 * 1024)
    print(f"\nInput file: {filepath}")
    print(f"File size:  {file_size_mb:.1f} MB")
    
    # Read sample data
    sample_bytes = min(sample_size_mb * 1024 * 1024, file_size)
    with open(filepath, 'rb') as f:
        sample_data = f.read(int(sample_bytes))
    
    sample_mb = len(sample_data) / (1024 * 1024)
    print(f"Sample:     {sample_mb:.1f} MB")
    print()
    
    results = []
    
    # ── Test 1: Gzip ──
    print("Testing GZIP...", end=" ", flush=True)
    start = time.time()
    compressed = gzip.compress(sample_data, compresslevel=6)
    compress_time = time.time() - start
    
    start = time.time()
    _ = gzip.decompress(compressed)
    decompress_time = time.time() - start
    
    ratio = len(sample_data) / len(compressed)
    compress_speed = sample_mb / max(compress_time, 0.001)
    decompress_speed = sample_mb / max(decompress_time, 0.001)
    
    results.append({
        "codec": "Gzip",
        "ratio": ratio,
        "compress_speed": compress_speed,
        "decompress_speed": decompress_speed,
        "splittable": False,
        "compressed_size": len(compressed)
    })
    print(f"ratio={ratio:.1f}×, compress={compress_speed:.0f} MB/s, decompress={decompress_speed:.0f} MB/s")
    
    # ── Test 2: BZip2 ──
    print("Testing BZIP2...", end=" ", flush=True)
    start = time.time()
    compressed = bz2.compress(sample_data, compresslevel=6)
    compress_time = time.time() - start
    
    start = time.time()
    _ = bz2.decompress(compressed)
    decompress_time = time.time() - start
    
    ratio = len(sample_data) / len(compressed)
    compress_speed = sample_mb / max(compress_time, 0.001)
    decompress_speed = sample_mb / max(decompress_time, 0.001)
    
    results.append({
        "codec": "BZip2",
        "ratio": ratio,
        "compress_speed": compress_speed,
        "decompress_speed": decompress_speed,
        "splittable": True,
        "compressed_size": len(compressed)
    })
    print(f"ratio={ratio:.1f}×, compress={compress_speed:.0f} MB/s, decompress={decompress_speed:.0f} MB/s")
    
    # ── Test 3: Zlib (similar to Snappy performance profile) ──
    # Python doesn't have native Snappy easily, so we use zlib level 1
    # as a rough proxy for fast compression
    import zlib
    print("Testing FAST (zlib-1, ~Snappy proxy)...", end=" ", flush=True)
    start = time.time()
    compressed = zlib.compress(sample_data, level=1)
    compress_time = time.time() - start
    
    start = time.time()
    _ = zlib.decompress(compressed)
    decompress_time = time.time() - start
    
    ratio = len(sample_data) / len(compressed)
    compress_speed = sample_mb / max(compress_time, 0.001)
    decompress_speed = sample_mb / max(decompress_time, 0.001)
    
    results.append({
        "codec": "Snappy*",
        "ratio": ratio,
        "compress_speed": compress_speed,
        "decompress_speed": decompress_speed,
        "splittable": False,
        "compressed_size": len(compressed)
    })
    print(f"ratio={ratio:.1f}×, compress={compress_speed:.0f} MB/s, decompress={decompress_speed:.0f} MB/s")
    print("  (* approximated with zlib level 1)")
    
    # ── Results table ──
    print("\n" + "─" * 70)
    print(f"{'CODEC':<12} {'RATIO':<8} {'COMPRESS':<14} {'DECOMPRESS':<14} {'SPLITTABLE':<12} {'SIZE'}")
    print("─" * 70)
    
    for r in results:
        size_mb = r['compressed_size'] / (1024 * 1024)
        print(f"{r['codec']:<12} {r['ratio']:<8.1f}× "
              f"{r['compress_speed']:<14.0f}MB/s "
              f"{r['decompress_speed']:<14.0f}MB/s "
              f"{'YES' if r['splittable'] else 'NO':<12} "
              f"{size_mb:.1f} MB")
    
    print("─" * 70)
    
    # ── Recommendations ──
    print("\n" + "═" * 70)
    print("RECOMMENDATIONS FOR YOUR DATA:")
    print("═" * 70)

    print("─" * 80)

        if splittable:

        else:

        if "100 ×" in name:

🔑 KEY TAKEAWAY

WHAT PROJECT 14 ESTABLISHED:

✅ THREE COMPRESSION POINTS:
   → Input (auto-detected from file extension — no code change)
   → Intermediate/Map output (reduces shuffle volume — Snappy best)
   → Final output (reduces HDFS storage — Gzip best for archival)

✅ SPLITTABILITY IS THE CRITICAL FACTOR:
   → Gzip: NOT splittable → 1 mapper per file (kills parallelism)
   → BZip2: SPLITTABLE → multiple mappers per file (preserves parallelism)
   → Snappy: NOT splittable → use many small files as workaround
   → SequenceFile with block compression: splittable + any codec (Project 30)

✅ CODEC SELECTION RULES:
   → Intermediate: ALWAYS Snappy (fastest, temporary data)
   → Archival output: Gzip (best ratio, rarely re-read)
   → Pipeline output: Snappy in many files OR BZip2
   → CPU-bound jobs: NO compression (or very light LZ4)

✅ COMPRESSION IS TRANSPARENT:
   → Mapper/Reducer code is IDENTICAL with or without compression
   → Only driver configuration changes
   → Input decompression is automatic (by file extension)
   → Output compression adds extension automatically
