PROJECT 20: WRITING ANALYSIS RESULTS TO MULTIPLE OUTPUT DIRECTORIES SIMULTANEOUSLY

🔴 HYPOTHETICAL PROBLEM

SCENARIO:
→ You are a data engineer at a healthcare analytics company
→ 200 million patient visit records processed through:
    - Map-Only ETL (Project 11) for cleaning
    - Reduce-Side Join (Project 17) for enriching with hospital data
    - Secondary Sort (Project 19) for grouping by patient, sorting by date
→ Now the compliance team needs THREE different outputs
   from a SINGLE MapReduce job:

OUTPUT REQUIREMENT 1: "by_department" directory
  → All records split into separate files by department name
  → /output/by_department/cardiology/part-r-00000
  → /output/by_department/neurology/part-r-00000
  → /output/by_department/oncology/part-r-00000
  → File name derived DYNAMICALLY from record content

OUTPUT REQUIREMENT 2: "flagged" directory
  → Patients with more than 20 visits in 30 days → potential fraud
  → /output/flagged/part-r-00000
  → Different format: only PATIENT_ID, VISIT_COUNT, DATE_RANGE

OUTPUT REQUIREMENT 3: "summary" directory
  → One line per patient: total visits, total charges, last visit date
  → /output/summary/part-r-00000

INPUT FORMAT (from Project 19 Secondary Sort output):
  P00001 | 2024-06-15 | V8830012 | Cardiology | Dr. Smith  | 1250.00
  P00001 | 2024-06-01 | V8820034 | Cardiology | Dr. Smith  | 890.00
  P00001 | 2024-05-10 | V7710045 | Neurology  | Dr. Jones  | 2100.00
  P00002 | 2024-04-22 | V8840078 | Oncology   | Dr. Wilson | 15000.00
  ...200 million more rows...

THE PROBLEM WITH DEFAULT MapReduce:
→ Standard job writes ALL output to ONE directory
→ ONE format: part-r-00000, part-r-00001, ...
→ Cannot write to /by_department/ AND /flagged/ AND /summary/
→ Cannot name files by department dynamically
→ Cannot output different key-value formats to different destinations

PREVIOUS WORKAROUND (BAD):
→ Run THREE separate MapReduce jobs
→ Job 1: filter + write by_department
→ Job 2: filter + write flagged
→ Job 3: aggregate + write summary
→ PROBLEM: Reads 200M records THREE times → 3x I/O cost
→ PROBLEM: Three jobs to schedule, monitor, handle failures

CORRECT SOLUTION: MultipleOutputs
→ ONE MapReduce job reads data ONCE
→ Writes to THREE output channels simultaneously
→ Dynamic file naming based on record content
→ Different key-value types per channel

✅ WHAT SHOULD DO

1. ALWAYS CLOSE MultipleOutputs IN cleanup()
   → MultipleOutputs buffers writes internally
   → If not closed: data loss — last batch never flushed to HDFS
   → This is the #1 bug with MultipleOutputs

   @Override
   protected void cleanup(Context context) throws ... {
       multipleOutputs.close();  // MANDATORY
   }

2. DEFINE NAMED OUTPUTS IN THE DRIVER, NOT THE MAPPER/REDUCER
   → Named outputs are JOB-LEVEL configuration
   → Must be set before job.waitForCompletion()
   → Mapper/Reducer only WRITE to pre-defined channels

3. USE DYNAMIC PATHS FOR DATA-DRIVEN FILE ORGANIZATION
   → Static named output: multipleOutputs.write("dept", key, value)
     → Creates: /output/dept-r-00000
   → Dynamic path: multipleOutputs.write("dept", key, value, "cardiology/")
     → Creates: /output/cardiology/dept-r-00000
   → The dynamic path can be ANY string derived from record data
   → Supports subdirectories: "2024/06/cardiology/"

4. SET LazyOutputFormat TO AVOID EMPTY FILES
   → Default: Hadoop creates part-r-NNNNN for EVERY reducer
     even if that reducer writes nothing to a channel
   → LazyOutputFormat creates files ONLY when first record is written
   → Prevents cluttering output with empty files

   LazyOutputFormat.setOutputFormatClass(job, TextOutputFormat.class);

5. SUPPRESS DEFAULT OUTPUT IF USING ONLY NAMED OUTPUTS
   → Default context.write() still creates part-r-NNNNN files
   → If you ONLY use multipleOutputs.write(), the default files are empty
   → Set NullOutputFormat as the job's output format to suppress them
   → OR use LazyOutputFormat to prevent empty default files

🚫 WHAT SHOULD NOT DO

1. DO NOT forget to close MultipleOutputs
   → Repeating because this causes SILENT DATA LOSS
   → No error thrown — job succeeds but output is incomplete
   → Always implement cleanup() method

2. DO NOT mix context.write() and multipleOutputs.write() carelessly
   → context.write() → goes to DEFAULT output (part-r-NNNNN)
   → multipleOutputs.write() → goes to NAMED output channel
   → If you use both, you get DEFAULT files + NAMED files
   → Usually you want ONE OR THE OTHER, not both
   → If only using MultipleOutputs: set NullOutputFormat for default

3. DO NOT create MultipleOutputs in the constructor
   → Create in setup() method
   → Constructor may be called before Context is fully initialized
   → setup() guarantees proper initialization order

4. DO NOT use dynamic paths with user-controlled input without sanitization
   → Dynamic path "cardiology/" creates directory /output/cardiology/
   → If department value is "../../etc/passwd" → PATH TRAVERSAL
   → ALWAYS sanitize: replace non-alphanumeric characters
   → departmentClean = department.replaceAll("[^a-zA-Z0-9]", "_")

5. DO NOT create excessive unique dynamic paths
   → Each unique dynamic path creates a SEPARATE output file PER REDUCER
   → 1000 unique departments × 100 reducers = 100,000 output files
   → HDFS NameNode metadata overhead (Project 1: small files problem)
   → RULE: If unique paths > 500, reconsider the design

6. DO NOT define named output with wrong key-value types
   → If named output defined with Text.class, IntWritable.class
   → But reducer writes IntWritable, Text → ClassCastException
   → Types must match EXACTLY between definition and write calls

🚫 WHEN NOT TO USE

1. SINGLE OUTPUT DESTINATION
   → If all records go to one directory in one format
   → Default context.write() is simpler and faster
   → MultipleOutputs adds overhead per write call

2. OUTPUT SPLITTING ALREADY HANDLED BY PARTITIONER
   → If you want records split by region and region IS the reduce key
   → Custom Partitioner (Project 10) already sends each region
     to a specific reducer → separate part-r-NNNNN files
   → MultipleOutputs is redundant here

3. POST-PROCESSING CAN HANDLE SPLITTING
   → If a simple script can split output after job completes
   → e.g., grep "Cardiology" output.txt > cardiology.txt
   → For small data: simpler than MultipleOutputs complexity

4. THOUSANDS OF UNIQUE OUTPUT PATHS
   → 10,000 unique departments × 50 reducers = 500,000 files
   → NameNode overwhelmed (small files problem — Project 1)
   → Consider: write to one file, then post-process
   → OR: use a column in the data for downstream filtering

5. WHEN USING HIVE/PIG/SPARK
   → Hive: INSERT OVERWRITE ... PARTITION (dept) handles this natively
   → Pig: STORE with MultiStorage
   → Spark: partitionBy() on DataFrameWriter
   → Raw MultipleOutputs in MapReduce is for Java-only pipelines

⚙️ HOW IT ONLY WORKS — INTERNAL MECHANICS

═══════════════════════════════════════════════════════════════
INTERNAL MECHANIC 1: HOW MultipleOutputs MANAGES FILE HANDLES
═══════════════════════════════════════════════════════════════

  When you call:
    multipleOutputs.write("dept", key, value, "cardiology/")

  INTERNALLY:
  
  Step 1 — Construct a unique file identifier
    → baseOutputPath = "cardiology/"
    → namedOutput = "dept"
    → taskId = reducer number (e.g., 00002)
    → fileId = "cardiology/dept-r-00002"

  Step 2 — Check if RecordWriter already exists for this fileId
    → MultipleOutputs maintains a CACHE of RecordWriters
    → HashMap<String, RecordWriter> writerCache

    ┌───────────────────────────────────────────────────────┐
    │ writerCache:                                          │
    │   "cardiology/dept-r-00002"  → RecordWriter_1        │
    │   "neurology/dept-r-00002"   → RecordWriter_2        │
    │   "flagged/flagged-r-00002"  → RecordWriter_3        │
    │   "summary/summary-r-00002"  → RecordWriter_4        │
    └───────────────────────────────────────────────────────┘

  Step 3 — If writer exists → use it directly
           If writer NOT exists → create new RecordWriter
    → New RecordWriter opens a NEW HDFS output stream
    → Uses the OutputFormat specified for this named output
    → Caches it for future writes to same fileId

  Step 4 — Write key-value through the RecordWriter
    → RecordWriter serializes key and value
    → Writes to HDFS output stream
    → Buffered internally (HDFS client-side buffer)

  Step 5 — On close() → flush and close ALL cached RecordWriters
    → THIS is why close() is mandatory
    → Without it: buffered data never reaches HDFS

═══════════════════════════════════════════════════════════════
INTERNAL MECHANIC 2: FILE NAMING CONVENTION
═══════════════════════════════════════════════════════════════

  Given:
    → Named output: "dept"
    → Dynamic base path: "cardiology/"
    → Reducer task number: 00002
    → Output directory: /output/

  DEFAULT output (context.write):
    /output/part-r-00002

  NAMED output WITHOUT dynamic path:
    multipleOutputs.write("dept", key, value)
    → /output/dept-r-00002

  NAMED output WITH dynamic path:
    multipleOutputs.write("dept", key, value, "cardiology/")
    → /output/cardiology/dept-r-00002

  NAMED output WITH nested dynamic path:
    multipleOutputs.write("dept", key, value, "2024/06/cardiology/")
    → /output/2024/06/cardiology/dept-r-00002

  NAMING PATTERN:
    {outputDir}/{dynamicBasePath}/{namedOutput}-{m|r}-{taskNumber}

    m = mapper output (map-only jobs)
    r = reducer output (jobs with reducers)

═══════════════════════════════════════════════════════════════
INTERNAL MECHANIC 3: HOW NAMED OUTPUTS ARE CONFIGURED
═══════════════════════════════════════════════════════════════

  When driver calls:
    MultipleOutputs.addNamedOutput(job, "dept",
        TextOutputFormat.class, Text.class, Text.class);

  INTERNALLY:
  
  Step 1 — Stores configuration in Job's Configuration object
    → Key: "mapreduce.multipleoutputs.namedOutput.dept.format"
      Value: "org.apache.hadoop.mapreduce.lib.output.TextOutputFormat"
    → Key: "mapreduce.multipleoutputs.namedOutput.dept.key"
      Value: "org.apache.hadoop.io.Text"
    → Key: "mapreduce.multipleoutputs.namedOutput.dept.value"
      Value: "org.apache.hadoop.io.Text"
    → Key: "mapreduce.multipleoutputs"
      Value: "dept flagged summary" (space-separated list)

  Step 2 — Configuration serialized and sent to all tasks
    → Every mapper and reducer receives these settings
    → MultipleOutputs constructor reads them to know valid channels

  Step 3 — At runtime, MultipleOutputs validates write() calls
    → If you write to undefined channel "unknown"
    → Exception: "Undefined named output 'unknown'"

═══════════════════════════════════════════════════════════════
INTERNAL MECHANIC 4: INTERACTION WITH LazyOutputFormat
═══════════════════════════════════════════════════════════════

  WITHOUT LazyOutputFormat:
    → Hadoop creates output files for ALL reducers immediately
    → Reducer 5 writes nothing to "flagged" channel
    → Still creates: /output/flagged/flagged-r-00005 (EMPTY file)
    → 100 reducers × 3 channels = up to 300 files even if most empty

  WITH LazyOutputFormat:
    → Wraps the real OutputFormat in a lazy proxy
    → File is created ONLY when first write() happens
    → Reducer 5 never writes to "flagged" → NO file created
    → Reduces file count significantly

  ┌─────────────────────────────────────────────────────────┐
  │  Without Lazy:  100 reducers × 3 channels = 300 files  │
  │  With Lazy:     Only files with actual data created     │
  │                 Might be 100 + 15 + 100 = 215 files    │
  │                 (not all reducers have flagged patients) │
  └─────────────────────────────────────────────────────────┘

═══════════════════════════════════════════════════════════════
INTERNAL MECHANIC 5: THREAD SAFETY AND RESOURCE MANAGEMENT
═══════════════════════════════════════════════════════════════

  → Each mapper/reducer task runs in its OWN JVM (separate process)
  → MultipleOutputs instance is PER-TASK, not shared
  → No thread safety concerns between tasks
  → BUT: Within one task, MultipleOutputs is NOT thread-safe
    → Do not share across threads within a single map/reduce call
    → Standard MapReduce is single-threaded per task → no issue

  → RESOURCE CONCERN: Each unique file = one open HDFS stream
    → 50 unique dynamic paths in one reducer = 50 open streams
    → Each stream uses network socket + memory buffer
    → Too many open streams → "Too many open files" OS error
    → LIMIT: Stay under 100 unique paths per reducer

⚠️ COMMON PITFALLS AND FAILURE MODES

PITFALL 1: "Output files are empty or missing last batch of records"
→ CAUSE: cleanup() not implemented → multipleOutputs.close() never called
→ MultipleOutputs buffers writes in RecordWriter
→ Without close(), final buffer never flushed
→ FIX: ALWAYS implement cleanup() with multipleOutputs.close()
→ PATTERN:
    private MultipleOutputs<Text, Text> multipleOutputs;
    
    setup()   → multipleOutputs = new MultipleOutputs<>(context);
    reduce()  → multipleOutputs.write(...)
    cleanup() → multipleOutputs.close();

PITFALL 2: "Empty part-r-NNNNN files alongside named output files"
→ CAUSE: Job still creates DEFAULT output files
→ If you only use multipleOutputs.write() and never context.write()
→ Default files are empty but still created
→ FIX: Use LazyOutputFormat:
    LazyOutputFormat.setOutputFormatClass(job, TextOutputFormat.class);
→ OR: Set NullOutputFormat as job output format:
    job.setOutputFormatClass(NullOutputFormat.class);

PITFALL 3: "Undefined named output 'xyz'" exception
→ CAUSE: Named output not registered in driver with addNamedOutput()
→ Every channel used in write() MUST be defined in driver first
→ FIX: Verify all channel names match between driver and reducer

PITFALL 4: "Too many open files" OS error
→ CAUSE: Too many unique dynamic paths per reducer
→ Each path = separate HDFS output stream = OS file descriptor
→ 500 unique departments × 1 reducer = 500 open streams
→ OS limit is typically 1024 or 4096 file descriptors
→ FIX: Reduce unique paths, or increase OS ulimit
→ BETTER FIX: Reconsider design — do you really need 500 output files?

PITFALL 5: "ClassCastException when writing to named output"
→ CAUSE: Key-value types in write() don't match addNamedOutput() definition
→ addNamedOutput(job, "flagged", ..., Text.class, IntWritable.class)
→ multipleOutputs.write("flagged", intKey, textValue)  ← SWAPPED
→ FIX: Match types exactly — first type = key, second = value

PITFALL 6: "Dynamic path creates unexpected directory structure"
→ CAUSE: Dynamic path string has leading/trailing issues
→ write("dept", k, v, "cardiology")  → /output/cardiology-dept-r-00000
→ write("dept", k, v, "cardiology/") → /output/cardiology/dept-r-00000
→ The trailing "/" makes it a DIRECTORY, without "/" it's a FILE PREFIX
→ FIX: Always end dynamic paths with "/" for directory semantics

PITFALL 7: "Named output files from different reducers have overlapping data"
→ CAUSE: Not a bug — this is EXPECTED behavior
→ Reducer 0 creates: /output/cardiology/dept-r-00000
→ Reducer 1 creates: /output/cardiology/dept-r-00001
→ Both contain cardiology records — from different partitions
→ Concatenating them gives ALL cardiology records
→ If you need ONE file per department → use 1 reducer (but loses parallelism)
→ OR: Post-process with hdfs dfs -getmerge

SCALING DIMENSION 2: NUMBER OF UNIQUE DYNAMIC PATHS

  ┌──────────────────────────┬──────────────────────────────────┐
  │ Unique Paths Per Reducer │ Impact                           │
  ├──────────────────────────┼──────────────────────────────────┤
  │ 1-10                     │ No issue                         │
  │ 10-100                   │ Watch open file descriptors      │
  │ 100-500                  │ Increase ulimit, monitor memory  │
  │ 500+                     │ DANGER — OS limits, NameNode     │
  │                          │ metadata explosion               │
  └──────────────────────────┴──────────────────────────────────┘

  TOTAL FILES = unique_paths × num_reducers
  50 departments × 100 reducers = 5,000 files
  → Manageable for HDFS
  
  5,000 departments × 100 reducers = 500,000 files
  → NameNode metadata: 500K × 150 bytes = 75MB
  → Not catastrophic but unnecessary
  → SOLUTION: Reduce granularity or post-process

SCALING DIMENSION 3: WRITE THROUGHPUT

  → Each write() call involves:
    1. HashMap lookup for cached RecordWriter (~10ns)
    2. Serialization of key + value (~100ns-1μs)
    3. Buffer append (~10ns)
    4. Occasional buffer flush to HDFS (~1ms when it happens)
  
  → Overhead vs context.write(): ~20% slower per record
  → For 200M records: ~40 extra seconds
  → Negligible compared to I/O time

SCALING DIMENSION 4: MEMORY USAGE

  → Each open RecordWriter has a write buffer
  → Default buffer: ~4KB per writer
  → 100 unique paths per reducer × 4KB = 400KB
  → Not a concern
  → BUT: If using SequenceFileOutputFormat → larger buffers
  → AND: Each writer holds HDFS DFSOutputStream → network resources

SCALING DIMENSION 5: INTERACTION WITH SECONDARY SORT (Project 19)

  → Secondary Sort + MultipleOutputs = powerful combination
  → Patient records arrive grouped by patient, sorted by date
  → Reducer computes statistics across pre-sorted group
  → Writes to multiple channels from same reduce() call
  → No additional shuffle overhead — MultipleOutputs is output-only

🔍 INTERPRETABILITY CONSIDERATIONS

HOW TO VERIFY MULTIPLE OUTPUTS ARE CORRECT:

1. CHECK DIRECTORY STRUCTURE
   hdfs dfs -ls -R /output/
   → Should show three directories: by_department/, flagged/, summary/
   → Each with appropriately named files

2. CHECK FILE COUNTS PER CHANNEL
   hdfs dfs -ls /output/by_department/cardiology/ | wc -l
   → Should equal number of reducers (or fewer with LazyOutputFormat)

3. CHECK RECORD COUNTS ACROSS CHANNELS
   → Total records in by_department/* should equal input record count
   → Records in flagged/ should be subset of total
   → Records in summary/ should equal unique patient count

4. CROSS-REFERENCE FLAGGED PATIENTS
   → Every patient in flagged/ should also appear in by_department/
   → Flagged visit count should match actual count in by_department/

5. USE COUNTERS FOR ACCOUNTING
   → Counter: DEPT_OUTPUT_RECORDS → total written to dept channel
   → Counter: FLAGGED_RECORDS → total written to flagged channel
   → Counter: SUMMARY_RECORDS → total written to summary channel
   → DEPT_OUTPUT_RECORDS should equal total input records

public class PatientVisitMultiOutput extends Configured implements Tool {

    // Channel names — defined as constants for consistency
    static final String CHANNEL_DEPT = "dept";
    static final String CHANNEL_FLAGGED = "flagged";
    static final String CHANNEL_SUMMARY = "summary";

    // ══════════════════════════════════════════════════════
    // MAPPER: Parse patient visit records
    //         Emit (patientId, full record)
    //         Grouping by patientId happens in shuffle
    // ══════════════════════════════════════════════════════

    public static class VisitMapper
            extends Mapper<LongWritable, Text, Text, Text> {

        private Text patientKey = new Text();

        @Override
        protected void map(LongWritable offset, Text line, Context context)
                throws IOException, InterruptedException {

            String record = line.toString().trim();

            // Skip empty lines and header
            if (record.isEmpty() || record.startsWith("PATIENT_ID")) {
                return;
            }

            // Parse: PATIENT_ID | VISIT_DATE | VISIT_ID | DEPARTMENT | DOCTOR | CHARGE
            String[] fields = record.split("\\|");

            if (fields.length < 6) {
                context.getCounter("QUALITY", "MALFORMED_RECORDS").increment(1);
                return;
            }

            String patientId = fields[0].trim();
            patientKey.set(patientId);

            context.write(patientKey, line);
            context.getCounter("QUALITY", "VALID_RECORDS").increment(1);
        }
    }

    // ══════════════════════════════════════════════════════
    // REDUCER: Process each patient's visits and write to
    //          THREE output channels simultaneously
    //
    // Channel 1 (dept):    Each record → dynamic path by department
    // Channel 2 (flagged): High-frequency patients → flat file
    // Channel 3 (summary): One summary line per patient
    // ══════════════════════════════════════════════════════

    public static class MultiOutputReducer
            extends Reducer<Text, Text, NullWritable, Text> {

        // ── MultipleOutputs instance — the core of this project ──
        private MultipleOutputs<Text, Text> multipleOutputs;
        private Text outputValue = new Text();

        private static final int FRAUD_THRESHOLD_VISITS = 20;
        private static final int FRAUD_THRESHOLD_DAYS = 30;
        private static final DateTimeFormatter DATE_FMT =
            DateTimeFormatter.ofPattern("yyyy-MM-dd");

        // ── SETUP: Create MultipleOutputs instance ──
        @Override
        protected void setup(Context context)
                throws IOException, InterruptedException {
            // MultipleOutputs reads named output configuration
            // from the Job's Configuration (set in driver)
            multipleOutputs = new MultipleOutputs<>(context);
        }

        @Override
        protected void reduce(Text patientId, Iterable<Text> visits,
                              Context context)
                throws IOException, InterruptedException {

            // Collect visit data for this patient
            // (We need to scan twice: once for dept output, once for stats)
            // Since Iterable can only be traversed once (Project 19 pitfall 7),
            // we collect into a list — acceptable because per-patient visit
            // count is bounded (typically < 1000)
            List<String[]> visitList = new ArrayList<>();

            for (Text visit : visits) {
                String[] fields = visit.toString().split("\\|");
                if (fields.length >= 6) {
                    visitList.add(fields);
                }
            }

            if (visitList.isEmpty()) {
                return;
            }

            // ════════════════════════════════════════════
            // CHANNEL 1: Write each record to dept channel
            //             with DYNAMIC PATH by department
            // ════════════════════════════════════════════

            double totalCharges = 0.0;
            String latestDate = "0000-00-00";
            String earliestDate = "9999-99-99";

            for (String[] fields : visitList) {
                String visitDate = fields[1].trim();
                String department = fields[3].trim();
                String chargeStr = fields[5].trim();

                // Sanitize department name for use as directory path
                // Remove special characters that could break HDFS paths
                String deptClean = department
                    .replaceAll("[^a-zA-Z0-9]", "_")
                    .toLowerCase();

                // Build the full record line
                String fullRecord = String.join(" | ",
                    fields[0].trim(), fields[1].trim(), fields[2].trim(),
                    fields[3].trim(), fields[4].trim(), fields[5].trim());
                outputValue.set(fullRecord);

                // ── WRITE TO DEPT CHANNEL WITH DYNAMIC PATH ──
                // This creates: /output/by_department/{deptClean}/dept-r-NNNNN
                multipleOutputs.write(
                    CHANNEL_DEPT,                    // named output channel
                    NullWritable.get(),              // key (we don't need one)
                    outputValue,                     // value (full record)
                    "by_department/" + deptClean + "/"  // dynamic base path
                );

                context.getCounter("OUTPUT", "DEPT_RECORDS").increment(1);

                // Accumulate stats for summary and flagging
                try {
                    totalCharges += Double.parseDouble(chargeStr);
                } catch (NumberFormatException e) {
                    // Skip bad charge values
                }

                if (visitDate.compareTo(latestDate) > 0) {
                    latestDate = visitDate;
                }
                if (visitDate.compareTo(earliestDate) < 0) {
                    earliestDate = visitDate;
                }
            }

            // ════════════════════════════════════════════
            // CHANNEL 2: Check for fraud pattern
            //             >20 visits in 30-day window
            // ════════════════════════════════════════════

            boolean isFlagged = false;

            if (visitList.size() >= FRAUD_THRESHOLD_VISITS) {
                // Check if there's a 30-day window with >=20 visits
                // Simple approach: check if total visits within
                // (latest - earliest) <= 30 days is >= threshold
                // More sophisticated: sliding window (production would do this)

                try {
                    LocalDate earliest = LocalDate.parse(earliestDate, DATE_FMT);
                    LocalDate latest = LocalDate.parse(latestDate, DATE_FMT);
                    long daySpan = ChronoUnit.DAYS.between(earliest, latest);

                    // Simple heuristic: if average frequency is high enough
                    if (daySpan <= FRAUD_THRESHOLD_DAYS
                        && visitList.size() >= FRAUD_THRESHOLD_VISITS) {
                        isFlagged = true;
                    }

                    // Also flag if visit rate exceeds threshold in ANY 30-day window
                    // Simplified: check overall rate
                    if (daySpan > 0) {
                        double visitsPerDay = (double) visitList.size() / daySpan;
                        if (visitsPerDay * FRAUD_THRESHOLD_DAYS >= FRAUD_THRESHOLD_VISITS) {
                            isFlagged = true;
                        }
                    }
                } catch (Exception e) {
                    // Date parsing failed — skip flagging
                }
            }

            if (isFlagged) {
                // ── WRITE TO FLAGGED CHANNEL ──
                // Different format: PATIENT_ID | VISIT_COUNT | DATE_RANGE | TOTAL_CHARGES
                String flaggedRecord = String.format("%s | %d | %s to %s | %.2f",
                    patientId.toString(),
                    visitList.size(),
                    earliestDate,
                    latestDate,
                    totalCharges
                );
                outputValue.set(flaggedRecord);

                multipleOutputs.write(
                    CHANNEL_FLAGGED,        // channel
                    NullWritable.get(),     // key
                    outputValue             // value
                    // No dynamic path — all flagged go to same directory
                );

                context.getCounter("OUTPUT", "FLAGGED_PATIENTS").increment(1);
            }

            // ════════════════════════════════════════════
            // CHANNEL 3: Write summary for EVERY patient
            //             One line: PATIENT_ID | TOTAL_VISITS | TOTAL_CHARGES | LAST_VISIT
            // ════════════════════════════════════════════

            String summaryRecord = String.format("%s | %d | %.2f | %s",
                patientId.toString(),
                visitList.size(),
                totalCharges,
                latestDate
            );
            outputValue.set(summaryRecord);

            multipleOutputs.write(
                CHANNEL_SUMMARY,        // channel
                NullWritable.get(),     // key
                outputValue             // value
            );

            context.getCounter("OUTPUT", "SUMMARY_RECORDS").increment(1);
        }

        // ── CLEANUP: MUST close MultipleOutputs ──
        // Without this: DATA LOSS — last buffer never flushed
        @Override
        protected void cleanup(Context context)
                throws IOException, InterruptedException {
            multipleOutputs.close();
        }
    }

    // ══════════════════════════════════════════════════════
    // DRIVER: Configure named outputs and submit job
    // ══════════════════════════════════════════════════════

    @Override
    public int run(String[] args) throws Exception {

        if (args.length != 3) {
            System.err.println(
                "Usage: PatientVisitMultiOutput <input> <output> <numReducers>");
            return -1;
        }

        Configuration conf = getConf();
        Job job = Job.getInstance(conf, "Patient Visit Multi-Output Analysis");
        job.setJarByClass(PatientVisitMultiOutput.class);

        // ── Mapper ──
        job.setMapperClass(VisitMapper.class);
        job.setMapOutputKeyClass(Text.class);
        job.setMapOutputValueClass(Text.class);

        // ── Reducer ──
        job.setReducerClass(MultiOutputReducer.class);
        job.setNumReduceTasks(Integer.parseInt(args[2]));

        // ── Input ──
        job.setInputFormatClass(TextInputFormat.class);
        FileInputFormat.addInputPath(job, new Path(args[0]));

        // ── Output base directory ──
        FileOutputFormat.setOutputPath(job, new Path(args[1]));

        // ════════════════════════════════════════════════════
        // DEFINE NAMED OUTPUTS — THE KEY CONFIGURATION
        // Each channel can have different OutputFormat and types
        // ════════════════════════════════════════════════════

        // Channel 1: Department records (dynamic subdirectories)
        MultipleOutputs.addNamedOutput(
            job,
            CHANNEL_DEPT,                    // channel name
            TextOutputFormat.class,          // output format
            NullWritable.class,              // key class
            Text.class                       // value class
        );

        // Channel 2: Flagged patients (flat directory)
        MultipleOutputs.addNamedOutput(
            job,
            CHANNEL_FLAGGED,
            TextOutputFormat.class,
            NullWritable.class,
            Text.class
        );

        // Channel 3: Patient summaries (flat directory)
        MultipleOutputs.addNamedOutput(
            job,
            CHANNEL_SUMMARY,
            TextOutputFormat.class,
            NullWritable.class,
            Text.class
        );

        // ════════════════════════════════════════════════════
        // USE LazyOutputFormat TO PREVENT EMPTY FILES
        // Without this: every reducer creates files for every channel
        // even if it has no data for that channel
        // ════════════════════════════════════════════════════

        LazyOutputFormat.setOutputFormatClass(job, TextOutputFormat.class);

        // NOTE: LazyOutputFormat replaces job.setOutputFormatClass()
        // It wraps TextOutputFormat in a lazy proxy
        // Default output files (part-r-NNNNN) are also lazy now
        // Since we only use multipleOutputs.write() and never context.write(),
        // no default files will be created

        // ── Submit ──
        System.out.println("=== PATIENT VISIT MULTI-OUTPUT ANALYSIS ===");
        System.out.println("Named outputs: dept, flagged, summary");
        System.out.println("Dept output uses dynamic paths by department name");
        System.out.println("LazyOutputFormat enabled — no empty files");

        boolean success = job.waitForCompletion(true);

        if (success) {
            System.out.println("\n=== JOB COMPLETED SUCCESSFULLY ===");

            long deptRecords = job.getCounters()
                .findCounter("OUTPUT", "DEPT_RECORDS").getValue();
            long flaggedPatients = job.getCounters()
                .findCounter("OUTPUT", "FLAGGED_PATIENTS").getValue();
            long summaryRecords = job.getCounters()
                .findCounter("OUTPUT", "SUMMARY_RECORDS").getValue();
            long validRecords = job.getCounters()
                .findCounter("QUALITY", "VALID_RECORDS").getValue();

            System.out.println("Input records:      " + validRecords);
            System.out.println("Dept records:       " + deptRecords
                + " (should equal input)");
            System.out.println("Flagged patients:   " + flaggedPatients);
            System.out.println("Summary records:    " + summaryRecords
                + " (should equal unique patients)");

            System.out.println("\nOutput structure:");
            System.out.println("  " + args[1] + "/by_department/{dept_name}/dept-r-NNNNN");
            System.out.println("  " + args[1] + "/flagged-r-NNNNN");
            System.out.println("  " + args[1] + "/summary-r-NNNNN");
        }

        return success ? 0 : 1;
    }

    public static void main(String[] args) throws Exception {
        int exitCode = ToolRunner.run(
            new Configuration(), new PatientVisitMultiOutput(), args);
        System.exit(exitCode);
    }
}

public class MapOnlyMultiOutput extends Configured implements Tool {

    // ══════════════════════════════════════════════════════
    // MAPPER: Route each record to department-specific file
    //         NO reducer — output goes directly from mapper
    //
    // WHY map-only:
    //   → No aggregation needed — just splitting/routing
    //   → No shuffle overhead (major performance gain)
    //   → Each mapper independently writes to its channels
    //
    // CAVEAT:
    //   → File naming uses "m" not "r": dept-m-00000
    //   → Multiple mappers may write to same department
    //   → More output files: num_mappers × num_departments
    // ══════════════════════════════════════════════════════

    public static class RoutingMapper
            extends Mapper<LongWritable, Text, NullWritable, Text> {

        private MultipleOutputs<NullWritable, Text> multipleOutputs;

        @Override
        protected void setup(Context context)
                throws IOException, InterruptedException {
            multipleOutputs = new MultipleOutputs<>(context);
        }

        @Override
        protected void map(LongWritable offset, Text line, Context context)
                throws IOException, InterruptedException {

            String record = line.toString().trim();
            if (record.isEmpty() || record.startsWith("PATIENT_ID")) {
                return;
            }

            String[] fields = record.split("\\|");
            if (fields.length < 6) {
                context.getCounter("QUALITY", "MALFORMED").increment(1);
                return;
            }

            String department = fields[3].trim()
                .replaceAll("[^a-zA-Z0-9]", "_")
                .toLowerCase();

            // Write directly from mapper — no shuffle, no reduce
            // File created: /output/by_department/{dept}/dept-m-NNNNN
            //                                              ^^ "m" for mapper
            multipleOutputs.write(
                "dept",
                NullWritable.get(),
                line,
                "by_department/" + department + "/"
            );

            context.getCounter("OUTPUT", "ROUTED_RECORDS").increment(1);
        }

        @Override
        protected void cleanup(Context context)
                throws IOException, InterruptedException {
            multipleOutputs.close();  // STILL MANDATORY in mapper
        }
    }

    @Override
    public int run(String[] args) throws Exception {

        if (args.length != 2) {
            System.err.println("Usage: MapOnlyMultiOutput <input> <output>");
            return -1;
        }

        Configuration conf = getConf();
        Job job = Job.getInstance(conf, "Map-Only Department Routing");
        job.setJarByClass(MapOnlyMultiOutput.class);

        job.setMapperClass(RoutingMapper.class);

        // ── ZERO REDUCERS — map-only job ──
        // (Project 11 pattern)
        job.setNumReduceTasks(0);

        job.setOutputKeyClass(NullWritable.class);
        job.setOutputValueClass(Text.class);

        FileInputFormat.addInputPath(job, new Path(args[0]));
        FileOutputFormat.setOutputPath(job, new Path(args[1]));

        // Define named output
        MultipleOutputs.addNamedOutput(job, "dept",
            TextOutputFormat.class, NullWritable.class, Text.class);

        // Lazy output to prevent empty files
        LazyOutputFormat.setOutputFormatClass(job, TextOutputFormat.class);

        System.out.println("=== MAP-ONLY MULTI-OUTPUT ROUTING ===");
        System.out.println("Zero reducers — no shuffle overhead");
        System.out.println("Each mapper independently routes to department files");

        boolean success = job.waitForCompletion(true);

        if (success) {
            long routed = job.getCounters()
                .findCounter("OUTPUT", "ROUTED_RECORDS").getValue();
            System.out.println("Records routed: " + routed);
        }

        return success ? 0 : 1;
    }

    public static void main(String[] args) throws Exception {
        int exitCode = ToolRunner.run(
            new Configuration(), new MapOnlyMultiOutput(), args);
        System.exit(exitCode);
    }
}
