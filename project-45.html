PROJECT 45: RESOURCEMANAGER HIGH AVAILABILITY AND RECOVERY

🔴 HYPOTHETICAL PROBLEM

SCENARIO:
→ You are the platform reliability lead at a logistics analytics company
→ Your Hadoop cluster is NOW production-hardened:
    - HDFS Federation with 3 namespaces (Project 43)
    - NameNode HA with automatic failover (Project 44)
    - Kerberos security (Project 41) + encryption (Project 42)
→ You have 2,000 DataNodes processing 15PB of shipping data
→ 400 concurrent jobs run at any given time:
    - 150 MapReduce ETL pipelines
    - 120 Hive analytical queries
    - 80 Pig data flow scripts
    - 50 Spark-on-YARN jobs

FRIDAY 11:23 PM — EVERYTHING BREAKS:
→ YARN ResourceManager process crashes (JVM OutOfMemoryError)
→ HDFS is FINE — NameNode HA keeps storage accessible
→ BUT:
    - ALL 400 running jobs lose their ApplicationMaster
    - No new jobs can be submitted
    - NodeManagers on 2,000 machines are orphaned
    - Running containers continue briefly, then timeout and die
    - Hive queries from the analytics dashboard → "Connection refused"
    - Scheduled Oozie workflows (Project 31) pile up in WAITING state

IMPACT WITHOUT RM HA:
→ 11:23 PM: ResourceManager crashes
→ 11:25 PM: All running ApplicationMasters detect RM is gone
→ 11:30 PM: Containers start timing out
→ 11:35 PM: All 400 jobs FAIL
→ 11:40 PM: On-call engineer receives 400 "job failed" alerts simultaneously
→ 12:00 AM: Engineer restarts ResourceManager manually
→ 12:05 AM: RM starts, loads state... but NO state was preserved
→ 12:10 AM: All 400 jobs must be RESUBMITTED manually
→ 12:30 AM: Queue backlog — 400 old jobs + 50 new jobs competing
→ 2:00 AM: Cluster stabilizes — 2.5 HOURS of total disruption
→ Morning: Finance team discovers overnight shipping cost analysis MISSING
→ Operations team: delivery route optimization ran with STALE data

THE QUESTION:
→ NameNode HA (Project 44) protects STORAGE
→ Can we do the SAME for COMPUTATION?
→ Can a Standby ResourceManager take over seamlessly?
→ Can RUNNING jobs SURVIVE the failover without restarting?
→ YES — That is ResourceManager High Availability + Work-Preserving Recovery

🔍 HOW TO APPROACH IT

STEP 1: Understand why RM failure is DIFFERENT from NN failure

Recall from Project 22:
→ ResourceManager is the MASTER of YARN
→ It does THREE things:
  1. SCHEDULING: Decides which job gets which resources
  2. APPLICATION TRACKING: Monitors ApplicationMasters
  3. RESOURCE ALLOCATION: Assigns containers to NodeManagers

Recall from Project 44:
→ NameNode HA needed JournalNodes for shared EditLog
→ Because NameNode state = persistent filesystem metadata
→ Every edit must be durably stored and replicated

KEY DIFFERENCE:
→ NameNode state is PERSISTENT (filesystem metadata — must survive forever)
→ ResourceManager state is TRANSIENT (running jobs — temporary by nature)
→ This means RM HA is SIMPLER than NN HA in some ways

  ┌─────────────────────────────────────────────────────────────┐
  │         NAMENODE HA vs RESOURCEMANAGER HA                    │
  ├─────────────────────────┬───────────────────────────────────┤
  │ NameNode HA             │ ResourceManager HA                │
  ├─────────────────────────┼───────────────────────────────────┤
  │ Needs JournalNodes      │ NO JournalNodes needed            │
  │ (shared EditLog)        │ (state stored in ZooKeeper        │
  │                         │  or filesystem state store)       │
  ├─────────────────────────┼───────────────────────────────────┤
  │ Standby continuously    │ Standby loads state from          │
  │ tails edits in real-time│ ZK/state-store upon promotion     │
  ├─────────────────────────┼───────────────────────────────────┤
  │ DataNodes report to     │ NodeManagers reconnect to         │
  │ BOTH Active & Standby   │ new Active RM automatically       │
  ├─────────────────────────┼───────────────────────────────────┤
  │ Fencing critical         │ Fencing via ZK ephemeral lock    │

💡 HOW CAN WE SOLVE IT

RM HA INTRODUCES THREE MECHANISMS:

═══════════════════════════════════════════════════════════
MECHANISM 1: ACTIVE/STANDBY WITH ZK LEADER ELECTION
═══════════════════════════════════════════════════════════

Same pattern as NameNode HA (Project 44) but SIMPLER:

→ Both RMs compete for an ephemeral znode in ZooKeeper
→ Winner becomes Active, loser stays Standby
→ No separate ZKFC process needed — RM has EMBEDDED leader election
→ No separate fencing needed — ZK ephemeral lock IS the fence

  ┌──────────────────────────────────────────────────────────┐
  │  NAMENODE HA:                                             │
  │    NameNode + separate ZKFC process + JournalNodes        │
  │    External fencing (sshfence/IPMI)                       │
  │                                                           │
  │  RESOURCEMANAGER HA:                                      │
  │    ResourceManager has BUILT-IN leader election            │
  │    No ZKFC needed                                         │
  │    No JournalNodes needed                                 │
  │    No external fencing needed                             │
  │    ZK ephemeral znode = the only fence                    │
  └──────────────────────────────────────────────────────────┘

WHY SIMPLER:
→ NameNode split-brain = metadata corruption = PERMANENT data loss
→ RM split-brain = duplicate job submissions = annoying but recoverable
→ The STAKES are lower, so the MECHANISMS can be simpler

ELECTION PROCESS:
  T+0s:    Both RMs start and connect to ZooKeeper
  T+0.1s:  RM1 creates ephemeral znode → becomes Active
  T+0.1s:  RM2 fails to create → stays Standby
  T+0.2s:  Active RM1 starts accepting client connections
  
  ...Active RM1 crashes...
  
  T+Xs:    ZK session for RM1 expires → ephemeral znode deleted
  T+Xs+1:  RM2's watcher fires → RM2 creates znode → becomes Active
  T+Xs+2:  RM2 loads state from ZK state store
  T+Xs+30: RM2 fully operational (state loaded, NMs reconnecting)

═══════════════════════════════════════════════════════════
MECHANISM 2: STATE STORE (Application State Persistence)
═══════════════════════════════════════════════════════════

PROBLEM:
→ When Active RM crashes, all in-memory state about running jobs is LOST
→ New Active RM needs to know: what was running? what was queued?
→ Without state persistence: ALL jobs must be resubmitted

SOLUTION — ZooKeeper State Store:
→ Active RM writes critical state to ZooKeeper CONTINUOUSLY
→ On failover, new Active reads this state and rebuilds in-memory state

WHAT IS STORED IN ZK:

  /rmstore/
  ├── ZKRMStateRoot/
  │   ├── RMAppRoot/
  │   │   ├── application_1705334400_0001    ← app submission info
  │   │   ├── application_1705334400_0002
  │   │   ├── application_1705334400_0003
  │   │   └── ...
  │   ├── RMDTSecretManagerRoot/             ← delegation tokens
  │   │   ├── RMDTSequentialNumber
  │   │   ├── RMDTMasterKey_1
  │   │   └── ...
  │   ├── AMRMTokenSecretManagerRoot/        ← AM-RM auth tokens
  │   └── EpochNode                          ← RM epoch (monotonic counter)

WHAT IS STORED PER APPLICATION:
  {
    "applicationId": "application_1705334400_0001",
    "submitTime": 1705334500000,
    "applicationSubmissionContext": {
      "applicationName": "daily_etl_pipeline",
      "queue": "production",
      "resource": {"memory": 4096, "vcores": 2},
      "amContainerSpec": { ... },
      "maxAppAttempts": 2
    },
    "currentAttempt": 1,
    "state": "RUNNING",
    "diagnostics": ""
  }

WRITE FREQUENCY:
→ State written on: application submit, state change, attempt change
→ NOT written on: every heartbeat or container allocation
→ Container-level state is recovered from NodeManagers (see Mechanism 3)

═══════════════════════════════════════════════════════════
MECHANISM 3: WORK-PRESERVING RM RESTART
═══════════════════════════════════════════════════════════

THE MOST IMPORTANT FEATURE — THIS IS WHAT MAKES RM HA TRULY USEFUL

WITHOUT Work-Preserving Restart:
→ RM crashes → new RM starts → all apps resubmitted from scratch
→ Running containers are KILLED
→ Completed map tasks must RE-RUN
→ Hours of computation WASTED

WITH Work-Preserving Restart:
→ RM crashes → new RM starts
→ NodeManagers keep containers RUNNING (they don't know RM crashed)
→ ApplicationMasters keep managing their jobs
→ New RM reconnects to NodeManagers and AMs
→ Running containers SURVIVE the failover
→ ZERO computation wasted

HOW IT WORKS:

  BEFORE CRASH (Normal Operation):
  ┌─────────────┐     heartbeat      ┌──────────────┐
  │ Active RM   │ ←─────────────── │ NodeManager 1 │
  │             │                    │ Container A   │
  │             │                    │ Container B   │
  │             │ ←─────────────── │ NodeManager 2 │
  │             │                    │ Container C   │
  └─────────────┘                    └──────────────┘
  
  DURING CRASH:
  ┌─────────────┐                    ┌──────────────┐
  │ Active RM   │  ← DEAD            │ NodeManager 1 │
  │   (crashed) │                    │ Container A ← still running!
  │             │                    │ Container B ← still running!
  └─────────────┘                    │              │
                                     │ "RM heartbeat │
                                     │  timed out... │
                                     │  I'll keep my │
                                     │  containers   │
                                     │  alive and    │
                                     │  wait for     │
                                     │  reconnect"   │
                                     └──────────────┘
  
  AFTER FAILOVER:
  ┌─────────────┐     re-register    ┌──────────────┐
  │ NEW Active  │ ←─────────────── │ NodeManager 1 │
  │ RM          │                    │ Container A ← still running!
  │             │  "Here are my      │ Container B ← still running!
  │ Rebuilds    │   running          │              │
  │ state from  │   containers"      │              │
  │ ZK + NM     │                    │              │
  │ reports     │ ←─────────────── │ NodeManager 2 │
  │             │                    │ Container C ← still running!
  └─────────────┘                    └──────────────┘
  
  STATE RECONSTRUCTION:
  1. New Active RM reads application state from ZK state store
  2. NodeManagers re-register and report ALL running containers
  3. ApplicationMasters re-register with new Active RM
  4. RM matches ZK state with NM reports → rebuilds complete picture
  5. Scheduling resumes — no jobs lost, no containers killed

═══════════════════════════════════════════════════════════
MECHANISM 4: NODEMANAGER RECOVERY
═══════════════════════════════════════════════════════════

PROBLEM:
→ If NodeManager itself crashes and restarts, it forgets its containers
→ All containers on that NM die, even though the processes may still run

SOLUTION — NM Recovery:
→ NodeManager writes container state to LOCAL disk
→ On restart, NM reads state and re-adopts running containers
→ Combined with RM HA: even NM restarts don't kill running work

NM RECOVERY STATE (written to local disk):
  /opt/hadoop/nm-recovery/
  ├── container_1705334400_0001_01_000001/
  │   ├── launch_context.json
  │   ├── resource_allocation.json
  │   └── process_pid
  ├── container_1705334400_0001_01_000002/
  └── ...

ON NM RESTART:
  1. NM reads recovery directory
  2. Checks if container processes are still alive (using saved PIDs)
  3. If alive: re-adopts container, resumes heartbeating to RM
  4. If dead: marks container as LOST, reports to RM

✅ WHAT SHOULD DO

1. REUSE EXISTING ZOOKEEPER ENSEMBLE
   → Project 44 already deployed ZK for NameNode HA
   → RM HA can use the SAME ZK ensemble
   → RM state store uses different znodes (/rmstore/) — no conflict
   → Leader election uses different znode path — no conflict

2. ENABLE WORK-PRESERVING RM RESTART FROM DAY ONE
   → yarn.resourcemanager.work-preserving-recovery.enabled = true
   → This is the ENTIRE POINT of RM HA
   → Without it, failover still kills all running containers
   → With it, containers survive seamlessly

3. ENABLE NODEMANAGER RECOVERY TOO
   → yarn.nodemanager.recovery.enabled = true
   → Protects against NM crashes, not just RM crashes
   → Containers survive NM restart if their processes are still alive
   → Storage: yarn.nodemanager.recovery.dir = /opt/hadoop/nm-recovery

4. CONFIGURE AM MAX RETRY HIGHER THAN DEFAULT
   → yarn.resourcemanager.am.max-attempts = 4 (default: 2)
   → During RM failover, AM might fail to heartbeat → AM attempt fails
   → Higher max-attempts gives AM more chances to reconnect
   → Each AM attempt reconnects to new Active RM

5. SET APPROPRIATE RM TIMEOUT FOR NM WAIT
   → yarn.resourcemanager.nm.liveness-monitor.interval-ms = 10000
   → After failover, NMs need time to re-register
   → If RM declares NMs dead too quickly, containers get killed
   → yarn.nm.liveness-monitor.expiry-interval-ms = 600000 (10 min)
   → Gives NMs ample time to reconnect after RM failover

6. TEST FAILOVER REGULARLY
   → Schedule monthly RM failover drills
   → Verify: running jobs survive, containers continue
   → Measure: failover time, job impact, queue recovery

🚫 WHAT SHOULD NOT DO

1. DO NOT CO-LOCATE ACTIVE RM WITH ACTIVE NAMENODE
   → Both are masters — both need dedicated resources
   → Single machine failure takes down BOTH storage and compute masters
   → Even with HA: you'd failover both simultaneously — added complexity

2. DO NOT USE FILESYSTEM STATE STORE WITH HDFS
   → Circular dependency: RM needs HDFS, HDFS needs RM (for replication)
   → If HDFS is down, RM cannot load state
   → Use ZooKeeper state store — independent of HDFS

3. DO NOT SET ZK SESSION TIMEOUT TOO LOW FOR RM
   → RM has heavier GC than ZKFC (larger heap, more state)
   → If RM GC pause > ZK session timeout → false failover
   → Recommended: yarn.resourcemanager.zk-timeout-ms = 30000 (30 sec)
   → Higher than NN ZKFC timeout because RM heap is typically larger

4. DO NOT FORGET TO SET RM IDS CONSISTENTLY
   → yarn.resourcemanager.ha.rm-ids = rm1,rm2
   → Must be identical on ALL nodes in the cluster
   → Mismatch → NMs connect to wrong RM → split-brain behavior

5. DO NOT DISABLE WORK-PRESERVING RECOVERY
   → Without it, RM HA is just "faster restart" — not true HA
   → All containers die on failover
   → This defeats the purpose of the entire HA setup
   → ALWAYS enable: yarn.resourcemanager.work-preserving-recovery.enabled=true

6. DO NOT IGNORE RM STATE STORE CLEANUP
   → ZK state store grows with every application submission
   → Old application state must be cleaned up
   → yarn.resourcemanager.state-store.max-completed-applications = 10000
   → If not limited, ZK storage bloats → ZK performance degrades

🚫 WHEN NOT TO USE

HARD RULES — RM HA IS WRONG WHEN:

1. SINGLE-NODE PSEUDO-DISTRIBUTED MODE
   → Need 2 machines minimum for RM HA
   → Plus ZooKeeper ensemble (reuse from NN HA)
   → If only 1 machine: RM HA is impossible

2. NO ZOOKEEPER AVAILABLE AND CANNOT DEPLOY ONE
   → RM HA requires ZooKeeper for leader election + state store
   → If you skipped NN HA (Project 44), you have no ZK
   → Deploy ZK first, then enable both NN HA and RM HA

3. ALL JOBS ARE IDEMPOTENT AND SHORT
   → If every job takes < 2 minutes
   → And resubmission is automated (Oozie retry)
   → RM failure + manual restart + resubmit = 10 minutes total
   → RM HA overhead may not be justified (but still recommended)

4. CLUSTER IS BEING DECOMMISSIONED
   → If migrating to Spark standalone or cloud-native
   → Investing in RM HA for a 3-month-remaining cluster = waste

═══════════════════════════════════════════════════════════
INTERNAL MECHANIC 1: EMBEDDED LEADER ELECTION
═══════════════════════════════════════════════════════════

Unlike NameNode HA (which needs external ZKFC process):
→ ResourceManager has BUILT-IN ActiveStandbyElector
→ No separate ZKFC daemon needed
→ RM itself manages ZK session and election

  RM1 starts:
    Step 1: Connect to ZooKeeper ensemble
    Step 2: Try to create ephemeral znode:
            /yarn-leader-election/rm-cluster/ActiveBreadCrumb
    Step 3: SUCCESS → RM1 becomes Active
    Step 4: Write Active RM address to ZK for client discovery

  RM2 starts:
    Step 1: Connect to ZooKeeper ensemble
    Step 2: Try to create same ephemeral znode
    Step 3: FAIL (already exists, held by RM1)
    Step 4: Set WATCHER on the znode
    Step 5: RM2 stays in Standby mode (does NOT accept client connections)

  RM1 crashes:
    Step 1: RM1's ZK session expires
    Step 2: Ephemeral znode is deleted by ZK
    Step 3: RM2's watcher fires: "znode deleted!"
    Step 4: RM2 creates ephemeral znode → becomes Active
    Step 5: RM2 loads state from ZK state store
    Step 6: RM2 starts accepting client connections

  NO FENCING NEEDED — WHY:
  → ZK ephemeral znode guarantees only ONE holder at a time
  → If old RM1 somehow recovers, it discovers it lost the lock
  → RM1 transitions itself to Standby
  → Worst case: brief period of TWO RMs thinking they're Active
  → Impact: possible duplicate container allocations (harmless — deduped by NMs)
  → Compare to NN split-brain: metadata corruption (CATASTROPHIC)

INTERNAL MECHANIC 2: ZK STATE STORE WRITE PROTOCOL
═══════════════════════════════════════════════════════════

Active RM writes state to ZK at specific lifecycle events:

  EVENT: Application Submitted
    RM writes to ZK:
      /rmstore/ZKRMStateRoot/RMAppRoot/application_001
      Data: {submitContext, queue, resources, maxAttempts}
    
  EVENT: Application Attempt Started
    RM writes to ZK:
      /rmstore/ZKRMStateRoot/RMAppRoot/application_001/attempt_01
      Data: {attemptId, amContainerId, startTime}
    
  EVENT: Application State Changed (RUNNING → FINISHED)
    RM updates ZK:
      /rmstore/ZKRMStateRoot/RMAppRoot/application_001
      Data: {state: FINISHED, finishTime, diagnostics}
    
  EVENT: Delegation Token Stored/Renewed
    RM writes to ZK:
      /rmstore/ZKRMStateRoot/RMDTSecretManagerRoot/token_001
      Data: {token, owner, renewer, expiry}

  NOT WRITTEN TO ZK:
    → Individual container allocations (too frequent, too many)
    → NodeManager heartbeats (recovered from NM re-registration)
    → Scheduler queue utilization (recomputed from live state)
    → Container resource usage metrics (transient)

  WHY SELECTIVE WRITES:
    → ZK is not designed for high-throughput writes
    → 1000 container allocations/second → would overwhelm ZK
    → Only write what is needed to RECONSTRUCT state
    → Container state is recovered from NodeManagers

INTERNAL MECHANIC 3: WORK-PRESERVING FAILOVER SEQUENCE
═══════════════════════════════════════════════════════════

Complete timeline of RM failover with work-preserving recovery:

  T+0.000s: Active RM on Machine A crashes
  
  T+0.000s: NodeManagers detect RM heartbeat missing
             BUT: NMs have a WAIT period before killing containers
             yarn.resourcemanager.connect.max-wait.ms = 900000 (15 min)
             → NMs will wait UP TO 15 minutes for RM to come back
             → During this wait: containers keep running
  
  T+0.000s: ApplicationMasters detect RM unavailable
             AMs also have retry logic:
             yarn.resourcemanager.am-rm-client.max-wait-ms = 900000
             → AMs keep managing their tasks
             → AMs cannot request NEW containers (no RM to ask)
             → But EXISTING containers continue processing
  
  T+30.0s:  ZK session for RM1 expires
             (yarn.resourcemanager.zk-timeout-ms = 30000)
             → Ephemeral znode deleted
  
  T+30.1s:  RM2's watcher fires
             RM2 begins transition to Active
  
  T+30.2s:  RM2 READS STATE FROM ZK:
             Step 1: Load all application records from /rmstore/
             Step 2: Load delegation tokens
             Step 3: Load RM epoch (increment by 1)
             Step 4: Rebuild scheduler state from application records
             → This takes 5-30 seconds depending on number of applications
  
  T+60.0s:  RM2 is now Active and accepting connections
  
  T+60.1s:  RM2 waits for NodeManagers to re-register
             
  T+63.0s:  NodeManager 1 detects new Active RM
             NM1 sends RE-REGISTRATION to RM2:
             {
               "nodeId": "nm1-host:45454",
               "totalResources": {"memory": 65536, "vcores": 16},
               "runningContainers": [
                 {"containerId": "container_001_01_000001", 
                  "resource": {"memory": 4096, "vcores": 2},
                  "state": "RUNNING"},
                 {"containerId": "container_001_01_000002",
                  "resource": {"memory": 2048, "vcores": 1},
                  "state": "RUNNING"}
               ]
             }
  
  T+65.0s:  RM2 receives NM1's re-registration
             RM2 matches containers with ZK application records:
             → container_001_01_000001 belongs to application_001
             → application_001 was RUNNING per ZK state
             → MATCH: container is valid → keep it alive
  
  T+70.0s:  ApplicationMaster for app_001 re-registers with RM2:
             "I am the AM for application_001, these are my tasks"
             RM2: "Acknowledged — you may continue scheduling"
  
  T+90.0s:  All NMs re-registered, all AMs re-registered
             RM2 has COMPLETE picture:
             → Application state from ZK
             → Container state from NM re-registration
             → AM state from AM re-registration
             → Scheduler rebuilds queue allocations
  
  T+90.0s:  RM2 resumes normal operation
             → New jobs can be submitted
             → Running jobs continue uninterrupted
             → No containers were killed
             → No computation was wasted

  TOTAL DISRUPTION: ~60-90 seconds
  COMPUTATION LOST: ZERO (containers survived)

INTERNAL MECHANIC 4: NODEMANAGER RECONNECTION PROTOCOL
═══════════════════════════════════════════════════════════

When NM loses connection to Active RM:

  Phase 1: DETECTION (0-10 seconds)
    → NM's heartbeat to RM fails
    → NM enters "RM connection retry" mode
    → NM reads yarn.resourcemanager.ha.rm-ids from config
    → NM discovers both RM addresses

  Phase 2: DISCOVERY (10-30 seconds)
    → NM tries to connect to rm1 address
    → If rm1 is Standby or dead → try rm2
    → If rm2 is Active → connect
    → NM uses ConfiguredRMFailoverProxyProvider (similar to HDFS client)

  Phase 3: RE-REGISTRATION (30-60 seconds)
    → NM sends NodeManager.registerNodeManager() to new Active RM
    → Includes: node resources + list of ALL running containers
    → New RM acknowledges and adds NM to its live node list
    → Containers that NM reports but RM doesn't know about:
      → RM asks NM to KILL those containers (orphaned from previous RM epoch)
    → Containers that RM expects but NM doesn't report:
      → RM marks those containers as LOST

  IMPORTANT — RM EPOCH:
  → Each new Active RM increments a monotonic epoch counter
  → Containers from previous epochs can be identified
  → Prevents stale containers from previous RM incarnation
  → Epoch stored in ZK: /rmstore/ZKRMStateRoot/EpochNode

INTERNAL MECHANIC 6: NODEMANAGER RECOVERY (LOCAL)
═══════════════════════════════════════════════════════════

NM Recovery is SEPARATE from RM HA but complementary:

  DURING NORMAL OPERATION:
    NM writes to local recovery directory:
    /opt/hadoop/nm-recovery/
    ├── nmstore.db                    ← LevelDB database
    │   Contains:
    │   → container_001: {launchContext, resources, pid: 12345}
    │   → container_002: {launchContext, resources, pid: 12346}
    │   → application_001: {appCredentials, acls}
    └── ...

  WHEN NM CRASHES AND RESTARTS:
    Step 1: NM reads nmstore.db
    Step 2: For each container in DB:
            → Check if process with saved PID is still alive
            → If alive: re-adopt container, resume management
            → If dead: mark as COMPLETED(FAILED)
    Step 3: Re-register with Active RM
    Step 4: Report re-adopted containers as RUNNING
    Step 5: Resume normal heartbeating

  WITHOUT NM RECOVERY:
    → NM restarts → forgets all containers
    → All container processes are orphaned (eventually killed by OS)
    → RM thinks containers are lost → fails tasks → restarts them
    → Computation wasted on that node

  WITH NM RECOVERY:
    → NM restarts → re-adopts living containers
    → No computation wasted
    → Combined with RM HA: cluster survives ANY single-process failure

⚠️ COMMON PITFALLS AND FAILURE MODES

PITFALL 1: STATE STORE CORRUPTED IN ZOOKEEPER
→ SYMPTOM: New Active RM fails to start with "InvalidStateStoreException"
→ CAUSE: ZK state store data corrupted (ZK crash during write, network issue)
→ FIX:
   - Delete corrupt state: zkCli.sh rmr /rmstore
   - Restart RM (it recreates state store)
   - IMPACT: All running applications are LOST (must resubmit)
   - PREVENTION: Monitor ZK health (same ZK ensemble as NN HA)

PITFALL 2: NODEMANAGERS DON'T RECONNECT AFTER FAILOVER
→ SYMPTOM: New Active RM shows 0 live NodeManagers
→ CAUSE: NM configuration doesn't list both RM addresses
→ FIX: Verify on ALL NMs:
   yarn.resourcemanager.ha.rm-ids = rm1,rm2
   yarn.resourcemanager.hostname.rm1 = rm1-host
   yarn.resourcemanager.hostname.rm2 = rm2-host
→ FIX: Check firewall allows NM → rm2-host:8031 (scheduler port)

PITFALL 3: WORK-PRESERVING RECOVERY ENABLED BUT CONTAINERS STILL DIE
→ SYMPTOM: After failover, all containers killed despite config
→ CAUSE 1: yarn.resourcemanager.work-preserving-recovery.enabled = false
   (check ALL nodes, not just RM)
→ CAUSE 2: NM wait timeout too short
   yarn.resourcemanager.connect.max-wait.ms = 10000 (10s — too short!)
   Set to: 900000 (15 minutes)
→ CAUSE 3: NM declares RM dead before new RM is ready
   yarn.resourcemanager.connect.retry-interval.ms = 30000
   Set to: 10000 (retry every 10 seconds)

PITFALL 4: RM FAILOVER TAKES 5+ MINUTES INSTEAD OF 60 SECONDS
→ SYMPTOM: Long delay between old RM crash and new RM becoming Active
→ CAUSE: ZK session timeout too high
→ FIX: yarn.resourcemanager.zk-timeout-ms = 30000 (30 seconds)
   Not higher unless GC pauses warrant it
→ ALSO CHECK: ZK ensemble health (if ZK is slow, session timeout is slow)

PITFALL 5: ZK STATE STORE GROWS UNBOUNDED
→ SYMPTOM: ZooKeeper running out of memory/disk
→ CAUSE: Completed application records never cleaned up
→ FIX: yarn.resourcemanager.state-store.max-completed-applications = 10000
→ Default is unlimited — ALWAYS set a limit
→ Monitor: /rmstore/ znode count in ZK

PITFALL 6: BOTH RMs STUCK IN STANDBY
→ SYMPTOM: No Active RM — all job submissions fail
→ CAUSE: ZK ensemble lost quorum — neither RM can win election
→ FIX: Restore ZK quorum first (same as Project 44 ZK recovery)
→ Then RM will auto-elect
→ TEMPORARY: Manual transition
   yarn rmadmin -transitionToActive rm1 --forcemanual
   (ONLY if you are CERTAIN the other RM is not Active)

PITFALL 7: SPLIT-BRAIN WITH DUPLICATE JOB SCHEDULING
→ SYMPTOM: Same job running twice on different sets of containers
→ CAUSE: Brief period where both RMs think they are Active
→ IMPACT: Duplicate output files, double resource usage
→ SELF-HEALING: RM epoch mechanism eventually kills stale containers
→ PREVENTION: Set ZK session timeout appropriately
→ COMPARE TO NN SPLIT-BRAIN: RM split-brain is annoying but not catastrophic

📈 SCALABILITY CONSIDERATIONS

RM HA OVERHEAD:

1. ZOOKEEPER WRITE LOAD
   → Each application submission = 1 ZK write (~1-5KB)
   → Each state change = 1 ZK update
   → 1000 jobs/hour = ~3000 ZK writes/hour = trivial for ZK
   → ZK can handle 10,000+ writes/second
   → RM HA adds negligible load to existing ZK ensemble

2. FAILOVER TIME vs CLUSTER SIZE
   ┌────────────────┬───────────────────┬────────────────────────┐
   │ Cluster Size   │ State Store Load  │ Expected Failover Time │
   ├────────────────┼───────────────────┼────────────────────────┤
   │ 100 NMs        │ ~500 apps in ZK   │ 30-45 seconds          │
   │ 500 NMs        │ ~2000 apps in ZK  │ 45-60 seconds          │
   │ 2000 NMs       │ ~5000 apps in ZK  │ 60-90 seconds          │
   │ 5000 NMs       │ ~10000 apps in ZK │ 90-120 seconds         │
   └────────────────┴───────────────────┴────────────────────────┘
   
   Bottleneck is NOT state store read (fast) but NM re-registration:
   → 5000 NMs all reconnecting simultaneously = thundering herd
   → New RM must process 5000 re-registrations with container lists
   → Stagger: NMs have random backoff before re-registration

3. STATE STORE SIZE IN ZK
   → Each application: ~2-10KB in ZK
   → 10,000 applications: ~100MB in ZK
   → ZK heap recommendation: 1-4GB for combined NN HA + RM HA
   → Set max-completed-applications to limit growth

4. NODEMANAGER RECOVERY DISK OVERHEAD
   → LevelDB database: ~1MB per 100 containers
   → Typical NM with 50 containers: ~500KB
   → Negligible disk impact
   → BUT: use local SSD for recovery directory (write latency matters)

    if [ "$rm1_state" = "active" ]; then

rm_failover() {
    echo "═══ PLANNED RM FAILOVER ═══"
    echo ""
    
    rm1_state=$(yarn rmadmin -getServiceState rm1 2>/dev/null || echo "UNREACHABLE")
    rm2_state=$(yarn rmadmin -getServiceState rm2 2>/dev/null || echo "UNREACHABLE")
    
    if [ "$rm1_state" = "active" ]; then
        FROM="rm1"; TO="rm2"
    elif [ "$rm2_state" = "active" ]; then
        FROM="rm2"; TO="rm1"
    else
        echo "  ❌ Cannot determine Active RM. Aborting."
        exit 1
    fi
    
    echo "  Current Active: $FROM"
    echo "  Failover to:    $TO"
    echo ""
    
    # Verify Standby is alive
    to_state=$(yarn rmadmin -getServiceState $TO 2>/dev/null)
    if [ "$to_state" != "standby" ]; then
        echo "  ❌ Target $TO is not in standby state (state=$to_state). Aborting."
        exit 1
    fi
    
    read -p "  Proceed with failover $FROM → $TO? (yes/no): " confirm
    if [ "$confirm" != "yes" ]; then
        echo "  Cancelled."
        exit 0
    fi
    
    echo ""
    echo "  Executing failover..."
    
    # For RM, graceful failover is: transition old Active to Standby,
    # then ZK election promotes the other automatically
    yarn rmadmin -transitionToStandby $FROM --forcemanual 2>&1 | sed 's/^/    /'
    
    sleep 10
    
    new_to_state=$(yarn rmadmin -getServiceState $TO 2>/dev/null)
    new_from_state=$(yarn rmadmin -getServiceState $FROM 2>/dev/null)
    
    echo ""
    echo "  Post-failover:"
    echo "    $FROM: $new_from_state"
    echo "    $TO:   $new_to_state"
    
    if [ "$new_to_state" = "active" ]; then
        echo ""
        echo "  ✅ Failover SUCCESSFUL"
    else
        echo ""
        echo "  ⚠️  $TO did not become Active automatically"
        echo "     Forcing transition..."
        yarn rmadmin -transitionToActive $TO --forcemanual 2>&1 | sed 's/^/    /'
    fi
}
