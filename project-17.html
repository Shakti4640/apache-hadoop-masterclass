PROJECT 17: JOINING TWO MASSIVE DATASETS — ORDERS MEETS PRODUCTS

🔴 HYPOTHETICAL PROBLEM

SCENARIO:
→ You are a data engineer at a global e-commerce marketplace
→ The quarterly business intelligence team needs a combined report:

DATASET 1 — Orders (LARGE: 120GB, 800 million records)
  ORDER_ID|CUSTOMER_ID|PRODUCT_ID|QUANTITY|UNIT_PRICE|ORDER_DATE
  ORD-00000001|C00542|P10234|2|49.99|2024-01-15
  ORD-00000002|C01287|P08912|1|129.50|2024-01-15
  ORD-00000003|C00542|P04567|3|19.99|2024-01-16

DATASET 2 — Product Reviews (LARGE: 80GB, 500 million records)
  REVIEW_ID|PRODUCT_ID|REVIEWER_ID|RATING|REVIEW_TEXT|REVIEW_DATE
  REV-00000001|P10234|C01100|5|Excellent quality|2024-01-20
  REV-00000002|P10234|C00542|4|Good value|2024-01-22
  REV-00000003|P08912|C03456|2|Disappointing|2024-01-18

REQUIRED OUTPUT:
  Join on PRODUCT_ID — for each order, attach all reviews for that product
  Used for: "Do products with higher review ratings sell more units?"

WHY MAP-SIDE JOIN FAILS:
→ Product Reviews: 80GB → HashMap = ~200GB in memory
→ Mapper heap: typically 1-4GB → IMPOSSIBLE to cache
→ Orders: 120GB → also impossible to cache
→ NEITHER dataset fits in memory
→ Map-Side Join (Project 16) CANNOT work here

WHY WE NEED REDUCE-SIDE JOIN:
→ The ONLY strategy that works for two large datasets
→ Both datasets processed by mappers
→ Mappers TAG each record with its source (orders vs reviews)
→ Records SHUFFLED to reducers by join key (PRODUCT_ID)
→ Reducer receives ALL records for same PRODUCT_ID
→ Reducer MATCHES orders with reviews

THE COST:
→ 120GB + 80GB = 200GB shuffled over network
→ This is EXPENSIVE — but it's the ONLY option
→ Understanding this cost is why Project 16 matters:
   "Use Map-Side Join whenever possible to AVOID this shuffle"

STEP 2: Understand TAGGING — the critical technique

WHY TAGGING?
→ Reducer receives values for ONE key (PRODUCT_ID)
→ But values come from TWO different datasets
→ Reducer must know: "Is this value an order or a review?"
→ SOLUTION: Prepend a TAG to each value

  Order mapper emits:   key="P10234", value="O|ORD-001|C00542|2|49.99|..."
  Review mapper emits:  key="P10234", value="R|REV-001|C01100|5|Excellent|..."
                                              ↑
                                              TAG prefix

  Reducer reads first character:
    'O' → this is an order record → add to orders list
    'R' → this is a review record → add to reviews list

STEP 3: Understand MultipleInputs

We have TWO datasets with DIFFERENT formats:
  → Orders: 6 fields, pipe-delimited
  → Reviews: 6 fields, pipe-delimited (but different columns)

Each dataset needs its OWN mapper:
  → OrderMapper: parses orders, tags with "O"
  → ReviewMapper: parses reviews, tags with "R"

MultipleInputs allows ONE job to read MULTIPLE input paths
with DIFFERENT mapper classes:

  MultipleInputs.addInputPath(job, ordersPath, TextInputFormat.class, OrderMapper.class);
  MultipleInputs.addInputPath(job, reviewsPath, TextInputFormat.class, ReviewMapper.class);

This replaces:
  job.setMapperClass()           ← single mapper
  FileInputFormat.addInputPath() ← single input

With:
  Multiple input paths, each with its own mapper class

SOLUTION: Tagged Reduce-Side Join with MultipleInputs

JOIN STRATEGY IN REDUCER:

Option A: Full cross-product (each order × each review)
  → 100 orders × 500 reviews = 50,000 output records per product
  → Data explosion — usually NOT what you want
  → Used for: detailed order-review pairing

Option B: Aggregate reviews, attach summary to each order (OUR CHOICE)
  → Calculate: avg_rating, review_count per product
  → Attach summary to each order for that product
  → 100 orders → 100 output records (with review stats)
  → Used for: analytics ("do higher-rated products sell more?")

Option C: Filter join (only orders for products with rating > 4)
  → Calculate avg rating from reviews
  → Only emit orders where avg_rating > threshold
  → Used for: quality-filtered reporting

✅ WHAT COULD BE DONE

APPROACH A: MultipleInputs + Tagged Values + Single Reducer (OUR CHOICE)
→ Two mapper classes via MultipleInputs
→ Tag values with source identifier
→ Reducer separates, aggregates, and joins
→ Most flexible — supports all join types

APPROACH B: Single Mapper + Source Detection
→ One mapper that detects record format
→ If line has ORDER_ID pattern → tag as order
→ If line has REVIEW_ID pattern → tag as review
→ Simpler setup but fragile (format detection can fail)

APPROACH C: Two separate jobs + merge
→ Job 1: Aggregate reviews per product → product_stats
→ Job 2: Map-Side Join orders with product_stats (Project 16)
→ product_stats is small → fits in cache
→ Two jobs but second is fast (map-side)

APPROACH D: Hive JOIN (Project 24)
→ CREATE TABLE orders (...); CREATE TABLE reviews (...);
→ SELECT * FROM orders o JOIN reviews r ON o.product_id = r.product_id;
→ Hive auto-generates MapReduce with optimal join strategy
→ Easier but less control

→ Demonstrates the fundamental Reduce-Side Join pattern

✅ WHAT SHOULD DO

1. USE MultipleInputs FOR CLEAN SEPARATION
   → Each dataset gets its own mapper class
   → No format detection logic needed
   → Type-safe parsing in each mapper
   → Clear code organization

2. CHOOSE A TAG FORMAT THAT'S UNAMBIGUOUS
   → Use a character that NEVER appears in data
   → "O|rest_of_record" — tag is single char before first pipe
   → Or use a control character: "\tO\t" + record
   → AVOID using same delimiter as data fields
   → If data has pipes, use tab or special char as tag separator

3. HANDLE THE "SMALL SIDE" MEMORY PROBLEM IN REDUCER
   → Reducer iterates values ONCE — cannot re-iterate
   → If doing cross-join: must buffer one side in memory
   → For orders × reviews: buffer the SMALLER side
   → If reviews per product < 1000: buffer reviews, stream orders
   → If orders per product < 100: buffer orders, stream reviews
   → ALWAYS buffer the side with fewer records per key

4. USE COUNTERS TO TRACK JOIN STATISTICS
   → COUNT_ORDERS_PROCESSED
   → COUNT_REVIEWS_PROCESSED
   → COUNT_PRODUCTS_WITH_BOTH (successful joins)
   → COUNT_PRODUCTS_ORDERS_ONLY (orders without reviews)
   → COUNT_PRODUCTS_REVIEWS_ONLY (reviews without orders)
   → These map to join type semantics

5. SET APPROPRIATE NUMBER OF REDUCERS
   → Total shuffle data: 200GB
   → Each reducer should handle: 1-2GB
   → 200GB / 1.5GB = ~130 reducers
   → Too few: reducer OOM or very slow
   → Too many: overhead exceeds benefit
   → RULE: numReducers = totalShuffleData / (1-2GB)

6. TEST WITH KNOWN SMALL DATA FIRST
   → Create 10 orders, 10 reviews for 5 products
   → Manually compute expected join output
   → Run job → compare → verify before scaling

🚫 WHAT SHOULD NOT DO

1. DO NOT FORGET TO BUFFER THE SMALLER SIDE
   → Reducer iterates values stream ONCE (Iterable, not Collection)
   → If you iterate once for orders, reviews are GONE
   → Must collect one side into ArrayList during iteration
   → Then iterate the other side, cross-referencing the list
   → Buffer the side with FEWER records per key

2. DO NOT BUFFER THE LARGER SIDE
   → Product P10234 has 50 orders but 50,000 reviews
   → Buffering 50,000 reviews = potential OOM
   → Buffer 50 orders (small), stream 50,000 reviews
   → ALWAYS profile your data to know which side is smaller per key

3. DO NOT USE MultipleInputs AND setMapperClass() TOGETHER
   → MultipleInputs.addInputPath() replaces setMapperClass()
   → If both set: undefined behavior (mapper may be overridden)
   → Use EITHER MultipleInputs OR setMapperClass(), not both

4. DO NOT ASSUME TAG ORDER IN REDUCER
   → Values arrive in UNDEFINED order within a reduce() call
   → First value might be order OR review — you don't know
   → Must check tag on EVERY value
   → DO NOT assume "all orders come before all reviews"
   → (Secondary Sort in Project 19 CAN guarantee order)

5. DO NOT IGNORE PRODUCTS WITH NO MATCHES
   → Product has orders but zero reviews → LEFT JOIN keeps it
   → Product has reviews but zero orders → RIGHT JOIN keeps it
   → INNER JOIN: skip products with only one side
   → Track these via counters for data quality monitoring

6. DO NOT EMIT INSIDE THE BUFFERING LOOP
   → WRONG: iterate values, buffer orders, emit as you go
   → Reviews haven't been seen yet when first orders arrive
   → Must complete FULL iteration before emitting
   → Two-pass logic: Pass 1 = buffer + separate, Pass 2 = emit

🚫 WHEN NOT TO USE

REDUCE-SIDE JOIN IS WRONG WHEN:

1. ONE DATASET IS SMALL ENOUGH TO CACHE
   → 120GB orders + 60MB customers → Map-Side Join (Project 16)
   → Reduce-Side would shuffle 120GB unnecessarily
   → ALWAYS check if Map-Side is possible first

2. JOIN KEY HAS EXTREME SKEW
   → Product "P00001" has 100 million orders + 5 million reviews
   → All 105 million records go to ONE reducer
   → That reducer takes hours while others finish in minutes
   → SOLUTION: Salted keys (Project 33) or pre-aggregation

3. CROSS-JOIN IS NEEDED (no join key)
   → "Every order paired with every promotion"
   → No common key → shuffle doesn't help
   → Use nested loop in mapper with distributed cache instead

4. REAL-TIME JOIN REQUIREMENT
   → Reduce-Side Join is batch — takes minutes to hours
   → For sub-second joins: use HBase or database

═══════════════════════════════════════════════════════════
MECHANIC 2: VALUE ITERATION — THE SINGLE-PASS PROBLEM
═══════════════════════════════════════════════════════════

Reducer receives: reduce(Text key, Iterable<Text> values, Context ctx)

The Iterable<Text> is NOT a collection — it's a STREAM:
  → Values are read from disk one at a time
  → Once you iterate past a value, it's GONE
  → You CANNOT iterate twice
  → You CANNOT call values.size()
  → You CANNOT access values by index

  WHY? Memory efficiency:
  → Product P00001 might have 10 million values
  → Storing all in memory = OOM
  → Streaming: only ONE value in memory at a time

  CONSEQUENCE FOR JOINS:
  → Must buffer ONE side during iteration
  → Stream the OTHER side in the same pass

  ┌──────────────────────────────────────────────────────────┐
  │  REDUCER ITERATION FOR JOIN:                              │
  │                                                           │
  │  List<String> orders = new ArrayList<>();                 │
  │  List<String> reviews = new ArrayList<>();                │
  │                                                           │
  │  for (Text value : values) {                              │
  │      String record = value.toString();                    │
  │      if (record.startsWith("O|")) {                       │
  │          orders.add(record.substring(2));  // Buffer       │
  │      } else if (record.startsWith("R|")) {                │
  │          reviews.add(record.substring(2)); // Buffer       │
  │      }                                                    │
  │  }                                                        │
  │                                                           │
  │  // NOW both lists are populated — can cross-reference    │
  │  // AFTER iteration completes                             │
  │                                                           │
  │  ⚠️ CAUTION: Both lists in memory simultaneously         │
  │  If product has 1M orders + 1M reviews = 2M strings      │
  │  → Potentially GBs of memory per reducer call             │
  │  → SOLUTION: Buffer smaller side only (see Mechanic 3)    │
  └──────────────────────────────────────────────────────────┘

═══════════════════════════════════════════════════════════
MECHANIC 3: OPTIMIZED BUFFERING STRATEGY
═══════════════════════════════════════════════════════════

PROBLEM: For popular products, both sides may be large
  Product P00001: 500,000 orders + 200,000 reviews

STRATEGY: Buffer SMALLER side, aggregate, then stream LARGER side

  Pass 1 (during iteration):
    → Check tag of each value
    → If review: accumulate into running aggregates
      (sum of ratings, count of reviews)
    → If order: buffer in list (or emit immediately if reviews
      were guaranteed to come first — but they're NOT)

  BETTER STRATEGY WITH TWO PASSES:
    Cannot re-iterate values stream...
    BUT: can use Secondary Sort (Project 19) to guarantee
    reviews come BEFORE orders in the value stream

    With Secondary Sort:
    → Sort key: (productId, tag) where tag "R" < "O"
    → Reviews arrive first → aggregate in memory (small)
    → Orders arrive second → emit enriched immediately
    → Memory: only review aggregates, NOT full review list

    Without Secondary Sort (this project):
    → Must buffer BOTH sides → memory risk for hot products
    → Mitigate: aggregate reviews on-the-fly (don't store full text)
    → Buffer orders (usually fewer per product than reviews)

═══════════════════════════════════════════════════════════
MECHANIC 4: TEXT OBJECT REUSE WARNING
═══════════════════════════════════════════════════════════

CRITICAL BUG SOURCE:

  for (Text value : values) {
      orders.add(value);  // ← BUG! Stores reference to SAME object
  }

  Hadoop REUSES the Text object in the iterator!
  → Every call to next() overwrites the SAME Text instance
  → After loop: all entries in orders point to LAST value

  FIX: Copy the string, not the reference:
  
  for (Text value : values) {
      orders.add(value.toString());  // ← Creates new String each time
  }

  This is the #1 reducer bug in Hadoop development.
  Every experienced Hadoop developer has hit this exactly once.

⚠️ COMMON PITFALLS AND FAILURE MODES

PITFALL 1: "All values in reducer list are identical (last record)"
→ CAUSE: Stored Text reference instead of String copy
→ Hadoop reuses Text object in iterator
→ FIX: orders.add(value.toString()) not orders.add(value)
→ This is THE most common Hadoop reducer bug

PITFALL 2: "Reducer OOM on popular products"
→ CAUSE: Product with millions of records buffered in memory
→ 1M orders × 100 bytes = 100MB per reducer call
→ Multiple hot products = GBs of memory
→ FIX: Aggregate reviews (don't store full text)
→ FIX: Use Secondary Sort (Project 19) to stream one side

PITFALL 3: "MultipleInputs mapper class not found"
→ CAUSE: Inner class not properly included in JAR
→ MultipleInputs stores class name as string in configuration
→ Remote node deserializes class name → ClassNotFoundException
→ FIX: Use static inner classes or top-level classes
→ FIX: Verify JAR contains all .class files

PITFALL 4: "Join produces zero results despite matching keys"
→ CAUSE: Key format mismatch between mappers
→ OrderMapper emits key="P10234" (no space)
→ ReviewMapper emits key="P10234 " (trailing space)
→ Shuffle groups by exact byte match → never grouped together
→ FIX: Trim keys in BOTH mappers

PITFALL 5: "INNER JOIN drops everything — no matches"
→ CAUSE: Tag is included in value but also corrupts parsing
→ Value: "O|ORD-001|P10234|..." 
→ Reducer splits by "|" → first field is "O" not "ORD-001"
→ FIX: Use tag separator different from data delimiter
→ OR: Use substring(2) to skip "O|" prefix consistently

PITFALL 6: "Job runs but output has wrong field count"
→ CAUSE: Order record has 6 fields, review has 6 fields
→ After join: expected 12 fields but some have 11 or 13
→ CAUSE: Delimiter character appears in review text
→ "Excellent, great quality" contains comma (if comma-delimited)
→ FIX: Use pipe delimiter, validate field counts in mapper

🔍 INTERPRETABILITY CONSIDERATIONS

HOW TO VERIFY REDUCE-SIDE JOIN:

1. PER-SOURCE RECORD COUNTERS
   → ORDERS_PROCESSED: 800,000,000
   → REVIEWS_PROCESSED: 500,000,000
   → PRODUCTS_WITH_ORDERS_AND_REVIEWS: 45,000
   → PRODUCTS_WITH_ORDERS_ONLY: 5,000
   → PRODUCTS_WITH_REVIEWS_ONLY: 2,000
   → JOIN_OUTPUT_RECORDS: depends on join type

2. VERIFY TAG SEPARATION IS CORRECT
   → In reducer: count records per tag
   → If 99% are tagged "O" and 1% "R" → possible tagging bug
   → Expected: roughly proportional to input sizes

3. SPOT-CHECK SPECIFIC PRODUCT
   → Pick product P10234
   → Count orders in input: grep P10234 orders.txt | wc -l
   → Count reviews in input: grep P10234 reviews.txt | wc -l
   → Verify output matches expected join result

4. OUTPUT SIZE SANITY CHECK
   → INNER JOIN: output ≤ min(orders_count, reviews_count)
   → LEFT JOIN on orders: output = orders_count
   → If output >> input → possible cross-join explosion

public class ReduceSideJoinJob extends Configured implements Tool {

    // ── Tag constants ──
    // These prefixes identify which dataset a record came from
    // Using characters that won't appear in actual data
    public static final String ORDER_TAG = "O~";
    public static final String REVIEW_TAG = "R~";

    // Join type configuration
    public static final String JOIN_TYPE_KEY = "join.type";
    public static final String JOIN_INNER = "INNER";
    public static final String JOIN_LEFT = "LEFT";
    public static final String JOIN_FULL_OUTER = "FULL_OUTER";


    // ════════════════════════════════════════════════════════
    // MAPPER 1: OrderMapper — reads order records
    // ════════════════════════════════════════════════════════
    //
    // Input: ORDER_ID|CUSTOMER_ID|PRODUCT_ID|QUANTITY|UNIT_PRICE|ORDER_DATE
    // Output: key=PRODUCT_ID, value="O~ORDER_ID|CUSTOMER_ID|QUANTITY|UNIT_PRICE|ORDER_DATE"
    //
    // Join key = PRODUCT_ID (the field we're joining on)
    // Tag = "O~" prepended to value (identifies as order)

    public static class OrderMapper
            extends Mapper<LongWritable, Text, Text, Text> {

        private Text joinKey = new Text();
        private Text taggedValue = new Text();

        enum OrderCounters {
            TOTAL_ORDER_LINES,
            VALID_ORDERS,
            INVALID_ORDERS,
            HEADER_SKIPPED
        }

        @Override
        protected void map(LongWritable offset, Text line, Context context)
                throws IOException, InterruptedException {

            context.getCounter(OrderCounters.TOTAL_ORDER_LINES).increment(1);

            String record = line.toString().trim();
            if (record.isEmpty()) return;

            if (record.startsWith("ORDER_ID")) {
                context.getCounter(OrderCounters.HEADER_SKIPPED).increment(1);
                return;
            }

            // Parse: ORDER_ID|CUSTOMER_ID|PRODUCT_ID|QUANTITY|UNIT_PRICE|ORDER_DATE
            String[] fields = record.split("\\|");
            if (fields.length != 6) {
                context.getCounter(OrderCounters.INVALID_ORDERS).increment(1);
                return;
            }

            String orderId   = fields[0].trim();
            String custId    = fields[1].trim();
            String productId = fields[2].trim();  // JOIN KEY
            String quantity  = fields[3].trim();
            String unitPrice = fields[4].trim();
            String orderDate = fields[5].trim();

            // ── Emit with join key and tagged value ──
            // Key = product ID (what we're joining on)
            // Value = TAG + order fields (without product ID — it's in the key)
            joinKey.set(productId);
            taggedValue.set(ORDER_TAG + orderId + "|" + custId + "|"
                + quantity + "|" + unitPrice + "|" + orderDate);

            context.write(joinKey, taggedValue);
            context.getCounter(OrderCounters.VALID_ORDERS).increment(1);
        }
    }


    // ════════════════════════════════════════════════════════
    // MAPPER 2: ReviewMapper — reads review records
    // ════════════════════════════════════════════════════════
    //
    // Input: REVIEW_ID|PRODUCT_ID|REVIEWER_ID|RATING|REVIEW_TEXT|REVIEW_DATE
    // Output: key=PRODUCT_ID, value="R~REVIEW_ID|REVIEWER_ID|RATING|REVIEW_TEXT|REVIEW_DATE"

    public static class ReviewMapper
            extends Mapper<LongWritable, Text, Text, Text> {

        private Text joinKey = new Text();
        private Text taggedValue = new Text();

        enum ReviewCounters {
            TOTAL_REVIEW_LINES,
            VALID_REVIEWS,
            INVALID_REVIEWS,
            HEADER_SKIPPED
        }

        @Override
        protected void map(LongWritable offset, Text line, Context context)
                throws IOException, InterruptedException {

            context.getCounter(ReviewCounters.TOTAL_REVIEW_LINES).increment(1);

            String record = line.toString().trim();
            if (record.isEmpty()) return;

            if (record.startsWith("REVIEW_ID")) {
                context.getCounter(ReviewCounters.HEADER_SKIPPED).increment(1);
                return;
            }

            // Parse: REVIEW_ID|PRODUCT_ID|REVIEWER_ID|RATING|REVIEW_TEXT|REVIEW_DATE
            String[] fields = record.split("\\|");
            if (fields.length != 6) {
                context.getCounter(ReviewCounters.INVALID_REVIEWS).increment(1);
                return;
            }

            String reviewId   = fields[0].trim();
            String productId  = fields[1].trim();  // JOIN KEY
            String reviewerId = fields[2].trim();
            String rating     = fields[3].trim();
            String reviewText = fields[4].trim();
            String reviewDate = fields[5].trim();

            // ── Emit with join key and tagged value ──
            joinKey.set(productId);
            taggedValue.set(REVIEW_TAG + reviewId + "|" + reviewerId + "|"
                + rating + "|" + reviewText + "|" + reviewDate);

            context.write(joinKey, taggedValue);
            context.getCounter(ReviewCounters.VALID_REVIEWS).increment(1);
        }
    }


    // ════════════════════════════════════════════════════════
    // REDUCER: Join orders with review aggregates
    // ════════════════════════════════════════════════════════
    //
    // For each PRODUCT_ID:
    //   1. Separate incoming values by tag (O~ vs R~)
    //   2. Aggregate reviews: avg rating, review count
    //   3. Attach review stats to each order
    //   4. Emit enriched orders
    //
    // Strategy: Aggregate reviews on-the-fly (constant memory)
    //           Buffer orders (usually fewer per product)
    //           After iteration: emit each order with review stats

    public static class JoinReducer
            extends Reducer<Text, Text, Text, Text> {

        private Text outputKey = new Text();
        private Text outputValue = new Text();
        private String joinType;

        enum JoinCounters {
            PRODUCTS_PROCESSED,
            PRODUCTS_WITH_BOTH,
            PRODUCTS_ORDERS_ONLY,
            PRODUCTS_REVIEWS_ONLY,
            ORDERS_JOINED,
            ORDERS_WITHOUT_REVIEWS,
            REVIEWS_AGGREGATED,
            OUTPUT_RECORDS
        }

        @Override
        protected void setup(Context context) {
            joinType = context.getConfiguration().get(JOIN_TYPE_KEY, JOIN_LEFT);
        }

        @Override
        protected void reduce(Text productId, Iterable<Text> values, Context context)
                throws IOException, InterruptedException {

            context.getCounter(JoinCounters.PRODUCTS_PROCESSED).increment(1);

            // ── PASS 1: Separate and aggregate ──
            // Buffer orders (typically fewer per product)
            // Aggregate reviews on-the-fly (memory efficient)
            List<String> orderRecords = new ArrayList<>();
            double ratingSum = 0.0;
            int reviewCount = 0;
            int highRatings = 0;  // ratings >= 4

            for (Text value : values) {
                // ══════════════════════════════════════════
                // CRITICAL: Use toString() to COPY the value
                // ══════════════════════════════════════════
                // Hadoop REUSES the Text object in the iterator
                // If we stored 'value' directly, all entries
                // would point to the LAST value after the loop
                String record = value.toString();

                if (record.startsWith(ORDER_TAG)) {
                    // Strip tag prefix and buffer
                    String orderData = record.substring(ORDER_TAG.length());
                    orderRecords.add(orderData);
                } else if (record.startsWith(REVIEW_TAG)) {
                    // Strip tag and aggregate review statistics
                    String reviewData = record.substring(REVIEW_TAG.length());
                    // reviewData: REVIEW_ID|REVIEWER_ID|RATING|REVIEW_TEXT|REVIEW_DATE
                    String[] reviewFields = reviewData.split("\\|");
                    if (reviewFields.length >= 3) {
                        try {
                            double rating = Double.parseDouble(reviewFields[2].trim());
                            ratingSum += rating;
                            reviewCount++;
                            if (rating >= 4.0) highRatings++;
                        } catch (NumberFormatException e) {
                            // Skip bad rating — don't crash
                        }
                    }
                    context.getCounter(JoinCounters.REVIEWS_AGGREGATED).increment(1);
                }
            }

            // ── Calculate review statistics ──
            double avgRating = reviewCount > 0 ? ratingSum / reviewCount : 0.0;
            double highRatingPct = reviewCount > 0
                ? (highRatings * 100.0 / reviewCount) : 0.0;

            // ── Track join categories ──
            boolean hasOrders = !orderRecords.isEmpty();
            boolean hasReviews = reviewCount > 0;

            if (hasOrders && hasReviews) {
                context.getCounter(JoinCounters.PRODUCTS_WITH_BOTH).increment(1);
            } else if (hasOrders) {
                context.getCounter(JoinCounters.PRODUCTS_ORDERS_ONLY).increment(1);
            } else if (hasReviews) {
                context.getCounter(JoinCounters.PRODUCTS_REVIEWS_ONLY).increment(1);
            }

            // ── PASS 2: Apply join semantics and emit ──
            String pid = productId.toString();
            String reviewStats = String.format("%.1f|%d|%.0f%%",
                avgRating, reviewCount, highRatingPct);

            if (hasOrders && hasReviews) {
                // BOTH sides present — emit for all join types
                for (String orderData : orderRecords) {
                    outputKey.set(pid);
                    outputValue.set(orderData + "|" + reviewStats);
                    context.write(outputKey, outputValue);
                    context.getCounter(JoinCounters.ORDERS_JOINED).increment(1);
                    context.getCounter(JoinCounters.OUTPUT_RECORDS).increment(1);
                }
            } else if (hasOrders && !hasReviews) {
                // Orders only — LEFT JOIN and FULL OUTER emit with "no reviews"
                if (joinType.equals(JOIN_LEFT) || joinType.equals(JOIN_FULL_OUTER)) {
                    for (String orderData : orderRecords) {
                        outputKey.set(pid);
                        outputValue.set(orderData + "|0.0|0|0%");
                        context.write(outputKey, outputValue);
                        context.getCounter(JoinCounters.ORDERS_WITHOUT_REVIEWS).increment(1);
                        context.getCounter(JoinCounters.OUTPUT_RECORDS).increment(1);
                    }
                }
                // INNER JOIN: skip (no output for unmatched orders)
            } else if (!hasOrders && hasReviews) {
                // Reviews only — FULL OUTER emits review summary
                if (joinType.equals(JOIN_FULL_OUTER)) {
                    outputKey.set(pid);
                    outputValue.set("NO_ORDERS|" + reviewStats);
                    context.write(outputKey, outputValue);
                    context.getCounter(JoinCounters.OUTPUT_RECORDS).increment(1);
                }
                // INNER and LEFT: skip (no orders to join)
            }
        }
    }


    // ════════════════════════════════════════════════════════
    // DRIVER: Configure MultipleInputs and launch job
    // ════════════════════════════════════════════════════════

    @Override
    public int run(String[] args) throws Exception {
        if (args.length < 4) {
            System.err.println("Usage: ReduceSideJoinJob <orders_input> <reviews_input> <output> <join_type>");
            System.err.println("  join_type: INNER | LEFT | FULL_OUTER");
            return 1;
        }

        String ordersPath  = args[0];
        String reviewsPath = args[1];
        String outputPath  = args[2];
        String joinType    = args[3].toUpperCase();

        Configuration conf = getConf();
        conf.set(JOIN_TYPE_KEY, joinType);

        Job job = Job.getInstance(conf,
            "Reduce-Side Join (" + joinType + "): Orders × Reviews");
        job.setJarByClass(ReduceSideJoinJob.class);

        // ══════════════════════════════════════════════════
        // MultipleInputs: Different mapper per input path
        // ══════════════════════════════════════════════════
        // This REPLACES both:
        //   job.setMapperClass(...)
        //   FileInputFormat.addInputPath(...)
        // Each input path gets its own mapper class

        MultipleInputs.addInputPath(job,
            new Path(ordersPath),
            TextInputFormat.class,
            OrderMapper.class);

        MultipleInputs.addInputPath(job,
            new Path(reviewsPath),
            TextInputFormat.class,
            ReviewMapper.class);

        // ── Reducer ──
        job.setReducerClass(JoinReducer.class);
        job.setNumReduceTasks(4);  // Adjust based on data size

        // NOTE: No Combiner — cannot combine across different sources
        // Combining "O|order_data" with "R|review_data" makes no sense

        // ── Output types ──
        // Map output types must match for BOTH mappers
        job.setMapOutputKeyClass(Text.class);
        job.setMapOutputValueClass(Text.class);
        job.setOutputKeyClass(Text.class);
        job.setOutputValueClass(Text.class);

        // ── Output path ──
        FileOutputFormat.setOutputPath(job, new Path(outputPath));

        System.out.println("═══ REDUCE-SIDE JOIN CONFIGURATION ═══");
        System.out.println("  Join type:     " + joinType);
        System.out.println("  Orders input:  " + ordersPath + " → OrderMapper");
        System.out.println("  Reviews input: " + reviewsPath + " → ReviewMapper");
        System.out.println("  Output:        " + outputPath);
        System.out.println("  Reducers:      4");
        System.out.println("  Tag format:    O~ (orders), R~ (reviews)");
        System.out.println("════════════════════════════════════════");

        boolean success = job.waitForCompletion(true);

        if (success) {
            printJoinReport(job, joinType);
        }

        return success ? 0 : 1;
    }


    private void printJoinReport(Job job, String joinType) throws IOException {
        System.out.println("\n╔══════════════════════════════════════════════════════════╗");
        System.out.println("║         REDUCE-SIDE JOIN REPORT (" + joinType + ")                   ║");
        System.out.println("╠══════════════════════════════════════════════════════════╣");

        long ordersValid = job.getCounters()
            .findCounter(OrderMapper.OrderCounters.VALID_ORDERS).getValue();
        long reviewsValid = job.getCounters()
            .findCounter(ReviewMapper.ReviewCounters.VALID_REVIEWS).getValue();

        long prodsProcessed = getJoinCounter(job, "PRODUCTS_PROCESSED");
        long prodsBoth = getJoinCounter(job, "PRODUCTS_WITH_BOTH");
        long prodsOrdersOnly = getJoinCounter(job, "PRODUCTS_ORDERS_ONLY");
        long prodsReviewsOnly = getJoinCounter(job, "PRODUCTS_REVIEWS_ONLY");
        long ordersJoined = getJoinCounter(job, "ORDERS_JOINED");
        long ordersNoReviews = getJoinCounter(job, "ORDERS_WITHOUT_REVIEWS");
        long reviewsAgg = getJoinCounter(job, "REVIEWS_AGGREGATED");
        long outputRecords = getJoinCounter(job, "OUTPUT_RECORDS");

        System.out.printf("║  Input orders:            %,d%n", ordersValid);
        System.out.printf("║  Input reviews:           %,d%n", reviewsValid);
        System.out.println("║──────────────────────────────────────────────────────────║");
        System.out.printf("║  Products processed:      %,d%n", prodsProcessed);
        System.out.printf("║    With orders + reviews: %,d%n", prodsBoth);
        System.out.printf("║    Orders only:           %,d%n", prodsOrdersOnly);
        System.out.printf("║    Reviews only:          %,d%n", prodsReviewsOnly);
        System.out.println("║──────────────────────────────────────────────────────────║");
        System.out.printf("║  Orders joined:           %,d%n", ordersJoined);
        System.out.printf("║  Orders without reviews:  %,d%n", ordersNoReviews);
        System.out.printf("║  Reviews aggregated:      %,d%n", reviewsAgg);
        System.out.println("║──────────────────────────────────────────────────────────║");
        System.out.printf("║  OUTPUT RECORDS:          %,d%n", outputRecords);
        System.out.println("╚══════════════════════════════════════════════════════════╝\n");
    }

    private long getJoinCounter(Job job, String name) throws IOException {
        return job.getCounters()
                  .findCounter(JoinReducer.JoinCounters.valueOf(name))
                  .getValue();
    }

    public static void main(String[] args) throws Exception {
        System.exit(ToolRunner.run(new Configuration(), new ReduceSideJoinJob(), args));
    }
}
