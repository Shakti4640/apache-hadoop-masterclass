   │                    KEY HIERARCHY                                  │
   │                                                                  │
   │  ┌─────────────────────────────────────────────────────┐        │
   │  │  LEVEL 1: KMS Master Key                             │        │
   │  │  → Protects all EZ Keys                              │        │
   │  │  → Stored in KMS keystore (Java KeyStore or HSM)     │        │

   │                          │ protects                              │
   │                          ▼                                       │
   │  ┌─────────────────────────────────────────────────────┐        │
   │  │  LEVEL 2: Encryption Zone Keys (EZ Keys)             │        │
   │  │  → One per encryption zone (e.g., /data/warehouse)   │        │
   │  │  → Never leaves KMS                                  │        │
   │  │  → Used to encrypt/decrypt DEKs                      │        │
   │  │  → Can be ROTATED (new version) without re-encrypting│        │
   │  │    existing data                                     │        │
   │  └───────────────────────┬─────────────────────────────┘        │
   │                          │ encrypts                              │

   │  ┌─────────────────────────────────────────────────────┐        │
   │  │  LEVEL 3: Data Encryption Keys (DEKs)                │        │
   │  │  → One per FILE                                      │        │
   │  │  → Generated by KMS when file is created             │        │
   │  │  → Encrypted by EZ Key → becomes EDEK                │        │
   │  │  → EDEK stored in NameNode file metadata             │        │
   │  │  → Plain DEK used by client to encrypt/decrypt data  │        │
   │  │  → Plain DEK NEVER stored — always derived from EDEK │        │
   │  └─────────────────────────────────────────────────────┘        │
   │                                                                  │
   │  EXAMPLE:                                                        │
   │  File: /data/warehouse/fact_orders/dt=2024-01-14/000000_0.orc   │
   │                                                                  │
   │  → Encryption Zone: /data/warehouse (EZ Key: "warehouse_key")   │
   │  → File's EDEK: stored in NameNode metadata                     │
   │  → When client reads file:                                      │
   │    1. Client gets EDEK from NameNode                            │
   │    2. Client sends EDEK to KMS: "decrypt this EDEK"             │
   │    3. KMS decrypts EDEK using EZ Key → returns plain DEK        │
   │    4. Client uses DEK to decrypt data blocks from DataNode      │
   │    5. Client sees plain text data                               │
   │    6. DEK discarded after use (never stored)                    │
   └──────────────────────────────────────────────────────────────────┘

💡 HOW CAN WE SOLVE IT

COMPLETE ENCRYPTION SOLUTION:

COMPONENT 1: Key Management Server (KMS)
→ Dedicated service for encryption key management
→ Kerberos-authenticated access (requires Project 41)
→ ACLs on key operations (who can encrypt/decrypt with which key)
→ Audit logging of all key operations
→ Key rotation support without data re-encryption

COMPONENT 2: HDFS Encryption Zones
→ Directories designated as encryption zones
→ All files in zone automatically encrypted
→ Each zone uses a specific EZ Key from KMS
→ Transparent to applications — read/write API unchanged
→ Different zones can use different keys (data classification)

COMPONENT 3: SSL/TLS for Data in Transit
→ Certificates for all services
→ HTTPS for Web UIs
→ TLS for HDFS data transfer protocol
→ TLS for RPC communication
→ Encrypted MapReduce shuffle

COMPONENT 4: SASL/RPC Wire Encryption
→ hadoop.rpc.protection=privacy (from Project 41 upgrade)
→ Encrypts ALL Hadoop RPC communication
→ Complements SSL/TLS for complete coverage

✅ WHAT SHOULD DO

1. DEPLOY KMS ON A DEDICATED HOST
   → KMS should NOT run on NameNode or DataNode
   → Compromise of KMS = ability to decrypt all data
   → Separate machine, hardened OS, restricted network access
   → Consider KMS HA (two instances) for availability

2. USE HARDWARE SECURITY MODULE (HSM) FOR KMS MASTER KEY
   → Software keystore is acceptable for development
   → Production: HSM (e.g., AWS CloudHSM, Thales Luna)
   → HSM makes master key extraction physically impossible
   → Most stringent compliance requires HSM

3. CREATE SEPARATE EZ KEYS PER DATA CLASSIFICATION
   → Different keys for different sensitivity levels
   → raw_zone_key: highest sensitivity (raw PII)
   → warehouse_key: high sensitivity (processed PII)
   → archive_key: historical data (different rotation)
   → If one key is compromised, only that zone is affected

4. IMPLEMENT KEY ROTATION POLICY
   → Rotate EZ Keys periodically (e.g., every 90 days)
   → Key rotation creates NEW version of the key
   → New files encrypted with new version
   → Old files remain readable with old version
   → NO data re-encryption needed (key versioning handles it)

5. SET KMS ACLs MATCHING HDFS PERMISSIONS
   → etl_service can GENERATE_EEK and DECRYPT_EEK on warehouse_key
   → analyst_bob can DECRYPT_EEK on warehouse_key (read-only)
   → qa_charlie CANNOT access warehouse_key at all
   → KMS ACLs are the SECOND layer of encryption access control

6. GENERATE PROPER SSL CERTIFICATES
   → Use internal CA (Certificate Authority) for the cluster
   → Each node gets its own certificate with correct hostname
   → Certificates signed by the internal CA
   → All nodes trust the internal CA
   → Certificate expiry: 1-2 years with automated renewal

7. ENCRYPT BOTH REST AND TRANSIT TOGETHER
   → Encrypting only at rest: network capture still reveals data
   → Encrypting only in transit: stolen disk still reveals data
   → BOTH are required for complete protection

🚫 WHAT SHOULD NOT DO

1. DO NOT USE SELF-SIGNED CERTIFICATES IN PRODUCTION
   → Self-signed = no chain of trust
   → Man-in-the-middle attacks possible
   → Use internal CA or commercial CA
   → Self-signed acceptable ONLY for development

2. DO NOT STORE KMS MASTER KEY ON THE SAME MACHINE AS DATA
   → If attacker gets DataNode + KMS master key = decryption possible
   → Physical separation between KMS and DataNodes
   → Ideally: KMS master key in HSM

3. DO NOT CREATE ONE ENCRYPTION ZONE FOR THE ENTIRE HDFS
   → Single key compromise = all data exposed
   → Separate zones for different data classifications
   → Enables granular access control per zone

4. DO NOT FORGET TO ENCRYPT THE MAPREDUCE SHUFFLE
   → Map output is written to local disk and transferred over network
   → Contains intermediate results with customer data
   → Enable encrypted shuffle: mapreduce.shuffle.ssl.enabled=true

5. DO NOT DISABLE CERTIFICATE HOSTNAME VERIFICATION
   → Tempting when certificates don't match hostnames
   → Disabling = man-in-the-middle attacks possible
   → FIX the certificates instead of disabling verification

6. DO NOT MOVE FILES BETWEEN ENCRYPTION ZONES WITH hdfs dfs -mv
   → Moving between zones requires re-encryption
   → HDFS does not support cross-zone moves
   → Use distcp with encryption or copy-then-delete pattern

7. DO NOT ENCRYPT EVERYTHING IF PERFORMANCE IS CRITICAL
   → Encryption adds 3-10% CPU overhead
   → Non-sensitive data (public reference tables) may not need encryption
   → Encrypt based on data classification, not blanket policy

8. DO NOT LOSE THE KMS MASTER KEY
   → If master key is lost, ALL encrypted data is PERMANENTLY inaccessible
   → Back up master key to offline secure storage
   → Multiple copies in different physical locations
   → Document recovery procedure and test annually

🚫 WHEN NOT TO USE

1. DO NOT USE TDE FOR NON-SENSITIVE PUBLIC DATA
   → Encryption adds CPU overhead (3-10% per read/write)
   → Public reference data (country codes, product categories)
     does not need encryption
   → Encrypt based on data classification

2. DO NOT USE HDFS ENCRYPTION FOR REAL-TIME LOW-LATENCY WORKLOADS
   → KMS decryption call adds ~5-20ms per file open
   → For HBase with millions of tiny random reads: significant
   → HBase has its own encryption (hbase.crypto.key.algorithm)
   → Use HBase-native encryption instead of HDFS TDE for HBase

3. DO NOT USE SOFTWARE KMS FOR PCI-DSS LEVEL 1 COMPLIANCE
   → PCI-DSS Level 1 requires HSM for key protection
   → Software keystores are vulnerable to memory dumps
   → Invest in HSM for payment card data

4. DO NOT USE SSL/TLS WITH EXPIRED CERTIFICATES
   → Expired certificates cause silent connection failures
   → Or worse: applications configured to skip verification
   → Implement certificate monitoring and auto-renewal

5. DO NOT USE ENCRYPTION AS A SUBSTITUTE FOR ACCESS CONTROL
   → Encryption protects data from unauthorized physical access
   → But anyone with KMS decrypt permission can read data
   → Encryption + ACLs + Kerberos = complete protection
   → Encryption alone is insufficient

MECHANIC 1: FILE WRITE TO ENCRYPTION ZONE — STEP BY STEP
═══════════════════════════════════════════════════════════

SCENARIO: etl_service writes cleaned data to /data/warehouse/

Step 1 — Client creates file
   Client: "Create /data/warehouse/fact_orders/dt=2024-01-14/000000_0.orc"
   Client → NameNode (authenticated via Kerberos, Project 41)

Step 2 — NameNode detects encryption zone
   NameNode checks: Is /data/warehouse an encryption zone?
   → YES, encryption zone with key name "warehouse_key"

Step 3 — NameNode requests EDEK from KMS
   NameNode → KMS: "Generate encrypted data encryption key
                     using EZ Key 'warehouse_key'"
   
   KMS internally:
   a. Generate random DEK (256-bit AES key)
   b. Encrypt DEK with warehouse_key → EDEK
   c. Return EDEK to NameNode
   d. Log: "GENERATE_EEK warehouse_key by nn/hadoop-master@COMPANY.COM"

Step 4 — NameNode stores EDEK in file metadata
   File metadata now contains:
   {
     path: "/data/warehouse/fact_orders/dt=2024-01-14/000000_0.orc",
     edek: <encrypted bytes>,
     ez_key_version: 3,
     cipher_suite: "AES/CTR/NoPadding",
     ...normal metadata (blocks, replication, permissions)
   }

Step 5 — NameNode returns EDEK to client
   NameNode → Client: "Here's the EDEK for your new file"

Step 6 — Client decrypts EDEK to get DEK
   Client → KMS: "Decrypt this EDEK using warehouse_key"
   
   KMS checks:
   a. Is client authenticated? (Kerberos ticket verified)
   b. Does client have DECRYPT_EEK permission on warehouse_key?
      → etl_service has DECRYPT_EEK → ALLOWED
   c. Decrypt EDEK → return plain DEK to client
   d. Log: "DECRYPT_EEK warehouse_key by etl_service@COMPANY.COM"

Step 7 — Client encrypts data and writes to DataNode
   Client: encrypt(plain_data, DEK) → encrypted_data
   Client → DataNode: write encrypted_data
   
   DataNode receives ONLY encrypted bytes
   DataNode stores encrypted bytes to local disk
   DataNode has NO access to DEK — cannot decrypt

   ┌────────────┐  encrypted   ┌────────────┐  encrypted  ┌──────┐
   │   Client   │  data        │  DataNode  │  bytes      │ Disk │
   │ (has DEK)  │────────────→ │ (no DEK)   │───────────→ │      │
   │            │              │            │             │      │
   │ plain →    │              │ stores as  │             │ blk_ │
   │ encrypt →  │              │ received   │             │ 1073 │
   │ send       │              │            │             │      │
   └────────────┘              └────────────┘             └──────┘

Step 8 — Client discards DEK
   DEK is held only in memory during the write operation
   Once file is closed → DEK is garbage collected
   DEK is NEVER stored on disk or in HDFS

MECHANIC 3: KEY ROTATION WITHOUT RE-ENCRYPTION
═══════════════════════════════════════════════════════════

SCENARIO: Security policy requires 90-day key rotation

Day 0: warehouse_key version 3 is active
   → All existing files have EDEKs encrypted with version 3

Day 90: Administrator rotates the key
   $ hadoop key roll warehouse_key

   KMS creates warehouse_key VERSION 4
   → Version 3 still exists (not deleted)
   → Version 4 becomes the "current" version

After rotation:
   NEW files:
   → NameNode requests EDEK from KMS
   → KMS generates DEK, encrypts with warehouse_key version 4
   → New EDEK stored in new file's metadata

   EXISTING files:
   → Their EDEKs were encrypted with version 3
   → When read, client sends EDEK to KMS
   → KMS sees version 3 → uses version 3 to decrypt → returns DEK
   → File is still readable WITHOUT re-encryption

   ┌────────────────────────────────────────────────────────┐
   │ File created before rotation:                          │
   │   EDEK encrypted with version 3                        │
   │   → KMS uses version 3 to decrypt → works ✅           │
   │                                                        │
   │ File created after rotation:                           │
   │   EDEK encrypted with version 4                        │
   │   → KMS uses version 4 to decrypt → works ✅           │
   │                                                        │
   │ KEY INSIGHT: No data re-encryption needed              │
   │ Old keys are kept for reading old files                │
   │ Only new files use the new key version                 │
   └────────────────────────────────────────────────────────┘

   OPTIONAL: Re-encrypt existing files (for strongest security)
   $ hadoop crypto -reencryptZone -path /data/warehouse
   → Reads each file's EDEK
   → Decrypts with old version → re-encrypts with new version
   → Updates EDEK in metadata
   → Data on disk is NOT touched (only metadata changes)

⚠️ COMMON PITFALLS AND FAILURE MODES

PITFALL 1: "Cannot create file in encryption zone — KMS unreachable"
→ CAUSE: KMS server is down or network partition
→ IMPACT: Cannot write ANY new files to encryption zones
→ IMPACT: CAN still read existing files (if EDEK cached by client)
→ FIX: KMS high availability (two KMS instances)
→ FIX: KMS load balancer with health checks

PITFALL 2: "Performance dropped 30% after enabling encryption"
→ CAUSE: KMS is a bottleneck — every file open requires KMS call
→ CAUSE: CPU-intensive encryption on client side
→ FIX: Enable EDEK caching on NameNode (dfs.namenode.list.encryption.zones.num.responses)
→ FIX: KMS EDEK cache (hadoop.kms.cache.enable=true)
→ FIX: Use AES-NI hardware acceleration (modern CPUs have this)
→ FIX: Increase KMS thread pool size

PITFALL 3: "Cannot move files between encryption zones"
→ CAUSE: HDFS does not support cross-zone rename/move
→ CAUSE: Different zones use different keys → move requires re-encryption
→ FIX: Use distcp to copy between zones (handles re-encryption)
→ FIX: Design pipeline to avoid cross-zone moves
→ FIX: Our pipeline design (Project 40) uses separate zones per stage

PITFALL 4: "Encryption zone creation fails — directory not empty"
→ CAUSE: Can only create encryption zone on EMPTY directory
→ FIX: Create encryption zone BEFORE loading any data
→ FIX: If data exists: create new encrypted dir → distcp → swap

PITFALL 5: "SSL handshake failure — certificate hostname mismatch"
→ CAUSE: Certificate CN/SAN doesn't match actual hostname
→ CAUSE: Using IP address instead of hostname
→ FIX: Generate certificates with correct hostnames AND IPs in SAN
→ FIX: Ensure DNS resolution is consistent across cluster

PITFALL 6: "KMS master key lost — all data inaccessible"
→ CAUSE: KMS keystore file corrupted or deleted
→ CAUSE: No backup of master key
→ IMPACT: PERMANENT DATA LOSS — no recovery possible
→ PREVENTION: Multiple offline backups of KMS keystore
→ PREVENTION: HSM stores key in tamper-proof hardware
→ PREVENTION: Test key recovery procedure annually

PITFALL 7: "MapReduce shuffle data unencrypted despite TLS"
→ CAUSE: Shuffle encryption requires separate configuration
→ CAUSE: mapreduce.shuffle.ssl.enabled not set to true
→ FIX: Enable shuffle SSL in mapred-site.xml
→ FIX: Configure shuffle SSL keystore/truststore

PITFALL 8: "Hive query fails on encrypted table — KMS permission denied"
→ CAUSE: Hive service account doesn't have DECRYPT_EEK on the EZ key
→ CAUSE: KMS ACLs not updated for Hive
→ FIX: Add hive user to KMS key ACL for DECRYPT_EEK operation

#!/bin/bash
set -e

echo "═══════════════════════════════════════════════════════"
echo "  CREATING HDFS ENCRYPTION ZONES"
echo "═══════════════════════════════════════════════════════"

KMS_HOST="kms-host.company.com"
KMS_URI="kms://http@${KMS_HOST}:9600/kms"

# ── Authenticate as hdfs superuser ──
kinit -kt /etc/hadoop/keytabs/hdfs.keytab hdfs@COMPANY.COM

# ── Step 1: Verify KMS connectivity from NameNode ──
echo "[1/4] Verifying KMS connectivity..."
hadoop key list -provider ${KMS_URI} | head -10
echo "  ✅ KMS accessible from NameNode"

# ── Step 2: Create encryption zones ──
# NOTE: Encryption zones can ONLY be created on EMPTY directories
# If directories have data, we need to:
#   1. Create new empty directory
#   2. Set as encryption zone
#   3. Copy data from old directory
#   4. Rename directories

echo ""
echo "[2/4] Creating encryption zones..."

# Function to create encryption zone safely
create_ez() {
    local path=$1
    local key=$2
    local description=$3

    echo "  Creating encryption zone: ${path} (key: ${key})"

    # Check if directory exists and is empty
    EXISTS=$(hdfs dfs -test -d ${path} && echo "yes" || echo "no")
    if [ "${EXISTS}" = "yes" ]; then
        COUNT=$(hdfs dfs -count ${path} | awk '{print $2}')
        if [ "${COUNT}" != "0" ]; then
            echo "    ⚠️  Directory not empty — migrating data..."
            # Rename existing directory
            hdfs dfs -mv ${path} ${path}_pre_encryption
            # Create fresh empty directory
            hdfs dfs -mkdir -p ${path}
            # Create encryption zone
            hdfs crypto -createZone -keyName ${key} -path ${path}
            # Copy data into encryption zone (will be encrypted)
            hdfs dfs -cp ${path}_pre_encryption/* ${path}/ 2>/dev/null || true

echo "═══════════════════════════════════════════════════════"

SSL_DIR="/opt/hadoop/ssl"
CA_DIR="${SSL_DIR}/ca"
KEYSTORE_PASS="hadoop-keystore-password"  # In production: use Vault/secrets manager
TRUSTSTORE_PASS="hadoop-truststore-password"
CA_PASS="ca-password-very-secure"

MASTER="hadoop-master"
KMS_HOST="kms-host.company.com"
NODES=("hadoop-node01" "hadoop-node02" "hadoop-node03")
ALL_HOSTS=("${MASTER}" "${KMS_HOST}" "${NODES[@]}")

mkdir -p ${CA_DIR}
mkdir -p ${SSL_DIR}/keystores

# ════════════════════════════════════════
# STEP 1: Create Internal Certificate Authority (CA)
# ════════════════════════════════════════

echo ""
echo "[1/5] Creating Internal Certificate Authority..."

# Generate CA private key
openssl genrsa -aes256 -passout pass:${CA_PASS} -out ${CA_DIR}/ca-key.pem 4096

# Generate CA certificate (self-signed, valid 10 years)
openssl req -new -x509 \
    -key ${CA_DIR}/ca-key.pem \
    -passin pass:${CA_PASS} \
    -out ${CA_DIR}/ca-cert.pem \
    -days 3650 \
    -subj "/C=US/ST=California/O=Company/OU=DataPlatform/CN=HadoopInternalCA"

echo "  ✅ CA certificate created: ${CA_DIR}/ca-cert.pem"

# Create truststore containing CA certificate (used by ALL nodes)
keytool -importcert \
    -alias hadoop-ca \
    -file ${CA_DIR}/ca-cert.pem \
    -keystore ${SSL_DIR}/truststore.jks \
    -storepass ${TRUSTSTORE_PASS} \
    -noprompt

echo "  ✅ Truststore created: ${SSL_DIR}/truststore.jks"

# ════════════════════════════════════════
# STEP 2: Generate per-node certificates
# ════════════════════════════════════════

echo ""
echo "[2/5] Generating per-node certificates..."

for HOST in "${ALL_HOSTS[@]}"; do
    echo "  Generating certificate for ${HOST}..."
    NODE_DIR="${SSL_DIR}/keystores/${HOST}"
    mkdir -p ${NODE_DIR}

    # Generate node private key + keystore
    keytool -genkeypair \
        -alias ${HOST} \
        -keyalg RSA \
        -keysize 2048 \
        -validity 730 \
        -dname "CN=${HOST},OU=DataPlatform,O=Company,L=SanFrancisco,ST=CA,C=US" \
        -keystore ${NODE_DIR}/keystore.jks \
        -storepass ${KEYSTORE_PASS} \
        -keypass ${KEYSTORE_PASS} \
        -ext san=dns:${HOST},dns:localhost,ip:127.0.0.1

    # Generate Certificate Signing Request (CSR)
    keytool -certreq \
        -alias ${HOST} \
        -keystore ${NODE_DIR}/keystore.jks \
        -storepass ${KEYSTORE_PASS} \
        -file ${NODE_DIR}/${HOST}.csr \
        -ext san=dns:${HOST},dns:localhost,ip:127.0.0.1

    # Sign CSR with internal CA
    openssl x509 -req \
        -in ${NODE_DIR}/${HOST}.csr \
        -CA ${CA_DIR}/ca-cert.pem \
        -CAkey ${CA_DIR}/ca-key.pem \
        -passin pass:${CA_PASS} \
        -CAcreateserial \
        -out ${NODE_DIR}/${HOST}-signed.pem \
        -days 730 \
        -extensions v3_req \
        -extfile <(cat << EOF
[v3_req]
subjectAltName = DNS:${HOST}, DNS:localhost, IP:127.0.0.1
EOF
)

    # Import CA certificate into node keystore (trust chain)
    keytool -importcert \
        -alias hadoop-ca \
        -file ${CA_DIR}/ca-cert.pem \
        -keystore ${NODE_DIR}/keystore.jks \
        -storepass ${KEYSTORE_PASS} \
        -noprompt

    # Import signed certificate into node keystore
    keytool -importcert \
        -alias ${HOST} \
        -file ${NODE_DIR}/${HOST}-signed.pem \
        -keystore ${NODE_DIR}/keystore.jks \
        -storepass ${KEYSTORE_PASS} \
        -noprompt

    echo "    ✅ ${HOST}: keystore ready"
done

# ════════════════════════════════════════
# STEP 3: Create SSL credential files
# ════════════════════════════════════════

echo ""
echo "[3/5] Creating SSL credential files..."

# Keystore password file (referenced by Hadoop config)
for HOST in "${ALL_HOSTS[@]}"; do
    NODE_DIR="${SSL_DIR}/keystores/${HOST}"
    echo "${KEYSTORE_PASS}" > ${NODE_DIR}/keystore.password
    chmod 400 ${NODE_DIR}/keystore.password
done

chmod 400 ${SSL_DIR}/truststore.password

import subprocess

class EncryptionAuditor:
    """
    Comprehensive encryption audit for Hadoop cluster.
    Verifies TDE, TLS, RPC encryption, and KMS health.
    """

    def __init__(self, kms_host="kms-host.company.com",
                 namenode_host="hadoop-master"):
        self.kms_host = kms_host
        self.nn_host = namenode_host
        self.findings = []
        self.passed = 0
        self.failed = 0
        self.warnings = 0
        self.timestamp = datetime.datetime.now().strftime("%Y-%m-%d %H:%M:%S")

    def run_cmd(self, cmd, timeout=30):
        """Execute shell command."""
        try:
            result = subprocess.run(
                cmd, shell=True, capture_output=True,
                text=True, timeout=timeout
            )
            return result.stdout.strip(), result.stderr.strip(), result.returncode
        except subprocess.TimeoutExpired:
            return "", "TIMEOUT", 1
        except Exception as e:
            return "", str(e), 1

    def check(self, name, condition, detail=""):
        """Record audit finding."""
        if condition:
            self.passed += 1
            status = "✅ PASS"
        else:
            self.failed += 1
            status = "❌ FAIL"
        self.findings.append({"name": name, "status": status, "detail": detail})

    def warn(self, name, detail=""):
        """Record warning."""
        self.warnings += 1
        self.findings.append({"name": name, "status": "⚠️  WARN", "detail": detail})

    # ═══════════════════════════════════════════
    # ENCRYPTION AT REST CHECKS
    # ═══════════════════════════════════════════

    def check_encryption_zones_exist(self):
        """Verify encryption zones are configured."""
        stdout, _, rc = self.run_cmd("hdfs crypto -listZones 2>/dev/null")
        zones = [l for l in stdout.split("\n") if l.strip() and not l.startswith("/")] if stdout else []
        # Recount properly
        zone_lines = [l for l in stdout.split("\n") if l.strip()] if stdout else []

        expected_zones = ["/data/raw", "/data/cleaned", "/data/warehouse", "/data/archive"]
        for zone in expected_zones:
            found = any(zone in line for line in zone_lines)
            self.check(
                f"Encryption zone: {zone}",
                found,
                f"{'Found' if found else 'MISSING'} in encryption zone list"
            )

    def check_kms_reachable(self):
        """Verify KMS is accessible."""
        stdout, stderr, rc = self.run_cmd(
            f"hadoop key list -provider kms://http@{self.kms_host}:9600/kms 2>/dev/null"
        )
        self.check(
            "KMS reachable",
            rc == 0 and "raw_zone_key" in stdout,
            f"KMS at {self.kms_host}:9600"
        )

    def check_kms_keys_exist(self):
        """Verify all expected encryption keys exist."""
        stdout, _, rc = self.run_cmd(
            f"hadoop key list -provider kms://http@{self.kms_host}:9600/kms 2>/dev/null"
        )
        expected_keys = ["raw_zone_key", "cleaned_zone_key", "warehouse_key", "archive_key"]
        for key in expected_keys:
            self.check(
                f"KMS key exists: {key}",
                key in stdout,
                f"{'Found' if key in stdout else 'MISSING'}"
            )

    def check_file_encryption_info(self):
        """Verify files in encryption zones have encryption metadata."""
        test_zones = {
            "/data/warehouse": "warehouse_key",
            "/data/raw": "raw_zone_key",
        }

        for zone, expected_key in test_zones.items():
            # Find a file in the zone
            stdout, _, rc = self.run_cmd(
                f"hdfs dfs -find {zone} -name '*.orc' 2>/dev/null | head -1"
            )
            if not stdout.strip():
                stdout, _, rc = self.run_cmd(
                    f"hdfs dfs -find {zone} -name '*' -type f 2>/dev/null | head -1"
                )

            if stdout.strip():
                file_path = stdout.strip()
                enc_info, _, _ = self.run_cmd(
                    f"hdfs crypto -getFileEncryptionInfo -path {file_path} 2>/dev/null"
                )
                self.check(
                    f"File encrypted in {zone}",
                    expected_key in enc_info,
                    f"File: {file_path}"
                )
            else:
                self.warn(
                    f"No files found in {zone}",
                    "Cannot verify file-level encryption"
                )

            f"Current: {stdout.strip()}"

    def check_https_connectivity(self):
        """Verify HTTPS endpoints are actually serving."""
        # Create SSL context that accepts self-signed certs for testing
        ctx = ssl.create_default_context()
        ctx.check_hostname = False
        ctx.verify_mode = ssl.CERT_NONE

        endpoints = [
            (f"https://{self.nn_host}:9871/", "NameNode HTTPS"),
            (f"https://{self.nn_host}:8090/", "ResourceManager HTTPS"),
        ]

        for url, name in endpoints:
            try:
                req = urllib.request.Request(url)
                response = urllib.request.urlopen(req, timeout=10, context=ctx)
                self.check(
                    f"{name} endpoint reachable",
                    response.status in (200, 401, 403),
                    f"HTTP {response.status}"
                )
            except urllib.error.HTTPError as e:
                # 401/403 means HTTPS works but auth required — that's OK
                self.check(
                    f"{name} endpoint reachable",
                    e.code in (401, 403),
                    f"HTTP {e.code} (auth required — HTTPS working)"
                )
            except Exception as e:
                self.check(
                    f"{name} endpoint reachable",
                    False,
                    f"Error: {e}"
                )

            stdout, _, rc = self.run_cmd(

            )

            if "versions" in stdout.lower():
                # Extract version count if possible
                self.check(
                    f"Key metadata accessible: {key}",
                    True,
                    "Key rotation status available"
                )
            else:
                self.warn(
                    f"Cannot check key rotation: {key}",
                    "Verify manually with 'hadoop key list -metadata'"
                )

    def generate_report(self):
        """Generate formatted encryption audit report."""
        report = []
        report.append("")
        report.append("═" * 70)
        report.append(f"  HADOOP ENCRYPTION AUDIT REPORT — {self.timestamp}")
        report.append("═" * 70)
        report.append("")
        report.append(f"  Summary: {self.passed} passed, "
                      f"{self.failed} failed, {self.warnings} warnings")

        for status_filter in ["❌ FAIL", "⚠️  WARN", "✅ PASS"]:
            items = [f for f in self.findings if f["status"] == status_filter]
            if items:
                report.append("")
                label = "FAILURES" if "FAIL" in status_filter else \
                    "WARNINGS" if "WARN" in status_filter else "PASSED"
                report.append(f"  {label}:")
                for item in items:
                    report.append(f"    {item['status']} {item['name']}")
                    if item['detail']:
                        report.append(f"         → {item['detail']}")

        report.append("")
        report.append("═" * 70)

        if self.failed > 0:
            report.append("  🔴 ENCRYPTION AUDIT: FAILED")
        elif self.warnings > 0:
            report.append("  🟡 ENCRYPTION AUDIT: PASSED WITH WARNINGS")
        else:
            report.append("  🟢 ENCRYPTION AUDIT: ALL CHECKS PASSED")

        report.append("═" * 70)
        return "\n".join(report)

    def run(self):

        print("")

        # At rest

        self.check_kms_keys_exist()

        self.check_https_namenode()
        self.check_https_yarn()
        self.check_shuffle_encryption()
        self.check_https_connectivity()

        # Certificates
        self.check_certificate_expiry()

        # Key management
        self.check_key_rotation()

        report = self.generate_report()
        print(report)

        return 2 if self.failed > 0 else (1 if self.warnings > 0 else 0)
