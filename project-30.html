PROJECT 30: CHOOSING THE RIGHT FILE FORMAT — TEXT vs SEQUENCEFILE vs AVRO vs PARQUET vs ORC

🔴 HYPOTHETICAL PROBLEM

SCENARIO:
→ You are the lead data engineer at the ride-sharing company
→ The complete data pipeline is operational:
   - Sqoop (Project 28) imports MySQL data as CSV text files
   - Flume (Project 29) streams logs as plain text to HDFS
   - Pig (Project 26) processes and outputs tab-delimited text
   - Hive (Projects 24-25) stores some tables as ORC, some as text
   - HBase (Project 27) handles real-time serving

→ The CTO calls an emergency meeting:

PROBLEM 1: STORAGE COSTS EXPLODING
   → HDFS cluster: 500TB used
   → Growth rate: 50TB/month
   → New hardware purchase: $2 million
   → CTO: "Can we reduce storage by 70% WITHOUT losing data?"

PROBLEM 2: QUERY PERFORMANCE IS TERRIBLE
   → Hive query: "SELECT city, SUM(fare) FROM rides WHERE year=2024"
   → Reads ALL columns (ride_id, rider_id, driver_id, city, country,
     fare, distance, date, payment...) even though only city+fare needed
   → 500GB table scan when only 50GB of columns are relevant
   → Query takes 45 minutes

PROBLEM 3: SCHEMA EVOLUTION NIGHTMARE
   → Marketing wants to add "promo_code" column to rides table
   → 2.8 billion existing rows don't have this column
   → Current format: CSV text — adding column breaks all downstream parsers
   → Must rewrite entire table + update every Pig script + update every Hive DDL
   → Estimated effort: 2 weeks of engineering + 3 days of reprocessing

PROBLEM 4: SMALL FILES FROM FLUME
   → Flume writes plain text files
   → Many files are not splittable when compressed with gzip
   → MapReduce launches one mapper per file (thousands of tiny mappers)
   → Job scheduling overhead dominates actual processing time

ROOT CAUSE — ALL FOUR PROBLEMS:
→ Using PLAIN TEXT as the storage format everywhere
→ Text is the WORST possible format for analytical workloads
→ The right file format solves ALL four problems simultaneously

THIS PROJECT IS THE DEFINITIVE GUIDE TO HADOOP FILE FORMATS.

STEP 2: Understand ROW vs COLUMN orientation

ROW-ORIENTED (Text, SequenceFile, Avro):
   Store: [row1_all_cols] [row2_all_cols] [row3_all_cols] ...
   
   Disk layout for 3 rows × 4 columns:
   ┌──────────────────────────────────────────┐
   │ Alice, NYC, 25.50, card                  │ ← row 1 (all cols)
   │ Bob, CHI, 18.00, cash                    │ ← row 2 (all cols)
   │ Carol, LON, 42.00, card                  │ ← row 3 (all cols)
   └──────────────────────────────────────────┘
   
   GOOD FOR: Reading/writing complete rows
   BAD FOR:  Reading only a few columns (must read ALL columns)

COLUMN-ORIENTED (Parquet, ORC):
   Store: [col1_all_rows] [col2_all_rows] [col3_all_rows] ...
   
   Disk layout for 3 rows × 4 columns:
   ┌──────────────────────────────────────────┐
   │ Alice, Bob, Carol                         │ ← name column
   │ NYC, CHI, LON                             │ ← city column
   │ 25.50, 18.00, 42.00                      │ ← fare column
   │ card, cash, card                          │ ← payment column
   └──────────────────────────────────────────┘
   
   GOOD FOR: Reading specific columns (skip unneeded columns entirely)
   GOOD FOR: Compression (similar values together compress better)
   BAD FOR:  Writing (must buffer all rows before writing columns)

STEP 3: Map our problems to format characteristics

   Storage costs     → Columnar formats compress 3-10× better than text
   Query performance → Column pruning reads only needed columns
   Schema evolution  → Avro/Parquet/ORC embed schema, support evolution
   Splittability     → SequenceFile/Avro/Parquet/ORC are splittable
                       even when compressed

💡 HOW CAN WE SOLVE IT

SOLUTION: Choose the right format for each use case

FORMAT 1 — PLAIN TEXT (CSV/TSV/JSON)
┌─────────────────────────────────────────────────────────┐
│ STRUCTURE:                                               │
│   Just text — one record per line, delimiter-separated   │
│   "1001,Alice,NYC,25.50,card\n"                         │
│                                                          │
│ SCHEMA:   None embedded — must be defined externally     │
│ COMPRESS: Optional (gzip, bzip2, snappy)                 │
│ SPLIT:    Uncompressed=yes, gzip=NO, bzip2=yes          │
│ ENCODING: UTF-8 text                                     │
│ SIZE:     Largest (no compression by default)            │
│ SPEED:    Slow reads (parse text to types every time)    │
│                                                          │
│ WHEN: Quick prototyping, human inspection, data exchange │
│ AVOID: Production analytics, large-scale storage         │
└─────────────────────────────────────────────────────────┘

FORMAT 2 — SEQUENCEFILE
┌─────────────────────────────────────────────────────────┐
│ STRUCTURE:                                               │
│   Binary key-value container                             │
│   Header + [sync_marker + record_block] repeated         │
│   Each record: key (Writable) + value (Writable)        │
│                                                          │
│ SCHEMA:   Key and value class names in header            │
│ COMPRESS: Block or record level (snappy, gzip, lzo)      │
│ SPLIT:    YES (sync markers every ~8KB enable splitting) │
│ ENCODING: Binary (Writable serialization)                │
│ SIZE:     30-50% smaller than text with compression      │
│ SPEED:    Faster than text (binary, no parsing)          │
│                                                          │
│ WHEN: Intermediate MapReduce output, small file merging  │
│ AVOID: Analytics queries, schema evolution needs         │
└─────────────────────────────────────────────────────────┘

FORMAT 3 — APACHE AVRO
┌─────────────────────────────────────────────────────────┐
│ STRUCTURE:                                               │
│   Header (JSON schema) + data blocks (binary)            │
│   Row-oriented — complete records stored together        │
│   Self-describing — schema embedded in every file        │
│                                                          │
│ SCHEMA:   JSON schema EMBEDDED in file header            │
│           Full schema evolution (add/remove/rename cols) │
│ COMPRESS: Block-level (snappy, deflate)                  │
│ SPLIT:    YES (sync markers between blocks)              │
│ ENCODING: Compact binary (variable-length integers)      │
│ SIZE:     50-70% smaller than text                       │
│ SPEED:    Fast serialization/deserialization              │
│                                                          │
│ WHEN: Data exchange, schema evolution, Sqoop output      │
│       Streaming ingestion, Kafka integration             │
│ AVOID: Column-heavy analytics (reads all columns)        │
└─────────────────────────────────────────────────────────┘

FORMAT 4 — APACHE PARQUET
┌─────────────────────────────────────────────────────────┐
│ STRUCTURE:                                               │
│   Row groups → column chunks → pages                     │
│   Column-oriented with row group boundaries              │
│   Footer contains schema + column statistics             │
│                                                          │
│   ┌──────────── Row Group 1 ────────────┐               │
│   │ Col A chunk │ Col B chunk │ Col C... │               │
│   │ [page][page]│ [page][page]│ ...      │               │
│   └─────────────────────────────────────┘               │
│   ┌──────────── Row Group 2 ────────────┐               │
│   │ Col A chunk │ Col B chunk │ Col C... │               │
│   └─────────────────────────────────────┘               │
│   Footer: schema + min/max stats per column per RG       │
│                                                          │
│ SCHEMA:   Embedded in footer (Thrift-encoded)            │
│           Schema evolution supported                     │
│ COMPRESS: Per-column (snappy, gzip, lzo, zstd)          │
│ SPLIT:    YES (row group boundaries)                     │
│ ENCODING: Dictionary, RLE, delta, bit-packing per column │
│ SIZE:     70-90% smaller than text                       │
│ SPEED:    Fastest for column-projection queries          │
│                                                          │
│ WHEN: Analytics, Spark, column-selective queries         │
│       Cross-ecosystem (Spark, Hive, Impala, Presto)     │
│ AVOID: Full-row access patterns, small files             │
└─────────────────────────────────────────────────────────┘

FORMAT 5 — APACHE ORC (Optimized Row Columnar)
┌─────────────────────────────────────────────────────────┐
│ STRUCTURE:                                               │
│   Stripes → column streams → index + data               │
│   Column-oriented with stripe boundaries                 │
│   File footer + stripe footers + postscript              │
│                                                          │
│   ┌──────────── Stripe 1 ──────────────┐               │
│   │ Index streams (min/max/bloom/count) │               │
│   │ Data streams (col A, col B, col C)  │               │
│   │ Stripe footer (column encodings)    │               │
│   └────────────────────────────────────┘               │
│   ┌──────────── Stripe 2 ──────────────┐               │
│   │ ...                                 │               │
│   └────────────────────────────────────┘               │
│   File footer: schema + stripe locations + statistics    │
│   Postscript: compression codec + version               │
│                                                          │
│ SCHEMA:   Embedded in file footer                        │
│           Schema evolution supported                     │
│ COMPRESS: Per-stripe (zlib, snappy, lzo, zstd)          │
│ SPLIT:    YES (stripe boundaries)                        │
│ ENCODING: RLE, dictionary, delta, direct per column      │
│ SIZE:     75-95% smaller than text (best compression)    │
│ SPEED:    Fastest for Hive (predicate pushdown + bloom)  │
│           Vectorized reading (batch of 1024 rows)        │
│                                                          │
│ FEATURES UNIQUE TO ORC:                                  │
│   → Built-in bloom filters per column                    │
│   → ACID transaction support in Hive                     │
│   → Vectorized query execution                           │
│   → Built-in column-level statistics                     │
│   → Type-aware compression (int vs string vs decimal)    │
│                                                          │
│ WHEN: Hive warehouse, heavy analytics, ACID tables       │
│ AVOID: Non-Hive ecosystems (Spark prefers Parquet)       │
└─────────────────────────────────────────────────────────┘

✅ WHAT SHOULD DO

1. USE COLUMNAR FORMATS (PARQUET/ORC) FOR ANALYTICS
   → Analytical queries typically SELECT 3-5 columns from 50+ column tables
   → Columnar format reads ONLY those 3-5 columns
   → 90% I/O reduction for typical analytical queries
   → Compression is dramatically better (similar values grouped)

2. USE AVRO FOR DATA EXCHANGE AND SCHEMA EVOLUTION
   → When schema changes frequently (new columns added monthly)
   → When data is shared between different systems (Kafka, Spark, Hive)
   → When you need the schema embedded with the data
   → Sqoop --as-avrodatafile is excellent for RDBMS imports

3. USE ORC FOR HIVE-CENTRIC WORKLOADS
   → If Hive is your primary query engine → ORC is optimal
   → Bloom filters, predicate pushdown, vectorized reads
   → ACID support for UPDATE/DELETE in Hive (Project 34)
   → Tightest integration with Hive optimizer

4. USE PARQUET FOR CROSS-ECOSYSTEM WORKLOADS
   → If data is queried by Spark, Impala, Presto, AND Hive
   → Parquet has broadest ecosystem support
   → Spark's native format — fastest reads in Spark
   → Project 48 (Spark migration) uses Parquet extensively

5. USE SEQUENCEFILE FOR INTERMEDIATE MAPREDUCE OUTPUT
   → Between chained MapReduce jobs (Project 21)
   → For merging small files into larger containers
   → For binary key-value data
   → Being replaced by Avro in modern pipelines

6. USE TEXT ONLY FOR DEBUGGING AND DATA EXCHANGE
   → Quick inspection: hdfs dfs -cat | head
   → Sending data to external systems that can't read binary
   → NEVER for production analytical storage

7. ALWAYS ENABLE COMPRESSION
   → Snappy: fast compression/decompression, moderate ratio
   → Zlib/Gzip: best compression ratio, slower
   → LZO: splittable when indexed, good balance
   → Zstd: best of both worlds (Hadoop 3.x+)

🚫 WHAT SHOULD NOT DO

1. DO NOT STORE PRODUCTION DATA AS PLAIN TEXT CSV
   → 3-10× more storage than columnar formats
   → No schema embedded — fragile parsing
   → No column pruning — reads everything
   → No predicate pushdown — scans everything
   → Gzip-compressed text is NOT splittable → one mapper per file

2. DO NOT USE PARQUET FOR HIVE-ONLY WORKLOADS
   → ORC has deeper Hive integration (bloom filters, ACID, vectorized)
   → Parquet works in Hive but ORC is optimized FOR Hive
   → Exception: if same data is also read by Spark → Parquet wins

3. DO NOT USE ORC OUTSIDE HIVE ECOSYSTEM
   → Spark can read ORC but Parquet is Spark's native format
   → Presto/Trino prefers Parquet
   → If cross-ecosystem → Parquet
   → If Hive-only → ORC

4. DO NOT USE AVRO FOR ANALYTICAL QUERIES
   → Avro is ROW-oriented — reads all columns always
   → SELECT city, SUM(fare) on Avro reads rider_id, driver_id, etc. too
   → Use Avro for ingestion → convert to Parquet/ORC for analytics

5. DO NOT IGNORE COMPRESSION CODEC SPLITTABILITY
   → Gzip-compressed text: NOT splittable → one mapper per file
   → Snappy-compressed text: NOT splittable
   → BUT: Snappy inside Avro/Parquet/ORC/SequenceFile IS splittable
     (container format has sync markers / row group boundaries)
   → RULE: Use compressed CONTAINER formats, not compressed TEXT

6. DO NOT MIX FORMATS WITHIN A HIVE TABLE
   → All files in a Hive table partition must be same format
   → Mixing ORC and text in same directory → query errors
   → Migrating format? Create new table, INSERT OVERWRITE

7. DO NOT CHOOSE FORMAT BASED ON WRITE SPEED ALONE
   → Text writes fastest (no encoding overhead)
   → Columnar formats write slower (buffering, encoding, statistics)
   → BUT: You write ONCE, read HUNDREDS of times
   → Optimize for READ performance, not write performance

🚫 WHEN NOT TO USE

TEXT — DO NOT USE WHEN:
→ Data exceeds 10GB (storage waste)
→ Queries select subset of columns (no column pruning)
→ Data needs schema enforcement (text has no schema)
→ Compressed data needs parallel processing (gzip not splittable)

SEQUENCEFILE — DO NOT USE WHEN:
→ Data is queried analytically (no column pruning)
→ Schema evolution is needed (no embedded schema)
→ Data is shared outside Hadoop (Java-specific serialization)
→ Modern alternatives exist (Avro is better in most cases)

AVRO — DO NOT USE WHEN:
→ Queries are column-selective analytics (reads all columns)
→ Hive is primary query engine with heavy predicate pushdown
→ Maximum compression is needed (columnar compresses better)

PARQUET — DO NOT USE WHEN:
→ Workload is write-heavy with few reads (write overhead high)
→ Full-row access pattern dominates (row-oriented is faster)
→ Hive-only ecosystem with ACID requirements (ORC better)
→ Files are very small (Parquet overhead per file is significant)

ORC — DO NOT USE WHEN:
→ Data is consumed by Spark as primary engine (Parquet preferred)
→ Data is shared with non-Hadoop systems (Parquet more portable)
→ Simple key-value storage needed (SequenceFile simpler)

⚙️ HOW IT ONLY WORKS — INTERNAL MECHANICS

═══════════════════════════════════════════════════════════
MECHANIC 1: HOW COLUMN PRUNING WORKS (Parquet/ORC)
═══════════════════════════════════════════════════════════

Query: SELECT city, SUM(fare) FROM rides WHERE year=2024

TABLE HAS 10 COLUMNS:
ride_id, rider_id, driver_id, city, country,
fare, distance, duration, payment, year

TEXT FORMAT (row-oriented):
→ Must read ALL 10 columns for EVERY row
→ 500GB table → reads 500GB from disk
→ Only 2 columns needed (city, fare) = 90% wasted I/O

PARQUET FORMAT (column-oriented):
→ File structure:
  Row Group 1 (128MB):
    ride_id chunk:   [page1][page2]...     ← SKIPPED
    rider_id chunk:  [page1][page2]...     ← SKIPPED
    driver_id chunk: [page1][page2]...     ← SKIPPED
    city chunk:      [page1][page2]...     ← READ
    country chunk:   [page1][page2]...     ← SKIPPED
    fare chunk:      [page1][page2]...     ← READ
    distance chunk:  [page1][page2]...     ← SKIPPED
    duration chunk:  [page1][page2]...     ← SKIPPED
    payment chunk:   [page1][page2]...     ← SKIPPED
    year chunk:      [page1][page2]...     ← READ (for filter)

→ Reads ONLY city + fare + year chunks
→ 500GB table → reads ~50GB from disk (3 of 10 columns)
→ 90% I/O ELIMINATED

═══════════════════════════════════════════════════════════
MECHANIC 2: HOW PREDICATE PUSHDOWN WORKS (ORC)
═══════════════════════════════════════════════════════════

Query: SELECT city, SUM(fare) FROM rides WHERE year = 2024

ORC FILE STRUCTURE:
   ┌──────────── Stripe 1 (64MB) ────────────────────────┐
   │ INDEX:                                               │
   │   year column: min=2022, max=2022                    │ ← year≠2024
   │   fare column: min=5.00, max=99.50                   │
   │ DATA: [column streams...]                            │
   └─────────────────────────────────────────────────────┘
   ┌──────────── Stripe 2 (64MB) ────────────────────────┐
   │ INDEX:                                               │
   │   year column: min=2023, max=2023                    │ ← year≠2024
   │   fare column: min=3.50, max=85.00                   │
   │ DATA: [column streams...]                            │
   └─────────────────────────────────────────────────────┘
   ┌──────────── Stripe 3 (64MB) ────────────────────────┐
   │ INDEX:                                               │
   │   year column: min=2024, max=2024                    │ ← MATCH!
   │   fare column: min=4.00, max=120.00                  │
   │ DATA: [column streams...]                            │
   └─────────────────────────────────────────────────────┘
   ┌──────────── Stripe 4 (64MB) ────────────────────────┐
   │ INDEX:                                               │
   │   year column: min=2024, max=2024                    │ ← MATCH!
   │ DATA: [column streams...]                            │
   └─────────────────────────────────────────────────────┘

PREDICATE PUSHDOWN:
→ Hive pushes "year = 2024" filter DOWN to the ORC reader
→ ORC reader checks stripe index BEFORE reading data
→ Stripe 1: year min=2022, max=2022 → 2024 NOT possible → SKIP
→ Stripe 2: year min=2023, max=2023 → 2024 NOT possible → SKIP
→ Stripe 3: year min=2024, max=2024 → 2024 IS possible → READ
→ Stripe 4: year min=2024, max=2024 → 2024 IS possible → READ

RESULT: Only 2 of 4 stripes read = 50% more I/O saved
COMBINED with column pruning: 90% (columns) × 50% (stripes) = 95% saved

WITH BLOOM FILTERS (ORC-specific):
→ Bloom filter per column per stripe
→ "Does value 'New_York' POSSIBLY exist in this stripe?"
→ If bloom says NO → skip entire stripe
→ Even more granular than min/max statistics

═══════════════════════════════════════════════════════════

WRITER SCHEMA (when data was written):
{
  "type": "record",
  "name": "Ride",
  "fields": [
    {"name": "ride_id", "type": "long"},
    {"name": "rider_id", "type": "long"},
    {"name": "fare", "type": "double"},
    {"name": "city", "type": "string"}
  ]
}

READER SCHEMA (what the application expects NOW):
{
  "type": "record",
  "name": "Ride",
  "fields": [
    {"name": "ride_id", "type": "long"},
    {"name": "rider_id", "type": "long"},
    {"name": "fare", "type": "double"},
    {"name": "city", "type": "string"},
    {"name": "promo_code", "type": ["null", "string"], "default": null}
  ]
}

EVOLUTION RULES:
→ NEW FIELD with DEFAULT: Reader sees default for old data (null)
→ REMOVED FIELD: Reader ignores it, writer's data still valid
→ RENAMED FIELD: Use aliases for backward compatibility
→ TYPE PROMOTION: int → long, float → double (safe widening)

NO DATA REWRITE NEEDED:
→ Old files (4 columns) read with new schema (5 columns)
→ promo_code returns null for all old records
→ New files include promo_code with actual values
→ Same Hive/Pig/Spark code works on both old and new files

MECHANIC 4: COMPRESSION INSIDE CONTAINER FORMATS
═══════════════════════════════════════════════════════════

WHY COLUMNAR COMPRESSES BETTER:

TEXT ROW (mixed types together):
   "Alice,NYC,25.50,card,Bob,CHI,18.00,cash,Carol,LON,42.00,card"
   → Mixed strings, numbers, cities → low compression ratio
   → Gzip: ~40% reduction

PARQUET COLUMN (same type together):
   City column:    "NYC,CHI,LON,NYC,NYC,CHI,LON,NYC,CHI..."
   → Dictionary: NYC=0, CHI=1, LON=2
   → Encoded as: 0,1,2,0,0,1,2,0,1... (tiny integers)
   → Then RLE: 0×1, 1×1, 2×1, 0×2, 1×1, 2×1, 0×1, 1×1
   → MASSIVE compression from dictionary + run-length encoding

   Fare column:    "25.50,18.00,42.00,25.50,25.50,18.00..."
   → Delta encoding: 25.50, -7.50, +24.00, -16.50...
   → Smaller numbers → fewer bits → better compression

COMPRESSION RATIOS (typical ride data):

   ┌──────────────────┬──────────────┬──────────────────┐
   │ FORMAT           │ RAW SIZE     │ COMPRESSED SIZE  │
   ├──────────────────┼──────────────┼──────────────────┤
   │ Text (CSV)       │ 100 GB       │ 100 GB (none)    │
   │ Text + gzip      │ 100 GB       │ 30 GB (70% ↓)   │
   │ SequenceFile+snp │ 100 GB       │ 45 GB (55% ↓)   │
   │ Avro + snappy    │ 100 GB       │ 25 GB (75% ↓)   │
   │ Parquet + snappy │ 100 GB       │ 12 GB (88% ↓)   │
   │ ORC + zlib       │ 100 GB       │ 8 GB  (92% ↓)   │
   │ ORC + zstd       │ 100 GB       │ 10 GB (90% ↓)   │
   └──────────────────┴──────────────┴──────────────────┘

═══════════════════════════════════════════════════════════
MECHANIC 5: SPLITTABILITY — WHY IT MATTERS
═══════════════════════════════════════════════════════════

MapReduce needs to split input files for parallel processing
(One mapper per split — Project 4 concept)

TEXT + GZIP:
   ┌──────────────── gzip file ────────────────┐
   │ Compressed stream — no split points       │
   │ Must decompress from START to read ANY    │
   │ part of the file                          │
   └───────────────────────────────────────────┘
   → ONE mapper reads entire file = NO parallelism
   → 10GB gzip file = 1 mapper working alone

PARQUET:
   ┌─── Row Group 1 ──┬─── Row Group 2 ──┬─── Row Group 3 ──┐
   │ (self-contained) │ (self-contained) │ (self-contained) │
   │ Each RG can be   │ Each RG can be   │ Each RG can be   │
   │ read independently│ read independently│ read independently│
   └──────────────────┴──────────────────┴──────────────────┘
   → THREE mappers, each reading one row group = 3× parallelism
   → Compression is WITHIN each row group, not across file

ORC:
   ┌─── Stripe 1 ──┬─── Stripe 2 ──┬─── Stripe 3 ──┐
   │ (independent) │ (independent) │ (independent) │
   └───────────────┴───────────────┴───────────────┘
   → Same principle: each stripe is independently readable

SEQUENCEFILE:
   ┌── sync ──┬── records ──┬── sync ──┬── records ──┬── sync ──┐
   → Sync markers every ~8KB allow splitting
   → Even with block compression, splits align to sync markers

AVRO:
   ┌── sync ──┬── block ──┬── sync ──┬── block ──┬── sync ──┐
   → Same sync marker pattern as SequenceFile
   → Each block independently compressed

⚠️ COMMON PITFALLS AND FAILURE MODES

PITFALL 1: "Converted to Parquet but queries are SLOWER"
→ CAUSE: Very small Parquet files (< 10MB each)
→ Parquet has per-file overhead (footer, metadata, column indexes)
→ 10,000 tiny Parquet files: 10,000× metadata reads
→ FIX: Merge small files into 128MB-1GB Parquet files
→ FIX: Set parquet.block.size = 128MB (row group size)

PITFALL 2: "ORC table is 92% smaller but write takes 5× longer"
→ CAUSE: Columnar write requires buffering all rows before writing columns
→ ORC with zlib compression: highest CPU cost
→ This is EXPECTED — trade write speed for read speed
→ FIX: Use snappy for faster writes (slightly less compression)
→ REMEMBER: Write once, read hundreds of times

PITFALL 3: "Schema evolution broke all my Pig scripts"
→ CAUSE: Added column to Avro schema without default value
→ Old files don't have the column → Avro throws error
→ FIX: ALWAYS add new fields with default values:
   {"name": "new_field", "type": ["null", "string"], "default": null}

PITFALL 4: "Parquet column pruning doesn't seem to work in Hive"
→ CAUSE: Using SELECT * (no column pruning possible)
→ Or: Hive not pushing projection down to Parquet reader
→ VERIFY: EXPLAIN query — look for "columns.types" and "columns"
→ FIX: Explicitly list needed columns in SELECT
→ FIX: SET hive.optimize.ppd = true (predicate pushdown)

PITFALL 5: "Gzip-compressed CSV creates one mapper per file"
→ CAUSE: Gzip is not splittable (see Mechanic 5 above)
→ IMPACT: 100 files × 1 mapper each = 100 serial tasks
→ FIX 1: Use bzip2 (splittable but slow)
→ FIX 2: Convert to Parquet/ORC/Avro (splittable with any codec)
→ FIX 3: Use LZO + index (splittable and fast)
→ BEST: Don't use compressed text — use container formats

PITFALL 6: "ORC bloom filter doesn't help my query"
→ CAUSE: Bloom filter only helps with equality predicates (=, IN)
→ Range queries (BETWEEN, >, <) use min/max stats, not bloom
→ Bloom filters on high-cardinality columns (ride_id) waste space
→ FIX: Set bloom filters on columns used in WHERE = conditions
→ BEST FOR: rider_id, city, payment_method (equality lookups)

class FormatDecisionEngine:
    """
    Recommends the optimal Hadoop file format based on workload
    characteristics. Codifies the decision framework from this project.
    """

    def __init__(self):
        self.scores = {
            'text': 0,
            'sequencefile': 0,
            'avro': 0,
            'parquet': 0,
            'orc': 0,
        }
        self.reasons = {k: [] for k in self.scores}

    def _add_score(self, fmt, points, reason):
        self.scores[fmt] += points
        self.reasons[fmt].append(f"+{points}: {reason}")

    def _subtract_score(self, fmt, points, reason):
        self.scores[fmt] -= points
        self.reasons[fmt].append(f"-{points}: {reason}")

    def evaluate_query_pattern(self, pattern):
        """
        pattern: 'column_selective', 'full_row', 'mixed'
        """
        if pattern == 'column_selective':
            self._add_score('parquet', 10, "Column-selective queries → column pruning")
            self._add_score('orc', 10, "Column-selective queries → column pruning")
            self._subtract_score('text', 5, "No column pruning — reads all columns")
            self._subtract_score('avro', 3, "Row-oriented — reads all columns")
            self._subtract_score('sequencefile', 3, "No column pruning")
        elif pattern == 'full_row':
            self._add_score('avro', 5, "Full-row access — row-oriented is efficient")
            self._add_score('text', 3, "Full-row access — simple sequential read")
            self._subtract_score('parquet', 2, "Columnar reassembly overhead for full rows")
            self._subtract_score('orc', 2, "Columnar reassembly overhead for full rows")
        elif pattern == 'mixed':
            self._add_score('parquet', 5, "Good for mixed workloads")
            self._add_score('orc', 5, "Good for mixed workloads")

    def evaluate_primary_engine(self, engine):
        """
        engine: 'hive', 'spark', 'both', 'pig', 'mapreduce'
        """
        if engine == 'hive':
            self._add_score('orc', 10, "ORC is Hive-optimized (vectorized, bloom, ACID)")
            self._add_score('parquet', 5, "Parquet works well in Hive too")
        elif engine == 'spark':
            self._add_score('parquet', 10, "Parquet is Spark's native format")
            self._add_score('orc', 3, "ORC supported in Spark but not native")
        elif engine == 'both':
            self._add_score('parquet', 8, "Best cross-ecosystem compatibility")
            self._add_score('orc', 5, "Works in both but Parquet is more portable")
        elif engine == 'pig':
            self._add_score('avro', 5, "Pig has native Avro support")
            self._add_score('text', 3, "Pig reads text easily with PigStorage")
        elif engine == 'mapreduce':
            self._add_score('sequencefile', 5, "Native MR format for KV pairs")
            self._add_score('avro', 4, "Good MR integration")

    def evaluate_schema_evolution(self, needed):
        """
        needed: True or False
        """
        if needed:
            self._add_score('avro', 10, "Best schema evolution (add/remove/rename)")
            self._add_score('parquet', 5, "Supports column addition with defaults")
            self._add_score('orc', 5, "Supports column addition with defaults")
            self._subtract_score('text', 8, "No schema — evolution breaks parsing")
            self._subtract_score('sequencefile', 5, "No schema evolution support")

    def evaluate_compression_priority(self, priority):
        """
        priority: 'maximum', 'balanced', 'minimal'
        """
        if priority == 'maximum':
            self._add_score('orc', 8, "Best compression (zlib/zstd + columnar encoding)")
            self._add_score('parquet', 7, "Excellent compression (dictionary + RLE)")
            self._subtract_score('text', 5, "Worst compression (or none)")
        elif priority == 'balanced':
            self._add_score('parquet', 5, "Good snappy compression balance")
            self._add_score('orc', 5, "Good snappy compression balance")
            self._add_score('avro', 3, "Moderate compression")
        elif priority == 'minimal':
            self._add_score('text', 3, "No compression overhead")

    def evaluate_data_volume(self, volume_gb):
        """
        volume_gb: estimated data size in GB
        """
        if volume_gb > 1000:
            self._add_score('orc', 5, f"{volume_gb}GB → columnar compression critical")
            self._add_score('parquet', 5, f"{volume_gb}GB → columnar compression critical")
            self._subtract_score('text', 8, f"{volume_gb}GB as text → massive storage waste")
        elif volume_gb > 100:
            self._add_score('parquet', 3, f"{volume_gb}GB → compression beneficial")
            self._add_score('orc', 3, f"{volume_gb}GB → compression beneficial")
        elif volume_gb < 1:
            self._add_score('text', 3, f"{volume_gb}GB → format overhead not worth it")
            self._subtract_score('parquet', 1, "Small data — Parquet overhead significant")

    def evaluate_write_frequency(self, frequency):
        """
        frequency: 'write_once', 'frequent_append', 'frequent_update'
        """
        if frequency == 'write_once':
            self._add_score('parquet', 3, "Write-once → optimize for reads")
            self._add_score('orc', 3, "Write-once → optimize for reads")
        elif frequency == 'frequent_append':
            self._add_score('avro', 5, "Fast row-oriented appends")
            self._add_score('text', 3, "Simplest append model")
            self._subtract_score('parquet', 2, "Columnar write buffering overhead")
        elif frequency == 'frequent_update':
            self._add_score('orc', 10, "ONLY format supporting ACID updates in Hive")
            self._subtract_score('parquet', 5, "No native update support")
            self._subtract_score('text', 5, "No update support")
            self._subtract_score('avro', 5, "No update support")

    def evaluate_predicate_pushdown(self, needed):
        """
        needed: True or False
        """
        if needed:
            self._add_score('orc', 8, "Best pushdown (stripe index + bloom filters)")
            self._add_score('parquet', 6, "Good pushdown (row group statistics)")
            self._subtract_score('text', 5, "No predicate pushdown")
            self._subtract_score('avro', 3, "No predicate pushdown")
            self._subtract_score('sequencefile', 3, "No predicate pushdown")

    def evaluate_human_readable(self, needed):
        """
        needed: True or False
        """
        if needed:
            self._add_score('text', 10, "Only human-readable format")
            self._subtract_score('parquet', 2, "Binary — needs tools to inspect")
            self._subtract_score('orc', 2, "Binary — needs tools to inspect")

    def recommend(self):
        """Return sorted recommendations with explanations."""
        sorted_formats = sorted(
            self.scores.items(),
            key=lambda x: x[1],
            reverse=True
        )
        return sorted_formats

    def print_report(self):
        """Print formatted recommendation report."""
        print("\n" + "=" * 70)
        print("FILE FORMAT RECOMMENDATION REPORT")
        print("=" * 70)

        recommendations = self.recommend()
        winner = recommendations[0]

        print(f"\n🏆 RECOMMENDED FORMAT: {winner[0].upper()} (score: {winner[1]})")
        print()

        print("RANKING:")
        for i, (fmt, score) in enumerate(recommendations):
            bar = "█" * max(score, 0) + "░" * max(0, 50 - max(score, 0))
            medal = ["🥇", "🥈", "🥉", "  ", "  "][i]
            print(f"  {medal} {fmt:<15} [{bar[:30]}] {score:+d} pts")

        print("\nDETAILED REASONING:")
        for fmt, score in recommendations:
            print(f"\n  ── {fmt.upper()} ({score:+d} points) ──")
            for reason in self.reasons[fmt]:
                print(f"    {reason}")

        print("\n" + "=" * 70)
        return winner[0]


class FormatConverter:
    """
    Generates Hive SQL to convert between file formats.
    """

    @staticmethod
    def generate_conversion_sql(source_db, source_table,
                                 target_db, target_table,
                                 target_format, compression='SNAPPY',
                                 bloom_columns=None, partitioned_by=None):
        """Generate SQL to convert a table to a different format."""
        lines = []
        lines.append(f"-- Convert {source_db}.{source_table} → "
                      f"{target_db}.{target_table} ({target_format})")
        lines.append(f"USE {target_db};")
        lines.append(f"")
        lines.append(f"DROP TABLE IF EXISTS {target_table};")
        lines.append(f"CREATE TABLE {target_table}")

        # Storage clause
        if target_format.upper() == 'ORC':
            lines.append(f"STORED AS ORC")
            tbl_props = [f"'orc.compress'='{compression}'"]
            if bloom_columns:
                tbl_props.append(
                    f"'orc.bloom.filter.columns'='{','.join(bloom_columns)}'")
            tbl_props.append("'orc.create.index'='true'")
            lines.append(f"TBLPROPERTIES ({', '.join(tbl_props)})")
        elif target_format.upper() == 'PARQUET':
            lines.append(f"STORED AS PARQUET")
            lines.append(
                f"TBLPROPERTIES ('parquet.compression'='{compression}')")
        elif target_format.upper() == 'AVRO':
            lines.append(f"STORED AS AVRO")
        elif target_format.upper() == 'SEQUENCEFILE':
            lines.append(f"STORED AS SEQUENCEFILE")
        else:
            lines.append(f"ROW FORMAT DELIMITED FIELDS TERMINATED BY ','")
            lines.append(f"STORED AS TEXTFILE")

        # Create as select
        lines.append(f"AS SELECT * FROM {source_db}.{source_table};")
        lines.append(f"")

        # Verification
        lines.append(f"-- Verify conversion")
        lines.append(
            f"SELECT COUNT(*) AS row_count FROM {target_table};")
        lines.append(f"DESCRIBE FORMATTED {target_table};")

        return '\n'.join(lines)

echo "[1] Tables created in all formats:"
for table in rides_text rides_sequencefile rides_avro rides_parquet rides_orc rides_orc_snappy; do

    format=$(hive -e "DESCRIBE FORMATTED rideshare_db.$table;" 2>/dev/null | grep -i "InputFormat" | head -1 | awk -F: '{print $2}' | xargs)
    echo "  ✅ $table: $count rows | Format: ${format##*.}"
done
echo ""

echo "[2] Storage sizes:"
printf "  %-25s %12s\n" "TABLE" "SIZE"
printf "  %-25s %12s\n" "-----" "----"
for table in rides_text rides_sequencefile rides_avro rides_parquet rides_orc rides_orc_snappy; do
    size=$(hdfs dfs -du -s -h $DB_PATH/$table 2>/dev/null | awk '{print $1$2}')
    printf "  %-25s %12s\n" "$table" "$size"

WHAT WE ESTABLISHED IN PROJECT 30:

✅ FILE FORMAT CHOICE IS THE SINGLE BIGGEST PERFORMANCE DECISION
   → Same data, same queries → 10-25× performance difference
   → Same data → 5-12× storage difference
   → Wrong format = wasted money, slow queries, brittle pipelines

✅ FIVE FORMATS, EACH WITH A PURPOSE:
   → Text: human readable, debugging, data exchange (NOT production)
   → SequenceFile: MapReduce intermediate, small file merging (legacy)
   → Avro: schema evolution, data ingestion, cross-system exchange
   → Parquet: columnar analytics, Spark-native, cross-ecosystem
   → ORC: columnar analytics, Hive-native, ACID, bloom filters

✅ COLUMNAR FORMATS (Parquet/ORC) DOMINATE FOR ANALYTICS
   → Column pruning: read only needed columns (90% I/O reduction)
   → Predicate pushdown: skip irrelevant data blocks
   → Better compression: similar values compress dramatically
   → Vectorized reads: batch processing of 1024 rows

✅ THE GOLDEN PIPELINE PATTERN:
   → Ingest: Sqoop/Flume → Avro (schema evolution, fast writes)
   → Process: Pig/MR → SequenceFile or Avro (intermediate)
   → Store: Hive warehouse → ORC (Hive) or Parquet (Spark)
   → Serve: HBase (random access — not a file format question)

✅ COMPRESSION + SPLITTABILITY MATRIX:
   → Text + gzip = NOT splittable → one mapper per file
   → Container formats (Avro/Parquet/ORC/SeqFile) + any codec = splittable
   → ALWAYS use container formats for compressed data

✅ SCHEMA EVOLUTION:
   → Avro: best (add/remove/rename columns safely)
   → Parquet/ORC: good (add columns with defaults)
   → Text/SeqFile: none (breaks on schema change)
