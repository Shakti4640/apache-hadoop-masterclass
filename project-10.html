PROJECT 10: DISTRIBUTING CUSTOMER DATA TO REGION-SPECIFIC FILES

🔴 HYPOTHETICAL PROBLEM

SCENARIO:
→ You work at a nationwide retail chain with operations in 4 regions:
    NORTH, SOUTH, EAST, WEST
→ Every day, 80 million customer transactions arrive in a single massive file
→ Each transaction line looks like:
    TXN_ID | CUSTOMER_ID | REGION | PRODUCT | AMOUNT | TIMESTAMP
→ The finance team for EACH region needs ONLY their region's data
→ Currently, the MapReduce job (from Projects 4-9 patterns) produces
   output files like:
    part-r-00000 (mixed data from all regions)
    part-r-00001 (mixed data from all regions)
    part-r-00002 (mixed data from all regions)

THE PAIN:
→ The NORTH finance team downloads ALL output files
→ They grep/filter for NORTH records — discarding 75% of data
→ SOUTH team does the same — discarding 75%
→ Every team downloads EVERYTHING, uses almost NOTHING
→ Massive wasted bandwidth, time, and frustration

WHAT WE ACTUALLY NEED:
→ part-r-00000 → contains ONLY North region data
→ part-r-00001 → contains ONLY South region data
→ part-r-00002 → contains ONLY East region data
→ part-r-00003 → contains ONLY West region data

WHO DECIDES WHICH RECORD GOES TO WHICH OUTPUT FILE?
→ THE PARTITIONER

🔍 HOW TO APPROACH IT

STEP 1: Understand what happens BETWEEN Map and Reduce

In Projects 4-9, we focused on Mapper logic and Reducer logic.
But there is a CRITICAL middle step we glossed over:

  MAPPER → [???] → REDUCER

That [???] is actually THREE sub-steps:

  MAPPER → PARTITION → SORT → SHUFFLE → REDUCER

┌──────────────────────────────────────────────────────────────────┐
│                  FULL MapReduce DATA FLOW                        │
│                                                                  │
│  MAP SIDE (each mapper machine):                                 │
│  ┌────────┐  ┌────────────┐  ┌──────────┐  ┌───────────────┐   │
│  │ MAPPER │→ │ PARTITIONER │→ │  SORTER  │→ │ COMBINER(opt) │   │
│  └────────┘  └────────────┘  └──────────┘  └───────────────┘   │
│                    │                                             │
│                    │ decides which REDUCER gets this key-value   │
│                    ▼                                             │
│  REDUCE SIDE:                                                    │
│  ┌─────────┐  ┌──────────┐  ┌─────────┐  ┌────────────────┐   │
│  │ SHUFFLE │→ │  MERGE   │→ │  SORT   │→ │    REDUCER     │   │
│  │ (fetch) │  │          │  │         │  │ (your logic)   │    │
│  └─────────┘  └──────────┘  └─────────┘  └────────────────┘   │
│                                                                  │
│  Each REDUCER writes ONE output file: part-r-NNNNN              │
│  Reducer 0 → part-r-00000                                       │
│  Reducer 1 → part-r-00001                                       │
│  Reducer 2 → part-r-00002                                       │
│  ...                                                             │
└──────────────────────────────────────────────────────────────────┘

STEP 2: Understand the DEFAULT partitioner

By default, Hadoop uses HashPartitioner:

    partition = (key.hashCode() & Integer.MAX_VALUE) % numReducers

This means:
→ Keys are RANDOMLY distributed across reducers
→ "NORTH" might go to reducer 0 OR reducer 2 — depends on hash
→ Same key ALWAYS goes to same reducer (deterministic hash)
→ But you CANNOT CONTROL which reducer gets which key

STEP 3: Realize we need a CUSTOM partitioner

We want:
→ NORTH records → ALWAYS reducer 0 → part-r-00000
→ SOUTH records → ALWAYS reducer 1 → part-r-00001
→ EAST records  → ALWAYS reducer 2 → part-r-00002
→ WEST records  → ALWAYS reducer 3 → part-r-00003

This requires OVERRIDING the default HashPartitioner
with our OWN logic.

💡 HOW CAN WE SOLVE IT

SOLUTION: Write a Custom Partitioner class

A Partitioner has ONE method to implement:

    getPartition(KEY key, VALUE value, int numReduceTasks)
    → Returns an integer: 0 to (numReduceTasks - 1)
    → This integer IS the reducer number

┌────────────────────────────────────────────────────────────┐
│                  PARTITIONER CONTRACT                       │
│                                                             │
│  INPUT:  A key-value pair emitted by the mapper             │
│  OUTPUT: An integer (reducer index)                         │
│                                                             │
│  RULE 1: Return value must be >= 0                          │
│  RULE 2: Return value must be < numReduceTasks              │
│  RULE 3: Same key MUST always return same partition          │
│          (otherwise identical keys go to different reducers  │
│           and aggregation BREAKS)                            │
│                                                             │
│  DIFFERENCE FROM COMBINER (Project 9):                      │
│  → Combiner reduces DATA VOLUME before shuffle              │
│  → Partitioner decides DATA ROUTING to reducers             │
│  → Combiner is OPTIONAL                                     │
│  → Partitioner ALWAYS exists (default = HashPartitioner)    │
└────────────────────────────────────────────────────────────┘

EXECUTION ORDER on map side:
  1. Mapper emits (key, value)
  2. Partitioner decides: this pair goes to reducer N
  3. Pair is written to a partition buffer for reducer N
  4. Buffer is sorted by key
  5. Combiner (if set, from Project 9) runs on sorted buffer
  6. Data is spilled to disk, ready for shuffle

✅ WHAT COULD BE DONE

APPROACH A: Region as the KEY — Custom Partitioner routes by key
→ Mapper emits: key=REGION, value=transaction_data
→ Partitioner maps REGION string to reducer index
→ Each reducer gets one region → one output file per region

APPROACH B: Composite key — Region embedded in key
→ Mapper emits: key=REGION_PRODUCT, value=amount
→ Partitioner extracts REGION from composite key
→ Same partitioner logic but key carries more info
→ (We explore composite keys fully in Project 13)

APPROACH C: Region as VALUE field — Partitioner inspects value
→ Mapper emits: key=PRODUCT, value=REGION|AMOUNT
→ Partitioner parses VALUE to extract region
→ LEGAL but BAD PRACTICE — Partitioner should use KEY

WE CHOOSE APPROACH A:
→ Cleanest separation
→ Partitioner only looks at key (standard pattern)
→ Easy to understand as first custom partitioner

✅ WHAT SHOULD DO

1. SET NUMBER OF REDUCERS = NUMBER OF REGIONS
   → 4 regions → 4 reducers
   → job.setNumReduceTasks(4)
   → Each reducer writes exactly one file
   → File-to-region mapping is deterministic

2. MAP REGION NAMES TO REDUCER INDICES EXPLICITLY
   → NORTH → 0
   → SOUTH → 1
   → EAST  → 2
   → WEST  → 3
   → Use a HashMap or switch statement — NOT hash function

3. HANDLE UNKNOWN REGIONS GRACEFULLY
   → What if data has "CENTRAL" or typo "NROTH"?
   → Route to a default reducer (e.g., reducer 0)
   → Or count as error — depends on business logic

4. TEST WITH KNOWN DATA FIRST
   → Create a small file with known region distribution
   → Verify each output file contains ONLY its region
   → Then scale to full dataset

5. DOCUMENT THE PARTITION MAPPING
   → Downstream consumers need to know:
     "part-r-00000 = NORTH, part-r-00001 = SOUTH..."
   → Without this, output files are meaningless numbers

🚫 WHAT SHOULD NOT DO

1. DO NOT use more reducers than regions
   → 4 regions but 10 reducers = 6 EMPTY output files
   → Empty part-r-00004 through part-r-00009 created
   → Wastes task slots, confuses consumers

2. DO NOT use fewer reducers than regions
   → 4 regions but 2 reducers = 2 regions MIXED in each file
   → Defeats the entire purpose

3. DO NOT return negative partition numbers
   → getPartition() returning -1 crashes the job
   → ArrayIndexOutOfBoundsException
   → ALWAYS use Math.abs() or explicit bounds checking

4. DO NOT make partition logic non-deterministic
   → Using random(), current time, etc. in getPartition()
   → Same key MUST always go to same reducer
   → Otherwise: identical keys split across reducers
   → Aggregation produces WRONG RESULTS

5. DO NOT ignore the Combiner interaction
   → Combiner (Project 9) runs AFTER partitioning on map side
   → Combiner runs PER PARTITION BUFFER
   → Combiner logic must still be valid within each partition
   → No conflict here — but understand the order

6. DO NOT hardcode partition count in partitioner
   → Use the numReduceTasks parameter passed to getPartition()
   → If someone changes job.setNumReduceTasks(), your
     partitioner must still work correctly

├──────────────────────────────────────┼─────────────────────────┤

│ (default — no special routing)       │ (built-in default)       │

│ (A-F → R0, G-M → R1, N-Z → R2)     │ Partitioner              │
├──────────────────────────────────────┼─────────────────────────┤

├──────────────────────────────────────┼─────────────────────────┤

└──────────────────────────────────────┴─────────────────────────┘

DECISION RULE:
→ If you DON'T CARE which reducer gets which key → HashPartitioner
→ If you NEED CONTROL over key-to-reducer mapping → Custom Partitioner
→ If output files must represent MEANINGFUL groups → Custom Partitioner

🚫 WHEN NOT TO USE

1. WHEN DEFAULT HASHING IS SUFFICIENT
   → Most aggregation jobs (sum, count, avg) don't care
     WHICH reducer processes WHICH keys
   → They only need: same key → same reducer (hash guarantees this)
   → Adding a custom partitioner adds complexity for zero benefit

2. WHEN CATEGORIES ARE DYNAMIC OR UNKNOWN
   → If regions change weekly (new regions added)
   → Hardcoded partitioner breaks with new data
   → Use HashPartitioner + post-processing filter instead

3. WHEN CATEGORIES ARE SEVERELY IMBALANCED
   → NORTH has 70% of transactions
   → SOUTH has 5%
   → Custom partitioner sends 70% to ONE reducer
   → That reducer becomes bottleneck (data skew)
   → (We solve this in Project 33)
   → Better to use HashPartitioner for balanced distribution

4. WHEN NUMBER OF CATEGORIES IS HUGE
   → 10,000 product categories → 10,000 reducers?
   → Each reducer has JVM overhead (~200MB minimum)
   → 10,000 reducers = 2TB of overhead RAM
   → Use MultipleOutputs (Project 20) instead

⚙️ HOW IT ONLY WORKS — INTERNAL MECHANICS

═══════════════════════════════════════════════════════════
MECHANIC 1: THE PARTITION BUFFER SYSTEM
═══════════════════════════════════════════════════════════

When a Mapper emits a key-value pair, here is what ACTUALLY happens:

Step 1: Mapper calls context.write(key, value)

Step 2: MapOutputBuffer receives the pair
        → This is a CIRCULAR BUFFER in memory
        → Default size: 100MB (mapreduce.task.io.sort.mb)

Step 3: Partitioner is called IMMEDIATELY
        → int partition = partitioner.getPartition(key, value, numReducers)
        → The partition number is STORED alongside the key-value pair
        → Buffer now holds: [partition_number | key | value]

Step 4: When buffer reaches 80% full (io.sort.spill.percent):
        → SPILL occurs
        → Data in buffer is:
          a) Sorted by (partition, then by key within partition)
          b) Combiner runs on each partition's sorted data (if set)
          c) Written to a spill file on local disk

        ┌─────────────── SPILL FILE LAYOUT ───────────────┐
        │                                                   │
        │  [Partition 0 data: key1,val | key2,val | ...]   │
        │  [Partition 1 data: key3,val | key5,val | ...]   │
        │  [Partition 2 data: key4,val | key7,val | ...]   │
        │  [Partition 3 data: key6,val | key8,val | ...]   │
        │                                                   │
        │  + INDEX FILE pointing to each partition's offset │
        └───────────────────────────────────────────────────┘

Step 5: After mapper finishes, ALL spill files are MERGED
        → One final sorted file per mapper
        → Still organized by partition sections

Step 6: SHUFFLE — Each reducer FETCHES its partition section
        → Reducer 0 contacts ALL mappers, asks for "partition 0 data"
        → Reducer 1 contacts ALL mappers, asks for "partition 1 data"
        → This is a PULL model (reducers pull from mappers)

        ┌─────────────────────────────────────────────────────┐
        │              SHUFFLE VISUALIZATION                    │
        │                                                       │
        │  Mapper 1 output:  [P0|P1|P2|P3]                    │
        │  Mapper 2 output:  [P0|P1|P2|P3]                    │
        │  Mapper 3 output:  [P0|P1|P2|P3]                    │
        │                                                       │
        │  Reducer 0 pulls: M1[P0] + M2[P0] + M3[P0]          │
        │  Reducer 1 pulls: M1[P1] + M2[P1] + M3[P1]          │
        │  Reducer 2 pulls: M1[P2] + M2[P2] + M3[P2]          │
        │  Reducer 3 pulls: M1[P3] + M2[P3] + M3[P3]          │
        └─────────────────────────────────────────────────────┘

═══════════════════════════════════════════════════════════
MECHANIC 2: HashPartitioner INTERNALS
═══════════════════════════════════════════════════════════

The default HashPartitioner source code is literally 3 lines:

    public int getPartition(K key, V value, int numReduceTasks) {
        return (key.hashCode() & Integer.MAX_VALUE) % numReduceTasks;
    }

WHY "& Integer.MAX_VALUE"?
→ hashCode() can return NEGATIVE numbers
→ Negative modulo in Java gives negative results
→ Partition index MUST be non-negative
→ Bitwise AND with 0x7FFFFFFF strips the sign bit
→ Example: hashCode() = -12345
  → -12345 & 0x7FFFFFFF = 2147471303
  → 2147471303 % 4 = 3 (valid partition index)

WHY THIS CAUSES RANDOM DISTRIBUTION:
→ "NORTH".hashCode() = 75457038
→ 75457038 % 4 = 2 → NORTH goes to reducer 2
→ "SOUTH".hashCode() = 79482265
→ 79482265 % 4 = 1 → SOUTH goes to reducer 1

→ No guarantee about which region goes where
→ Multiple regions CAN map to same reducer
→ This is why we need CUSTOM partitioner for our problem

═══════════════════════════════════════════════════════════
MECHANIC 3: CUSTOM PARTITIONER LIFECYCLE
═══════════════════════════════════════════════════════════

1. Job submission: Driver sets job.setPartitionerClass(MyPartitioner.class)
2. Hadoop serializes the Job configuration
3. Each mapper JVM instantiates MyPartitioner via reflection
4. For EVERY key-value pair emitted by mapper:
   → MyPartitioner.getPartition() is called
   → Returns integer → stored in output buffer
5. Partitioner instance lives for entire mapper lifetime
6. ONE partitioner instance per mapper task
7. Partitioner is NOT called on reducer side — only mapper side

⚠️ COMMON PITFALLS AND FAILURE MODES

PITFALL 1: "Partition index out of range"
→ ERROR: java.lang.ArrayIndexOutOfBoundsException
→ CAUSE: getPartition() returned value >= numReduceTasks
→ EXAMPLE: You return 4 but only 4 reducers (valid: 0,1,2,3)
→ FIX: Always validate: return Math.min(index, numReduceTasks - 1)

PITFALL 2: "Output files exist but some are empty"
→ CAUSE: No data maps to that partition
→ EXAMPLE: No WEST region transactions today
→ Reducer 3 runs, produces empty part-r-00003
→ NOT AN ERROR — just empty partition
→ But downstream scripts may fail on empty files

PITFALL 3: "Job is extremely slow — one reducer takes forever"
→ CAUSE: Data skew — one partition has 80% of records
→ DIAGNOSIS: Check reducer progress in YARN UI
  → 3 reducers finish in 5 minutes
  → 1 reducer still at 20% after 30 minutes
→ ROOT CAUSE: That region has far more data
→ FIX OPTIONS:
  a) Sub-partition the large region (Project 33)
  b) Accept the imbalance if tolerable
  c) Use Combiner (Project 9) to reduce data volume

PITFALL 4: "Custom partitioner not being used"
→ CAUSE: Forgot job.setPartitionerClass() in driver
→ Hadoop silently uses HashPartitioner
→ Output files are mixed — no error, just wrong results
→ ALWAYS verify with small test data first

PITFALL 5: "Number of reducers doesn't match partition logic"
→ CAUSE: Partitioner hardcodes 4 regions
→ But job.setNumReduceTasks(8) — 8 reducers
→ Partitioner returns 0-3, reducers 4-7 get nothing
→ 4 empty files wasted
→ FIX: Partitioner should USE numReduceTasks parameter
       OR driver should explicitly set matching count

PITFALL 6: "Combiner output differs from expected"
→ CAUSE: Combiner runs PER PARTITION SECTION in spill
→ If you're counting unique values, combiner may
   de-duplicate WITHIN one spill but not across spills
→ Same pitfall as Project 9 — Combiner must be idempotent
   and associative for aggregation operations

📈 SCALABILITY CONSIDERATIONS

PARTITIONER AND CLUSTER SCALE:

┌─────────────────────────────────────────────────────────────┐
│ FACTOR              │ IMPACT                                │
├─────────────────────┼───────────────────────────────────────┤
│ Number of reducers  │ Each reducer = 1 JVM container        │
│                     │ More reducers = more parallelism       │
│                     │ BUT: more shuffle connections          │
│                     │ M mappers × R reducers = M×R fetches  │
├─────────────────────┼───────────────────────────────────────┤
│ Reducer count       │ TOO FEW: each reducer overwhelmed     │
│ sweet spot          │ TOO MANY: overhead exceeds benefit     │
│                     │ RULE OF THUMB:                         │
│                     │   0.95 × numNodes × maxReducersPerNode│
│                     │   OR based on output data size:        │
│                     │   totalOutputSize / (256MB to 1GB)     │
├─────────────────────┼───────────────────────────────────────┤
│ Partition balance   │ Skewed partitions = wasted cluster     │
│                     │ 1 slow reducer holds up ENTIRE job     │
│                     │ Job completion = slowest reducer        │
├─────────────────────┼───────────────────────────────────────┤
│ Shuffle overhead    │ Custom partitioner doesn't change      │
│                     │ shuffle VOLUME — same data transferred │
│                     │ But BALANCED partitions mean shuffle    │
│                     │ completes evenly across reducers        │
└─────────────────────┴───────────────────────────────────────┘

SCALING OUR REGION EXAMPLE:

4 regions → 4 reducers → WORKS at any data size
BUT:
  → If NORTH = 60% of data, reducer 0 processes 3× more
  → 100 node cluster: 99 nodes idle waiting for 1 reducer
  → SOLUTION at scale: Sub-partition NORTH into NORTH_A, NORTH_B, NORTH_C
  → 6 reducers now, NORTH split across 3 → better balance
  → (Full skew handling in Project 33)

NETWORK IMPACT:
  → Shuffle transfers ALL mapper output to reducers over network
  → Custom partitioner doesn't reduce transfer volume
  → Combiner (Project 9) reduces volume BEFORE shuffle
  → Best practice: COMBINE first, then PARTITION
  → Execution order guarantees this: Partition → Sort → Combine → Shuffle

🔍 INTERPRETABILITY CONSIDERATIONS

HOW TO VERIFY YOUR PARTITIONER IS WORKING:

1. CHECK OUTPUT FILE SIZES
   hdfs dfs -ls /output/region_data/
   
   -rw-r--r--  part-r-00000    245MB  ← NORTH
   -rw-r--r--  part-r-00001    198MB  ← SOUTH
   -rw-r--r--  part-r-00002    312MB  ← EAST
   -rw-r--r--  part-r-00003    156MB  ← WEST
   
   → If sizes match expected regional distribution → CORRECT
   → If all same size → probably using HashPartitioner (random)

2. SPOT-CHECK FILE CONTENTS
   hdfs dfs -cat /output/region_data/part-r-00000 | head -5
   → ALL lines should show NORTH
   → If mixed regions appear → partitioner not working

3. USE JOB COUNTERS (from Project 7)
   → Add custom counters in partitioner:
     context.getCounter("Partitions", "NORTH").increment(1)
   → After job: check counter values match expected distribution

4. MAPPER OUTPUT RECORDS vs REDUCER INPUT RECORDS
   → Built-in counter: "Map output records" should equal
     sum of all "Reduce input records" (minus combiner reduction)
   → If mismatch → data loss during shuffle (rare but possible)

5. OUTPUT FILE NAMING CONVENTION
   → part-r-00000 = Reducer 0's output
   → part-r-00001 = Reducer 1's output
   → The "r" means REDUCER output
   → "m" would mean MAP-ONLY output (Project 11)
   → The number = reducer index = partition number

// FILE: RegionPartitionJob.java

import org.apache.hadoop.conf.Configuration;
import org.apache.hadoop.conf.Configured;
import org.apache.hadoop.fs.Path;
import org.apache.hadoop.io.IntWritable;
import org.apache.hadoop.io.LongWritable;
import org.apache.hadoop.io.Text;
import org.apache.hadoop.mapreduce.Job;
import org.apache.hadoop.mapreduce.Mapper;
import org.apache.hadoop.mapreduce.Partitioner;
import org.apache.hadoop.mapreduce.Reducer;
import org.apache.hadoop.mapreduce.lib.input.FileInputFormat;
import org.apache.hadoop.mapreduce.lib.output.FileOutputFormat;
import org.apache.hadoop.util.Tool;
import org.apache.hadoop.util.ToolRunner;

import java.io.IOException;
import java.util.HashMap;
import java.util.Map;

public class RegionPartitionJob extends Configured implements Tool {

    // ════════════════════════════════════════════════════════
    // MAPPER: Parse transaction lines → emit (region, amount)
    // ════════════════════════════════════════════════════════
    //
    // Input line format:
    //   TXN001|CUST042|NORTH|Laptop|1299.99|2024-01-15
    //   TXN002|CUST017|SOUTH|Phone|899.50|2024-01-15
    //
    // Emit: key = region (Text), value = full transaction line (Text)
    //
    // WHY region as key?
    //   → Partitioner routes by KEY
    //   → Region in key = partitioner can read it directly
    //   → No need to parse value in partitioner (clean design)

    public static class TransactionMapper
            extends Mapper<LongWritable, Text, Text, Text> {

        private Text regionKey = new Text();
        private Text transactionValue = new Text();

        // Counter group for tracking parse results
        enum MapCounters {
            VALID_RECORDS,
            INVALID_RECORDS,
            UNKNOWN_REGION
        }

        @Override
        protected void map(LongWritable offset, Text line, Context context)
                throws IOException, InterruptedException {

            String record = line.toString().trim();

            // Skip empty lines or headers
            if (record.isEmpty() || record.startsWith("TXN_ID")) {
                return;
            }

            String[] fields = record.split("\\|");

            // Validate: expect 6 fields
            // TXN_ID | CUSTOMER_ID | REGION | PRODUCT | AMOUNT | TIMESTAMP
            if (fields.length != 6) {
                context.getCounter(MapCounters.INVALID_RECORDS).increment(1);
                return;
            }

            String region = fields[2].trim().toUpperCase();

            // Validate region is one of expected values
            if (!region.equals("NORTH") && !region.equals("SOUTH") &&
                !region.equals("EAST") && !region.equals("WEST")) {
                context.getCounter(MapCounters.UNKNOWN_REGION).increment(1);
                // Still emit — partitioner will route to default
                // This way we don't lose data silently
            }

            regionKey.set(region);
            transactionValue.set(record);

            context.write(regionKey, transactionValue);
            context.getCounter(MapCounters.VALID_RECORDS).increment(1);
        }
    }


    // ════════════════════════════════════════════════════════
    // CUSTOM PARTITIONER: Route regions to specific reducers
    // ════════════════════════════════════════════════════════
    //
    // THIS IS THE CORE OF PROJECT 10
    //
    // Contract:
    //   → Input: key (Text = region), value, numReduceTasks
    //   → Output: integer 0 to (numReduceTasks - 1)
    //
    // Our mapping:
    //   NORTH → 0
    //   SOUTH → 1
    //   EAST  → 2
    //   WEST  → 3
    //   Unknown → 0 (fallback — don't crash, route safely)

    public static class RegionPartitioner
            extends Partitioner<Text, Text> {

        // Static mapping — initialized once, used for all calls
        private static final Map<String, Integer> REGION_MAP = new HashMap<>();

        static {
            REGION_MAP.put("NORTH", 0);
            REGION_MAP.put("SOUTH", 1);
            REGION_MAP.put("EAST",  2);
            REGION_MAP.put("WEST",  3);
        }

        @Override
        public int getPartition(Text key, Text value, int numReduceTasks) {
            // ── CRITICAL: Extract region from key ──
            String region = key.toString().toUpperCase().trim();

            // Look up the partition index
            Integer partitionIndex = REGION_MAP.get(region);

            if (partitionIndex != null) {
                // Known region → route to designated reducer
                //
                // WHY we check against numReduceTasks:
                // If someone runs with fewer reducers than our mapping expects,
                // we must not return an index >= numReduceTasks
                return partitionIndex % numReduceTasks;
            } else {
                // Unknown region → route to reducer 0 (safe fallback)
                //
                // ALTERNATIVE: Could use HashPartitioner logic here:
                //   return (key.hashCode() & Integer.MAX_VALUE) % numReduceTasks;
                // But explicit routing is more predictable
                return 0;
            }
        }
    }


    // ════════════════════════════════════════════════════════
    // REDUCER: Collect all transactions for this partition
    // ════════════════════════════════════════════════════════
    //
    // Since partitioner guarantees each reducer gets ONE region,
    // the reducer simply writes all records.
    //
    // In a real scenario, you might also:
    //   → Calculate regional totals (sum amounts)
    //   → Count transactions per product within region
    //   → We keep it simple to focus on PARTITIONING concept
    //
    // We also calculate regional summary stats to demonstrate
    // that reducer sees ONLY its region's data.

    public static class RegionReducer
            extends Reducer<Text, Text, Text, Text> {

        enum ReduceCounters {
            NORTH_RECORDS,
            SOUTH_RECORDS,
            EAST_RECORDS,
            WEST_RECORDS,
            TOTAL_AMOUNT_CENTS
        }

        @Override
        protected void reduce(Text region, Iterable<Text> transactions,
                              Context context)
                throws IOException, InterruptedException {

            long recordCount = 0;
            double totalAmount = 0.0;

            for (Text transaction : transactions) {
                String record = transaction.toString();
                String[] fields = record.split("\\|");

                // Write each transaction to output
                // Output format: REGION \t full_record
                context.write(region, transaction);

                // Track statistics
                recordCount++;
                if (fields.length >= 5) {
                    try {
                        totalAmount += Double.parseDouble(fields[4].trim());
                    } catch (NumberFormatException e) {
                        // Skip bad amounts — don't crash the job
                    }
                }
            }

            // Increment region-specific counters
            String regionName = region.toString().toUpperCase();
            switch (regionName) {
                case "NORTH":
                    context.getCounter(ReduceCounters.NORTH_RECORDS)
                           .increment(recordCount);
                    break;
                case "SOUTH":
                    context.getCounter(ReduceCounters.SOUTH_RECORDS)
                           .increment(recordCount);
                    break;
                case "EAST":
                    context.getCounter(ReduceCounters.EAST_RECORDS)
                           .increment(recordCount);
                    break;
                case "WEST":
                    context.getCounter(ReduceCounters.WEST_RECORDS)
                           .increment(recordCount);
                    break;
            }
        }
    }


    // ════════════════════════════════════════════════════════
    // DRIVER: Wire everything together
    // ════════════════════════════════════════════════════════

    @Override
    public int run(String[] args) throws Exception {

        if (args.length != 2) {
            System.err.println("Usage: RegionPartitionJob <input_path> <output_path>");
            System.err.println("  input_path:  HDFS path to transaction files");
            System.err.println("  output_path: HDFS path for region-partitioned output");
            return 1;
        }

        Configuration conf = getConf();
        Job job = Job.getInstance(conf, "Region Partitioned Transaction Analysis");
        job.setJarByClass(RegionPartitionJob.class);

        // ── Set Mapper ──
        job.setMapperClass(TransactionMapper.class);

        // ── Set CUSTOM Partitioner — THE KEY LINE ──
        // Without this line: HashPartitioner used → regions mixed
        // With this line: Our logic routes each region to specific reducer
        job.setPartitionerClass(RegionPartitioner.class);

        // ── Set Reducer ──
        job.setReducerClass(RegionReducer.class);

        // ── CRITICAL: Number of reducers MUST match partition count ──
        // Our partitioner maps to indices 0,1,2,3
        // So we need EXACTLY 4 reducers
        job.setNumReduceTasks(4);

        // ── Set output key-value types ──
        job.setMapOutputKeyClass(Text.class);
        job.setMapOutputValueClass(Text.class);
        job.setOutputKeyClass(Text.class);
        job.setOutputValueClass(Text.class);

        // ── Set input/output paths ──
        FileInputFormat.addInputPath(job, new Path(args[0]));
        FileOutputFormat.setOutputPath(job, new Path(args[1]));

        // ── Run and report ──
        boolean success = job.waitForCompletion(true);

        // Print partition summary from counters
        if (success) {
            System.out.println("\n═══ REGION PARTITION SUMMARY ═══");
            System.out.println("NORTH records (part-r-00000): " +
                job.getCounters()
                   .findCounter(RegionReducer.ReduceCounters.NORTH_RECORDS)
                   .getValue());
            System.out.println("SOUTH records (part-r-00001): " +
                job.getCounters()
                   .findCounter(RegionReducer.ReduceCounters.SOUTH_RECORDS)
                   .getValue());
            System.out.println("EAST records  (part-r-00002): " +
                job.getCounters()
                   .findCounter(RegionReducer.ReduceCounters.EAST_RECORDS)
                   .getValue());
            System.out.println("WEST records  (part-r-00003): " +
                job.getCounters()
                   .findCounter(RegionReducer.ReduceCounters.WEST_RECORDS)
                   .getValue());
            System.out.println("════════════════════════════════\n");
        }

        return success ? 0 : 1;
    }

    public static void main(String[] args) throws Exception {
        int exitCode = ToolRunner.run(new Configuration(),
                                      new RegionPartitionJob(), args);
        System.exit(exitCode);
    }
}

import random
import os

def generate_large_transactions(filepath, num_records, region_distribution):
    """
    Generate realistic transaction data with controlled region distribution.
    
    region_distribution: dict like {"NORTH": 0.40, "SOUTH": 0.25, "EAST": 0.20, "WEST": 0.15}
    This creates SKEWED data to test partitioner behavior.
    """
    products = [
        ("Laptop", 800, 2000), ("Phone", 400, 1200),
        ("Tablet", 200, 800), ("Watch", 100, 500),
        ("Headphones", 20, 300), ("Cable", 5, 30),
        ("Monitor", 200, 800), ("Keyboard", 30, 150),
        ("Mouse", 10, 80), ("SSD", 50, 300)
    ]
    
    # Build weighted region list for random selection
    regions = []
    for region, weight in region_distribution.items():
        regions.extend([region] * int(weight * 1000))
    
    with open(filepath, 'w') as f:
        for i in range(1, num_records + 1):
            txn_id = f"TXN{i:08d}"
            cust_id = f"CUST{random.randint(1, 100000):06d}"
            region = random.choice(regions)
            product, min_price, max_price = random.choice(products)
            amount = round(random.uniform(min_price, max_price), 2)
            
            # Random date in January 2024
            day = random.randint(1, 31)
            timestamp = f"2024-01-{day:02d}"
            
            line = f"{txn_id}|{cust_id}|{region}|{product}|{amount}|{timestamp}\n"
            f.write(line)
    
    file_size_mb = os.path.getsize(filepath) / (1024 * 1024)
    print(f"Generated {num_records:,} records ({file_size_mb:.1f} MB)")
    print(f"Region distribution: {region_distribution}")

🔑 KEY TAKEAWAY

WHAT PROJECT 10 ESTABLISHED:

✅ PARTITIONER controls which reducer receives which key-value pair
✅ Default HashPartitioner distributes randomly (by hash)
✅ Custom Partitioner gives EXPLICIT control over routing
✅ Number of reducers MUST match your partition logic
✅ Each reducer writes ONE output file (part-r-NNNNN)
✅ Partitioner runs on MAP SIDE — before shuffle
✅ Execution order: Map → Partition → Sort → Combine → Shuffle → Reduce

CONNECTION TO PROJECT 9 (Combiner):
→ Project 9: Combiner reduces DATA VOLUME before network transfer
→ Project 10: Partitioner controls DATA ROUTING to reducers
→ Both run on map side, but serve completely different purposes
→ Combiner = optimization (less data shipped)
→ Partitioner = routing (right data to right reducer)
→ They WORK TOGETHER: partition first, then combine within partition

WHAT WE DID NOT COVER YET:
→ Map-only jobs where we need NO reducers at all (Project 11)
→ Composite keys that carry multiple pieces of info (Project 13)
→ Global sorting where partition boundaries must be calculated (Project 18)
→ Data skew where one partition overwhelms its reducer (Project 33)

🌉 BRIDGE TO PROJECT 11

PROJECT 10 gave us control over HOW data reaches reducers.

But what if we DON'T NEED reducers at all?

PROJECT 11 asks: "We have messy hospital records with
missing fields, invalid dates, and duplicate entries.
We just need to CLEAN them — no aggregation, no grouping."

This introduces:
→ Map-Only Jobs (zero reducers)
→ NullWritable (when you don't need a key or value)
→ Data cleansing as an ETL pattern
→ Why skipping the reducer saves MASSIVE time
   (no shuffle = no network transfer = no sort overhead)

The insight: Not every problem needs Map AND Reduce.
Sometimes Map alone is the entire solution.
