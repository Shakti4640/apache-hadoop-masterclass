PROJECT 33: PROCESSING 1TB OF DATA — DIAGNOSING AND FIXING SKEW

🔴 HYPOTHETICAL PROBLEM

SCENARIO:
→ You are the performance engineer at a social media analytics company
→ Your pipeline processes 1TB of user activity logs daily
→ The job: count interactions (likes, shares, comments) per user
→ Expected runtime: ~25 minutes on your 50-node cluster

WHAT ACTUALLY HAPPENS:
→ 199 out of 200 reducers finish in 12 minutes
→ 1 reducer takes 3 HOURS 47 MINUTES
→ Total job time: 3 hours 47 minutes (bottlenecked by ONE reducer)
→ Cluster sits 98% idle for 3.5 hours waiting for that one task

THE ROOT CAUSE:
→ Your platform has celebrity accounts
→ @PopStar has 200 million followers → 48 million interactions/day
→ @RegularUser has 300 followers → 12 interactions/day
→ ALL 48 million @PopStar records go to ONE reducer
  (because same key = same reducer, as learned in Project 10)

→ How do you detect that skew IS the problem (not just slow hardware)?

→ How do you ensure @PopStar's 48M records get processed
  in parallel across MULTIPLE reducers, then combined correctly?

🔍 HOW TO APPROACH IT

STEP 1: Confirm skew is the actual problem (not guessing)

  SYMPTOM: "One task takes much longer than others"
  
  BUT THIS COULD ALSO BE:
  → Slow hardware on one node (bad disk, network issues)
  → Speculative execution fighting a slow but correct task
  → Memory pressure causing garbage collection pauses
  → Uneven input splits (not skew — different problem)
  
  DEFINITIVE DIAGNOSIS requires looking at MULTIPLE metrics:

  ┌──────────────────────────────────────────────────────────┐
  │              SKEW DIAGNOSIS CHECKLIST                      │
  ├──────────────────────────────────────────────────────────┤
  │                                                           │
  │  1. TASK RUNTIME DISTRIBUTION                             │
  │     → Look at ALL reducer runtimes                        │
  │     → If most finish in T and few take 10T+ → SKEW       │
  │     → If all take similar time → NOT skew                 │
  │                                                           │
  │  2. RECORDS PER REDUCER                                   │
  │     → Check "Reduce input records" counter per task       │
  │     → If one reducer gets 1000x more records → SKEW      │
  │     → If roughly equal → NOT skew                         │
  │                                                           │
  │  3. BYTES READ PER REDUCER                                │
  │     → Check "Reduce shuffle bytes" per task               │
  │     → Massive variance → SKEW                             │
  │                                                           │
  │  4. IS THE SLOW TASK ON A SLOW MACHINE?                   │
  │     → Run same task on different node (speculative exec)  │

💡 HOW CAN WE SOLVE IT

═══════════════════════════════════════════════════════════════
SOLUTION 1: COMBINER (First line of defense — from Project 9)
═══════════════════════════════════════════════════════════════

  BEFORE combiner:
    Mapper 1 emits: (@PopStar, 1) × 2,000,000 times
    Mapper 2 emits: (@PopStar, 1) × 1,500,000 times
    ...
    ALL 48,000,000 records shuffle to ONE reducer

  AFTER combiner:
    Mapper 1 combines locally: (@PopStar, 2000000)  ← ONE record!
    Mapper 2 combines locally: (@PopStar, 1500000)  ← ONE record!
    ...
    Only ~200 records shuffle to reducer (one per mapper)
    Reducer sums: 2000000 + 1500000 + ... = 48,000,000

  EFFECTIVENESS:
  → EXCELLENT for SUM, COUNT (fully associative + commutative)
  → GOOD for MIN, MAX
  → USELESS for MEDIAN, DISTINCT (cannot combine partial results)
  → REDUCES shuffle volume dramatically
  → May COMPLETELY solve skew for simple aggregations

  LIMITATION:
  → Combiner runs on MAP side — reduces shuffle
  → But if reducer still needs complex processing per key → still skewed
  → For our COUNT scenario: combiner MIGHT be sufficient
  → For complex scenarios: need salting

═══════════════════════════════════════════════════════════════
SOLUTION 2: KEY SALTING (The universal skew fix)
═══════════════════════════════════════════════════════════════

  CONCEPT: Split a hot key into N sub-keys using a random salt
  
  BEFORE SALTING:
    Key: @PopStar → ALL 48M records → Reducer 7 (alone)
  
  AFTER SALTING (salt range: 0-9):
    Key: @PopStar_0 → ~4.8M records → Reducer 2
    Key: @PopStar_1 → ~4.8M records → Reducer 5
    Key: @PopStar_2 → ~4.8M records → Reducer 8
    ...
    Key: @PopStar_9 → ~4.8M records → Reducer 15
    
    Now 10 reducers share the load!
    Each processes ~4.8M instead of one processing 48M.

  THE CATCH: 
  → After salting, results are PARTIAL
  → @PopStar_0 count = 4,800,000
  → @PopStar_1 count = 4,799,500
  → We need a SECOND MapReduce job to combine:
    → Strip salt: @PopStar_0 → @PopStar
    → Sum partials: 4,800,000 + 4,799,500 + ... = 48,000,000

  TWO-PASS APPROACH:

  ┌──────────────────────────────────────────────────────┐
  │                                                       │
  │  PASS 1: Salted Aggregation                           │
  │  ┌─────────┐   ┌─────────────┐   ┌───────────────┐  │
  │  │ Mapper  │──→│ Combiner    │──→│ Reducer       │  │
  │  │ Add salt│   │ Local sum   │   │ Sum per       │  │
  │  │ to key  │   │ per salted  │   │ salted key    │  │
  │  │         │   │ key         │   │               │  │
  │  └─────────┘   └─────────────┘   └───────┬───────┘  │
  │                                           │          │
  │  Output: @PopStar_0  4800000              │          │
  │          @PopStar_1  4799500              │          │
  │          @RegUser_0  12                   │          │
  │                                           ▼          │
  │  PASS 2: De-salt and Final Aggregation                │
  │  ┌─────────┐                    ┌───────────────┐    │
  │  │ Mapper  │───────────────────→│ Reducer       │    │
  │  │ Strip   │                    │ Sum partials  │    │
  │  │ salt    │                    │ per real key  │    │
  │  │ suffix  │                    │               │    │
  │  └─────────┘                    └───────────────┘    │
  │                                                       │
  │  Output: @PopStar    48000000                         │
  │          @RegUser    12                               │
  │                                                       │

SOLUTION 3: ADAPTIVE SALTING (Salt only hot keys)
═══════════════════════════════════════════════════════════════

  PROBLEM WITH UNIFORM SALTING:
  → @RegularUser has 12 records → split into 10 salted keys
  → Each salted key has 1-2 records → massive overhead
  → 50 million users × 10 salts = 500 million keys
  → Second pass must process 500M keys instead of 50M

  BETTER: Only salt keys that NEED it

  HOW:
  → Pre-analyze: find keys with frequency > threshold
  → Store hot keys in a file → load via Distributed Cache (Project 15)
  → Mapper checks: is this key hot?
    → YES: add random salt
    → NO: emit as-is (no salt needed)
  → Second pass only needs to de-salt the hot keys

  THRESHOLD CALCULATION:
    total_records = 5,000,000,000
    num_reducers = 200
    ideal_per_reducer = 25,000,000
    threshold = ideal_per_reducer × 3 = 75,000,000
    → Salt any key with > 75M records
    → In practice: top ~500 users get salted
    → Other 49,999,500 users processed normally

✅ WHAT SHOULD DO

1. ALWAYS DIAGNOSE BEFORE FIXING
   → Run the diagnostic job FIRST
   → Measure actual key distribution
   → If top key is only 2x median → skew is mild → combiner may suffice
   → If top key is 1000x median → skew is severe → salting required

2. TRY COMBINER FIRST (Project 9)
   → Simplest fix — one line of code: job.setCombinerClass(...)
   → For SUM/COUNT: often reduces skew enough
   → If insufficient: then add salting

3. CHOOSE SALT FACTOR BASED ON DATA
   → salt_factor = hot_key_count / ideal_per_reducer
   → @PopStar: 48M records, ideal 5M per reducer → salt_factor = 10
   → Too few salts: still skewed
   → Too many salts: overhead in second pass
   → RULE OF THUMB: salt_factor = max(1, key_count / (2 × median_count))

4. USE DISTRIBUTED CACHE FOR HOT KEY LIST (Project 15)
   → Don't hardcode hot keys in mapper code
   → Hot keys change over time (new celebrities emerge)
   → Load from file at mapper startup → easy to update

5. MONITOR WITH CUSTOM COUNTERS (Project 7)
   → Count salted vs. non-salted records
   → Track per-reducer record counts in second pass
   → Verify skew is actually resolved

6. CHAIN JOBS PROPERLY (Project 21 pattern, or Project 31 Oozie)
   → Phase 0 → Phase 1 → Phase 2 must execute in order
   → Use ControlledJob or Oozie workflow for dependency management

🚫 WHAT SHOULD NOT DO

1. DO NOT SALT ALL KEYS UNIFORMLY
   → 50M users × 10 salts = 500M intermediate keys
   → Second pass processes 10x more data than necessary
   → Only salt keys that exceed the threshold

2. DO NOT USE TOO MANY REDUCERS AS A "FIX"
   → Increasing from 200 to 2000 reducers does NOT fix skew
   → @PopStar still maps to ONE reducer (hash mod N)
   → You just get 1999 fast reducers and 1 slow one
   → More reducers = more overhead, same bottleneck

3. DO NOT INCREASE REDUCER MEMORY AS THE ONLY FIX
   → Giving the slow reducer 64GB RAM helps with OOM
   → But it still takes 3+ hours to process 48M records
   → You need PARALLELISM, not just more resources per task

4. DO NOT IGNORE MAP-SIDE SKEW
   → If one input file is 500GB and others are 1GB
   → That's INPUT skew, not reduce skew
   → Fix: ensure uniform split sizes (CombineFileInputFormat)

5. DO NOT SALT JOIN KEYS WITHOUT MATCHING BOTH SIDES
   → If salting for a join (Project 17 pattern):
     → Salt the large table's keys randomly
     → REPLICATE the small table's rows for each salt value
     → Otherwise: salted keys won't match unsalted keys = data loss

6. DO NOT ASSUME SKEW IS CONSTANT
   → Today's hot key may be cold tomorrow (viral content)
   → Run diagnostic job periodically
   → Or use adaptive salting that adjusts at runtime

🚫 WHEN NOT TO USE

1. WHEN DATA IS NATURALLY UNIFORM
   → If key distribution is roughly even → salting adds overhead for no gain
   → Run diagnostic first — don't assume skew exists

2. WHEN COMBINER ALREADY SOLVES IT
   → If combiner reduces 48M records to 200 records per key
   → Reducer processes 200 records in milliseconds → no skew problem
   → Adding salting on top is unnecessary complexity

3. WHEN THE OPERATION IS NOT PARALLELIZABLE
   → Computing exact global median requires ALL data in order
   → Cannot salt and combine partial medians correctly
   → Need different approach: sampling or approximate algorithms

4. WHEN SKEW IS IN MAP PHASE (not reduce)
   → Salting keys fixes REDUCE skew only
   → If one mapper processes a 500GB file → that's input split skew
   → Fix: CombineFileInputFormat or custom split logic (Project 32)

5. WHEN MIGRATING TO SPARK IS FEASIBLE
   → Spark has built-in skew hints (spark.sql.adaptive.skewJoin)
   → Project 48 covers Spark migration
   → If you're starting fresh → consider Spark over MapReduce for skewed data

⚙️ HOW IT ONLY WORKS — INTERNAL MECHANICS

═══════════════════════════════════════════════════════════════
MECHANIC 1: WHY SKEW HAPPENS — THE PARTITIONER MATH
═══════════════════════════════════════════════════════════════

  Default HashPartitioner (from Project 10):
  
    partition = (key.hashCode() & Integer.MAX_VALUE) % numReducers
  
  For @PopStar:
    hashCode("@PopStar") = 1234567 (example)
    partition = (1234567 & 0x7FFFFFFF) % 200 = 167
    
    ALL 48,000,000 @PopStar records → Reducer 167
    
  For @RegularUser:
    hashCode("@RegularUser") = 9876543
    partition = (9876543 & 0x7FFFFFFF) % 200 = 143
    
    ALL 12 @RegularUser records → Reducer 143
    
  THE PARTITIONER IS DOING ITS JOB CORRECTLY
  → Same key MUST go to same reducer (correctness guarantee)
  → The problem is not the partitioner — it's the DATA

═══════════════════════════════════════════════════════════════
MECHANIC 2: HOW SALTING BREAKS THE SINGLE-REDUCER CONSTRAINT
═══════════════════════════════════════════════════════════════

  MAPPER (with salting):
  
    Input record: "@PopStar liked photo_123"
    
    salt = random.nextInt(10)  → e.g., 7
    salted_key = "@PopStar" + "_" + "7" = "@PopStar_7"
    
    emit("@PopStar_7", 1)
    
  PARTITIONER now sees "@PopStar_7":
    hashCode("@PopStar_7") = 5551234  (DIFFERENT from "@PopStar")
    partition = (5551234 & 0x7FFFFFFF) % 200 = 34
    → Goes to Reducer 34 (NOT Reducer 167!)
    
  Different salts produce different hash codes:
    "@PopStar_0" → Reducer 89
    "@PopStar_1" → Reducer 12
    "@PopStar_2" → Reducer 156
    ...
    "@PopStar_9" → Reducer 34
    
  48M records now SPREAD across ~10 reducers
  Each processes ~4.8M instead of one processing 48M
  
  TIME IMPACT:
    Before: 1 reducer × 48M records = 3.8 hours
    After:  10 reducers × 4.8M each = ~23 minutes (parallel)

═══════════════════════════════════════════════════════════════
MECHANIC 3: WHY TWO PASSES ARE NEEDED
═══════════════════════════════════════════════════════════════

  After Pass 1:
    Reducer 89 output:  @PopStar_0    4,812,345
    Reducer 12 output:  @PopStar_1    4,798,001
    Reducer 156 output: @PopStar_2    4,803,777
    ...
    
  These are PARTIAL results — not the final answer.
  The business wants: @PopStar    48,000,000
  
  Pass 2 mapper strips the salt:
    Input: "@PopStar_0    4812345"
    Strip: key = "@PopStar_0".substring(0, lastIndexOf("_")) = "@PopStar"
    Emit: ("@PopStar", 4812345)
    
  Pass 2 reducer sums partials:
    @PopStar: [4812345, 4798001, 4803777, ...] → SUM = 48,000,000
    
  CRITICAL: Pass 2 has NO skew because:
  → Each hot key has only 10 partial records (one per salt)
  → 500 hot keys × 10 partials = 5,000 records
  → Plus 49.5M cold keys with 1 record each
  → Distribution is nearly uniform → no skew

⚠️ COMMON PITFALLS AND FAILURE MODES

PITFALL 1: "Salting made the job SLOWER"
→ CAUSE: Salted ALL 50M keys, not just hot ones
→ Pass 2 now processes 500M keys instead of 50M
→ Overhead of second job exceeds skew savings
→ SOLUTION: Adaptive salting — only salt keys above threshold

PITFALL 2: "Results are wrong after salting — counts are too low"
→ CAUSE: Pass 2 reducer uses MAX instead of SUM
→ Or: de-salt logic strips wrong characters from key
→ Or: some records lost between Pass 1 and Pass 2
→ SOLUTION: Validate — sum of Pass 1 output = sum of Pass 2 output

PITFALL 3: "Hot key list is stale — new celebrity went viral yesterday"
→ CAUSE: Hot keys identified last month, not updated
→ New viral user creates skew that adaptive salting doesn't catch
→ SOLUTION: Run diagnostic job daily/weekly before main pipeline
→ Or: use uniform salting (less efficient but always catches new hot keys)

PITFALL 4: "Combiner didn't help at all"
→ CAUSE: Combiner only runs when there's spill to disk
→ If mapper output fits in memory buffer (io.sort.mb) → no combiner
→ SOLUTION: Reduce io.sort.mb to force spills → combiner runs
→ Or: combiner may run but key is still unique per record (JOIN case)

PITFALL 5: "De-salt in Pass 2 strips wrong characters"
→ CAUSE: Key naturally contains underscore + number
→ "user_42" → strip → "user" (WRONG — "42" is part of the key, not salt)
→ SOLUTION: Use a delimiter that never appears in keys
→ Better: use a structured salt format: key + "\t" + salt
→ Or: prefix salt: "7_@PopStar" instead of "@PopStar_7"

PITFALL 6: "Skew returned after fixing — different key this time"
→ CAUSE: Viral content is unpredictable
→ SOLUTION: Build skew detection INTO the pipeline (not as afterthought)
→ Run Phase 0 (diagnostic) every time before Phase 1
→ Dynamically generate hot key list

import org.apache.hadoop.io.LongWritable;

import org.apache.hadoop.mapreduce.Mapper;

import java.io.IOException;
import java.util.Random;

public class SkewDiagnosticMapper 
        extends Mapper<LongWritable, Text, Text, LongWritable> {

    private Text outKey = new Text();
    private static final LongWritable ONE = new LongWritable(1);
    
    // Sampling rate: process 1 in every N records
    // Reduces diagnostic job time from hours to minutes
    private int sampleRate;
    private Random random = new Random();
    
    enum DiagCounters {
        TOTAL_INPUT_RECORDS,
        SAMPLED_RECORDS,
        PARSE_ERRORS
    }

    @Override
    protected void setup(Context context) throws IOException {
        // Configurable sample rate — default 100 means sample 1%
        sampleRate = context.getConfiguration().getInt("skew.diag.sample.rate", 100);
    }

    @Override
    protected void map(LongWritable key, Text value, Context context) 
            throws IOException, InterruptedException {
        
        context.getCounter(DiagCounters.TOTAL_INPUT_RECORDS).increment(1);
        
        // ── Sampling: skip most records for speed ──
        if (random.nextInt(sampleRate) != 0) {
            return;
        }
        
        context.getCounter(DiagCounters.SAMPLED_RECORDS).increment(1);
        
        // ── Parse user from activity log ──
        // Format: "timestamp\tuser_id\taction\ttarget"
        String line = value.toString();
        String[] parts = line.split("\t");
        
        if (parts.length < 2) {
            context.getCounter(DiagCounters.PARSE_ERRORS).increment(1);
            return;
        }
        
        String userId = parts[1].trim();
        outKey.set(userId);
        context.write(outKey, ONE);
    }
}

import org.apache.hadoop.io.LongWritable;
import org.apache.hadoop.io.Text;
import org.apache.hadoop.mapreduce.Reducer;

import java.io.IOException;

public class SkewDiagnosticReducer 
        extends Reducer<Text, LongWritable, Text, LongWritable> {

    private LongWritable outValue = new LongWritable();
    private int sampleRate;
    
    enum DiagReduceCounters {
        UNIQUE_KEYS,
        HOT_KEYS_FOUND,
        TOTAL_ESTIMATED_RECORDS
    }

    @Override
    protected void setup(Context context) throws IOException {
        sampleRate = context.getConfiguration().getInt("skew.diag.sample.rate", 100);
    }

    @Override
    protected void reduce(Text key, Iterable<LongWritable> values, Context context) 
            throws IOException, InterruptedException {
        
        long sampledCount = 0;
        for (LongWritable val : values) {
            sampledCount += val.get();
        }
        
        // Estimate true count from sample
        long estimatedCount = sampledCount * sampleRate;
        
        context.getCounter(DiagReduceCounters.UNIQUE_KEYS).increment(1);
        context.getCounter(DiagReduceCounters.TOTAL_ESTIMATED_RECORDS)
               .increment(estimatedCount);
        
        // Threshold: keys with estimated count > 10M are "hot"
        long threshold = context.getConfiguration()
                                .getLong("skew.diag.hot.threshold", 10000000);
        if (estimatedCount > threshold) {
            context.getCounter(DiagReduceCounters.HOT_KEYS_FOUND).increment(1);
        }
        
        outValue.set(estimatedCount);
        context.write(key, outValue);
    }
}

package com.social.skew;

import org.apache.hadoop.conf.Configuration;
import org.apache.hadoop.fs.*;
import java.io.*;
import java.util.*;

public class HotKeyExtractor {

    /**
     * Read diagnostic output and create hot_keys.txt
     * Format per line: key\tsalt_factor
     */
    public static int extractHotKeys(
            Configuration conf, 
            String diagnosticOutputPath,
            String hotKeysOutputPath,
            long hotThreshold,
            long targetPerReducer) throws IOException {
        
        FileSystem fs = FileSystem.get(conf);
        Map<String, Integer> hotKeys = new TreeMap<>();
        
        // ── Read all part files from diagnostic output ──
        FileStatus[] files = fs.listStatus(new Path(diagnosticOutputPath), 
            path -> path.getName().startsWith("part-r-"));
        
        long totalRecords = 0;
        long maxCount = 0;
        
        for (FileStatus file : files) {
            BufferedReader reader = new BufferedReader(
                new InputStreamReader(fs.open(file.getPath())));
            
            String line;
            while ((line = reader.readLine()) != null) {
                String[] parts = line.split("\t");
                if (parts.length != 2) continue;
                
                String key = parts[0];
                long count = Long.parseLong(parts[1]);
                totalRecords += count;
                maxCount = Math.max(maxCount, count);
                
                if (count > hotThreshold) {
                    // Calculate salt factor for this key
                    int saltFactor = (int) Math.ceil((double) count / targetPerReducer);
                    saltFactor = Math.max(2, Math.min(saltFactor, 100));
                    hotKeys.put(key, saltFactor);
                }
            }
            reader.close();
        }
        
        // ── Write hot keys file ──
        Path hotKeysPath = new Path(hotKeysOutputPath);
        FSDataOutputStream out = fs.create(hotKeysPath, true);
        BufferedWriter writer = new BufferedWriter(new OutputStreamWriter(out));
        
        for (Map.Entry<String, Integer> entry : hotKeys.entrySet()) {
            writer.write(entry.getKey() + "\t" + entry.getValue());
            writer.newLine();
        }
        writer.close();
        
        // ── Print summary ──
        System.out.println("\n  ═══ HOT KEY ANALYSIS ═══");
        System.out.printf("  Total estimated records:  %,d%n", totalRecords);
        System.out.printf("  Max key count:            %,d%n", maxCount);
        System.out.printf("  Hot threshold:            %,d%n", hotThreshold);
        System.out.printf("  Hot keys found:           %d%n", hotKeys.size());
        System.out.printf("  Hot keys file:            %s%n", hotKeysOutputPath);
        
        if (!hotKeys.isEmpty()) {
            System.out.println("\n  Top hot keys:");
            hotKeys.entrySet().stream()
                .sorted((a, b) -> b.getValue() - a.getValue())
                .limit(10)
                .forEach(e -> System.out.printf("    %-30s salt_factor=%d%n", 
                    e.getKey(), e.getValue()));
        }
        
        return hotKeys.size();
    }
}

package com.social.skew;

import org.apache.hadoop.io.LongWritable;
import org.apache.hadoop.io.Text;
import org.apache.hadoop.mapreduce.Mapper;

import java.io.*;
import java.net.URI;
import java.util.HashMap;
import java.util.Map;
import java.util.Random;

public class SaltedAggregationMapper 
        extends Mapper<LongWritable, Text, Text, LongWritable> {

    private Text outKey = new Text();
    private static final LongWritable ONE = new LongWritable(1);
    private Random random = new Random();
    
    // Hot key lookup: key → salt_factor
    // Loaded from Distributed Cache in setup()
    private Map<String, Integer> hotKeys = new HashMap<>();
    
    enum SaltCounters {
        TOTAL_RECORDS,
        RECORDS_SALTED,
        RECORDS_NOT_SALTED,
        HOT_KEYS_LOADED,
        PARSE_ERRORS
    }

    @Override
    protected void setup(Context context) throws IOException {
        // ── Load hot keys from Distributed Cache (Project 15 pattern) ──
        URI[] cacheFiles = context.getCacheFiles();
        
        if (cacheFiles != null && cacheFiles.length > 0) {
            // The file is in the local working directory of the task
            // when added via addCacheFile
            Path localPath = new Path(cacheFiles[0].getPath());
            String fileName = localPath.getName();
            
            BufferedReader reader = new BufferedReader(
                new FileReader(fileName));
            
            String line;
            while ((line = reader.readLine()) != null) {
                String[] parts = line.split("\t");
                if (parts.length == 2) {
                    hotKeys.put(parts[0], Integer.parseInt(parts[1]));
                }
            }
            reader.close();
        }
        
        context.getCounter(SaltCounters.HOT_KEYS_LOADED)
               .increment(hotKeys.size());
    }

    @Override
    protected void map(LongWritable key, Text value, Context context) 
            throws IOException, InterruptedException {
        
        context.getCounter(SaltCounters.TOTAL_RECORDS).increment(1);
        
        // ── Parse user from activity log ──
        String line = value.toString();
        String[] parts = line.split("\t");
        
        if (parts.length < 2) {
            context.getCounter(SaltCounters.PARSE_ERRORS).increment(1);
            return;
        }
        
        String userId = parts[1].trim();
        
        // ── Adaptive salting decision ──
        if (hotKeys.containsKey(userId)) {
            // HOT KEY: Add random salt to distribute across reducers
            int saltFactor = hotKeys.get(userId);
            int salt = random.nextInt(saltFactor);
            
            // Use a delimiter that won't appear in user IDs
            // §§ is safe — no user ID contains this
            outKey.set(userId + "§§" + salt);
            context.getCounter(SaltCounters.RECORDS_SALTED).increment(1);
        } else {
            // COLD KEY: No salt needed — small number of records
            outKey.set(userId);
            context.getCounter(SaltCounters.RECORDS_NOT_SALTED).increment(1);
        }
        
        context.write(outKey, ONE);
    }
}

package com.social.skew;

import org.apache.hadoop.io.LongWritable;
import org.apache.hadoop.io.Text;
import org.apache.hadoop.mapreduce.Mapper;

import java.io.IOException;

public class DesaltMapper 
        extends Mapper<LongWritable, Text, Text, LongWritable> {

    private Text outKey = new Text();
    private LongWritable outValue = new LongWritable();
    
    // Salt delimiter — must match what SaltedAggregationMapper used
    private static final String SALT_DELIMITER = "§§";
    
    enum DesaltCounters {
        RECORDS_DESALTED,
        RECORDS_ALREADY_CLEAN,
        TOTAL_RECORDS
    }

    @Override
    protected void map(LongWritable key, Text value, Context context) 
            throws IOException, InterruptedException {
        
        context.getCounter(DesaltCounters.TOTAL_RECORDS).increment(1);
        
        // ── Parse Phase 1 output: "key\tcount" ──
        String line = value.toString();
        String[] parts = line.split("\t");
        
        if (parts.length != 2) return;
        
        String saltedKey = parts[0];
        long count = Long.parseLong(parts[1]);
        
        // ── Strip salt if present ──
        int delimIndex = saltedKey.lastIndexOf(SALT_DELIMITER);
        
        if (delimIndex > 0) {
            // Verify the suffix after delimiter is a number (actual salt)
            String suffix = saltedKey.substring(delimIndex + SALT_DELIMITER.length());
            try {
                Integer.parseInt(suffix);
                // Valid salt — strip it
                String originalKey = saltedKey.substring(0, delimIndex);
                outKey.set(originalKey);
                context.getCounter(DesaltCounters.RECORDS_DESALTED).increment(1);
            } catch (NumberFormatException e) {
                // Not a salt suffix — key naturally contains delimiter
                outKey.set(saltedKey);
                context.getCounter(DesaltCounters.RECORDS_ALREADY_CLEAN).increment(1);
            }
        } else {
            // No salt — cold key passed through unchanged
            outKey.set(saltedKey);
            context.getCounter(DesaltCounters.RECORDS_ALREADY_CLEAN).increment(1);
        }
        
        outValue.set(count);
        context.write(outKey, outValue);
    }
}

package com.social.skew;

import org.apache.hadoop.conf.Configuration;
import org.apache.hadoop.conf.Configured;
import org.apache.hadoop.fs.FileSystem;
import org.apache.hadoop.fs.Path;
import org.apache.hadoop.io.LongWritable;
import org.apache.hadoop.io.Text;
import org.apache.hadoop.mapreduce.Job;
import org.apache.hadoop.mapreduce.lib.input.FileInputFormat;
import org.apache.hadoop.mapreduce.lib.input.TextInputFormat;
import org.apache.hadoop.mapreduce.lib.output.FileOutputFormat;
import org.apache.hadoop.mapreduce.lib.output.TextOutputFormat;
import org.apache.hadoop.util.Tool;
import org.apache.hadoop.util.ToolRunner;

import java.net.URI;

public class SkewFixDriver extends Configured implements Tool {

    // ── Path constants ──
    private static final String DIAG_OUTPUT = "/tmp/skew_pipeline/diagnostic";
    private static final String HOT_KEYS_FILE = "/tmp/skew_pipeline/hot_keys.txt";
    private static final String PHASE1_OUTPUT = "/tmp/skew_pipeline/salted_aggregation";
    
    @Override
    public int run(String[] args) throws Exception {
        
        if (args.length < 2) {
            System.err.println("Usage: SkewFixDriver <input-path> <output-path> [num-reducers]");
            System.err.println("  input-path:   HDFS path to activity logs");
            System.err.println("  output-path:  HDFS path for final results");
            System.err.println("  num-reducers: Number of reducers (default: 200)");
            return 1;
        }
        
        String inputPath = args[0];
        String finalOutputPath = args[1];
        int numReducers = args.length > 2 ? Integer.parseInt(args[2]) : 200;
        
        Configuration conf = getConf();
        FileSystem fs = FileSystem.get(conf);
        
        // ── Clean up intermediate paths ──
        fs.delete(new Path(DIAG_OUTPUT), true);
        fs.delete(new Path(HOT_KEYS_FILE), true);
        fs.delete(new Path(PHASE1_OUTPUT), true);
        fs.delete(new Path(finalOutputPath), true);
        
        long pipelineStart = System.currentTimeMillis();
        
        System.out.println("═══════════════════════════════════════════════════════");
        System.out.println("  SKEW-FIX PIPELINE: THREE-PHASE EXECUTION");
        System.out.println("═══════════════════════════════════════════════════════");
        
        // ══════════════════════════════════════════════════════
        // PHASE 0: DIAGNOSTIC — Detect hot keys
        // ══════════════════════════════════════════════════════
        
        System.out.println("\n  ┌─────────────────────────────────────────────┐");
        System.out.println("  │  PHASE 0: SKEW DIAGNOSTIC (Sampling 1%)    │");
        System.out.println("  └─────────────────────────────────────────────┘\n");
        
        long phase0Start = System.currentTimeMillis();
        
        Job diagJob = Job.getInstance(conf, "Phase 0: Skew Diagnostic");
        diagJob.setJarByClass(SkewFixDriver.class);
        
        // Sampling rate: 100 means process 1 in 100 records
        diagJob.getConfiguration().setInt("skew.diag.sample.rate", 100);
        diagJob.getConfiguration().setLong("skew.diag.hot.threshold", 10000000);
        
        diagJob.setMapperClass(SkewDiagnosticMapper.class);
        diagJob.setCombinerClass(SkewDiagnosticCombiner.class);
        diagJob.setReducerClass(SkewDiagnosticReducer.class);
        diagJob.setNumReduceTasks(10);  // Small — just counting
        
        diagJob.setOutputKeyClass(Text.class);
        diagJob.setOutputValueClass(LongWritable.class);
        
        FileInputFormat.addInputPath(diagJob, new Path(inputPath));
        FileOutputFormat.setOutputPath(diagJob, new Path(DIAG_OUTPUT));
        
        if (!diagJob.waitForCompletion(true)) {
            System.err.println("  🔴 Phase 0 FAILED");
            return 1;
        }
        
        long phase0Time = (System.currentTimeMillis() - phase0Start) / 1000;
        System.out.printf("  ✅ Phase 0 completed in %d seconds%n", phase0Time);
        
        // ── Extract hot keys from diagnostic output ──
        System.out.println("\n  Analyzing key distribution...");
        
        long totalRecords = diagJob.getCounters()
            .findCounter(SkewDiagnosticReducer.DiagReduceCounters.TOTAL_ESTIMATED_RECORDS)
            .getValue();
        long targetPerReducer = totalRecords / numReducers;
        long hotThreshold = targetPerReducer * 3;
        
        System.out.printf("  Total estimated records: %,d%n", totalRecords);
        System.out.printf("  Target per reducer:      %,d%n", targetPerReducer);
        System.out.printf("  Hot key threshold:       %,d%n", hotThreshold);
        
        int hotKeyCount = HotKeyExtractor.extractHotKeys(
            conf, DIAG_OUTPUT, HOT_KEYS_FILE, hotThreshold, targetPerReducer);
        
        if (hotKeyCount == 0) {
            System.out.println("\n  ⚠️  No hot keys detected — running standard aggregation");
            return runStandardAggregation(conf, inputPath, finalOutputPath, numReducers);
        }
        
        // ══════════════════════════════════════════════════════
        // PHASE 1: SALTED AGGREGATION
        // ══════════════════════════════════════════════════════
        
        System.out.println("\n  ┌─────────────────────────────────────────────┐");
        System.out.println("  │  PHASE 1: SALTED AGGREGATION                │");
        System.out.println("  └─────────────────────────────────────────────┘\n");
        
        long phase1Start = System.currentTimeMillis();
        
        Job saltedJob = Job.getInstance(conf, "Phase 1: Salted Aggregation");
        saltedJob.setJarByClass(SkewFixDriver.class);
        
        // ── Add hot keys file to Distributed Cache (Project 15) ──
        saltedJob.addCacheFile(new URI(HOT_KEYS_FILE));
        
        saltedJob.setMapperClass(SaltedAggregationMapper.class);
        saltedJob.setCombinerClass(SaltedAggregationCombiner.class);
        saltedJob.setReducerClass(SaltedAggregationReducer.class);
        saltedJob.setNumReduceTasks(numReducers);
        
        saltedJob.setOutputKeyClass(Text.class);
        saltedJob.setOutputValueClass(LongWritable.class);
        
        FileInputFormat.addInputPath(saltedJob, new Path(inputPath));
        FileOutputFormat.setOutputPath(saltedJob, new Path(PHASE1_OUTPUT));
        
        if (!saltedJob.waitForCompletion(true)) {
            System.err.println("  🔴 Phase 1 FAILED");
            return 1;
        }
        
        long phase1Time = (System.currentTimeMillis() - phase1Start) / 1000;
        System.out.printf("  ✅ Phase 1 completed in %d seconds%n", phase1Time);
        
        // ── Print Phase 1 skew metrics ──
        long saltedRecords = saltedJob.getCounters()
            .findCounter(SaltedAggregationMapper.SaltCounters.RECORDS_SALTED)
            .getValue();
        long unsaltedRecords = saltedJob.getCounters()
            .findCounter(SaltedAggregationMapper.SaltCounters.RECORDS_NOT_SALTED)
            .getValue();
        long maxPerKey = saltedJob.getCounters()
            .findCounter(SaltedAggregationReducer.ReduceCounters.MAX_RECORDS_PER_KEY)
            .getValue();
        
        System.out.printf("  Records salted:     %,d (%.1f%%)%n", 
            saltedRecords, 
            100.0 * saltedRecords / (saltedRecords + unsaltedRecords));
        System.out.printf("  Records not salted: %,d%n", unsaltedRecords);
        System.out.printf("  Max count per salted key: %,d%n", maxPerKey);
        
        // ══════════════════════════════════════════════════════
        // PHASE 2: DE-SALT AND FINAL COMBINE
        // ══════════════════════════════════════════════════════
        
        System.out.println("\n  ┌─────────────────────────────────────────────┐");
        System.out.println("  │  PHASE 2: DE-SALT AND FINAL COMBINE         │");
        System.out.println("  └─────────────────────────────────────────────┘\n");
        
        long phase2Start = System.currentTimeMillis();
        
        Job desaltJob = Job.getInstance(conf, "Phase 2: De-salt and Final Combine");
        desaltJob.setJarByClass(SkewFixDriver.class);
        
        desaltJob.setInputFormatClass(TextInputFormat.class);
        desaltJob.setMapperClass(DesaltMapper.class);
        desaltJob.setCombinerClass(SaltedAggregationCombiner.class);
        desaltJob.setReducerClass(DesaltReducer.class);
        
        // Fewer reducers — Phase 2 data is MUCH smaller
        int phase2Reducers = Math.max(10, numReducers / 10);
        desaltJob.setNumReduceTasks(phase2Reducers);
        
        desaltJob.setOutputKeyClass(Text.class);
        desaltJob.setOutputValueClass(LongWritable.class);
        
        FileInputFormat.addInputPath(desaltJob, new Path(PHASE1_OUTPUT));
        FileOutputFormat.setOutputPath(desaltJob, new Path(finalOutputPath));
        
        if (!desaltJob.waitForCompletion(true)) {
            System.err.println("  🔴 Phase 2 FAILED");
            return 1;
        }
        
        long phase2Time = (System.currentTimeMillis() - phase2Start) / 1000;
        System.out.printf("  ✅ Phase 2 completed in %d seconds%n", phase2Time);
        
        // ══════════════════════════════════════════════════════
        // FINAL REPORT
        // ══════════════════════════════════════════════════════
        
        long totalTime = (System.currentTimeMillis() - pipelineStart) / 1000;
        
        long uniqueUsers = desaltJob.getCounters()
            .findCounter(DesaltReducer.FinalCounters.UNIQUE_USERS_FINAL)
            .getValue();
        long totalInteractions = desaltJob.getCounters()
            .findCounter(DesaltReducer.FinalCounters.TOTAL_INTERACTIONS_FINAL)
            .getValue();
        long maxUserInteractions = desaltJob.getCounters()
            .findCounter(DesaltReducer.FinalCounters.MAX_USER_INTERACTIONS)
            .getValue();
        
        System.out.println("\n═══════════════════════════════════════════════════════");
        System.out.println("  PIPELINE COMPLETE — FINAL REPORT");
        System.out.println("═══════════════════════════════════════════════════════");
        System.out.printf("  Phase 0 (Diagnostic):  %d seconds%n", phase0Time);
        System.out.printf("  Phase 1 (Salted Agg):  %d seconds%n", phase1Time);
        System.out.printf("  Phase 2 (De-salt):     %d seconds%n", phase2Time);
        System.out.printf("  Total pipeline time:   %d seconds (%.1f minutes)%n", 
            totalTime, totalTime / 60.0);
        System.out.println("  ─────────────────────────────────────────────────────");
        System.out.printf("  Unique users:          %,d%n", uniqueUsers);
        System.out.printf("  Total interactions:    %,d%n", totalInteractions);
        System.out.printf("  Max per user:          %,d%n", maxUserInteractions);
        System.out.printf("  Hot keys salted:       %d%n", hotKeyCount);
        System.out.printf("  Output path:           %s%n", finalOutputPath);
        System.out.println("═══════════════════════════════════════════════════════");
        
        // ── Cleanup intermediate data ──
        fs.delete(new Path(DIAG_OUTPUT), true);
        fs.delete(new Path(PHASE1_OUTPUT), true);
        // Keep hot_keys.txt for reference
        
        return 0;
    }
    
    /**
     * Fallback: Standard aggregation when no skew detected.
     * Simple word-count pattern without salting overhead.
     */
    private int runStandardAggregation(
            Configuration conf, String inputPath, 
            String outputPath, int numReducers) throws Exception {
        
        System.out.println("\n  Running standard aggregation (no salting needed)...");
        
        Job job = Job.getInstance(conf, "Standard Aggregation (No Skew)");
        job.setJarByClass(SkewFixDriver.class);
        
        // Reuse the diagnostic mapper but without sampling
        job.setMapperClass(StandardCountMapper.class);
        job.setCombinerClass(SaltedAggregationCombiner.class);
        job.setReducerClass(DesaltReducer.class);
        job.setNumReduceTasks(numReducers);
        
        job.setOutputKeyClass(Text.class);
        job.setOutputValueClass(LongWritable.class);
        
        FileInputFormat.addInputPath(job, new Path(inputPath));
        FileOutputFormat.setOutputPath(job, new Path(outputPath));
        
        boolean success = job.waitForCompletion(true);
        return success ? 0 : 1;
    }
    
    public static void main(String[] args) throws Exception {

    }

package com.social.skew;

import org.apache.hadoop.io.LongWritable;
import org.apache.hadoop.io.Text;
import org.apache.hadoop.mapreduce.Reducer;

import java.io.IOException;

public class SkewDiagnosticCombiner 
        extends Reducer<Text, LongWritable, Text, LongWritable> {

    private LongWritable outValue = new LongWritable();

    @Override
    protected void reduce(Text key, Iterable<LongWritable> values, Context context)
            throws IOException, InterruptedException {
        
        long sum = 0;
        for (LongWritable val : values) {
            sum += val.get();
        }
        outValue.set(sum);
        context.write(key, outValue);
    }
}

// ═══════════════════════════════════════════════════════════════
// Used when diagnostic detects no skew — simple count without salting.

package com.social.skew;

import org.apache.hadoop.io.LongWritable;
import org.apache.hadoop.io.Text;
import org.apache.hadoop.mapreduce.Mapper;

import java.io.IOException;

public class StandardCountMapper 
        extends Mapper<LongWritable, Text, Text, LongWritable> {

    private Text outKey = new Text();
    private static final LongWritable ONE = new LongWritable(1);

    @Override
    protected void map(LongWritable key, Text value, Context context) 
            throws IOException, InterruptedException {
        
        String line = value.toString();
        String[] parts = line.split("\t");
        
        if (parts.length < 2) return;
        
        String userId = parts[1].trim();
        outKey.set(userId);
        context.write(outKey, ONE);
    }
}

// FILE 12: TestDataGenerator.java — GENERATE SKEWED TEST DATA
// ═══════════════════════════════════════════════════════════════
// Creates realistic skewed activity logs for testing.
// Follows Zipf distribution — few users have massive activity.

package com.social.skew;

import java.io.*;
import java.util.Random;

public class TestDataGenerator {

    /**
     * Generate skewed activity log data.
     * 
     * @param outputPath    Local file path
     * @param totalRecords  Total number of records to generate
     * @param numUsers      Total number of unique users
     * @param skewFactor    Higher = more skewed (1.0 = uniform, 2.0 = heavy skew)
     */
    public static void generate(
            String outputPath, long totalRecords, 
            int numUsers, double skewFactor) throws IOException {
        
        Random rand = new Random(42);
        String[] actions = {"like", "share", "comment", "view", "follow"};
        
        // ── Generate Zipf-distributed user weights ──
        // Zipf: probability of rank r ∝ 1/r^skewFactor
        double[] cumulativeWeights = new double[numUsers];
        double totalWeight = 0;
        
        for (int i = 0; i < numUsers; i++) {
            totalWeight += 1.0 / Math.pow(i + 1, skewFactor);
            cumulativeWeights[i] = totalWeight;
        }
        
        // Normalize to [0, 1]
        for (int i = 0; i < numUsers; i++) {
            cumulativeWeights[i] /= totalWeight;
        }
        
        // ── Pre-calculate expected distribution for top users ──
        System.out.println("\n  Expected distribution (top 10 users):");
        for (int i = 0; i < Math.min(10, numUsers); i++) {
            double prevCum = (i == 0) ? 0 : cumulativeWeights[i - 1];
            double prob = cumulativeWeights[i] - prevCum;
            long expectedRecords = (long)(prob * totalRecords);
            System.out.printf("    user_%07d: ~%,d records (%.4f%%)%n",
                i, expectedRecords, prob * 100);
        }
        
        // ── Generate records ──
        System.out.printf("\n  Generating %,d records...\n", totalRecords);
        long startTime = System.currentTimeMillis();
        long baseTimestamp = 1705276800000L;  // 2024-01-15 00:00:00
        
        try (BufferedWriter writer = new BufferedWriter(
                new FileWriter(outputPath), 1024 * 1024)) {
            
            for (long i = 0; i < totalRecords; i++) {
                // Select user using Zipf distribution
                double r = rand.nextDouble();
                int userRank = binarySearch(cumulativeWeights, r);
                
                String userId = String.format("user_%07d", userRank);
                String action = actions[rand.nextInt(actions.length)];
                long timestamp = baseTimestamp + (i * 10);  // 10ms apart
                String target = "item_" + rand.nextInt(1000000);
                
                // Format: timestamp\tuser_id\taction\ttarget
                writer.write(String.format("%d\t%s\t%s\t%s", 
                    timestamp, userId, action, target));
                writer.newLine();
                
                // Progress reporting
                if (i > 0 && i % 10000000 == 0) {
                    double elapsed = (System.currentTimeMillis() - startTime) / 1000.0;
                    double rate = i / elapsed;
                    System.out.printf("    %,d records (%.0f records/sec)%n", i, rate);
                }
            }
        }
        
        long elapsed = (System.currentTimeMillis() - startTime) / 1000;
        File f = new File(outputPath);
        
        System.out.printf("\n  ✅ Generated: %s%n", outputPath);
        System.out.printf("     Size: %.2f GB%n", f.length() / (1024.0 * 1024.0 * 1024.0));
        System.out.printf("     Time: %d seconds%n", elapsed);
        System.out.printf("     Rate: %,d records/sec%n", totalRecords / Math.max(1, elapsed));
    }
    
    /**
     * Binary search in cumulative weight array.
     * Returns the index where random value falls.
     */
    private static int binarySearch(double[] cumWeights, double value) {
        int lo = 0;
        int hi = cumWeights.length - 1;
        
        while (lo < hi) {
            int mid = (lo + hi) / 2;
            if (cumWeights[mid] < value) {
                lo = mid + 1;
            } else {
                hi = mid;
            }
        }
        return lo;
    }

        String outputDir = args.length > 0 ? args[0] : "/tmp/skew_test_data";
        new File(outputDir).mkdirs();
        
        System.out.println("═══════════════════════════════════════════════════════");

        System.out.println("═══════════════════════════════════════════════════════");

            outputDir + "/activity_log_small.tsv",

        System.out.println("    hadoop jar skew-fix.jar com.social.skew.SkewFixDriver \\");
        System.out.println("      /data/social/activity /data/social/output 200");

import org.apache.hadoop.mapreduce.lib.input.FileInputFormat;
import org.apache.hadoop.mapreduce.lib.output.FileOutputFormat;
import org.apache.hadoop.util.Tool;
import org.apache.hadoop.util.ToolRunner;

public class StandardCountDriver extends Configured implements Tool {

    @Override
    public int run(String[] args) throws Exception {
        
        if (args.length < 2) {
            System.err.println("Usage: StandardCountDriver <input> <output> [reducers]");
            return 1;
        }
        
        int numReducers = args.length > 2 ? Integer.parseInt(args[2]) : 200;
        
        Job job = Job.getInstance(getConf(), "Standard Count (No Skew Fix — Baseline)");
        job.setJarByClass(StandardCountDriver.class);
        
        job.setMapperClass(StandardCountMapper.class);
        job.setCombinerClass(SaltedAggregationCombiner.class);  // Reuse combiner
        job.setReducerClass(DesaltReducer.class);               // Reuse reducer (sums)
        job.setNumReduceTasks(numReducers);
        
        job.setOutputKeyClass(Text.class);
        job.setOutputValueClass(LongWritable.class);
        
        FileInputFormat.addInputPath(job, new Path(args[0]));
        FileOutputFormat.setOutputPath(job, new Path(args[1]));
        
        return job.waitForCompletion(true) ? 0 : 1;
    }
    
    public static void main(String[] args) throws Exception {
        int exitCode = ToolRunner.run(new Configuration(), new StandardCountDriver(), args);
        System.exit(exitCode);
    }
}

✅ THE SKEW PROBLEM:
   → Zipf's Law causes extreme data imbalance
   → One reducer processes 1000x more data than median
   → Job bottlenecked by single slowest task
   → 98% of cluster sits idle waiting

✅ THE THREE-PHASE SOLUTION:
   → Phase 0: DIAGNOSE — sample data, identify hot keys
   → Phase 1: SALT — distribute hot keys across multiple reducers
   → Phase 2: DE-SALT — combine partial results into final answer
   → Result: 8-15x speedup on heavily skewed data

✅ ADAPTIVE SALTING vs. UNIFORM SALTING:
   → Adaptive: salt only hot keys → minimal overhead
   → Uniform: salt all keys → simpler but wasteful for cold keys
   → Adaptive is preferred for production workloads
