STEP 1: Understand what Kerberos IS

   Kerberos is a NETWORK AUTHENTICATION PROTOCOL.
   It proves identity using CRYPTOGRAPHIC TICKETS.
   Named after the three-headed dog guarding the underworld.

   THE CORE PROBLEM KERBEROS SOLVES:
   → In a network, how does Server B KNOW that the request
     truly came from User A and not an impersonator?
   → Without Kerberos: "Trust me, I'm User A" (honor system)
   → With Kerberos: "Here's a cryptographic ticket from a trusted
     authority that proves I'm User A" (verified identity)

✅ WHAT SHOULD DO

1. SET UP A DEDICATED KDC SERVER
   → KDC should NOT run on a Hadoop node
   → Separate machine — its compromise means total cluster compromise
   → High availability KDC recommended (primary + replica)
   → KDC is the MOST CRITICAL security component

2. USE KEYTABS FOR SERVICES, NOT PASSWORDS
   → Keytab = file containing encrypted credentials
   → Services use keytabs for automated authentication
   → No human enters passwords for daemon startup
   → Keytab files must be readable ONLY by the service user
   → chmod 400 on every keytab file

3. USE SHORT TICKET LIFETIMES
   → Default TGT lifetime: 24 hours
   → Default renewable lifetime: 7 days
   → Shorter = more secure but more renewal overhead
   → Production recommendation: 24h TGT, 7d renewable

4. IMPLEMENT DELEGATION TOKENS FOR MAPREDUCE
   → When user submits MapReduce job:
     User authenticates with Kerberos → gets HDFS delegation token
     Token is embedded in the job → tasks use token (not user's TGT)
   → Why: MapReduce tasks run on remote machines
     They cannot access user's TGT → need delegation tokens
   → Hadoop handles this AUTOMATICALLY when security is enabled

5. SET UP HDFS SUPERUSER CAREFULLY
   → hdfs user is the HDFS superuser — can access EVERYTHING
   → Lock down hdfs keytab with strictest permissions
   → Never use hdfs account for routine operations
   → Create separate service accounts for each pipeline component

6. ENABLE HDFS AUDIT LOGGING
   → Every file access logged: who, what, when, from where
   → Essential for compliance (GDPR, PCI-DSS)
   → Send audit logs to SIEM for real-time monitoring

7. TEST SECURITY CONFIGURATION BEFORE PRODUCTION
   → Verify authentication works for all services
   → Verify unauthorized access is BLOCKED
   → Verify pipeline (Project 40) still works with Kerberos
   → Verify Web UIs are protected with SPNEGO

🚫 WHAT SHOULD NOT DO

1. DO NOT STORE KEYTABS IN HDFS
   → Keytabs are like password files — never put in shared storage
   → Store on local filesystem with strict permissions (400)
   → Each node gets ONLY the keytabs for services running on it

2. DO NOT USE THE SAME PRINCIPAL FOR MULTIPLE SERVICES
   → NameNode and DataNode must have DIFFERENT principals
   → If one service is compromised, the other is still protected
   → Principle of least privilege

3. DO NOT DISABLE SECURITY "TEMPORARILY" FOR DEBUGGING
   → Once enabled, never disable — even briefly
   → Disabling security = all permissions become unenforced
   → Data exposure during "temporary" window is permanent

4. DO NOT SKIP MUTUAL AUTHENTICATION
   → Client authenticates to server (standard)
   → Server ALSO authenticates to client (mutual)
   → Without mutual: man-in-the-middle can impersonate NameNode
   → Hadoop supports mutual authentication — always enable it

5. DO NOT FORGET ABOUT WEB UIS
   → NameNode UI (port 9870) shows file browser
   → Without SPNEGO: anyone can browse HDFS via web browser
   → Enable HTTP authentication for all Web UIs

6. DO NOT IGNORE CLOCK SYNCHRONIZATION
   → Kerberos tickets have timestamps
   → If clocks differ by > 5 minutes → authentication FAILS
   → Run NTP on EVERY node in the cluster
   → Clock skew is the #1 cause of Kerberos failures

7. DO NOT GRANT HDFS SUPERUSER TO APPLICATION ACCOUNTS
   → etl_service does NOT need superuser access
   → Grant access ONLY to directories the service needs
   → Use HDFS ACLs for fine-grained control

8. DO NOT HARDCODE KEYTAB PATHS IN APPLICATION CODE
   → Use configuration properties
   → Different environments have different paths
   → Keytab paths should be in Oozie job properties (Project 31)

🚫 WHEN NOT TO USE

1. DO NOT USE KERBEROS FOR SINGLE-USER DEVELOPMENT CLUSTERS
   → If you are the only user on a local pseudo-distributed setup
   → Kerberos overhead adds significant operational complexity
   → Use simple authentication for development (Project 8)
   → Enable Kerberos when moving to shared/production environments

2. DO NOT USE KERBEROS ALONE FOR FINE-GRAINED AUTHORIZATION
   → Kerberos provides AUTHENTICATION (who are you?)
   → NOT AUTHORIZATION (what can you do?)
   → Kerberos + HDFS ACLs + Ranger/Sentry = complete security
   → Kerberos without ACLs = everyone authenticated but still full access

3. DO NOT USE FILE-BASED KEYTABS IN CONTAINERIZED ENVIRONMENTS
   → Containers are ephemeral — keytab distribution is complex
   → Consider: Kubernetes secrets, HashiCorp Vault integration
   → Or: token-based authentication for short-lived containers

4. DO NOT USE KERBEROS FOR CROSS-REALM ACCESS WITHOUT TRUST
   → If two departments have separate Kerberos realms
   → Cross-realm trust must be explicitly configured
   → Without trust: users from DEPT_A.COM cannot access DEPT_B.COM

5. DO NOT USE MIT KERBEROS KDC FOR > 10,000 PRINCIPALS WITHOUT TUNING
   → Default MIT KDC scales to moderate sizes
   → Large enterprises: use Active Directory as KDC
   → Or: FreeIPA which combines KDC + LDAP + DNS

SCENARIO: analyst_bob wants to list files in HDFS

STEP 1 — User obtains TGT (Ticket Granting Ticket)

   Bob runs: kinit analyst_bob@COMPANY.COM
   (enters password)

   ┌────────┐                              ┌──────────────┐
   │  Bob   │  AS_REQ:                     │     KDC      │
   │        │  "I am analyst_bob,          │              │
   │        │   timestamp encrypted with   │  ┌────────┐  │
   │        │   my password hash"          │  │   AS   │  │
   │        │ ────────────────────────────→│  │        │  │
   │        │                              │  │ Lookup │  │
   │        │                              │  │ Bob's  │  │
   │        │                              │  │ key    │  │
   │        │  AS_REP:                     │  │ Decrypt│  │
   │        │  TGT = {                     │  │ Verify │  │
   │        │    identity: analyst_bob,    │  └────────┘  │
   │        │    expiry: +24 hours,        │              │
   │        │    session_key: SK1          │              │
   │        │  } encrypted with KDC's key  │              │
   │        │  +                           │              │
   │        │  SK1 encrypted with Bob's    │              │
   │        │  password hash               │              │
   │        │ ←────────────────────────────│              │
   └────────┘                              └──────────────┘

   RESULT: Bob now has a TGT valid for 24 hours
   Bob's password is NEVER sent over the network
   Only encrypted timestamps and session keys are exchanged

STEP 2 — User obtains Service Ticket for NameNode

   Bob runs: hdfs dfs -ls /data/
   Hadoop client automatically requests service ticket

   ┌────────┐                              ┌──────────────┐
   │  Bob   │  TGS_REQ:                   │     KDC      │
   │  (has  │  "I have this TGT,          │              │
   │  TGT)  │   I need a ticket for       │  ┌────────┐  │
   │        │   nn/hadoop-master"          │  │  TGS   │  │
   │        │ ────────────────────────────→│  │        │  │
   │        │                              │  │ Verify │  │
   │        │                              │  │ TGT    │  │
   │        │  TGS_REP:                   │  │ Create │  │
   │        │  ServiceTicket = {           │  │ ticket │  │
   │        │    identity: analyst_bob,    │  └────────┘  │
   │        │    service: nn/hadoop-master,│              │
   │        │    expiry: +24 hours,        │              │
   │        │    session_key: SK2          │              │
   │        │  } encrypted with NN's key   │              │
   │        │ ←────────────────────────────│              │
   └────────┘                              └──────────────┘

   RESULT: Bob has a service ticket for NameNode
   NameNode's key is needed to decrypt it — only NameNode can read it

STEP 3 — User presents Service Ticket to NameNode

   ┌────────┐                              ┌──────────────┐
   │  Bob   │  AP_REQ:                    │   NameNode   │
   │  (has  │  ServiceTicket +            │              │
   │  svc   │  Authenticator (timestamp   │  Decrypt     │
   │  ticket│  encrypted with SK2)        │  ticket with │
   │        │ ────────────────────────────→│  its keytab  │
   │        │                              │              │
   │        │                              │  Verify:     │
   │        │                              │  - Not expired│
   │        │                              │  - Timestamp  │
   │        │                              │    is fresh   │
   │        │                              │  - Identity   │
   │        │  AP_REP:                    │    is valid   │
   │        │  "Authenticated as          │              │
   │        │   analyst_bob@COMPANY.COM"  │  Check HDFS  │
   │        │  + directory listing         │  permissions │
   │        │ ←────────────────────────────│  for Bob     │
   └────────┘                              └──────────────┘

   RESULT: NameNode KNOWS this is really analyst_bob
   → Checks HDFS permissions for /data/
   → Returns directory listing if Bob has read permission
   → Returns "Permission denied" if Bob lacks access
   → Logs access in audit log regardless

═══════════════════════════════════════════════════════════
MECHANIC 2: DELEGATION TOKENS FOR MAPREDUCE
═══════════════════════════════════════════════════════════

PROBLEM:
→ Bob submits a MapReduce job from his machine
→ Job spawns 100 map tasks across 50 different nodes
→ Each map task needs to read from HDFS
→ Map tasks run as Bob — but they're on REMOTE machines
→ They cannot access Bob's TGT (it's on Bob's machine)
→ They cannot run kinit (no password available)

SOLUTION: Delegation Tokens

   ┌────────┐  "Give me a delegation   ┌──────────────┐
   │  Bob   │   token for HDFS"        │   NameNode   │
   │  (has  │ ─────────────────────────→│              │
   │  TGT)  │                          │  Create      │
   │        │  DelegationToken = {      │  token:      │
   │        │    owner: analyst_bob,    │  - Signed by │
   │        │    renewer: yarn,         │    NameNode  │
   │        │    expiry: +24 hours,     │  - Tracks    │
   │        │    token_id: 12345,       │    token ID  │
   │        │    signature: <signed>    │              │
   │        │  }                        │              │
   │        │ ←─────────────────────────│              │
   └───┬────┘                          └──────────────┘
       │
       │  Embeds token in job submission
       ▼
   ┌──────────────────────────────────────────────────┐
   │  YARN ApplicationMaster                          │
   │  Distributes token to all task containers         │
   │                                                  │
   │  ┌──────────┐ ┌──────────┐ ┌──────────┐        │
   │  │ Map Task │ │ Map Task │ │ Map Task │        │
   │  │ Node 01  │ │ Node 02  │ │ Node 03  │        │
   │  │          │ │          │ │          │        │
   │  │ Uses     │ │ Uses     │ │ Uses     │        │
   │  │ deleg.   │ │ deleg.   │ │ deleg.   │        │
   │  │ token    │ │ token    │ │ token    │        │
   │  └─────┬────┘ └─────┬────┘ └─────┬────┘        │
   │        │             │            │              │
   └────────┼─────────────┼────────────┼──────────────┘
            │             │            │
            ▼             ▼            ▼
      ┌──────────────────────────────────────┐
      │           NameNode                    │
      │                                      │
      │  Receives delegation token            │
      │  Verifies: signature, expiry, owner   │
      │  Treats request as analyst_bob        │
      │  Applies Bob's HDFS permissions       │
      └──────────────────────────────────────┘

   KEY PROPERTIES OF DELEGATION TOKENS:
   → Created by authenticated user (requires Kerberos first)
   → Has an OWNER (who it represents) and RENEWER (who can extend it)
   → Expires after configured time (default 24 hours)
   → Can be RENEWED by the renewer (YARN renews for long jobs)
   → Can be CANCELLED explicitly
   → NameNode tracks all active tokens in memory

MECHANIC 3: KEYTAB-BASED SERVICE AUTHENTICATION
═══════════════════════════════════════════════════════════

PROBLEM:
→ NameNode starts at boot time — no human to type password
→ DataNodes start automatically — no human available
→ Services need AUTOMATED authentication

SOLUTION: Keytab files

   A keytab file contains:
   ┌─────────────────────────────────────────────────────┐
   │ KEYTAB FILE: /etc/hadoop/conf/nn.keytab             │
   │                                                     │
   │ Principal: nn/hadoop-master@COMPANY.COM             │
   │ Key Version: 3                                      │
   │ Encryption: aes256-cts-hmac-sha1-96                 │
   │ Key: <encrypted bytes>                              │
   │                                                     │
   │ Principal: HTTP/hadoop-master@COMPANY.COM           │
   │ Key Version: 2                                      │
   │ Encryption: aes256-cts-hmac-sha1-96                 │
   │ Key: <encrypted bytes>                              │
   └─────────────────────────────────────────────────────┘

   WHEN NAMENODE STARTS:
   1. Reads keytab file
   2. Extracts encrypted key for nn/hadoop-master@COMPANY.COM
   3. Uses key to authenticate to KDC (same as kinit but automated)
   4. Obtains TGT
   5. Now NameNode is authenticated — can prove its identity

   SECURITY OF KEYTAB:
   → File permissions: -r-------- (400) — only service user can read
   → Stored on LOCAL filesystem — never in HDFS
   → Each node gets ONLY keytabs for services running on it
   → Compromise of keytab = compromise of that service's identity
   → Equivalent to knowing the service's password

MECHANIC 4: HDFS PERMISSION MODEL WITH KERBEROS
═══════════════════════════════════════════════════════════

WITHOUT KERBEROS:
   → User sets HADOOP_USER_NAME=hdfs → instant superuser
   → Permissions exist but are UNENFORCED
   → Anyone can bypass any restriction

WITH KERBEROS:
   → User identity comes from Kerberos ticket — CANNOT be faked
   → HDFS checks ticket identity against file permissions
   → Superuser (hdfs) requires hdfs keytab — not just a username

   PERMISSION CHECK FLOW:
   ┌──────────────────────────────────────────────────────────┐
   │ Request: analyst_bob wants to read /data/warehouse/      │
   │                                                          │
   │ Step 1: Verify Kerberos ticket                           │
   │   → Is ticket valid? (not expired, not forged)           │
   │   → Identity: analyst_bob@COMPANY.COM                    │
   │                                                          │
   │ Step 2: Map Kerberos principal to OS user                │
   │   → auth_to_local rules: analyst_bob@COMPANY.COM → bob   │
   │                                                          │
   │ Step 3: Check HDFS permissions                           │
   │   /data/warehouse/  owner=hdfs  group=analytics  755     │
   │   → bob is not owner (hdfs)                              │
   │   → Is bob in group "analytics"? Check group mapping     │
   │   → YES → group permission = r-x → READ ALLOWED          │
   │                                                          │
   │ Step 4: Check HDFS ACLs (if extended ACLs exist)         │
   │   getfacl /data/warehouse/                               │
   │   user::rwx                                              │
   │   user:etl_service:rwx                                   │
   │   group::r-x                                             │
   │   group:datascience:r-x                                  │
   │   mask::r-x                                              │
   │   other::---                                              │

⚠️ COMMON PITFALLS AND FAILURE MODES

PITFALL 1: "GSSException: No valid credentials provided"
→ THE #1 most common Kerberos error
→ CAUSE 1: TGT expired — run kinit again
→ CAUSE 2: Keytab file not found or not readable
→ CAUSE 3: Principal name mismatch (typo in config)
→ CAUSE 4: Clock skew > 5 minutes between client and KDC
→ DEBUG: klist to check current tickets
→ DEBUG: KRB5_TRACE=/dev/stderr for detailed protocol trace

PITFALL 2: "Clock skew too great"
→ THE #2 most common Kerberos error
→ Kerberos requires clocks synchronized within 5 minutes (default)
→ FIX: Install and configure NTP on EVERY node
→ FIX: ntpdate -s ntp.company.com on all machines
→ VERIFY: date on each node — must match within seconds

PITFALL 3: "NameNode won't start after enabling security"
→ CAUSE: Keytab path wrong in hdfs-site.xml
→ CAUSE: Keytab not readable by hdfs user (permission issue)
→ CAUSE: Principal in keytab doesn't match config
→ DEBUG: Start NameNode manually with debug logging:
   HADOOP_OPTS="-Dsun.security.krb5.debug=true" hdfs namenode

PITFALL 4: "MapReduce job fails with 'token not found'"
→ CAUSE: Delegation token expired during long-running job
→ CAUSE: Token renewal not configured correctly
→ FIX: Increase token max lifetime
→ FIX: Ensure YARN is configured as token renewer
→ FIX: Set mapreduce.job.hdfs-servers for cross-NameNode jobs

PITFALL 5: "Web UI shows 401 Unauthorized"
→ CAUSE: SPNEGO (HTTP Kerberos) not configured
→ CAUSE: Browser not configured for Kerberos
→ FIX: Configure HTTP/ principal and keytab
→ FIX: Browser: Firefox → about:config → network.negotiate-auth.trusted-uris

PITFALL 6: "User can access files they shouldn't"
→ CAUSE: auth_to_local rules mapping principal to wrong OS user
→ CAUSE: OS group membership not matching HDFS group
→ CAUSE: HDFS permissions too permissive (world-readable)
→ DEBUG: hdfs groups <username> to check group membership
→ DEBUG: hdfs dfs -ls -d /data/warehouse/ to check permissions

PITFALL 7: "Pipeline (Project 40) breaks after enabling Kerberos"
→ CAUSE: Oozie not configured with Kerberos
→ CAUSE: Sqoop not passing delegation tokens
→ CAUSE: Hive metastore not authenticated
→ FIX: Configure each ecosystem component individually
→ FIX: Test each pipeline stage with kinit before Oozie automation
→ FIX: Oozie credential configuration for Hive and HBase

PITFALL 8: "Cross-realm authentication fails"
→ CAUSE: No trust relationship between realms
→ CAUSE: krbtgt/REALM_B@REALM_A principal not created
→ FIX: Establish cross-realm trust in both KDCs
→ FIX: Configure auth_to_local for cross-realm principal mapping

DELEGATION TOKEN SCALING:
→ NameNode tracks ALL active delegation tokens in memory
→ Each token: ~200 bytes of heap
→ 100,000 concurrent tokens: ~20MB (negligible)
→ 10 million concurrent tokens: ~2GB (significant)
→ Token count = concurrent_jobs × services_accessed_per_job

KDC PERFORMANCE:
→ Each authentication: ~1-5ms
→ Single MIT KDC: ~5000 authentications/second
→ At cluster startup: ALL services authenticate simultaneously
→ 200 nodes × 3 services = 600 simultaneous requests
→ Burst handling: KDC must handle startup spike

TICKET CACHE PERFORMANCE:
→ TGTs are cached locally — not re-requested for each operation
→ Service tickets are cached — one request per service per ticket lifetime
→ Performance impact after initial authentication: MINIMAL
→ Hadoop operations add ~1-2ms for ticket verification overhead

#!/bin/bash
set -e

echo "═══════════════════════════════════════════════════════"

echo "═══════════════════════════════════════════════════════"

KDC_HOST="kdc.company.com"

DOMAIN="company.com"

# ── Step 1: Install Kerberos packages ──
echo "[1/6] Installing Kerberos packages..."
sudo apt-get update -qq
sudo apt-get install -y krb5-kdc krb5-admin-server krb5-config

# ── Step 2: Configure krb5.conf ──
echo "[2/6] Configuring /etc/krb5.conf..."

sudo cat > /etc/krb5.conf << EOF
[libdefaults]
    default_realm = ${REALM}
    dns_lookup_realm = false
    dns_lookup_kdc = false
    ticket_lifetime = 24h
    renew_lifetime = 7d
    forwardable = true
    # Clock skew tolerance (default 5 minutes)
    clockskew = 300
    # Default encryption types
    default_tkt_enctypes = aes256-cts-hmac-sha1-96 aes128-cts-hmac-sha1-96
    default_tgs_enctypes = aes256-cts-hmac-sha1-96 aes128-cts-hmac-sha1-96

[realms]
    ${REALM} = {
        kdc = ${KDC_HOST}
        admin_server = ${KDC_HOST}
        default_domain = ${DOMAIN}
    }

[domain_realm]
    .${DOMAIN} = ${REALM}
    ${DOMAIN} = ${REALM}

[logging]
    kdc = FILE:/var/log/kerberos/krb5kdc.log
    admin_server = FILE:/var/log/kerberos/kadmin.log
    default = FILE:/var/log/kerberos/krb5lib.log
EOF

# ── Step 3: Configure KDC ──
echo "[3/6] Configuring KDC..."

sudo cat > /etc/krb5kdc/kdc.conf << EOF
[kdcdefaults]
    kdc_ports = 88
    kdc_tcp_ports = 88

[realms]
    ${REALM} = {
        kadmind_port = 749
        max_life = 24h 0m 0s
        max_renewable_life = 7d 0h 0m 0s
        master_key_type = aes256-cts-hmac-sha1-96
        supported_enctypes = aes256-cts-hmac-sha1-96:normal aes128-cts-hmac-sha1-96:normal
        # Ticket flags
        default_principal_flags = +renewable, +forwardable
    }
EOF

# ── Step 4: Create KDC database ──
echo "[4/6] Creating KDC database (this will prompt for master password)..."
sudo mkdir -p /var/log/kerberos
sudo kdb5_util create -s -r ${REALM}

# ── Step 5: Configure admin ACL ──
echo "[5/6] Configuring admin ACL..."
sudo cat > /etc/krb5kdc/kadm5.acl << EOF
# Format: principal  permissions  [target_principal]
*/admin@${REALM}    *
hadoop_admin@${REALM}  *
EOF

# ── Step 6: Start KDC services ──
echo "[6/6] Starting KDC services..."
sudo systemctl enable krb5-kdc krb5-admin-server
sudo systemctl start krb5-kdc krb5-admin-server
sudo systemctl status krb5-kdc krb5-admin-server

echo "   Admin port: 749"

# ════════════════════════════════════════

# (must match HDFS group mappings)

# ════════════════════════════════════════

# ── Raw data zone — ETL service writes, others read ──
hdfs dfs -chown -R etl_service:etl_team /data/raw

# etl_service: rwx (read, write, execute)
# etl_team:    r-x (read, execute)

hdfs dfs -chown -R etl_service:etl_team /data/warehouse
hdfs dfs -chmod -R 750 /data/warehouse

# ── Serving zone — ETL writes, HBase reads ──
hdfs dfs -chown -R etl_service:hadoop /data/serving
hdfs dfs -chmod -R 750 /data/serving

# ── Quality reports — ETL writes, engineers read ──
hdfs dfs -chown -R etl_service:etl_team /data/quality
hdfs dfs -chmod -R 750 /data/quality

# ── Archive — hdfs owns, restricted access ──
hdfs dfs -chown -R hdfs:hadoop /data/archive

# FILE 11: security_audit.py
# PURPOSE: Automated security audit of the Kerberized cluster

import subprocess
import sys
import os
import datetime


class HadoopSecurityAuditor:
    """
    Automated security audit for Kerberized Hadoop cluster.
    Checks authentication, authorization, permissions, and configuration.
    """

    def __init__(self):
        self.findings = []
        self.timestamp = datetime.datetime.now().strftime("%Y-%m-%d %H:%M:%S")
        self.passed = 0
        self.failed = 0
        self.warnings = 0

    def run_cmd(self, cmd, timeout=30):
        """Execute shell command and return output."""
        try:
            result = subprocess.run(
                cmd, shell=True, capture_output=True,
                text=True, timeout=timeout
            )
            return result.stdout.strip(), result.stderr.strip(), result.returncode
        except subprocess.TimeoutExpired:
            return "", "TIMEOUT", 1
        except Exception as e:
            return "", str(e), 1

    def check(self, name, condition, detail=""):
        """Record an audit check result."""
        if condition:
            self.passed += 1
            status = "✅ PASS"
        else:
            self.failed += 1
            status = "❌ FAIL"
        self.findings.append({
            "name": name,
            "status": status,
            "detail": detail,
            "passed": condition
        })

    def warn(self, name, detail=""):
        """Record a warning."""
        self.warnings += 1
        self.findings.append({
            "name": name,
            "status": "⚠️  WARN",
            "detail": detail,
            "passed": None
        })

    # ════════════════════════════════════════
    # AUDIT CHECKS
    # ════════════════════════════════════════

    def check_kerberos_enabled(self):
        """Verify Hadoop is configured for Kerberos authentication."""
        stdout, _, rc = self.run_cmd(
            "hdfs getconf -confKey hadoop.security.authentication"
        )
        self.check(
            "Kerberos authentication enabled",
            stdout.strip() == "kerberos",
            f"Current value: {stdout.strip()}"
        )

    def check_authorization_enabled(self):
        """Verify service-level authorization is enabled."""
        stdout, _, rc = self.run_cmd(
            "hdfs getconf -confKey hadoop.security.authorization"
        )
        self.check(
            "Service-level authorization enabled",
            stdout.strip() == "true",
            f"Current value: {stdout.strip()}"
        )

    def check_block_tokens_enabled(self):
        """Verify block access tokens are enabled."""
        stdout, _, rc = self.run_cmd(
            "hdfs getconf -confKey dfs.block.access.token.enable"
        )
        self.check(
            "Block access tokens enabled",
            stdout.strip() == "true",
            f"Current value: {stdout.strip()}"
        )

    def check_acls_enabled(self):
        """Verify HDFS ACLs are enabled."""
        stdout, _, rc = self.run_cmd(
            "hdfs getconf -confKey dfs.namenode.acls.enabled"
        )
        self.check(
            "HDFS ACLs enabled",
            stdout.strip() == "true",
            f"Current value: {stdout.strip()}"
        )

    def check_valid_ticket(self):
        """Verify current user has valid Kerberos ticket."""
        stdout, stderr, rc = self.run_cmd("klist -s")
        self.check(
            "Valid Kerberos ticket exists",
            rc == 0,
            "Run 'kinit' to obtain a ticket" if rc != 0 else "Ticket valid"
        )

    def check_keytab_permissions(self):
        """Verify keytab files have correct permissions."""
        keytab_dir = "/etc/hadoop/keytabs"
        if not os.path.exists(keytab_dir):
            self.warn("Keytab directory check", f"{keytab_dir} not found on this host")
            return

        stdout, _, _ = self.run_cmd(f"ls -la {keytab_dir}/*.keytab 2>/dev/null")
        if not stdout:
            self.warn("No keytab files found", f"Directory: {keytab_dir}")
            return

        for line in stdout.split("\n"):
            if not line.strip():
                continue
            parts = line.split()
            if len(parts) < 9:
                continue
            perms = parts[0]
            filename = parts[-1]

            # Keytab should be -r-------- (400) or -rw------- (600)
            is_secure = (perms == "-r--------" or perms == "-rw-------")
            self.check(
                f"Keytab permission: {os.path.basename(filename)}",
                is_secure,
                f"Permissions: {perms} {'(OK)' if is_secure else '(should be 400)'}"
            )

    def check_unauthenticated_access_blocked(self):
        """Verify that unauthenticated HDFS access is denied."""
        # Destroy any existing tickets first
        self.run_cmd("kdestroy 2>/dev/null")

        stdout, stderr, rc = self.run_cmd(
            "HADOOP_USER_NAME=root hdfs dfs -ls / 2>&1"
        )

        # Re-authenticate for remaining checks
        self.run_cmd(
            "kinit -kt /etc/hadoop/keytabs/hdfs.keytab hdfs@COMPANY.COM 2>/dev/null"
        )

        access_denied = (
            "GSSException" in stderr or
            "GSSException" in stdout or
            "No valid credentials" in stderr or
            "No valid credentials" in stdout or
            rc != 0
        )

        self.check(
            "Unauthenticated access blocked",
            access_denied,
            "Unauthenticated access should be DENIED"
        )

    def check_directory_permissions(self):
        """Verify critical directories have correct permissions."""
        critical_dirs = {
            "/data/raw":       {"expected_perms": "drwxr-x---", "expected_owner": "etl_service"},
            "/data/warehouse": {"expected_perms": "drwxr-x---", "expected_owner": "etl_service"},
            "/data/archive":   {"expected_perms": "drwx------", "expected_owner": "hdfs"},
        }

        for path, expected in critical_dirs.items():
            stdout, stderr, rc = self.run_cmd(f"hdfs dfs -ls -d {path} 2>/dev/null")
            if rc != 0 or not stdout:
                self.warn(f"Directory check: {path}", f"Cannot access: {stderr}")
                continue

            parts = stdout.split()
            actual_perms = parts[0] if len(parts) > 0 else "unknown"
            actual_owner = parts[2] if len(parts) > 2 else "unknown"

            self.check(
                f"Directory permissions: {path}",
                actual_perms == expected["expected_perms"],
                f"Expected: {expected['expected_perms']}, Actual: {actual_perms}"
            )

            self.check(
                f"Directory ownership: {path}",
                actual_owner == expected["expected_owner"],
                f"Expected: {expected['expected_owner']}, Actual: {actual_owner}"
            )

    def check_web_ui_authentication(self):
        """Verify Web UIs require authentication."""
        stdout, _, rc = self.run_cmd(
            "hdfs getconf -confKey hadoop.http.authentication.type"
        )
        self.check(
            "Web UI authentication type",
            stdout.strip() == "kerberos",
            f"Current: {stdout.strip()} (should be 'kerberos')"
        )

    def check_container_executor(self):
        """Verify LinuxContainerExecutor is configured."""
        stdout, _, rc = self.run_cmd(
            "hdfs getconf -confKey yarn.nodemanager.container-executor.class"
        )
        is_linux_executor = "LinuxContainerExecutor" in stdout
        self.check(
            "LinuxContainerExecutor enabled",
            is_linux_executor,
            f"Current: {stdout.strip()}"
        )

    def check_rpc_protection(self):
        """Verify RPC protection level."""
        stdout, _, rc = self.run_cmd(
            "hdfs getconf -confKey hadoop.rpc.protection"
        )
        protection = stdout.strip()
        self.check(
            "RPC protection level",
            protection in ("authentication", "integrity", "privacy"),
            f"Current: {protection}"
        )
        if protection == "authentication":
            self.warn(
                "RPC protection upgrade recommended",
                "Consider 'privacy' for encrypted RPC (Project 42)"
            )

    def check_ntp_synchronization(self):
        """Verify NTP is running (critical for Kerberos)."""
        stdout, _, rc = self.run_cmd("ntpstat 2>/dev/null || timedatectl show --property=NTPSynchronized 2>/dev/null")
        is_synced = ("synchronised" in stdout.lower() or "ntp=yes" in stdout.lower()
                     or "ntpsynchronized=yes" in stdout.lower())
        self.check(
            "NTP synchronization",
            is_synced,
            "Clock sync is CRITICAL for Kerberos — max 5 min skew"
        )

    # ════════════════════════════════════════
    # REPORT GENERATION
    # ════════════════════════════════════════

    def generate_report(self):
        """Generate formatted audit report."""
        report = []
        report.append("")
        report.append("═" * 70)
        report.append(f"  HADOOP SECURITY AUDIT REPORT — {self.timestamp}")
        report.append("═" * 70)

        report.append("")
        report.append(f"  Summary: {self.passed} passed, "
                      f"{self.failed} failed, {self.warnings} warnings")
        report.append("")

        # Group by status
        for status_filter in ["❌ FAIL", "⚠️  WARN", "✅ PASS"]:
            items = [f for f in self.findings if f["status"] == status_filter]
            if items:
                report.append(f"  {status_filter.split()[0]} {'FAILURES' if 'FAIL' in status_filter else 'WARNINGS' if 'WARN' in status_filter else 'PASSED'}:")
                for item in items:
                    report.append(f"    {item['status']} {item['name']}")
                    if item['detail']:
                        report.append(f"         → {item['detail']}")
                report.append("")

        report.append("═" * 70)

        if self.failed > 0:
            report.append("  🔴 AUDIT RESULT: FAILED — Critical security issues found")
        elif self.warnings > 0:
            report.append("  🟡 AUDIT RESULT: PASSED WITH WARNINGS")
        else:
            report.append("  🟢 AUDIT RESULT: PASSED — All checks OK")

        report.append("═" * 70)
        return "\n".join(report)

    def run(self):
        """Execute all audit checks."""
        print("  Running security audit checks...")
        print("")

        self.check_kerberos_enabled()
        self.check_authorization_enabled()
        self.check_block_tokens_enabled()
        self.check_acls_enabled()
        self.check_valid_ticket()
        self.check_keytab_permissions()
        self.check_unauthenticated_access_blocked()
        self.check_directory_permissions()
        self.check_web_ui_authentication()
        self.check_container_executor()
        self.check_rpc_protection()
        self.check_ntp_synchronization()

        report = self.generate_report()
        print(report)

        return 2 if self.failed > 0 else (1 if self.warnings > 0 else 0)


if __name__ == "__main__":
    auditor = HadoopSecurityAuditor()
    exit_code = auditor.run()
    sys.exit(exit_code)

✅ THE PROBLEM: Without authentication, Hadoop security is an illusion
   → Anyone can claim to be any user
   → Rogue services can join the cluster
   → HDFS permissions are meaningless without identity verification
   → Compliance with GDPR, PCI-DSS, SOX is impossible

✅ KERBEROS PROVIDES:
   → AUTHENTICATION: Cryptographic proof of identity
   → MUTUAL AUTHENTICATION: Services prove identity to each other
   → DELEGATION: MapReduce tasks authenticate on behalf of users
   → AUDIT TRAIL: Every access tied to verified identity

✅ WHAT KERBEROS DOES NOT PROVIDE:
   → AUTHORIZATION: Use HDFS ACLs + hadoop-policy.xml
   → ENCRYPTION: Use SSL/TLS + HDFS Encryption Zones (Project 42)
   → Fine-grained table access: Use Apache Ranger or Sentry

✅ OPERATIONAL REALITY:
   → Kerberos adds SIGNIFICANT operational complexity
