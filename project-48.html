PROJECT 48: MIGRATING A HADOOP PIPELINE TO APACHE SPARK — SIDE-BY-SIDE

🔴 HYPOTHETICAL PROBLEM

SCENARIO:
→ You are the lead data engineer at a logistics company
→ Your existing Hadoop pipeline (built across Projects 21, 40, 46):
    - Ingests 2TB of shipment tracking events daily via Flume
    - Cleans and normalizes raw data (MapReduce ETL)
    - Joins shipment data with customer master (Reduce-Side Join)
    - Aggregates delivery metrics per region per day
    - Chains 5 MapReduce jobs via Oozie
    - Loads final results into HBase for dashboards
→ Total pipeline runtime: 6.5 HOURS

BUSINESS PRESSURE:
→ Operations team needs metrics refreshed every 2 hours, not 8
→ Data science team wants to run iterative ML on same data
→ MapReduce rewrites each intermediate result to HDFS disk
→ 5-job chain means 5 full HDFS write-read cycles
→ Each cycle: serialize → write HDFS → schedule new job → deserialize → read HDFS
→ CTO asks: "Can Spark cut this to under 1 hour WITHOUT rebuilding infrastructure?"

THE CORE CHALLENGE:
→ Migrating incrementally — NOT rip-and-replace
→ Running Spark ON the existing YARN cluster
→ Understanding EXACTLY where Spark wins and where MapReduce still fits
→ Avoiding the trap of "rewrite everything in Spark" without understanding WHY

SPARK CORE ABSTRACTION: RDD (Resilient Distributed Dataset)

✅ WHAT SHOULD DO

1. INSTALL SPARK ON EXISTING CLUSTER WITHOUT DISRUPTION
   → Spark is a library — untar it on edge node
   → Point it to existing YARN ResourceManager
   → No changes to HDFS, no changes to running jobs

2. VALIDATE OUTPUT PARITY BEFORE RETIRING MAPREDUCE
   → Run BOTH pipelines in parallel for 2 weeks
   → Compare output row counts, checksums, sample values
   → Only retire MapReduce after 100% match confirmed

3. USE DATAFRAME API OVER RAW RDD API
   → DataFrame/Dataset API enables Catalyst optimizer
   → Catalyst generates optimized physical execution plan
   → Raw RDD API skips optimization — use only for custom logic

4. LEVERAGE SPARK SQL FOR EXISTING HIVE TABLES
   → Spark reads Hive metastore directly
   → All Hive tables (Project 24, 25, 34) instantly queryable
   → spark.sql("SELECT * FROM hive_table") — zero migration

5. USE BROADCAST JOINS FOR SMALL DIMENSION TABLES
   → Replaces Distributed Cache pattern (Project 15, 16)
   → Spark broadcasts small table to all executors
   → Eliminates shuffle for star-schema joins (Project 34)

6. CACHE INTERMEDIATE RESULTS THAT ARE REUSED
   → df.cache() keeps data in executor memory
   → Replaces HDFS intermediate writes entirely
   → Critical for iterative algorithms (ML training loops)

7. MONITOR VIA SPARK UI
   → http://driver-host:4040 during execution
   → Shows DAG visualization, stage breakdown, task metrics
   → Far more detailed than MapReduce job counters (Project 7)

🚫 WHAT SHOULD NOT DO

1. DO NOT rewrite everything at once
   → Migrate one job at a time
   → Validate each independently
   → Parallel-run before cutover
   → Big-bang rewrites fail 70% of the time in production

2. DO NOT use .groupByKey() where .reduceByKey() works
   → groupByKey() shuffles ALL values to reducer — like no combiner
   → reduceByKey() combines locally first — like having a combiner (Project 9)
   → Performance difference: 10-100× on skewed data

3. DO NOT ignore executor memory configuration
   → Spark holds data IN MEMORY
   → Undersized executors → constant garbage collection → slower than MapReduce
   → Oversized executors → GC pauses exceed 30 seconds → tasks killed
   → Sweet spot: 4-8 cores per executor, 16-32GB per executor

4. DO NOT cache everything blindly
   → Memory is finite — caching one dataset may evict another
   → Only cache datasets that are READ MULTIPLE TIMES
   → Single-use datasets: DO NOT cache — let Spark stream through

5. DO NOT assume Spark is ALWAYS faster
   → Single-pass ETL on 50TB with no joins → MapReduce may match Spark
   → MapReduce has lower memory overhead — stable for enormous shuffles
   → Spark shuffle with insufficient memory → spills to disk → same speed
   → See "When MapReduce Still Wins" section below

6. DO NOT skip partitioning strategy
   → Default Spark partitions: 200 (spark.sql.shuffle.partitions)
   → For TB-scale: needs 2000-10000 partitions
   → Too few → OOM errors, too many → scheduling overhead
   → Rule: target 128-256MB per partition (matches HDFS block size)

7. DO NOT mix Spark versions carelessly
   → Spark 2.x → Spark 3.x has breaking API changes
   → Adaptive Query Execution (AQE) only in Spark 3.0+
   → Dynamic Partition Pruning only in Spark 3.0+
   → Pin your version and test upgrades in staging

WHEN MAPREDUCE GENUINELY STILL WINS:
1. Pure I/O-bound single-pass ETL on petabyte scale
   → Disk throughput is the bottleneck, not computation
   → Spark's in-memory advantage is irrelevant
   → MR's lower memory footprint means more tasks per node

2. Environments with extreme memory constraints
   → MR designed for 1-2GB containers
   → Spark executors need 8-32GB to be effective

3. Regulatory environments requiring audit trail stability
   → MR pipeline unchanged for 5 years = proven compliance
   → Rewriting in Spark requires re-certification

🚫 WHEN NOT TO USE

DO NOT MIGRATE TO SPARK WHEN:

1. The MapReduce pipeline is meeting all SLAs comfortably
   → "If it ain't broke, don't fix it" applies
   → Migration cost (engineering time, testing, risk) must be justified

2. Your team has zero Spark experience
   → Spark's debugging is HARDER than MapReduce
   → Memory tuning requires deep understanding
   → Budget 2-4 weeks of learning before production migration

3. The pipeline is being decommissioned within 6 months
   → Migration effort wasted on dying system
   → Focus engineering time on the replacement

4. Data fits on a single machine
   → Neither Spark nor MapReduce needed
   → Pandas, DuckDB, or even SQL Server handles it
   → (Same principle from Project 1 — Hadoop is for BIG data)

5. You need sub-100ms latency
   → Spark Structured Streaming: seconds latency
   → For milliseconds: use Kafka Streams or Apache Flink
   → Spark is NOT a real-time system despite "streaming" label

═══════════════════════════════════════════════════════════
MECHANIC 1: WHY SPARK IS FASTER — THE DISK I/O ELIMINATION
═══════════════════════════════════════════════════════════

MAPREDUCE 5-JOB CHAIN (from Project 21):

  Read HDFS → Map1 → Shuffle → Reduce1 → WRITE HDFS
  Read HDFS → Map2 → Shuffle → Reduce2 → WRITE HDFS
  Read HDFS → Map3 → Shuffle → Reduce3 → WRITE HDFS
  Read HDFS → Map4 → Shuffle → Reduce4 → WRITE HDFS
  Read HDFS → Map5 → Shuffle → Reduce5 → WRITE HDFS

  DISK I/O COUNT:
  → 5 HDFS reads + 5 HDFS writes = 10 HDFS operations
  → Each HDFS write: replicated 3× = 15 actual disk writes
  → Each shuffle: Map spills to local disk, Reduce reads from disk
  → Total disk I/O: ~25-30 full dataset scans

SPARK SINGLE APPLICATION:

  Read HDFS → transform → transform → transform → transform → WRITE HDFS

  DISK I/O COUNT:
  → 1 HDFS read + 1 HDFS write = 2 HDFS operations
  → Intermediate data: stays in executor JVM memory
  → If memory insufficient: spills to LOCAL disk (not HDFS, no replication)
  → Total disk I/O: 2-4 full dataset scans

  SPEEDUP = 25/3 ≈ 8× just from eliminating I/O
  (Real-world: 5-100× depending on pipeline complexity)


═══════════════════════════════════════════════════════════
MECHANIC 2: RDD LINEAGE vs. REPLICATION
═══════════════════════════════════════════════════════════

HDFS FAULT TOLERANCE (Project 2):
→ Every block replicated 3 times
→ If a node dies, data is already on 2 other nodes
→ Cost: 3× storage, 3× write bandwidth

SPARK FAULT TOLERANCE:
→ RDDs are NOT replicated by default
→ Instead, Spark remembers HOW to recompute each RDD
→ This "recipe" is called the LINEAGE GRAPH

  Example lineage:
  
  raw = sc.textFile("hdfs:///logs")           ← reads from HDFS
  clean = raw.filter(lambda x: len(x) > 0)   ← remembers: filter on raw
  pairs = clean.map(lambda x: (x[0], 1))     ← remembers: map on clean
  counts = pairs.reduceByKey(lambda a,b: a+b) ← remembers: reduce on pairs

  If executor holding "pairs" partition 7 crashes:
  → Spark does NOT look for a replica
  → Spark replays: read HDFS block 7 → filter → map → recovered!
  → Only recomputes the LOST partition, not the entire dataset

  LINEAGE GRAPH:
  
  textFile ──→ filter ──→ map ──→ reduceByKey
  (HDFS)       (clean)    (pairs)  (counts)
      │            │          │         │
      └── If this partition lost, trace back to HDFS and replay


═══════════════════════════════════════════════════════════
MECHANIC 3: DAG EXECUTION ENGINE (vs. MapReduce linear)
═══════════════════════════════════════════════════════════

MAPREDUCE EXECUTION MODEL:
→ Rigid: Map → Shuffle → Reduce
→ Every job MUST have a Map phase
→ Every job MUST have exactly one Shuffle
→ To do Map → Reduce → Map → Reduce, you need 2 SEPARATE JOBS

SPARK DAG (Directed Acyclic Graph) MODEL:

  User writes:
    logs.filter(...).map(...).join(customers).groupBy(...).count()

  Spark builds internal DAG:

    ┌────────┐     ┌────────┐     ┌──────────┐
    │ Scan   │────→│ Filter │────→│ Map      │──┐
    │ (logs) │     │        │     │          │  │
    └────────┘     └────────┘     └──────────┘  │
                                                 │  ┌──────┐
                                                 ├─→│ Join │──→┌─────────┐──→┌───────┐
    ┌────────────┐                               │  │      │   │ GroupBy │   │ Count │
    │ Scan       │───────────────────────────────┘  └──────┘   └─────────┘   └───────┘
    │ (customers)│
    └────────────┘

  DAG SCHEDULER splits this into STAGES:

  Stage boundaries occur at SHUFFLE points (wide dependencies):
  
  STAGE 1: Scan logs → Filter → Map        (pipelined, no shuffle)
  STAGE 2: Scan customers                   (parallel with Stage 1)
  STAGE 3: Join (requires shuffle of both)  (depends on Stage 1, 2)
  STAGE 4: GroupBy → Count (shuffle)        (depends on Stage 3)

  WITHIN each stage: tasks are PIPELINED
  → Filter does NOT wait for all Scan to finish
  → Each record flows: Scan → Filter → Map → buffer
  → This is like Unix pipes: cat file | grep pattern | cut -f1

  BETWEEN stages: shuffle barrier (like MapReduce shuffle)
  → All tasks in Stage 1 must complete before Stage 3 starts
  → Data is repartitioned and exchanged across network

MECHANIC 5: CATALYST OPTIMIZER (DataFrame/SQL only)
═══════════════════════════════════════════════════════════

  When you write Spark SQL or DataFrame operations:
  
  df = spark.read.parquet("hdfs:///data/shipments")
  result = df.filter(df.status == "delivered") \
              .join(customers, "customer_id") \
              .groupBy("region") \
              .agg({"delivery_time": "avg"})

  Catalyst performs 4 optimization phases:

  PHASE 1: ANALYSIS
  → Resolves column names against schema catalog
  → Validates data types
  → Links to Hive metastore if needed

  PHASE 2: LOGICAL OPTIMIZATION
  → Predicate pushdown: push filter BEFORE join
    BEFORE: Read all shipments → Join → Filter delivered
    AFTER:  Read shipments WHERE delivered → Join (fewer rows!)
  
  → Column pruning: read only needed columns from Parquet
    BEFORE: Read all 50 columns
    AFTER:  Read only (customer_id, status, region, delivery_time)

  → Constant folding: pre-compute constant expressions

  PHASE 3: PHYSICAL PLANNING
  → Chooses join strategy:
    - BroadcastHashJoin if one side < 10MB (configurable)
    - SortMergeJoin for large-large joins
    - ShuffledHashJoin as fallback
  → Chooses aggregation strategy
  → Generates multiple candidate plans, picks cheapest by cost model

  PHASE 4: CODE GENERATION (Tungsten)
  → Generates Java bytecode at runtime
  → Avoids virtual function dispatch overhead
  → Operates on raw memory (off-heap) instead of Java objects
  → Called "Whole-Stage Code Generation"
  → Can process billions of rows/second per core

  THIS IS WHY DataFrame API IS FASTER THAN RDD API:
  → RDD: you write the execution plan
  → DataFrame: Catalyst writes a BETTER execution plan for you

⚠️ COMMON PITFALLS AND FAILURE MODES

PITFALL 1: "Spark job OOM-killed but MapReduce handled same data fine"
→ REASON: MapReduce processes one record at a time, writes to disk
→ Spark tries to hold partitions in memory
→ FIX: Increase spark.executor.memory, reduce partition size,
   use .persist(StorageLevel.MEMORY_AND_DISK) instead of .cache()

PITFALL 2: "Spark job is SLOWER than MapReduce for simple word count"
→ REASON: Spark overhead (Driver, Executors, DAG scheduling)
   exceeds the benefit for trivially simple single-pass jobs
→ FIX: Accept that Spark wins on COMPLEX pipelines, not simple ones
   Single Map + Single Reduce on 100TB → MR may be equal or faster

PITFALL 3: "Job works on 10GB test data but fails on 10TB production"
→ REASON: spark.sql.shuffle.partitions=200 (default)
   10TB / 200 partitions = 50GB per partition → OOM
→ FIX: Set shuffle partitions = data_size_GB * 4
   10TB → 40,000 partitions (each ~256MB)

PITFALL 4: "Broadcast join fails with OOM on driver"
→ REASON: Broadcast collects entire small table to Driver JVM
   then ships to all executors
   If "small" table is 5GB and Driver has 2GB → crash
→ FIX: spark.sql.autoBroadcastJoinThreshold = 10485760 (10MB default)
   Increase only if Driver has enough memory
   Or disable: set to -1 and force SortMergeJoin

PITFALL 5: "GC pauses causing executor lost / heartbeat timeout"
→ REASON: Large Java heap (>32GB) causes long GC pauses
→ FIX: Keep executor memory ≤ 32GB
   Use more executors with less memory each
   Enable G1GC: --conf spark.executor.extraJavaOptions="-XX:+UseG1GC"

PITFALL 6: "Data skew in Spark join — one task takes 10× longer"
→ SAME problem as Project 33 (data skew in MapReduce)
→ Spark-specific fixes:
   - AQE (Adaptive Query Execution) in Spark 3.0+ auto-handles some skew
   - spark.sql.adaptive.enabled=true
   - spark.sql.adaptive.skewJoin.enabled=true
   - Manual: salt the skewed key (same technique as Project 33/35)

PITFALL 7: "Migrated pipeline gives different results than MapReduce"
→ REASON: Floating point aggregation order differs
   MapReduce: deterministic partition assignment
   Spark: may reorder operations due to optimization
→ FIX: Use BigDecimal for financial data
   Compare results with tolerance (0.001%) not exact match
   Sort outputs before comparison

PITFALL 8: "Spark Streaming micro-batch latency too high"
→ REASON: Micro-batch minimum is ~100ms, typical 1-10 seconds
→ FIX: For true millisecond latency, Spark is wrong tool
   Use Kafka Streams or Apache Flink
   Spark Structured Streaming is "near real-time", not "real-time"

public class CleanShipmentsMapper extends Mapper<LongWritable, Text, NullWritable, Text> {
    
    private long validRecords = 0;
    private long invalidRecords = 0;
    
    @Override
    protected void map(LongWritable key, Text value, Context context) 
            throws IOException, InterruptedException {
        
        String line = value.toString();
        
        // Skip header
        if (line.startsWith("shipment_id")) return;
        
        String[] fields = line.split(",");
        
        // Validation rules (same as Project 11 pattern)
        if (fields.length != 8) { invalidRecords++; return; }
        if (fields[4].trim().isEmpty()) { invalidRecords++; return; }  // Missing region
        
        try {
            double cost = Double.parseDouble(fields[6]);
            if (cost < 0) { invalidRecords++; return; }  // Negative cost
        } catch (NumberFormatException e) {
            invalidRecords++; return;
        }
        
        validRecords++;
        context.write(NullWritable.get(), value);
    }
    
    @Override
    protected void cleanup(Context context) throws IOException, InterruptedException {
        context.getCounter("QUALITY", "VALID_RECORDS").increment(validRecords);
        context.getCounter("QUALITY", "INVALID_RECORDS").increment(invalidRecords);
    }
}

class RegionAggMapper extends Mapper<LongWritable, Text, Text, DoubleWritable> {
    private Text regionKey = new Text();
    private DoubleWritable costValue = new DoubleWritable();
    
    @Override
    protected void map(LongWritable key, Text value, Context context)
            throws IOException, InterruptedException {
        String[] fields = value.toString().split(",");
        if (fields.length >= 7 && !fields[0].startsWith("shipment_id")) {
            String region = fields[4];
            double cost = Double.parseDouble(fields[6]);
            regionKey.set(region);
            costValue.set(cost);
            context.write(regionKey, costValue);
        }
    }
}

class RegionAggReducer extends Reducer<Text, DoubleWritable, Text, Text> {
    @Override
    protected void reduce(Text key, Iterable<DoubleWritable> values, Context context)
            throws IOException, InterruptedException {
        double totalCost = 0;
        long count = 0;
        for (DoubleWritable val : values) {
            totalCost += val.get();
            count++;
        }
        double avgCost = totalCost / count;
        context.write(key, new Text(String.format("%d\t%.2f\t%.2f", count, totalCost, avgCost)));
    }
}

    protected void reduce(Text key, Iterable<Text> values, Context context)
            throws IOException, InterruptedException {

        for (Text val : values) {
            String record = val.toString();
            if (record.startsWith("CUST|")) {
                customerInfo = record.substring(5);
            } else if (record.startsWith("SHIP|")) {
                shipments.add(record.substring(5));
            }
        }

            String[] custFields = customerInfo.split(",");
            String custName = custFields.length > 1 ? custFields[1] : "UNKNOWN";
            String tier = custFields.length > 2 ? custFields[2] : "UNKNOWN";
            
            for (String shipment : shipments) {
                context.write(NullWritable.get(), 
                    new Text(shipment + "," + custName + "," + tier));
            }
        }
    }

      → CleanShipmentsMapper with 0 reducers
      → Manual field validation in map() method
      → Counter increments for valid/invalid records
      → WRITES cleaned data to HDFS intermediate path
    
    Spark equivalent:
      → DataFrame filter + validation
      → NO HDFS write — stays in memory as next stage input
      → Accumulators replace Counters
    """

    total_raw = shipments_raw.count()
    print(f"[CLEAN] Raw records: {total_raw}")

    # ── Apply cleaning rules (same logic as MapReduce mapper) ──

    shipments_cleaned = shipments_raw \
        .filter(
            # Rule 1: Region must not be null or empty
            (F.col("region").isNotNull()) &
            (F.trim(F.col("region")) != "") &

            # Rule 2: Cost must be positive
            (F.col("cost") > 0) &
            (F.col("cost").isNotNull()) &

            # Rule 3: Status must be valid
            (F.col("status").isin(
                "shipped", "in_transit", "delivered", "returned", "lost"
            )) &

            # Rule 4: Shipment ID must not be null
            (F.col("shipment_id").isNotNull())
        ) \
        .withColumn(
            # Normalize: trim whitespace from region
            "region", F.trim(F.upper(F.col("region")))
        ) \
        .withColumn(
            # Parse timestamp string to proper timestamp type
            "event_timestamp",
            F.to_timestamp(F.col("timestamp"), "yyyy-MM-dd'T'HH:mm:ss")
        ) \
        .drop("timestamp")  # Drop original string column

    total_clean = shipments_cleaned.count()
    invalid_count = total_raw - total_clean

    print(f"[CLEAN] Valid records:   {total_clean}")
    print(f"[CLEAN] Invalid records: {invalid_count}")
    print(f"[CLEAN] Rejection rate:  {(invalid_count/total_raw)*100:.1f}%")

    # ═══════════════════════════════════════════════════════
    # KEY DIFFERENCE FROM MAPREDUCE:
    # In MR: cleaned data is WRITTEN to HDFS here
    #         hdfs:///data/logistics/mr_cleaned/
    #         This takes minutes for TB-scale data
    #
    # In Spark: cleaned data stays as in-memory DataFrame
    #           NO disk I/O between stages
    #           This is WHERE Spark gains its speed advantage
    # ═══════════════════════════════════════════════════════

    return shipments_cleaned

def enrich_with_customers(shipments_cleaned, customers):
    """
    MapReduce equivalent:
      → MultipleInputs with ShipmentTagMapper + CustomerTagMapper
      → Each mapper tags records with source prefix ("SHIP|", "CUST|")
      → JoinReducer receives all records for same customer_id
      → Separates customer info from shipment records
      → Emits enriched records
      → READS cleaned data FROM HDFS (another full scan!)
      → WRITES enriched data TO HDFS (another full write!)
    
    Spark equivalent:
      → Single .join() call
      → Spark automatically chooses join strategy:
        - BroadcastHashJoin if customers < 10MB (our case!)
        - SortMergeJoin for large-large joins
      → NO HDFS read/write between stages
    """

    # ── Check if broadcast join will be used ──
    # customers.csv is small (~300KB for 10,000 records)
    # Spark will automatically broadcast it to all executors
    # This is equivalent to Distributed Cache Map-Side Join (Project 16)
    # but Spark decides this AUTOMATICALLY via Catalyst optimizer

    print(f"[JOIN] Customer table size: {customers.count()} rows")
    print(f"[JOIN] Strategy: Spark will auto-select BroadcastHashJoin")

    enriched = shipments_cleaned.join(
        customers,
        on="customer_id",
        how="inner"  # Only keep shipments with matching customers
    )

    # ── Verify join result ──
    print(f"[JOIN] Enriched records: {enriched.count()}")
    print(f"[JOIN] Sample enriched record:")
    enriched.show(3, truncate=False)

    # ── Show the execution plan — see Catalyst's join decision ──
    print("[JOIN] Physical execution plan:")
    enriched.explain(mode="formatted")
    # Look for "BroadcastHashJoin" in the output
    # This means Spark shipped the small table to all executors
    # ZERO shuffle for the large shipments table!

    # ═══════════════════════════════════════════════════════
    # COMPARISON: MapReduce Reduce-Side Join vs Spark
    #
    # MapReduce:
    #   → BOTH tables fully shuffled across network
    #   → All records with same key go to same reducer
    #   → Reducer must buffer all records for one key in memory
    #   → If one customer has 1M shipments → reducer OOM
    #   → Requires: 2 custom Mappers + 1 Reducer + MultipleInputs

    #
    # Spark (broadcast):
    #   → Small table broadcast to all executors (once, ~300KB)
    #   → Large table NOT shuffled at all
    #   → Hash lookup per record — O(1) per join
    #   → No OOM risk from key skew
    #   → Lines of code: 1
    # ═══════════════════════════════════════════════════════

    return enriched

# STAGE 4: REGION AGGREGATION
# (Replaces: MapReduce Job 2 — RegionAggMapper + RegionAggReducer)
# (Pattern from Project 5)
# ══════════════════════════════════════════════════════════════

def aggregate_by_region(enriched):
    """
    MapReduce equivalent:
      → RegionAggMapper: emit(region, cost) for each record
      → Combiner: local sum/count per region per mapper
      → Shuffle: all records for same region to same reducer
      → RegionAggReducer: compute total_cost, count, avg_cost
      → READS enriched data FROM HDFS
      → WRITES aggregated data TO HDFS
    
    Spark equivalent:
      → .groupBy("region").agg(...)
      → Spark internally applies partial aggregation (like Combiner)
      → Then shuffles partial results
      → Then final aggregation
      → All decided by Catalyst optimizer
    """

    region_metrics = enriched \
        .groupBy("region") \
        .agg(
            F.count("*").alias("shipment_count"),
            F.round(F.sum("cost"), 2).alias("total_cost"),
            F.round(F.avg("cost"), 2).alias("avg_cost"),
            F.round(F.min("cost"), 2).alias("min_cost"),
            F.round(F.max("cost"), 2).alias("max_cost"),
            F.countDistinct("customer_id").alias("unique_customers"),

            # Additional metrics that would need ANOTHER MapReduce job
            F.sum(F.when(F.col("status") == "delivered", 1).otherwise(0))
                .alias("delivered_count"),
            F.round(
                F.avg(F.when(F.col("delivery_days") > 0, F.col("delivery_days"))),
                1
            ).alias("avg_delivery_days")
        ) \
        .withColumn(
            "delivery_rate",
            F.round(F.col("delivered_count") / F.col("shipment_count") * 100, 1)
        ) \
        .orderBy(F.desc("shipment_count"))

    print("[AGGREGATE] Region metrics:")
    region_metrics.show(truncate=False)

    # ═══════════════════════════════════════════════════════
    # KEY INSIGHT: In MapReduce, computing delivery_rate
    # would require EITHER:
    #   a) A separate MapReduce job for delivered counts
    #   b) Complex composite values in the reducer
    #
    # In Spark: it's just another column expression
    # ALL aggregations computed in a SINGLE PASS
    # ═══════════════════════════════════════════════════════

    return region_metrics

# (Replaces: MapReduce Job with Secondary Sort from Project 19)

def top_products_per_region(enriched, top_n=5):
    """
    MapReduce equivalent:
      → CompositeKey: (region, product_count) with custom comparators
      → Custom Partitioner to ensure same region goes to same reducer
      → Custom GroupingComparator to group by region only
      → Custom SortComparator to sort by count descending within group
      → Lines of Java code: ~200
      → Requires understanding of 3 separate comparator contracts
    
    Spark equivalent:
      → Window function with row_number()
      → 5 lines of code
      → Catalyst handles all partitioning and sorting internally
    """

    # Step 1: Count shipments per product per region
    product_region_counts = enriched \
        .groupBy("region", "product_id") \
        .agg(
            F.count("*").alias("product_count"),
            F.round(F.sum("cost"), 2).alias("product_revenue")
        )

    # Step 2: Rank products within each region using Window function
    # This is the Spark equivalent of Secondary Sort
    window_spec = Window \
        .partitionBy("region") \
        .orderBy(F.desc("product_count"))

    top_products = product_region_counts \
        .withColumn("rank", F.row_number().over(window_spec)) \
        .filter(F.col("rank") <= top_n) \
        .orderBy("region", "rank")

    print(f"[TOP PRODUCTS] Top {top_n} products per region:")
    top_products.show(50, truncate=False)

    # ═══════════════════════════════════════════════════════
    # COMPARISON: Secondary Sort complexity
    #
    # MapReduce (Project 19):
    #   → CompositeKey class: 50 lines
    #   → Custom Partitioner: 20 lines
    #   → GroupingComparator: 20 lines
    #   → SortComparator: 20 lines
    #   → Mapper: 30 lines
    #   → Reducer: 30 lines
    #   → Driver: 30 lines
    #   → TOTAL: ~200 lines of Java
    #
    # Spark:
    #   → Window function: 5 lines
    #   → Catalyst generates optimal execution plan
    #   → TOTAL: 5 lines of Python
    # ═══════════════════════════════════════════════════════

    return top_products

# STAGE 7: CACHING STRATEGY — THE CORE SPARK ADVANTAGE
# ══════════════════════════════════════════════════════════════

def demonstrate_caching_advantage(spark, shipments_raw, customers):
    """
    This demonstrates WHY Spark is fundamentally faster for
    multi-stage pipelines.
    
    MapReduce: Each job reads from HDFS, writes to HDFS
    Spark: Cache intermediate results in memory, reuse across stages
    """

    print("\n" + "=" * 70)
    print("CACHING DEMONSTRATION: Memory vs Disk")
    print("=" * 70)

    # ── WITHOUT CACHING (simulates MapReduce behavior) ──
    spark.catalog.clearCache()

    start_no_cache = time.time()

    # Each of these triggers a FULL re-read from HDFS + re-computation
    shipments_cleaned = clean_shipments(shipments_raw)

    # Query 1: Count by status (full pipeline runs)
    status_counts = shipments_cleaned.groupBy("status").count()
    status_counts.show()

    # Query 2: Count by region (full pipeline runs AGAIN from HDFS!)
    region_counts = shipments_cleaned.groupBy("region").count()
    region_counts.show()

    # Query 3: Average cost (full pipeline runs AGAIN from HDFS!)
    avg_cost = shipments_cleaned.agg(F.avg("cost")).collect()
    print(f"Average cost: {avg_cost[0][0]:.2f}")

    time_no_cache = time.time() - start_no_cache

    # ── WITH CACHING (Spark's advantage) ──
    spark.catalog.clearCache()

    start_with_cache = time.time()

    shipments_cleaned = clean_shipments(shipments_raw)

    # CACHE the cleaned DataFrame in memory
    # This is the KEY operation that eliminates HDFS re-reads
    shipments_cleaned.cache()

    # First query triggers computation AND caches result
    status_counts = shipments_cleaned.groupBy("status").count()
    status_counts.show()

    # Second query reads from MEMORY — no HDFS, no re-cleaning
    region_counts = shipments_cleaned.groupBy("region").count()
    region_counts.show()

    # Third query also reads from MEMORY
    avg_cost = shipments_cleaned.agg(F.avg("cost")).collect()
    print(f"Average cost: {avg_cost[0][0]:.2f}")

    time_with_cache = time.time() - start_with_cache

    print(f"\n[CACHE RESULTS]")
    print(f"  Without caching: {time_no_cache:.2f} seconds")
    print(f"  With caching:    {time_with_cache:.2f} seconds")
    print(f"  Speedup:         {time_no_cache/time_with_cache:.1f}x")
    print(f"\n  At TB scale with 5+ downstream queries,")
    print(f"  this speedup becomes 10-100x")

    # Unpersist to free memory
    shipments_cleaned.unpersist()

    return time_no_cache, time_with_cache

# STAGE 9: SIDE-BY-SIDE RDD API vs DATAFRAME API
# (Understanding WHEN to use each)
# ══════════════════════════════════════════════════════════════

def demonstrate_rdd_vs_dataframe(spark, shipments_raw):
    """
    Spark has TWO APIs:
    1. RDD API — low-level, functional, like writing MapReduce
    2. DataFrame API — high-level, SQL-like, optimizer-enhanced
    
    Rule: Use DataFrame API by default
    Use RDD only for custom logic that cannot be expressed in SQL
    """

    print("\n" + "=" * 70)
    print("RDD API vs DATAFRAME API — Side by Side")
    print("=" * 70)

    # ════════════════════════════════
    # APPROACH 1: RDD API (low-level)
    # Closest to MapReduce thinking
    # ════════════════════════════════

    start_rdd = time.time()

    rdd_result = shipments_raw.rdd \
        .filter(lambda row: row.region is not None and row.region.strip() != "") \
        .filter(lambda row: row.cost is not None and row.cost > 0) \
        .map(lambda row: (row.region.strip().upper(), (row.cost, 1))) \
        .reduceByKey(lambda a, b: (a[0] + b[0], a[1] + b[1])) \
        .mapValues(lambda v: (v[0], v[1], round(v[0] / v[1], 2))) \
        .sortBy(lambda x: x[1][1], ascending=False) \
        .collect()

    time_rdd = time.time() - start_rdd

    print("\n[RDD API] Region aggregation:")
    print(f"{'Region':<12} {'Total Cost':>12} {'Count':>8} {'Avg Cost':>10}")
    print("-" * 44)
    for region, (total, count, avg) in rdd_result:
        print(f"{region:<12} {total:>12.2f} {count:>8} {avg:>10.2f}")
    print(f"[RDD API] Time: {time_rdd:.2f}s")

    # ════════════════════════════════
    # APPROACH 2: DataFrame API (high-level)
    # Catalyst optimizer generates efficient plan
    # ════════════════════════════════

    start_df = time.time()

    df_result = shipments_raw \
        .filter(
            (F.col("region").isNotNull()) &
            (F.trim(F.col("region")) != "") &
            (F.col("cost") > 0)
        ) \
        .withColumn("region", F.trim(F.upper(F.col("region")))) \
        .groupBy("region") \
        .agg(
            F.round(F.sum("cost"), 2).alias("total_cost"),
            F.count("*").alias("count"),
            F.round(F.avg("cost"), 2).alias("avg_cost")
        ) \
        .orderBy(F.desc("count"))

    df_result.show()
    time_df = time.time() - start_df

    print(f"[DataFrame API] Time: {time_df:.2f}s")

    # ════════════════════════════════
    # APPROACH 3: Spark SQL (pure SQL)
    # Same Catalyst optimizer as DataFrame
    # ════════════════════════════════

    start_sql = time.time()

    shipments_raw.createOrReplaceTempView("shipments")

    sql_result = spark.sql("""
        SELECT 
            UPPER(TRIM(region)) as region,
            ROUND(SUM(cost), 2) as total_cost,
            COUNT(*) as count,
            ROUND(AVG(cost), 2) as avg_cost
        FROM shipments
        WHERE region IS NOT NULL 
          AND TRIM(region) != ''
          AND cost > 0
        GROUP BY UPPER(TRIM(region))
        ORDER BY count DESC
    """)

    sql_result.show()
    time_sql = time.time() - start_sql

    print(f"[Spark SQL] Time: {time_sql:.2f}s")

    # ── Comparison ──
    print("\n" + "=" * 50)
    print("PERFORMANCE COMPARISON:")
    print(f"  RDD API:       {time_rdd:.2f}s (no optimizer)")
    print(f"  DataFrame API: {time_df:.2f}s (Catalyst optimized)")
    print(f"  Spark SQL:     {time_sql:.2f}s (Catalyst optimized)")
    print(f"\n  DataFrame & SQL use Catalyst → same execution plan")
    print(f"  RDD bypasses optimizer → typically 2-10x slower")
    print("=" * 50)

# STAGE 10: BROADCAST VARIABLE (Replaces Distributed Cache)
# (Pattern from Project 15 and Project 16)

def demonstrate_broadcast_join(spark, shipments_cleaned, customers):
    """
    MapReduce Distributed Cache (Project 15):
      → addCacheFile() in Driver
      → Load file in Mapper.setup()
      → Build HashMap manually
      → Lookup in map() method
      → Works only for small files
    
    Spark Broadcast Variable:
      → Automatic for small DataFrames in joins
      → Manual via sc.broadcast() for custom logic
      → Serialized ONCE, sent to all executors
      → Stored in executor memory for all tasks
    """

    print("\n" + "=" * 70)
    print("BROADCAST JOIN vs SHUFFLE JOIN")
    print("=" * 70)

    # ── Method 1: Automatic broadcast (Catalyst decides) ──
    # Spark checks: is customers < autoBroadcastJoinThreshold (10MB)?
    # If yes → BroadcastHashJoin
    # If no  → SortMergeJoin

    auto_join = shipments_cleaned.join(customers, "customer_id")
    print("\n[AUTO] Execution plan:")
    auto_join.explain(mode="simple")
    # Look for "BroadcastHashJoin" in output

    # ── Method 2: Force broadcast (when you KNOW table is small) ──
    forced_broadcast = shipments_cleaned.join(
        F.broadcast(customers),  # Explicit broadcast hint
        "customer_id"
    )
    print("\n[FORCED BROADCAST] Execution plan:")
    forced_broadcast.explain(mode="simple")

    # ── Method 3: Manual broadcast for RDD-level custom logic ──
    # Equivalent to Project 15 Distributed Cache + HashMap pattern
    sc = spark.sparkContext

    # Collect small table to driver, then broadcast
    customer_map = {
        row.customer_id: (row.customer_name, row.tier)
        for row in customers.collect()
    }
    broadcast_customers = sc.broadcast(customer_map)

    # Use in RDD transformation (like Mapper with DistributedCache)
    def enrich_record(row):
        cust_info = broadcast_customers.value.get(row.customer_id, ("UNKNOWN", "UNKNOWN"))
        return (
            row.shipment_id,
            row.customer_id,
            cust_info[0],  # customer_name
            cust_info[1],  # tier
            row.region,
            row.cost
        )

    enriched_rdd = shipments_cleaned.rdd.map(enrich_record)
    print(f"\n[MANUAL BROADCAST] Sample enriched records:")
    for record in enriched_rdd.take(5):
        print(f"  {record}")

    # ═══════════════════════════════════════════════════════
    # BROADCAST MECHANICS:
    #
    # Driver JVM:
    #   customer_map dict (~300KB) serialized via Torrent broadcast
    #   → Sent to BlockManager on each executor
    #   → Each executor deserializes ONCE
    #   → All tasks on that executor share the SAME object
    #
    # vs MapReduce Distributed Cache:
    #   → File copied to EACH task's local dir via HDFS
    #   → Each Mapper.setup() reads and parses the file
    #   → If 1000 mappers → file parsed 1000 times
    #   → Spark: parsed ONCE per executor (not per task)
    # ═══════════════════════════════════════════════════════

# STAGE 11: ACCUMULATOR (Replaces MapReduce Counters)
# (Pattern from Project 7)
# ══════════════════════════════════════════════════════════════

def demonstrate_accumulators(spark, shipments_raw):
    """
    MapReduce Counters (Project 7):
      → context.getCounter("GROUP", "NAME").increment(1)
      → Read after job completes via job.getCounters()
    
    Spark Accumulators:
      → sc.longAccumulator("name")
      → Increment inside transformations
      → Read on Driver after action completes
    """

    sc = spark.sparkContext

    # Define accumulators (like defining Counter groups)
    valid_counter = sc.longAccumulator("valid_records")
    invalid_cost_counter = sc.longAccumulator("invalid_cost")
    missing_region_counter = sc.longAccumulator("missing_region")
    total_counter = sc.longAccumulator("total_records")

    def validate_and_count(row):
        total_counter.add(1)

        if row.region is None or row.region.strip() == "":
            missing_region_counter.add(1)
            return False

        if row.cost is None or row.cost <= 0:
            invalid_cost_counter.add(1)
            return False

        valid_counter.add(1)
        return True

    # Apply validation with counting
    validated_rdd = shipments_raw.rdd.filter(validate_and_count)

    # Trigger execution (accumulators only update during actions)
    validated_count = validated_rdd.count()

    # Read accumulator values (like reading MR counters after job)
    print("\n[ACCUMULATORS] Quality Counters (like MR Job Counters):")
    print(f"  Total records:    {total_counter.value}")
    print(f"  Valid records:    {valid_counter.value}")
    print(f"  Invalid cost:     {invalid_cost_counter.value}")
    print(f"  Missing region:   {missing_region_counter.value}")
    print(f"  Validated count:  {validated_count}")

    # ═══════════════════════════════════════════════════════
    # WARNING: Accumulators in Spark have a subtlety:
    # → If a task is retried (due to failure or speculation),
    #   the accumulator may be incremented MULTIPLE TIMES
    # → MapReduce counters are EXACTLY-ONCE per record
    # → For exact counts, use DataFrame .count() after filter
