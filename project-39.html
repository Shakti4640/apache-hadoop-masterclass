WHAT HAPPENS WITHOUT QUEUE MANAGEMENT:

   Monday 9:00 AM:
   → Data Science intern submits a job requesting 200 cores, 1.6TB RAM
   → Job consumes 50% of entire cluster
   → Business Analytics team submits dashboard queries
   → ALL queries queued behind the ML job
   → Dashboard shows "Loading..." for 3 hours
   → VP of Sales calls your manager

   Tuesday 8:00 PM:
   → Nightly ETL pipeline starts (200 jobs)
   → Marketing team launches campaign analysis (heavy)
   → Both compete for same resources
   → ETL finishes at 9 AM instead of 6 AM
   → Morning reports delayed 3 hours
   → CEO notices

   Wednesday 2:00 PM:
   → QA engineer runs test job that accidentally requests 300 cores
   → Cluster is saturated
   → Finance team's regulatory job cannot start
   → Compliance deadline missed
   → Legal department involved

   COST OF NO QUEUE MANAGEMENT:
   → Political battles between teams ("my job is more important")
   → Unpredictable job completion times
   → Production SLA violations
   → Wasted cluster capacity (some teams hoard, others starve)
   → Engineer time spent manually killing jobs

THE QUESTION:
→ How do we guarantee each team gets fair access?
→ How do we protect critical workloads from being starved?
→ How do we prevent any single team from consuming the entire cluster?
→ How do we allow elastic scaling when others are idle?
→ How do we enforce these rules automatically WITHOUT human intervention?

🔍 HOW TO APPROACH IT

STEP 1: Understand YARN's role (from Project 22 — not repeated here)

   YARN ResourceManager has a SCHEDULER component
   → The scheduler decides WHICH pending application gets resources NEXT
   → It does NOT monitor or manage applications — only ALLOCATES
   → Three built-in schedulers:

💡 HOW CAN WE SOLVE IT

SOLUTION: CAPACITY SCHEDULER WITH HIERARCHICAL QUEUES

THE KEY CONCEPTS:

1. GUARANTEED CAPACITY (minimum)
   → Each queue is GUARANTEED this percentage of the cluster
   → Even if all other queues are submitting jobs
   → This is the "floor" — your queue WILL get at least this much

2. MAXIMUM CAPACITY (elastic ceiling)
   → If OTHER queues are idle, your queue can use MORE than its guarantee
   → Up to this maximum percentage
   → Resources are RETURNED when other queues need them
   → This prevents waste — idle capacity is borrowed, not lost

3. PREEMPTION
   → When a queue needs its guaranteed resources back
   → But another queue is using them (borrowed via elasticity)
   → The scheduler can KILL tasks in the borrowing queue
   → To return resources to the queue that owns them
   → CRITICAL for SLA enforcement

4. USER LIMITS
   → Within a queue, limit how much ONE user can consume
   → Prevents a single intern from hogging the team's entire allocation
   → Example: Each user can use at most 50% of their queue

5. APPLICATION LIMITS
   → Maximum number of concurrent applications per queue
   → Prevents a runaway script from submitting 10,000 jobs
   → Controls scheduling overhead

6. QUEUE ACLs (Access Control Lists)
   → WHO can submit jobs to which queue
   → WHO can administer (kill jobs in) which queue
   → Prevents teams from accidentally using wrong queue
   → Enforces organizational boundaries

✅ WHAT COULD BE DONE

COMPREHENSIVE MULTI-TENANT STRATEGY:

1. CONFIGURE CAPACITY SCHEDULER WITH QUEUE HIERARCHY
   → Define queues matching organizational structure
   → Set guaranteed and maximum capacities
   → Enable elastic sharing for efficient utilization

2. IMPLEMENT QUEUE ACLs
   → Each team can ONLY submit to their queue
   → Admins can submit to ANY queue
   → Queue admins can kill jobs in their queue only

3. CONFIGURE USER LIMITS WITHIN QUEUES
   → No single user consumes more than X% of their queue
   → Minimum user limit percentage for fairness within team

4. ENABLE PREEMPTION FOR CRITICAL QUEUES
   → Finance queue: preemption enabled (regulatory compliance)
   → ETL queue: preemption enabled (SLA-bound)
   → DataScience queue: preemptable (can be killed to free resources)

5. SET APPLICATION LIMITS
   → QA queue: max 5 concurrent applications
   → Production queues: higher limits
   → Prevents runaway submission scripts

6. CONFIGURE QUEUE ORDERING POLICY
   → Within each queue: FIFO or Fair ordering
   → Business analytics: Fair (many small queries share resources)
   → ETL: FIFO (pipeline order matters)

7. MONITOR AND ALERT
   → Track queue utilization via ResourceManager UI (Project 7)
   → Alert when queues consistently hit maximum capacity
   → Alert when preemption events exceed threshold

✅ WHAT SHOULD DO

1. DESIGN QUEUE HIERARCHY TO MATCH ORGANIZATION
   → Queues should mirror team/department structure
   → Maximum 3 levels deep (root → department → team)
   → Deeper hierarchies become hard to reason about

2. ALWAYS SET MAXIMUM CAPACITY > MINIMUM CAPACITY
   → This enables elastic sharing
   → A queue with max = min CANNOT borrow idle resources
   → Wastes cluster capacity when that queue is idle

3. GUARANTEE THAT MINIMUM CAPACITIES SUM TO 100% AT EACH LEVEL
   → Children of "production" must sum to 100% of production
   → Children of "root" must sum to 100%
   → This is a HARD REQUIREMENT — scheduler will reject otherwise

4. ENABLE PREEMPTION CAREFULLY
   → Only for queues with hard SLAs
   → Set reasonable preemption timeouts (not instant)
   → Monitor preemption events — too many = queue design problem

5. USE QUEUE PLACEMENT RULES
   → Auto-route jobs to correct queue based on user/group
   → Prevents teams from needing to specify queue manually
   → Reduces human error

6. TEST QUEUE CONFIGURATION BEFORE PRODUCTION
   → Use Project 38 testing techniques on a test cluster
   → Verify that limits are actually enforced
   → Simulate resource contention scenarios

7. SET UP QUEUE-LEVEL MONITORING
   → Per-queue: pending applications, used resources, wait time
   → Alerting: If business queue wait time > 5 minutes
   → Capacity planning: If any queue consistently at 95%+

8. DOCUMENT QUEUE OWNERSHIP AND SLAs
   → Each queue has a designated owner team
   → Each queue has documented SLA expectations
   → Escalation path when SLAs are violated

🚫 WHAT SHOULD NOT DO

1. DO NOT USE FIFO SCHEDULER IN PRODUCTION
   → FIFO = one queue, first come first served
   → One large job blocks EVERYTHING behind it
   → Zero isolation between teams
   → Acceptable ONLY for single-user development clusters

2. DO NOT CREATE A QUEUE FOR EVERY INDIVIDUAL USER
   → Queue overhead is non-trivial for the scheduler
   → 200 queues = slow scheduling decisions
   → Use user-limits WITHIN team queues instead
   → Maximum recommended: 50-100 queues

3. DO NOT SET MAXIMUM CAPACITY TO 100% FOR ALL QUEUES
   → If every queue can use 100%, guarantees become meaningless
   → When all queues are active, no queue has priority
   → Set maximums thoughtfully based on burst needs

4. DO NOT ENABLE PREEMPTION EVERYWHERE
   → Preemption kills running tasks — work is LOST
   → Tasks must restart from beginning (or last checkpoint)
   → Enable ONLY for queues with hard SLAs
   → Too much preemption = cluster thrashing

5. DO NOT CHANGE QUEUE CONFIGURATION DURING PEAK HOURS
   → Configuration changes require ResourceManager refresh
   → Can cause brief scheduling disruption
   → Apply changes during maintenance windows
   → Test in staging first (Project 38)

6. DO NOT IGNORE THE "DEFAULT" QUEUE
   → Jobs submitted without specifying a queue go to "default"
   → If default has large capacity → accidental resource hogging
   → Set default queue to minimal capacity (5-10%)
   → Or disable it entirely with ACLs

7. DO NOT RELY ON QUEUE NAMES FOR SECURITY
   → Queue ACLs enforce access control
   → But users can specify any queue name in job configuration
   → Without ACLs, any user can submit to any queue
   → ACLs are the ENFORCEMENT mechanism, not queue names

8. DO NOT CONFUSE CAPACITY SCHEDULER WITH FAIR SCHEDULER CONFIGURATION
   → They use DIFFERENT configuration files
   → Capacity: capacity-scheduler.xml
   → Fair: fair-scheduler.xml
   → They use DIFFERENT property names
   → Mixing them causes silent misconfigurations

🚫 WHEN NOT TO USE

1. DO NOT USE MULTI-TENANT QUEUE MANAGEMENT WHEN:
   → Single team uses the cluster exclusively
   → FIFO is sufficient — no contention exists
   → Overhead of queue management adds no value

2. DO NOT USE CAPACITY SCHEDULER WHEN:
   → Workloads are highly unpredictable in size and timing
   → You need fine-grained fair sharing within queues
   → Fair Scheduler handles dynamic workloads better

3. DO NOT USE FAIR SCHEDULER WHEN:
   → You need strict capacity guarantees with percentages
   → Deep hierarchical queue structure is required
   → Organization demands "this team gets exactly 30%"

4. DO NOT USE PREEMPTION WHEN:
   → Jobs are not idempotent (cannot safely restart killed tasks)
   → Task startup cost is very high (long initialization)
   → All workloads are equally important (no priority)
   → Cluster utilization is below 70% (no resource pressure)

5. DO NOT USE COMPLEX QUEUE HIERARCHIES WHEN:
   → You have fewer than 3 teams
   → All teams have similar workloads and priorities
   → Flat queue structure is simpler to manage and debug

6. DO NOT USE ON-PREMISE QUEUE MANAGEMENT WHEN:
   → Cloud deployment allows separate clusters per team
   → Autoscaling eliminates resource contention
   → (Bridge to Project 49: Cloud Hadoop)

MECHANIC 1: CAPACITY SCHEDULER — ALLOCATION DECISION FLOW
═══════════════════════════════════════════════════════════

When a NodeManager sends a HEARTBEAT to ResourceManager
(from Project 22 — YARN Architecture):

Step 1 — NodeManager heartbeat arrives
   "I have 8GB free, 4 vcores available"

Step 2 — Scheduler checks which queues have PENDING requests
   Queue "etl": 3 pending containers (24GB, 12 vcores needed)
   Queue "business": 1 pending container (4GB, 2 vcores needed)
   Queue "datascience": 5 pending containers (40GB, 20 vcores needed)

Step 3 — Scheduler calculates CURRENT USAGE vs CAPACITY

   ┌────────────────┬──────────┬──────────┬──────────────────────┐
   │ QUEUE          │ CAPACITY │ CURRENT  │ STATUS               │
   │                │ (min%)   │ USAGE %  │                      │
   ├────────────────┼──────────┼──────────┼──────────────────────┤
   │ etl            │ 28%      │ 22%      │ UNDER capacity ← ★  │
   │ business       │ 14.7%    │ 14.7%    │ AT capacity          │
   │ datascience    │ 20%      │ 35%      │ OVER capacity        │
   │                │          │          │ (borrowed from idle) │
   └────────────────┴──────────┴──────────┴──────────────────────┘

Step 4 — Scheduler picks queue MOST UNDER its capacity
   → "etl" is most under-served (22% vs 28% guarantee)
   → ETL queue gets FIRST priority for this heartbeat's resources

Step 5 — Within chosen queue, pick application
   → If queue policy is FIFO: oldest pending application
   → If queue policy is FAIR: application with least resources

Step 6 — Allocate container
   → ResourceManager tells NodeManager:
     "Start container for application_1234 with 4GB, 2 vcores"
   → NodeManager launches the container

Step 7 — Update accounting
   → ETL queue usage: 22% → 23.2%
   → Next heartbeat: re-evaluate which queue is most under-served

THIS HAPPENS ON EVERY HEARTBEAT (~1 second intervals)

MECHANIC 2: ELASTIC SHARING — HOW BORROWING WORKS
═══════════════════════════════════════════════════════════

SCENARIO: 2:00 AM — Only ETL queue has active jobs

   ┌────────────────┬──────────┬──────────┬──────────────────┐
   │ QUEUE          │ MIN      │ MAX      │ ACTIVE JOBS      │
   ├────────────────┼──────────┼──────────┼──────────────────┤
   │ etl            │ 28%      │ 60%      │ 45 jobs running  │
   │ business       │ 14.7%    │ 40%      │ 0 jobs           │
   │ marketing      │ 9.8%     │ 30%      │ 0 jobs           │
   │ finance        │ 17.5%    │ 50%      │ 0 jobs           │
   │ datascience    │ 20%      │ 40%      │ 0 jobs           │
   │ qa             │ 10%      │ 15%      │ 0 jobs           │
   └────────────────┴──────────┴──────────┴──────────────────┘

   WHAT HAPPENS:
   → ETL is the ONLY queue with pending work
   → Business, marketing, finance, datascience, qa are all idle
   → Their guaranteed capacity (72%) is UNUSED
   → Capacity Scheduler lets ETL BORROW this idle capacity
   → ETL can grow up to its MAXIMUM of 60%
   → Remaining 40% stays truly idle (no queue can use > its max)

   CLUSTER USAGE AT 2:00 AM:
   → ETL: using 60% (its maximum)
   → Others: 0%
   → Cluster utilization: 60% (better than 28% without elasticity)

   NOW 9:00 AM — Business Analytics team starts submitting:
   → Business queue has pending jobs but 0% allocated
   → Scheduler sees business is UNDER capacity (0% vs 14.7% guarantee)
   → NEW containers go to business queue FIRST
   → As ETL tasks COMPLETE, their resources are NOT re-given to ETL
   → Instead, freed resources go to business (most under-served)
   → ETL gradually shrinks from 60% back toward 28%
   → Business gradually grows toward 14.7%

   THIS IS "NATURAL" RESOURCE RETURN — no tasks killed
   Resources return as tasks naturally finish

MECHANIC 3: PREEMPTION — FORCED RESOURCE RETURN
═══════════════════════════════════════════════════════════

SCENARIO: Finance submits urgent regulatory job at month-end

   PROBLEM:
   → Finance queue guaranteed: 17.5%
   → Finance queue current usage: 0%
   → DataScience queue current usage: 35% (borrowed from idle queues)
   → DataScience tasks are LONG RUNNING (2-3 hours each)
   → Natural completion won't free resources for HOURS
   → Finance NEEDS its 17.5% NOW

   WITHOUT PREEMPTION:
   → Finance waits until DataScience tasks finish naturally
   → Could be HOURS
   → Regulatory deadline missed

   WITH PREEMPTION:
   Step 1 — Scheduler detects finance is under guaranteed capacity
   Step 2 — Scheduler identifies which queues are OVER their capacity
            DataScience is at 35% but guaranteed only 20%
            → 15% is "borrowed" capacity → eligible for preemption
   Step 3 — Preemption monitor waits for preemption delay (e.g., 30 seconds)
            Gives over-capacity queues chance to release voluntarily
   Step 4 — If resources still not returned → scheduler sends KILL signal
            Specific containers in DataScience queue are terminated
   Step 5 — Freed resources allocated to Finance queue
   Step 6 — Killed DataScience tasks are RE-SCHEDULED
            They go back to pending state in DataScience queue
            Will run when resources become available (within DataScience's share)

   PREEMPTION FLOW:
   ┌──────────┐     ┌────────────┐     ┌──────────────┐     ┌──────────┐
   │ Finance  │     │ Scheduler  │     │ DataScience  │     │ Finance  │
   │ submits  │────→│ detects    │────→│ container    │────→│ gets its │
   │ job      │     │ under-     │     │ KILLED       │     │ 17.5%    │
   │          │     │ served     │     │ (preempted)  │     │ back     │
   └──────────┘     └────────────┘     └──────────────┘     └──────────┘
                         │
                    30s delay
                    (grace period)

MECHANIC 5: USER LIMITS WITHIN A QUEUE
═══════════════════════════════════════════════════════════

SCENARIO: Business Analytics queue has 3 analysts

   WHAT THIS MEANS:

   → If 2 analysts are active → each gets 50% (235GB)
   → If 3 analysts are active → each gets 33% (155GB)
   → If 4 analysts are active → each gets 25% (117GB)

   THE "minimum" means: each user gets AT LEAST this percentage
   when queue is fully contested

   user-limit-factor controls the MAXIMUM a single user can use:
   → user-limit-factor = 1.0 → user can use up to 100% of queue
   → user-limit-factor = 0.5 → user can use up to 50% of queue
   → user-limit-factor = 2.0 → user can use up to 200% of queue
     (if queue has borrowed capacity from idle queues)

⚠️ COMMON PITFALLS AND FAILURE MODES

PITFALL 1: "Queue capacities don't sum to 100%"
→ SYMPTOM: ResourceManager fails to start or scheduler rejects config
→ CAUSE: Children of a parent queue must sum to EXACTLY 100%
→ Example: production(70) + research(20) + development(10) = 100% ✅
→ Example: production(70) + research(20) + development(15) = 105% ❌
→ FIX: Always verify sums at every level of hierarchy

PITFALL 2: "Jobs stuck in ACCEPTED state, never start running"
→ SYMPTOM: Job shows in ResourceManager UI as ACCEPTED but 0% progress
→ CAUSE 1: Queue is at maximum capacity and cannot borrow
→ CAUSE 2: User limit reached within the queue
→ CAUSE 3: ACL rejects the user but error message is swallowed
→ DEBUG: Check ResourceManager UI → Scheduler tab → queue utilization
→ DEBUG: Check application logs for "Queue does not have permission"

PITFALL 3: "Preemption is killing too many tasks"
→ SYMPTOM: DataScience jobs never finish — constantly restarted
→ CAUSE: Preemption timeout too aggressive (e.g., 5 seconds)
→ CAUSE: DataScience queue max-capacity too high → borrows too much
→ FIX: Increase preemption delay (yarn.resourcemanager.monitor.
       capacity.preemption.max-wait-before-kill)
→ FIX: Lower max-capacity of non-critical queues

PITFALL 4: "Cluster utilization is only 40% despite pending jobs"
→ SYMPTOM: Queues have pending apps but cluster has free resources
→ CAUSE: max-capacity set too low on active queues
→ CAUSE: Queue ACLs preventing jobs from being placed
→ FIX: Increase max-capacity to allow elastic borrowing
→ FIX: Check that default queue routing works correctly

PITFALL 5: "Configuration changes not taking effect"
→ SYMPTOM: Modified capacity-scheduler.xml but behavior unchanged
→ CAUSE: Scheduler must be REFRESHED after config change
→ FIX: Run: yarn rmadmin -refreshQueues
→ NOTE: Some changes require ResourceManager restart (scheduler class change)

PITFALL 6: "Cannot delete a queue that has running applications"
→ SYMPTOM: Want to remove deprecated queue but apps are running in it
→ CAUSE: YARN does not allow removing queues with active applications
→ FIX: First move or kill all applications in the queue
→ FIX: Set queue state to STOPPED → drain → remove

PITFALL 7: "Jobs submitted to wrong queue despite ACLs"
→ SYMPTOM: QA team's jobs appear in production queue
→ CAUSE: Queue placement rules override ACLs in evaluation order
→ CAUSE: Default queue is "production" instead of "qa"
→ FIX: Review placement rules priority
→ FIX: Set yarn.scheduler.capacity.queue-mappings correctly

PITFALL 8: "Fair Scheduler config in Capacity Scheduler cluster"
→ SYMPTOM: Fair scheduler properties silently ignored
→ CAUSE: Using fair-scheduler.xml properties in capacity-scheduler.xml
→ CAUSE: Different schedulers use COMPLETELY different property names
→ FIX: Verify which scheduler is active in yarn-site.xml
→ FIX: Use correct property namespace for your scheduler

SCHEDULER PERFORMANCE LIMITS:

   CAPACITY SCHEDULER:
   → Handles up to ~100 queues efficiently
   → Beyond 100: scheduling decisions slow down
   → Each heartbeat evaluates ALL queues for allocation
   → Mitigation: Flatten hierarchy, merge similar teams

   FAIR SCHEDULER:
   → Handles dynamic workloads better than Capacity
   → But preemption calculations expensive with many queues
   → Beyond 50 active queues: preemption overhead noticeable

CAPACITY PLANNING FOR QUEUE DESIGN:

   FORMULA: Queue minimum capacity in real resources
   
   queue_min_memory = cluster_total_memory × queue_capacity_percent
   queue_min_vcores = cluster_total_vcores × queue_capacity_percent

   EXAMPLE for our 50-node cluster:
   → Total: 3200 GB RAM, 400 vcores
   → ETL queue (28%): 896 GB, 112 vcores guaranteed
   → Business queue (14.7%): 470 GB, 59 vcores guaranteed
   → Finance queue (17.5%): 560 GB, 70 vcores guaranteed

   QUESTION: Is 470 GB enough for Business Analytics?
   → Their queries typically need 4 GB each
   → 470 / 4 = 117 concurrent query tasks
   → They run ~50-100 queries/day
   → If queries take 5 minutes each: 100 × 5 / 60 = ~8 hours
   → With 117 concurrent tasks: most queries run immediately ✅

   QUESTION: Is 896 GB enough for ETL?
   → 200 jobs per night, each needing 4-8 GB per mapper
   → With 896 GB: ~112-224 concurrent mappers
   → Pipeline has dependencies → not all 200 run simultaneously
   → Typical concurrency: 30-50 jobs → well within capacity ✅

# FILE 5: queue_monitor.py
# PURPOSE: Automated queue monitoring and alerting script
# Connects to ResourceManager REST API
# Run periodically via cron or Oozie (Project 31)
# ══════════════════════════════════════════════════════════════

import json
import urllib.request
import datetime
import sys


class QueueMonitor:
    """
    Monitors YARN queue health via ResourceManager REST API.
    Checks for SLA violations, capacity issues, and starvation.
    """

    def __init__(self, rm_host="localhost", rm_port=8088):
        self.rm_url = f"http://{rm_host}:{rm_port}"
        self.alerts = []
        self.timestamp = datetime.datetime.now().strftime("%Y-%m-%d %H:%M:%S")

    def fetch_scheduler_info(self):
        """Fetch current scheduler state from ResourceManager REST API."""
        url = f"{self.rm_url}/ws/v1/cluster/scheduler"
        try:
            req = urllib.request.Request(url)
            req.add_header("Accept", "application/json")
            response = urllib.request.urlopen(req, timeout=10)
            data = json.loads(response.read().decode())
            return data
        except Exception as e:
            self.alerts.append({
                "severity": "CRITICAL",
                "queue": "CLUSTER",
                "message": f"Cannot reach ResourceManager: {e}"
            })
            return None

    def fetch_cluster_metrics(self):
        """Fetch overall cluster metrics."""
        url = f"{self.rm_url}/ws/v1/cluster/metrics"
        try:
            req = urllib.request.Request(url)
            req.add_header("Accept", "application/json")
            response = urllib.request.urlopen(req, timeout=10)
            data = json.loads(response.read().decode())
            return data.get("clusterMetrics", {})
        except Exception as e:
            return {}

    def parse_queues(self, scheduler_info):
        """Recursively extract all leaf queues from scheduler info."""
        queues = []
        if scheduler_info is None:
            return queues

        scheduler = scheduler_info.get("scheduler", {})
        scheduler_info_inner = scheduler.get("schedulerInfo", {})

        self._extract_queues(scheduler_info_inner, queues)
        return queues

    def _extract_queues(self, queue_data, result):
        """Recursively traverse queue hierarchy."""
        if "queues" in queue_data:
            # Parent queue — recurse into children
            children = queue_data["queues"].get("queue", [])
            if isinstance(children, dict):
                children = [children]
            for child in children:
                self._extract_queues(child, result)
        else:
            # Leaf queue — add to results
            result.append({
                "name": queue_data.get("queueName", "unknown"),
                "capacity": queue_data.get("capacity", 0),
                "usedCapacity": queue_data.get("usedCapacity", 0),
                "maxCapacity": queue_data.get("maxCapacity", 0),
                "absoluteCapacity": queue_data.get("absoluteCapacity", 0),
                "absoluteUsedCapacity": queue_data.get("absoluteUsedCapacity", 0),
                "absoluteMaxCapacity": queue_data.get("absoluteMaxCapacity", 0),
                "numApplications": queue_data.get("numApplications", 0),
                "numPendingApplications": queue_data.get("numPendingApplications", 0),
                "numActiveApplications": queue_data.get("numActiveApplications", 0),
                "state": queue_data.get("state", "UNKNOWN"),
                "usedResources": queue_data.get("resourcesUsed", {}),
            })

    def check_queue_health(self, queues):
        """Run all health checks on each queue."""
        for q in queues:
            self._check_starvation(q)
            self._check_over_capacity(q)
            self._check_pending_backlog(q)
            self._check_queue_state(q)
            self._check_utilization_waste(q)

    def _check_starvation(self, queue):
        """
        ALERT: Queue has pending apps but usage is below guaranteed capacity.
        This means the queue is being starved — its resources are borrowed.
        """
        if (queue["numPendingApplications"] > 0 and
                queue["absoluteUsedCapacity"] < queue["absoluteCapacity"] * 0.8):
            self.alerts.append({
                "severity": "WARNING",
                "queue": queue["name"],
                "message": (
                    f"Queue STARVING: {queue['numPendingApplications']} pending apps, "
                    f"but using only {queue['absoluteUsedCapacity']:.1f}% "
                    f"of guaranteed {queue['absoluteCapacity']:.1f}%"
                )
            })

    def _check_over_capacity(self, queue):
        """
        ALERT: Queue consistently at or near maximum capacity.
        Indicates need for more cluster resources or queue rebalancing.
        """
        if queue["absoluteMaxCapacity"] > 0:
            usage_ratio = queue["absoluteUsedCapacity"] / queue["absoluteMaxCapacity"]
            if usage_ratio > 0.95:
                self.alerts.append({
                    "severity": "WARNING",
                    "queue": queue["name"],
                    "message": (
                        f"Queue at CEILING: using {queue['absoluteUsedCapacity']:.1f}% "
                        f"of max {queue['absoluteMaxCapacity']:.1f}% "
                        f"({queue['numPendingApplications']} apps waiting)"
                    )
                })

    def _check_pending_backlog(self, queue):
        """
        ALERT: Too many pending applications — SLA risk.
        """
        # Different thresholds for different queue types
        critical_queues = {"etl", "finance"}
        threshold = 5 if queue["name"] in critical_queues else 15

        if queue["numPendingApplications"] > threshold:
            severity = "CRITICAL" if queue["name"] in critical_queues else "WARNING"
            self.alerts.append({
                "severity": severity,
                "queue": queue["name"],
                "message": (
                    f"HIGH BACKLOG: {queue['numPendingApplications']} pending applications "
                    f"(threshold: {threshold})"
                )
            })

    def _check_queue_state(self, queue):
        """
        ALERT: Queue is not in RUNNING state.
        """
        if queue["state"] != "RUNNING":
            self.alerts.append({
                "severity": "CRITICAL",
                "queue": queue["name"],
                "message": f"Queue state is {queue['state']} — not RUNNING"
            })

    def _check_utilization_waste(self, queue):
        """
        INFO: Queue has significant capacity but zero usage for extended periods.
        Indicates over-provisioning — capacity could be reallocated.
        """
        if (queue["absoluteCapacity"] > 10 and
                queue["numApplications"] == 0 and
                queue["absoluteUsedCapacity"] == 0):
            self.alerts.append({
                "severity": "INFO",
                "queue": queue["name"],
                "message": (
                    f"Queue IDLE with {queue['absoluteCapacity']:.1f}% guaranteed capacity. "
                    f"Consider reducing if consistently unused."
                )
            })

    def check_cluster_health(self, metrics):
        """Check overall cluster health."""
        if not metrics:
            return

        total_mb = metrics.get("totalMB", 1)
        used_mb = metrics.get("allocatedMB", 0)
        utilization = (used_mb / total_mb) * 100 if total_mb > 0 else 0

        # Cluster over-utilized
        if utilization > 95:
            self.alerts.append({
                "severity": "CRITICAL",
                "queue": "CLUSTER",
                "message": (
                    f"Cluster utilization at {utilization:.1f}% — "
                    f"consider adding nodes or reducing workload"
                )
            })

        # Cluster under-utilized (waste)
        if utilization < 20:
            pending_apps = metrics.get("appsPending", 0)
            if pending_apps == 0:
                self.alerts.append({
                    "severity": "INFO",
                    "queue": "CLUSTER",
                    "message": (
                        f"Cluster utilization only {utilization:.1f}% with no pending apps — "
                        f"consider scaling down (cost savings)"
                    )
                })

        # Dead nodes
        lost_nodes = metrics.get("lostNodes", 0)
        unhealthy_nodes = metrics.get("unhealthyNodes", 0)
        if lost_nodes > 0 or unhealthy_nodes > 0:
            self.alerts.append({
                "severity": "CRITICAL",
                "queue": "CLUSTER",
                "message": (
                    f"Node health issue: {lost_nodes} lost, "
                    f"{unhealthy_nodes} unhealthy"
                )
            })

    def generate_report(self, queues, metrics):
        """Generate formatted monitoring report."""
        report = []
        report.append("=" * 75)
        report.append(f"  YARN QUEUE HEALTH REPORT — {self.timestamp}")
        report.append("=" * 75)

        # ── Cluster Summary ──
        if metrics:
            total_gb = metrics.get("totalMB", 0) / 1024
            used_gb = metrics.get("allocatedMB", 0) / 1024
            total_vcores = metrics.get("totalVirtualCores", 0)
            used_vcores = metrics.get("allocatedVirtualCores", 0)
            active_nodes = metrics.get("activeNodes", 0)
            lost_nodes = metrics.get("lostNodes", 0)

            report.append("")
            report.append("  CLUSTER SUMMARY:")
            report.append(f"  ├─ Memory:  {used_gb:.0f} GB / {total_gb:.0f} GB "
                          f"({(used_gb/total_gb*100) if total_gb > 0 else 0:.1f}%)")
            report.append(f"  ├─ VCores:  {used_vcores} / {total_vcores} "
                          f"({(used_vcores/total_vcores*100) if total_vcores > 0 else 0:.1f}%)")
            report.append(f"  ├─ Nodes:   {active_nodes} active, {lost_nodes} lost")
            report.append(f"  └─ Apps:    {metrics.get('appsRunning', 0)} running, "
                          f"{metrics.get('appsPending', 0)} pending")

        # ── Queue Details ──
        report.append("")
        report.append("  QUEUE DETAILS:")
        report.append("  " + "-" * 71)
        report.append(f"  {'Queue':<30} {'Used%':>7} {'Guar%':>7} "
                      f"{'Max%':>7} {'Active':>7} {'Pending':>8}")
        report.append("  " + "-" * 71)

        for q in queues:
            status_icon = "🟢" if q["numPendingApplications"] == 0 else "🟡"
            if q["state"] != "RUNNING":
                status_icon = "🔴"

            report.append(
                f"  {status_icon} {q['name']:<28} "
                f"{q['absoluteUsedCapacity']:>6.1f}% "
                f"{q['absoluteCapacity']:>6.1f}% "
                f"{q['absoluteMaxCapacity']:>6.1f}% "
                f"{q['numActiveApplications']:>7} "
                f"{q['numPendingApplications']:>8}"
            )

        report.append("  " + "-" * 71)

        # ── Alerts ──
        if self.alerts:
            report.append("")
            report.append("  ALERTS:")

            for severity in ["CRITICAL", "WARNING", "INFO"]:
                severity_alerts = [a for a in self.alerts if a["severity"] == severity]
                if severity_alerts:
                    icon = {"CRITICAL": "🔴", "WARNING": "🟡", "INFO": "🔵"}[severity]
                    for alert in severity_alerts:
                        report.append(
                            f"  {icon} [{severity}] {alert['queue']}: {alert['message']}"
                        )
        else:
            report.append("")
            report.append("  ✅ NO ALERTS — All queues healthy")

        report.append("")
        report.append("=" * 75)
        return "\n".join(report)

    def run(self):
        """Execute full monitoring cycle."""
        # Fetch data
        scheduler_info = self.fetch_scheduler_info()
        metrics = self.fetch_cluster_metrics()

        # Parse queues
        queues = self.parse_queues(scheduler_info)

        # Run health checks
        self.check_queue_health(queues)
        self.check_cluster_health(metrics)

        # Generate report
        report = self.generate_report(queues, metrics)
        print(report)

        # Return exit code based on alerts
        critical_count = sum(1 for a in self.alerts if a["severity"] == "CRITICAL")
        if critical_count > 0:
            return 2  # Critical
        warning_count = sum(1 for a in self.alerts if a["severity"] == "WARNING")
        if warning_count > 0:
            return 1  # Warning
        return 0  # Healthy

import unittest
import xml.etree.ElementTree as ET
import os


class TestQueueConfiguration(unittest.TestCase):
    """
    Validate capacity-scheduler.xml configuration.
    Run BEFORE deploying to cluster — catches math errors,
    missing properties, and logical inconsistencies.
    """

    @classmethod
    def setUpClass(cls):
        """Parse capacity-scheduler.xml into a properties dict."""
        config_path = os.path.join(
            os.environ.get("HADOOP_HOME", "/opt/hadoop"),
            "etc/hadoop/capacity-scheduler.xml"
        )

        # If file doesn't exist, create from test fixture
        if not os.path.exists(config_path):
            config_path = "capacity-scheduler.xml"

        cls.properties = {}
        try:
            tree = ET.parse(config_path)
            root = tree.getroot()
            for prop in root.findall("property"):
                name = prop.find("name").text
                value = prop.find("value").text
                cls.properties[name] = value.strip() if value else ""
        except FileNotFoundError:
            cls.properties = {}

    def get_prop(self, key, default=None):
        """Get property value with optional default."""
        return self.properties.get(key, default)

    def get_float_prop(self, key, default=0.0):
        """Get property as float."""
        val = self.get_prop(key)
        if val is None:
            return default
        try:
            return float(val)
        except ValueError:
            return default

    # ────────────────────────────────────────────
    # TEST: Root level capacities sum to 100%

    def test_root_children_sum_to_100(self):

            "yarn.scheduler.capacity.root.queues", "")
        root_queues = [q.strip() for q in root_queues_str.split(",") if q.strip()]

        total = 0.0
        for queue in root_queues:

            total += capacity

        self.assertAlmostEqual(total, 100.0, places=1,

    # ────────────────────────────────────────────
    # TEST: Queue mappings are syntactically valid
    # ────────────────────────────────────────────
    def test_queue_mappings_syntax(self):
        mappings_str = self.get_prop(
            "yarn.scheduler.capacity.queue-mappings", "")

        if not mappings_str:
            return  # No mappings configured

        mappings = [m.strip() for m in mappings_str.split(",") if m.strip()]

        for mapping in mappings:
            parts = mapping.split(":")
            self.assertGreaterEqual(len(parts), 3,
                msg=f"Invalid mapping syntax: '{mapping}' — "
                    f"expected format u:USER:QUEUE or g:GROUP:QUEUE")
            self.assertIn(parts[0], ["u", "g"],
                msg=f"Invalid mapping type '{parts[0]}' in '{mapping}' — "
                    f"must be 'u' (user) or 'g' (group)")

    # ────────────────────────────────────────────
    # TEST: Finance queue can burst for regulatory needs
    # ────────────────────────────────────────────
    def test_finance_queue_burst_capacity(self):
        finance_cap = self.get_float_prop(
            "yarn.scheduler.capacity.root.production.finance.capacity")
        finance_max = self.get_float_prop(
            "yarn.scheduler.capacity.root.production.finance.maximum-capacity")

        # Finance must be able to at least double its guaranteed capacity
        burst_ratio = finance_max / finance_cap if finance_cap > 0 else 0
        self.assertGreaterEqual(burst_ratio, 2.0,
            msg=f"Finance queue burst ratio is {burst_ratio:.1f}x — "
                f"should be at least 2x for month-end regulatory jobs")
