PROJECT 22: WHO GETS THE RESOURCES? UNDERSTANDING YARN FROM INSIDE

🔴 HYPOTHETICAL PROBLEM

SCENARIO:
→ You are the Hadoop cluster administrator at a financial services company
→ 50-node cluster shared by THREE teams:
    - Risk Analytics team: Runs heavy overnight batch pipelines (Project 21)
    - Marketing team: Runs ad-hoc Hive queries during business hours
    - Data Science team: Runs experimental MapReduce jobs sporadically

MONDAY MORNING DISASTER:
→ 8:00 AM: Risk Analytics pipeline (Project 21 pattern) is still running
   from overnight — consuming 80% of cluster resources
→ 8:15 AM: Marketing VP submits urgent Hive query for board meeting at 9 AM
→ 8:15 AM: Query status shows "ACCEPTED" — not "RUNNING"
→ 8:30 AM: Still "ACCEPTED" — no resources available
→ 8:45 AM: Marketing VP calls you: "WHY ISN'T MY QUERY RUNNING?"
→ 8:50 AM: Data Science submits 5 experimental jobs simultaneously
→ 9:00 AM: Board meeting starts — Marketing has NO data

POST-MORTEM QUESTIONS:
→ WHO decided Risk Analytics gets 80% of the cluster?
→ WHY couldn't Marketing's query get even a small slice?
→ WHAT component manages resource allocation?
→ HOW does a MapReduce job actually acquire CPU and memory?
→ WHERE does the decision happen — master node or worker nodes?
→ WHEN does a job go from "ACCEPTED" to "RUNNING"?

THESE ARE ALL YARN QUESTIONS.

You configured YARN in Project 8 (yarn-site.xml) but treated it
as a black box. Now you need to understand EVERY internal component
to prevent this disaster from happening again.

🚫 WHAT SHOULD NOT DO

1. DO NOT confuse ResourceManager with the old JobTracker
   → JobTracker (Hadoop 1.x) managed BOTH resources AND tasks
   → ResourceManager ONLY manages resources
   → Task management is ApplicationMaster's job
   → This separation is what makes YARN scalable

2. DO NOT assume containers are persistent
   → Containers are SHORT-LIVED — one task per container
   → When mapper finishes, its container is RELEASED
   → Reducer gets a NEW container (possibly on different node)
   → No state carries between containers
   → (Exception: ApplicationMaster container lives for entire job)

3. DO NOT run ApplicationMaster on the same node as NameNode
   → NameNode is memory-intensive (metadata in RAM — Project 2)
   → ApplicationMaster also needs memory
   → Competing for resources on master node causes instability
   → RULE: Dedicate master node to RM + NN daemons only
   → Configure yarn.nodemanager.resource.memory-mb = 0 on master
     OR simply don't run NodeManager on the master node

4. DO NOT ignore the ApplicationMaster's resource cost
   → Every running job consumes ONE container for its AM
   → Default AM container: 1536MB RAM, 1 vcore
   → 50 concurrent jobs = 50 AM containers = 75GB RAM just for AMs
   → On a small cluster this is significant overhead

5. DO NOT set container memory larger than NodeManager capacity
   → If NodeManager has 32GB available
   → And you request a 48GB container → NEVER gets allocated
   → Job sits in ACCEPTED state forever
   → No error message — just infinite waiting (common trap!)

6. DO NOT treat vcore allocation as strict CPU isolation
   → YARN vcores are a SCHEDULING abstraction
   → By default, CPU isolation is NOT enforced
   → A container with 1 vcore CAN use all CPU cores on the machine
   → Strict CPU isolation requires Linux cgroups configuration
   → Memory IS strictly enforced (container killed if exceeded)

🚫 WHEN NOT TO USE

1. YARN IS NOT OPTIONAL IN HADOOP 2.x+
   → If you run MapReduce on a cluster, YARN is ALWAYS involved
   → Even pseudo-distributed mode uses YARN (Project 8)
   → The question is not "when to use YARN" but
     "how deeply to configure YARN"

2. WHEN NOT TO DEEPLY CONFIGURE YARN:
   → Single-user development cluster → default settings fine
   → Less than 5 nodes → overhead of queue management not worth it
   → All jobs are similar size → no resource contention

3. WHEN YOU MUST DEEPLY CONFIGURE YARN:
   → Multiple teams sharing cluster (our scenario)
   → Mixed workloads: batch + interactive queries
   → Jobs of vastly different sizes (1-node vs 100-node)
   → SLA requirements: "Marketing queries must finish in 10 minutes"
   → Cluster utilization below 60% (waste of expensive hardware)

4. WHEN YARN ALONE IS NOT ENOUGH:
   → Need sub-second scheduling → YARN overhead too high
   → Kubernetes/Mesos may be better for microservice workloads
   → Spark Standalone scheduler for pure Spark clusters
   → But for Hadoop ecosystem: YARN is THE resource manager

INTERNAL MECHANIC 2: HEARTBEAT PROTOCOL
═══════════════════════════════════════════════════════════════

  TWO separate heartbeat channels:

  CHANNEL A: NodeManager → ResourceManager (every 1 second default)
  ┌──────────────────────────────────────────────────────────────┐
  │ NodeManager heartbeat contains:                               │
  │  → Node health status (healthy/unhealthy)                    │
  │  → Available resources (free memory, free vcores)            │
  │  → Running containers list                                   │
  │  → Completed containers since last heartbeat                 │
  │                                                               │
  │ ResourceManager response contains:                            │
  │  → New containers to launch on this node                     │
  │  → Containers to kill (preemption or cleanup)                │
  │  → Updated cluster state information                         │
  │                                                               │
  │ If RM receives no heartbeat for 10 minutes:                  │
  │  → Declares node DEAD                                        │
  │  → All containers on that node marked FAILED                 │
  │  → ApplicationMasters notified to retry those tasks          │
  └──────────────────────────────────────────────────────────────┘

  CHANNEL B: ApplicationMaster → ResourceManager (every 1 second)
  ┌──────────────────────────────────────────────────────────────┐
  │ AM heartbeat contains:                                        │
  │  → Resource requests (new containers needed)                 │
  │  → Released containers (tasks completed)                     │
  │  → Application progress (% mappers done, % reducers done)   │
  │                                                               │
  │ ResourceManager response contains:                            │
  │  → Newly allocated containers (node, memory, vcores)         │
  │  → This is HOW AM gets containers — via heartbeat response   │
  │                                                               │
  │ CRITICAL INSIGHT:                                             │
  │  → Container allocation is NOT a separate RPC call           │
  │  → It's PIGGYBACKED on the heartbeat response                │
  │  → AM sends request → next heartbeat response has allocation │
  │  → This means allocation latency = heartbeat interval (~1s)  │
  └──────────────────────────────────────────────────────────────┘

INTERNAL MECHANIC 3: CONTAINER ALLOCATION ALGORITHM
═══════════════════════════════════════════════════════════════

  When Scheduler receives container request from AM:

  REQUEST: "I need 50 containers, each 2048MB + 1 vcore,
            preferring nodes where my input data lives"

  SCHEDULER ALGORITHM (CapacityScheduler):

  ┌─────────────────────────────────────────────────────────────┐
  │                                                             │
  │  FOR each NodeManager heartbeat that reports free resources:│
  │                                                             │
  │    1. CHECK QUEUE: Is the requesting app's queue under      │
  │       its capacity limit?                                   │
  │       → If over limit: SKIP (no allocation)                │
  │       → If under limit: proceed                            │
  │                                                             │
  │    2. CHECK NODE CAPACITY: Does this node have enough       │
  │       free memory + vcores for the requested container?    │
  │       → If not: SKIP this node                             │
  │                                                             │
  │    3. CHECK DATA LOCALITY:                                  │
  │       → LEVEL 1 — NODE LOCAL: Does this node have the      │
  │         input data block? (BEST — zero network transfer)   │
  │       → LEVEL 2 — RACK LOCAL: Is data on another node      │
  │         in the SAME rack? (GOOD — intra-rack transfer)     │
  │       → LEVEL 3 — OFF RACK: Data is on a different rack    │
  │         (WORST — cross-rack transfer)                      │
  │                                                             │
  │    4. ALLOCATE: Assign container on this node               │
  │       → Decrement node's available resources               │
  │       → Add to allocated list in heartbeat response to NM  │
  │       → Notify AM in next AM heartbeat response            │
  │                                                             │
  │  LOCALITY SCHEDULING DETAIL:                                │
  │    → Scheduler WAITS for node-local allocation              │
  │    → After N heartbeat cycles without node-local match:     │
  │      relax to rack-local                                   │
  │    → After M more cycles: relax to off-rack                │
  │    → Controlled by: yarn.scheduler.capacity.                │
  │      node-locality-delay                                   │
  │                                                             │
  └─────────────────────────────────────────────────────────────┘

⚠️ COMMON PITFALLS AND FAILURE MODES

PITFALL 1: "Job stuck in ACCEPTED state forever"
→ CAUSE 1: Requested container size exceeds NodeManager capacity
   → mapreduce.map.memory.mb = 8192 but NM only has 4096 available
   → Container can NEVER be allocated
   → No error — just infinite waiting
→ CAUSE 2: Queue is at maximum-capacity and all resources consumed
   → Other jobs in same queue using everything
→ CAUSE 3: All NodeManagers are unhealthy or decommissioned
→ FIX: Check RM UI → Application → Diagnostics message
→ FIX: Verify container request size ≤ yarn.scheduler.maximum-allocation-mb

PITFALL 2: "Container killed by YARN for exceeding memory limits"
→ CAUSE: JVM heap + off-heap memory exceeds container allocation
→ Container allocated: 2048MB
→ JVM heap (-Xmx): 1536MB
→ JVM overhead (metaspace, threads, buffers): ~600MB
→ Total: 2136MB > 2048MB → KILLED
→ FIX: Container memory should be 1.5x-2x JVM heap
→ (Deep dive in Project 23 — Memory Tuning)

PITFALL 3: "ApplicationMaster killed — job fails after hours of work"
→ CAUSE: AM ran out of memory tracking too many tasks
→ 10,000 mappers × task state tracking = significant AM memory
→ Default AM memory: 1536MB — insufficient for very large jobs
→ FIX: yarn.app.mapreduce.am.resource.mb = 4096 for large jobs

PITFALL 4: "Cluster shows 40% utilization but jobs are queued"
→ CAUSE: Resources fragmented across nodes
→ Each node has 1GB free but no node has the 4GB container needs
→ Like having $100 in pennies — can't buy a $1 item
→ FIX: Right-size containers, avoid requesting unnecessarily large ones

PITFALL 5: "Node declared UNHEALTHY — containers not scheduled"
→ CAUSE: Disk usage exceeds threshold (default: 90%)
→ NodeManager health checker: disk, custom scripts
→ Unhealthy node: RM stops scheduling containers there
→ FIX: Clean disk space, or increase threshold:
   yarn.nodemanager.disk-health-checker.max-disk-utilization-per-disk-percentage

PITFALL 6: "Job runs but data locality is 0%"
→ CAUSE: Cluster too busy — no node-local containers available
→ Scheduler gave up waiting for local containers
→ All data transferred over network → much slower
→ FIX: Increase node-locality-delay setting
→ FIX: Ensure cluster is not over-committed

SCALING DIMENSION 1: ResourceManager AS BOTTLENECK

  → RM processes ALL heartbeats from ALL NodeManagers
  → RM processes ALL container requests from ALL ApplicationMasters
  → 10,000 nodes × 1 heartbeat/sec = 10,000 heartbeats/sec
  → 500 concurrent jobs × 1 AM heartbeat/sec = 500 requests/sec
  → Total: ~10,500 RPCs/second

  │ <100 nodes   │ Default settings fine                   │
  │ 100-1000     │ Tune RM heap: 4-8GB                    │
  │ 1000-5000    │ Tune RM heap: 8-16GB, tune heartbeat   │
  │ 5000-10000   │ RM HA required (Project 45)            │

  └──────────────┴─────────────────────────────────────────┘

SCALING DIMENSION 2: NUMBER OF CONCURRENT APPLICATIONS

  → Each application has ONE ApplicationMaster container
  → AM container: 1536MB default
  → 100 concurrent apps: 150GB just for AMs
  → On 50-node cluster with 64GB each: 150GB/3200GB = 4.7% overhead
  → On 10-node cluster with 16GB each: 150GB/160GB = 93.75% → DISASTER

  RULE: max_concurrent_apps ≤ total_cluster_memory / (10 × AM_memory)

SCALING DIMENSION 3: CONTAINER ALLOCATION THROUGHPUT

  → CapacityScheduler: ~10,000 container allocations/second
  → FairScheduler: ~5,000 container allocations/second
  → For most workloads: not a bottleneck
  → For Spark with thousands of short tasks: may become bottleneck

    private void printClusterOverview(YarnClient client)
            throws YarnException, IOException {

        System.out.println("=".repeat(70));
        System.out.println("YARN CLUSTER OVERVIEW");
        System.out.println("=".repeat(70));

        List<NodeReport> nodes = client.getNodeReports(NodeState.RUNNING);

        long totalMemoryMB = 0;
        long usedMemoryMB = 0;
        int totalVcores = 0;
        int usedVcores = 0;
        int totalContainers = 0;

        for (NodeReport node : nodes) {
            totalMemoryMB += node.getCapability().getMemorySize();
            usedMemoryMB += node.getUsed().getMemorySize();
            totalVcores += node.getCapability().getVirtualCores();
            usedVcores += node.getUsed().getVirtualCores();
            totalContainers += node.getNumContainers();
        }

        long availableMemoryMB = totalMemoryMB - usedMemoryMB;
        int availableVcores = totalVcores - usedVcores;
        double memoryUtilization = totalMemoryMB > 0
            ? (double) usedMemoryMB / totalMemoryMB * 100 : 0;

        System.out.println("  Active nodes:       " + nodes.size());
        System.out.println("  Running containers: " + totalContainers);
        System.out.println();
        System.out.println("  Memory:");
        System.out.printf("    Total:     %,d MB (%.1f GB)%n",
            totalMemoryMB, totalMemoryMB / 1024.0);
        System.out.printf("    Used:      %,d MB (%.1f GB) [%.1f%%]%n",
            usedMemoryMB, usedMemoryMB / 1024.0, memoryUtilization);
        System.out.printf("    Available: %,d MB (%.1f GB)%n",
            availableMemoryMB, availableMemoryMB / 1024.0);
        System.out.println();
        System.out.println("  VCores:");
        System.out.printf("    Total:     %d%n", totalVcores);
        System.out.printf("    Used:      %d [%.1f%%]%n",
            usedVcores, totalVcores > 0
                ? (double) usedVcores / totalVcores * 100 : 0);
        System.out.printf("    Available: %d%n", availableVcores);
    }

    private void printResourceCalculation(YarnClient client)
            throws YarnException, IOException {

        System.out.println("\n" + "=".repeat(70));
        System.out.println("RESOURCE CALCULATION GUIDE");
        System.out.println("=".repeat(70));

        List<NodeReport> nodes = client.getNodeReports(NodeState.RUNNING);

        if (nodes.isEmpty()) {
            System.out.println("  No running nodes available");
            return;
        }

        // Take first node as example
        NodeReport sampleNode = nodes.get(0);
        long nodeMemoryMB = sampleNode.getCapability().getMemorySize();
        int nodeVcores = sampleNode.getCapability().getVirtualCores();

        int numNodes = nodes.size();
        long totalClusterMemoryMB = 0;
        int totalClusterVcores = 0;

        for (NodeReport node : nodes) {
            totalClusterMemoryMB += node.getCapability().getMemorySize();
            totalClusterVcores += node.getCapability().getVirtualCores();
        }

        // Calculate how many containers of different sizes fit
        int[] containerSizesMB = {1024, 2048, 4096, 8192};

        System.out.println("\n  Cluster Summary:");
        System.out.printf("    Nodes:            %d%n", numNodes);
        System.out.printf("    Total Memory:     %,d MB (%.1f GB)%n",
            totalClusterMemoryMB, totalClusterMemoryMB / 1024.0);
        System.out.printf("    Total VCores:     %d%n", totalClusterVcores);

        System.out.println("\n  Sample Node: " + sampleNode.getNodeId().getHost());
        System.out.printf("    Memory:  %,d MB (%.1f GB)%n",
            nodeMemoryMB, nodeMemoryMB / 1024.0);
        System.out.printf("    VCores:  %d%n", nodeVcores);

        System.out.println("\n  Container Capacity Planning:");
        System.out.printf("    %-15s %-15s %-15s %-20s%n",
            "Container Size", "Per Node", "Per Cluster", "Recommended For");
        System.out.println("    " + "-".repeat(65));

        for (int sizeMB : containerSizesMB) {
            // Containers per node limited by memory and vcores
            int perNodeByMem = (int) (nodeMemoryMB / sizeMB);
            int perNodeByVcore = nodeVcores; // 1 vcore per container
            int perNode = Math.min(perNodeByMem, perNodeByVcore);
            int perCluster = perNode * numNodes;

            String recommendation;
            if (sizeMB <= 1024) {
                recommendation = "Light mappers";
            } else if (sizeMB <= 2048) {
                recommendation = "Standard map/reduce";
            } else if (sizeMB <= 4096) {
                recommendation = "Heavy reduce tasks";
            } else {
                recommendation = "Large aggregations";
            }

            System.out.printf("    %-15s %-15d %-15d %-20s%n",
                sizeMB + " MB", perNode, perCluster, recommendation);
        }

        // ApplicationMaster overhead calculation
        System.out.println("\n  ApplicationMaster Overhead:");
        int amMemoryMB = 1536; // default
        int maxConcurrentApps = (int) (totalClusterMemoryMB * 0.10 / amMemoryMB);
        System.out.printf("    AM container size:    %d MB%n", amMemoryMB);
        System.out.printf("    Max concurrent apps:  ~%d (using 10%% of cluster for AMs)%n",
            maxConcurrentApps);
        System.out.printf("    AM overhead at max:   %,d MB (%.1f GB)%n",
            maxConcurrentApps * amMemoryMB,
            maxConcurrentApps * amMemoryMB / 1024.0);

        // Queue capacity in absolute terms
        System.out.println("\n  Queue Capacity (Absolute Resources):");
        String[] queueNames = {"risk_analytics", "marketing", "data_science"};
        float[] capacities = {0.50f, 0.30f, 0.20f};
        float[] maxCapacities = {0.80f, 0.60f, 0.50f};

            "Queue", "Guaranteed", "Guar(GB)", "Max", "Max(GB)");
        System.out.println("    " + "-".repeat(68));

        for (int i = 0; i < queueNames.length; i++) {
            long guarMemMB = (long) (totalClusterMemoryMB * capacities[i]);
            long maxMemMB = (long) (totalClusterMemoryMB * maxCapacities[i]);

            System.out.printf("    %-20s %-12s %-12.1f %-12s %-12.1f%n",
                queueNames[i],
                String.format("%.0f%%", capacities[i] * 100),
                guarMemMB / 1024.0,
                String.format("%.0f%%", maxCapacities[i] * 100),
                maxMemMB / 1024.0);
        }

        // Data locality impact estimation
        System.out.println("\n  Data Locality Impact Estimation:");
        System.out.println("    ┌───────────────────┬──────────────────┬──────────────────┐");
        System.out.println("    │ Locality Level    │ Network Cost     │ Typical Speed    │");
        System.out.println("    ├───────────────────┼──────────────────┼──────────────────┤");
        System.out.println("    │ NODE_LOCAL        │ Zero             │ Disk speed only  │");
        System.out.println("    │ RACK_LOCAL        │ Intra-rack       │ ~80% of local    │");
        System.out.println("    │ OFF_RACK          │ Cross-rack       │ ~50% of local    │");
        System.out.println("    └───────────────────┴──────────────────┴──────────────────┘");
        System.out.println("    Goal: >80% NODE_LOCAL for optimal performance");
    }

def format_mb(mb):
    """Format MB to human-readable."""
    if mb >= 1024 * 1024:
        return f"{mb / (1024*1024):.1f} TB"
    elif mb >= 1024:
        return f"{mb / 1024:.1f} GB"
    else:
        return f"{mb} MB"

def progress_bar(used, total, width=30):
    """Create ASCII progress bar."""
    if total == 0:
        return "[" + " " * width + "] 0%"
    pct = used / total
    filled = int(pct * width)
    bar = "█" * filled + "░" * (width - filled)
    return f"[{bar}] {pct*100:.1f}%"

WHAT PROJECT 22 ESTABLISHED:

✅ PROBLEM: Without understanding YARN, you cannot diagnose
   why jobs are stuck, why resources are wasted, or why
   teams fight over cluster capacity

✅ THE FOUR YARN COMPONENTS:
   1. ResourceManager (RM) — ONE per cluster
      → Scheduler: allocates containers to applications
      → ApplicationsManager: accepts jobs, launches AMs
   2. NodeManager (NM) — ONE per worker node
      → Launches/monitors/kills containers on that node
      → Reports available resources via heartbeat
   3. ApplicationMaster (AM) — ONE per running job
      → Negotiates containers from RM for its tasks
      → Manages task execution, retries, progress
   4. Container — atomic resource allocation unit
      → Fixed memory + vcores on a specific node
      → Runs ONE task (mapper, reducer, or AM)

✅ THE 12-STEP JOB LIFECYCLE:
   Client → RM accepts → Scheduler allocates AM container →
   NM launches AM → AM calculates needs → AM requests containers →
   Scheduler allocates in batches → AM launches tasks on NMs →
   Tasks execute → Tasks complete → AM reports to RM → Done

✅ HEARTBEAT PROTOCOL:
   → NM → RM: "I'm alive, here's my free resources"
   → AM → RM: "I need containers" / "I'm done with these"
   → Container allocation piggybacked on heartbeat responses
   → No heartbeat for 10 min → node declared DEAD

✅ CAPACITY SCHEDULER:
   → Queues with guaranteed capacity percentages
   → Elastic borrowing of idle capacity
   → Preemption to reclaim borrowed resources
   → Solves the Monday morning disaster

Project 22 explained HOW YARN allocates containers.

If you request a 2048MB container and your task uses 2100MB:
