PROJECT 31: ORCHESTRATING A NIGHTLY ETL PIPELINE WITH APACHE OOZIE

🔴 HYPOTHETICAL PROBLEM

SCENARIO:
→ You are the lead data engineer at a nationwide retail chain
→ Every night at 2:00 AM, the following MUST happen in EXACT order:

  STEP 1: Import yesterday's sales from MySQL → HDFS       (Sqoop — Project 28)
  STEP 2: Clean and validate the raw data                   (MapReduce — Project 11)
  STEP 3: Transform into Hive ORC tables                    (Hive — Project 24)
  STEP 4: Generate regional summary reports                 (Hive — Project 25)
  STEP 5: Export summary back to MySQL for dashboards       (Sqoop export)
  STEP 6: Send email notification on success or failure

CURRENT STATE (the mess):
→ You built this using Project 21's JobControl chaining
→ A cron job triggers a shell script at 2 AM
→ The shell script calls each step sequentially

WHAT KEEPS BREAKING:
→ Step 2 started before Step 1 finished (race condition)
→ Step 3 failed but Step 4 ran anyway (no error propagation)
→ Weekend runs failed silently — nobody knew until Monday
→ The cron job ran even on holidays when no data existed
→ Retry logic? You wrote nested if-else in bash. It's 400 lines.
→ Last week the script ran TWICE because someone restarted cron

THE QUESTION:
→ How do you orchestrate a multi-step Hadoop pipeline
  with dependency management, time scheduling, failure handling,
  conditional logic, and parallel execution — WITHOUT brittle scripts?

THE ANSWER: Apache Oozie

STEP 2: Understand why Project 21's approach breaks at scale

  Project 21 (JobControl):
  → Java-level chaining — works for 2-3 MapReduce jobs
  → Cannot schedule Sqoop, Hive, Pig, shell scripts
  → No time-based triggering
  → No Web UI for monitoring
  → No retry/failure notification built in
  → Lives inside a single JVM — if that JVM dies, everything stops

  Oozie:
  → Server-side orchestration — survives client disconnection
  → Supports ALL Hadoop ecosystem actions natively
  → XML-based workflow definition — version controllable
  → Web UI for monitoring running/failed/succeeded workflows
  → Built-in retry, timeout, error handling
  → Coordinator for recurring time-based execution

✅ WHAT SHOULD DO

1. KEEP WORKFLOW XML IN VERSION CONTROL (Git)
   → Treat workflows as code — review, branch, merge
   → Store alongside the Hive scripts and MR jars they reference

2. USE PARAMETERIZED WORKFLOWS
   → Pass dates, paths, database names as parameters
   → NEVER hardcode dates like "2024-01-15" in workflow XML
   → Use EL functions: ${coord:formatTime(coord:nominalTime(), 'yyyy-MM-dd')}

3. DESIGN FOR IDEMPOTENCY
   → Re-running yesterday's pipeline should produce identical results
   → Hive: INSERT OVERWRITE (not INSERT INTO) — replaces partition
   → HDFS: Delete output before writing — or use timestamped paths

4. SET MEANINGFUL RETRY POLICIES
   → Transient failures (network glitch): retry 3 times, 5-minute intervals
   → Permanent failures (bad data): retry once, then kill + alert
   → Configure per-action, not globally

5. USE THE <global> SECTION FOR SHARED CONFIGURATION
   → JobTracker, NameNode, queue name — define once
   → All actions inherit these values

6. DEPLOY WORKFLOW TO HDFS (not local filesystem)
   → Oozie reads workflow.xml FROM HDFS
   → All referenced scripts, JARs, config files must be in HDFS too
   → Standard structure:
     /user/oozie/workflows/sales-etl/
       ├── workflow.xml
       ├── coordinator.xml
       ├── lib/
       │   └── sales-cleaner.jar
       ├── scripts/
       │   ├── transform.hql
       │   └── summary.hql
       └── config/
           └── job.properties

🚫 WHAT SHOULD NOT DO

1. DO NOT use Oozie for sub-second scheduling
   → Oozie polling interval: 5 minutes (default)
   → Minimum practical granularity: ~5-10 minutes
   → Need second-level? Use Airflow, NiFi, or streaming

2. DO NOT put business logic IN the workflow XML
   → Workflow XML defines FLOW, not LOGIC
   → SQL goes in .hql files, Java in .jar files
   → Workflow only REFERENCES these artifacts

3. DO NOT skip the <error to="kill-node"> on any action
   → EVERY action must define an error transition
   → Without it: workflow hangs indefinitely on failure
   → Oozie does NOT auto-fail — it follows YOUR graph

4. DO NOT hardcode cluster addresses in workflow XML
   → Use ${jobTracker} and ${nameNode} variables
   → Pass them via job.properties file
   → Allows same workflow to run on dev/staging/prod

5. DO NOT create monolithic mega-workflows
   → If workflow exceeds 15-20 actions → split into sub-workflows
   → Use <sub-workflow> action to call child workflows
   → Easier to debug, test, and reuse individual pieces

6. DO NOT ignore timezone in Coordinator
   → Oozie defaults to UTC
   → "2 AM" in UTC is NOT "2 AM" in your local timezone
   → Explicitly set: <timezone>America/New_York</timezone>

🚫 WHEN NOT TO USE

1. WHEN YOUR PIPELINE HAS NO HADOOP COMPONENTS
   → Oozie is tightly coupled to the Hadoop ecosystem
   → If you're orchestrating REST APIs, Kubernetes jobs, cloud functions
   → Use Apache Airflow, Prefect, or Dagster instead

2. WHEN YOU NEED DYNAMIC DAG GENERATION
   → Oozie workflows are STATIC XML — defined at deploy time
   → Cannot generate workflow steps based on runtime data
   → Airflow can generate DAGs dynamically with Python

3. WHEN YOU NEED INTERACTIVE DEBUGGING
   → Oozie debugging is painful — read XML, check logs, pray
   → No breakpoints, no step-through, no variable inspection
   → For development: test each action independently first

4. WHEN SCHEDULING GRANULARITY < 5 MINUTES
   → Oozie's polling model is not designed for rapid scheduling
   → Internal check frequency is configurable but not designed for real-time

═══════════════════════════════════════════════════════════════

MECHANIC 4: COORDINATOR TRIGGER LOGIC
═══════════════════════════════════════════════════════════════

  Coordinator checks TWO types of conditions:

  TYPE A: TIME-BASED TRIGGER
    <frequency>${coord:days(1)}</frequency>
    → Oozie's internal scheduler creates a "materialized action"
       at each frequency interval
    → Each materialized action = one workflow instance

  TYPE B: DATA-BASED TRIGGER
    <dataset name="daily_sales"
             frequency="${coord:days(1)}"
             initial-instance="2024-01-01T00:00Z"
             timezone="America/New_York">
      <uri-template>hdfs://namenode:9000/data/sales/${YEAR}/${MONTH}/${DAY}</uri-template>
      <done-flag>_SUCCESS</done-flag>
    </dataset>

    → Oozie checks HDFS for the existence of _SUCCESS flag file
    → Only triggers workflow when data directory + flag exists
    → Checks every oozie.service.coord.check.interval (default 5 min)

  COMBINED: Time creates the schedule, data gates the execution
    → "Run every day at 2 AM, BUT ONLY IF today's data exists"

⚠️ COMMON PITFALLS AND FAILURE MODES

PITFALL 1: "Workflow submitted but nothing happens"
→ CAUSE: Oozie cannot find workflow.xml on HDFS
→ The oozie.wf.application.path in job.properties must point
  to the HDFS DIRECTORY containing workflow.xml
→ NOT the file itself: /user/oozie/workflows/sales-etl/
  NOT: /user/oozie/workflows/sales-etl/workflow.xml

PITFALL 2: "Action failed: Main class not found"
→ CAUSE: JAR files not in the lib/ subdirectory of workflow path
→ Oozie automatically adds everything in <workflow-path>/lib/ to classpath
→ Structure MUST be:
  /user/oozie/workflows/sales-etl/
    ├── workflow.xml
    └── lib/
        └── your-mapreduce-job.jar    ← MUST be here

PITFALL 3: "EL expression not resolving — shows ${variable} literally"
→ CAUSE: Variable not defined in job.properties or coordinator
→ Every ${...} in workflow.xml must have a corresponding definition
→ Check: oozie validate workflow.xml (validates before submission)

PITFALL 4: "Coordinator creates actions but they stay in WAITING forever"
→ CAUSE: Data dependency not met — _SUCCESS flag missing
→ Your upstream process must create this flag file
→ MapReduce creates it automatically, custom scripts DO NOT
→ Add: hdfs dfs -touchz /data/sales/2024/01/15/_SUCCESS

PITFALL 5: "Hive action fails with ClassNotFoundException"
→ CAUSE: Hive action needs Hive libraries on the Oozie sharelib
→ Oozie uses a shared library in HDFS: /user/oozie/share/lib/
→ Must be populated during Oozie setup:
  oozie-setup.sh sharelib create -fs hdfs://namenode:9000

PITFALL 6: "Fork/Join validation error"
→ CAUSE: Oozie enforces that EVERY path from a fork MUST reach the join
→ You cannot have one fork path go to end while another goes to join
→ ALL parallel paths must converge at the SAME join node

PITFALL 7: "Workflow succeeded but Hive table is empty"
→ CAUSE: Hive action ran but SQL had wrong partition date
→ The action "succeeded" because Hive query executed without SQL error
→ An empty result is NOT a failure from Oozie's perspective
→ SOLUTION: Add a validation step after Hive that checks row count

📈 SCALABILITY CONSIDERATIONS

1. OOZIE SERVER SCALING
   → Single Oozie server can handle ~10,000 concurrent workflows
   → Bottleneck: state database (MySQL/PostgreSQL)
   → For larger scale: tune database connection pool, use SSD storage
   → Oozie HA: Multiple Oozie servers behind load balancer (ZK-based)

2. WORKFLOW COMPLEXITY LIMITS
   → XML parsing becomes slow for workflows with 100+ nodes
   → Use sub-workflows to break into manageable pieces
   → Each sub-workflow is an independent execution unit

3. COORDINATOR MATERIALIZATION PRESSURE
   → A coordinator running every 5 minutes creates 288 actions/day
   → 100 coordinators × 365 days = 10.5 million action records
   → Purge old records: oozie.service.PurgeService.older.than=30

4. LAUNCHER MAPPER OVERHEAD
   → Each action launches a YARN container just for the launcher
   → 10-action workflow = 10 launcher containers (sequential)
   → This adds ~30-60 seconds overhead PER ACTION
   → For fast actions (simple shell commands), overhead > actual work
   → Oozie 5.x introduced "Launcher AM" to reduce this overhead

5. CONCURRENT WORKFLOW LIMIT
   → Default: oozie.service.CallableQueueService.threads=10
   → Controls how many actions Oozie processes concurrently
   → Increase for clusters with many parallel workflows
   → But: more threads = more database connections = tune together

-- Create database if not exists (idempotent)
CREATE DATABASE IF NOT EXISTS ${DATABASE_NAME};
USE ${DATABASE_NAME};

-- Raw sales staging table (external — points to Sqoop output)
CREATE EXTERNAL TABLE IF NOT EXISTS raw_sales (
    transaction_id    BIGINT,
    customer_id       INT,
    product_id        INT,
    product_name      STRING,
    category          STRING,
    quantity          INT,
    unit_price        DECIMAL(10,2),
    total_amount      DECIMAL(10,2),
    sale_date         STRING,
    store_id          INT,
    region            STRING
)
ROW FORMAT DELIMITED
FIELDS TERMINATED BY ','
STORED AS TEXTFILE
LOCATION '/data/sales/cleaned/';

-- Transformed sales table (managed — ORC format, partitioned)
-- As learned in Project 25: partitioned for query performance
-- As learned in Project 30: ORC for compression and speed
CREATE TABLE IF NOT EXISTS transformed_sales (
    transaction_id    BIGINT,
    customer_id       INT,
    product_id        INT,
    product_name      STRING,
    category          STRING,
    quantity          INT,
    unit_price        DECIMAL(10,2),
    total_amount      DECIMAL(10,2),
    store_id          INT
)
PARTITIONED BY (region STRING, sale_date STRING)
STORED AS ORC
TBLPROPERTIES ('orc.compress'='SNAPPY');

-- Daily summary table
CREATE TABLE IF NOT EXISTS daily_sales_summary (
    sale_date         STRING,
    region            STRING,
    total_transactions BIGINT,
    total_revenue     DECIMAL(15,2),
    avg_order_value   DECIMAL(10,2),
    unique_customers  BIGINT,
    top_category      STRING
)
PARTITIONED BY (report_date STRING)
STORED AS ORC;

-- FILE 3: scripts/transform_sales.hql — REGIONAL TRANSFORMATION
-- ═══════════════════════════════════════════════════════════════

USE ${DATABASE_NAME};

-- Enable dynamic partitioning (Project 25 concept)
SET hive.exec.dynamic.partition=true;
SET hive.exec.dynamic.partition.mode=nonstrict;

-- INSERT OVERWRITE ensures idempotency (re-run replaces, not appends)
INSERT OVERWRITE TABLE transformed_sales
PARTITION (region='${REGION}', sale_date='${DATE_PARTITION}')
SELECT
    transaction_id,
    customer_id,
    product_id,
    product_name,
    UPPER(TRIM(category)) AS category,
    quantity,
    unit_price,
    CAST(quantity * unit_price AS DECIMAL(10,2)) AS total_amount,
    store_id
FROM raw_sales
WHERE region = '${REGION}'
  AND sale_date = '${DATE_PARTITION}'
  AND transaction_id IS NOT NULL
  AND quantity > 0
  AND unit_price > 0;
-- ═══════════════════════════════════════════════════════════════
-- FILE 4: scripts/generate_summary.hql — COMBINED SUMMARY
-- ═══════════════════════════════════════════════════════════════

USE ${DATABASE_NAME};

SET hive.exec.dynamic.partition=true;
SET hive.exec.dynamic.partition.mode=nonstrict;

-- Generate daily summary across all regions
INSERT OVERWRITE TABLE daily_sales_summary
PARTITION (report_date='${DATE_PARTITION}')
SELECT
    sale_date,
    region,
    COUNT(*)                            AS total_transactions,
    SUM(total_amount)                   AS total_revenue,
    AVG(total_amount)                   AS avg_order_value,
    COUNT(DISTINCT customer_id)         AS unique_customers,
    -- Subquery to find top category per region
    (SELECT category 
     FROM transformed_sales t2 
     WHERE t2.region = t1.region 
       AND t2.sale_date = '${DATE_PARTITION}'
     GROUP BY category 
     ORDER BY SUM(total_amount) DESC 
     LIMIT 1)                           AS top_category
FROM transformed_sales t1
WHERE sale_date = '${DATE_PARTITION}'
GROUP BY sale_date, region;

package com.retail.etl;

import org.apache.hadoop.io.LongWritable;
import org.apache.hadoop.io.NullWritable;
import org.apache.hadoop.io.Text;
import org.apache.hadoop.mapreduce.Mapper;
import org.apache.hadoop.mapreduce.Counter;

import java.io.IOException;

public class SalesCleanerMapper 
    extends Mapper<LongWritable, Text, NullWritable, Text> {
    
    // Custom counters for pipeline monitoring
    // (Visible in Oozie UI → YARN application → Counters)
    enum CleaningCounters {
        TOTAL_RECORDS,
        VALID_RECORDS,
        NULL_TRANSACTION_ID,
        NEGATIVE_QUANTITY,
        NEGATIVE_PRICE,
        MALFORMED_RECORDS,
        DUPLICATE_HEADERS
    }
    
    private Text outputValue = new Text();
    
    @Override
    protected void map(LongWritable key, Text value, Context context) 
            throws IOException, InterruptedException {
        
        context.getCounter(CleaningCounters.TOTAL_RECORDS).increment(1);
        
        String line = value.toString().trim();
        
        // Skip empty lines
        if (line.isEmpty()) {
            return;
        }
        
        // Skip header rows (Sqoop sometimes includes them)
        if (line.startsWith("transaction_id") || line.startsWith("TRANSACTION_ID")) {
            context.getCounter(CleaningCounters.DUPLICATE_HEADERS).increment(1);
            return;
        }
        
        // Parse CSV fields
        String[] fields = line.split(",");
        
        // Validate field count
        // Expected: transaction_id, customer_id, product_id, product_name,
        //           category, quantity, unit_price, total_amount, 
        //           sale_date, store_id, region
        if (fields.length != 11) {
            context.getCounter(CleaningCounters.MALFORMED_RECORDS).increment(1);
            return;
        }
        
        try {
            // Validate transaction_id is not null/empty
            long transactionId = Long.parseLong(fields[0].trim());
            if (transactionId <= 0) {
                context.getCounter(CleaningCounters.NULL_TRANSACTION_ID).increment(1);
                return;
            }
            
            // Validate quantity is positive
            int quantity = Integer.parseInt(fields[5].trim());
            if (quantity <= 0) {
                context.getCounter(CleaningCounters.NEGATIVE_QUANTITY).increment(1);
                return;
            }
            
            // Validate price is positive
            double unitPrice = Double.parseDouble(fields[6].trim());
            if (unitPrice <= 0) {
                context.getCounter(CleaningCounters.NEGATIVE_PRICE).increment(1);
                return;
            }
            
            // Recalculate total_amount (don't trust source system)
            double correctTotal = quantity * unitPrice;
            
            // Clean and normalize fields
            StringBuilder cleaned = new StringBuilder();
            cleaned.append(transactionId).append(",");                          // transaction_id
            cleaned.append(fields[1].trim()).append(",");                       // customer_id
            cleaned.append(fields[2].trim()).append(",");                       // product_id
            cleaned.append(fields[3].trim().replace(",", "")).append(",");      // product_name (remove internal commas)
            cleaned.append(fields[4].trim().toUpperCase()).append(",");         // category (standardize to uppercase)
            cleaned.append(quantity).append(",");                               // quantity
            cleaned.append(String.format("%.2f", unitPrice)).append(",");       // unit_price
            cleaned.append(String.format("%.2f", correctTotal)).append(",");    // total_amount (recalculated)
            cleaned.append(fields[8].trim()).append(",");                       // sale_date
            cleaned.append(fields[9].trim()).append(",");                       // store_id
            cleaned.append(fields[10].trim().toLowerCase());                    // region (standardize to lowercase)
            
            outputValue.set(cleaned.toString());
            context.write(NullWritable.get(), outputValue);
            context.getCounter(CleaningCounters.VALID_RECORDS).increment(1);
            
        } catch (NumberFormatException e) {
            context.getCounter(CleaningCounters.MALFORMED_RECORDS).increment(1);
            // Record is silently dropped — counter tracks how many
        }
    }
}

import requests
import json
import time
from datetime import datetime, timedelta

class OozieMonitor:
    """
    Monitor Oozie workflows and coordinators programmatically.
    Uses the Oozie REST API (documented at oozie.apache.org).
    """
    
    def __init__(self, oozie_url="http://localhost:11000/oozie"):
        self.base_url = oozie_url
        self.api_url = f"{oozie_url}/v2"
    
    def get_system_status(self):
        """Check if Oozie server is operational."""
        response = requests.get(f"{self.api_url}/admin/status")
        return response.json()
    
    def get_workflow_info(self, job_id):
        """Get detailed info about a specific workflow."""
        response = requests.get(f"{self.api_url}/job/{job_id}?show=info")
        return response.json()
    
    def get_workflow_actions(self, job_id):
        """Get all actions within a workflow."""
        info = self.get_workflow_info(job_id)
        return info.get('actions', [])
    
    def list_workflows(self, status=None, name=None, limit=50):
        """List workflows with optional filters."""
        params = {
            'jobtype': 'wf',
            'len': limit
        }
        
        filters = []
        if status:
            filters.append(f"status={status}")
        if name:
            filters.append(f"name={name}")
        
        if filters:
            params['filter'] = ';'.join(filters)
        
        response = requests.get(f"{self.api_url}/jobs", params=params)
        return response.json().get('workflows', [])
    
    def list_coordinators(self, status=None, limit=50):
        """List coordinator jobs."""
        params = {
            'jobtype': 'coordinator',
            'len': limit
        }
        
        if status:
            params['filter'] = f"status={status}"
        
        response = requests.get(f"{self.api_url}/jobs", params=params)
        return response.json().get('coordinatorjobs', [])
    
    def get_failed_actions_detail(self, job_id):
        """Extract detailed failure info for debugging."""
        actions = self.get_workflow_actions(job_id)
        failed = []
        
        for action in actions:
            if action.get('status') in ['ERROR', 'FAILED', 'KILLED']:
                failed.append({
                    'name': action.get('name'),
                    'type': action.get('type'),
                    'status': action.get('status'),
                    'error_code': action.get('errorCode'),
                    'error_message': action.get('errorMessage'),
                    'yarn_app_id': action.get('externalId'),
                    'console_url': action.get('consoleUrl'),
                    'start_time': action.get('startTime'),
                    'end_time': action.get('endTime')
                })
        
        return failed
    
    def generate_daily_report(self, pipeline_name="sales-etl-pipeline"):
        """
        Generate a daily health report for a pipeline.
        Shows success rate, average duration, failures.
        """
        # Get last 30 days of workflows
        all_workflows = self.list_workflows(name=pipeline_name, limit=30)
        
        report = {
            'pipeline': pipeline_name,
            'report_date': datetime.now().strftime('%Y-%m-%d'),
            'total_runs': len(all_workflows),
            'succeeded': 0,
            'failed': 0,
            'running': 0,
            'avg_duration_minutes': 0,
            'failures': []
        }
        
        durations = []
        
        for wf in all_workflows:
            status = wf.get('status')
            
            if status == 'SUCCEEDED':
                report['succeeded'] += 1
                
                # Calculate duration
                start = wf.get('startTime')
                end = wf.get('endTime')
                if start and end:
                    # Parse Oozie timestamp format
                    try:
                        start_dt = datetime.strptime(
                            start[:19], '%a, %d %b %Y %H:%M:%S')
                        end_dt = datetime.strptime(
                            end[:19], '%a, %d %b %Y %H:%M:%S')
                        duration = (end_dt - start_dt).total_seconds() / 60
                        durations.append(duration)
                    except ValueError:
                        pass
                        
            elif status in ['KILLED', 'FAILED']:
                report['failed'] += 1
                # Get failure details
                failures = self.get_failed_actions_detail(wf['id'])
                for f in failures:
                    report['failures'].append({
                        'workflow_id': wf['id'],
                        'date': wf.get('startTime', 'unknown'),
                        'failed_action': f['name'],
                        'error': f['error_message']
                    })
                    
            elif status == 'RUNNING':
                report['running'] += 1
        
        if durations:
            report['avg_duration_minutes'] = round(
                sum(durations) / len(durations), 1)
        
        report['success_rate'] = (
            f"{(report['succeeded'] / max(report['total_runs'], 1)) * 100:.1f}%"
        )
        
        return report
    
    def wait_for_completion(self, job_id, poll_interval=30, timeout=7200):
        """
        Block until workflow completes or timeout.
        Returns final status.
        """
        start_time = time.time()
        
        while True:
            elapsed = time.time() - start_time
            if elapsed > timeout:
                return {
                    'status': 'TIMEOUT',
                    'message': f'Exceeded {timeout}s timeout'
                }
            
            info = self.get_workflow_info(job_id)
            status = info.get('status')
            
            timestamp = datetime.now().strftime('%H:%M:%S')
            
            # Find currently running action
            running_action = 'unknown'
            for action in info.get('actions', []):
                if action.get('status') == 'RUNNING':
                    running_action = action.get('name')
                    break
            
            print(f"  [{timestamp}] {status} | Action: {running_action} | "
                  f"Elapsed: {elapsed:.0f}s")
            
            if status in ['SUCCEEDED', 'KILLED', 'FAILED']:
                return {
                    'status': status,
                    'duration_seconds': elapsed,
                    'info': info
                }
            
            time.sleep(poll_interval)
