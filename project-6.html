PROJECT 6: FINDING THE MOST POPULAR PRODUCT IN A SMALL SHOP

🔴 HYPOTHETICAL PROBLEM

SCENARIO:
→ You work at an e-commerce company with 50,000 products
→ Transaction logs in HDFS: /ecommerce/raw/transactions/2024/01/
→ Each line: transaction_id,customer_id,product_id,product_name,quantity,price,timestamp
→ Data: ~5GB per month (millions of transactions)

MANAGEMENT ASKS THREE QUESTIONS:

  Q1: What are the TOP 10 most purchased products this month?
  Q2: Rank ALL 50,000 products by total quantity sold (descending)
  Q3: Which product generated the MOST revenue?

WHY THIS IS HARDER THAN PROJECTS 4 AND 5:

  Project 4: Count by category → output SORTED BY KEY (call_type)
  Project 5: Aggregate by station → output SORTED BY KEY (station_id)
  
  THIS PROJECT: We need output SORTED BY VALUE (the count/revenue)
  
  MapReduce AUTOMATICALLY sorts by KEY during shuffle
  MapReduce NEVER automatically sorts by VALUE
  
  So our output from a count job looks like:
    laptop      15234      ← sorted alphabetically by product name
    mouse       89012      ← NOT sorted by count
    phone       45678
    shirt       102345
    watch       23456

  But management wants:
    shirt       102345     ← sorted by count DESCENDING
    mouse       89012
    phone       45678
    watch       23456
    laptop      15234

  HOW DO WE GET FROM KEY-SORTED TO VALUE-SORTED?
  → This requires EITHER:
    A) A second MapReduce job that INVERTS key and value
    B) In-mapper Top-N technique (for "Top 10" only)
    C) WritableComparable with custom sorting (for full ranking)
  
  WE WILL LEARN ALL THREE APPROACHES.

🔍 HOW TO APPROACH IT

FROM PROJECT 4 WE KNOW:
→ Count pattern: emit (key, 1) → sum all 1s in reducer
FROM PROJECT 5 WE KNOW:
→ Aggregation patterns: sum, max, min, average
→ Complete 7-stage data flow including shuffle sorting

NEW CHALLENGE: SORTING BY VALUE

MapReduce guarantees:
→ Keys are SORTED when they reach the reducer
→ All values for one key go to the SAME reducer
→ Values for a given key are NOT sorted (arbitrary order)

MapReduce does NOT guarantee:
→ Any ordering by VALUE
→ Global ordering across multiple reducers (each reducer sorts independently)

THREE STRATEGIES TO SORT BY VALUE:

┌──────────────────────────────────────────────────────────────────────┐
│ STRATEGY A: JOB CHAINING (Two MapReduce Jobs)                       │
│                                                                      │
│ Job 1: Count products                                                │
│   Input:  Raw transaction logs                                       │
│   Output: (product_name, count) — sorted by product_name            │
│                                                                      │
│ Job 2: Sort by count                                                 │
│   Input:  Output of Job 1                                            │
│   Mapper: SWAP key and value → emit (count, product_name)           │
│   Shuffle: Automatically sorts by count (now the key!)              │
│   Reducer: SWAP back → emit (product_name, count)                   │
│   Output: Products sorted by count                                   │
│                                                                      │
│ KEY INSIGHT: Make the VALUE become the KEY                           │
│ → MapReduce sorts by key automatically                               │
│ → If count IS the key → output sorted by count                      │
│                                                                      │
│ PROBLEM: Default sort is ASCENDING                                   │
│ → We want DESCENDING (highest count first)                          │
│ → Solution: Use a custom comparator or negate the count             │
│                                                                      │
└──────────────────────────────────────────────────────────────────────┘

┌──────────────────────────────────────────────────────────────────────┐
│ STRATEGY B: IN-MAPPER TOP-N (Single Job, Approximate for Top N)     │
│                                                                      │
│ Each mapper maintains a local TreeMap of top N products              │
│ In cleanup(): emit only the top N from this mapper                  │
│ Reducer receives top-N from EACH mapper → finds global top N        │
│                                                                      │
│ ADVANTAGE: Single job, minimal shuffle data                          │
│ LIMITATION: Only works for "Top N" — not full ranking               │
│                                                                      │
└──────────────────────────────────────────────────────────────────────┘

┌──────────────────────────────────────────────────────────────────────┐
│ STRATEGY C: WritableComparable (Custom Sort Key)                    │
│                                                                      │
│ Create a custom key class that contains BOTH product and count      │
│ Implement compareTo() to sort by count descending                   │
│ MapReduce uses YOUR compareTo() during shuffle sort                 │
│                                                                      │
│ ADVANTAGE: Full control over sort order                              │
│ FOUNDATION: This leads to Secondary Sort (Project 19)               │
│                                                                      │
└──────────────────────────────────────────────────────────────────────┘

═══ JOB 2: Sort by Count (KEY-VALUE INVERSION) ═══

Input:    Output of Job 1 (tab-delimited: product\tcount)

  "laptop\t3"  → emit(IntWritable(3), Text("laptop"))    ← SWAPPED!
  "mouse\t3"   → emit(IntWritable(3), Text("mouse"))
  "phone\t7"   → emit(IntWritable(7), Text("phone"))
  "shirt\t12"  → emit(IntWritable(12), Text("shirt"))
  "watch\t5"   → emit(IntWritable(5), Text("watch"))

  3  → ["laptop", "mouse"]     ← ascending order (default)
  5  → ["watch"]
  7  → ["phone"]
  12 → ["shirt"]

  Option 1: Negate the count → emit(-12, "shirt"), (-7, "phone")...
  Option 2: Custom WritableComparable with reversed compareTo()

Reducer (with descending sort):
  12 → ["shirt"]   → emit("shirt", 12)     ← RANK 1
  7  → ["phone"]   → emit("phone", 7)      ← RANK 2
  5  → ["watch"]   → emit("watch", 5)      ← RANK 3
  3  → ["laptop", "mouse"] → emit both     ← RANK 4 (tie)

Output: /ecommerce/output/product_ranking/part-r-00000
  shirt     12       ← NOW SORTED BY COUNT DESCENDING
  phone     7
  watch     5
  laptop    3
  mouse     3

STRATEGY B — IN-MAPPER TOP-N — DETAILED DATA FLOW:

Mapper 1 processes Block 1:
  Maintains HashMap: {laptop:2, mouse:3, shirt:8, phone:4, ...}
  In cleanup(): sort by value, emit TOP 10 only
  → ("shirt", 8), ("phone", 4), ("mouse", 3), ("laptop", 2), ...

Mapper 2 processes Block 2:
  Maintains HashMap: {shirt:4, watch:5, phone:3, laptop:1, ...}
  In cleanup(): sort by value, emit TOP 10 only
  → ("watch", 5), ("shirt", 4), ("phone", 3), ("laptop", 1), ...

Reducer receives:
  "laptop" → [2, 1]        → sum = 3
  "mouse"  → [3]           → sum = 3
  "phone"  → [4, 3]        → sum = 7
  "shirt"  → [8, 4]        → sum = 12
  "watch"  → [5]           → sum = 5

Reducer maintains top-10 TreeMap, emits at end:
  shirt: 12, phone: 7, watch: 5, laptop: 3, mouse: 3

⚠️ LIMITATION: If a product is in top-10 GLOBALLY but NOT in
   top-10 of ANY individual mapper → IT GETS LOST
   This is APPROXIMATE for products near the boundary
   EXACT for clearly dominant products

FOR EXACT TOP-N: Use Job Chaining (Strategy A)

UNDERSTANDING WritableComparable — THE KEY INTERFACE:

Every MapReduce KEY must implement WritableComparable<T>
This interface requires THREE methods:

┌──────────────────────────────────────────────────────────────────────┐
│ public interface WritableComparable<T> extends Writable, Comparable<T>│
│                                                                      │
│ METHOD 1: write(DataOutput out)                                     │
│   → Serialize this object to bytes (for network transfer)           │
│   → Called when mapper sends key through shuffle                    │
│                                                                      │
│ METHOD 2: readFields(DataInput in)                                  │
│   → Deserialize bytes back to object                                │
│   → Called when reducer receives key from shuffle                   │
│                                                                      │
│ METHOD 3: compareTo(T other)                                        │
│   → Define sort order between two keys                              │
│   → Return negative: this < other                                   │
│   → Return zero: this == other                                      │
│   → Return positive: this > other                                   │
│   → MapReduce calls this DURING SHUFFLE SORT                        │
│   → THIS IS HOW YOU CONTROL SORT ORDER                              │
│                                                                      │
│ ALSO NEEDED:                                                         │
│ hashCode(): Used by HashPartitioner to assign keys to reducers      │
│ equals(): Used for key grouping in reducer                          │
│                                                                      │
└──────────────────────────────────────────────────────────────────────┘

BUILT-IN WritableComparable CLASSES AND THEIR SORT ORDER:

┌────────────────────┬──────────────────────────────────────────────┐
│ CLASS              │ DEFAULT compareTo() BEHAVIOR                 │
├────────────────────┼──────────────────────────────────────────────┤
│ Text               │ Lexicographic (alphabetical) ascending       │
│                    │ "apple" < "banana" < "cherry"                │
├────────────────────┼──────────────────────────────────────────────┤
│ IntWritable        │ Numeric ascending                            │
│                    │ 1 < 5 < 10 < 100                            │
├────────────────────┼──────────────────────────────────────────────┤
│ LongWritable       │ Numeric ascending                            │
│                    │ Same as IntWritable but for long values      │
├────────────────────┼──────────────────────────────────────────────┤
│ DoubleWritable     │ Numeric ascending                            │
│                    │ 1.5 < 2.7 < 10.3                            │
├────────────────────┼──────────────────────────────────────────────┤
│ NullWritable       │ All instances are equal                      │
│                    │ compareTo() always returns 0                 │
└────────────────────┴──────────────────────────────────────────────┘

TO SORT DESCENDING: Override compareTo() to REVERSE the comparison
→ return -1 * Integer.compare(this.count, other.count)
→ OR: return other.count - this.count
→ This makes higher counts sort FIRST

✅ WHAT SHOULD DO

1. USE Job Chaining for EXACT full ranking
   → Two jobs: Count → Sort
   → Job 2 reads Job 1's output from HDFS
   → Guaranteed correct ordering
   → Works for any data size

2. USE In-Mapper Top-N for approximate fast results
   → Single job, much faster
   → Perfect when you ONLY need top 10/20/100
   → Acceptable when borderline products can be missed
   → Great for dashboards, monitoring, alerts

3. ALWAYS use 1 reducer for global ranking
   → Multiple reducers = each sorts independently
   → Reducer 0 has rank 1-1000, Reducer 1 has rank 1001-2000
   → But they are NOT globally sorted across files
   → For true global sort: either 1 reducer or TotalOrderPartitioner (Project 18)

4. USE cleanup() method in mapper for aggregation-then-emit pattern
   → setup(): Called ONCE before first map() call — initialize data structures
   → map(): Called per line — accumulate into data structures
   → cleanup(): Called ONCE after last map() call — emit aggregated results
   → This is the IN-MAPPER AGGREGATION pattern

5. HANDLE ties in ranking
   → Two products with count=100 should share the same rank
   → Or: secondary sort by product name for deterministic ordering
   → Document your tie-breaking strategy

6. CONSIDER memory when using in-mapper aggregation
   → HashMap of 50,000 products × ~50 bytes per entry = ~2.5MB — fine
   → HashMap of 50,000,000 products → 2.5GB — may exceed mapper memory
   → For very high cardinality: use Job Chaining instead

🚫 WHAT SHOULD NOT DO

1. DO NOT assume MapReduce sorts output by value
   → It ONLY sorts by key
   → "I set the count as value and expected it sorted" — WRONG
   → You must explicitly make the count a KEY for sorting

2. DO NOT use multiple reducers for global ranking without TotalOrderPartitioner
   → 3 reducers produce 3 independently sorted files
   → part-r-00000 might have: shirt(12), phone(7)
   → part-r-00001 might have: watch(15), hat(2)
   → watch(15) should be rank 1 but it's in file 1, not file 0
   → USE 1 reducer for small output, TotalOrderPartitioner for large (Project 18)

3. DO NOT collect ALL values into a List in the reducer
   → For top-N: you only need to track N items
   → Collecting all 50,000 products into a List: wasteful but works
   → Collecting all 50,000,000 products: OutOfMemoryError
   → USE: TreeMap with size limit, or PriorityQueue

4. DO NOT negate Long.MIN_VALUE for descending sort
   → -Long.MIN_VALUE == Long.MIN_VALUE (overflow!)
   → Safe for counts that fit in normal range
   → For safety: use custom WritableComparable instead of negation trick

5. DO NOT forget to sum quantities (not just count transactions)
   → Transaction: laptop, quantity=3 → this is 3 units sold, not 1
   → Project 4 counted occurrences (each line = 1)
   → This project sums QUANTITIES (each line has a quantity field)
   → emit(product, quantity) not emit(product, 1)

6. DO NOT chain jobs by passing output path as a string
   → Use the actual Path object from Job 1's output
   → Verify Job 1 succeeded BEFORE starting Job 2
   → If Job 1 fails and Job 2 reads empty/missing directory → wrong results

   → Financial reports, regulatory submissions
   → Products near the top-N boundary may be missed
   → Use job chaining for exactness

2. DO NOT use job chaining when a single-pass solution exists
   → If you only need the SINGLE most popular product:
     → One job with reducer tracking running max is enough
     → No need for a second sorting job
   → Job chaining adds overhead: extra HDFS write + read + job startup

3. DO NOT use 1 reducer for ranking millions of unique keys
   → 1 reducer = 1 machine processes ALL data
   → 50 million products × 50 bytes each = 2.5GB in one reducer
   → Reducer becomes bottleneck — job takes hours
   → Use TotalOrderPartitioner (Project 18) for large-scale global sort

4. DO NOT use MapReduce for interactive ranking queries
   → "Show me top 10 products RIGHT NOW"
   → MapReduce takes minutes to start
   → Pre-compute rankings in batch, serve from HBase/database
   → Or use Hive (Project 24) for SQL-style ORDER BY

MECHANIC 3: HOW SortComparator CONTROLS SORT ORDER

┌──────────────────────────────────────────────────────────────────────┐
│                                                                      │
│ During shuffle, MapReduce sorts keys using a COMPARATOR              │
│                                                                      │
│ DEFAULT: Uses the key class's compareTo() method                    │
│   → IntWritable.compareTo(): ascending numeric order                │
│   → Text.compareTo(): ascending lexicographic order                 │
│                                                                      │
│ OVERRIDE: Set a custom SortComparator on the job                    │
│   job.setSortComparatorClass(DescendingIntComparator.class)         │
│                                                                      │
│ The comparator is called MILLIONS of times during sort              │
│ It operates on RAW BYTES (not deserialized objects) for speed:      │
│                                                                      │
│   public class DescendingIntComparator                               │
│       extends WritableComparator {                                   │
│                                                                      │
│       public int compare(byte[] b1, int s1, int l1,                 │
│                          byte[] b2, int s2, int l2) {               │
│           // Read ints from raw bytes                                │
│           int val1 = readInt(b1, s1);                                │
│           int val2 = readInt(b2, s2);                                │
│           // REVERSE comparison for descending                       │
│           return -1 * Integer.compare(val1, val2);                   │
│       }                                                              │
│   }                                                                  │
│                                                                      │
│ WHY RAW BYTES?                                                       │
│ → Deserializing millions of keys into Java objects is SLOW          │
│ → Comparing raw bytes avoids object creation overhead               │
│ → WritableComparator provides utility methods: readInt, readLong    │
│ → This is a critical performance optimization in MapReduce          │
│                                                                      │
└──────────────────────────────────────────────────────────────────────┘

MECHANIC 4: HOW IN-MAPPER AGGREGATION WORKS WITH setup/map/cleanup

┌──────────────────────────────────────────────────────────────────────┐
│                                                                      │
│ MAPPER LIFECYCLE (called by MapReduce framework):                    │
│                                                                      │
│   ① setup(Context context)                                          │
│      → Called ONCE when mapper task starts                           │
│      → Initialize data structures (HashMap, TreeMap)                │
│      → Read configuration parameters                                 │
│      → Load side data (Project 15)                                   │
│                                                                      │
│   ② map(key, value, context)                                        │
│      → Called ONCE PER INPUT RECORD                                 │
│      → Process the record                                            │
│      → Can emit immediately OR accumulate in data structure          │
│                                                                      │
│   ③ cleanup(Context context)                                        │
│      → Called ONCE when all records processed                        │
│      → Emit accumulated results                                      │
│      → Close resources                                               │
│                                                                      │
│ FOR IN-MAPPER TOP-N:                                                 │
│                                                                      │
│   setup():                                                           │
│     productCounts = new HashMap<String, Long>()                      │
│                                                                      │
│   map():                                                             │
│     parse product and quantity from line                              │
│     productCounts.merge(product, quantity, Long::sum)                │
│     // NO context.write() here — accumulating                       │
│                                                                      │
│   cleanup():                                                         │
│     sort productCounts by value descending                           │
│     emit TOP N entries via context.write()                           │
│                                                                      │
│ RESULT: Each mapper emits at most N records instead of millions     │
│ SHUFFLE: Only N × numMappers records transferred (tiny!)            │
│ REDUCER: Receives N × numMappers records, finds global top N        │
│                                                                      │
└──────────────────────────────────────────────────────────────────────┘

⚠️ COMMON PITFALLS AND FAILURE MODES

PITFALL 1: "Descending sort shows negative numbers instead of high counts"
→ REASON: Used negation trick (emit -count) but forgot to negate back
→ Output shows: -102345, -89012, -45678...
→ FIX: In reducer, negate back: context.write(product, new IntWritable(-key.get()))
→ BETTER FIX: Use custom comparator — cleaner, no negation needed

PITFALL 2: "Products with same count appear in random order"
→ REASON: MapReduce only guarantees sort by key
→ If key is count (e.g., 100), two products with count=100 are "equal"
→ Their relative order is undefined
→ FIX: Use composite key (count + product_name) with custom comparator
→ Compare by count first (desc), then product_name (asc) for ties

PITFALL 3: "In-mapper top-N misses a product that should be in top 10"
→ REASON: Product was split across many mappers, none saw enough to rank it
→ EXAMPLE: Product X has 10,000 sales split evenly across 100 mappers
→ Each mapper sees only 100 sales for X — not in any mapper's top 10
→ But globally, X has 10,000 sales — should be in top 10
→ FIX: Use Job Chaining for exact results
→ MITIGATION: Use top-100 in mapper to catch borderline products

PITFALL 4: "Job 2 fails — cannot read Job 1 output"

import java.io.IOException;
import java.util.*;

import org.apache.hadoop.conf.Configuration;
import org.apache.hadoop.fs.Path;
import org.apache.hadoop.fs.FileSystem;
import org.apache.hadoop.io.IntWritable;
import org.apache.hadoop.io.LongWritable;
import org.apache.hadoop.io.NullWritable;
import org.apache.hadoop.io.Text;
import org.apache.hadoop.io.WritableComparable;
import org.apache.hadoop.io.WritableComparator;
import org.apache.hadoop.mapreduce.Job;
import org.apache.hadoop.mapreduce.Mapper;
import org.apache.hadoop.mapreduce.Reducer;
import org.apache.hadoop.mapreduce.lib.input.FileInputFormat;
import org.apache.hadoop.mapreduce.lib.output.FileOutputFormat;


public class ProductRanking {

    // ================================================================
    // APPROACH A: JOB CHAINING (Count → Sort)
    // ================================================================

    // ──── JOB 1 MAPPER: Parse transactions, emit (product, quantity) ────
    public static class ProductCountMapper
            extends Mapper<LongWritable, Text, Text, LongWritable> {

        private final Text productKey = new Text();
        private final LongWritable quantityValue = new LongWritable();

        // Column indices: tx_id(0), cust_id(1), prod_id(2), 
        // prod_name(3), quantity(4), price(5), timestamp(6)
        private static final int COL_PRODUCT_NAME = 3;
        private static final int COL_QUANTITY = 4;

        @Override
        protected void map(LongWritable key, Text value, Context context)
                throws IOException, InterruptedException {

            String line = value.toString();

            // Skip header (Project 4 pattern)
            if (line.startsWith("transaction_id")) {
                context.getCounter("ProductRanking", "headers_skipped").increment(1);
                return;
            }

            try {
                String[] fields = line.split(",");
                if (fields.length < 7) {
                    context.getCounter("ProductRanking", "malformed_lines").increment(1);
                    return;
                }

                String productName = fields[COL_PRODUCT_NAME].trim();
                long quantity = Long.parseLong(fields[COL_QUANTITY].trim());

                if (quantity <= 0) {
                    context.getCounter("ProductRanking", "invalid_quantity").increment(1);
                    return;
                }

                // Emit product name as key, QUANTITY as value (not 1!)
                productKey.set(productName);
                quantityValue.set(quantity);
                context.write(productKey, quantityValue);

                context.getCounter("ProductRanking", "valid_transactions").increment(1);
                context.getCounter("ProductRanking", "total_quantity").increment(quantity);

            } catch (Exception e) {
                context.getCounter("ProductRanking", "parse_errors").increment(1);
            }
        }
    }

    // ──── JOB 1 REDUCER: Sum quantities per product ────
    public static class ProductCountReducer
            extends Reducer<Text, LongWritable, Text, LongWritable> {

        private final LongWritable totalQuantity = new LongWritable();

        @Override
        protected void reduce(Text key, Iterable<LongWritable> values, Context context)
                throws IOException, InterruptedException {

            long sum = 0;
            for (LongWritable val : values) {
                sum += val.get();
            }

            totalQuantity.set(sum);
            context.write(key, totalQuantity);
        }
    }

    // ──── JOB 2 MAPPER: Invert key-value (product,count) → (count,product) ────
    public static class InvertMapper
            extends Mapper<LongWritable, Text, LongWritable, Text> {

        private final LongWritable countKey = new LongWritable();
        private final Text productValue = new Text();

        @Override
        protected void map(LongWritable key, Text value, Context context)
                throws IOException, InterruptedException {

            // Input: "product_name\tcount" (tab-delimited from Job 1)
            String line = value.toString();
            String[] parts = line.split("\t");

            if (parts.length == 2) {
                try {
                    String product = parts[0].trim();
                    long count = Long.parseLong(parts[1].trim());

                    // INVERT: count becomes key, product becomes value
                    countKey.set(count);
                    productValue.set(product);
                    context.write(countKey, productValue);

                } catch (NumberFormatException e) {
                    context.getCounter("SortJob", "parse_errors").increment(1);
                }
            }
        }
    }

    // ──── DESCENDING SORT COMPARATOR ────
    // This makes MapReduce sort keys (counts) in DESCENDING order
    public static class DescendingLongComparator extends WritableComparator {

        protected DescendingLongComparator() {
            super(LongWritable.class, true);
        }

        @Override
        public int compare(WritableComparable a, WritableComparable b) {
            LongWritable lw1 = (LongWritable) a;
            LongWritable lw2 = (LongWritable) b;
            // REVERSE comparison: b.compareTo(a) instead of a.compareTo(b)
            return -1 * Long.compare(lw1.get(), lw2.get());
        }
    }

    // ──── JOB 2 REDUCER: Invert back (count,product) → (product,count) ────
    public static class InvertBackReducer
            extends Reducer<LongWritable, Text, Text, LongWritable> {

        private int rank = 0;

        @Override
        protected void reduce(LongWritable key, Iterable<Text> values, Context context)
                throws IOException, InterruptedException {

            // All products with THIS count arrive here
            // Since keys are sorted descending, first call has highest count
            for (Text product : values) {
                rank++;
                // Emit: product_name \t count
                context.write(new Text(rank + ".\t" + product.toString()), key);
            }
        }
    }


    // ================================================================
    // APPROACH B: IN-MAPPER TOP-N (Single Job)
    // ================================================================

    // ──── MAPPER: Accumulate counts in memory, emit top N in cleanup ────
    public static class TopNMapper
            extends Mapper<LongWritable, Text, Text, LongWritable> {

        private Map<String, Long> productCounts;
        private int topN;

        @Override
        protected void setup(Context context) {
            productCounts = new HashMap<>();
            topN = context.getConfiguration().getInt("topN", 10);
        }

        @Override
        protected void map(LongWritable key, Text value, Context context)
                throws IOException, InterruptedException {

            String line = value.toString();
            if (line.startsWith("transaction_id")) return;

            try {
                String[] fields = line.split(",");
                if (fields.length >= 7) {
                    String product = fields[3].trim();
                    long quantity = Long.parseLong(fields[4].trim());

                    if (quantity > 0) {
                        // Accumulate in HashMap (no emit yet!)
                        productCounts.merge(product, quantity, Long::sum);
                    }
                }
            } catch (Exception e) {
                // Skip malformed
            }
        }

        @Override
        protected void cleanup(Context context)
                throws IOException, InterruptedException {

            // Sort by value descending, emit top N
            List<Map.Entry<String, Long>> sorted = new ArrayList<>(productCounts.entrySet());
            sorted.sort((a, b) -> Long.compare(b.getValue(), a.getValue()));

            Text productKey = new Text();
            LongWritable countValue = new LongWritable();

            int emitted = 0;
            for (Map.Entry<String, Long> entry : sorted) {
                if (emitted >= topN) break;

                productKey.set(entry.getKey());
                countValue.set(entry.getValue());
                context.write(productKey, countValue);
                emitted++;
            }

            context.getCounter("TopN", "mapper_unique_products").increment(productCounts.size());
            context.getCounter("TopN", "mapper_emitted").increment(emitted);
        }
    }

    // ──── REDUCER: Find global top N from all mappers' top N ────
    public static class TopNReducer
            extends Reducer<Text, LongWritable, Text, LongWritable> {

        private TreeMap<Long, List<String>> topNMap;
        private int topN;

        @Override
        protected void setup(Context context) {
            topNMap = new TreeMap<>(Collections.reverseOrder()); // descending
            topN = context.getConfiguration().getInt("topN", 10);
        }

        @Override
        protected void reduce(Text key, Iterable<LongWritable> values, Context context)
                throws IOException, InterruptedException {

            // Sum counts from all mappers for this product
            long totalCount = 0;
            for (LongWritable val : values) {
                totalCount += val.get();
            }

            // Add to TreeMap (sorted by count descending)
            topNMap.computeIfAbsent(totalCount, k -> new ArrayList<>()).add(key.toString());

            // Trim to keep only top N entries
            int currentSize = topNMap.values().stream().mapToInt(List::size).sum();
            while (currentSize > topN && !topNMap.isEmpty()) {
                Map.Entry<Long, List<String>> lastEntry = topNMap.lastEntry();
                List<String> products = lastEntry.getValue();
                products.remove(products.size() - 1);
                if (products.isEmpty()) {
                    topNMap.pollLastEntry();
                }
                currentSize--;
            }
        }

        @Override
        protected void cleanup(Context context)
                throws IOException, InterruptedException {

            // Emit final top N
            Text productKey = new Text();
            LongWritable countValue = new LongWritable();
            int rank = 0;

            for (Map.Entry<Long, List<String>> entry : topNMap.entrySet()) {
                for (String product : entry.getValue()) {
                    rank++;
                    if (rank > topN) break;
                    productKey.set(rank + ".\t" + product);
                    countValue.set(entry.getKey());
                    context.write(productKey, countValue);
                }
                if (rank > topN) break;
            }
        }
    }

    // DRIVER: Supports all three approaches
    // ================================================================

    public static void main(String[] args) throws Exception {

        if (args.length < 3) {
            System.err.println("Usage: ProductRanking <mode> <input> <output> [topN]");
            System.err.println("  mode: 'chain' (exact full ranking) or 'topn' (fast top-N)");
            System.err.println("  topN: number of top products (default 10, only for 'topn' mode)");
            System.exit(1);
        }

        String mode = args[0].toLowerCase();
        String inputPath = args[1];
        String outputPath = args[2];
        int topN = args.length > 3 ? Integer.parseInt(args[3]) : 10;

        Configuration conf = new Configuration();
        FileSystem fs = FileSystem.get(conf);

        if (mode.equals("chain")) {
            // ════════════════════════════════════════
            // APPROACH A: JOB CHAINING
            // ════════════════════════════════════════

            String intermediatePath = outputPath + "_intermediate";

            // ──── JOB 1: Count products ────
            Path intermediateOut = new Path(intermediatePath);
            if (fs.exists(intermediateOut)) fs.delete(intermediateOut, true);

            Job job1 = Job.getInstance(conf, "Product Count (Job 1 of 2)");
            job1.setJarByClass(ProductRanking.class);
            job1.setMapperClass(ProductCountMapper.class);
            job1.setReducerClass(ProductCountReducer.class);
            job1.setOutputKeyClass(Text.class);
            job1.setOutputValueClass(LongWritable.class);
            job1.setNumReduceTasks(1);
            FileInputFormat.addInputPath(job1, new Path(inputPath));
            FileOutputFormat.setOutputPath(job1, intermediateOut);

            System.out.println("=== JOB 1: Counting products ===");
            long start1 = System.currentTimeMillis();
            boolean success1 = job1.waitForCompletion(true);
            long elapsed1 = System.currentTimeMillis() - start1;
            System.out.println("Job 1: " + (success1 ? "SUCCESS" : "FAILED") +
                             " (" + elapsed1/1000 + "s)");

            if (!success1) {
                System.err.println("Job 1 failed — aborting pipeline");
                System.exit(1);
            }

            // ──── JOB 2: Sort by count descending ────
            Path finalOut = new Path(outputPath);
            if (fs.exists(finalOut)) fs.delete(finalOut, true);

            Job job2 = Job.getInstance(conf, "Sort by Count Descending (Job 2 of 2)");
            job2.setJarByClass(ProductRanking.class);
            job2.setMapperClass(InvertMapper.class);
            job2.setReducerClass(InvertBackReducer.class);

            // Map output: (LongWritable count, Text product)
            job2.setMapOutputKeyClass(LongWritable.class);
            job2.setMapOutputValueClass(Text.class);

            // Final output: (Text "rank.\tproduct", LongWritable count)
            job2.setOutputKeyClass(Text.class);
            job2.setOutputValueClass(LongWritable.class);

            // CRITICAL: Set descending sort comparator
            job2.setSortComparatorClass(DescendingLongComparator.class);

            job2.setNumReduceTasks(1); // Global ordering requires 1 reducer
            FileInputFormat.addInputPath(job2, intermediateOut);
            FileOutputFormat.setOutputPath(job2, finalOut);

            System.out.println("\n=== JOB 2: Sorting by count descending ===");
            long start2 = System.currentTimeMillis();
            boolean success2 = job2.waitForCompletion(true);
            long elapsed2 = System.currentTimeMillis() - start2;
            System.out.println("Job 2: " + (success2 ? "SUCCESS" : "FAILED") +
                             " (" + elapsed2/1000 + "s)");
            System.out.println("Total pipeline: " + (elapsed1 + elapsed2)/1000 + "s");

            System.exit(success2 ? 0 : 1);

        } else if (mode.equals("topn")) {
            // ════════════════════════════════════════
            // APPROACH B: IN-MAPPER TOP-N
            // ════════════════════════════════════════

            conf.setInt("topN", topN);

            Path outPath = new Path(outputPath);
            if (fs.exists(outPath)) fs.delete(outPath, true);

            Job job = Job.getInstance(conf, "Top " + topN + " Products");
            job.setJarByClass(ProductRanking.class);
            job.setMapperClass(TopNMapper.class);
            job.setReducerClass(TopNReducer.class);
            job.setOutputKeyClass(Text.class);
            job.setOutputValueClass(LongWritable.class);
            job.setNumReduceTasks(1);
            FileInputFormat.addInputPath(job, new Path(inputPath));
            FileOutputFormat.setOutputPath(job, outPath);

            System.out.println("=== TOP " + topN + " PRODUCTS (In-Mapper Approach) ===");
            long start = System.currentTimeMillis();
            boolean success = job.waitForCompletion(true);
            long elapsed = System.currentTimeMillis() - start;
            System.out.println("Job: " + (success ? "SUCCESS" : "FAILED") +
                             " (" + elapsed/1000 + "s)");

            System.exit(success ? 0 : 1);
        } else {
            System.err.println("Unknown mode: " + mode);
            System.err.println("Use 'chain' or 'topn'");
            System.exit(1);
        }
    }
}

WHAT WE ESTABLISHED IN PROJECT 6:

✅ VALUE-SORTING IS NOT FREE IN MAPREDUCE:
   → MapReduce sorts by KEY automatically during shuffle
   → Sorting by VALUE requires explicit strategies
   → This is a fundamental MapReduce limitation and design consideration

✅ THREE STRATEGIES FOR VALUE-BASED SORTING:

   STRATEGY A — JOB CHAINING:
   → Job 1: Aggregate (count/sum per key)
   → Job 2: Invert key-value → shuffle sorts by former-value
   → EXACT results, works at any scale
   → Cost: 2× job overhead + intermediate HDFS I/O
   
   STRATEGY B — IN-MAPPER TOP-N:
   → Single job with HashMap accumulation in map()
   → Emit only top N in cleanup()
   → APPROXIMATE results, very fast, minimal shuffle
   → Limitation: borderline products may be missed
   
   STRATEGY C — CUSTOM COMPARATOR:
   → WritableComparator with reversed comparison
   → Controls sort order during shuffle phase
   → Operates on raw bytes for performance
   → Foundation for Secondary Sort (Project 19)

✅ MAPPER LIFECYCLE: setup() → map() × N → cleanup()
   → setup(): Initialize data structures (called ONCE)
   → map(): Process records (called PER LINE)
   → cleanup(): Emit accumulated results (called ONCE)
   → This pattern enables in-mapper aggregation

✅ JOB CHAINING MECHANICS:
   → Jobs connected via HDFS intermediate output
   → No in-memory state passing between jobs
   → Must verify Job 1 success before starting Job 2
   → Intermediate data is small (aggregated) → Job 2 is fast

✅ WritableComparable INTERFACE:
   → write(): Serialize to bytes
   → readFields(): Deserialize from bytes
   → compareTo(): Define sort order
   → hashCode(): Used by partitioner
   → Controls how MapReduce sorts and groups keys
