PROJECT 25: PARTITIONING AND BUCKETING A HIVE TABLE FOR FASTER QUERIES

🔴 HYPOTHETICAL PROBLEM

SCENARIO:
→ You are a data engineer at a global ride-sharing company
→ Your Hive warehouse (from Project 24 setup) holds a "rides" table
→ 2.8 BILLION rows covering 3 years of ride data
→ Schema: ride_id, rider_id, driver_id, city, country,
          fare, distance_km, ride_date, payment_method

WHAT ANALYSTS ASK EVERY DAY:
→ "Total revenue in New York for December 2024"
→ "Average fare per city for Q3 2024"
→ "Compare cash vs. card payments in London last week"

WHAT HAPPENS WITH A NORMAL HIVE TABLE (Project 24 style):
→ Query: SELECT SUM(fare) FROM rides WHERE country='US' AND city='New York'
         AND ride_date BETWEEN '2024-12-01' AND '2024-12-31';
→ Hive launches MapReduce that scans ALL 2.8 BILLION rows
→ 98% of data is IRRELEVANT (not New York, not December)
→ Query takes 4.5 HOURS
→ Analysts are furious — they need answers in MINUTES

THE CORE WASTE:
→ Full table scan on 2.8 billion rows to find ~12 million relevant rows
→ Reading 800GB from HDFS when only 3GB is needed
→ This is like searching EVERY book in a library
   when you know you need "History, Shelf 7, Row 3"

HADOOP ALREADY HAS THE DATA — THE PROBLEM IS HOW IT'S ORGANIZED.
PARTITIONING AND BUCKETING SOLVE THIS.

🔍 HOW TO APPROACH IT

STEP 1: Understand WHY full table scans happen

In Project 24, we created tables where ALL data sits in ONE directory:
   /user/hive/warehouse/rides/
       ├── 000000_0    (huge file)
       ├── 000001_0    (huge file)
       ├── 000002_0    (huge file)
       └── ... (hundreds of files, ALL mixed together)

Every query reads EVERY file regardless of the WHERE clause.
Hive has NO WAY to skip irrelevant data.

STEP 2: Identify the TWO optimization strategies

┌─────────────────────────────────────────────────────────────────┐
│                                                                 │
│  STRATEGY 1: PARTITIONING                                       │
│  → Physically organize data into SUBDIRECTORIES by column value │
│  → Query on partition column = skip entire directories          │
│  → Analogy: Organizing a library by FLOOR (History, Science...) │
│                                                                 │
│  STRATEGY 2: BUCKETING                                          │
│  → Within each partition, split into FIXED number of files      │
│  → Based on HASH of a column value                              │
│  → Analogy: Within the History floor, organize by SHELF NUMBER  │
│  → Enables efficient sampling and map-side joins                │
│                                                                 │
│  COMBINED: Partitioning + Bucketing = surgical data access      │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘

STEP 3: Map the query patterns to the strategy

   Query Pattern: "WHERE country='US' AND city='New York'"
   → PARTITION BY (country, city)
   → Hive reads ONLY /rides/country=US/city=New_York/
   → Skips ALL other countries and cities entirely

   Query Pattern: "JOIN rides ON rider_id"
   → BUCKET BY (rider_id)
   → Matching rider_ids guaranteed in same bucket number
   → Enables map-side join (no shuffle needed — see Project 16)

TWO TYPES OF PARTITIONING:

TYPE 1 — STATIC PARTITIONING:
→ You explicitly specify the partition value in INSERT
→ INSERT INTO rides_opt PARTITION(country='US', city='New_York')
→ You KNOW what value goes where
→ Best for: controlled ETL pipelines

TYPE 2 — DYNAMIC PARTITIONING:
→ Hive reads the column value from the data itself
→ INSERT INTO rides_opt PARTITION(country, city) SELECT ... FROM rides
→ Hive automatically creates directories for each unique combination
→ Best for: bulk loading historical data with many partition values

✅ WHAT SHOULD DO

1. CHOOSE PARTITION COLUMNS BASED ON QUERY PATTERNS
   → Partition by columns that appear in WHERE clauses
   → Partition by columns with LOW-TO-MEDIUM cardinality
     (country: ~200 values ✅, city: ~10,000 values ⚠️, ride_id: billions ❌)

2. LIMIT PARTITION DEPTH
   → 1-3 levels of partitioning is ideal
   → Example: PARTITION BY (year, month, country) = 3 levels
   → More levels = more directories = more NameNode metadata pressure

3. CHOOSE BUCKET COLUMNS BASED ON JOIN/SAMPLE PATTERNS
   → Bucket by columns used in JOIN conditions
   → Bucket by columns used in frequent GROUP BY
   → Choose columns with HIGH cardinality (rider_id, driver_id)

4. SET BUCKET COUNT AS POWER OF 2 OR MEANINGFUL NUMBER
   → Common choices: 4, 8, 16, 32, 64, 128, 256
   → Target: each bucket file should be ~128MB-256MB
   → If partition has 1GB data → 4-8 buckets
   → If partition has 100GB data → 256-512 buckets

5. COMBINE WITH ORC/PARQUET (Project 30 concepts)
   → Partitioning + Bucketing + Columnar format = maximum performance
   → ORC with partitioning enables predicate pushdown WITHIN files too

6. USE DYNAMIC PARTITIONING FOR BULK HISTORICAL LOADS
   → Static partitioning for daily incremental loads
   → Dynamic partitioning for initial migration

🚫 WHAT SHOULD NOT DO

1. DO NOT PARTITION BY HIGH-CARDINALITY COLUMNS
   → Partitioning by ride_id (billions of unique values)
     creates BILLIONS of directories
   → Each directory = at least one file = one HDFS block = NameNode metadata
   → NameNode will crash (recall Project 1: ~150 bytes per file in memory)

   ┌──────────────────────────────────────────────────────────┐
   │ COLUMN             │ CARDINALITY  │ PARTITION? │ BUCKET? │
   ├────────────────────┼──────────────┼───────────┼─────────┤
   │ country            │ ~200         │ ✅ YES    │ ❌ NO   │
   │ city               │ ~10,000      │ ⚠️ MAYBE │ ✅ YES  │
   │ payment_method     │ 4            │ ✅ YES    │ ❌ NO   │
   │ ride_date          │ ~1,100       │ ✅ YES    │ ❌ NO   │
   │ rider_id           │ 50 million   │ ❌ NO     │ ✅ YES  │
   │ ride_id            │ 2.8 billion  │ ❌ NO     │ ❌ NO   │
   │ fare (continuous)  │ infinite     │ ❌ NO     │ ❌ NO   │
   └──────────────────────────────────────────────────────────┘

2. DO NOT ENABLE DYNAMIC PARTITIONING WITHOUT SAFETY LIMITS
   → Default: Hive BLOCKS dynamic partitioning (safety feature)
   → If data has unexpected values, it can create thousands of partitions
   → ALWAYS set hive.exec.max.dynamic.partitions

3. DO NOT ASSUME BUCKETING MEANS SORTING
   → Bucketed data is HASHED, not sorted
   → If you need sorted data within buckets, explicitly use SORTED BY
   → Bucketing alone guarantees same key → same bucket, NOT ordering

4. DO NOT MIX INSERT AND INSERT OVERWRITE CARELESSLY
   → INSERT INTO: appends to existing partition (duplicates possible)
   → INSERT OVERWRITE: replaces entire partition
   → Wrong choice = data duplication or data loss

5. DO NOT CREATE EMPTY PARTITIONS
   → Adding partition metadata without data wastes metastore space
   → ALTER TABLE ADD PARTITION creates metadata even with no files
   → Let dynamic partitioning create only partitions that have data

6. DO NOT FORGET TO SET hive.enforce.bucketing=true (Hive < 2.0)
   → Without this, Hive ignores CLUSTERED BY during INSERT
   → Data lands in wrong bucket files
   → Joins and sampling silently produce WRONG results
   → Hive 2.0+ enforces automatically

🚫 WHEN NOT TO USE

DO NOT USE PARTITIONING WHEN:

1. CARDINALITY IS TOO HIGH
   → Column has millions of unique values
   → Creates millions of directories → NameNode stress
   → Use BUCKETING instead for high-cardinality columns

2. CARDINALITY IS TOO LOW
   → Column has only 2 values (e.g., gender: M/F)
   → Each partition has half the data — barely any pruning benefit
   → Exception: if combined with other partition columns

3. QUERIES NEVER FILTER ON THAT COLUMN
   → Partitioning by "country" is useless if every query is
     "SELECT * FROM rides" with no WHERE on country
   → Partitioning without pruning = extra overhead for nothing

4. TABLE IS SMALL
   → Table with 1 million rows (< 1GB)
   → Partitioning creates many tiny files
   → Small files problem (Project 1 pitfall) makes things WORSE

DO NOT USE BUCKETING WHEN:

1. NO JOIN OR SAMPLING PATTERNS EXIST
   → Bucketing helps JOINs and TABLESAMPLE queries
   → If you never join or sample, bucketing adds write-time overhead
     with no read-time benefit

2. DATA VOLUME PER PARTITION IS TINY
   → 10MB partition bucketed into 32 buckets = 32 files of 300KB each
   → Small files problem again — worse performance, not better

3. BUCKET COLUMN DISTRIBUTION IS EXTREMELY SKEWED
   → 90% of rider_ids are in 5 cities
   → Hash distribution still skews if input is skewed
   → Some buckets get 100× more data than others

MECHANIC 1: HOW PARTITIONING WORKS INTERNALLY

PHYSICAL STRUCTURE:
→ Each partition = a SUBDIRECTORY in HDFS
→ Directory name encodes the partition column value

Table location: /user/hive/warehouse/rides_partitioned/

When you INSERT data with country='US', city='New_York':

HDFS creates:
/user/hive/warehouse/rides_partitioned/
    └── country=US/

              ├── 000000_0    (actual data file — NO country/city columns inside)

CRITICAL DETAIL:
→ The partition columns are NOT stored inside the data files
→ They are INFERRED from the directory path
→ This SAVES storage space
→ When Hive reads, it reconstructs the column from the path

═══════════════════════════════════════════════════════════
MECHANIC 2: PARTITION PRUNING — THE PERFORMANCE MAGIC
═══════════════════════════════════════════════════════════

Query: SELECT SUM(fare) FROM rides_partitioned
       WHERE country='US' AND city='New_York';

STEP 1 — Query hits Hive Metastore
   Hive asks: "Which partitions match country='US' AND city='New_York'?"
   Metastore returns: ONE partition path
   
STEP 2 — Hive generates MapReduce job
   InputPaths = ["/warehouse/rides_partitioned/country=US/city=New_York/"]
   NOT the entire table directory!

STEP 3 — MapReduce reads ONLY those files
   → Instead of 800GB across all directories
   → Reads only 3GB in one subdirectory
   → 99.6% of data NEVER TOUCHED

THIS IS PARTITION PRUNING:
→ The Metastore acts as an INDEX
→ Directories act as PHYSICAL SEGMENTS
→ Irrelevant segments are NEVER read from disk

VERIFICATION — You can see pruning in EXPLAIN:
   EXPLAIN SELECT SUM(fare) FROM rides_partitioned
           WHERE country='US' AND city='New_York';
   
   Output includes:
   "partition pruning: country=US/city=New_York"
   "number of partitions: 1"   ← instead of thousands

═══════════════════════════════════════════════════════════
MECHANIC 3: HOW BUCKETING WORKS INTERNALLY  
═══════════════════════════════════════════════════════════

CREATE TABLE rides_bucketed (
    ride_id BIGINT, rider_id BIGINT, fare DOUBLE, ...
) CLUSTERED BY (rider_id) INTO 8 BUCKETS;

WHEN DATA IS INSERTED:

For each row, Hive computes:
   bucket_number = hash(rider_id) MOD 8

   rider_id = 1001 → hash(1001) MOD 8 = 3 → goes to bucket file 000003_0
   rider_id = 2050 → hash(2050) MOD 8 = 6 → goes to bucket file 000006_0
   rider_id = 3777 → hash(3777) MOD 8 = 3 → goes to bucket file 000003_0

PHYSICAL RESULT:
/user/hive/warehouse/rides_bucketed/
    ├── 000000_0    (all rows where hash(rider_id) MOD 8 = 0)
    ├── 000001_0    (all rows where hash(rider_id) MOD 8 = 1)
    ├── 000002_0    (all rows where hash(rider_id) MOD 8 = 2)
    ├── 000003_0    (rider_id 1001 and 3777 are HERE)
    ├── 000004_0
    ├── 000005_0
    ├── 000006_0    (rider_id 2050 is HERE)
    └── 000007_0

GUARANTEE:
→ ALL rows with the same rider_id are in the SAME bucket file
→ This is deterministic — hash function always produces same result

═══════════════════════════════════════════════════════════
MECHANIC 4: WHY BUCKETING ENABLES MAP-SIDE JOINS
═══════════════════════════════════════════════════════════

(Builds on Map-Side Join concept from Project 16)

TWO TABLES, BOTH BUCKETED BY rider_id INTO 8 BUCKETS:

Table A: rides_bucketed (8 bucket files)
Table B: riders_bucketed (8 bucket files)

JOIN rides_bucketed a JOIN riders_bucketed b ON a.rider_id = b.rider_id

WITHOUT BUCKETING:
→ Full shuffle-sort across network (Reduce-Side Join from Project 17)
→ ALL data from both tables shipped across cluster
→ Massive network I/O

WITH BUCKETING (same column, same bucket count):
→ Bucket 0 of Table A only needs to join with Bucket 0 of Table B
→ Bucket 1 of Table A only needs to join with Bucket 1 of Table B
→ ...
→ Each mapper joins ONE pair of buckets LOCALLY

   ┌─────────────┐    ┌─────────────┐
   │ A: bucket_0  │←──→│ B: bucket_0  │  → Mapper 0 joins locally
   │ A: bucket_1  │←──→│ B: bucket_1  │  → Mapper 1 joins locally
   │ A: bucket_2  │←──→│ B: bucket_2  │  → Mapper 2 joins locally
   │     ...      │    │     ...      │
   │ A: bucket_7  │←──→│ B: bucket_7  │  → Mapper 7 joins locally
   └─────────────┘    └─────────────┘
   
   NO SHUFFLE PHASE → dramatically faster

REQUIREMENT:
→ Both tables MUST be bucketed by the JOIN column
→ Bucket count of one MUST be a multiple of the other
→ Set: hive.optimize.bucketmapjoin = true

THREE LEVELS OF PRUNING STACKED.

⚠️ COMMON PITFALLS AND FAILURE MODES

PITFALL 1: "Dynamic partitioning created 50,000 directories"
→ CAUSE: Partitioned by a column with too many unique values
→ SYMPTOM: Job runs for hours, NameNode memory spikes
→ EXAMPLE: PARTITION BY (ride_date) on 3 years = 1,095 partitions (OK)
           PARTITION BY (ride_date, city) = 1,095 × 500 = 547,500 (DISASTER)
→ FIX: Set hive.exec.max.dynamic.partitions = reasonable limit (e.g., 5000)
       Rethink partition strategy — maybe partition by year/month only

PITFALL 2: "My bucketed table query is still slow"
→ CAUSE: Data was loaded with plain INSERT without enforcing bucketing
→ WHAT HAPPENS: Files are created but data is NOT hash-distributed
   → File names look right (000000_0, 000001_0...)
   → But rows are randomly placed, NOT by hash(bucket_column)
→ VERIFICATION:
   SELECT rider_id, INPUT__FILE__NAME FROM rides_bucketed LIMIT 20;
   → If rider_ids in same file have different hash(id) MOD 8 → BROKEN
→ FIX: Hive 2.0+ enforces automatically
       Hive 1.x: SET hive.enforce.bucketing = true BEFORE INSERT

PITFALL 3: "Partition column appears in SELECT but values are NULL"
→ CAUSE: Querying a partitioned table but partition columns
         not properly registered in metastore
→ COMMON WITH: External tables where you ADD PARTITION manually
→ FIX: Verify with SHOW PARTITIONS tablename
       Run MSCK REPAIR TABLE tablename to discover partitions

PITFALL 4: "INSERT OVERWRITE wiped out wrong partitions"
→ CAUSE: Dynamic partitioning with INSERT OVERWRITE
→ BEHAVIOR: INSERT OVERWRITE with dynamic partitions only overwrites
            partitions that appear in the SELECT results
→ DANGER: If source data is filtered, unmentioned partitions survive
          BUT if you expected them to be cleared — confusion
→ FIX: Always verify with SHOW PARTITIONS before and after

PITFALL 5: "Small files explosion after daily partitioned loads"
→ CAUSE: Daily INSERT creates one small file per mapper per partition
→ EXAMPLE: 100 mappers × 365 days = 36,500 tiny files in one year
→ FIX: Use INSERT OVERWRITE (rewrites partition as fewer files)
       Or periodically run: ALTER TABLE ... CONCATENATE (ORC/RCFile)
       Or set: hive.merge.mapfiles = true
              hive.merge.mapredfiles = true
              hive.merge.smallfiles.avgsize = 128000000

PITFALL 6: "Bucket map join produces wrong results silently"
→ CAUSE: Tables claim to be bucketed but actual files don't match
→ Hive trusts metadata — doesn't verify at query time
→ If you loaded data bypassing Hive (e.g., direct HDFS put),
  bucket structure is violated
→ FIX: ALWAYS load bucketed tables through Hive INSERT, never direct HDFS copy

└─────────────────────┴────────────────────────────────────────┘

BUCKET SCALABILITY:

┌─────────────────────┬────────────────────────────────────────┐
│ METRIC              │ GUIDANCE                               │
├─────────────────────┼────────────────────────────────────────┤
│ Bucket count        │ Target each bucket file = 128MB-1GB    │
│                     │ Too few buckets → files too large      │
│                     │ Too many buckets → small files problem  │
├─────────────────────┼────────────────────────────────────────┤
│ Bucket count for    │ Must be identical or multiple between  │
│ joins               │ the two tables being joined            │
│                     │ Plan bucket count at DESIGN TIME       │
├─────────────────────┼────────────────────────────────────────┤
│ Changing bucket     │ REQUIRES full table rewrite            │
│ count               │ Cannot add/remove buckets incrementally│
├─────────────────────┼────────────────────────────────────────┤
│ Sampling efficiency │ 1 bucket out of N → reads ~1/N of data│
│                     │ N=100 → 1% sample by reading 1 file   │
└─────────────────────┴────────────────────────────────────────┘

COMBINED SCALABILITY FORMULA:

Total files = (number_of_partitions) × (number_of_buckets)

EXAMPLE:
→ Partitioned by (year, month, country): 3 years × 12 months × 50 countries = 1,800 partitions
→ Bucketed into 16 buckets
→ Total files: 1,800 × 16 = 28,800 files
→ At 128MB each: 28,800 × 128MB = 3.6TB → reasonable for HDFS

DANGER EXAMPLE:
→ Partitioned by (date, city): 1,095 days × 500 cities = 547,500 partitions
→ Bucketed into 32 buckets
→ Total files: 547,500 × 32 = 17,520,000 files
→ NameNode memory: 17.5M × 150 bytes = 2.6GB JUST for this one table
→ THIS WILL DEGRADE THE ENTIRE CLUSTER

# PYTHON UTILITY: Partition Health Monitor
# ═══════════════════════════════════════════════════════════════════
# This script connects to Hive and checks partition health
# Identifies: empty partitions, skewed partitions, small file issues

import subprocess
import re
import sys

class PartitionHealthChecker:
    """
    Checks the health of a partitioned Hive table.
    Identifies common problems that degrade performance.
    """
    
    def __init__(self, database, table):
        self.database = database
        self.table = table
        self.partitions = []
        self.partition_stats = []
    
    def run_hive_query(self, query):
        """Execute a Hive query and return output lines."""
        cmd = ['hive', '-e', f'USE {self.database}; {query}']
        try:
            result = subprocess.run(
                cmd,
                capture_output=True,
                text=True,
                timeout=120
            )
            if result.returncode != 0:
                print(f"  [ERROR] Hive query failed: {result.stderr[:200]}")
                return []
            return [line.strip() for line in result.stdout.strip().split('\n')
                    if line.strip() and not line.startswith('OK')
                    and not line.startswith('Time taken')]
        except subprocess.TimeoutExpired:
            print("  [ERROR] Query timed out after 120 seconds")
            return []
        except FileNotFoundError:
            print("  [ERROR] Hive CLI not found — ensure Hive is in PATH")
            return []
    
    def run_hdfs_command(self, path):
        """List HDFS directory contents."""
        cmd = ['hdfs', 'dfs', '-ls', path]
        try:
            result = subprocess.run(cmd, capture_output=True, text=True, timeout=30)
            if result.returncode != 0:
                return []
            lines = result.stdout.strip().split('\n')
            return [l for l in lines if l.strip() and not l.startswith('Found')]
        except (subprocess.TimeoutExpired, FileNotFoundError):
            return []
    
    def get_partitions(self):
        """Retrieve all partitions for the table."""
        print(f"\n[1] Discovering partitions for {self.database}.{self.table}...")
        
        lines = self.run_hive_query(f'SHOW PARTITIONS {self.table}')
        self.partitions = [line for line in lines if '=' in line]
        
        print(f"    Found {len(self.partitions)} partitions")
        return self.partitions
    
    def check_partition_sizes(self):
        """Check file count and size for each partition."""
        print(f"\n[2] Analyzing partition file structure...")
        
        # Get table location
        desc_lines = self.run_hive_query(
            f"DESCRIBE FORMATTED {self.table}"
        )
        
        table_location = None
        for line in desc_lines:
            if 'Location:' in line or 'location:' in line:
                parts = line.split()
                for part in parts:
                    if part.startswith('hdfs://') or part.startswith('/'):
                        table_location = part
                        break
        
        if not table_location:
            print("    [WARNING] Could not determine table location")
            # Fallback to default warehouse path
            table_location = (
                f"/user/hive/warehouse/{self.database}.db/{self.table}"
            )
        
        print(f"    Table location: {table_location}")
        
        self.partition_stats = []
        
        for partition in self.partitions[:20]:  # Limit to first 20 for demo
            partition_path = f"{table_location}/{partition.replace(',', '/')}"
            
            files = self.run_hdfs_command(partition_path)
            
            file_count = len(files)
            total_size = 0
            
            for f in files:
                # Parse HDFS -ls output: permissions repl user group size date time path
                parts = f.split()
                if len(parts) >= 5:
                    try:
                        total_size += int(parts[4])
                    except ValueError:
                        pass
            
            stat = {
                'partition': partition,
                'file_count': file_count,
                'total_bytes': total_size,
                'total_mb': round(total_size / (1024 * 1024), 2),
                'avg_file_mb': round(
                    total_size / (1024 * 1024 * max(file_count, 1)), 2
                )
            }
            self.partition_stats.append(stat)
        
        return self.partition_stats
    
    def diagnose(self):
        """Run all checks and produce a health report."""
        print("=" * 70)
        print(f"PARTITION HEALTH REPORT: {self.database}.{self.table}")
        print("=" * 70)
        
        # Step 1: Get partitions
        partitions = self.get_partitions()
        if not partitions:
            print("\n  [CRITICAL] No partitions found!")
            print("  → Table may not be partitioned")
            print("  → Or no data has been inserted")
            return
        
        # Step 2: Check partition count
        print(f"\n[3] Partition Count Analysis:")
        if len(partitions) > 100000:
            print(f"    [CRITICAL] {len(partitions)} partitions — TOO MANY!")
            print("    → NameNode metadata pressure is high")
            print("    → Consider reducing partition granularity")
        elif len(partitions) > 10000:
            print(f"    [WARNING] {len(partitions)} partitions — getting high")
            print("    → Monitor NameNode heap usage")
        else:
            print(f"    [OK] {len(partitions)} partitions — healthy range")
        
        # Step 3: Check file structure
        stats = self.check_partition_sizes()
        
        if not stats:
            print("\n  [WARNING] Could not retrieve file statistics")
            return
        
        # Step 4: Identify problems
        print(f"\n[4] Partition-Level Diagnostics:")
        print(f"    {'Partition':<45} {'Files':<8} {'Size(MB)':<12} {'Avg File(MB)':<14} {'Status'}")
        print(f"    {'-'*45} {'-'*8} {'-'*12} {'-'*14} {'-'*20}")
        
        empty_partitions = []
        small_file_partitions = []
        skewed_partitions = []
        healthy_partitions = []
        
        sizes = [s['total_mb'] for s in stats if s['total_mb'] > 0]
        avg_size = sum(sizes) / max(len(sizes), 1)
        
        for stat in stats:
            status = "✅ OK"
            
            if stat['total_bytes'] == 0:
                status = "🔴 EMPTY"
                empty_partitions.append(stat)
            elif stat['avg_file_mb'] < 10 and stat['file_count'] > 4:
                status = "🟡 SMALL FILES"
                small_file_partitions.append(stat)
            elif avg_size > 0 and stat['total_mb'] > avg_size * 5:
                status = "🟠 SKEWED (large)"
                skewed_partitions.append(stat)
            elif avg_size > 0 and stat['total_mb'] < avg_size * 0.1 and stat['total_mb'] > 0:
                status = "🟠 SKEWED (tiny)"
                skewed_partitions.append(stat)
            else:
                healthy_partitions.append(stat)
            
            partition_display = stat['partition'][:44]
            print(f"    {partition_display:<45} {stat['file_count']:<8} "
                  f"{stat['total_mb']:<12} {stat['avg_file_mb']:<14} {status}")
        
        # Step 5: Summary
        print(f"\n[5] SUMMARY:")
        print(f"    Total partitions analyzed: {len(stats)}")
        print(f"    Healthy:                   {len(healthy_partitions)}")
        print(f"    Empty:                     {len(empty_partitions)}")
        print(f"    Small file issue:          {len(small_file_partitions)}")
        print(f"    Skewed:                    {len(skewed_partitions)}")
        
        # Step 6: Recommendations
        print(f"\n[6] RECOMMENDATIONS:")
        
        if empty_partitions:
            print(f"\n    🔴 EMPTY PARTITIONS ({len(empty_partitions)}):")
            print(f"    → Drop empty partitions to clean metastore:")
            for ep in empty_partitions[:3]:
                parts = ep['partition'].split('/')
                part_spec = ', '.join(
                    [f"{p.split('=')[0]}='{p.split('=')[1]}'" for p in parts]
                )
                print(f"       ALTER TABLE {self.table} DROP PARTITION ({part_spec});")
        
        if small_file_partitions:
            print(f"\n    🟡 SMALL FILE PARTITIONS ({len(small_file_partitions)}):")
            print(f"    → Run CONCATENATE to merge small ORC files:")
            for sf in small_file_partitions[:3]:
                parts = sf['partition'].split('/')
                part_spec = ', '.join(
                    [f"{p.split('=')[0]}='{p.split('=')[1]}'" for p in parts]
                )
                print(f"       ALTER TABLE {self.table} "
                      f"PARTITION ({part_spec}) CONCATENATE;")
            print(f"    → Or enable auto-merge:")
            print(f"       SET hive.merge.mapredfiles = true;")
            print(f"       SET hive.merge.smallfiles.avgsize = 134217728;")
        
        if skewed_partitions:
            print(f"\n    🟠 SKEWED PARTITIONS ({len(skewed_partitions)}):")
            print(f"    → Data distribution is uneven across partitions")
            print(f"    → Consider sub-partitioning large partitions")
            print(f"    → Or use bucketing within partitions for better distribution")
        
        if not (empty_partitions or small_file_partitions or skewed_partitions):
            print(f"\n    ✅ ALL PARTITIONS HEALTHY — No action needed")
        
        print(f"\n{'=' * 70}")
        print(f"REPORT COMPLETE")
        print(f"{'=' * 70}")
