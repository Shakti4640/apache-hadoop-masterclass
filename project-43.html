PROJECT 43: HDFS FEDERATION — SCALING THE NAMENODE BEYOND ONE NAMESPACE

🔴 HYPOTHETICAL PROBLEM

SCENARIO:
→ You are the lead data platform architect at a global media streaming company
→ Your Hadoop cluster has grown to 5,000 DataNodes over 4 years
→ The SINGLE NameNode manages:
    - 800 million files
    - 1.2 billion blocks
    - Metadata consumes 180GB of JVM heap

WHAT IS HAPPENING:
→ NameNode garbage collection pauses last 45-90 seconds
→ During GC pauses, the ENTIRE cluster is FROZEN
    - No file creates
    - No file reads
    - No MapReduce tasks can start
    - All 5,000 DataNodes wait helplessly
→ Adding more files makes the problem WORSE every week
→ NameNode startup takes 35 minutes (loading FSImage + replaying EditLog)
→ After a restart, the cluster is unusable for another 20 minutes
   while DataNodes report all their blocks (safe mode)

BUSINESS IMPACT:
→ 4 separate business units share this cluster:
    - Video Analytics team:     300 million files
    - Recommendation Engine:    250 million files  
    - Billing/Finance:          150 million files
    - Content Delivery Logs:    100 million files
→ A GC pause in NameNode freezes ALL teams simultaneously
→ Finance team's critical month-end job fails because
   Video Analytics flooded the namespace with tiny thumbnails
→ One team's namespace pollution affects EVERYONE

THE ROOT CAUSE:
→ Everything we built in Project 2 relies on ONE NameNode
→ That single NameNode is a SCALING BOTTLENECK
→ Project 44 will solve the AVAILABILITY problem (what if it crashes)
→ THIS project solves the SCALABILITY problem (what if it's too slow)

QUESTION:
→ Can we have MULTIPLE NameNodes, each managing a PORTION
   of the total namespace, all sharing the SAME DataNodes?
→ YES — That is HDFS Federation.

🔍 HOW TO APPROACH IT

STEP 1: Understand why the NameNode is the bottleneck

Recall from Project 2:
→ NameNode stores ALL metadata IN MEMORY
→ Each file object ≈ 150 bytes of heap
→ Each block object ≈ 150 bytes of heap
→ A file with 3 blocks = 150 + (3 × 150) = 600 bytes

Current state:
→ 800 million files × 150 bytes          = 120 GB
→ 1.2 billion blocks × 150 bytes         =  60 GB (after dedup)
→ Total NameNode heap requirement         ≈ 180 GB

JVM HEAP PROBLEM:
→ Java GC on 180GB heap = catastrophic stop-the-world pauses
→ Even G1GC or ZGC struggle at this scale
→ You CANNOT just add more RAM indefinitely
→ JVM heap beyond 200GB becomes operationally dangerous

KEY INSIGHT:
→ The NameNode does TWO distinct jobs:
   Job 1: NAMESPACE MANAGEMENT (directory tree, file→block mapping)
   Job 2: BLOCK MANAGEMENT (block→DataNode mapping, replication)

→ These two jobs CAN be separated
→ Multiple NameNodes can each manage their OWN namespace
→ DataNodes can serve blocks for ALL NameNodes simultaneously

RESULT:
→ NameNode 1 heap: ~80GB (manages /video + /content)
→ NameNode 2 heap: ~50GB (manages /recommendations)
→ NameNode 3 heap: ~50GB (manages /billing + /finance)
→ NO single node needs 180GB heap
→ GC pauses affect only ONE team's namespace
→ Startup time per NameNode: ~10 minutes instead of 35

💡 HOW CAN WE SOLVE IT

FEDERATION INTRODUCES THREE NEW CONCEPTS:

CONCEPT 1: MULTIPLE INDEPENDENT NAMESPACES

Before Federation:
  /  (root) ← managed by ONE NameNode
  ├── /video/
  ├── /recommendations/
  ├── /billing/
  └── /content-delivery/

After Federation:
  NameNode-1 manages:  /video/  AND  /content-delivery/
  NameNode-2 manages:  /recommendations/
  NameNode-3 manages:  /billing/

→ Each NameNode is COMPLETELY INDEPENDENT
→ They do NOT communicate with each other
→ They do NOT share namespace state
→ Failure of NameNode-2 does NOT affect NameNode-1 or NameNode-3

═══════════════════════════════════════════════════════════

CONCEPT 2: BLOCK POOLS

→ A Block Pool = the set of ALL blocks belonging to ONE NameNode's namespace
→ Each NameNode has exactly ONE Block Pool
→ Block Pool has a globally unique ID (like: BP-12345-10.0.0.1-1234567890)
→ DataNodes store blocks from MULTIPLE Block Pools on the SAME disks

Physical disk layout on a DataNode:
  /data/hdfs/datanode/
  ├── BP-111-10.0.0.1-1700000001/     ← Block Pool for NameNode 1
  │   ├── current/
  │   │   ├── blk_0001
  │   │   ├── blk_0001.meta
  │   │   ├── blk_0045
  │   │   └── blk_0045.meta
  │   └── tmp/
  ├── BP-222-10.0.0.2-1700000002/     ← Block Pool for NameNode 2
  │   ├── current/
  │   │   ├── blk_1001
  │   │   └── blk_1001.meta
  │   └── tmp/
  └── BP-333-10.0.0.3-1700000003/     ← Block Pool for NameNode 3
      ├── current/
      │   ├── blk_2001
      │   └── blk_2001.meta
      └── tmp/

KEY RULE:
→ Block IDs are unique WITHIN a Block Pool (not globally)
→ blk_0001 in BP-111 is DIFFERENT from blk_0001 in BP-222
→ The combination (Block Pool ID + Block ID) is globally unique

═══════════════════════════════════════════════════════════

CONCEPT 3: VIEWFS — CLIENT-SIDE MOUNT TABLE

PROBLEM:
→ Without ViewFS, clients must know WHICH NameNode to contact
→ Client 1: hdfs://namenode1:9000/video/file.txt
→ Client 2: hdfs://namenode2:9000/recommendations/model.bin
→ This breaks existing applications that expect hdfs://cluster/...

SOLUTION — ViewFS:
→ A client-side VIRTUAL filesystem
→ Maps paths to the correct NameNode transparently

  Client sees:         Actual location:
  /video/            → hdfs://namenode1:9000/video/
  /content-delivery/ → hdfs://namenode1:9000/content-delivery/
  /recommendations/  → hdfs://namenode2:9000/recommendations/
  /billing/          → hdfs://namenode3:9000/billing/

→ Applications use: viewfs://cluster-federation/video/file.txt
→ ViewFS resolves it to: hdfs://namenode1:9000/video/file.txt
→ Existing code needs ZERO changes (if using fs.defaultFS)

✅ WHAT COULD BE DONE

FEDERATION DEPLOYMENT STRATEGIES:

STRATEGY 1: BY BUSINESS UNIT (Recommended for our scenario)
  NameNode-1: /video, /content-delivery    (Media team)
  NameNode-2: /recommendations             (ML team)
  NameNode-3: /billing, /finance           (Finance team)
  
  PROS: Team isolation, independent maintenance windows
  CONS: Uneven namespace sizes may cause imbalance

STRATEGY 2: BY DATA LIFECYCLE
  NameNode-1: /raw         (ingestion landing zone)
  NameNode-2: /processed   (ETL outputs)
  NameNode-3: /warehouse   (Hive tables, analytics)
  NameNode-4: /archive     (cold data, rarely accessed)
  
  PROS: Matches data flow pipeline
  CONS: /raw may have millions of tiny files while /archive is calm

STRATEGY 3: BY ACCESS PATTERN
  NameNode-1: /hot-data    (frequently accessed, high throughput)
  NameNode-2: /warm-data   (moderate access)
  NameNode-3: /cold-data   (rarely accessed, large archive)
  
  PROS: Tune each NameNode's heap and GC for its workload
  CONS: Requires data migration as data ages

STRATEGY 4: HYBRID (Production reality)
  NameNode-1: /team-video, /team-video-staging
  NameNode-2: /team-ml, /team-ml-experiments
  NameNode-3: /shared/warehouse, /shared/reference-data
  NameNode-4: /archive
  
  PROS: Balances all concerns
  CONS: More complex ViewFS mount table

✅ WHAT SHOULD DO

1. PLAN NAMESPACE PARTITIONING BEFORE DEPLOYMENT
   → Analyze current namespace distribution:
     hdfs dfs -count /video /recommendations /billing /content-delivery
   → Output shows: DIR_COUNT FILE_COUNT SIZE PATH
   → Use this to balance NameNodes by file count, NOT by data size
   → File COUNT drives heap usage — data SIZE drives DataNode storage

2. SIZE EACH NAMENODE'S HEAP INDEPENDENTLY
   → NameNode managing 300M files: needs ~90GB heap
   → NameNode managing 100M files: needs ~30GB heap
   → Formula: heap_gb ≈ (num_files + num_blocks) × 300 bytes / 1GB
   → Add 30% safety margin

3. DEPLOY EACH NAMENODE ON A DEDICATED PHYSICAL MACHINE
   → Each NameNode needs its own:
     - FSImage storage (local SSD recommended)
     - EditLog storage (separate disk or JournalNodes — Project 44)
     - Sufficient RAM for its namespace portion
   → NEVER co-locate two NameNodes on the same machine

4. IMPLEMENT VIEWFS FROM DAY ONE
   → Even if starting with 2 NameNodes
   → Makes future namespace splits transparent to applications
   → All client configurations point to viewfs:// not hdfs://

5. MONITOR EACH NAMENODE INDEPENDENTLY
   → Separate JMX monitoring per NameNode
   → Separate alerting thresholds per NameNode
   → Track: heap usage, GC time, RPC queue length, block reports per NN

6. PLAN FOR ENCRYPTION ZONES PER NAMESPACE
   → Recall Project 42: encryption zones are per-directory
   → With Federation, each NameNode can have its own KMS policies
   → Finance NameNode may have stricter encryption than Video NameNode

🚫 WHAT SHOULD NOT DO

1. DO NOT CONFUSE FEDERATION WITH HIGH AVAILABILITY
   → Federation = MULTIPLE NameNodes for SCALABILITY
     (each manages a DIFFERENT portion of the namespace)
   → HA = TWO NameNodes for RELIABILITY
     (Active/Standby managing the SAME namespace)
   → In production you need BOTH:
     Each federated NameNode should ALSO have an HA standby
     (We configure this in Project 44)

   ┌─────────────────────────────────────────────────────┐
   │           FEDERATION + HA COMBINED                   │
   │                                                      │
   │  Namespace 1:  NN1-Active  ←→  NN1-Standby         │
   │  Namespace 2:  NN2-Active  ←→  NN2-Standby         │
   │  Namespace 3:  NN3-Active  ←→  NN3-Standby         │
   │                                                      │
   │  6 NameNode processes total                          │
   │  3 active + 3 standby                               │
   │  ALL sharing the same pool of DataNodes              │
   └─────────────────────────────────────────────────────┘

2. DO NOT SPLIT NAMESPACES TOO GRANULARLY
   → 20 NameNodes for 20 subdirectories = operational nightmare
   → Each NameNode needs its own monitoring, backup, HA standby
   → RULE OF THUMB: 2-5 federated NameNodes covers most use cases
   → Only go beyond 5 if individual namespaces still exceed 500M files

3. DO NOT ASSUME CROSS-NAMESPACE OPERATIONS WORK
   → hdfs dfs -mv /video/file.txt /billing/file.txt
     ↑ This FAILS if /video and /billing are on different NameNodes
   → There is NO atomic cross-namespace rename or move
   → You must: copy to destination NameNode, then delete from source
   → distcp is your tool for cross-namespace data transfer

4. DO NOT IGNORE BLOCK POOL STORAGE BALANCE
   → DataNode disk may fill up with blocks from one Block Pool
   → The HDFS balancer only balances WITHIN a single namespace
   → You need to monitor per-Block-Pool disk usage on each DataNode

5. DO NOT CHANGE NAMESPACE ASSIGNMENTS AFTER DEPLOYMENT WITHOUT PLANNING
   → Moving /recommendations from NameNode-2 to NameNode-1 requires:
     - Full data copy via distcp
     - Metadata is NOT transferable between NameNodes
     - Application downtime during cutover
     - ViewFS mount table update on ALL clients

🚫 WHEN NOT TO USE

HARD RULES — FEDERATION IS WRONG WHEN:

1. YOUR PROBLEM IS AVAILABILITY, NOT SCALABILITY
   → NameNode crashes = cluster down
   → Federation gives you MORE NameNodes but each is still a SPOF
   → You need HA (Project 44), not Federation
   → Federation + HA together = correct production setup

2. YOUR NAMESPACE IS SMALL BUT DATA VOLUME IS HUGE
   → 10 million files but 500TB of data
   → NameNode heap: only ~5GB — perfectly manageable
   → The bottleneck is DataNode storage, not NameNode memory
   → Add more DataNodes, not more NameNodes

3. YOUR APPLICATIONS HEAVILY CROSS-REFERENCE NAMESPACES
   → Application reads /video/metadata.json, joins with /billing/invoice.csv
   → With Federation these may be on different NameNodes
   → Cross-namespace operations are expensive (distcp, no atomic moves)
   → Either put related data under the same NameNode's namespace
     or redesign the application

4. YOU ARE MIGRATING TO CLOUD OBJECT STORAGE
   → If moving to S3/ADLS (Project 49), NameNode scaling is moot
   → Cloud storage has its own metadata service
   → Investing in Federation for a cluster with 1-year remaining lifetime
     is wasted effort

5. SMALL CLUSTER (< 500 NODES)
   → Federation overhead (multiple NN machines, ViewFS config management,
     operational complexity) rarely justified for small clusters
   → Exception: if file count is extremely high even on a small cluster

⚙️ HOW IT ONLY WORKS — INTERNAL MECHANICS

═══════════════════════════════════════════════════════════
INTERNAL MECHANIC 1: NAMESERVICE ID AND BLOCK POOL MAPPING
═══════════════════════════════════════════════════════════

Every federated NameNode is identified by a NAMESERVICE ID:
  ns1 → NameNode managing /video, /content-delivery
  ns2 → NameNode managing /recommendations
  ns3 → NameNode managing /billing, /finance

During NameNode FORMAT, a Block Pool ID is generated:
  ns1 → BP-1001-10.0.1.1-1700000001
  ns2 → BP-2002-10.0.1.2-1700000002
  ns3 → BP-3003-10.0.1.3-1700000003

This mapping is PERMANENT — it cannot be changed after format.

═══════════════════════════════════════════════════════════
INTERNAL MECHANIC 2: DATANODE REGISTRATION WITH MULTIPLE NN
═══════════════════════════════════════════════════════════

WITHOUT Federation (Project 2):
  DataNode → registers with → 1 NameNode
  DataNode → sends heartbeat to → 1 NameNode
  DataNode → sends block report to → 1 NameNode

WITH Federation:
  DataNode → registers with → ALL NameNodes
  DataNode → sends heartbeat to → ALL NameNodes (independently)
  DataNode → sends block report to → ALL NameNodes (per Block Pool)

  ┌────────────┐
  │ DataNode 1 │
  │            │──heartbeat──→ NameNode 1 (ns1)
  │            │──heartbeat──→ NameNode 2 (ns2)
  │            │──heartbeat──→ NameNode 3 (ns3)
  │            │
  │ Stores:    │──block report (BP-1001 blocks)──→ NameNode 1
  │ BP-1001:   │──block report (BP-2002 blocks)──→ NameNode 2
  │  blk_0001  │──block report (BP-3003 blocks)──→ NameNode 3
  │  blk_0045  │
  │ BP-2002:   │
  │  blk_1001  │
  │ BP-3003:   │
  │  blk_2001  │
  └────────────┘

IMPORTANT:
→ Each NameNode ONLY knows about blocks in its OWN Block Pool
→ NameNode 1 has NO knowledge that blk_1001 (BP-2002) exists on DataNode 1
→ NameNode 2 has NO knowledge that blk_0001 (BP-1001) exists on DataNode 1
→ They are COMPLETELY ISOLATED at the metadata level

═══════════════════════════════════════════════════════════
INTERNAL MECHANIC 3: VIEWFS RESOLUTION PATH
═══════════════════════════════════════════════════════════

When a client runs:
  hadoop fs -cat viewfs://my-cluster/video/trending/2024-01-15.csv

Step 1 — Client reads core-site.xml
  fs.defaultFS = viewfs://my-cluster

Step 2 — Client reads ViewFS mount table from core-site.xml
  /video           → hdfs://ns1/video
  /content-delivery → hdfs://ns1/content-delivery
  /recommendations → hdfs://ns2/recommendations
  /billing         → hdfs://ns3/billing

Step 3 — Path resolution
  /video/trending/2024-01-15.csv
  ↓ matches mount point /video
  ↓ resolves to hdfs://ns1/video/trending/2024-01-15.csv

Step 4 — Nameservice resolution (from hdfs-site.xml)
  ns1 → namenode1-host:9000

Step 5 — Client contacts NameNode 1 at namenode1-host:9000
  "Give me block locations for /video/trending/2024-01-15.csv"

Step 6 — NameNode 1 responds with block locations
  Block blk_0088 → [DataNode3:50010, DataNode7:50010, DataNode12:50010]

Step 7 — Client reads directly from DataNode3
  (data locality — same as non-federated HDFS from Project 2)

ENTIRE RESOLUTION: < 5ms overhead compared to non-federated setup

═══════════════════════════════════════════════════════════
INTERNAL MECHANIC 4: CLUSTER ID — THE GLUE
═══════════════════════════════════════════════════════════

All NameNodes in a federation share the SAME Cluster ID.

  Cluster ID: CID-abc123-my-cluster

  NameNode 1 (ns1): CID-abc123-my-cluster, BP-1001-...
  NameNode 2 (ns2): CID-abc123-my-cluster, BP-2002-...
  NameNode 3 (ns3): CID-abc123-my-cluster, BP-3003-...

  DataNode registration check:
    "Is this NameNode's Cluster ID the same as mine?"
    → YES: Register and serve blocks
    → NO:  Refuse connection (prevents cross-cluster contamination)

  The Cluster ID is set during the FIRST NameNode format.
  Subsequent NameNodes must use -clusterid to join the same federation.

    ├── BP-2002-10.0.1.2-1700000002/    ← Block Pool for ns2

            │           └── blk_2001.meta
            └── rbw/

DISK SPACE SHARING:
→ All Block Pools share the SAME physical disks
→ No pre-allocated disk partitions per Block Pool
→ First-come-first-served disk usage
→ Risk: One namespace's data growth can fill the disk,
  affecting ALL namespaces on that DataNode
→ Mitigation: Configure dfs.datanode.du.reserved per volume

═══════════════════════════════════════════════════════════
INTERNAL MECHANIC 6: BALANCER IN FEDERATED MODE
═══════════════════════════════════════════════════════════

The HDFS Balancer (hdfs balancer) works PER NAMESERVICE:

  hdfs balancer -Ddfs.nameservices=ns1
  → Balances blocks ONLY within Block Pool BP-1001
  → Moves blocks between DataNodes to equalize disk usage
  → Has NO visibility into BP-2002 or BP-3003

  To balance all namespaces:
  → Run separate balancer instances for each nameservice
  → Or use a script that iterates over all nameservices

CROSS-BLOCK-POOL BALANCE:
→ There is NO built-in cross-Block-Pool balancer
→ If BP-1001 uses 80% of disk and BP-2002 uses 5%,
   Hadoop cannot automatically rebalance between them
→ This is a namespace planning problem, not a runtime problem

⚠️ COMMON PITFALLS AND FAILURE MODES

PITFALL 1: FORMATTING ADDITIONAL NAMENODES INCORRECTLY
→ SYMPTOM: Second NameNode refuses to start or DataNodes won't register
→ CAUSE: Did not specify -clusterid during format
→ FIX:
   First NameNode:   hdfs namenode -format -clusterid my-cluster-id
   Second NameNode:  hdfs namenode -format -clusterid my-cluster-id
   → Both MUST have the SAME Cluster ID
   → Block Pool IDs will be different (correct behavior)

PITFALL 2: VIEWFS MOUNT TABLE NOT DEPLOYED TO ALL CLIENTS
→ SYMPTOM: MapReduce jobs fail with "No mount point for path /video"
→ CAUSE: Worker nodes have old core-site.xml without ViewFS entries
→ FIX: Distribute updated core-site.xml to ALL nodes
→ Use configuration management (Ansible/Puppet/Ambari)

SCALING LIMITS OF FEDERATION:

1. DATANODE LIMIT
   → Each DataNode must heartbeat to ALL NameNodes
   → With 10 NameNodes: DataNode sends 10 heartbeats per interval
   → Practical limit: 5-8 federated NameNodes before DataNode overhead matters

2. VIEWFS COMPLEXITY
   → Mount table grows with number of mount points
   → Client must resolve mount point on every file operation
   → Deep mount tables (100+ entries) add measurable latency
   → Keep mount table shallow: mount at /team-level, not /project-level

3. OPERATIONAL OVERHEAD
   → N NameNodes = N FSImages to backup
   → N NameNodes = N EditLog streams
   → N NameNodes = N HA standby nodes (with HA enabled)
   → N NameNodes = N monitoring dashboards
   → Total machines for NameNode layer: 2N (with HA) + JournalNodes

   Example: 4-way federation with HA
   → 4 Active NameNodes (4 machines)
   → 4 Standby NameNodes (4 machines)
   → 3 JournalNodes (shared across all, 3 machines)
   → Total: 11 machines JUST for the metadata layer

4. NAMESPACE REBALANCING
   → If namespace A grows from 100M to 400M files while B stays at 100M
   → You cannot dynamically "move" directories between NameNodes
   → Must plan for growth or accept periodic painful migrations

# Analyzes namespace distribution and recommends rebalancing

"""
PURPOSE:
Before deploying Federation or when planning namespace splits,
this tool analyzes the current HDFS namespace to recommend
how to partition directories across multiple NameNodes.

USAGE:
  python3 federation_namespace_analyzer.py

PREREQUISITES:
  - Hadoop CLI available in PATH
  - Access to HDFS cluster (single NameNode or existing federation)
"""

import subprocess
import json
import sys
from collections import defaultdict

class NamespaceAnalyzer:
    """
    Analyzes HDFS namespace distribution to plan Federation deployment.
    
    The key insight: Federation partitions by FILE COUNT (which drives
    NameNode heap), NOT by data SIZE (which drives DataNode storage).
    """
    
    def __init__(self, hdfs_uri=None):
        self.hdfs_uri = hdfs_uri  # None = use default fs
        self.directory_stats = {}
        
    def _run_hdfs_command(self, args):
        """Execute an HDFS command and return stdout."""
        cmd = ["hdfs", "dfs"]
        if self.hdfs_uri:
            cmd.extend(["-fs", self.hdfs_uri])
        cmd.extend(args)
        
        try:
            result = subprocess.run(
                cmd,
                capture_output=True,
                text=True,
                timeout=300
            )
            return result.stdout.strip()
        except subprocess.TimeoutExpired:
            print(f"  ⚠️  Command timed out: {' '.join(cmd)}")
            return ""
        except FileNotFoundError:
            print("  ❌ 'hdfs' command not found. Ensure Hadoop is in PATH.")
            sys.exit(1)
    
    def analyze_top_level_directories(self):
        """
        Get file count, directory count, and size for each top-level directory.
        This is the PRIMARY input for Federation planning.
        """
        print("═" * 70)
        print("  HDFS NAMESPACE ANALYSIS FOR FEDERATION PLANNING")
        print("═" * 70)
        print()
        
        # List top-level directories
        ls_output = self._run_hdfs_command(["-ls", "/"])
        if not ls_output:
            print("  ❌ Cannot list root directory. Check HDFS access.")
            return
        
        top_dirs = []
        for line in ls_output.split('\n'):
            parts = line.split()
            if len(parts) >= 8 and parts[0].startswith('d'):
                dir_path = parts[-1]
                top_dirs.append(dir_path)
        
        print(f"  Found {len(top_dirs)} top-level directories")
        print()
        
        # Get detailed counts for each directory
        # hdfs dfs -count outputs: DIR_COUNT FILE_COUNT CONTENT_SIZE PATH
        total_files = 0
        total_dirs = 0
        total_size = 0
        
        print(f"  {'Directory':<30} {'Dirs':>12} {'Files':>15} {'Size (GB)':>12} {'% Files':>10}")
        print(f"  {'─' * 30} {'─' * 12} {'─' * 15} {'─' * 12} {'─' * 10}")
        
        dir_file_counts = {}
        
        for dir_path in top_dirs:
            count_output = self._run_hdfs_command(["-count", dir_path])
            if count_output:
                parts = count_output.split()
                if len(parts) >= 4:
                    dir_count = int(parts[0])
                    file_count = int(parts[1])
                    size_bytes = int(parts[2])
                    
                    self.directory_stats[dir_path] = {
                        'dirs': dir_count,
                        'files': file_count,
                        'size_bytes': size_bytes,
                        'size_gb': size_bytes / (1024 ** 3)
                    }
                    
                    dir_file_counts[dir_path] = file_count
                    total_files += file_count
                    total_dirs += dir_count
                    total_size += size_bytes
        
        # Print with percentages
        for dir_path in sorted(self.directory_stats.keys()):
            stats = self.directory_stats[dir_path]
            file_pct = (stats['files'] / total_files * 100) if total_files > 0 else 0
            
            print(f"  {dir_path:<30} {stats['dirs']:>12,} {stats['files']:>15,} "
                  f"{stats['size_gb']:>11.1f} {file_pct:>9.1f}%")
        
        print(f"  {'─' * 30} {'─' * 12} {'─' * 15} {'─' * 12} {'─' * 10}")
        print(f"  {'TOTAL':<30} {total_dirs:>12,} {total_files:>15,} "
              f"{total_size / (1024**3):>11.1f} {'100.0':>9}%")
        
        return dir_file_counts, total_files
    
    def estimate_namenode_heap(self, file_count, block_multiplier=1.5):
        """
        Estimate NameNode heap requirement.
        
        Formula:
          heap_bytes = (num_files + num_blocks) × ~300 bytes
          num_blocks ≈ num_files × block_multiplier
          (block_multiplier accounts for multi-block files)
        
        Returns heap in GB with 30% safety margin.
        """
        estimated_blocks = int(file_count * block_multiplier)
        total_objects = file_count + estimated_blocks
        heap_bytes = total_objects * 300  # ~150 bytes per file + ~150 bytes per block
        heap_gb = heap_bytes / (1024 ** 3)
        heap_with_margin = heap_gb * 1.3  # 30% safety margin
        return heap_with_margin
    
    def recommend_partition(self, dir_file_counts, total_files, 
                            max_namenodes=4, max_heap_per_nn_gb=80):
        """
        Recommend how to partition namespaces across NameNodes.
        
        Strategy: Bin-packing — assign directories to NameNodes
        such that no NameNode exceeds max_heap_per_nn_gb.
        
        Uses First Fit Decreasing (FFD) heuristic.
        """
        print()
        print("═" * 70)
        print("  FEDERATION PARTITION RECOMMENDATION")
        print("═" * 70)
        print()
        
        # Calculate current single-NN heap requirement
        current_heap = self.estimate_namenode_heap(total_files)
        print(f"  Current state (single NameNode):")
        print(f"    Total files:         {total_files:,}")
        print(f"    Estimated heap need: {current_heap:.1f} GB")
        print(f"    Max recommended:     {max_heap_per_nn_gb} GB per NameNode")
        print()
        
        if current_heap <= max_heap_per_nn_gb:
            print(f"  ✅ Current heap ({current_heap:.1f}GB) fits within single NameNode limit.")
            print(f"     Federation is NOT required for scalability.")
            print(f"     Consider Federation only for namespace isolation.")
            return
        
        # Sort directories by file count (descending) for FFD bin packing
        sorted_dirs = sorted(dir_file_counts.items(), key=lambda x: x[1], reverse=True)
        
        # Bin packing: First Fit Decreasing
        namenodes = []  # List of (namenode_id, [dirs], total_files, heap_gb)
        
        for dir_path, file_count in sorted_dirs:
            dir_heap = self.estimate_namenode_heap(file_count)
            placed = False
            
            for i, nn in enumerate(namenodes):
                nn_id, nn_dirs, nn_files, nn_heap = nn
                if nn_heap + dir_heap <= max_heap_per_nn_gb and len(namenodes) <= max_namenodes:
                    nn_dirs.append(dir_path)
                    namenodes[i] = (nn_id, nn_dirs, nn_files + file_count, nn_heap + dir_heap)
                    placed = True
                    break
            
            if not placed:
                nn_id = f"ns{len(namenodes) + 1}"
                namenodes.append((nn_id, [dir_path], file_count, dir_heap))
        
        # Print recommendation
        print(f"  Recommended: {len(namenodes)} NameNodes")
        print()
        
        for nn_id, nn_dirs, nn_files, nn_heap in namenodes:
            print(f"  ┌─ {nn_id} ──────────────────────────────────────────┐")
            print(f"  │  Files: {nn_files:>15,}                            │")
            print(f"  │  Estimated Heap: {nn_heap:>8.1f} GB                │")
            print(f"  │  Directories:                                      │")
            for d in nn_dirs:
                dir_files = dir_file_counts[d]
                print(f"  │    {d:<30} ({dir_files:>12,} files)  │")
            print(f"  └────────────────────────────────────────────────────┘")
            print()
        
        # Generate ViewFS mount table snippet
        print("  ── GENERATED VIEWFS MOUNT TABLE ──")
        print()
        print("  <!-- Add to core-site.xml on ALL nodes -->")
        for nn_id, nn_dirs, _, _ in namenodes:
            for d in nn_dirs:
                dir_name = d.lstrip('/')
                print(f"  <property>")
                print(f"    <name>fs.viewfs.mounttable.my-cluster.link.{d}</name>")
                print(f"    <value>hdfs://{nn_id}{d}</value>")
                print(f"  </property>")
        
        # Generate hdfs-site.xml nameservices snippet
        print()
        print("  <!-- Add to hdfs-site.xml -->")
        ns_list = ",".join([nn[0] for nn in namenodes])
        print(f"  <property>")
        print(f"    <name>dfs.nameservices</name>")
        print(f"    <value>{ns_list}</value>")
        print(f"  </property>")
        
        return namenodes
    
    def generate_migration_plan(self, namenodes, dir_file_counts):
        """
        Generate step-by-step migration plan from single NameNode to Federation.
        """
        print()
        print("═" * 70)
        print("  MIGRATION PLAN: SINGLE NAMENODE → FEDERATION")
        print("═" * 70)
        print()
        
        steps = [
            "PHASE 1: PREPARATION (No Downtime)",
            "─" * 50,
            "  1.1  Provision new NameNode machines",
            f"        → Need {len(namenodes)} machines (+ {len(namenodes)} for HA standbys)",
            "  1.2  Install Hadoop on new machines (same version as existing)",
            "  1.3  Prepare hdfs-site.xml with all nameservices",
            "  1.4  Prepare core-site.xml with ViewFS mount table",

# Orchestrates cross-namespace data transfers in federated HDFS
# ════════════════════════════════════════════════════════════════

"""
PURPOSE:
Since cross-namespace mv/rename is NOT supported in Federation,
data movement between namespaces requires distcp.

This script orchestrates reliable cross-namespace transfers with:
  - Pre-transfer validation
  - Checksum verification
  - Atomic cutover (copy → verify → delete source)
  - Progress tracking and resumability
"""

import subprocess
import sys
import time
import hashlib
import json
import os

class FederationDistCpOrchestrator:
    """
    Manages reliable data transfer between federated HDFS namespaces.
    
    Why this exists:
    → hdfs dfs -mv /ns1-path /ns2-path FAILS across namespaces
    → distcp works but needs orchestration for production reliability
    → This wraps distcp with validation, retry, and cleanup logic
    """
    
    def __init__(self, state_file="/tmp/federation_distcp_state.json"):
        self.state_file = state_file
        self.state = self._load_state()
    
    def _load_state(self):
        """Load previous transfer state for resumability."""
        if os.path.exists(self.state_file):
            with open(self.state_file, 'r') as f:
                return json.load(f)
        return {"transfers": {}}
    
    def _save_state(self):
        """Persist transfer state."""
        with open(self.state_file, 'w') as f:
            json.dump(self.state, f, indent=2)
    
    def _run_cmd(self, cmd, timeout=3600):
        """Run a shell command with timeout."""
        print(f"  CMD: {' '.join(cmd)}")
        result = subprocess.run(
            cmd,
            capture_output=True,
            text=True,
            timeout=timeout
        )
        return result.returncode, result.stdout, result.stderr
    
    def _get_file_count(self, hdfs_path):
        """Get file count at an HDFS path."""
        rc, stdout, _ = self._run_cmd(["hdfs", "dfs", "-count", hdfs_path])
        if rc == 0 and stdout.strip():
            parts = stdout.strip().split()
            return int(parts[1])  # file count
        return -1
    
    def _get_content_size(self, hdfs_path):
        """Get total content size at an HDFS path."""
        rc, stdout, _ = self._run_cmd(["hdfs", "dfs", "-count", hdfs_path])
        if rc == 0 and stdout.strip():
            parts = stdout.strip().split()
            return int(parts[2])  # content size in bytes
        return -1
    
    def _path_exists(self, hdfs_path):
        """Check if an HDFS path exists."""
        rc, _, _ = self._run_cmd(["hdfs", "dfs", "-test", "-e", hdfs_path])
        return rc == 0
    
    def transfer(self, source_path, dest_path, 
                 delete_source_after=False,
                 bandwidth_mb=100,
                 num_maps=20):
        """
        Transfer data between federated namespaces.
        
        Parameters:
          source_path:         Full HDFS URI (e.g., hdfs://ns1/video/archive)
          dest_path:           Full HDFS URI (e.g., hdfs://ns2/video/archive)
          delete_source_after: If True, delete source AFTER verified transfer
          bandwidth_mb:        Bandwidth limit per mapper in MB
          num_maps:            Number of parallel copy mappers
        
        Flow:
          1. Pre-validate source exists
          2. Run distcp with checksum verification
          3. Post-validate file count and size match
          4. Optionally delete source
          5. Record completion state
        """
        transfer_id = f"{source_path}→{dest_path}"
        
        print()
        print("═" * 70)
        print(f"  CROSS-NAMESPACE TRANSFER")
        print(f"  Source: {source_path}")
        print(f"  Dest:   {dest_path}")
        print("═" * 70)
        
        # Check if already completed
        if self.state["transfers"].get(transfer_id, {}).get("status") == "COMPLETED":
            print("  ✅ Transfer already completed (found in state file). Skipping.")
            return True
        
        # ── STEP 1: PRE-VALIDATION ──
        print()
        print("  ── Step 1: Pre-Validation ──")
        
        if not self._path_exists(source_path):
            print(f"  ❌ Source path does not exist: {source_path}")
            return False
        
        source_files = self._get_file_count(source_path)
        source_size = self._get_content_size(source_path)
        
        print(f"  Source files: {source_files:,}")
        print(f"  Source size:  {source_size / (1024**3):.2f} GB")
        
        if source_files <= 0:
            print("  ⚠️  Source is empty. Nothing to transfer.")
            return True
        
        # Record state
        self.state["transfers"][transfer_id] = {
            "status": "IN_PROGRESS",
            "source_files": source_files,
            "source_size": source_size,
            "start_time": time.strftime("%Y-%m-%d %H:%M:%S")
        }
        self._save_state()
        
        # ── STEP 2: RUN DISTCP ──
        print()
        print("  ── Step 2: Running DistCp ──")
        
        distcp_cmd = [
            "hadoop", "distcp",
            "-bandwidth", str(bandwidth_mb),
            "-m", str(num_maps),
            "-skipcrccheck",          # Skip CRC initially (faster)
            "-strategy", "dynamic",   # Dynamic task assignment for skew handling
            "-log", f"/tmp/distcp-log-{int(time.time())}",
            source_path,
            dest_path
        ]
        
        start_time = time.time()
        rc, stdout, stderr = self._run_cmd(distcp_cmd, timeout=86400)  # 24hr timeout
        elapsed = time.time() - start_time
        
        if rc != 0:
            print(f"  ❌ DistCp FAILED (exit code {rc})")
            print(f"  STDERR: {stderr[:500]}")
            self.state["transfers"][transfer_id]["status"] = "FAILED"
            self.state["transfers"][transfer_id]["error"] = stderr[:500]
            self._save_state()
            return False
        
        print(f"  DistCp completed in {elapsed:.0f} seconds ({elapsed/60:.1f} minutes)")
        
        # ── STEP 3: POST-VALIDATION ──
        print()
        print("  ── Step 3: Post-Validation ──")
        
        dest_files = self._get_file_count(dest_path)
        dest_size = self._get_content_size(dest_path)
        
        print(f"  Source: {source_files:,} files, {source_size / (1024**3):.2f} GB")
        print(f"  Dest:   {dest_files:,} files, {dest_size / (1024**3):.2f} GB")
        
        files_match = source_files == dest_files
        # Allow 1% size tolerance for metadata differences
        size_match = abs(source_size - dest_size) < (source_size * 0.01)
        
        if files_match and size_match:
            print("  ✅ Validation PASSED — file counts and sizes match")
        else:
            print("  ❌ Validation FAILED")
            if not files_match:
                print(f"     File count mismatch: source={source_files}, dest={dest_files}")
            if not size_match:
                print(f"     Size mismatch: source={source_size}, dest={dest_size}")
            self.state["transfers"][transfer_id]["status"] = "VALIDATION_FAILED"
            self._save_state()
            return False
        
        # ── STEP 4: OPTIONAL SOURCE DELETION ──
        if delete_source_after:
            print()
            print("  ── Step 4: Source Cleanup ──")
            print(f"  ⚠️  Deleting source: {source_path}")
            
            rc, _, stderr = self._run_cmd([
                "hdfs", "dfs", "-rm", "-r", "-skipTrash", source_path
            ])
            
            if rc == 0:
                print("  ✅ Source deleted successfully")
            else:
                print(f"  ⚠️  Source deletion failed: {stderr[:200]}")
                print("     Data is safely at destination. Manual cleanup needed.")
        
        # ── STEP 5: RECORD COMPLETION ──
        self.state["transfers"][transfer_id].update({
            "status": "COMPLETED",
            "dest_files": dest_files,
            "dest_size": dest_size,
            "elapsed_seconds": elapsed,
            "end_time": time.strftime("%Y-%m-%d %H:%M:%S")
        })
        self._save_state()
        
        print()
        print("  ✅ TRANSFER COMPLETE")
        return True

🔑 KEY TAKEAWAY

WHAT WE ESTABLISHED IN PROJECT 43:

✅ THE PROBLEM:
   → Single NameNode cannot scale beyond ~500M files without
     crippling GC pauses and 30+ minute restart times
   → Recall Project 2: ALL metadata lives in ONE NameNode's memory
   → This creates a SCALABILITY ceiling that vertical scaling cannot fix

✅ THE SOLUTION — HDFS FEDERATION:
   → Multiple INDEPENDENT NameNodes, each managing its own NAMESPACE
   → Each NameNode has its own BLOCK POOL (set of blocks)
   → ALL NameNodes share the SAME DataNodes
   → DataNodes store blocks from ALL Block Pools on the same disks

✅ KEY ARCHITECTURE:
   → NameNodes do NOT communicate with each other
   → DataNodes register with and heartbeat to ALL NameNodes
   → Block Pool ID + Block ID = globally unique block identifier
   → Cluster ID (shared) ensures all NameNodes belong to same cluster

✅ VIEWFS — THE GLUE:
   → Client-side mount table maps paths to the correct NameNode
   → Applications see ONE unified namespace
   → Transparently routes /video → hdfs://ns1, /billing → hdfs://ns3
   → Existing applications need ZERO code changes

✅ CRITICAL DISTINCTIONS:
   → Federation ≠ High Availability
   → Federation = SCALABILITY (multiple NameNodes for more metadata capacity)
   → HA = RELIABILITY (standby NameNode for crash recovery)
   → Production requires BOTH: each federated NN has its own HA standby

✅ WHAT DOES NOT WORK:
   → Cross-namespace atomic operations (mv, rename)
   → Cross-Block-Pool balancing
   → Cluster-wide snapshots (snapshots are per-NameNode)
