PROJECT 2: STORING A 10GB TEXT FILE NOBODY CAN OPEN LOCALLY

🔴 HYPOTHETICAL PROBLEM

SCENARIO:
→ You work at a telecom company
→ The billing system generates a single CSV file every night
→ File: "call_records_20240115.csv" — exactly 10GB
→ Contains: caller_id, receiver_id, duration, timestamp, cell_tower_id

💡 HOW CAN WE SOLVE IT

ANSWER TO QUESTION 1: WHY SPLIT INTO BLOCKS?

Without blocks (traditional filesystem):
┌───────────────────────────────────────────┐
│         10GB FILE ON ONE DISK             │
│  [══════════════════════════════════════]  │
│  → One machine reads it: 10GB / 200MB/s   │
│    = 50 seconds JUST TO READ              │
│  → If disk fails: ENTIRE file lost        │
│  → Cannot parallelize: one machine only   │
└───────────────────────────────────────────┘

With blocks (HDFS):
┌───────────────────────────────────────────────────────────┐
│              10GB FILE SPLIT INTO 128MB BLOCKS             │
│                                                            │
│  10GB / 128MB = 80 BLOCKS                                  │
│                                                            │
│  DataNode 1: [Blk01][Blk05][Blk09]...[Blk77]  (20 blocks)│
│  DataNode 2: [Blk02][Blk06][Blk10]...[Blk78]  (20 blocks)│
│  DataNode 3: [Blk03][Blk07][Blk11]...[Blk79]  (20 blocks)│
│  DataNode 4: [Blk04][Blk08][Blk12]...[Blk80]  (20 blocks)│
│                                                            │
│  → 4 machines read in parallel:                            │
│    Each reads 2.5GB / 200MB/s = 12.5 seconds              │
│  → 4× faster than single machine                          │
│  → If DataNode 2 dies: blocks exist on other nodes         │
└───────────────────────────────────────────────────────────┘

ANSWER TO QUESTION 2: WHY 128MB SPECIFICALLY?

The block size is a TRADEOFF between two forces:

FORCE A — Seek Time vs. Transfer Time
→ Typical disk seek time: ~10ms
→ Typical disk transfer rate: ~100-200 MB/s
→ Goal: Make seek time < 1% of total block read time
→ 10ms seek + (128MB / 150MB/s) ≈ 10ms + 850ms = 860ms
→ Seek is ~1.2% of total — ACCEPTABLE
→ If block were 1MB: 10ms + 6.7ms = 16.7ms → Seek is 60% — TERRIBLE

FORCE B — NameNode Memory
→ Each block entry costs ~150 bytes in NameNode memory
→ Smaller blocks = MORE blocks = MORE NameNode memory
→ 10GB file with 128MB blocks = 80 blocks × 150 bytes = 12KB metadata
→ 10GB file with 1MB blocks = 10,240 blocks × 150 bytes = 1.5MB metadata
→ At petabyte scale this difference becomes ENORMOUS

THE MATH:
┌─────────────┬──────────────┬────────────────────┬──────────────────┐
│ Block Size  │ Blocks for   │ NameNode Memory    │ Seek Overhead    │
│             │ 10GB file    │ for 10GB file      │                  │
├─────────────┼──────────────┼────────────────────┼──────────────────┤
│ 1MB         │ 10,240       │ 1.5 MB             │ 60% (terrible)   │
│ 16MB        │ 640          │ 96 KB              │ 9% (poor)        │
│ 64MB        │ 160          │ 24 KB              │ 2.3% (ok)        │
│ 128MB       │ 80           │ 12 KB              │ 1.2% (good)      │ ← DEFAULT
│ 256MB       │ 40           │ 6 KB               │ 0.6% (great)     │
│ 1GB         │ 10           │ 1.5 KB             │ 0.15% (great)    │
└─────────────┴──────────────┴────────────────────┴──────────────────┘

WHY NOT 1GB?
→ MapReduce assigns ONE mapper per block (typically)
→ 10 blocks = only 10 mappers = poor parallelism
→ 128MB gives 80 mappers = excellent parallelism

128MB is the SWEET SPOT between:
  → NameNode memory efficiency
  → Disk seek efficiency  
  → MapReduce parallelism

✅ WHAT COULD BE DONE

ANSWER TO QUESTION 3: WHAT DOES NAMENODE ACTUALLY STORE?

The NameNode maintains TWO critical data structures IN MEMORY:

STRUCTURE 1: NAMESPACE IMAGE (fsimage)
→ The complete directory tree of HDFS
→ Every file, every directory, permissions, timestamps
→ Like a phone book — maps file names to block lists

  /
  ├── user/
  │   └── telecom/
  │       └── call_records_20240115.csv
  │           → file_id: 1001
  │           → owner: telecom_etl
  │           → permissions: 644
  │           → size: 10,737,418,240 bytes (10GB)
  │           → replication: 3
  │           → block_size: 134,217,728 (128MB)
  │           → blocks: [blk_001, blk_002, ..., blk_080]
  │
  └── tmp/
      └── staging/

STRUCTURE 2: BLOCK-TO-DATANODE MAPPING
→ Which DataNode holds which block replica
→ NOT persisted to disk — rebuilt from DataNode heartbeats at startup

  blk_001 → [DataNode1:50010, DataNode3:50010, DataNode5:50010]
  blk_002 → [DataNode2:50010, DataNode4:50010, DataNode1:50010]
  blk_003 → [DataNode3:50010, DataNode5:50010, DataNode2:50010]
  ...
  blk_080 → [DataNode4:50010, DataNode1:50010, DataNode3:50010]

PERSISTENCE MECHANISM:
→ fsimage: Full snapshot written to disk periodically
→ edits log: Every change (create file, delete, rename) written immediately
→ At startup: fsimage + edits log = current state
→ SecondaryNameNode periodically merges edits into fsimage
  (to prevent edits log from growing forever)

  ┌────────────────────────────────────────────────────────┐
  │                 NAMENODE PERSISTENCE                    │
  │                                                        │
  │  [fsimage_0]  +  [edits_1, edits_2, ..., edits_N]     │
  │       ↓                        ↓                       │
  │  SecondaryNameNode merges them periodically             │
  │       ↓                                                │
  │  [fsimage_N]  (new checkpoint)                         │
  │                                                        │
  │  ⚠️ SecondaryNameNode is NOT a backup NameNode!        │
  │     It is a CHECKPOINTING helper only.                 │
  │     If NameNode dies, cluster is DOWN.                 │
  │     (Fixed in Project 44: NameNode HA)                 │
  └────────────────────────────────────────────────────────┘

ANSWER TO QUESTION 4: WHAT DOES DATANODE ACTUALLY STORE ON DISK?

Each DataNode stores blocks as REGULAR FILES on its local filesystem:

  DataNode 1 local disk:
  /opt/hadoop/data/datanode/current/BP-xxx/current/finalized/
  ├── subdir0/
  │   ├── blk_001          ← actual data (128MB of raw bytes)
  │   ├── blk_001.meta     ← checksum file (for corruption detection)
  │   ├── blk_005
  │   ├── blk_005.meta
  │   └── ...
  └── subdir1/
      ├── blk_009
      ├── blk_009.meta
      └── ...

IMPORTANT DETAILS:
→ blk_001 is literally 128MB of the original file's bytes
→ blk_001.meta contains CRC32 checksums (one per 512 bytes)
→ On READ, DataNode verifies checksums → detects bit rot
→ If corruption detected → DataNode reports to NameNode
   → NameNode triggers re-replication from a healthy copy

✅ WHAT SHOULD DO

ANSWER TO QUESTION 5: HOW DOES REPLICATION ACTUALLY HAPPEN?

RACK AWARENESS STRATEGY:
→ Hadoop knows which DataNodes are in which server rack
→ Replication placement follows a specific policy:

  For replication factor = 3:
  
  Replica 1: Same node as the writer (or random node if client is outside cluster)
  Replica 2: A node on a DIFFERENT rack
  Replica 3: A different node on the SAME rack as Replica 2

  WHY THIS SPECIFIC PLACEMENT?
  
  ┌─────── Rack 1 ──────┐    ┌─────── Rack 2 ──────┐
  │                      │    │                      │
  │  [DN1: Replica 1] ◄─┼────┼─► [DN3: Replica 2]   │
  │                      │    │   [DN4: Replica 3]   │
  │  [DN2]               │    │                      │
  └──────────────────────┘    └──────────────────────┘

  → If Rack 1 loses power → Replicas 2 & 3 on Rack 2 survive
  → If DN3 dies → Replica 3 on DN4 (same rack) survives
  → Writing only crosses rack switch ONCE (not twice)
    → Minimizes inter-rack bandwidth usage

PIPELINE REPLICATION (from Project 1, now in detail):

  Client writes blk_001:
  
  Step 1: Client asks NameNode "Where should I put this block?"
  Step 2: NameNode returns: [DN1, DN3, DN4]
  Step 3: Client opens pipeline:
  
  ┌────────┐  data packets  ┌──────┐  forward  ┌──────┐  forward  ┌──────┐
  │ Client │ ─────────────→ │ DN1  │ ────────→ │ DN3  │ ────────→ │ DN4  │
  └────────┘                └──────┘           └──────┘           └──────┘
       ↑                         │                  │                  │
       └─────── ACK ─────────────┴───── ACK ────────┴────── ACK ──────┘
  
  Step 4: Data flows in 64KB PACKETS (not entire 128MB at once)
  Step 5: Each packet is ACKnowledged back through the pipeline
  Step 6: When all packets ACKed → block is committed
  Step 7: Client tells NameNode "blk_001 written to DN1, DN3, DN4"
  Step 8: NameNode records the mapping

1. ALWAYS verify block health after uploading large files
   → hdfs fsck /path/to/file -files -blocks -locations

2. UNDERSTAND the 128MB default — change it ONLY with reason
   → Larger blocks (256MB) for very large sequential files
   → Smaller blocks (64MB) NEVER recommended in modern Hadoop

3. MONITOR NameNode heap usage as file count grows
   → Each file + block costs ~150 bytes of NameNode heap
   → 10 million files with 3 blocks each = 30M entries × 150B ≈ 4.5GB heap

4. ALWAYS check replication factor matches your fault tolerance needs
   → Development: 1 (saves disk, no fault tolerance needed)
   → Production: 3 (survives any single node failure)
   → Critical data: 5+ (survives rack-level failures)

🚫 WHAT SHOULD NOT DO

1. DO NOT store millions of small files in HDFS
   → 1 million files × 1KB each = 1GB of data
   → BUT: 1 million files × 150 bytes metadata = 150MB of NameNode heap
   → Meanwhile: 1 single 1GB file = ~8 blocks × 150B = 1.2KB metadata
   → RATIO: Small files use 125,000× MORE NameNode memory per byte of data

   ┌──────────────────────────────────────────────────────────────┐
   │          THE SMALL FILES PROBLEM — VISUALIZED                │
   │                                                              │
   │  1,000,000 × 1KB files:                                     │
   │    Data stored:    1 GB                                      │
   │    NameNode heap:  ~150 MB (just for metadata!)              │
   │    Blocks created: 1,000,000 (one per file minimum)          │
   │                                                              │
   │  1 × 1GB file:                                               │
   │    Data stored:    1 GB                                      │
   │    NameNode heap:  ~1.2 KB                                   │
   │    Blocks created: 8                                         │
   │                                                              │
   │  SAME DATA, 125,000× MORE NAMENODE PRESSURE                 │
   └──────────────────────────────────────────────────────────────┘

2. DO NOT format the NameNode after initial setup
   → "hdfs namenode -format" DESTROYS all metadata
   → All files become inaccessible (data still on DataNodes but orphaned)
   → Only format ONCE during initial cluster setup

3. DO NOT set replication factor higher than number of DataNodes
   → 3 replicas with 2 DataNodes → one replica can never be placed
   → NameNode will constantly warn about under-replicated blocks
   → Wastes NameNode effort trying to place impossible replicas

4. DO NOT ignore under-replicated blocks warnings
   → Means some blocks have fewer copies than expected
   → One more node failure could cause DATA LOSS
   → Check with: hdfs dfsadmin -report

5. DO NOT let NameNode's edits log grow unbounded
   → SecondaryNameNode must be running and checkpointing
   → If edits log is huge, NameNode restart takes HOURS
   → (It replays every edit since last checkpoint)

6. DO NOT put the NameNode metadata on the same disk as DataNode storage
   → NameNode metadata writes must be fast and reliable
   → DataNode I/O will compete and slow down metadata operations
   → In production: NameNode gets dedicated SSD/RAID disks

🚫 WHEN NOT TO USE

HDFS IS THE WRONG STORAGE WHEN:

1. YOU NEED RANDOM READ/WRITE ACCESS
   → HDFS is optimized for SEQUENTIAL reads of large files
   → Reading bytes 5000-6000 from a 10GB file is inefficient
   → HDFS must find the right block, seek to the offset
   → Use HBase (Project 27) or a traditional database instead

2. YOU NEED LOW-LATENCY ACCESS
   → HDFS read involves: Client → NameNode (get block locations) → DataNode
   → Minimum latency: several milliseconds
   → For sub-millisecond access: use Redis, Memcached

3. YOU HAVE MANY SMALL FILES
   → Explained above in "What Should Not Do"
   → Solutions: HAR files, SequenceFiles (covered in Project 30)

4. YOU NEED POSIX-COMPLIANT FILESYSTEM
   → HDFS does NOT support:
     → File append efficiently (append is supported but slow)
     → Random writes (cannot modify byte 5000 of a file)
     → Hard links or symbolic links
     → File locking for concurrent writers
   → Need these? Use a regular filesystem (ext4, XFS)

5. YOUR DATA CHANGES FREQUENTLY
   → HDFS is WRITE-ONCE, READ-MANY
   → Updating one record means rewriting the entire file
   → Use a database or HBase for mutable data

⚙️ HOW IT ONLY WORKS — INTERNAL MECHANICS

MECHANIC 1: THE COMPLETE FILE WRITE JOURNEY

When you run: hdfs dfs -put call_records.csv /user/telecom/

┌──────────────────────────────────────────────────────────────────────┐
│ STEP-BY-STEP INTERNAL FLOW                                          │
│                                                                      │
│ ① Client calls NameNode.create("/user/telecom/call_records.csv")    │
│    → NameNode checks:                                                │
│      → Does parent directory exist? YES → continue                   │
│      → Does file already exist? NO → continue                        │
│      → Does client have write permission? YES → continue             │
│    → NameNode creates file entry in namespace (edits log)            │
│    → Returns SUCCESS to client (no DataNodes involved yet!)          │
│                                                                      │
│ ② Client starts writing data                                        │
│    → Client's DFSOutputStream splits data into PACKETS (64KB each)  │
│    → Packets accumulate into a BLOCK (128MB worth)                   │
│                                                                      │
│ ③ For each block, client asks NameNode for DataNode list             │
│    Client: "Where should I write block blk_001?"                     │
│    NameNode: "Write to [DN1, DN3, DN5]" (rack-aware selection)       │
│                                                                      │
│ ④ Client creates replication pipeline                                │
│    Client → DN1 → DN3 → DN5                                         │
│                                                                      │
│ ⑤ Packets flow through pipeline                                     │
│    ┌────────┐  pkt1  ┌─────┐  pkt1  ┌─────┐  pkt1  ┌─────┐        │
│    │ Client │──────→ │ DN1 │──────→ │ DN3 │──────→ │ DN5 │         │
│    └────────┘        └─────┘        └─────┘        └─────┘         │
│         ↑               │               │               │           │
│         └───── ACK ─────┴──── ACK ──────┴──── ACK ──────┘           │
│                                                                      │
│    → Client maintains TWO queues:                                    │
│      → DATA QUEUE: packets waiting to be sent                        │
│      → ACK QUEUE: packets sent but not yet acknowledged              │
│    → When ACK received: packet removed from ACK queue                │
│    → If pipeline fails: unACKed packets moved back to data queue     │
│      → Failed DataNode removed from pipeline                         │
│      → Pipeline rebuilt with remaining DataNodes                     │
│      → NameNode schedules re-replication to replace failed node      │
│                                                                      │
│ ⑥ After 128MB of packets → block complete                           │
│    → Client tells NameNode "blk_001 finalized on DN1, DN3, DN5"     │
│    → Process repeats for blk_002, blk_003, ..., blk_080             │
│                                                                      │
│ ⑦ All blocks written → Client calls NameNode.complete()             │
│    → NameNode verifies minimum replication met for all blocks        │
│    → File marked as COMPLETE in namespace                            │
│    → File is now visible to other clients                            │
└──────────────────────────────────────────────────────────────────────┘

MECHANIC 3: HEARTBEAT AND BLOCK REPORT SYSTEM

┌──────────────────────────────────────────────────────────────────────┐
│                                                                      │
│ HEARTBEAT (every 3 seconds):                                        │
│   DataNode → NameNode: "I am alive"                                 │
│   NameNode → DataNode: "OK" + optional COMMANDS:                    │
│     → "Replicate blk_005 to DN7"                                    │
│     → "Delete blk_012 (no longer needed)"                           │
│     → "Shut down (decommissioning)"                                 │
│                                                                      │
│ BLOCK REPORT (every 6 hours by default):                            │
│   DataNode → NameNode: "Here is my complete block list"             │
│     → [blk_001, blk_005, blk_009, blk_013, ...]                    │
│   NameNode uses this to:                                             │
│     → Detect missing replicas (trigger re-replication)               │
│     → Detect extra replicas (trigger deletion)                       │
│     → Rebuild block-to-DataNode mapping after restart                │
│                                                                      │
│ DEAD NODE DETECTION:                                                 │
│   → No heartbeat for 10 minutes + 30 seconds (default)              │
│     = dfs.namenode.heartbeat.recheck-interval (5 min) × 2           │
│       + 10 × heartbeat interval (3s × 10 = 30s)                     │
│   → NameNode marks DataNode as DEAD                                  │
│   → All blocks on dead node: under-replicated                        │
│   → NameNode commands other DataNodes to re-replicate                │
│                                                                      │
│ SELF-HEALING EXAMPLE:                                                │
│                                                                      │
│   Before failure:                                                    │
│     blk_001: DN1 ✓, DN3 ✓, DN5 ✓  (3 replicas — healthy)          │
│                                                                      │
│   DN5 crashes:                                                       │
│     blk_001: DN1 ✓, DN3 ✓, DN5 ✗  (2 replicas — under-replicated) │
│                                                                      │
│   NameNode detects after 10.5 minutes:                               │
│     → Finds blk_001 is under-replicated                              │
│     → Selects DN2 as new target (rack-aware)                         │
│     → Commands DN1: "Send blk_001 to DN2"                            │
│                                                                      │
│   After re-replication:                                               │
│     blk_001: DN1 ✓, DN3 ✓, DN2 ✓  (3 replicas — healthy again)    │
│                                                                      │
│   ALL AUTOMATIC — NO HUMAN INTERVENTION                              │
└──────────────────────────────────────────────────────────────────────┘

⚠️ COMMON PITFALLS AND FAILURE MODES

PITFALL 1: "I uploaded a 500MB file but HDFS shows it using 1.5GB"
→ REASON: Replication factor = 3
→ 500MB × 3 replicas = 1.5GB physical disk usage
→ NOT a bug — this is the COST of fault tolerance
→ hdfs dfs -du -s -h /path shows LOGICAL size
→ hdfs dfs -du -s -h /path × replication = PHYSICAL size

PITFALL 2: "File upload succeeded but fsck shows CORRUPT blocks"
→ POSSIBLE CAUSES:
  → Network corruption during pipeline transfer
  → Disk bit rot on DataNode
  → Abrupt DataNode shutdown during write
→ SOLUTION: HDFS automatically detects via checksums
  → Re-replicate from healthy copy
  → If ALL replicas corrupt → data loss (extremely rare with rep=3)

PITFALL 3: "NameNode takes 30 minutes to start"
→ REASON: Loading fsimage + replaying edits log
→ If edits log has millions of entries → slow replay
→ SOLUTION: Ensure SecondaryNameNode is running and checkpointing
→ Check: ls $HADOOP_HOME/data/namenode/current/
  → If edits_* files are huge → checkpointing is broken

PITFALL 4: "Cannot write files — SafeMode"
→ After NameNode starts, it enters SAFE MODE
→ In SafeMode: reads allowed, writes BLOCKED
→ NameNode waits for DataNodes to report their blocks
→ Exits SafeMode when 99.9% of blocks have minimum replicas reported
→ If DataNodes are slow to start → SafeMode persists
→ NEVER force-leave SafeMode in production:
  hdfs dfsadmin -safemode leave  ← DANGEROUS if blocks truly missing
→ Wait for DataNodes to report, or investigate why they haven't

PITFALL 5: "Deleted a file but disk space didn't free up"
→ REASON: HDFS Trash is enabled by default
→ Deleted files go to /user/<username>/.Trash/
→ Actual deletion happens after trash interval (default: disabled in some configs)
→ To bypass trash: hdfs dfs -rm -skipTrash /path/to/file
→ To empty trash: hdfs dfs -expunge

PITFALL 6: "Block not found but file exists"
→ REASON: NameNode metadata says block exists but no DataNode has it
→ This means ALL replicas of that block are lost
→ File is CORRUPT — that portion of data is unrecoverable
→ Prevention: maintain replication factor ≥ 3, monitor under-replicated blocks

📈 SCALABILITY CONSIDERATIONS

NAMENODE MEMORY — THE SCALING CEILING:

┌──────────────────────┬─────────────┬───────────────────────────┐
│ Cluster Data Size    │ Approx Files│ NameNode Heap Needed      │
│ (with rep=3)         │ + Blocks    │                           │
├──────────────────────┼─────────────┼───────────────────────────┤
│ 1 TB                 │ ~24K blocks │ ~4 MB (trivial)           │
├──────────────────────┼─────────────┼───────────────────────────┤
│ 100 TB               │ ~2.4M blocks│ ~360 MB                   │
├──────────────────────┼─────────────┼───────────────────────────┤
│ 1 PB                 │ ~24M blocks │ ~3.6 GB                   │
├──────────────────────┼─────────────┼───────────────────────────┤
│ 10 PB                │ ~240M blocks│ ~36 GB                    │
├──────────────────────┼─────────────┼───────────────────────────┤
│ 100 PB               │ ~2.4B blocks│ ~360 GB ← PROBLEM!       │
│                      │             │ (single JVM cannot handle)│
└──────────────────────┴─────────────┴───────────────────────────┘

→ At ~100PB with default block size, NameNode hits JVM limits
→ SOLUTION: HDFS Federation (multiple NameNodes) — Project 43
→ SOLUTION: Increase block size to 256MB+ to halve block count

DATANODE SCALING:
→ Each DataNode can handle thousands of blocks
→ Disk I/O is the bottleneck, not block count
→ Typical DataNode: 12-24 disks × 4-16TB each = 48-384TB per node
→ Adding DataNodes = linear storage growth
→ HDFS automatically balances blocks across new nodes
  → hdfs balancer — redistributes blocks for even disk usage

NETWORK SCALING:
→ Write bandwidth: limited by pipeline (sequential through 3 nodes)
→ Read bandwidth: scales with replicas (3 replicas = 3 possible readers)
→ Inter-rack bandwidth: often 10× less than intra-rack
→ Rack-aware replication minimizes cross-rack traffic

# Using subprocess to call hdfs commands (works without extra libraries)

import subprocess
import json
import re

def run_hdfs_command(cmd):
    """Execute an HDFS command and return output."""
    result = subprocess.run(cmd, shell=True, capture_output=True, text=True)
    return result.stdout, result.stderr

def get_file_info(hdfs_path):
    """Get detailed file information from HDFS."""
    print(f"\n{'='*70}")
    print(f"HDFS FILE ANALYSIS: {hdfs_path}")
    print(f"{'='*70}")
    
    # Basic file stats
    stdout, _ = run_hdfs_command(f'hdfs dfs -stat "%n|%b|%r|%o|%y" {hdfs_path}')
    if stdout.strip():
        parts = stdout.strip().split('|')
        name = parts[0]
        size_bytes = int(parts[1])
        replication = int(parts[2])
        block_size = int(parts[3])
        mod_time = parts[4]
        
        num_blocks = (size_bytes + block_size - 1) // block_size  # ceiling division
        
        print(f"\n  File Name:        {name}")
        print(f"  File Size:        {size_bytes:,} bytes ({size_bytes/(1024*1024):.1f} MB)")
        print(f"  Replication:      {replication}")
        print(f"  Block Size:       {block_size:,} bytes ({block_size/(1024*1024):.0f} MB)")
        print(f"  Expected Blocks:  {num_blocks}")
        print(f"  Physical Storage: {size_bytes * replication / (1024*1024):.1f} MB (with replication)")
        print(f"  Last Modified:    {mod_time}")
        
        # NameNode memory estimate
        nn_memory = num_blocks * 150  # ~150 bytes per block entry
        print(f"\n  NameNode Memory Impact:")
        print(f"    Block entries:  {num_blocks} × 150 bytes = {nn_memory:,} bytes")
        print(f"    File entry:     ~{250} bytes (path, permissions, etc.)")
        print(f"    Total:          ~{nn_memory + 250:,} bytes")
    
    # Block-level details via fsck
    print(f"\n  {'─'*50}")
    print(f"  BLOCK-LEVEL DETAILS:")
    print(f"  {'─'*50}")
    
    stdout, _ = run_hdfs_command(f'hdfs fsck {hdfs_path} -files -blocks -locations 2>/dev/null')
    
    block_pattern = re.compile(r'(\d+)\.\s+\S+:(blk_\d+).*?len=(\d+)\s+\[(.*?)\]')
    blocks = block_pattern.findall(stdout)
    
    total_block_bytes = 0
    for idx, block_id, length, locations in blocks:
        length = int(length)
        total_block_bytes += length
        loc_list = locations.split(', ') if locations else ['unknown']
        
        print(f"\n    Block {idx}:")
        print(f"      ID:        {block_id}")
        print(f"      Size:      {length:,} bytes ({length/(1024*1024):.1f} MB)")
        print(f"      Locations: {loc_list}")
        
        if length < block_size:
            print(f"      Note:      PARTIAL BLOCK (last block of file)")
    
    if blocks:
        print(f"\n  {'─'*50}")
        print(f"  SUMMARY:")
        print(f"    Total blocks found: {len(blocks)}")
        print(f"    Full blocks:        {len([b for b in blocks if int(b[2]) == block_size])}")
        print(f"    Partial blocks:     {len([b for b in blocks if int(b[2]) < block_size])}")
        print(f"    Total block bytes:  {total_block_bytes:,} (should match file size)")

def compare_block_strategies(hdfs_dir):
    """Compare different block size strategies for files in a directory."""
    print(f"\n{'='*70}")
    print(f"BLOCK STRATEGY COMPARISON")
    print(f"{'='*70}")
    
    stdout, _ = run_hdfs_command(f'hdfs dfs -ls {hdfs_dir}')
    
    files = []
    for line in stdout.strip().split('\n'):
        if line.startswith('-'):
            parts = line.split()
            path = parts[-1]
            size = int(parts[4])
            files.append((path, size))
    
    print(f"\n  {'File':<55} {'Size':>10} {'Blocks':>8}")
    print(f"  {'─'*55} {'─'*10} {'─'*8}")
    
    for path, size in files:
        fsck_out, _ = run_hdfs_command(f'hdfs fsck {path} -files -blocks 2>/dev/null')
        block_match = re.search(r'(\d+)\s+block\(s\)', fsck_out)
        num_blocks = int(block_match.group(1)) if block_match else 0
        
        filename = path.split('/')[-1]
        print(f"  {filename:<55} {size/(1024*1024):>8.1f}MB {num_blocks:>6}")

def demonstrate_namenode_scaling():
    """Show how NameNode memory grows with file count."""
    print(f"\n{'='*70}")
    print(f"NAMENODE MEMORY SCALING DEMONSTRATION")
    print(f"{'='*70}")
    
    scenarios = [
        ("Small startup",     100,    "1 GB",     128),
        ("Growing company",   10000,  "100 GB",   128),
        ("Enterprise",        100000, "10 TB",    128),
        ("Big Data company",  1000000,"1 PB",     128),
        ("Web giant",         10000000,"100 PB",  128),
        ("Web giant (256MB)", 10000000,"100 PB",  256),
    ]
    
    print(f"\n  {'Scenario':<22} {'Files':>12} {'Data':>8} {'BlkSize':>8} {'~Blocks':>12} {'NN Heap':>10}")
    print(f"  {'─'*22} {'─'*12} {'─'*8} {'─'*8} {'─'*12} {'─'*10}")
    
    for name, num_files, data_size, blk_mb in scenarios:
        # Estimate: average file has ~5 blocks
        if blk_mb == 256:
            avg_blocks = 2.5
        else:
            avg_blocks = 5
        
        total_blocks = int(num_files * avg_blocks)
        # Each file entry ~250 bytes, each block entry ~150 bytes
        nn_bytes = (num_files * 250) + (total_blocks * 150)
        
        if nn_bytes < 1024*1024:
            nn_str = f"{nn_bytes/1024:.0f} KB"
        elif nn_bytes < 1024*1024*1024:
            nn_str = f"{nn_bytes/(1024*1024):.0f} MB"
        else:
            nn_str = f"{nn_bytes/(1024*1024*1024):.1f} GB"
        
        print(f"  {name:<22} {num_files:>12,} {data_size:>8} {blk_mb:>6}MB {total_blocks:>12,} {nn_str:>10}")
    
    print(f"\n  KEY INSIGHT:")
    print(f"  → NameNode memory is proportional to NUMBER of files and blocks")
    print(f"  → NOT proportional to DATA size")
    print(f"  → 1 file of 1TB uses less NameNode memory than 1 million files of 1KB")
