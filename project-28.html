PROJECT 28: IMPORTING A MYSQL DATABASE INTO HDFS AUTOMATICALLY

🔴 HYPOTHETICAL PROBLEM

SCENARIO:
→ You are a data engineer at the ride-sharing company
→ The company's OPERATIONAL systems run on MySQL:
   - riders table: 50 million rows (user registrations, profiles)
   - drivers table: 500,000 rows (driver info, license details)
   - rides table: 2.8 billion rows (every ride ever taken)
   - payments table: 2.8 billion rows (payment transactions)
   - cities table: 500 rows (city metadata, surge pricing zones)

→ The analytics team needs this data in the Hadoop ecosystem:
   - Hive warehouse (Project 24-25) for SQL analytics
   - Pig pipelines (Project 26) for ETL transformations
   - HBase (Project 27) for real-time serving

CURRENT PAINFUL PROCESS:
→ DBA runs: mysqldump --tab rides > rides.tsv (takes 18 hours)
→ Engineer runs: scp rides.tsv hadoop-node:/tmp/ (takes 6 hours)
→ Engineer runs: hdfs dfs -put /tmp/rides.tsv /data/ (takes 3 hours)
→ Total: 27 HOURS for ONE table
→ Must be repeated for EVERY table, EVERY day
→ No parallelism — single thread export
→ No incremental — re-exports ALL 2.8 billion rows daily
→ No schema awareness — Hive table must be manually created
→ If MySQL schema changes → manual Hive DDL update required
→ Error handling: none — if transfer fails at hour 20, start over

WHAT WE NEED:
→ Parallel extraction from MySQL (use all 8 cores)
→ Direct write to HDFS (skip local disk entirely)
→ Incremental import (only rows changed since yesterday)
→ Auto-create Hive tables matching MySQL schema
→ Import directly as ORC/Parquet (not raw text)
→ Handle billions of rows reliably
→ One command to import, one command to export

THIS IS EXACTLY WHAT APACHE SQOOP DOES.

STEP 2: Understand Sqoop's core capability

Sqoop uses MAPREDUCE under the hood:
→ It generates a MapReduce job where each MAPPER reads a CHUNK of the table
→ Multiple mappers read from MySQL in PARALLEL via JDBC
→ Each mapper writes its chunk DIRECTLY to HDFS
→ No intermediate local storage needed
→ Uses the split-by column to divide the work

EXAMPLE: Importing 100 million rows with 4 mappers

💡 HOW CAN WE SOLVE IT

SOLUTION: Apache Sqoop — Bidirectional RDBMS ↔ Hadoop connector

SQOOP HAS TWO CORE OPERATIONS:

OPERATION 1: IMPORT (RDBMS → Hadoop)
   sqoop import --connect jdbc:mysql://... --table rides --target-dir /data/rides
   → Reads from MySQL
   → Writes to HDFS (or directly to Hive/HBase)

OPERATION 2: EXPORT (Hadoop → RDBMS)
   sqoop export --connect jdbc:mysql://... --table analytics --export-dir /output/
   → Reads from HDFS
   → Writes to MySQL (for dashboards, APIs)

IMPORT MODES:

MODE 1 — Full Import (all rows, every time)
   → Best for: small tables, first-time load
   → Drawback: re-imports unchanged data

MODE 2 — Incremental Append (new rows only)
   → --incremental append --check-column ride_id --last-value 1000000
   → Only imports rows where ride_id > 1000000
   → Best for: append-only tables (new rides added daily)

MODE 3 — Incremental LastModified (changed rows)
   → --incremental lastmodified --check-column updated_at --last-value "2024-03-22"
   → Imports rows where updated_at > '2024-03-22'
   → Best for: tables where existing rows are updated (profiles, settings)

IMPORT TARGETS:

TARGET 1 — HDFS directory (default)
   → Writes text/CSV/Parquet/Avro files to specified directory
   → Most flexible — downstream Pig/MR can process

TARGET 2 — Hive table (--hive-import)
   → Creates Hive table matching MySQL schema
   → Loads data into the Hive warehouse
   → Immediate SQL query capability (Project 24)

TARGET 3 — HBase table (--hbase-table)
   → Writes each row to HBase with specified row key
   → Immediate random-access capability (Project 27)

TARGET 4 — Hive with HCatalog (--hcatalog-table)
   → Uses HCatalog for Hive integration
   → Supports ORC/Parquet format directly
   → Respects Hive partitioning

✅ WHAT SHOULD DO

1. ALWAYS USE --split-by FOR PARALLEL IMPORT
   → Sqoop splits the table into ranges using this column
   → Column MUST be numeric or date type with even distribution
   → Best choice: primary key (auto-increment ID)
   → Without --split-by: Sqoop uses primary key automatically
   → If no primary key exists: MUST specify --split-by or use -m 1

2. CHOOSE THE RIGHT NUMBER OF MAPPERS
   → Default: 4 mappers
   → Too few → slow import (not enough parallelism)
   → Too many → overwhelm MySQL (too many concurrent connections)
   → RULE: 4-8 mappers for typical MySQL server
   → RULE: Up to 16-20 for dedicated MySQL read replica
   → NEVER point Sqoop at your production master — use a READ REPLICA

3. USE --direct FOR MYSQL-SPECIFIC OPTIMIZATION
   → --direct flag uses mysqldump/mysqlimport native tools
   → 2-3× faster than JDBC for MySQL specifically
   → Bypasses JDBC overhead
   → Limitation: doesn't work with all column types or --hive-import

4. SPECIFY --as-parquetfile OR --as-avrodatafile FOR PRODUCTION
   → Default output is CSV text — inefficient for large datasets
   → Parquet: columnar, compressed, fast for analytics (Project 30)
   → Avro: row-based, schema evolution support (Project 30)
   → ORC: use --hcatalog-table with Hive (Project 25)

5. USE SQOOP JOBS FOR RECURRING IMPORTS
   → sqoop job --create daily_rides_import -- import ...
   → Saves all parameters — run with: sqoop job --exec daily_rides_import
   → Automatically tracks --last-value for incremental imports
   → Integrates with Oozie scheduler (Project 31)

6. ALWAYS TEST WITH --where CLAUSE FIRST
   → Before importing 2.8 billion rows, test with:
     sqoop import ... --where "ride_id < 10000"
   → Verify schema mapping, data quality, target directory
   → Then run full import

7. USE --null-string AND --null-non-string
   → MySQL NULLs need explicit handling
   → --null-string '\\N' --null-non-string '\\N'
   → Matches Hive's default NULL representation
   → Without this: NULLs may import as the string "null"

🚫 WHAT SHOULD NOT DO

1. DO NOT POINT SQOOP AT PRODUCTION DATABASE MASTER
   → Sqoop runs parallel SELECT queries with BETWEEN clauses
   → 8 mappers = 8 concurrent long-running queries
   → Can lock tables, consume I/O, slow down production app
   → ALWAYS use a READ REPLICA or snapshot

2. DO NOT IMPORT WITHOUT --split-by ON LARGE TABLES
   → Without split column, Sqoop uses single mapper (-m 1)
   → Single mapper = single JDBC connection = sequential read
   → 2.8 billion rows through one connection = DAYS
   → With 8 mappers and good split column = HOURS

3. DO NOT USE NON-UNIFORM SPLIT COLUMNS
   → If split column has gaps (IDs 1-1000, then 1000000-2000000)
   → Sqoop divides range evenly: mapper 0 gets 1-500000 (only 1000 rows)
   → Mapper 1 gets 500001-1000000 (zero rows!)
   → Mapper 2 gets 1000001-1500000 (500000 rows — all the work)
   → MASSIVE SKEW → some mappers finish instantly, one takes forever
   → FIX: Use a uniformly distributed column or --boundary-query

4. DO NOT FORGET TO ESCAPE DELIMITERS IN DATA
   → If MySQL data contains commas and you import as CSV
   → Data is corrupted — field values split incorrectly
   → FIX: --enclosed-by '"' --escaped-by '\\'
   → OR: Use Parquet/Avro/ORC (binary formats — no delimiter issues)

5. DO NOT IMPORT BLOB/CLOB COLUMNS WITHOUT --inline-lob-limit
   → Large binary objects are written to separate files by default
   → Creates thousands of small files (NameNode pressure — Project 1)
   → FIX: --inline-lob-limit 16777216 (inline LOBs up to 16MB)
   → OR: Exclude BLOB columns: --columns "id,name,email,city"

6. DO NOT RE-IMPORT FULL TABLE DAILY FOR APPEND-ONLY DATA
   → If table only gets new rows (never updates)
   → Full import reads ALL existing rows again — wasteful
   → USE: --incremental append --check-column id --last-value N
   → Sqoop imports ONLY rows where id > N

7. DO NOT IMPORT TO AN EXISTING HDFS DIRECTORY
   → Sqoop fails if target directory already exists
   → Use --delete-target-dir for full reimports
   → Use --append for incremental imports to existing directory
   → Use --target-dir with date-stamped paths for snapshots

🚫 WHEN NOT TO USE

DO NOT USE SQOOP WHEN:

1. SOURCE IS NOT A RELATIONAL DATABASE
   → Sqoop works ONLY with JDBC-compatible databases
   → For log files → use Flume (Project 29)
   → For APIs/JSON feeds → use custom scripts or NiFi
   → For NoSQL (MongoDB, Cassandra) → use their native Hadoop connectors

2. YOU NEED REAL-TIME / STREAMING DATA
   → Sqoop is BATCH — runs as a MapReduce job
   → Minimum import time: minutes (even for small tables)
   → For sub-second data ingestion → use Kafka, Flume, Spark Streaming
   → For Change Data Capture (CDC) → use Debezium, Maxwell, or GoldenGate

3. DATA VOLUME IS TINY AND ONE-TIME
   → 1000-row lookup table imported once
   → Just export CSV and hdfs dfs -put (Project 3)
   → Sqoop overhead (MapReduce job startup) is overkill

4. YOU NEED COMPLEX TRANSFORMATIONS DURING IMPORT
   → Sqoop does minimal transformation (column selection, WHERE filter)
   → For complex ETL → import raw data first, then Pig/Hive transforms
   → Don't try to do everything in the --query parameter

5. SOURCE DATABASE CANNOT HANDLE PARALLEL READS
   → Sqoop opens N concurrent connections (N = num-mappers)
   → Small MySQL instance may crash under 8 parallel full-table scans
   → If DB is fragile → use -m 1 (single mapper) or mysqldump + put

6. SQOOP IS BEING DEPRECATED IN YOUR STACK
   → Sqoop 1 is in maintenance mode (no major new features)
   → Sqoop 2 was abandoned
   → Modern alternatives: Apache NiFi, Airbyte, Spark JDBC,
     cloud-native services (AWS DMS, GCP Dataflow)
   → Sqoop knowledge still valuable for existing pipelines

MECHANIC 1: HOW SQOOP IMPORT WORKS INTERNALLY
═══════════════════════════════════════════════════════════

Command: sqoop import --connect jdbc:mysql://db:3306/rideshare
         --table rides --split-by ride_id --num-mappers 4
         --target-dir /data/rides

STEP 1 — METADATA INSPECTION
   → Sqoop connects to MySQL via JDBC
   → Reads table schema: DESCRIBE rides
   → Gets column names, types, primary key
   → Determines split column boundaries:
     SELECT MIN(ride_id), MAX(ride_id) FROM rides
     → MIN = 1, MAX = 100,000,000

STEP 2 — CODE GENERATION
   → Sqoop generates a Java class for the table
   → Class has: fields matching MySQL columns, 
     serialization/deserialization methods,
     parse() and toString() methods
   → Compiled into a JAR on the fly
   → This JAR is used by mappers to read/write records

   // Auto-generated (simplified):
   public class rides implements DBWritable, Writable {
       private Long ride_id;
       private Long rider_id;
       private String city;
       private Double fare;
       
       public void readFields(ResultSet rs) {
           this.ride_id = rs.getLong("ride_id");
           this.rider_id = rs.getLong("rider_id");
           // ...
       }
       
       public String toString() {
           return ride_id + "," + rider_id + "," + city + "," + fare;
       }
   }

STEP 3 — SPLIT CALCULATION
   → Range: 1 to 100,000,000
   → 4 mappers → 4 equal splits:
     Split 0: ride_id BETWEEN 1 AND 25,000,000
     Split 1: ride_id BETWEEN 25,000,001 AND 50,000,000
     Split 2: ride_id BETWEEN 50,000,001 AND 75,000,000
     Split 3: ride_id BETWEEN 75,000,001 AND 100,000,000

STEP 4 — MAPREDUCE JOB EXECUTION
   → Sqoop submits a Map-only job (no reducers — like Project 11)
   → Each mapper:
     1. Opens its own JDBC connection to MySQL
     2. Executes: SELECT * FROM rides WHERE ride_id BETWEEN X AND Y
     3. Reads ResultSet row by row
     4. Converts each row using generated class
     5. Writes to HDFS output file

   ┌──────────────────────────────────────────────────────────┐
   │                    MapReduce Job                          │
   │                                                          │
   │  Mapper 0:                                               │
   │    JDBC → SELECT * FROM rides WHERE ride_id BETWEEN      │
   │           1 AND 25000000                                  │
   │    → Writes /data/rides/part-m-00000                     │
   │                                                          │
   │  Mapper 1:                                               │
   │    JDBC → SELECT * FROM rides WHERE ride_id BETWEEN      │
   │           25000001 AND 50000000                           │
   │    → Writes /data/rides/part-m-00001                     │
   │                                                          │
   │  Mapper 2:                                               │
   │    JDBC → SELECT * FROM rides WHERE ride_id BETWEEN      │
   │           50000001 AND 75000000                           │
   │    → Writes /data/rides/part-m-00002                     │
   │                                                          │
   │  Mapper 3:                                               │
   │    JDBC → SELECT * FROM rides WHERE ride_id BETWEEN      │

   │    → Writes /data/rides/part-m-00003                     │

═══════════════════════════════════════════════════════════
MECHANIC 2: HOW INCREMENTAL IMPORT WORKS
═══════════════════════════════════════════════════════════

APPEND MODE (for auto-increment IDs):

Day 1: Full import
   sqoop import ... --table rides --target-dir /data/rides
   → Imports all 100M rows
   → Sqoop notes: last imported ride_id = 100,000,000

Day 2: Incremental import
   sqoop import ... --table rides --target-dir /data/rides
         --incremental append
         --check-column ride_id
         --last-value 100000000
         --append
   
   Sqoop generates:
   SELECT * FROM rides WHERE ride_id > 100000000
   → Only reads new rows added since Day 1
   → Appends new part-m files to /data/rides/
   → Notes: last imported ride_id = 100,250,000

Day 3: Next incremental
   → --last-value 100250000
   → Only imports ride_id > 100,250,000

WHEN USING SQOOP JOBS:
   → Sqoop saves --last-value automatically in metastore
   → Next run picks up where previous left off
   → No manual tracking needed

LASTMODIFIED MODE (for mutable rows):

   sqoop import ... --table rider_profiles
         --incremental lastmodified
         --check-column updated_at
         --last-value "2024-03-22 00:00:00"
         --merge-key rider_id
   
   Sqoop generates:
   SELECT * FROM rider_profiles WHERE updated_at > '2024-03-22 00:00:00'
   
   --merge-key: If a rider_id already exists in HDFS,
                the new version REPLACES the old one
   → Requires an additional MapReduce merge step
   → More expensive but handles updates correctly

MECHANIC 3: HOW --hive-import WORKS
═══════════════════════════════════════════════════════════

   sqoop import --connect jdbc:mysql://db:3306/rideshare
         --table rides --hive-import --create-hive-table
         --hive-database rideshare_db
         --hive-table rides_from_mysql

STEP 1 — Normal HDFS import (same as above)
   → Writes to temporary HDFS directory

STEP 2 — Schema translation
   → MySQL INT → Hive INT
   → MySQL VARCHAR(255) → Hive STRING
   → MySQL DECIMAL(10,2) → Hive DOUBLE
   → MySQL DATETIME → Hive STRING (or TIMESTAMP)
   → MySQL BLOB → Hive BINARY

STEP 3 — Hive DDL generation and execution
   CREATE TABLE rideshare_db.rides_from_mysql (
       ride_id BIGINT,
       rider_id BIGINT,
       driver_id BIGINT,
       city STRING,
       fare DOUBLE,
       ...
   )
   ROW FORMAT DELIMITED FIELDS TERMINATED BY ','
   STORED AS TEXTFILE;

STEP 4 — Data move
   → Moves data from temporary dir to Hive warehouse location
   → /user/hive/warehouse/rideshare_db.db/rides_from_mysql/

STEP 5 — Hive table is immediately queryable
   → SELECT COUNT(*) FROM rideshare_db.rides_from_mysql;
   → Works immediately — data + schema both present

═══════════════════════════════════════════════════════════
MECHANIC 4: HOW --hbase-table WORKS
═══════════════════════════════════════════════════════════

   sqoop import --connect jdbc:mysql://db:3306/rideshare
         --table riders
         --hbase-table rider_profiles
         --column-family info
         --hbase-row-key rider_id

Each MySQL row becomes an HBase Put:
   MySQL row: (1001, 'Alice', 'alice@example.com', 'New_York')
   
   HBase Put:
     Row Key: "1001"
     info:rider_id = "1001"
     info:name = "Alice"
     info:email = "alice@example.com"
     info:city = "New_York"

ALL non-key columns go into the specified column family.
Row key comes from --hbase-row-key column.

MECHANIC 5: HOW SQOOP EXPORT WORKS (Hadoop → RDBMS)
═══════════════════════════════════════════════════════════

   sqoop export --connect jdbc:mysql://db:3306/rideshare
         --table daily_city_revenue
         --export-dir /data/pig_output/etl/daily_city_summary

STEP 1 — Read HDFS files
   → Each mapper reads one or more HDFS part files

STEP 2 — Parse records
   → Uses generated Java class to parse each line

STEP 3 — Batch INSERT into MySQL
   → Each mapper opens JDBC connection
   → Executes batch INSERT statements:
     INSERT INTO daily_city_revenue VALUES (?,?,?,?), (?,?,?,?), ...
   → Batch size controlled by --batch and --records-per-statement

STEP 4 — Commit
   → Each mapper commits its batch
   → NOTE: If one mapper fails, its data is NOT in MySQL
     but other mappers' data IS → PARTIAL EXPORT possible
   → Use --staging-table for atomic exports (all or nothing)

⚠️ COMMON PITFALLS AND FAILURE MODES

PITFALL 1: "Import fails with: No primary key could be found"
→ CAUSE: Table has no primary key — Sqoop can't determine split column
→ FIX 1: --split-by explicit_column (choose a numeric column)
→ FIX 2: -m 1 (use single mapper — slow but works)
→ FIX 3: Add a primary key to the MySQL table

PITFALL 2: "Data skew — one mapper takes 10× longer than others"
→ CAUSE: Split column has non-uniform distribution
→ EXAMPLE: ride_id has gap from 1000 to 10,000,000
   → Mapper 0: ride_id 1-2,500,000 (only 1000 rows — gap)
   → Mapper 1: ride_id 2,500,001-5,000,000 (zero rows!)
   → Mapper 2: ride_id 5,000,001-7,500,000 (zero rows!)
   → Mapper 3: ride_id 7,500,001-10,000,000 (all 9 million rows!)
→ FIX: --boundary-query "SELECT 1, 10000000 FROM rides"
   → Or use a more uniformly distributed column
   → Or use --split-by with string column: --split-by city
     (Sqoop 1.4.7+ supports string split with -Dorg.apache.sqoop
      .splitter.allow_text_splitter=true)

PITFALL 3: "Hive import creates table with wrong types"
→ CAUSE: MySQL → Hive type mapping is not always perfect
→ EXAMPLE: MySQL TINYINT(1) → Hive BOOLEAN (MySQL uses TINYINT for booleans)
   → But if your TINYINT stores actual numbers (0-255) → data loss
→ FIX: --map-column-hive "status=INT,flag=BOOLEAN"
   → Explicit type mapping overrides

PITFALL 4: "Import succeeds but Hive query returns NULLs for some columns"
→ CAUSE: MySQL NULLs imported as string "null" instead of Hive NULL
→ FIX: --null-string '\\N' --null-non-string '\\N'
→ Hive recognizes \N as NULL (its default serialization)

PITFALL 5: "Export fails with duplicate key errors"
→ CAUSE: Target MySQL table has unique constraint
→ Sqoop retries failed batches → some rows inserted twice
→ FIX 1: --update-mode allowinsert (uses INSERT ... ON DUPLICATE KEY UPDATE)
→ FIX 2: --update-key id --update-mode updateonly (UPDATE instead of INSERT)
→ FIX 3: Truncate target table before export

PITFALL 6: "Import takes forever — MySQL is slow"
→ CAUSE: Each mapper runs a SELECT with BETWEEN clause
→ If table has no index on split column → FULL TABLE SCAN per mapper
→ 4 mappers × full table scan = 4× the work on MySQL
→ FIX: Ensure split column is INDEXED in MySQL
→ FIX: Use --direct flag for MySQL (uses mysqldump native protocol)

PITFALL 7: "Free-form query import fails with $CONDITIONS error"
→ CAUSE: Using --query without $CONDITIONS placeholder
→ Sqoop needs to inject WHERE clause for each mapper's range
→ CORRECT: --query "SELECT * FROM rides WHERE city='NYC' AND $CONDITIONS"
→ WRONG:   --query "SELECT * FROM rides WHERE city='NYC'"
→ Also MUST specify --split-by when using --query

# PART I: PYTHON UTILITY — Sqoop Command Generator & Monitor

import subprocess
import time
import json
import os
from datetime import datetime


class SqoopManager:
    """
    Manages Sqoop import/export operations programmatically.
    Generates commands, tracks execution, validates results.
    """

    def __init__(self, jdbc_url, username, password):
        self.jdbc_url = jdbc_url
        self.username = username
        self.password = password
        self.import_log = []

    def _build_base_args(self):
        """Common connection arguments."""
        return [
            '--connect', self.jdbc_url,
            '--username', self.username,
            '--password', self.password
        ]

    def _run_command(self, command, args):
        """Execute a Sqoop command and return results."""
        full_cmd = ['sqoop', command] + args
        cmd_str = ' '.join(full_cmd).replace(self.password, '****')
        print(f"  [CMD] {cmd_str}")

        start_time = time.time()
        try:
            result = subprocess.run(
                full_cmd,
                capture_output=True,
                text=True,
                timeout=3600  # 1 hour timeout
            )
            elapsed = time.time() - start_time

            log_entry = {
                'command': command,
                'timestamp': datetime.now().isoformat(),
                'elapsed_seconds': round(elapsed, 2),
                'return_code': result.returncode,
                'success': result.returncode == 0
            }
            self.import_log.append(log_entry)

            if result.returncode == 0:
                print(f"  [OK] Completed in {elapsed:.1f}s")
            else:
                print(f"  [FAIL] Return code {result.returncode}")
                print(f"  [STDERR] {result.stderr[:500]}")

            return result
        except subprocess.TimeoutExpired:
            print(f"  [TIMEOUT] Command exceeded 1 hour")
            return None
        except FileNotFoundError:
            print(f"  [ERROR] Sqoop not found in PATH")
            return None

    def list_tables(self):
        """List all tables in the database."""
        args = self._build_base_args()
        result = self._run_command('list-tables', args)
        if result and result.returncode == 0:
            tables = [
                line.strip() for line in result.stdout.strip().split('\n')
                if line.strip() and not line.startswith('Warning')
                and not line.startswith('SLF4J')
            ]
            return tables
        return []

    def import_table(self, table_name, target_dir, split_by=None,
                     num_mappers=4, file_format='text',
                     columns=None, where=None,
                     hive_import=False, hive_database=None,
                     hive_table=None, incremental_mode=None,
                     check_column=None, last_value=None,
                     delete_target=True):
        """
        Generate and execute a Sqoop import command.

        Parameters match Sqoop flags directly.
        """
        print(f"\n[IMPORT] Table: {table_name}")
        print(f"  Target: {target_dir}")
        print(f"  Format: {file_format} | Mappers: {num_mappers}")

        args = self._build_base_args()
        args.extend(['--table', table_name])

        # Target directory
        if not hive_import:
            args.extend(['--target-dir', target_dir])

        # Split column
        if split_by:
            args.extend(['--split-by', split_by])

        # Mapper count
        args.extend(['--num-mappers', str(num_mappers)])

        # File format
        if file_format == 'parquet':
            args.append('--as-parquetfile')
        elif file_format == 'avro':
            args.append('--as-avrodatafile')
        elif file_format == 'sequencefile':
            args.append('--as-sequencefile')
        else:
            args.extend(['--fields-terminated-by', ','])

        # Null handling
        args.extend([
            '--null-string', '\\N',
            '--null-non-string', '\\N'
        ])

        # Column selection
        if columns:
            args.extend(['--columns', ','.join(columns)])

        # WHERE filter
        if where:
            args.extend(['--where', where])

        # Hive import
        if hive_import:
            args.append('--hive-import')
            args.append('--create-hive-table')
            if hive_database:
                args.extend(['--hive-database', hive_database])
            if hive_table:
                args.extend(['--hive-table', hive_table])

        # Incremental mode
        if incremental_mode:
            args.extend(['--incremental', incremental_mode])
            if check_column:
                args.extend(['--check-column', check_column])
            if last_value is not None:
                args.extend(['--last-value', str(last_value)])
            if incremental_mode == 'append':
                args.append('--append')
            elif incremental_mode == 'lastmodified':
                args.extend(['--merge-key', split_by or 'id'])

        # Delete target directory for full imports
        if delete_target and not hive_import and not incremental_mode:
            args.append('--delete-target-dir')

        return self._run_command('import', args)

    def export_table(self, table_name, export_dir,
                     num_mappers=2, update_key=None,
                     update_mode=None, staging_table=None):
        """Generate and execute a Sqoop export command."""
        print(f"\n[EXPORT] From: {export_dir}")
        print(f"  To MySQL table: {table_name}")

        args = self._build_base_args()
        args.extend([
            '--table', table_name,
            '--export-dir', export_dir,
            '--num-mappers', str(num_mappers),
            '--input-fields-terminated-by', ',',
            '--batch'
        ])

        if update_key:
            args.extend(['--update-key', update_key])
        if update_mode:
            args.extend(['--update-mode', update_mode])
        if staging_table:
            args.extend([
                '--staging-table', staging_table,
                '--clear-staging-table'
            ])

        return self._run_command('export', args)

    def create_saved_job(self, job_name, import_args):
        """Create a saved Sqoop job for recurring imports."""
        print(f"\n[JOB] Creating saved job: {job_name}")

        # Delete existing job
        subprocess.run(
            ['sqoop', 'job', '--delete', job_name],
            capture_output=True, text=True
        )

        full_cmd = ['sqoop', 'job', '--create', job_name, '--'] + import_args
        result = subprocess.run(
            full_cmd,
            capture_output=True, text=True
        )

        if result.returncode == 0:
            print(f"  [OK] Job '{job_name}' created")
        else:
            print(f"  [FAIL] {result.stderr[:300]}")

        return result

    def validate_import(self, table_name, hdfs_path):
        """Compare MySQL row count with HDFS row count."""
        print(f"\n[VALIDATE] {table_name}")

        # Get MySQL count
        mysql_cmd = (
            f"mysql -u {self.username} -p{self.password} -N -e "
            f"\"SELECT COUNT(*) FROM {table_name}\""
        )
        db_name = self.jdbc_url.split('/')[-1].split('?')[0]
        mysql_cmd = (
            f"mysql -u {self.username} -p{self.password} -N -e "
            f"\"SELECT COUNT(*) FROM {db_name}.{table_name}\""
        )

        try:
            mysql_result = subprocess.run(
                mysql_cmd, shell=True,
                capture_output=True, text=True, timeout=60
            )
            mysql_count = int(mysql_result.stdout.strip())
        except (ValueError, subprocess.TimeoutExpired):
            mysql_count = -1
            print("  [WARN] Could not get MySQL count")

        # Get HDFS count
        try:
            hdfs_result = subprocess.run(
                ['hdfs', 'dfs', '-cat', f'{hdfs_path}/part-m-*'],
                capture_output=True, text=True, timeout=120
            )
            hdfs_count = len([
                line for line in hdfs_result.stdout.strip().split('\n')
                if line.strip()
            ])
        except subprocess.TimeoutExpired:
            hdfs_count = -1
            print("  [WARN] Could not get HDFS count")

        print(f"  MySQL count: {mysql_count}")
        print(f"  HDFS count:  {hdfs_count}")

        if mysql_count == hdfs_count:
            print(f"  ✅ COUNTS MATCH")
            return True
        elif mysql_count >= 0 and hdfs_count >= 0:
            diff = abs(mysql_count - hdfs_count)
            pct = (diff / max(mysql_count, 1)) * 100
            print(f"  ⚠️ DIFFERENCE: {diff} rows ({pct:.2f}%)")
            return False
        else:
            print(f"  ❓ COULD NOT VALIDATE")
            return None

    def print_summary(self):
        """Print execution summary."""
        print("\n" + "=" * 70)
        print("SQOOP EXECUTION SUMMARY")
        print("=" * 70)

        total = len(self.import_log)
        success = sum(1 for e in self.import_log if e['success'])
        failed = total - success
        total_time = sum(e['elapsed_seconds'] for e in self.import_log)

        print(f"  Total operations: {total}")
        print(f"  Successful:       {success}")
        print(f"  Failed:           {failed}")
        print(f"  Total time:       {total_time:.1f}s")
        print()

        for entry in self.import_log:
            status = "✅" if entry['success'] else "❌"
            print(f"  {status} {entry['command']} | "
                  f"{entry['elapsed_seconds']}s | "
                  f"{entry['timestamp']}")

        print("=" * 70)
