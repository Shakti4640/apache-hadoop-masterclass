PROJECT 29: STREAMING LIVE WEB SERVER LOGS INTO HDFS

🔴 HYPOTHETICAL PROBLEM

SCENARIO:
→ You are a data engineer at the ride-sharing company
→ The data pipeline now handles:
   - MySQL data via Sqoop (Project 28) → HDFS/Hive/HBase
   - Batch processing via Pig (Project 26) and Hive (Projects 24-25)
   - Real-time serving via HBase (Project 27)

→ BUT there is a massive data source NOT yet captured:

SOURCE 1: Web Server Access Logs (200 servers)
   → Each server generates ~2.5GB of access logs per day
   → Total: 500GB/day of raw HTTP request logs
   → Format: Apache Combined Log Format
   → 192.168.1.1 - - [22/Mar/2024:14:30:22] "GET /api/ride/request HTTP/1.1" 200 1234

SOURCE 2: Application Event Logs (50 app servers)
   → JSON event streams: ride_requested, ride_started, ride_completed, payment_processed
   → Each server emits ~100 events/second
   → Total: 5000 events/second = 432 million events/day

SOURCE 3: Mobile SDK Telemetry (real-time)
   → App crash reports, GPS pings, UI interaction events
   → Sent to a centralized endpoint
   → Must be captured and stored for offline analysis

CURRENT BROKEN APPROACH:
→ Cron job runs every hour on each web server:
   scp /var/log/httpd/access.log hadoop-master:/staging/
→ Then another cron job:
   hdfs dfs -put /staging/*.log /data/logs/

WHAT GOES WRONG:
→ Server 47 crashed at 3:00 AM — logs from 2:00-3:00 LOST FOREVER
→ HDFS was down for maintenance 1:00-2:00 AM
   → 200 servers × 1 hour of logs queued on local disk
   → Staging directory overflows → logs dropped
→ Network glitch causes scp to fail silently
   → No retry mechanism → data gap
→ Duplicate uploads when cron runs twice
→ Thousands of small files in HDFS (one per server per hour)
   → NameNode memory pressure (Project 1 pitfall)
→ NO monitoring — nobody knows data is missing until analyst complains

REQUIREMENTS FOR A REAL SOLUTION:
→ RELIABLE: guaranteed delivery — no data loss even during failures
→ CONTINUOUS: stream data as it's generated, not hourly batches
→ SCALABLE: handle 200+ sources without manual configuration per server
→ BUFFERED: survive temporary HDFS downtime without losing data
→ SMART: roll files by size/time, avoid small files problem
→ MONITORED: know immediately when data flow stops

THIS IS EXACTLY WHAT APACHE FLUME DOES.

STEP 2: Understand Flume's core architecture

FLUME HAS ONE FUNDAMENTAL CONCEPT: THE AGENT

An AGENT has exactly THREE components:

   ┌──────────────────────────────────────────────────┐
   │                  FLUME AGENT                      │
   │                                                   │
   │  ┌──────────┐   ┌──────────┐   ┌──────────┐    │
   │  │  SOURCE   │──→│ CHANNEL  │──→│   SINK   │    │
   │  │          │   │          │   │          │     │
   │  │ (where   │   │ (buffer  │   │ (where   │     │
   │  │  data    │   │  data    │   │  data    │     │
   │  │  comes   │   │  between │   │  goes)   │     │
   │  │  from)   │   │  source  │   │          │     │
   │  │          │   │  and     │   │          │     │

   │  └──────────┘   └──────────┘   └──────────┘    │

   SOURCE: Receives or pulls data from an external system
   → Exec source: tails a log file (tail -F /var/log/access.log)
   → Spooling directory source: watches a directory for new files
   → HTTP source: listens on a port for HTTP POST events
   → Syslog source: receives syslog messages
   → Avro source: receives data from another Flume agent
   → Kafka source: reads from Kafka topics
   → Netcat source: listens on a TCP port (testing only)

   CHANNEL: Buffers data between source and sink
   → Memory channel: fast but data lost if agent crashes
   → File channel: slower but survives agent crash (durable)
   → Kafka channel: uses Kafka as the buffer (scalable, durable)

   SINK: Delivers data to the destination
   → HDFS sink: writes to HDFS (our primary use case)
   → HBase sink: writes to HBase (Project 27)
   → Avro sink: sends to another Flume agent (multi-hop)
   → Logger sink: writes to Flume log (testing only)
   → Kafka sink: writes to Kafka topic
   → Null sink: discards data (testing only)

💡 HOW CAN WE SOLVE IT

SOLUTION ARCHITECTURE: Multi-tier Flume deployment

TIER 1 — COLLECTION AGENTS (on each web/app server)
   → One Flume agent per server
   → Source: tail local log file
   → Channel: memory (local buffering)
   → Sink: Avro sink → sends to Tier 2 aggregator

TIER 2 — AGGREGATION AGENTS (on 2-3 dedicated nodes)
   → Receives data from many Tier 1 agents
   → Source: Avro source (listens for Tier 1 connections)
   → Channel: File channel (durable — survives crash)
   → Sink: HDFS sink (writes to HDFS)

WHY TWO TIERS:
→ Tier 1 agents are lightweight — minimal impact on web servers
→ Tier 2 agents handle HDFS writing complexity (file rolling, etc.)
→ If HDFS is slow/down, Tier 2 buffers in file channel
→ Tier 1 agents don't need to know about HDFS at all
→ Easy to add/remove web servers without reconfiguring HDFS sink

✅ WHAT SHOULD DO

1. USE FILE CHANNEL IN PRODUCTION — ALWAYS
   → Memory channel is faster but LOSES data on agent crash
   → File channel writes to local disk → survives crash
   → Performance difference: ~10-20% slower
   → Data safety: INFINITELY more reliable
   → RULE: Memory channel for development/testing ONLY

2. CONFIGURE HDFS SINK FILE ROLLING PROPERLY
   → HDFS sink creates new files based on:
     - hdfs.rollInterval: seconds before rolling (default 30)
     - hdfs.rollSize: bytes before rolling (default 1024)
     - hdfs.rollCount: events before rolling (default 10)
   → DEFAULT SETTINGS CREATE TINY FILES — ALWAYS OVERRIDE
   → Production settings:
     hdfs.rollInterval = 3600 (roll every hour)
     hdfs.rollSize = 134217728 (roll at 128MB)
     hdfs.rollCount = 0 (disable count-based rolling)
   → This prevents the small files problem (Project 1)

3. USE TIME-BASED HDFS DIRECTORY PATHS
   → hdfs.path = /data/logs/%Y/%m/%d/%H
   → Creates: /data/logs/2024/03/22/14/
   → Natural partitioning for Hive (Project 25)
   → Easy to create Hive external table pointing to these paths

4. USE BATCHING FOR HDFS WRITES
   → hdfs.batchSize = 1000 (write 1000 events per HDFS append)
   → Reduces number of HDFS operations
   → Balances latency vs. throughput

5. SET UP MULTIPLE SINKS FOR REDUNDANCY
   → Failover sink group: if primary HDFS sink fails, use backup
   → Load-balancing sink group: distribute across multiple HDFS paths

6. MONITOR CHANNEL CAPACITY
   → If channel fills up → source backs up → data loss possible
   → Set channel capacity large enough for peak throughput
   → Alert when channel is > 80% full

🚫 WHAT SHOULD NOT DO

1. DO NOT USE EXEC SOURCE WITH "cat" FOR LARGE FILES
   → exec source with "cat /var/log/huge.log" reads ENTIRE file every restart
   → Sends duplicate data each time agent restarts
   → USE: "tail -F /var/log/access.log" for continuous following
   → OR: Spooling Directory Source for completed log files

2. DO NOT USE MEMORY CHANNEL IN PRODUCTION
   → Agent crash = ALL buffered events LOST
   → JVM heap full = OutOfMemoryError = data loss
   → ONLY acceptable for non-critical, high-throughput scenarios
     where occasional data loss is tolerable

3. DO NOT USE DEFAULT HDFS ROLL SETTINGS
   → Default: roll every 30 seconds, every 1024 bytes, every 10 events
   → Creates THOUSANDS of tiny files per hour
   → 200 servers × 24 hours × 120 rolls/hour = 576,000 files/day
   → NameNode will be overwhelmed within days
   → ALWAYS set: rollInterval=3600, rollSize=128MB, rollCount=0

4. DO NOT FORGET hdfs.fileType AND COMPRESSION
   → Default: SequenceFile (includes overhead)
   → For raw logs: hdfs.fileType = DataStream (plain text)
   → For compression: hdfs.codeC = snappy or gzip
   → For structured data: hdfs.fileType = CompressedStream

5. DO NOT RUN FLUME AGENT AS ROOT
   → Security risk — if agent is compromised, attacker has root
   → Create dedicated "flume" user with minimal permissions
   → Only needs read access to log files and write to channel directory

6. DO NOT IGNORE INTERCEPTORS FOR DATA QUALITY
   → Interceptors modify events BEFORE they reach the channel
   → Timestamp interceptor: adds event timestamp (critical for time-based paths)
   → Host interceptor: adds source hostname to each event
   → Regex filtering: drop events matching patterns (health checks, etc.)
   → Static interceptor: add constant headers (datacenter, environment)
   → WITHOUT timestamp interceptor: %Y/%m/%d in HDFS path uses PROCESSING time
     not EVENT time → late-arriving events go to wrong directory

7. DO NOT CREATE ONE AGENT PER LOG FILE
   → Flume supports multiple sources, channels, and sinks per agent
   → Configure one agent with multiple source-channel-sink pipelines
   → Reduces operational overhead

🚫 WHEN NOT TO USE

DO NOT USE FLUME WHEN:

1. YOU NEED MESSAGE REPLAY / REPROCESSING
   → Flume delivers events once — no replay from source
   → If downstream processing fails, data may need to be re-ingested
   → Use Kafka for replayable event streams

2. YOU NEED PUB/SUB PATTERN (multiple consumers)
   → Flume sends to ONE sink (or sink group)
   → Cannot have multiple independent consumers reading same stream
   → Use Kafka for fan-out to multiple consumers

3. DATA IS IN A RELATIONAL DATABASE
   → Use Sqoop (Project 28) for structured RDBMS data
   → Flume is for log files and event streams

4. YOU NEED COMPLEX EVENT PROCESSING
   → Flume interceptors do simple transformations
   → For complex routing, filtering, enrichment:
     → Use Apache NiFi (visual dataflow)
     → Or Kafka Streams / Flink (stream processing)

5. YOU NEED EXACTLY-ONCE DELIVERY SEMANTICS
   → Flume guarantees AT-LEAST-ONCE with file channel
   → Duplicates possible during recovery after crash
   → If exact-once is required → use Kafka with transactions

6. YOUR DESTINATION IS NOT HADOOP
   → Flume is optimized for HDFS/HBase sinks
   → For Elasticsearch → use Logstash or Filebeat
   → For cloud storage → use cloud-native ingestion tools
   → Flume CAN write elsewhere but it's not its strength

⚙️ HOW IT ONLY WORKS — INTERNAL MECHANICS

═══════════════════════════════════════════════════════════
MECHANIC 1: THE FLUME EVENT — Fundamental Data Unit
═══════════════════════════════════════════════════════════

Every piece of data in Flume is an EVENT:

   ┌───────────────────────────────────────────┐
   │              FLUME EVENT                    │
   │                                            │
   │  Headers: {                                │
   │      "timestamp": "1711115422000",         │
   │      "host": "webserver-47",               │
   │      "category": "access_log"              │
   │  }                                         │
   │                                            │
   │  Body: (byte array)                        │
   │  "192.168.1.1 - - [22/Mar/2024:14:30:22]  │
   │   GET /api/ride/request HTTP/1.1 200 1234" │
   │                                            │
   └───────────────────────────────────────────┘

   HEADERS = metadata key-value pairs (String → String)
   → Added by interceptors (timestamp, host, static values)
   → Used by HDFS sink for path substitution (%{host}, %{category})
   → Used by channel selectors for routing

   BODY = raw data payload (byte[])
   → One log line, one JSON event, one syslog message
   → Flume does NOT parse the body — treats it as opaque bytes
   → Parsing happens downstream (Pig/Hive/MapReduce)

═══════════════════════════════════════════════════════════
MECHANIC 2: SOURCE → CHANNEL — The Put Transaction
═══════════════════════════════════════════════════════════

When a SOURCE receives data:

   STEP 1: Source creates Event objects from incoming data
           (e.g., each line from tail -F becomes one Event)

   STEP 2: Source begins a TRANSACTION on the channel
           channel.getTransaction().begin()

   STEP 3: Source puts events into the channel (batch)
           for each event:
               channel.put(event)

   STEP 4: Source COMMITS the transaction
           channel.getTransaction().commit()

   STEP 5: If commit fails → ROLLBACK
           channel.getTransaction().rollback()
           → Events are NOT in the channel
           → Source will retry

   THIS IS WHY FLUME IS RELIABLE:
   → Events are either ALL in the channel or NONE
   → No partial writes
   → Transaction ensures atomicity

═══════════════════════════════════════════════════════════
MECHANIC 3: CHANNEL → SINK — The Take Transaction
═══════════════════════════════════════════════════════════

When a SINK is ready to deliver:

   STEP 1: Sink begins a TRANSACTION on the channel
           channel.getTransaction().begin()

   STEP 2: Sink TAKES events from the channel (batch)
           for i in range(batchSize):
               event = channel.take()

   STEP 3: Sink writes events to destination (HDFS, HBase, etc.)

   STEP 4: If write succeeds → COMMIT
           channel.getTransaction().commit()
           → Events removed from channel permanently

   STEP 5: If write fails → ROLLBACK
           channel.getTransaction().rollback()
           → Events remain in channel for retry
           → NOTHING IS LOST

   END-TO-END GUARANTEE:
   Source Transaction: data enters channel reliably
   Sink Transaction: data leaves channel reliably
   → If agent crashes between source commit and sink commit:
     → File channel: events are still on disk → recovered on restart
     → Memory channel: events are LOST (in-memory only)

═══════════════════════════════════════════════════════════
MECHANIC 4: FILE CHANNEL — Durability on Disk
═══════════════════════════════════════════════════════════

File channel stores events in a WAL-like structure:

   /var/flume/file-channel/
   ├── log-1.dat     (event data — append-only log file)
   ├── log-2.dat     (next log file after rotation)
   ├── checkpoint/   (current channel state)
   │   ├── checkpoint.dat
   │   └── checkpoint.meta
   └── data/         (overflow storage)

   WRITE: Events appended to log-N.dat (sequential I/O — fast)
   READ: Events read from log and marked as taken
   COMMIT: Taken events marked as permanently removed
   CHECKPOINT: Periodically snapshot channel state to checkpoint/

   ON CRASH RECOVERY:
   → Agent restarts
   → File channel reads checkpoint
   → Replays log files from last checkpoint
   → Reconstructs exact channel state before crash
   → Un-committed events are available for re-delivery
   → ZERO DATA LOSS (assuming disk is intact)

═══════════════════════════════════════════════════════════
MECHANIC 5: HDFS SINK — File Management
═══════════════════════════════════════════════════════════

HDFS sink manages file lifecycle:

   STATE 1: OPEN
   → Sink opens a new file: /data/logs/2024/03/22/14/events.1711115422.tmp
   → ".tmp" suffix indicates file is being written
   → Events are appended to this file

   STATE 2: ROLLING
   → Trigger: rollInterval (time), rollSize (bytes), or rollCount (events)
   → Sink closes current file
   → Renames: events.1711115422.tmp → events.1711115422 (removes .tmp)
   → Opens new file for next batch

   STATE 3: IDLE TIMEOUT
   → If no events arrive for hdfs.idleTimeout seconds
   → Close current file (even if under rollSize)
   → Prevents stale .tmp files

   FILE NAMING:
   hdfs.path = /data/logs/%Y/%m/%d/%H
   hdfs.filePrefix = events
   hdfs.fileSuffix = .log
   hdfs.inUsePrefix = .
   hdfs.inUseSuffix = .tmp

   Active file:  /data/logs/2024/03/22/14/.events.1711115422.log.tmp
   Completed:    /data/logs/2024/03/22/14/events.1711115422.log

   TIME SUBSTITUTION IN PATH:
   → %Y = year (2024)
   → %m = month (03)
   → %d = day (22)
   → %H = hour (14)
   → Uses event HEADER timestamp (if timestamp interceptor is configured)
   → Falls back to system clock if no timestamp header

═══════════════════════════════════════════════════════════
MECHANIC 6: INTERCEPTORS — Event Modification Pipeline
═══════════════════════════════════════════════════════════

Interceptors modify events BETWEEN source and channel:

   Source → [Interceptor 1] → [Interceptor 2] → [Interceptor N] → Channel

   TIMESTAMP INTERCEPTOR:
   → Adds header: "timestamp" = current time in milliseconds
   → CRITICAL for time-based HDFS paths
   → Without this: %Y/%m/%d uses system clock at write time
     which may differ from event generation time

   HOST INTERCEPTOR:
   → Adds header: "host" = hostname of the machine
   → Useful for: hdfs.path = /data/logs/%{host}/%Y/%m/%d

   STATIC INTERCEPTOR:
   → Adds constant header: "datacenter" = "us-east-1"
   → Tag events with metadata not in the original log

   REGEX FILTERING INTERCEPTOR:
   → Drop events matching a pattern
   → Example: drop health check requests
     regex = "^.*GET /health.*$"
     excludeEvents = true

   REGEX EXTRACTOR INTERCEPTOR:
   → Extract fields from event body into headers
   → Example: extract HTTP status code from access log
     regex = "HTTP/1.[01]\" (\\d{3})"
     serializers = s1
     serializers.s1.name = status_code

   INTERCEPTORS ARE CHAINED:
   → Order matters — each interceptor sees output of previous
   → timestamp → host → regex_filter → static

⚠️ COMMON PITFALLS AND FAILURE MODES

PITFALL 1: "HDFS has millions of tiny files from Flume"
→ CAUSE: Default roll settings (30s interval, 1024 bytes, 10 events)
→ IMPACT: NameNode memory exhaustion (Project 1 — 150 bytes per file)
→ FIX:
   agent.sinks.hdfs_sink.hdfs.rollInterval = 3600
   agent.sinks.hdfs_sink.hdfs.rollSize = 134217728
   agent.sinks.hdfs_sink.hdfs.rollCount = 0
→ Creates ~1 file per hour per agent (manageable)

PITFALL 2: ".tmp files accumulate in HDFS — never closed"
→ CAUSE 1: Agent crashed while file was open
   → .tmp files from previous run are orphaned
   → Hive/MapReduce may skip .tmp files → data invisible
→ CAUSE 2: hdfs.idleTimeout not set
   → Low-volume source → file stays open for hours
→ FIX:
   agent.sinks.hdfs_sink.hdfs.idleTimeout = 300
   → Close file if no events for 5 minutes
→ RECOVERY: Manually rename orphaned .tmp files:
   hdfs dfs -mv /data/logs/path/.events.tmp /data/logs/path/events.recovered

PITFALL 3: "Events land in wrong time-based directory"
→ CAUSE: No timestamp interceptor configured
→ %Y/%m/%d/%H uses PROCESSING time (when sink writes to HDFS)
→ If events are buffered for 2 hours → land in wrong hour directory
→ FIX: ALWAYS add timestamp interceptor:
   agent.sources.src.interceptors = ts
   agent.sources.src.interceptors.ts.type = timestamp
→ Now %Y/%m/%d/%H uses EVENT time from header

PITFALL 4: "Agent starts but no data flows"
→ CAUSE 1: Source cannot read log file (permissions)
→ CAUSE 2: Channel directory doesn't exist or isn't writable
→ CAUSE 3: HDFS sink cannot connect to NameNode
→ CAUSE 4: Typo in configuration property name (silent failure)
→ DEBUG:
   → Check flume.log: tail -f $FLUME_HOME/logs/flume.log
   → Look for: "Source starting", "Channel starting", "Sink starting"
   → Look for: ERROR or WARN messages
   → Use logger sink first to verify source is producing events

PITFALL 5: "Exec source loses data on agent restart"
→ CAUSE: "tail -F" starts from END of file on restart
→ Events generated between crash and restart are LOST
→ FIX 1: Use Spooling Directory Source instead
   → Tracks which files have been processed
   → Resumes from where it left off
→ FIX 2: Use Taildir Source (Flume 1.7+)
   → Tracks file position in a JSON position file
   → Resumes from exact byte offset after restart

PITFALL 6: "Memory channel causes OutOfMemoryError"
→ CAUSE: Channel capacity too large for JVM heap
→ Default capacity: 100 events — usually fine
→ If set to 1,000,000 events with 1KB each = 1GB heap needed
→ FIX: Size channel capacity based on available heap
→ OR: Use file channel (disk-backed, no heap pressure)

PITFALL 7: "Duplicate events after agent crash recovery"
→ CAUSE: File channel provides at-least-once delivery
→ If sink wrote to HDFS but didn't commit channel transaction:
   → On restart, same events are taken and written AGAIN
→ IMPACT: Some duplicate lines in HDFS files
→ FIX: Downstream deduplication in Pig/Hive (acceptable trade-off)
→ This is BETTER than data loss (at-most-once)

SCALING STRATEGIES:

1. VERTICAL SCALING (per agent):
   → Increase channel capacity
   → Increase batch size (hdfs.batchSize)
   → Increase JVM heap (-Xmx)
   → Use multiple channels per agent

2. HORIZONTAL SCALING (multiple agents):
   → One collection agent per source server
   → Multiple aggregation agents behind load balancer
   → Each aggregation agent writes to different HDFS path

3. FAN-IN ARCHITECTURE:
   200 web servers → 4 aggregation agents → HDFS
   
   Each aggregation agent handles ~50 source agents
   Each writes to HDFS independently
   No coordination needed between aggregators

CAPACITY PLANNING:

   Events per second: 5000
   Average event size: 500 bytes
   Data rate: 5000 × 500 = 2.5 MB/sec = 216 GB/day

   File channel disk: 2.5 MB/sec × 600 sec buffer = 1.5 GB minimum
   Memory channel: 5000 events × 500 bytes = 2.5 MB minimum heap
   HDFS files: 216 GB/day ÷ 128 MB per file = ~1,700 files/day
   NameNode impact: 1,700 × 150 bytes = 255 KB/day (negligible)

import os
import subprocess
import time
import json
from datetime import datetime


class FlumeConfigGenerator:
    """
    Generates Flume agent configuration files from structured parameters.
    Eliminates configuration errors from manual editing.
    """

    def __init__(self, agent_name):
        self.agent_name = agent_name
        self.sources = {}
        self.channels = {}
        self.sinks = {}
        self.wiring = {}

    def add_exec_source(self, name, command, batch_size=100,
                        interceptors=None):
        """Add an exec source (tail -F, etc.)."""
        self.sources[name] = {
            'type': 'exec',
            'command': command,
            'batchSize': batch_size,
        }
        if interceptors:
            self.sources[name]['interceptors'] = interceptors

    def add_spooldir_source(self, name, spool_dir, batch_size=500,
                            delete_policy='immediate', interceptors=None):
        """Add a spooling directory source."""
        self.sources[name] = {
            'type': 'spooldir',
            'spoolDir': spool_dir,
            'batchSize': batch_size,
            'deletePolicy': delete_policy,
            'fileHeader': 'true',
            'fileHeaderKey': 'source_file',
        }
        if interceptors:
            self.sources[name]['interceptors'] = interceptors

    def add_avro_source(self, name, bind='0.0.0.0', port=4141, threads=8):
        """Add an Avro source (for multi-tier)."""
        self.sources[name] = {
            'type': 'avro',
            'bind': bind,
            'port': port,
            'threads': threads,
        }

    def add_http_source(self, name, port=8888, interceptors=None):
        """Add an HTTP source."""
        self.sources[name] = {
            'type': 'http',
            'port': port,
            'handler': 'org.apache.flume.source.http.JSONHandler',
        }
        if interceptors:
            self.sources[name]['interceptors'] = interceptors

    def add_memory_channel(self, name, capacity=10000,
                           transaction_capacity=1000):
        """Add a memory channel."""
        self.channels[name] = {
            'type': 'memory',
            'capacity': capacity,
            'transactionCapacity': transaction_capacity,
        }

    def add_file_channel(self, name, checkpoint_dir, data_dir,
                         capacity=10000000, transaction_capacity=10000):
        """Add a file channel (durable)."""
        self.channels[name] = {
            'type': 'file',
            'checkpointDir': checkpoint_dir,
            'dataDirs': data_dir,
            'capacity': capacity,
            'transactionCapacity': transaction_capacity,
        }

    def add_hdfs_sink(self, name, hdfs_path, file_prefix='events',
                      file_suffix='.log', file_type='DataStream',
                      roll_interval=3600, roll_size=134217728,
                      roll_count=0, idle_timeout=300,
                      batch_size=1000, codec=None):
        """Add an HDFS sink with production-safe defaults."""
        self.sinks[name] = {
            'type': 'hdfs',
            'hdfs.path': hdfs_path,
            'hdfs.filePrefix': file_prefix,
            'hdfs.fileSuffix': file_suffix,
            'hdfs.fileType': file_type,
            'hdfs.rollInterval': roll_interval,
            'hdfs.rollSize': roll_size,
            'hdfs.rollCount': roll_count,
            'hdfs.idleTimeout': idle_timeout,
            'hdfs.batchSize': batch_size,
            'hdfs.useLocalTimeStamp': 'false',
            'hdfs.round': 'true',
            'hdfs.roundValue': 60,
            'hdfs.roundUnit': 'minute',
        }
        if codec:
            self.sinks[name]['hdfs.codeC'] = codec
            self.sinks[name]['hdfs.fileType'] = 'CompressedStream'

    def add_avro_sink(self, name, hostname, port=4141, batch_size=100):
        """Add an Avro sink (for multi-tier)."""
        self.sinks[name] = {
            'type': 'avro',
            'hostname': hostname,
            'port': port,
            'batch-size': batch_size,
        }

    def add_logger_sink(self, name):
        """Add a logger sink (debugging only)."""
        self.sinks[name] = {'type': 'logger'}

    def wire(self, source_name, channel_name, sink_name):
        """Connect source → channel → sink."""
        self.wiring[source_name] = {
            'channel': channel_name,
            'sink': sink_name,
        }

    def _build_interceptor_config(self, source_name, interceptors):
        """Generate interceptor configuration lines."""
        lines = []
        agent = self.agent_name
        src = source_name

        names = []
        for inter in interceptors:
            names.append(inter['name'])

        lines.append(
            f"{agent}.sources.{src}.interceptors = {' '.join(names)}")

        for inter in interceptors:
            prefix = f"{agent}.sources.{src}.interceptors.{inter['name']}"
            lines.append(f"{prefix}.type = {inter['type']}")
            for key, value in inter.get('params', {}).items():
                lines.append(f"{prefix}.{key} = {value}")

        return lines

    def generate(self):
        """Generate the complete configuration file content."""
        agent = self.agent_name
        lines = []

        lines.append(f"# Auto-generated Flume configuration")
        lines.append(f"# Agent: {agent}")
        lines.append(f"# Generated: {datetime.now().isoformat()}")
        lines.append("")

        # Component names
        source_names = ' '.join(self.sources.keys())
        channel_names = ' '.join(self.channels.keys())
        sink_names = ' '.join(self.sinks.keys())

        lines.append(f"{agent}.sources = {source_names}")
        lines.append(f"{agent}.channels = {channel_names}")
        lines.append(f"{agent}.sinks = {sink_names}")
        lines.append("")

        # Sources
        for name, config in self.sources.items():
            lines.append(f"# Source: {name}")
            interceptors = config.pop('interceptors', None)
            for key, value in config.items():
                lines.append(f"{agent}.sources.{name}.{key} = {value}")
            if interceptors:
                lines.extend(
                    self._build_interceptor_config(name, interceptors))
            lines.append("")

        # Channels
        for name, config in self.channels.items():
            lines.append(f"# Channel: {name}")
            for key, value in config.items():
                lines.append(f"{agent}.channels.{name}.{key} = {value}")
            lines.append("")

        # Sinks
        for name, config in self.sinks.items():
            lines.append(f"# Sink: {name}")
            for key, value in config.items():
                lines.append(f"{agent}.sinks.{name}.{key} = {value}")
            lines.append("")

        # Wiring
        lines.append("# Wiring: source → channel → sink")
        for src_name, wire in self.wiring.items():
            ch = wire['channel']
            snk = wire['sink']
            lines.append(f"{agent}.sources.{src_name}.channels = {ch}")
            lines.append(f"{agent}.sinks.{snk}.channel = {ch}")
        lines.append("")

        return '\n'.join(lines)

    def save(self, filepath):
        """Save configuration to file."""
        content = self.generate()
        with open(filepath, 'w') as f:
            f.write(content)
        print(f"Configuration saved to: {filepath}")
        return filepath


class FlumeMonitor:
    """
    Monitors Flume agents and HDFS output for health checks.
    """

    def __init__(self):
        self.checks = []

    def check_agent_running(self, agent_name):
        """Check if a Flume agent process is running."""
        try:
            result = subprocess.run(
                ['pgrep', '-f', f'--name {agent_name}'],
                capture_output=True, text=True, timeout=5
            )
            running = result.returncode == 0
            self.checks.append({
                'check': f'Agent {agent_name} running',
                'status': '✅' if running else '❌',
                'detail': f'PID: {result.stdout.strip()}' if running
                          else 'Not found'
            })
            return running
        except (subprocess.TimeoutExpired, FileNotFoundError):
            self.checks.append({
                'check': f'Agent {agent_name} running',
                'status': '❓',
                'detail': 'Could not check'
            })
            return None

    def check_hdfs_output(self, hdfs_path, max_age_minutes=60):
        """Check if HDFS path has recent data."""
        try:
            result = subprocess.run(
                ['hdfs', 'dfs', '-ls', '-R', hdfs_path],
                capture_output=True, text=True, timeout=30
            )
            if result.returncode != 0:
                self.checks.append({
                    'check': f'HDFS path {hdfs_path}',
                    'status': '❌',
                    'detail': 'Path does not exist'
                })
                return False

            lines = [l for l in result.stdout.strip().split('\n')
                     if l.strip() and not l.startswith('Found')
                     and not l.strip().startswith('d')]
            file_count = len(lines)
            tmp_count = sum(1 for l in lines if '.tmp' in l)

            self.checks.append({
                'check': f'HDFS path {hdfs_path}',
                'status': '✅' if file_count > 0 else '⚠️',
                'detail': f'{file_count} files ({tmp_count} active .tmp)'
            })
            return file_count > 0

        except (subprocess.TimeoutExpired, FileNotFoundError):
            self.checks.append({
                'check': f'HDFS path {hdfs_path}',
                'status': '❓',
                'detail': 'Could not check HDFS'
            })
            return None

    def check_tmp_files(self, hdfs_path):
        """Check for orphaned .tmp files (indicates agent crash)."""
        try:
            result = subprocess.run(
                ['hdfs', 'dfs', '-ls', '-R', hdfs_path],
                capture_output=True, text=True, timeout=30
            )
            if result.returncode != 0:
                return 0

            tmp_files = [l for l in result.stdout.strip().split('\n')
                         if '.tmp' in l and not l.startswith('Found')]

            if len(tmp_files) > 5:
                self.checks.append({
                    'check': f'Orphaned .tmp files in {hdfs_path}',
                    'status': '⚠️',
                    'detail': f'{len(tmp_files)} .tmp files — '
                              f'possible agent crash'
                })
            else:
                self.checks.append({
                    'check': f'Orphaned .tmp files in {hdfs_path}',
                    'status': '✅',
                    'detail': f'{len(tmp_files)} active .tmp files (normal)'
                })

            return len(tmp_files)
        except (subprocess.TimeoutExpired, FileNotFoundError):
            return -1

    def check_file_sizes(self, hdfs_path, min_size_mb=1, max_size_mb=256):
        """Check if HDFS files are within expected size range."""
        try:
            result = subprocess.run(
                ['hdfs', 'dfs', '-ls', '-R', hdfs_path],
                capture_output=True, text=True, timeout=30
            )
            if result.returncode != 0:
                return

            tiny_files = 0
            normal_files = 0
            huge_files = 0

            for line in result.stdout.strip().split('\n'):
                parts = line.split()
                if len(parts) >= 5 and not line.startswith('Found'):
                    try:
                        if parts[0].startswith('d'):
                            continue
                        size_bytes = int(parts[4])
                        size_mb = size_bytes / (1024 * 1024)
                        if size_mb < min_size_mb:
                            tiny_files += 1
                        elif size_mb > max_size_mb:
                            huge_files += 1
                        else:
                            normal_files += 1
                    except (ValueError, IndexError):
                        pass

            status = '✅' if tiny_files < 5 else '⚠️'
            self.checks.append({
                'check': f'File sizes in {hdfs_path}',
                'status': status,
                'detail': f'Normal: {normal_files} | '
                          f'Tiny(<{min_size_mb}MB): {tiny_files} | '
                          f'Huge(>{max_size_mb}MB): {huge_files}'
            })

        except (subprocess.TimeoutExpired, FileNotFoundError):
            pass

    def print_report(self):
        """Print health check report."""
        print("\n" + "=" * 70)
        print("FLUME HEALTH CHECK REPORT")
        print(f"Time: {datetime.now().isoformat()}")
        print("=" * 70)

        for check in self.checks:
            print(f"  {check['status']} {check['check']}")
            print(f"     → {check['detail']}")
            print()

        # Summary
        ok = sum(1 for c in self.checks if c['status'] == '✅')
        warn = sum(1 for c in self.checks if c['status'] == '⚠️')
        fail = sum(1 for c in self.checks if c['status'] == '❌')
        unknown = sum(1 for c in self.checks if c['status'] == '❓')

        print(f"  Summary: {ok} OK | {warn} Warnings | "
              f"{fail} Failed | {unknown} Unknown")
        print("=" * 70)
