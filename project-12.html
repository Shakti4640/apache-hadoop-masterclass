PROJECT 12: PROCESSING CSV, TSV, AND JSON — NOT EVERYTHING IS PLAIN TEXT

🔴 HYPOTHETICAL PROBLEM

SCENARIO:
→ You are a data engineer at a multinational insurance company
→ Three different systems feed data into your Hadoop cluster:

SYSTEM A — Claims Department (CSV with quoted fields):
  "CLM-001","John O'Brien","2024-01-15","Accident, car collision","15,250.00"
  "CLM-002","María García","2024-01-16","Flood damage, roof","8,900.50"
  
  PROBLEM: Commas INSIDE quoted fields
  → Naive split(",") breaks "Accident, car collision" into two fields
  → Amount "15,250.00" becomes "15" and "250.00"

SYSTEM B — Policy Administration (TSV):
  POL-001\tJane Smith\tPremium Plan\t2024-01-01\t2025-01-01\t1200.00
  POL-002\tBob Lee\tBasic Plan\t2024-02-01\t2025-02-01\t600.00
  
  PROBLEM: Tab-separated but some description fields contain tabs
  → "Premium\tPlan" in a description breaks field alignment

SYSTEM C — Agent Reports (JSON, one object per line):
  {"agent_id":"AGT-001","name":"Alice Wu","region":"NORTH","sales":45,"revenue":67500.00}
  {"agent_id":"AGT-002","name":"Tom Ray","region":"SOUTH","sales":32,"revenue":48000.00}
  
  PROBLEM: Default TextInputFormat treats entire JSON as a single string
  → You must PARSE JSON inside the mapper
  → No built-in Hadoop JSON InputFormat

CURRENT APPROACH (BROKEN):
→ All three files processed with default TextInputFormat
→ Mapper splits each line by comma → WRONG for CSV with quotes
→ Mapper splits by tab → WRONG for TSV with embedded tabs
→ Mapper gets raw JSON string → needs manual parsing

THE QUESTION:
→ How does Hadoop ACTUALLY read files and create records?
→ How do we handle formats that don't fit the "one line = one record" assumption?
→ What InputFormat options exist and when to use each?

🔍 HOW TO APPROACH IT

STEP 1: Understand what InputFormat ACTUALLY does

In Projects 4-11, we never questioned HOW the mapper receives data.
We just wrote: map(LongWritable key, Text value, Context ctx)
and assumed each line appears as a separate call.

BUT WHO DECIDES:
  → Where to split the file into chunks? (InputSplits)
  → Where one record ends and the next begins? (RecordReader)
  → What the KEY and VALUE types are?

ANSWER: The InputFormat class does ALL of this.

STEP 2: Understand the DEFAULT — TextInputFormat

Every job in Projects 4-11 used TextInputFormat implicitly:

  KEY:   LongWritable = byte offset of line start in file
  VALUE: Text = the entire line (up to newline character)

  Line 1 starts at byte 0:    key=0,   value="first line content"
  Line 2 starts at byte 45:   key=45,  value="second line content"
  Line 3 starts at byte 92:   key=92,  value="third line content"

  RecordReader: LineRecordReader
    → Reads until \n or \r\n
    → Handles lines that SPAN block boundaries
      (explained in Project 1 internal mechanics)
    → ONE line = ONE record = ONE mapper call

  THIS WORKS WHEN:
    → Each line is one complete record
    → No field contains newline characters
    → Delimiter is NOT the same character used within field values

  THIS BREAKS WHEN:
    → CSV with quoted fields containing commas
    → JSON with nested objects spanning lines
    → Multi-line records (XML elements)
    → Binary data (no line concept)

💡 HOW CAN WE SOLVE IT

HADOOP PROVIDES MULTIPLE BUILT-IN InputFormat OPTIONS:

FOR OUR THREE SYSTEMS:

System A (CSV with quotes):
  → TextInputFormat for line reading
  → CUSTOM CSV PARSER in mapper (not split by comma — use proper parser)
  → OpenCSV library or manual state-machine parser

System B (TSV):
  → KeyValueTextInputFormat (splits on first TAB)
  → OR TextInputFormat + manual TAB split in mapper
  → KeyValueTextInputFormat only splits on FIRST delimiter
    → Good for: key\tvalue format
    → Bad for: field1\tfield2\tfield3\t... (need all fields)

System C (JSON):
  → TextInputFormat (each line is complete JSON object)
  → Parse JSON in mapper using Jackson/Gson library
  → OR use third-party JsonInputFormat (not built into Hadoop)

SOLUTION STRUCTURE:
  → One job per input format (or MultipleInputs in one job — Project 17)
  → Each job uses appropriate InputFormat + parsing strategy
  → All jobs output in SAME standardized format for downstream

✅ WHAT COULD BE DONE

APPROACH A: TextInputFormat + Smart parsing in Mapper (OUR CHOICE)
→ Use default TextInputFormat for all three formats
→ Write format-specific parsing logic inside mapper
→ CSV: proper quote-aware parser (not naive comma split)
→ TSV: split by tab with field count validation
→ JSON: parse with lightweight JSON library
→ Most practical — works for 95% of real-world cases

APPROACH B: Different InputFormat per source
→ System A: TextInputFormat + CSV parser
→ System B: KeyValueTextInputFormat

→ Write CSVInputFormat, TSVInputFormat, JsonInputFormat
→ Maximum control but massive development effort
→ Justified only for very complex formats

→ Convert everything to pipe-delimited on ingest

→ Practical but adds an extra processing step

WE CHOOSE APPROACH A because:

→ Shows how to handle multiple formats

✅ WHAT SHOULD DO

1. USE A PROPER CSV PARSER — NEVER SPLIT BY COMMA
   → line.split(",") FAILS on: "value1, with comma","value2"
   → Use a state machine: track whether inside quotes or not
   → Or use OpenCSV library (add to job classpath)
   → This is the #1 mistake in production Hadoop jobs

2. UNDERSTAND KeyValueTextInputFormat BEFORE USING IT
   → Splits line on FIRST occurrence of separator (default: TAB)
   → Everything BEFORE first TAB = key
   → Everything AFTER first TAB = value
   → If line has no TAB: entire line = key, value = empty
   → Configurable separator: conf.set("mapreduce.input.keyvaluelinerecordreader.key.value.separator", ",")

3. USE NLineInputFormat WHEN PROCESSING MUST BE PER-N-LINES
   → Normal: split by block size (128MB) → variable lines per mapper
   → NLineInputFormat: exactly N lines per split
   → Useful when: each mapper must process exactly N records
   → Set N: conf.setInt("mapreduce.input.lineinputformat.linespermap", 100)

4. ADD PARSING LIBRARIES TO HADOOP JOB CLASSPATH
   → JSON parsing: add jackson-core, jackson-databind JARs
   → CSV parsing: add opencsv JAR
   → Two methods:
     a) Include in JAR: jar -cvf job.jar -C classes . lib/*.jar
     b) Distributed cache: job.addFileToClassPath(new Path("/libs/jackson.jar"))
   → Method (a) is simpler for small libraries

5. VALIDATE AFTER PARSING — NOT JUST BEFORE
   → Parsing may succeed but produce wrong values
   → CSV parser returns 5 fields but you expect 6 → parser ate a delimiter
   → Always check field count AFTER parsing
   → Apply Project 11 validation patterns on parsed fields

6. HANDLE CHARACTER ENCODING EXPLICITLY
   → UTF-8 is assumed by TextInputFormat
   → If source system sends Latin-1 or Windows-1252:
     → Special characters corrupt: José → JosÃ©
   → Convert encoding before uploading to HDFS
   → Or handle in mapper: new String(bytes, "ISO-8859-1")

🚫 WHAT SHOULD NOT DO

1. DO NOT USE line.split(",") FOR CSV
   → Breaks on: "Smith, Jr.",John,45
   → split(",") → ["\"Smith", " Jr.\"", "John", "45"]
   → Correct parse: ["Smith, Jr.", "John", "45"]
   → This bug silently corrupts data — no error thrown

2. DO NOT ASSUME ONE LINE = ONE RECORD FOR ALL FORMATS
   → JSON arrays may span multiple lines:
     {
       "name": "John",
       "address": "123 Main St"
     }
   → TextInputFormat would feed 4 separate calls to mapper
   → Each call gets a fragment, not a complete record
   → For multi-line JSON: use custom RecordReader (Project 32)
   → For single-line JSON (JSON Lines format): TextInputFormat works

3. DO NOT PARSE IN THE REDUCER WHAT SHOULD BE PARSED IN THE MAPPER
   → Parse raw format in mapper → emit structured key-value
   → Reducer should receive CLEAN, TYPED data
   → If reducer needs to re-parse, you're doing double work

4. DO NOT IGNORE HEADER ROWS
   → CSV files often start with column headers
   → First line of first split might be a header
   → TextInputFormat has no built-in header skip
   → Must check in mapper: skip if line starts with column names
   → Same pattern as Project 11 (REJECTED_HEADER_ROW)

5. DO NOT CHANGE InputFormat WITHOUT UNDERSTANDING KEY TYPE
   → TextInputFormat: key = LongWritable (byte offset)
   → KeyValueTextInputFormat: key = Text (part before separator)
   → Mapper signature MUST match InputFormat's key-value types
   → Mismatch → ClassCastException at runtime

6. DO NOT USE KeyValueTextInputFormat FOR MULTI-FIELD RECORDS
   → It only splits into TWO parts: key and value
   → "field1\tfield2\tfield3" → key="field1", value="field2\tfield3"
   → You still need to split value by tab in mapper
   → Rarely useful for typical data processing

🚫 WHEN NOT TO USE

1. KeyValueTextInputFormat — WHEN NOT TO USE
   → Multi-field records (more than key + value)
   → When separator character appears in data
   → When you need the byte offset as key (use TextInputFormat)

2. NLineInputFormat — WHEN NOT TO USE
   → When records are single-line (standard TextInputFormat suffices)
   → When N is very small (1 line per split = millions of splits)
   → When file has variable-length records that don't align to N lines

3. TextInputFormat — WHEN NOT TO USE
   → Binary files (images, compressed blocks, serialized objects)
   → Multi-line records (JSON objects, XML elements)
   → Files without newline characters as record boundaries

4. ANY BUILT-IN InputFormat — WHEN NOT TO USE
   → Proprietary binary formats
   → Database-specific dump formats
   → Records delimited by custom byte sequences
   → These require Custom InputFormat (Project 32)

⚙️ HOW IT ONLY WORKS — INTERNAL MECHANICS

═══════════════════════════════════════════════════════════
MECHANIC 1: InputFormat → InputSplit → RecordReader LIFECYCLE
═══════════════════════════════════════════════════════════

PHASE 1: JOB SUBMISSION (runs on CLIENT machine)

  JobClient calls InputFormat.getSplits(jobContext):
  
  For TextInputFormat:
  ┌──────────────────────────────────────────────────────────┐
  │  1. List all input files from FileInputFormat.addInputPath│
  │  2. For each file:                                        │
  │     a. Get file size from NameNode                        │
  │     b. Get block locations from NameNode                  │
  │     c. Calculate split size:                              │
  │        splitSize = max(minSize, min(maxSize, blockSize))  │
  │        Default: max(1, min(Long.MAX, 128MB)) = 128MB     │
  │     d. Create FileSplit objects:                           │
  │        FileSplit(path, start=0, length=128MB, hosts)      │
  │        FileSplit(path, start=128MB, length=128MB, hosts)  │
  │        ...                                                │
  │  3. Return list of all splits                             │
  │  4. Number of splits = number of mapper tasks             │
  └──────────────────────────────────────────────────────────┘

PHASE 2: TASK EXECUTION (runs on each DataNode)

  For each split, Hadoop:
  1. Assigns split to a mapper task (preferably on node with data)
  2. Calls InputFormat.createRecordReader(split, context)
  3. RecordReader opens the file at split's start offset
  4. Mapper framework calls RecordReader.nextKeyValue() in a loop:
  
  ┌──────────────────────────────────────────────────────────┐
  │  while (recordReader.nextKeyValue()) {                    │
  │      KEY key = recordReader.getCurrentKey();              │
  │      VALUE value = recordReader.getCurrentValue();        │
  │      mapper.map(key, value, context);                     │
  │  }                                                        │
  └──────────────────────────────────────────────────────────┘

═══════════════════════════════════════════════════════════
MECHANIC 2: HOW TextInputFormat HANDLES BLOCK BOUNDARIES
═══════════════════════════════════════════════════════════

PROBLEM: A 300MB file split into blocks:
  Block 1: bytes 0 - 128MB
  Block 2: bytes 128MB - 256MB
  Block 3: bytes 256MB - 300MB

But a text line might SPAN the boundary:
  Byte 128MB-5: "...end of line A\nstart of line"
  Byte 128MB+10: " B which continues here\nline C..."
                  ↑
                  Block boundary at exactly byte 128MB
                  Line B spans blocks 1 and 2!

SOLUTION: LineRecordReader does two things:

For Split 1 (bytes 0 to 128MB):
  → Reads from byte 0
  → When reaching byte 128MB, CONTINUES reading past boundary
  → Finishes the current line (reads into Block 2's territory)
  → Returns the complete line

For Split 2 (bytes 128MB to 256MB):
  → Starts at byte 128MB
  → SKIPS forward to the NEXT newline character
  → Begins reading from the first COMPLETE line
  → This avoids double-processing the spanning line

  ┌──────────────────────────────────────────────────────┐
  │  BLOCK 1                    │  BLOCK 2               │
  │  ...lineA\nlineB_part1      │  lineB_part2\nlineC... │
  │                              │                        │
  │  Split 1 reads:             │  Split 2 reads:        │
  │  lineA                      │  lineC                 │
  │  lineB (reads past boundary)│  (skips lineB_part2)   │
  │                              │                        │
  │  RESULT: lineB processed    │                        │
  │  EXACTLY ONCE by Split 1    │                        │
  └──────────────────────────────────────────────────────┘

  EXCEPTION: Split 1 does NOT skip to first newline
  because it starts at byte 0 (beginning of file).
  Only splits starting mid-file skip partial first line.

═══════════════════════════════════════════════════════════
MECHANIC 3: KeyValueTextInputFormat INTERNALS
═══════════════════════════════════════════════════════════

Uses KeyValueLineRecordReader:
  1. Reads a line (same as TextInputFormat)
  2. Finds FIRST occurrence of separator (default: TAB)
  3. Everything before separator → KEY (Text)
  4. Everything after separator → VALUE (Text)

  Line: "user123\tJohn Smith\t45\tNew York"
  
  Key:   "user123"
  Value: "John Smith\t45\tNew York"
  
  Note: Only splits on FIRST tab!
  The remaining tabs stay in the value.
  
  If no separator found:
  Key:   entire line
  Value: empty Text("")

  Configuration:
  conf.set("mapreduce.input.keyvaluelinerecordreader.key.value.separator", ",")
  → Changes separator from TAB to comma

═══════════════════════════════════════════════════════════
MECHANIC 4: NLineInputFormat INTERNALS
═══════════════════════════════════════════════════════════

Standard TextInputFormat: split by BYTES (128MB blocks)
  → Each mapper gets variable number of lines
  → Mapper 1 might get 1 million lines
  → Mapper 2 might get 1.1 million lines (different line lengths)

NLineInputFormat: split by LINE COUNT
  → Each mapper gets exactly N lines
  → Configuration: mapreduce.input.lineinputformat.linespermap = N

  Example: File has 10,000 lines, N = 1000
  → 10 splits created → 10 mapper tasks
  → Each mapper processes exactly 1000 lines

  HOW IT CREATES SPLITS:
  1. Opens file and counts newline characters
  2. Every N newlines → new split boundary
  3. Split boundaries are at byte offsets of the Nth newline
  4. Uses LineRecordReader internally (same as TextInputFormat)

  USE CASE:
  → When each record requires EQUAL processing time
  → When you need deterministic splits for testing
  → When multi-line records need exactly N lines per record
     (e.g., every 3 lines = 1 record → set N=3)

═══════════════════════════════════════════════════════════
MECHANIC 5: WHY InputFormat CHOICE AFFECTS PERFORMANCE
═══════════════════════════════════════════════════════════

  TextInputFormat:
  → getSplits() is FAST (reads only block metadata from NameNode)
  → No file content read during split planning
  → O(1) per file for split computation

  NLineInputFormat:
  → getSplits() is SLOW (must read ENTIRE file to count lines)
  → Reads all bytes to find newline positions
  → O(file_size) for split computation
  → For 50GB file: reads 50GB just to plan splits
  → Fine for small files, problematic for huge files

  KeyValueTextInputFormat:
  → getSplits() same as TextInputFormat (block-based)
  → Parsing overhead in RecordReader: find separator per line
  → Negligible performance difference from TextInputFormat

⚠️ COMMON PITFALLS AND FAILURE MODES

PITFALL 1: "CSV parsing produces wrong number of fields"
→ CAUSE: Using split(",") instead of proper CSV parser
→ INPUT: "Smith, Jr.",John,45
→ WRONG: ["\"Smith", " Jr.\"", "John", "45"] — 4 fields
→ RIGHT: ["Smith, Jr.", "John", "45"] — 3 fields
→ FIX: Implement quote-aware CSV parser (shown in code below)

PITFALL 2: "JSON parsing fails with NumberFormatException"
→ CAUSE: JSON value is string but treated as number
→ INPUT: {"amount": "15250.00"} ← string, not number
→ CODE: jsonNode.get("amount").asDouble() ← works
→ CODE: Double.parseDouble(jsonNode.get("amount").toString())
        → toString() returns "\"15250.00\"" with quotes!
→ FIX: Use .asText() then parse, or .asDouble() directly

PITFALL 3: "KeyValueTextInputFormat gives empty value"
→ CAUSE: Line has no TAB character
→ INPUT: "user123,John Smith,45" (comma-separated, not tab)
→ RESULT: key="user123,John Smith,45", value=""
→ FIX: Either use correct separator configuration
       OR switch to TextInputFormat + manual parsing

PITFALL 4: "NLineInputFormat creates too many mappers"
→ CAUSE: N set too small for large file
→ EXAMPLE: 10 million line file, N=10
→ RESULT: 1 million mappers requested
→ Each mapper: JVM startup + setup overhead = several seconds
→ 1 million × 5 seconds = 58 DAYS of overhead alone
→ FIX: Set N to produce reasonable mapper count (100-10000 mappers)

PITFALL 5: "First record in every mapper is corrupt"
→ CAUSE: Block boundary split a line, mapper starts mid-line
→ BUT: TextInputFormat handles this automatically (Mechanic 2 above)
→ If you see this: you're using a custom RecordReader incorrectly
→ OR: you're reading raw bytes instead of using RecordReader

PITFALL 6: "Mapper signature mismatch causes ClassCastException"
→ CAUSE: InputFormat produces different key type than mapper expects
→ TextInputFormat → key = LongWritable
→ KeyValueTextInputFormat → key = Text
→ Mapper<LongWritable, Text, ...> with KeyValueTextInputFormat → CRASH
→ FIX: Mapper key type must match InputFormat key type
→ ERROR MESSAGE: java.lang.ClassCastException:
   org.apache.hadoop.io.Text cannot be cast to org.apache.hadoop.io.LongWritable

PITFALL 7: "JSON library not found at runtime"
→ CAUSE: Jackson/Gson JAR not in Hadoop classpath
→ Compiles fine (JAR in local classpath)
→ Fails on cluster: ClassNotFoundException: com.fasterxml.jackson...
→ FIX: Bundle library in your job JAR
       OR use job.addFileToClassPath()
       OR use Hadoop's built-in org.apache.hadoop.shaded.com.fasterxml.jackson

CSV PARSING OVERHEAD:
  → Simple split(","): ~0.1 microsecond per line
  → Quote-aware parser: ~0.5 microsecond per line
  → 5× slower BUT still negligible vs disk I/O
  → For 1 billion records: 500 seconds of extra parse time
  → Distributed across 1000 mappers: 0.5 seconds each
  → NOT a bottleneck

JSON PARSING OVERHEAD:
  → Jackson library: ~1-2 microseconds per line
  → 10-20× slower than simple split
  → For 1 billion records: 2000 seconds extra
  → Distributed across 1000 mappers: 2 seconds each
  → Still NOT a bottleneck (disk I/O dominates)
  → DO NOT pre-optimize parsing — focus on I/O and shuffle

  → InputFormat choice rarely affects overall job time

import java.util.ArrayList;
import java.util.List;

public class CSVParserUtil {

    /**
     * Parse a CSV line respecting quoted fields.
     * 
     * Handles:
     *   → Commas inside quoted fields: "Smith, Jr."
     *   → Escaped quotes inside fields: "He said ""hello"""
     *   → Mixed quoted and unquoted fields
     *   → Trailing commas
     * 
     * WHY NOT use line.split(",")?
     *   → "John","New York, NY","45" 
     *   → split gives: ["\"John\"", "\"New York", " NY\"", "\"45\""]  WRONG
     *   → parser gives: ["John", "New York, NY", "45"]  CORRECT
     * 
     * This is a STATE MACHINE parser:
     *   State 1: FIELD_START — beginning of a new field
     *   State 2: IN_QUOTED — inside a quoted field
     *   State 3: IN_UNQUOTED — inside an unquoted field
     *   State 4: QUOTE_IN_QUOTED — saw quote inside quoted field
     */
    public static String[] parseCSVLine(String line, char delimiter, char quote) {
        
        if (line == null || line.isEmpty()) {
            return new String[0];
        }
        
        List<String> fields = new ArrayList<>();
        StringBuilder currentField = new StringBuilder();
        boolean inQuotes = false;
        
        for (int i = 0; i < line.length(); i++) {
            char c = line.charAt(i);
            
            if (inQuotes) {
                if (c == quote) {
                    // Check for escaped quote (double quote: "")
                    if (i + 1 < line.length() && line.charAt(i + 1) == quote) {
                        // Escaped quote — add single quote to field
                        currentField.append(quote);
                        i++;  // Skip next quote
                    } else {
                        // End of quoted field
                        inQuotes = false;
                    }
                } else {
                    // Regular character inside quotes — include everything
                    currentField.append(c);
                }
            } else {
                if (c == quote) {
                    // Start of quoted field
                    inQuotes = true;
                } else if (c == delimiter) {
                    // Field separator — emit current field
                    fields.add(currentField.toString().trim());
                    currentField.setLength(0);  // Reset for next field
                } else {
                    // Regular character outside quotes
                    currentField.append(c);
                }
            }
        }
        
        // Don't forget the last field (no trailing delimiter)
        fields.add(currentField.toString().trim());
        
        return fields.toArray(new String[0]);
    }
    
    /**
     * Convenience method with default comma delimiter and double-quote.
     */
    public static String[] parseCSVLine(String line) {
        return parseCSVLine(line, ',', '"');
    }
}

public class CSVClaimsProcessor extends Configured implements Tool {

    // ════════════════════════════════════════════════════════
    // MAPPER: Parse quoted CSV → emit (claim_type, amount)
    // ════════════════════════════════════════════════════════
    //
    // Input CSV format (with quotes and commas in fields):
    //   "CLM-001","John O'Brien","2024-01-15","Accident, car collision","15,250.00"
    //
    // Fields after parsing:
    //   [0] Claim ID
    //   [1] Customer Name
    //   [2] Date
    //   [3] Claim Type (may contain commas!)
    //   [4] Amount (may contain commas in thousands!)
    //
    // Emit: key = claim type category, value = amount

    public static class CSVMapper
            extends Mapper<LongWritable, Text, Text, DoubleWritable> {

        private Text claimTypeKey = new Text();
        private DoubleWritable amountValue = new DoubleWritable();

        enum CSVCounters {
            TOTAL_LINES,
            PARSED_OK,
            PARSE_ERROR_FIELD_COUNT,
            PARSE_ERROR_AMOUNT,
            HEADER_SKIPPED
        }

        @Override
        protected void map(LongWritable offset, Text line, Context context)
                throws IOException, InterruptedException {

            context.getCounter(CSVCounters.TOTAL_LINES).increment(1);

            String rawLine = line.toString().trim();
            if (rawLine.isEmpty()) return;

            // Skip header row
            if (rawLine.startsWith("\"CLM_ID\"") || rawLine.startsWith("CLM_ID")) {
                context.getCounter(CSVCounters.HEADER_SKIPPED).increment(1);
                return;
            }

            // ── USE PROPER CSV PARSER — NOT split(",") ──
            // This is the KEY lesson of this project
            String[] fields = CSVParserUtil.parseCSVLine(rawLine);

            // Validate field count
            if (fields.length != 5) {
                context.getCounter(CSVCounters.PARSE_ERROR_FIELD_COUNT).increment(1);
                return;
            }

            String claimId   = fields[0];
            String custName  = fields[1];
            String date      = fields[2];
            String claimType = fields[3];   // "Accident, car collision" → correctly parsed
            String amountStr = fields[4];

            // Parse amount — remove thousands separator
            // "15,250.00" → "15250.00" → 15250.00
            try {
                String cleanAmount = amountStr.replace(",", "");
                double amount = Double.parseDouble(cleanAmount);
                
                // Extract claim category (first word of claim type)
                String category = claimType.split("[,\\s]")[0].trim();
                if (category.isEmpty()) category = "Unknown";

                claimTypeKey.set(category);
                amountValue.set(amount);
                context.write(claimTypeKey, amountValue);
                context.getCounter(CSVCounters.PARSED_OK).increment(1);

            } catch (NumberFormatException e) {
                context.getCounter(CSVCounters.PARSE_ERROR_AMOUNT).increment(1);
            }
        }
    }


    // ════════════════════════════════════════════════════════
    // REDUCER: Sum amounts per claim type
    // ════════════════════════════════════════════════════════

    public static class SumReducer
            extends Reducer<Text, DoubleWritable, Text, DoubleWritable> {

        private DoubleWritable totalAmount = new DoubleWritable();

        @Override
        protected void reduce(Text claimType, Iterable<DoubleWritable> amounts,
                              Context context) throws IOException, InterruptedException {

            double sum = 0.0;
            for (DoubleWritable amount : amounts) {
                sum += amount.get();
            }
            totalAmount.set(Math.round(sum * 100.0) / 100.0);  // Round to 2 decimals
            context.write(claimType, totalAmount);
        }
    }


    @Override
    public int run(String[] args) throws Exception {
        if (args.length != 2) {
            System.err.println("Usage: CSVClaimsProcessor <input> <output>");
            return 1;
        }

        Configuration conf = getConf();
        Job job = Job.getInstance(conf, "CSV Claims Processor - Quoted Field Handling");
        job.setJarByClass(CSVClaimsProcessor.class);

        // ── EXPLICITLY set InputFormat (default is TextInputFormat anyway) ──
        // We're explicit here to demonstrate the concept
        job.setInputFormatClass(TextInputFormat.class);

        job.setMapperClass(CSVMapper.class);
        job.setReducerClass(SumReducer.class);
        job.setCombinerClass(SumReducer.class);  // Combiner safe for sum (Project 9)

        job.setOutputKeyClass(Text.class);
        job.setOutputValueClass(DoubleWritable.class);

        FileInputFormat.addInputPath(job, new Path(args[0]));
        FileOutputFormat.setOutputPath(job, new Path(args[1]));

        return job.waitForCompletion(true) ? 0 : 1;
    }

    public static void main(String[] args) throws Exception {
        System.exit(ToolRunner.run(new Configuration(), new CSVClaimsProcessor(), args));
    }
}

    //   POL-001\tJane Smith\tPremium Plan\t2024-01-01\t2025-01-01\t1200.00
    //
    // KeyValueTextInputFormat splits on FIRST TAB:
    //   KEY:   "POL-001"

    //

    // This is different from TextInputFormat mappers!

    public static class PolicyMapper
            extends Mapper<Text, Text, Text, IntWritable> {
            //              ↑     ↑
            //              Text, Text ← KeyValueTextInputFormat types
            //              NOT LongWritable, Text

        private Text planTypeKey = new Text();
        private final IntWritable one = new IntWritable(1);

        enum TSVCounters {
            TOTAL_RECORDS,
            PARSED_OK,
            WRONG_FIELD_COUNT,
            HEADER_SKIPPED
        }

        @Override
        protected void map(Text policyId, Text restOfLine, Context context)
                throws IOException, InterruptedException {

            context.getCounter(TSVCounters.TOTAL_RECORDS).increment(1);

            // Skip header
            if (policyId.toString().startsWith("POL_ID") ||
                policyId.toString().startsWith("POLICY")) {
                context.getCounter(TSVCounters.HEADER_SKIPPED).increment(1);
                return;
            }

            // Parse the remaining fields (after first TAB)
            // VALUE = "Jane Smith\tPremium Plan\t2024-01-01\t2025-01-01\t1200.00"
            String[] remainingFields = restOfLine.toString().split("\t");

            // Expected remaining fields: name, plan, start_date, end_date, amount
            if (remainingFields.length != 5) {
                context.getCounter(TSVCounters.WRONG_FIELD_COUNT).increment(1);
                return;
            }

            String customerName = remainingFields[0].trim();
            String planType     = remainingFields[1].trim();
            String startDate    = remainingFields[2].trim();
            String endDate      = remainingFields[3].trim();
            String amountStr    = remainingFields[4].trim();

            // Emit: count policies per plan type
            planTypeKey.set(planType);
            context.write(planTypeKey, one);
            context.getCounter(TSVCounters.PARSED_OK).increment(1);
        }
    }


    // ════════════════════════════════════════════════════════
    // REDUCER: Count policies per plan type

                sum += count.get();
            }
            result.set(sum);
            context.write(planType, result);
        }
    }

public class JSONAgentProcessor extends Configured implements Tool {

    // ════════════════════════════════════════════════════════
    // MAPPER: Parse JSON lines → extract region and revenue
    // ════════════════════════════════════════════════════════
    //
    // Input (JSON Lines format — one JSON object per line):
    //   {"agent_id":"AGT-001","name":"Alice Wu","region":"NORTH","sales":45,"revenue":67500.00}
    //
    // We use TextInputFormat (each line = one JSON string)
    // Then parse JSON manually (lightweight — no external library)
    //
    // WHY manual JSON parsing instead of Jackson?
    //   → Avoids JAR dependency issues on cluster
    //   → For simple flat JSON, manual parsing is sufficient
    //   → For complex nested JSON, USE Jackson (add to classpath)
    //
    // For production, use Jackson:
    //   ObjectMapper mapper = new ObjectMapper();
    //   JsonNode root = mapper.readTree(line.toString());
    //   String region = root.get("region").asText();
    //   double revenue = root.get("revenue").asDouble();

    public static class JSONMapper
            extends Mapper<LongWritable, Text, Text, DoubleWritable> {

        private Text regionKey = new Text();
        private DoubleWritable revenueValue = new DoubleWritable();

        enum JSONCounters {
            TOTAL_LINES,
            PARSED_OK,
            PARSE_ERROR,
            EMPTY_LINE,
            MISSING_FIELD
        }

        @Override
        protected void map(LongWritable offset, Text line, Context context)
                throws IOException, InterruptedException {

            context.getCounter(JSONCounters.TOTAL_LINES).increment(1);

            String jsonStr = line.toString().trim();
            if (jsonStr.isEmpty()) {
                context.getCounter(JSONCounters.EMPTY_LINE).increment(1);
                return;
            }

            try {
                // ── LIGHTWEIGHT JSON PARSER ──
                // For flat JSON objects only
                // Extracts value for a given key from JSON string
                
                String region = extractJsonStringValue(jsonStr, "region");
                String revenueStr = extractJsonValue(jsonStr, "revenue");

                if (region == null || revenueStr == null) {
                    context.getCounter(JSONCounters.MISSING_FIELD).increment(1);
                    return;
                }

                double revenue = Double.parseDouble(revenueStr);

                regionKey.set(region.toUpperCase());
                revenueValue.set(revenue);
                context.write(regionKey, revenueValue);
                context.getCounter(JSONCounters.PARSED_OK).increment(1);

            } catch (Exception e) {
                context.getCounter(JSONCounters.PARSE_ERROR).increment(1);
            }
        }

        /**
         * Extract a string value from flat JSON.
         * Input: {"key1":"val1","key2":"val2"}
         * extractJsonStringValue(json, "key1") → "val1"
         */
        private String extractJsonStringValue(String json, String key) {
            String searchKey = "\"" + key + "\"";
            int keyIndex = json.indexOf(searchKey);
            if (keyIndex == -1) return null;

            // Find the colon after the key
            int colonIndex = json.indexOf(':', keyIndex + searchKey.length());
            if (colonIndex == -1) return null;

            // Find the opening quote of the value
            int valueStart = json.indexOf('"', colonIndex + 1);
            if (valueStart == -1) return null;

            // Find the closing quote of the value
            int valueEnd = json.indexOf('"', valueStart + 1);
            if (valueEnd == -1) return null;

            return json.substring(valueStart + 1, valueEnd);
        }

        /**
         * Extract a numeric value from flat JSON (no quotes around value).
         * Input: {"key1":"val1","key2":123.45}
         * extractJsonValue(json, "key2") → "123.45"
         */
        private String extractJsonValue(String json, String key) {
            String searchKey = "\"" + key + "\"";
            int keyIndex = json.indexOf(searchKey);
            if (keyIndex == -1) return null;

            int colonIndex = json.indexOf(':', keyIndex + searchKey.length());
            if (colonIndex == -1) return null;

            // Skip whitespace after colon
            int valueStart = colonIndex + 1;
            while (valueStart < json.length() && json.charAt(valueStart) == ' ') {
                valueStart++;
            }

            // Find end of value (comma, closing brace, or end of string)
            int valueEnd = valueStart;
            while (valueEnd < json.length()) {
                char c = json.charAt(valueEnd);
                if (c == ',' || c == '}' || c == ' ') break;
                valueEnd++;
            }

            String value = json.substring(valueStart, valueEnd).trim();
            // Remove quotes if present (string value for numeric field)
            if (value.startsWith("\"") && value.endsWith("\"")) {
                value = value.substring(1, value.length() - 1);
            }
            return value;
        }
    }


    // ════════════════════════════════════════════════════════
    // REDUCER: Sum revenue per region
    // ════════════════════════════════════════════════════════

    public static class RevenueSumReducer
            extends Reducer<Text, DoubleWritable, Text, DoubleWritable> {

        private DoubleWritable totalRevenue = new DoubleWritable();

        @Override
        protected void reduce(Text region, Iterable<DoubleWritable> revenues,
                              Context context) throws IOException, InterruptedException {
            double sum = 0.0;
            for (DoubleWritable revenue : revenues) {
                sum += revenue.get();
            }
            totalRevenue.set(Math.round(sum * 100.0) / 100.0);
            context.write(region, totalRevenue);
        }
    }

        if (args.length != 2) {
            System.err.println("Usage: JSONAgentProcessor <input> <output>");
            return 1;
        }

        Configuration conf = getConf();

        // ═══════════════════════════════════════════════════
        // DEMONSTRATE NLineInputFormat
        // ═══════════════════════════════════════════════════
        // Instead of splitting by block size (128MB),
        // split so each mapper gets exactly 100 lines.
        //
        // WHY USE THIS FOR JSON?
        //   → JSON parsing is CPU-intensive
        //   → Equal line count = equal processing time per mapper
        //   → More predictable resource usage
        //
        // FOR SMALL FILES: NLineInputFormat ensures at least
        // some parallelism even when file < 128MB
        // (TextInputFormat would create only 1 mapper)

        conf.setInt("mapreduce.input.lineinputformat.linespermap", 100);

        Job job = Job.getInstance(conf, "JSON Agent Processor - NLineInputFormat");
        job.setJarByClass(JSONAgentProcessor.class);

        // ═══════════════════════════════════════════════════
        // KEY LINE: Set InputFormat to NLineInputFormat
        // ═══════════════════════════════════════════════════
        // Each mapper gets exactly 100 lines (configured above)
        // Key type: LongWritable (byte offset) — same as TextInputFormat
        // Value type: Text (full line) — same as TextInputFormat
        job.setInputFormatClass(NLineInputFormat.class);

                   .getValue() > 0 ?

            System.out.println("════════════════════════════════\n");
        }

        return success ? 0 : 1;
    }

def demonstrate_csv_parsing_problem():
    """
    Visually prove why naive comma splitting destroys data
    and why a state-machine parser is essential.
    """
    
    test_lines = [
        '"CLM-001","John O\'Brien","2024-01-15","Accident, car collision","15,250.00"',
        '"CLM-005","Jane ""JJ"" Smith","2024-01-17","Fire, electrical fault","45,000.00"',
        '"CLM-011","Roy Adams","2024-01-20","Flood damage, river overflow","31,000.00"',
    ]
    
    print("=" * 75)
    print("PROOF: WHY split(',') FAILS ON QUOTED CSV")
    print("=" * 75)
    
    for line in test_lines:
        print(f"\nRAW LINE:\n  {line}")
        
        # ── WRONG WAY: Naive split ──
        naive_result = line.split(",")
        print(f"\nNAIVE split(',') → {len(naive_result)} fields:")
        for i, field in enumerate(naive_result):
            print(f"  [{i}] {field}")
        
        # ── RIGHT WAY: Quote-aware parser ──
        proper_result = parse_csv_properly(line)
        print(f"\nPROPER CSV parser → {len(proper_result)} fields:")
        for i, field in enumerate(proper_result):
            print(f"  [{i}] {field}")
        
        # ── Impact on analysis ──
        if len(naive_result) != len(proper_result):
            print(f"\n  ⚠️  FIELD COUNT MISMATCH: naive={len(naive_result)}, "
                  f"proper={len(proper_result)}")
            print(f"  → Naive parsing would CORRUPT this record")
            print(f"  → In Hadoop: mapper validation rejects it OR produces wrong results")
        
        print("-" * 75)


def parse_csv_properly(line, delimiter=',', quote='"'):
    """
    Python implementation of the same state-machine CSV parser
    used in our Java CSVParserUtil.
    
    States:
      FIELD_START: Beginning of new field
      IN_QUOTED:   Inside a quoted field (commas are literal)
      IN_UNQUOTED: Inside an unquoted field
    """
    fields = []
    current_field = []
    in_quotes = False
    i = 0
    
    while i < len(line):
        c = line[i]
        
        if in_quotes:
            if c == quote:
                # Check for escaped quote (double quote)
                if i + 1 < len(line) and line[i + 1] == quote:
                    current_field.append(quote)
                    i += 1  # Skip next quote
                else:
                    in_quotes = False
            else:
                current_field.append(c)
        else:
            if c == quote:
                in_quotes = True
            elif c == delimiter:
                fields.append(''.join(current_field).strip())
                current_field = []
            else:
                current_field.append(c)
        
        i += 1
    
    # Last field
    fields.append(''.join(current_field).strip())
    
    return fields

def demonstrate_json_parsing():
    """
    Show JSON parsing approaches in Python (mirrors Java mapper logic).
    """
    
    print("\n" + "=" * 75)
    print("JSON PARSING APPROACHES FOR HADOOP MAPPERS")
    print("=" * 75)
    
    json_line = '{"agent_id":"AGT-001","name":"Alice Wu","region":"NORTH","sales":45,"revenue":67500.00}'
    
    print(f"\nRAW JSON LINE:")
    print(f"  {json_line}")
    
    # ── Approach 1: Manual string parsing (as in our Java code) ──
    print(f"\n1. MANUAL STRING PARSING (no external library):")
    print(f"   → Used in our Java mapper to avoid JAR dependencies")
    
    import json as json_lib
    
    # Simulate manual extraction
    def extract_value(json_str, key):
        search = f'"{key}":'
        idx = json_str.find(search)
        if idx == -1:
            return None
        value_start = idx + len(search)
        if json_str[value_start] == '"':
            # String value
            end = json_str.index('"', value_start + 1)
            return json_str[value_start + 1:end]
        else:
            # Numeric value
            end = value_start
            while end < len(json_str) and json_str[end] not in ',}':
                end += 1
            return json_str[value_start:end].strip()
    
    print(f"   region:  {extract_value(json_line, 'region')}")
    print(f"   revenue: {extract_value(json_line, 'revenue')}")
    print(f"   sales:   {extract_value(json_line, 'sales')}")
    
    # ── Approach 2: JSON library (recommended for production) ──
    print(f"\n2. JSON LIBRARY PARSING (Jackson in Java, json in Python):")
    parsed = json_lib.loads(json_line)
    for key, value in parsed.items():
        print(f"   {key}: {value} ({type(value).__name__})")
    
    print(f"\n   → Library correctly types: strings, ints, floats")
    print(f"   → Manual parser returns everything as string")
    print(f"   → For production Hadoop jobs: USE JACKSON LIBRARY")
    print(f"   → Bundle jackson-core + jackson-databind in your JAR")
    
    # ── Approach 3: What NOT to do ──
    print(f"\n3. WHAT NOT TO DO:")
    print(f"   → json_str.replace('{{', '').replace('}}', '').split(',')")
    print(f"   → Breaks on: nested objects, arrays, commas in strings")
    print(f"   → Same problem as CSV naive splitting")

WHAT PROJECT 12 ESTABLISHED:

✅ InputFormat has TWO responsibilities:
   1. getSplits() → how to divide input into mapper tasks
   2. createRecordReader() → how to parse records for mapper

✅ TextInputFormat (default):
   → Key = byte offset (LongWritable), Value = full line (Text)
   → Splits by block size (128MB)
   → Handles block boundary line spanning automatically
   → Works for 80% of use cases

✅ KeyValueTextInputFormat:
   → Key = text before first separator (Text)
   → Value = text after first separator (Text)
   → Mapper signature changes: Mapper<Text, Text, ...>
   → Only useful for 2-part key-value records

✅ NLineInputFormat:
   → Same key-value as TextInputFormat
   → But splits by LINE COUNT instead of byte size
   → Guarantees equal records per mapper
   → Overhead: must read entire file to count lines

✅ CSV PARSING:
   → NEVER use split(",") on quoted CSV
   → Use state-machine parser that tracks quote state
   → Commas inside quotes are literal, not delimiters

✅ JSON PARSING:
   → JSON Lines (one object per line) works with TextInputFormat
   → Parse JSON in mapper using library or manual extraction
   → Bundle JSON library JAR with your Hadoop job
