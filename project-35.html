PROJECT 35: DESIGNING AN HBASE ROW KEY FOR BILLION-RECORD TIME SERIES

🔴 HYPOTHETICAL PROBLEM

SCENARIO:
→ You are the architect at a smart city IoT platform
→ 500,000 sensors across the city: traffic, weather, air quality, noise
→ Each sensor reports every 5 seconds
→ That's 100,000 readings PER SECOND (500K sensors / 5 seconds)
→ Daily: 8.64 BILLION readings
→ 3-year retention: ~9.5 TRILLION records

REQUIREMENTS:

│  PATTERN 2: TIME RANGE SCAN (Analysis)                        │
│  "Get ALL readings for sensor AIR-00712 between                │

│  → Must return in < 500ms                                     │
│  → Used by: anomaly detection, trend analysis                 │
│                                                               │
│  PATTERN 3: MULTI-SENSOR SCAN (Geographic)                    │
│  "Get latest readings for ALL sensors in zone DOWNTOWN"       │
│  → ~2,000 sensors in a zone                                   │
│  → Must return in < 2 seconds                                 │

FIRST ATTEMPT (naive row key):

  Row key = sensor_id + timestamp
  Example: "TRAFFIC-42159_1705312800000"

  WHAT GOES CATASTROPHICALLY WRONG:

  Problem 1: WRITE HOT-SPOTTING
  → All sensors report at roughly the same time
  → Timestamps increase monotonically: ...800, ...805, ...810
  → HBase stores rows in SORTED order
  → ALL new writes go to the LAST region (highest key range)
  → ONE region server handles 100% of write load
  → Other 99 region servers sit IDLE
  
  ┌──────────────────────────────────────────────────────┐
  │  Region 1: [A... - D...]     → idle                  │
  │  Region 2: [D... - H...]     → idle                  │
  │  Region 3: [H... - N...]     → idle                  │
  │  ...                                                  │
  │  Region 50: [T... - Z...]    → ALL WRITES HERE 🔥    │
  │                                 (100K writes/sec)     │
  │                                 Region server crashes │
  └──────────────────────────────────────────────────────┘

  Problem 2: SEQUENTIAL SCAN INEFFICIENCY
  → "Get latest reading for TRAFFIC-42159"
  → Row key starts with sensor_id → all that sensor's data is contiguous
  → BUT "latest" means scanning to the END of that sensor's key range
  → With 3 years of data: ~190M rows to skip to reach the latest

  Problem 3: REGION SPLIT IMBALANCE
  → HBase auto-splits regions when they grow too large
  → All data lands in one region → splits create 2 regions
  → Both still receive ALL writes → splits again
  → Creates a "split cascade" but never achieves balance

THE QUESTION:
→ How do you design a row key that:
  - Distributes writes EVENLY across ALL region servers
  - Supports fast point lookups by sensor + time
  - Supports efficient time range scans per sensor
  - Supports geographic grouping for zone queries
  - Scales to trillions of records over years

STEP 2: Identify the row key design constraints

  ┌────────────────────────────────────────────────────────────┐
  │              ROW KEY DESIGN CONSTRAINTS                     │
  │                                                             │
  │  CONSTRAINT 1: Write distribution                           │
  │  → Keys must spread across ALL regions uniformly            │
  │  → No monotonically increasing component at the START       │
  │  → Timestamp at start = hot-spot guaranteed                 │
  │                                                             │
  │  CONSTRAINT 2: Read locality for single-sensor queries      │
  │  → All data for one sensor should be CONTIGUOUS             │
  │  → Enables efficient range scan within a sensor             │
  │  → sensor_id must be part of the key                        │
  │                                                             │
  │  CONSTRAINT 3: Time ordering within a sensor                │
  │  → "Latest reading" should be the FIRST row found           │
  │  → Reverse timestamp: newest data sorts FIRST               │
  │  → Long.MAX_VALUE - timestamp = reverse chronological       │
  │                                                             │
  │  CONSTRAINT 4: Key compactness                              │
  │  → Row key is stored with EVERY cell                        │
  │  → Trillion rows × long key = terabytes of overhead         │
  │  → Short keys save significant storage                      │
  │                                                             │
  │  CONSTRAINT 5: Geographic grouping                          │
  │  → Zone queries scan a PREFIX range                         │
  │  → Zone identifier near the start of key helps              │
  │  → BUT conflicts with write distribution                    │
  │                                                             │
  │  THESE CONSTRAINTS CONFLICT — we must make tradeoffs        │

STEP 3: Evaluate row key design patterns

  PATTERN A: sensor_id + timestamp (naive)
    Key: "TRAFFIC-42159_1705312800000"
    ✅ Read locality per sensor
    ❌ Write hot-spot (alphabetical clustering by sensor prefix)
    ❌ Latest reading requires full scan to end of sensor range
    ❌ Long key (30+ bytes)

  PATTERN B: hash(sensor_id) + sensor_id + reverse_timestamp
    Key: "a7" + "TRAFFIC-42159" + "8294687199999"
    ✅ Write distribution (hash prefix spreads across regions)
    ✅ Read locality (same hash + sensor → contiguous)
    ✅ Latest reading is FIRST row (reverse timestamp)
    ⚠️ Geographic queries require knowing all sensor IDs in zone
    ⚠️ Medium key length (~25 bytes)

  PATTERN C: salt + sensor_id + reverse_timestamp
    Key: "07" + "TRAFFIC-42159" + "8294687199999"
    Salt = sensor_id.hashCode() % num_salts
    ✅ Write distribution (salt spreads across regions)
    ✅ Read locality (same salt + sensor → contiguous)
    ✅ Latest reading is FIRST row
    ✅ Predictable salt for any sensor (deterministic hash)
    ⚠️ Zone queries still require knowing sensor IDs
    ⚠️ Salt count must match region count for optimal spread

  PATTERN D: salt + zone_id + sensor_type + sensor_id + reverse_timestamp
    Key: "07" + "DOWNTOWN" + "TRAFFIC" + "42159" + "8294687199999"
    ✅ Write distribution
    ✅ Read locality per sensor
    ✅ Geographic scan by zone prefix
    ✅ Type-based scan (all traffic sensors in zone)
    ❌ Longer key (~40 bytes)
    ❌ Zone assignment changes require data migration

  WE CHOOSE PATTERN C (with elements of D for zone support)
  → Best balance of write distribution and read efficiency
  → Zone mapping stored in a separate lookup table

💡 HOW CAN WE SOLVE IT

FINAL ROW KEY DESIGN:

═══════════════════════════════════════════════════════════════
THE ROW KEY STRUCTURE (binary, not string)
═══════════════════════════════════════════════════════════════

  ┌──────┬───────────────┬──────────────────────────┐
  │ SALT │ SENSOR_HASH   │ REVERSE_TIMESTAMP        │
  │ 1 B  │ 4 bytes       │ 8 bytes                  │
  │      │ (int)         │ (long)                   │
  └──────┴───────────────┴──────────────────────────┘
  Total: 13 bytes (compact!)

  SALT (1 byte):
  → Value: 0x00 to 0xFF (256 possible salts)
  → Calculated: (sensor_id.hashCode() & 0xFF)
  → DETERMINISTIC: same sensor always gets same salt
  → Purpose: distribute writes across 256 key ranges
  → 256 salts → can pre-split into 256 regions

  SENSOR_HASH (4 bytes):
  → Value: Murmur3 hash of sensor_id string
  → Compact representation of sensor identity
  → ALL rows for one sensor have same hash → contiguous
  → Collision risk: ~1 in 4 billion (acceptable for 500K sensors)
  → Store sensor_id → hash mapping in separate table

  REVERSE_TIMESTAMP (8 bytes):
  → Value: Long.MAX_VALUE - epoch_millis
  → Newest data has SMALLEST value → sorts FIRST
  → "Get latest" = scan first row for this sensor
  → Time range scan: compute reverse(start) and reverse(end)

  EXAMPLES:
  
  Sensor "TRAFFIC-42159" at 2024-01-15 10:00:00.000 UTC:
    salt = hashCode("TRAFFIC-42159") & 0xFF = 0x4B
    sensor_hash = Murmur3("TRAFFIC-42159") = 0xA7F3219C
    timestamp = 1705312800000
    reverse_ts = 9223372036854775807 - 1705312800000 = 9221666724054775807
    
    Row key (hex): 4B A7F3219C 7FFE96C8A25C03FF
    Row key (13 bytes)

  Same sensor, 5 seconds later (10:00:05):
    salt = 0x4B (same — deterministic)
    sensor_hash = 0xA7F3219C (same sensor)
    reverse_ts = 9221666724054770807 (smaller — sorts AFTER previous!)
    
    Wait — smaller reverse_ts sorts BEFORE larger.
    10:00:05 has SMALLER reverse_ts than 10:00:00.
    So 10:00:05 sorts BEFORE 10:00:00.
    → NEWEST DATA COMES FIRST ✓

✅ WHAT SHOULD DO

1. USE BINARY ROW KEYS (not strings)
   → String "1705312800000" = 13 bytes
   → Long as bytes = 8 bytes (always)
   → Savings compound across trillions of rows
   → Use Bytes.toBytes() for all conversions

2. PRE-SPLIT REGIONS AT TABLE CREATION
   → Default: table starts with ONE region
   → All writes go to one server until it splits
   → Pre-split into 256 regions (matching salt range)
   → Immediate write distribution from the first second

3. USE REVERSE TIMESTAMP FOR "LATEST FIRST" QUERIES
   → Long.MAX_VALUE - timestamp
   → Most recent data sorts first in lexicographic order
   → "Get latest" = scan with limit 1 (instant)
   → Time range: reverse(end_time) to reverse(start_time)

4. SET TTL ON DATA COLUMN FAMILY
   → 3-year retention: TTL = 94608000 seconds
   → HBase automatically deletes expired cells during compaction
   → No manual cleanup needed
   → Metadata CF: no TTL (keep forever)

5. ENABLE BLOOM FILTERS ON THE TABLE
   → ROW-level bloom filter: "Is this exact row key in this store file?"
   → Avoids reading store files that don't contain the key
   → Especially useful for point lookups
   → Small memory cost, large read performance gain

6. ENABLE SNAPPY COMPRESSION ON COLUMN FAMILIES
   → Sensor data compresses well (repeated patterns)
   → SNAPPY: fast compression/decompression
   → Typically 4-8x compression on sensor data
   → Reduces disk I/O and storage costs

7. BUILD A SENSOR REGISTRY TABLE
   → Separate small table mapping sensor_id → hash, zone, type
   → Used by applications to compute row key from sensor_id
   → Supports zone-based queries (look up all sensors in zone)

🚫 WHAT SHOULD NOT DO

1. DO NOT PUT TIMESTAMP AT THE START OF ROW KEY
   → All current-time writes cluster in one region
   → Classic hot-spotting pattern
   → Even with reverse timestamp at start: all "latest" reads cluster

2. DO NOT USE STRING ENCODING FOR NUMERIC VALUES
   → String "42159" = 5 bytes, sorts lexicographically ("9" > "10")
   → Int as bytes = 4 bytes, sorts correctly numerically
   → Bytes.toBytes(42159) for correct binary sort order

3. DO NOT USE RANDOM SALT (non-deterministic)
   → Random salt means you don't know WHERE a sensor's data lives
   → Point lookup requires scanning ALL 256 salt regions
   → Salt MUST be deterministic: hash(sensor_id) mod N

4. DO NOT CREATE TOO MANY COLUMN FAMILIES
   → Each CF = separate storage file (HFile) on disk
   → Flushing one CF forces flush of ALL CFs
   → 10 CFs with different write rates = constant unnecessary flushes
   → MAXIMUM: 2-3 column families per table

5. DO NOT STORE LARGE BLOBS IN HBASE
   → Cell values should be < 10MB (ideally < 100KB)
   → Large values cause heap pressure and compaction storms
   → If sensor sends images: store in HDFS, put path in HBase

6. DO NOT SKIP PRE-SPLITTING
   → Without pre-split: single region → hot-spot → split → hot-spot → split
   → This "split cascade" takes hours to stabilize
   → Pre-splitting gives instant balanced distribution

7. DO NOT USE AUTO-INCREMENTING ROW KEYS
   → Sequential IDs (1, 2, 3, ...) all go to last region
   → Same problem as timestamp-first key
   → Always hash or salt to distribute

🚫 WHEN NOT TO USE

1. WHEN YOU DON'T NEED RANDOM ACCESS
   → If ALL queries are batch scans over entire dataset
   → Use HDFS + Hive/MapReduce (Project 34) instead
   → HBase overhead not justified for batch-only workloads

2. WHEN DATA FITS ON ONE SERVER
   → < 100GB total data, < 1000 reads/sec
   → PostgreSQL with TimescaleDB extension is simpler
   → HBase distributed overhead not needed

3. WHEN YOU NEED COMPLEX QUERIES
   → HBase has NO SQL (only get, put, scan, delete)
   → Multi-table joins → not possible in HBase
   → Complex aggregations → use Hive over HBase or Phoenix

4. WHEN WRITE VOLUME IS LOW
   → < 1000 writes/sec → any database handles this
   → HBase shines at > 10,000 writes/sec
   → Below that threshold: operational complexity not worth it

5. WHEN ALL QUERIES ARE BY SECONDARY ATTRIBUTES
   → Row key is the ONLY native index
   → If most queries are "find sensors by zone" (not by sensor_id)
   → Need secondary indexes (Phoenix) or denormalized tables

MECHANIC 1: HOW ROW KEY DETERMINES REGION PLACEMENT
═══════════════════════════════════════════════════════════════

  HBase table with 256 pre-split regions:
  
  Region boundaries (split points):
    Region 0:   [0x00...] to [0x01...)
    Region 1:   [0x01...] to [0x02...)
    Region 2:   [0x02...] to [0x03...)
    ...
    Region 255: [0xFF...] to [end)
  
  Write arrives: sensor "TRAFFIC-42159"
    salt = hashCode("TRAFFIC-42159") & 0xFF = 0x4B
    Row key starts with 0x4B
    → Goes to Region 75 (0x4B = 75 decimal)
    → Region 75 lives on RegionServer 7 (75 mod 10 servers)
  
  Write arrives: sensor "WEATHER-00312"
    salt = hashCode("WEATHER-00312") & 0xFF = 0xC2
    Row key starts with 0xC2
    → Goes to Region 194
    → Region 194 lives on RegionServer 4
  
  RESULT: 500,000 sensors distributed across 256 regions
  Each region handles ~1,950 sensors
  Each of 10 region servers handles ~25-26 regions
  Write load: ~10,000 writes/sec per server (balanced!)

═══════════════════════════════════════════════════════════════
MECHANIC 2: HOW REVERSE TIMESTAMP ENABLES "LATEST FIRST"
═══════════════════════════════════════════════════════════════

  Sensor "TRAFFIC-42159" writes at these times:
  
  Time                    │ Epoch millis      │ Reverse timestamp
  2024-01-15 10:00:00    │ 1705312800000     │ 9221666724054775807
  2024-01-15 10:00:05    │ 1705312805000     │ 9221666724054770807
  2024-01-15 10:00:10    │ 1705312810000     │ 9221666724054765807
  
  Row keys (sorted by HBase):
  
  [0x4B][0xA7F3219C][9221666724054765807]  ← 10:00:10 (NEWEST, smallest reverse_ts)
  [0x4B][0xA7F3219C][9221666724054770807]  ← 10:00:05
  [0x4B][0xA7F3219C][9221666724054775807]  ← 10:00:00 (OLDEST, largest reverse_ts)
  
  "Get latest reading":
    Scan with:
      startRow = [0x4B][0xA7F3219C][0[REDACTED:CREDIT_DEBIT_NUMBER]0]
      stopRow  = [0x4B][0xA7F3219D][0[REDACTED:CREDIT_DEBIT_NUMBER]0]
      limit    = 1
    → Returns FIRST row = 10:00:10 (newest!) ✓
    → Reads exactly 1 row, no scanning ✓

  "Get readings from 10:00:00 to 10:00:10":
    reverse(10:00:10) = 9221666724054765807
    reverse(10:00:00) = 9221666724054775807
    
    Scan with:
      startRow = [0x4B][0xA7F3219C][9221666724054765807]  (reverse of END time)
      stopRow  = [0x4B][0xA7F3219C][9221666724054775808]  (reverse of START time + 1)
    → Returns all 3 rows in reverse chronological order ✓

═══════════════════════════════════════════════════════════════
MECHANIC 3: HOW PRE-SPLITTING WORKS
═══════════════════════════════════════════════════════════════

  WITHOUT pre-splitting:
  
  Time 0:  [──────────── Region 1 ────────────────]
           All 100K writes/sec → ONE server
  
  Time 1:  Region 1 exceeds max size → auto-split
           [── Region 1a ──][── Region 1b ──]
           Still: most writes to 1b (newer keys)
  
  Time 2:  1b splits again
           [── 1a ──][── 1b-a ──][── 1b-b ──]
           STILL unbalanced — takes hours to stabilize

  WITH pre-splitting (256 regions from the start):
  
  Time 0:  [R0][R1][R2]...[R127]...[R255]
           100K writes/sec ÷ 256 regions = 390 writes/region/sec
           Distributed across 10 servers = ~10K writes/server/sec
           BALANCED FROM THE FIRST WRITE ✓

  SPLIT POINTS: byte array boundaries
  
  splits[0]  = [0x01]
  splits[1]  = [0x02]
  splits[2]  = [0x03]
  ...
  splits[254] = [0xFF]
  
  These are passed to HBaseAdmin.createTable(tableDesc, splits)

═══════════════════════════════════════════════════════════════
MECHANIC 4: HOW BLOOM FILTERS SPEED UP POINT LOOKUPS
═══════════════════════════════════════════════════════════════

  HBase stores data in HFiles (immutable sorted files):
  
  Region for salt=0x4B might have:
    HFile 1: rows from last compaction (millions of rows)
    HFile 2: recent flush (thousands of rows)
    HFile 3: most recent flush (hundreds of rows)
    MemStore: current writes (in memory)
  
  Point lookup for sensor "TRAFFIC-42159" latest reading:
  
  WITHOUT bloom filter:
    → Open HFile 1 → binary search for key → not found
    → Open HFile 2 → binary search → not found  
    → Open HFile 3 → binary search → FOUND
    → Check MemStore → maybe newer version
    3 disk reads for 1 lookup (each binary search reads index blocks)
  
  WITH bloom filter:
    → Check HFile 1 bloom: "Is this key in this file?" → NO → SKIP
    → Check HFile 2 bloom: → NO → SKIP
    → Check HFile 3 bloom: → YES → binary search → FOUND
    → Check MemStore
    1 disk read instead of 3 (bloom filters are in memory/block cache)

  BLOOM FILTER MEMORY COST:
  → ~10 bits per row key
  → 1 billion rows × 10 bits = ~1.2GB of bloom filter data
  → Stored in HFile, cached in block cache
  → WELL worth it for point lookup heavy workloads

⚠️ COMMON PITFALLS AND FAILURE MODES

PITFALL 1: "Writes are balanced but reads are slow"
→ CAUSE: Salt forces client to know which region to query
→ For "get latest for TRAFFIC-42159": compute salt, query ONE region ✓
→ For "get latest for ALL sensors": must query ALL 256 regions
→ SOLUTION: For multi-sensor queries, use parallel scans
→ Each thread scans one salt bucket → merge results client-side

PITFALL 2: "Hash collisions — two sensors map to same hash"
→ CAUSE: Murmur3 hash collision (rare but possible)
→ 500K sensors with 4-byte hash: ~0.003% collision probability
→ SOLUTION: Include original sensor_id in a column (d:sid)
→ On collision: both sensors' data interleaves (sorted by reverse_ts)
→ Client filters by d:sid column to separate

PITFALL 3: "Scans return data for wrong sensor"
→ CAUSE: Stop row not set correctly — scan bleeds into next sensor's data
→ Stop row must be sensor_hash + 1 (next sensor's hash boundary)
→ SOLUTION: Always set precise start and stop rows:
  startRow = [salt][sensor_hash][reverse(end_time)]
  stopRow  = [salt][sensor_hash + 1][0x00]

PITFALL 4: "Region server OOM during bulk load"
→ CAUSE: Too many concurrent writes to one region
→ MemStore fills up → flush → too many HFiles → compaction storm
→ SOLUTION: Tune MemStore size (hbase.hregion.memstore.flush.size)
→ For heavy write: increase to 256MB or 512MB
→ Also: use bulk load (Project 36) for historical data

PITFALL 5: "Compaction takes forever and blocks reads"
→ CAUSE: Time series data with TTL creates massive tombstone files
→ When TTL expires, cells marked as tombstones during compaction
→ Major compaction on a TB-scale region takes hours
→ SOLUTION: Use region-level TTL, schedule compaction during off-peak
→ Set hbase.hstore.compaction.max to limit files per compaction

PITFALL 6: "Pre-split regions become unbalanced over time"
→ CAUSE: Some sensors write more frequently than others
→ Deterministic salt means "heavy" sensors always go to same region
→ SOLUTION: Monitor region sizes, manually split hot regions
→ Or: use multiple salt bytes for finer distribution

2. ROW KEY SIZE IMPACT ON STORAGE

   Our key: 13 bytes (1 salt + 4 hash + 8 reverse_ts)
   
   Per cell storage: key + column family + qualifier + timestamp + value
   = 13 + 1 + 1 + 8 + 4 = 27 bytes overhead per cell
   
   With 7 data columns per reading:
   = 7 × 27 bytes overhead + 7 × avg 4 bytes value = 217 bytes per reading
   
   Daily: 8.6B × 217 bytes = ~1.87TB raw
   With SNAPPY compression (~4:1): ~467GB/day
   3-year retention: ~511TB compressed
   
   COMPARISON with string key (30 bytes):
   = 7 × (30 + 1 + 1 + 8 + 4) = 308 bytes per reading overhead
   = 8.6B × 308 = ~2.65TB raw/day
   SAVINGS from binary key: 780GB/day → 285TB over 3 years

// FILE 1: SensorRowKey.java — ROW KEY BUILDER AND PARSER
// ═══════════════════════════════════════════════════════════════
// Encapsulates all row key logic in one place.
// Every component that reads/writes HBase uses this class.

package com.smartcity.hbase;

import org.apache.hadoop.hbase.util.Bytes;
import org.apache.hadoop.hbase.util.MurmurHash3;

import java.nio.ByteBuffer;
import java.text.SimpleDateFormat;
import java.util.Date;
import java.util.TimeZone;

public class SensorRowKey {

    // ── Row key structure constants ──
    public static final int SALT_LENGTH = 1;
    public static final int SENSOR_HASH_LENGTH = 4;
    public static final int REVERSE_TS_LENGTH = 8;
    public static final int ROW_KEY_LENGTH = SALT_LENGTH + SENSOR_HASH_LENGTH + REVERSE_TS_LENGTH;  // 13 bytes

    // ── Salt configuration ──
    public static final int NUM_SALTS = 256;  // 1 byte = 256 possible values

    // ── Parsed fields ──
    private byte salt;
    private int sensorHash;
    private long reverseTimestamp;
    private long originalTimestamp;

    // ═══════════════════════════════════════════════
    // CONSTRUCTION: Build row key from components
    // ═══════════════════════════════════════════════

    /**
     * Create a row key for writing a sensor reading.
     *
     * @param sensorId  The sensor identifier string (e.g., "TRAFFIC-42159")
     * @param timestamp Epoch milliseconds of the reading
     * @return 13-byte row key as byte array
     */
    public static byte[] createRowKey(String sensorId, long timestamp) {
        byte salt = computeSalt(sensorId);
        int sensorHash = computeSensorHash(sensorId);
        long reverseTs = Long.MAX_VALUE - timestamp;

        ByteBuffer buffer = ByteBuffer.allocate(ROW_KEY_LENGTH);
        buffer.put(salt);
        buffer.putInt(sensorHash);
        buffer.putLong(reverseTs);

        return buffer.array();
    }

    /**
     * Compute deterministic salt from sensor ID.
     * Same sensor ALWAYS gets same salt → same region.
     * Distributes sensors uniformly across 256 buckets.
     */
    public static byte computeSalt(String sensorId) {
        return (byte)(sensorId.hashCode() & 0xFF);
    }

    /**
     * Compute 4-byte hash of sensor ID using Murmur3.
     * Murmur3 provides excellent distribution with low collision rate.
     * All readings for same sensor have identical hash → contiguous rows.
     */
    public static int computeSensorHash(String sensorId) {
        byte[] idBytes = Bytes.toBytes(sensorId);
        // MurmurHash3 from HBase utils — consistent across cluster
        return MurmurHash3.murmurhash3x8632(idBytes, 0, idBytes.length, 42);
    }

    /**
     * Compute reverse timestamp for "newest first" ordering.
     */
    public static long reverseTimestamp(long timestamp) {
        return Long.MAX_VALUE - timestamp;
    }

    /**
     * Recover original timestamp from reverse timestamp.
     */
    public static long originalTimestamp(long reverseTs) {
        return Long.MAX_VALUE - reverseTs;
    }

    // ═══════════════════════════════════════════════
    // SCAN KEYS: Build start/stop rows for queries
    // ═══════════════════════════════════════════════

    /**
     * Create start row for scanning a specific sensor.
     * Used for: "Get latest reading" (with limit=1)
     * Used for: "Get all readings" (without limit)
     *
     * Start row is the LOWEST possible key for this sensor.
     * Since reverse_ts for the NEWEST reading is smallest,
     * this returns newest data first.
     */
    public static byte[] createScanStartRow(String sensorId) {
        byte salt = computeSalt(sensorId);
        int sensorHash = computeSensorHash(sensorId);

        ByteBuffer buffer = ByteBuffer.allocate(ROW_KEY_LENGTH);
        buffer.put(salt);
        buffer.putInt(sensorHash);
        buffer.putLong(0L);  // Smallest reverse_ts = newest time

        return buffer.array();
    }

    /**
     * Create stop row for scanning a specific sensor.
     * Stop row is EXCLUSIVE — scan stops BEFORE this key.
     * We use sensor_hash + 1 to capture ALL of this sensor's rows.
     */
    public static byte[] createScanStopRow(String sensorId) {
        byte salt = computeSalt(sensorId);
        int sensorHash = computeSensorHash(sensorId);

        ByteBuffer buffer = ByteBuffer.allocate(SALT_LENGTH + SENSOR_HASH_LENGTH);
        buffer.put(salt);
        buffer.putInt(sensorHash + 1);  // Next sensor's hash boundary

        return buffer.array();
    }

    /**
     * Create start/stop rows for a TIME RANGE scan on one sensor.
     *
     * @param sensorId The sensor to query
     * @param startTime Start of time range (inclusive), epoch millis
     * @param endTime   End of time range (inclusive), epoch millis
     * @return byte[2][] where [0]=startRow, [1]=stopRow
     */
    public static byte[][] createTimeRangeScanKeys(
            String sensorId, long startTime, long endTime) {

        byte salt = computeSalt(sensorId);
        int sensorHash = computeSensorHash(sensorId);

        // IMPORTANT: Reverse the time logic for scan boundaries
        // Newest data (endTime) has SMALLEST reverse_ts → is the START of scan
        // Oldest data (startTime) has LARGEST reverse_ts → is the STOP of scan
        long scanStartReverseTs = reverseTimestamp(endTime);
        long scanStopReverseTs = reverseTimestamp(startTime) + 1;  // +1 for inclusive

        ByteBuffer startRow = ByteBuffer.allocate(ROW_KEY_LENGTH);
        startRow.put(salt);
        startRow.putInt(sensorHash);
        startRow.putLong(scanStartReverseTs);

        ByteBuffer stopRow = ByteBuffer.allocate(ROW_KEY_LENGTH);
        stopRow.put(salt);
        stopRow.putInt(sensorHash);
        stopRow.putLong(scanStopReverseTs);

        return new byte[][] { startRow.array(), stopRow.array() };
    }

    // ═══════════════════════════════════════════════
    // PARSING: Decode row key back to components
    // ═══════════════════════════════════════════════

    /**
     * Parse a binary row key into its components.
     */
    public static SensorRowKey parse(byte[] rowKey) {
        if (rowKey.length < ROW_KEY_LENGTH) {
            throw new IllegalArgumentException(
                "Row key too short: " + rowKey.length + " bytes (expected " + ROW_KEY_LENGTH + ")");
        }

        SensorRowKey parsed = new SensorRowKey();
        ByteBuffer buffer = ByteBuffer.wrap(rowKey);

        parsed.salt = buffer.get();
        parsed.sensorHash = buffer.getInt();
        parsed.reverseTimestamp = buffer.getLong();
        parsed.originalTimestamp = Long.MAX_VALUE - parsed.reverseTimestamp;

        return parsed;
    }

    // ═══════════════════════════════════════════════
    // PRE-SPLIT KEYS: For table creation
    // ═══════════════════════════════════════════════

    /**
     * Generate split points for pre-splitting the table into 256 regions.
     * Each region handles one salt value (0x00 to 0xFF).
     *
     * @return byte[][] array of 255 split points (256 regions)
     */
    public static byte[][] generateSplitKeys() {
        byte[][] splits = new byte[NUM_SALTS - 1][];

        for (int i = 1; i < NUM_SALTS; i++) {
            splits[i - 1] = new byte[] { (byte) i };
        }

        return splits;
    }

    // ═══════════════════════════════════════════════
    // GETTERS
    // ═══════════════════════════════════════════════

    public byte getSalt() { return salt; }
    public int getSensorHash() { return sensorHash; }
    public long getReverseTimestamp() { return reverseTimestamp; }
    public long getOriginalTimestamp() { return originalTimestamp; }

    public String getTimestampString() {
        SimpleDateFormat sdf = new SimpleDateFormat("yyyy-MM-dd HH:mm:ss.SSS");
        sdf.setTimeZone(TimeZone.getTimeZone("UTC"));
        return sdf.format(new Date(originalTimestamp));
    }

    @Override
    public String toString() {
        return String.format(
            "SensorRowKey{salt=0x%02X, sensorHash=0x%08X, timestamp=%s (%d)}",
            salt & 0xFF, sensorHash, getTimestampString(), originalTimestamp);
    }
}

package com.smartcity.hbase;

import org.apache.hadoop.conf.Configuration;

import org.apache.hadoop.hbase.util.Bytes;

import java.io.IOException;

public class SensorTableManager {

    // ── Table and column family names ──
    public static final TableName TABLE_NAME = TableName.valueOf("sensor_readings");
    public static final byte[] CF_DATA = Bytes.toBytes("d");       // Data column family
    public static final byte[] CF_META = Bytes.toBytes("m");       // Metadata column family

    // ── Data column qualifiers (short names save storage) ──
    public static final byte[] COL_SENSOR_ID = Bytes.toBytes("sid");
    public static final byte[] COL_TEMPERATURE = Bytes.toBytes("t");
    public static final byte[] COL_HUMIDITY = Bytes.toBytes("h");
    public static final byte[] COL_PRESSURE = Bytes.toBytes("p");
    public static final byte[] COL_AIR_QUALITY = Bytes.toBytes("aq");
    public static final byte[] COL_NOISE_LEVEL = Bytes.toBytes("nl");
    public static final byte[] COL_STATUS = Bytes.toBytes("s");

    // ── Metadata column qualifiers ──
    public static final byte[] COL_ZONE = Bytes.toBytes("zone");
    public static final byte[] COL_TYPE = Bytes.toBytes("type");

    // ── TTL: 3 years in seconds ──
    private static final int DATA_TTL = 3 * 365 * 24 * 60 * 60;  // ~94.6M seconds

    /**
     * Create the sensor_readings table with all optimizations.
     */
    public static void createTable(Configuration conf) throws IOException {
        try (Connection connection = ConnectionFactory.createConnection(conf);
             Admin admin = connection.getAdmin()) {

            if (admin.tableExists(TABLE_NAME)) {
                System.out.println("  Table already exists: " + TABLE_NAME);
                return;
            }

            System.out.println("  Creating table: " + TABLE_NAME);

            // ── Data column family configuration ──
            ColumnFamilyDescriptor dataCF = ColumnFamilyDescriptorBuilder
                .newBuilder(CF_DATA)
                // Bloom filter: ROW-level for point lookups
                // "Is this exact row key in this store file?"
                .setBloomFilterType(BloomType.ROW)
                // SNAPPY compression: fast decode, good compression
                .setCompressionType(Compression.Algorithm.SNAPPY)
                // Compaction compression (same as storage)
                .setCompactionCompressionType(Compression.Algorithm.SNAPPY)
                // TTL: auto-delete data older than 3 years
                .setTimeToLive(DATA_TTL)
                // Max versions: keep only latest version per cell
                .setMaxVersions(1)
                // Block size: 64KB (good for scan-heavy workload)
                .setBlocksize(65536)
                // Block cache: enable for read-heavy patterns
                .setBlockCacheEnabled(true)
                // In-memory: don't force into memory (too much data)
                .setInMemory(false)
                .build();

            // ── Metadata column family configuration ──
            ColumnFamilyDescriptor metaCF = ColumnFamilyDescriptorBuilder
                .newBuilder(CF_META)
                .setBloomFilterType(BloomType.ROW)
                .setCompressionType(Compression.Algorithm.SNAPPY)
                // No TTL for metadata — keep forever
                .setTimeToLive(HConstants.FOREVER)
                .setMaxVersions(1)
                .setBlockCacheEnabled(true)
                // Metadata is small — keep in memory for fast access
                .setInMemory(true)
                .build();

            // ── Table descriptor ──
            TableDescriptor tableDesc = TableDescriptorBuilder
                .newBuilder(TABLE_NAME)
                .setColumnFamily(dataCF)
                .setColumnFamily(metaCF)
                // Region split policy: based on size
                .setValue("hbase.hregion.max.filesize", "10737418240")  // 10GB per region
                // MemStore flush size
                .setValue("hbase.hregion.memstore.flush.size", "268435456")  // 256MB
                .build();

            // ── Pre-split into 256 regions ──
            byte[][] splitKeys = SensorRowKey.generateSplitKeys();

            admin.createTable(tableDesc, splitKeys);

            System.out.println("  ✅ Table created with " + (splitKeys.length + 1) + " pre-split regions");
            System.out.println("  Column families: 'd' (data, TTL=3yr), 'm' (metadata, no TTL)");
            System.out.println("  Bloom filter: ROW on both CFs");
            System.out.println("  Compression: SNAPPY on both CFs");
        }
    }

    /**
     * Create the sensor registry table.
     * Maps sensor_id → hash, zone, type for lookup support.
     */
    public static void createRegistryTable(Configuration conf) throws IOException {
        TableName registryTable = TableName.valueOf("sensor_registry");

                .newBuilder(registryTable)

                .build();

            System.out.println("  ✅ Sensor registry table created");
        }
    }

// FILE 3: SensorDataWriter.java — HIGH-THROUGHPUT WRITE CLIENT
// ═══════════════════════════════════════════════════════════════
// Writes sensor readings at 100K/sec using buffered mutations.
// Uses the row key design from SensorRowKey.

package com.smartcity.hbase;

import org.apache.hadoop.conf.Configuration;
import org.apache.hadoop.hbase.client.*;
import org.apache.hadoop.hbase.util.Bytes;

import java.io.IOException;
import java.util.ArrayList;
import java.util.List;

public class SensorDataWriter implements AutoCloseable {

    private final Connection connection;
    private final BufferedMutator mutator;
    private long writtenCount = 0;

    // ── Write buffer configuration ──
    private static final long WRITE_BUFFER_SIZE = 8 * 1024 * 1024;  // 8MB buffer
    // Buffer collects mutations and flushes in batch → higher throughput

    public SensorDataWriter(Configuration conf) throws IOException {
        this.connection = ConnectionFactory.createConnection(conf);

        BufferedMutatorParams params = new BufferedMutatorParams(SensorTableManager.TABLE_NAME)
            .writeBufferSize(WRITE_BUFFER_SIZE)
            .listener((e, mutator) -> {
                // Error callback — log failed mutations
                System.err.printf("  🔴 Write failed: %s%n", e.getMessage());
                for (int i = 0; i < e.getNumExceptions(); i++) {
                    System.err.printf("    Row: %s, Error: %s%n",
                        Bytes.toHex(e.getRow(i).getRow()),
                        e.getCause(i).getMessage());
                }
            });

        this.mutator = connection.getBufferedMutator(params);
    }

    /**
     * Write a single sensor reading.
     * Buffered internally — actual write happens when buffer fills.
     */
    public void writeReading(
            String sensorId, long timestamp,
            float temperature, float humidity, float pressure,
            int airQuality, float noiseLevel, byte status) throws IOException {

        // ── Build row key using our design ──
        byte[] rowKey = SensorRowKey.createRowKey(sensorId, timestamp);

        // ── Create Put mutation ──
        Put put = new Put(rowKey);

        // Data columns (column family "d")
        put.addColumn(SensorTableManager.CF_DATA, SensorTableManager.COL_SENSOR_ID,
                      Bytes.toBytes(sensorId));
        put.addColumn(SensorTableManager.CF_DATA, SensorTableManager.COL_TEMPERATURE,
                      Bytes.toBytes(temperature));
        put.addColumn(SensorTableManager.CF_DATA, SensorTableManager.COL_HUMIDITY,
                      Bytes.toBytes(humidity));
        put.addColumn(SensorTableManager.CF_DATA, SensorTableManager.COL_PRESSURE,
                      Bytes.toBytes(pressure));
        put.addColumn(SensorTableManager.CF_DATA, SensorTableManager.COL_AIR_QUALITY,
                      Bytes.toBytes(airQuality));
        put.addColumn(SensorTableManager.CF_DATA, SensorTableManager.COL_NOISE_LEVEL,
                      Bytes.toBytes(noiseLevel));
        put.addColumn(SensorTableManager.CF_DATA, SensorTableManager.COL_STATUS,
                      new byte[] { status });

        // ── Buffer the mutation ──
        mutator.mutate(put);
        writtenCount++;
    }

    /**
     * Write a batch of readings efficiently.
     * Pre-sorts by row key for optimal region-server locality.
     */
    public void writeBatch(List<SensorReading> readings) throws IOException {
        List<Mutation> mutations = new ArrayList<>(readings.size());

        for (SensorReading reading : readings) {
            byte[] rowKey = SensorRowKey.createRowKey(
                reading.getSensorId(), reading.getTimestamp());

            Put put = new Put(rowKey);
            put.addColumn(SensorTableManager.CF_DATA, SensorTableManager.COL_SENSOR_ID,
                          Bytes.toBytes(reading.getSensorId()));
            put.addColumn(SensorTableManager.CF_DATA, SensorTableManager.COL_TEMPERATURE,
                          Bytes.toBytes(reading.getTemperature()));
            put.addColumn(SensorTableManager.CF_DATA, SensorTableManager.COL_HUMIDITY,
                          Bytes.toBytes(reading.getHumidity()));
            put.addColumn(SensorTableManager.CF_DATA, SensorTableManager.COL_PRESSURE,
                          Bytes.toBytes(reading.getPressure()));
            put.addColumn(SensorTableManager.CF_DATA, SensorTableManager.COL_STATUS,
                          new byte[] { (byte) reading.getStatus() });

            mutations.add(put);
        }

        mutator.mutate(mutations);
        writtenCount += readings.size();
    }

    /**
     * Register a sensor in the registry table.
     */
    public void registerSensor(String sensorId, String zone, String type) throws IOException {
        try (Table registry = connection.getTable(
                org.apache.hadoop.hbase.TableName.valueOf("sensor_registry"))) {

            Put put = new Put(Bytes.toBytes(sensorId));
            put.addColumn(Bytes.toBytes("i"), Bytes.toBytes("hash"),
                          Bytes.toBytes(SensorRowKey.computeSensorHash(sensorId)));
            put.addColumn(Bytes.toBytes("i"), Bytes.toBytes("salt"),
                          new byte[] { SensorRowKey.computeSalt(sensorId) });
            put.addColumn(Bytes.toBytes("i"), Bytes.toBytes("zone"),
                          Bytes.toBytes(zone));
            put.addColumn(Bytes.toBytes("i"), Bytes.toBytes("type"),
                          Bytes.toBytes(type));

            registry.put(put);
        }
    }

    /**
     * Flush any buffered mutations to HBase.
     */
    public void flush() throws IOException {
        mutator.flush();
    }

    public long getWrittenCount() {
        return writtenCount;
    }

public class SensorDataReader implements AutoCloseable {

    private final Table table;
    private final Table registry;

    private static final SimpleDateFormat DATE_FORMAT = new SimpleDateFormat("yyyy-MM-dd HH:mm:ss.SSS");
    static {
        DATE_FORMAT.setTimeZone(TimeZone.getTimeZone("UTC"));

    public SensorDataReader(Configuration conf) throws IOException {
        this.connection = ConnectionFactory.createConnection(conf);
        this.table = connection.getTable(SensorTableManager.TABLE_NAME);
        this.registry = connection.getTable(
            org.apache.hadoop.hbase.TableName.valueOf("sensor_registry"));

    // ═══════════════════════════════════════════════
    // PATTERN 1: Get latest reading for one sensor
    // Expected: < 10ms
    // ═══════════════════════════════════════════════

    /**
     * Get the most recent reading for a specific sensor.
     * 
     * HOW IT WORKS:
     * → Compute salt + hash from sensor_id (deterministic)
     * → Scan starting from [salt][hash][0x00...] (smallest reverse_ts = newest)
     * → Limit to 1 result → returns immediately after first row
     * → Because of reverse timestamp: first row IS the latest reading
     */
    public Map<String, Object> getLatestReading(String sensorId) throws IOException {
        long startNanos = System.nanoTime();

        byte[] startRow = SensorRowKey.createScanStartRow(sensorId);
        byte[] stopRow = SensorRowKey.createScanStopRow(sensorId);

        Scan scan = new Scan()
            .withStartRow(startRow)
            .withStopRow(stopRow)
            .setLimit(1)                          // Only need the FIRST row (newest)
            .addFamily(SensorTableManager.CF_DATA) // Only read data columns
            .setCaching(1)                         // Only expect 1 row
            .setSmall(true);                       // Small scan optimization

        Map<String, Object> result = null;

        try (ResultScanner scanner = table.getScanner(scan)) {
            Result row = scanner.next();
            if (row != null && !row.isEmpty()) {
                result = parseResult(row);
            }
        }

        long elapsedMicros = (System.nanoTime() - startNanos) / 1000;

        if (result != null) {
            result.put("_query_time_us", elapsedMicros);
        }

        return result;
    }

    // ═══════════════════════════════════════════════
    // PATTERN 2: Time range scan for one sensor
    // Expected: < 500ms
    // ═══════════════════════════════════════════════

    /**
     * Get all readings for a sensor within a time range.
     *
     * HOW IT WORKS:
     * → Compute time range scan keys (reverse timestamps)
     * → startRow = [salt][hash][reverse(endTime)]   (newest in range)
     * → stopRow  = [salt][hash][reverse(startTime)]  (oldest in range)
     * → Sequential scan — all rows are contiguous for this sensor
     * → Returns in reverse chronological order (newest first)
     */
    public List<Map<String, Object>> getTimeRange(
            String sensorId, long startTime, long endTime) throws IOException {
        
        long startNanos = System.nanoTime();

        byte[][] scanKeys = SensorRowKey.createTimeRangeScanKeys(
            sensorId, startTime, endTime);

        Scan scan = new Scan()
            .withStartRow(scanKeys[0])
            .withStopRow(scanKeys[1])
            .addFamily(SensorTableManager.CF_DATA)
            .setCaching(1000)    // Prefetch 1000 rows per RPC (tune based on row size)
            .setBatch(0);        // No column batching — read all columns per row

        List<Map<String, Object>> results = new ArrayList<>();

        try (ResultScanner scanner = table.getScanner(scan)) {
            for (Result row : scanner) {
                if (row != null && !row.isEmpty()) {
                    results.add(parseResult(row));
                }
            }
        }

        long elapsedMs = (System.nanoTime() - startNanos) / 1_000_000;

        System.out.printf("  Time range scan: %s [%s — %s] → %d readings in %dms%n",
            sensorId,
            DATE_FORMAT.format(new Date(startTime)),
            DATE_FORMAT.format(new Date(endTime)),
            results.size(), elapsedMs);

        return results;
    }

    // ═══════════════════════════════════════════════
    // PATTERN 3: Latest readings for all sensors in a zone
    // Expected: < 2 seconds
    // ═══════════════════════════════════════════════

    /**
     * Get latest reading for every sensor in a geographic zone.
     *
     * HOW IT WORKS:
     * → Step 1: Query sensor_registry for all sensors in zone
     * → Step 2: For each sensor, run getLatestReading() in PARALLEL
     * → Uses thread pool to parallelize across salt regions
     * → Each individual lookup is < 10ms
     * → 2000 sensors × 10ms / 20 threads = ~1 second
     */
    public List<Map<String, Object>> getZoneLatest(String zone) throws IOException {
        long startNanos = System.nanoTime();

        // Step 1: Get all sensors in zone from registry
        List<String> sensorIds = getSensorsInZone(zone);
        System.out.printf("  Zone '%s': %d sensors found%n", zone, sensorIds.size());

        // Step 2: Parallel latest-reading lookups
        int threadCount = Math.min(20, sensorIds.size());
        ExecutorService executor = Executors.newFixedThreadPool(threadCount);
        List<Future<Map<String, Object>>> futures = new ArrayList<>();

        for (String sensorId : sensorIds) {
            futures.add(executor.submit(() -> getLatestReading(sensorId)));
        }

        // Step 3: Collect results
        List<Map<String, Object>> results = new ArrayList<>();
        for (Future<Map<String, Object>> future : futures) {
            try {
                Map<String, Object> reading = future.get(5, TimeUnit.SECONDS);
                if (reading != null) {
                    results.add(reading);
                }
            } catch (Exception e) {
                System.err.printf("  ⚠️ Failed to get reading: %s%n", e.getMessage());
            }
        }

        executor.shutdown();

        long elapsedMs = (System.nanoTime() - startNanos) / 1_000_000;
        System.out.printf("  Zone query: %d readings in %dms (%.1f ms/sensor)%n",
            results.size(), elapsedMs,
            (double) elapsedMs / Math.max(1, sensorIds.size()));

        return results;
    }

    /**
     * Look up all sensor IDs in a zone from the registry table.
     */
    private List<String> getSensorsInZone(String zone) throws IOException {
        List<String> sensorIds = new ArrayList<>();

        Scan scan = new Scan()
            .addColumn(Bytes.toBytes("i"), Bytes.toBytes("zone"));

        try (ResultScanner scanner = registry.getScanner(scan)) {
            for (Result row : scanner) {
                byte[] zoneValue = row.getValue(Bytes.toBytes("i"), Bytes.toBytes("zone"));
                if (zoneValue != null && Bytes.toString(zoneValue).equals(zone)) {
                    sensorIds.add(Bytes.toString(row.getRow()));
                }
            }
        }

        return sensorIds;
    }

    // PATTERN 4 (read side): Verify write distribution
    // ═══════════════════════════════════════════════

    /**
     * Count readings per salt bucket to verify distribution.
     * Used for monitoring write balance across regions.
     */
    public Map<Integer, Long> getDistributionBySalt(int sampleLimit) throws IOException {
        Map<Integer, Long> distribution = new TreeMap<>();

        for (int salt = 0; salt < SensorRowKey.NUM_SALTS; salt++) {
            byte[] startRow = new byte[] { (byte) salt };
            byte[] stopRow = (salt < 255) ? new byte[] { (byte)(salt + 1) } : null;

            Scan scan = new Scan()
                .withStartRow(startRow);
            if (stopRow != null) {
                scan.withStopRow(stopRow);
            }
            scan.setLimit(sampleLimit)
                .setCaching(sampleLimit)
                .setFilter(new org.apache.hadoop.hbase.filter.FirstKeyOnlyFilter());

            long count = 0;
            try (ResultScanner scanner = table.getScanner(scan)) {
                for (Result row : scanner) {
                    count++;
                }
            }

            if (count > 0) {
                distribution.put(salt, count);
            }
        }

        return distribution;
    }

    // RESULT PARSING
    // ═══════════════════════════════════════════════

    /**
     * Parse an HBase Result into a readable map.
     */
    private Map<String, Object> parseResult(Result row) {
        Map<String, Object> reading = new LinkedHashMap<>();

        // Parse row key
        SensorRowKey key = SensorRowKey.parse(row.getRow());
        reading.put("timestamp", key.getTimestampString());
        reading.put("epoch_ms", key.getOriginalTimestamp());
        reading.put("salt", String.format("0x%02X", key.getSalt() & 0xFF));
        reading.put("sensor_hash", String.format("0x%08X", key.getSensorHash()));

        // Parse data columns
        byte[] sid = row.getValue(SensorTableManager.CF_DATA, SensorTableManager.COL_SENSOR_ID);
        if (sid != null) reading.put("sensor_id", Bytes.toString(sid));

        byte[] temp = row.getValue(SensorTableManager.CF_DATA, SensorTableManager.COL_TEMPERATURE);
        if (temp != null) reading.put("temperature", Bytes.toFloat(temp));

        byte[] hum = row.getValue(SensorTableManager.CF_DATA, SensorTableManager.COL_HUMIDITY);
        if (hum != null) reading.put("humidity", Bytes.toFloat(hum));

        byte[] pres = row.getValue(SensorTableManager.CF_DATA, SensorTableManager.COL_PRESSURE);
        if (pres != null) reading.put("pressure", Bytes.toFloat(pres));

        byte[] aq = row.getValue(SensorTableManager.CF_DATA, SensorTableManager.COL_AIR_QUALITY);
        if (aq != null) reading.put("air_quality", Bytes.toInt(aq));

        byte[] nl = row.getValue(SensorTableManager.CF_DATA, SensorTableManager.COL_NOISE_LEVEL);
        if (nl != null) reading.put("noise_level", Bytes.toFloat(nl));

        byte[] status = row.getValue(SensorTableManager.CF_DATA, SensorTableManager.COL_STATUS);
        if (status != null) {
            int s = status[0] & 0xFF;
            String[] statusNames = {"NORMAL", "WARNING", "CRITICAL", "OFFLINE"};
            reading.put("status", s < statusNames.length ? statusNames[s] : "UNKNOWN");
        }

        return reading;
    }

    @Override
    public void close() throws IOException {
        table.close();
        registry.close();
        connection.close();
    }
